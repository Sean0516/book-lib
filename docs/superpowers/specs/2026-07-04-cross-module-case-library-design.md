# 跨模块完整案例库扩展设计

## 背景

现有深度精选区已经包含 6 篇 JVM 与并发深度文章，但完整教学案例只有“Full GC 导致
TP99 飙升”一篇。知识讲解能够支持单点问答，案例数量却不足以训练候选人完成现象判断、
证据收集、根因定位、止损修复和架构权衡的完整过程。

本次扩展建立独立案例库，覆盖 JVM、数据库、Redis、消息队列和系统设计五个模块。现有
Full GC 案例继续保留，新增 15 篇完整案例后，全站共有 16 篇完整案例。

## 目标与范围

- 五个模块各新增 3 篇完整案例，共新增 15 篇。
- 每个模块包含 2 个线上故障案例和 1 个架构设计案例。
- 每篇深度对齐现有 Full GC 案例，不使用只有结论的精简模板。
- 案例与相关知识文章双向链接，并进入导航、侧边栏和本地搜索。
- 案例明确为基于常见生产模式构造的教学材料，不虚构真实公司经历。

本阶段不引入登录、学习进度、在线答题、后端服务或自动评分系统，也不同时扩展 RAG 和
Agent 案例。

## 设计选择

采用独立案例库，而不是把案例附加到知识文章末尾或用一个综合项目串联所有主题。

理由：

1. 每篇案例可以独立搜索、收藏和在移动端阅读。
2. 故障案例与架构设计案例可以使用各自适合的叙事结构。
3. 知识文章保持聚焦，通过双向链接建立“原理—实践”关系。
4. 后续新增模块时，只需增加目录和入口，不会让现有文章持续膨胀。

## 案例选题

| 模块 | 线上故障案例一 | 线上故障案例二 | 架构设计案例 |
|---|---|---|---|
| JVM | CPU 持续 100% 排查 | 线程池耗尽与服务雪崩 | 低延迟 Java 服务设计 |
| 数据库 | 慢 SQL 导致接口超时 | 死锁与锁等待激增 | 高并发订单存储设计 |
| Redis | 热 Key 导致单节点过载 | 缓存击穿与数据不一致 | 高可用缓存体系设计 |
| 消息队列 | 消息积压与消费延迟 | 消息重复、丢失与乱序 | 可靠事件驱动架构设计 |
| 系统设计 | 秒杀流量击穿系统 | 下游故障引发级联雪崩 | 高并发订单系统设计 |

数据库案例以 MySQL 为主要语境，Redis 案例覆盖 Cluster 和本地缓存，消息队列案例使用
Kafka/RocketMQ 共通原理并在实现差异处明确标注，避免把单一产品行为错误推广为通用
结论。

## 内容结构

### 线上故障案例

每篇故障案例包含：

1. 事故背景：业务链路、系统规模、容量基线和变更背景。
2. 用户影响：错误率、延迟、吞吐和影响范围。
3. 时间线：发现、升级、止损、恢复和复盘节点。
4. 分阶段证据：监控、日志、线程栈、SQL、执行计划或配置。
5. 假设树：候选原因、验证方法和排除证据。
6. 定位过程：可执行命令、查询、日志样例或伪代码。
7. 根因：触发条件、故障机制和传播路径。
8. 临时止损：收益、风险、回滚条件和操作顺序。
9. 长期修复：代码、配置、架构和流程改进。
10. 指标验收：恢复标准、观察窗口和防复发告警。
11. 面试推演：三级追问、评分标准和常见失分点。

### 架构设计案例

每篇架构设计案例包含：

1. 需求澄清和明确假设。
2. 流量、存储、连接和带宽容量估算。
3. 核心数据模型与接口。
4. 组件关系、请求链路和数据流。
5. 至少两种候选方案及其取舍。
6. 一致性、幂等、降级和故障恢复设计。
7. 热点、扩容、发布和迁移路径。
8. 可观测指标、SLO 和容量水位。
9. 面试官追问、评分标准和常见失分点。

两类案例都必须至少包含一张 Mermaid 图、一组可执行命令或伪代码，以及一张指标或方案
对比表。案例数据在背景、时间线、根因和验收章节中必须保持一致。

## 阅读与关联体验

- 案例首页按模块和类型浏览，并标注建议前置知识。
- 每篇案例顶部提供难度、适用岗位、建议演练时长和训练目标。
- 正文采用“先给现象、逐步披露证据、最后给答案”的顺序，读者可以暂停并自行判断。
- 页面底部链接相关深度文章、同模块案例和下一篇建议。
- 相关知识文章增加“实战案例”入口，案例反向链接原理文章。
- 本地搜索收录案例正文，但继续排除 `docs/superpowers/` 内部设计和计划文档。

## 移动端要求

- 表格和代码块允许横向滚动，不挤压正文宽度。
- Mermaid 图在窄屏保持可缩放或横向滚动。
- 时间线、证据和答案使用清晰的二三级标题，避免单段过长。
- 正文字号和行距沿用现有移动端优化。
- 打印样式保持案例标题、表格和代码块可读。

## 验收与测试

### 内容验收

- 新增文件数量为 15 篇案例正文和 6 个入口页。
- 每个模块严格包含 2 个故障案例和 1 个架构设计案例。
- 每篇包含图、命令或伪代码、指标或方案表、追问和评分标准。
- 不出现未完成占位符、虚构公司背书或相互矛盾的数据。
- 故障案例能从现象和证据推导到根因，止损措施不替代长期修复。

### 工程验收

- `npm run build` 和 GitHub Pages 的 `/book-lib/` 基础路径构建通过。
- VitePress 不产生死链。
- 导航、侧边栏、上一篇/下一篇和知识文章双向链接可用。
- 本地搜索可以按案例标题、根因和技术关键词命中。
- 桌面端和 390 像素宽移动端检查表格、代码块、Mermaid 图和侧边栏。

## 实施顺序

按模块分批交付，每批完成入口、3 篇案例、关联链接和构建验证：

1. JVM
2. 数据库
3. Redis
4. 消息队列
5. 系统设计
6. 全站导航、搜索和移动端回归

这样可以在每个模块完成后立即检查内容质量，避免 15 篇全部写完才发现模板、数据或阅读
体验存在系统性问题。
