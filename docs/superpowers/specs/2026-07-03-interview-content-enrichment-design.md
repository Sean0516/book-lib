# 面试知识库深度内容升级设计

## 背景

现有知识库已经覆盖基础 8、架构 12、AI 6 共 26 个模块、520 道题。覆盖广度足够，
但内容普遍使用相同的“结论—答案—追问—指标—易错点”模板。审计结果显示，核心模块中
没有代码块、表格或 Mermaid 图，案例也较少；因此当前主要问题不是题目数量，而是缺少
可视化原理、工程证据、真实取舍和连续追问。

## 目标

第一阶段建立“广度题库 + 深度精选”的双层知识体系：

- 保留 520 道原题，继续承担快速检索和知识覆盖。
- 新增 8 个深度模块，每个模块精选 6 道题，共 48 道。
- 每个深度模块增加 1 个完整教学案例，共 8 个。
- 每道题同时支持面试冲刺和系统学习。
- 内容覆盖高级 Java 后端、架构师/技术负责人和 AI 架构师三条路线。

第一阶段不重写其余 472 道题，不把所有原文扩成长篇课程，也不增加需要后端服务的学习
进度、账户或在线答题功能。

## 设计选择

采用独立 `deep-dives/` 内容区，而不是直接扩写原题。

理由：

1. 原题库保留检索速度和完整覆盖。
2. 深度内容可以使用更长篇幅、图表和代码，而不让原模块膨胀到难以阅读。
3. 48 道题可以按统一质量标准逐步交付。
4. 后续扩展到更多题时，只需新增深度文章并从原题建立关联。

## 双层文章模板

### 第一层：面试速答

文章开头在首屏附近提供：

- 适用岗位：高级后端、架构师、AI 架构师。
- 难度：基础追问、进阶、架构场景。
- 建议回答时长。
- 60–90 秒可直接口述的答案。
- 面试官评分点。
- 一句话记忆锚点。
- 常见失分表达。

速答必须自然、可口述，不使用只有书面阅读才成立的长句。场景题使用固定五段逻辑：

1. 目标与约束
2. 核心设计
3. 权衡与替代
4. 风险与兜底
5. 指标与验证

### 第二层：深度展开

每篇文章根据主题选择以下内容，避免为了套模板而机械重复：

1. 问题背景、假设和适用边界。
2. 核心机制图、时序图或数据流图。
3. 原理拆解和关键状态变化。
4. Java、SQL、配置或伪代码示例。
5. 方案对比表。
6. 具体业务场景和输入数据。
7. 设计取舍、替代方案和不适用条件。
8. 故障模式、降级、回滚和恢复。
9. 指标定义、计算方式、参考基线和验证方法。
10. 面试官三级追问树。
11. 自测题及 25 分评分标准。

不要求每篇机械出现 11 个同名标题，但必须覆盖速答、原理、工程落地、追问和验证五类
信息。

## 48 道深度题

### JVM 与并发

1. G1、ZGC、Shenandoah 如何选，如何建立 GC 调优基线？
2. 线上内存泄漏如何从指标定位到对象引用链？
3. 线程池参数如何根据 CPU、IO、下游容量和延迟目标确定？
4. 锁竞争如何定位，什么时候用分段、无锁或异步化？
5. JMM、happens-before、volatile 与锁如何解释可见性？
6. CPU 飙高如何关联线程、堆栈、GC 和热点代码？

### 数据库

1. 联合索引如何设计，如何用执行计划验证？
2. MVCC、Read View、Undo Log 与隔离级别如何协作？
3. 慢 SQL 如何从业务现象定位到 SQL、索引和数据分布？
4. 锁等待和死锁如何复现、定位并治理？
5. 分库分表的拆分键、路由、扩容和全局查询如何设计？
6. 大表在线迁移如何做到可回滚、可校验和低风险切换？

### 缓存与消息

1. 缓存与数据库一致性如何控制不一致窗口？
2. 热点 Key、大 Key 和缓存击穿如何发现与治理？
3. 缓存雪崩时如何分层止损并恢复？
4. 消息如何做到可证明的不丢失？
5. 重复消费如何通过业务幂等处理？
6. 消息积压如何估算清理时间、扩容并避免二次冲击？

### 系统设计与容量规划

1. 如何从 DAU、峰值系数和请求链路估算 QPS？
2. 如何将 QPS 转换为 CPU、线程、连接和存储容量？
3. 压测如何构造流量模型并识别伪瓶颈？
4. 热点流量如何识别、隔离和削峰？
5. 容量不足时如何设计水平扩容、垂直扩容和降级顺序？
6. 如何定义安全水位、容量预算和成本边界？

### 分布式一致性

1. CAP 和一致性模型如何映射到真实业务决策？
2. 幂等键、状态机和去重记录如何协作？
3. 重试如何避免放大流量和重复副作用？
4. 分布式锁如何处理续租、主从切换和 fencing token？
5. 本地消息表、事务消息、TCC、Saga 如何选择？
6. 补偿状态机如何处理悬挂、空回滚和人工介入？

### 稳定性与高可用

1. SLI、SLO、错误预算如何落到发布和治理决策？
2. 限流、熔断、降级如何分层配合？
3. 故障隔离如何覆盖流量、线程池、连接池、租户和数据？
4. 同城双活、异地多活如何选择并治理数据冲突？
5. RTO、RPO 如何转化为备份、复制和演练方案？
6. 线上事故如何完成发现、止血、恢复、复盘和防复发？

### RAG

1. 文档切块如何按语义、文档类型和评测结果设计？
2. 向量检索、关键词检索和混合检索如何选择？
3. Reranker 放在哪里，如何评估收益与成本？
4. 多租户权限如何贯穿采集、索引、召回和生成？
5. RAG 如何建立检索、生成和端到端三层评测？
6. 企业 RAG 如何设计更新、缓存、降级、观测和成本治理？

### Agent

1. Agent 与确定性 Workflow 的边界如何判断？
2. 工具协议、参数校验、超时、重试和幂等如何设计？
3. 短期记忆、长期记忆和业务状态如何分层？
4. 最小权限、审批、沙箱和审计如何防止工具越权？
5. Agent 如何检测循环、纠偏、终止和人工接管？
6. Agent 的轨迹、工具调用、成本和任务成功率如何观测？

## 8 个教学案例

案例明确标注为“基于常见生产模式构造的教学案例”，不虚构真实公司背书。

### Full GC 导致 TP99 飙升

给出流量、堆大小、晋升速率、停顿时间和时间线；展示指标定位、GC 日志、堆转储引用链、
止血动作和长期修复。

### 订单慢 SQL 与索引治理

给出表结构、数据分布、SQL、执行计划和线上指标；展示索引选择、回表、锁影响、灰度和
回滚。

### Redis 热点失效击穿数据库

给出 Key 访问分布、过期时间、DB 容量和故障曲线；比较互斥锁、逻辑过期、预热和多级
缓存。

### 大促容量规划

从业务目标推导峰值 QPS、依赖放大系数、线程连接容量、安全水位、压测模型和降级预案。

### 跨服务订单一致性

覆盖订单、库存、支付、消息链路；比较事务消息、Saga 和补偿状态机，展示异常状态与人工
介入。

### 支付链路稳定性改造

给出 SLO、依赖拓扑、故障传播路径；设计隔离、限流、熔断、降级、多活和演练。

### 企业知识库 RAG 从 PoC 到生产

覆盖采集、切块、索引、混合检索、权限、引用、评测、更新和成本，并给出失败查询样本。

### Agent 工具越权与执行失控

展示 Prompt Injection、错误参数、循环调用和高风险工具执行链；设计权限、审批、预算、
终止和审计。

## 指标写作规范

指标不能只列名称。每个关键指标至少说明：

- 定义：它测量什么。
- 计算方式：分子、分母或统计口径。
- 数据来源：日志、指标、Trace、数据库或评测集。
- 参考基线：使用“示例基线”并说明需按业务校准。
- 决策作用：指标变化后采取什么动作。

例如：

```text
缓存命中率 = 命中请求数 / 可缓存请求数
示例基线：稳定期 > 95%，核心热点 > 99%
数据来源：网关或缓存客户端指标
决策：低于基线时检查 Key 设计、TTL、穿透和版本切换
```

不得把示例数字描述为普遍适用的行业标准。

## 代码、图表和案例规范

- 代码必须能说明当前问题，不放与主题无关的大段样板代码。
- Java 示例优先使用现代、清晰、可独立理解的代码。
- SQL 示例同时给表结构、查询条件和执行计划关注点。
- 配置示例说明版本和关键参数语义。
- Mermaid 图控制节点数量，手机端仍应可横向查看。
- 对比表必须包含“适用场景、收益、代价、风险”。
- 案例使用一致的输入数字，图、公式和结论不能互相矛盾。
- 不引用无法验证的事故细节，不冒充真实生产经历。

## 追问和评分规范

每道题至少设置三级追问：

1. 原理追问：确认基础机制是否理解。
2. 工程追问：要求给出方案、边界和异常处理。
3. 架构追问：要求权衡替代方案并量化验证。

评分沿用五维 25 分模型：

- 正确性：5 分
- 深度：5 分
- 权衡推理：5 分
- 表达结构：5 分
- 指标与可运维性：5 分

文章给出评分锚点，不为所有回答提供唯一措辞。

## 交付阶段

### 阶段 1：模板与样板模块

先完成 JVM 与并发模块的 6 道题和 1 个案例，用于验证模板、图表、代码和手机阅读体验。

### 阶段 2：经典后端与架构

完成数据库、缓存与消息、容量规划、分布式一致性、稳定性，共 30 道题和 5 个案例。

### 阶段 3：AI 架构

完成 RAG、Agent，共 12 道题和 2 个案例。

每个阶段单独构建、审校和提交；上一阶段通过质量检查后再继续，避免将同一质量问题复制
到全部 48 篇。

## 验收标准

第一阶段整体完成时：

- 深度精选包含 48 道题和 8 个案例。
- 每个模块至少有 2 张 Mermaid 图和 3 段代码、SQL 或配置。
- 每道题至少有 1 个包含“适用场景、收益、代价、风险”的对比表。
- 每道题有 60–90 秒速答和至少三级追问。
- 每个案例包含背景数据、时间线或状态流、证据、决策、实施、验证和复盘。
- 所有示例指标注明口径，示例基线不冒充通用标准。
- 原题和深度文章双向可达。
- VitePress 本地搜索可检索新内容。
- `npm run build` 在默认路径与 `/book-lib/` 部署路径下均成功。
- 390px 手机视口下标题、表格、代码和 Mermaid 图可读或可横向滚动，页面本身不溢出。
