# 基础模块4 - 计算机基础（OS / 网络 / Linux） - 标准答案库

## 1. 用户态和内核态区别？

**核心结论**  
用户态和内核态的本质区别在于权限等级不同，用户态负责业务逻辑执行，内核态负责系统资源和硬件控制。

**标准答案**  
操作系统通常会把 CPU 执行权限分成用户态和内核态。用户态权限较低，应用程序大部分代码都运行在这里，不能直接操作硬件和关键系统资源；内核态权限高，可以访问设备、管理内存、调度线程、处理系统调用。之所以要这样分层，是为了安全和稳定，如果所有程序都直接操作硬件，一旦写错就可能拖垮整个系统。工程上，应用程序要做文件读写、网络收发、进程调度相关操作时，往往都要通过系统调用从用户态切到内核态。面试里如果能补一句“很多性能问题其实就是用户态和内核态频繁切换的成本问题”，会更完整。

**追问点答案**  
- 为什么不能让应用一直跑在内核态？  
因为那样隔离性太差，普通程序出错就可能破坏整个系统。  
- 系统调用为什么会有性能成本？  
因为它涉及权限切换、上下文保存恢复以及内核处理路径。  
- Java 程序平时主要跑在哪？  
绝大多数业务代码都在用户态，只有遇到系统调用时才进入内核态。

**原理补充**  
这道题的关键不是记住两个名词，而是理解操作系统为什么要做权限隔离。用户态和内核态本质上是在平衡“开放应用执行能力”和“保护系统核心资源”。

**架构扩展**  
线上服务很多性能问题，表面上看是“CPU 高”或者“响应慢”，根因却是大量系统调用和频繁态切换。比如高频小包网络调用、频繁磁盘 IO、日志刷盘过重，都会让应用不断进出内核态。成熟系统通常会通过批量 IO、异步 IO、缓冲、零拷贝等方式减少切换成本，而不是只盯业务代码。

**工程指标**  
- system CPU 占比：看 CPU 时间是不是大量花在内核态。  
- 上下文切换次数：判断系统调用和调度成本是否过高。  
- 系统调用热点：看是不是某类 IO 操作过于频繁。  
- 磁盘 / 网络中断负载：辅助判断内核态工作是否异常繁忙。  
- 请求 TP99：很多内核态开销最终会放大到尾延迟。

**易错点**  
- 只说“权限不同”但不会解释为什么  
- 不知道系统调用和态切换的关系  
- 不会把它和实际性能问题联系起来

## 2. 上下文切换为什么贵？

**核心结论**  
上下文切换贵，不是因为“切一次很慢”，而是因为它会中断当前执行、保存恢复现场，还会破坏 CPU 缓存局部性。

**标准答案**  
上下文切换指的是 CPU 从当前执行单元切换到另一个执行单元时，需要保存当前上下文并恢复目标上下文。这里的上下文包括寄存器、程序计数器、栈指针、调度状态，进程切换时还可能涉及地址空间和页表切换。它之所以昂贵，不只是切换动作本身，还有一个经常被忽略的问题，就是 CPU Cache 和 TLB 的局部性会被打乱，导致切换回来后还要重新热身。工程上，线程过多、锁竞争严重、频繁阻塞唤醒、过度切分任务，都会把上下文切换次数拉高。面试里如果能补一句“高并发不等于线程越多越好”，就很贴近真实系统。

**追问点答案**  
- 进程切换和线程切换哪个更重？  
通常进程切换更重，因为它还涉及地址空间等更多上下文。  
- 为什么线程太多会变慢？  
因为线程一多，CPU 时间会被大量花在线程调度和切换上。  
- 什么问题容易导致切换暴涨？  
锁竞争、线程池配置过大、阻塞 IO 和忙等都很常见。

**原理补充**  
上下文切换本质上是一种“非业务工作”，它不直接产生业务价值，但却会吞掉 CPU 时间和缓存效率。

**架构扩展**  
很多线上系统一看到吞吐不够，就本能地把线程池调大，结果反而更慢。比如阻塞型接口服务，本来瓶颈在数据库和下游，线程一多只是让更多线程在等待，最终变成 CPU 也高、延迟也高。更合理的做法是先分清问题在计算、锁、IO 还是下游容量，再决定是加线程、异步化、限流，还是压缩临界区。

**工程指标**  
- context switch 次数：直接看切换是否异常频繁。  
- run queue 长度：看是不是太多线程在争 CPU。  
- CPU user / system 比例：辅助判断成本更多在业务执行还是系统调度。  
- 锁等待时间：看线程切换是否由竞争触发。  
- 线程池活跃线程数：判断是不是线程过配。

**易错点**  
- 以为上下文切换只是“保存寄存器”  
- 不知道缓存失效也是成本  
- 一味加线程解决吞吐问题

## 3. 阻塞 IO 和非阻塞 IO 区别？

**核心结论**  
阻塞 IO 是线程等数据准备好才继续，非阻塞 IO 是线程先返回，之后再决定重试、轮询或配合事件机制继续处理。

**标准答案**  
阻塞 IO 的特点是，当应用发起读写请求后，如果数据还没准备好，线程会一直等待，直到内核把数据准备完成；非阻塞 IO 则不会让线程一直卡住，如果当前还没准备好，会立刻返回一个状态，告诉调用方稍后再试。之所以有这两种模式，是因为不同系统对编程复杂度和资源利用率的要求不同。阻塞 IO 编程模型简单，但线程容易被大量占住；非阻塞 IO 更灵活，更适合高并发连接场景，但逻辑也更复杂。工程上，传统同步 Servlet 模型更像阻塞 IO，Netty、NIO 这类框架则更常结合非阻塞 IO 和事件驱动。面试时最好强调：非阻塞不等于高性能，关键还要看有没有配套事件机制，否则可能只是把阻塞改成忙轮询。

**追问点答案**  
- 非阻塞 IO 就一定不会卡线程吗？  
不一定，如果你自己不断轮询，也可能把线程和 CPU 浪费掉。  
- 为什么阻塞 IO 在线上也还大量存在？  
因为很多业务系统并发量没高到必须复杂化，阻塞模型更直观、开发成本更低。  
- Java 哪些场景常用非阻塞 IO？  
高连接网关、IM、消息中间件、RPC 框架常见。

**原理补充**  
这里真正的核心差异不是“快和慢”，而是线程在等待数据时是否被占住。它决定了系统到底是用线程换简单，还是用事件模型换容量。

**架构扩展**  
线上最常见的误判是把“非阻塞”直接理解成“吞吐更高”。实际上，如果你的瓶颈在数据库、下游服务、业务计算，而不是连接数，盲目上非阻塞框架只会增加复杂度。更合理的做法是根据连接规模、线程成本和编程复杂度来选模型，高连接长连接场景再优先考虑非阻塞。

**工程指标**  
- 活跃连接数：判断是否真的需要事件驱动模型。  
- 线程数：看阻塞模型是否已经把线程打满。  
- CPU 空转占比：看非阻塞实现是否退化成忙轮询。  
- 单机 QPS / 连接数承载：评估 IO 模型是否匹配业务规模。  
- 请求等待时长：判断线程是不是主要卡在 IO 等待。

**易错点**  
- 把非阻塞等同于异步  
- 以为非阻塞一定更快  
- 忽略编程复杂度和场景匹配

## 4. 同步和异步区别？

**核心结论**  
同步和异步描述的是结果通知方式，不是线程是否阻塞；阻塞和非阻塞描述的是等待期间线程状态，两者不是一个维度。

**标准答案**  
同步的核心是调用方发起请求后，要等结果返回才能继续后续逻辑；异步则是调用方发出请求后先继续做别的事，等结果好了再通过回调、通知、Future、消息等方式拿结果。之所以容易混淆，是因为很多同步调用往往伴随阻塞，很多异步模型又常和非阻塞搭配，但它们本质上不是同一层概念。同步/异步讲的是“结果什么时候交付给调用方”，阻塞/非阻塞讲的是“等待期间线程有没有被占住”。工程上，线程池提交任务后立即 `get()`，这就是异步接口写法、同步结果语义；消息队列投递后异步消费，则是典型异步协作。面试里如果能把这两个维度拆开解释，会明显比只背定义更强。

**追问点答案**  
- 异步一定非阻塞吗？  
不一定，异步接口内部实现也可能阻塞某些线程。  
- 同步一定阻塞吗？  
通常业务语义上会等待结果，但底层实现不一定全程都是阻塞模型。  
- 为什么很多系统喜欢“看起来异步、实际上同步”？  
因为这样接口统一，但实际并没有真正减少等待成本。

**原理补充**  
同步/异步是控制流问题，阻塞/非阻塞是执行等待问题。面试里能把这两个维度拆干净，说明概念很扎实。

**架构扩展**  
真实系统里，很多“异步改造失败”不是技术不行，而是接口虽然异步化了，调用方却还是立即等待结果，最终只增加复杂度没增加容量。更合理的异步改造通常要连业务链路一起改，让调用方真的能接受延迟处理、补偿或最终一致。

**工程指标**  
- 调用等待时长：看调用方是不是实际上还在同步等待。  
- 回调 / Future 完成耗时：判断异步链路是否真的独立运行。  
- 线程占用时长：看异步是否真正减少了线程阻塞。  
- 消息积压或异步任务堆积：看异步链路有没有变成新瓶颈。  
- 用户可感知延迟：验证异步化到底有没有改善体验。

**易错点**  
- 把同步和阻塞当同义词  
- 把异步和非阻塞当同义词  
- 不会举真实场景解释四种组合

## 5. 零拷贝是什么？

**核心结论**  
零拷贝的目标不是绝对“0 次复制”，而是尽量减少用户态和内核态之间不必要的数据拷贝与上下文切换。

**标准答案**  
传统数据传输路径里，数据经常需要在磁盘、内核缓冲区、用户缓冲区和网卡之间来回复制；零拷贝的核心思想，就是尽量减少这些不必要的数据搬运，尤其是避免把数据从内核拷到用户态再拷回内核态。这样做的好处是减少 CPU 消耗、内存带宽占用和上下文切换，提高吞吐。工程上，像 `sendfile`、mmap 配合 DMA、Netty 的一些优化路径，都是零拷贝思想的体现。面试里最好强调一句：零拷贝更准确地说是“少拷贝”，重点在于减少不必要的中转。

**追问点答案**  
- 为什么文件传输特别适合零拷贝？  
因为很多时候数据并不需要在用户态被加工，只是从文件直接送到网络。  
- 零拷贝为什么能提升性能？  
因为少了用户态参与的数据复制和上下文切换。  
- Java 里有没有体现？  
NIO、Netty、文件传输接口都可能利用底层零拷贝能力。

**原理补充**  
零拷贝的底层价值在于“让 CPU 少做搬运工，让 DMA 和内核路径承担更多数据移动工作”。

**架构扩展**  
网关、文件下载服务、日志传输、消息代理这类系统特别适合利用零拷贝，因为它们很多时候是在搬运数据而不是深度处理数据。相反，如果业务必须解析、加密、压缩、转换内容，零拷贝收益就会下降。更合理的做法是先判断链路是不是“传输型工作负载”。

**工程指标**  
- CPU 使用率：看数据搬运是否吃掉过多 CPU。  
- 网络吞吐：验证传输优化有没有带来实际提升。  
- 每请求内存拷贝量：评估是否存在过多中间缓冲。  
- system CPU 占比：辅助判断内核路径是否更高效。  
- 文件传输 TP99：看大文件场景下优化是否有效。

**易错点**  
- 以为零拷贝真的完全没有复制  
- 不知道它主要优化的是哪类场景  
- 不会把它和 `sendfile`、Netty 联系起来

## 6. TCP 三次握手为什么是三次？

**核心结论**  
三次握手的核心目的是让通信双方都确认“自己能发、也能收”，同时建立初始序列号基础。

**标准答案**  
TCP 建立连接时，客户端先发 SYN，服务端收到后回复 SYN+ACK，最后客户端再回 ACK，这就是三次握手。第一次表示客户端想建立连接，第二次表示服务端收到了客户端的请求并愿意建立连接，第三次则表示客户端也收到了服务端的响应，这样双方都确认了双向通信能力。之所以不能少，是因为 TCP 是全双工可靠连接，光知道一边发得到、还不够，要确认双方的发送和接收路径都可用。工程上，这三次握手还承担了同步初始序列号、避免历史失效连接误入当前会话的作用。面试里如果能说出“两次不够是因为服务端无法确认客户端是否收到了自己的确认”，就比较完整。

**追问点答案**  
- 为什么不是两次？  
因为两次只能让服务端确认客户端能发，无法确认客户端是否收到了服务端的响应。  
- 为什么要有初始序列号？  
因为 TCP 要做可靠传输和乱序重组，需要双方同步起始序列。  
- 三次握手期间应用层能发业务数据吗？  
通常要等连接建立完成后才进入稳定传输。

**原理补充**  
TCP 握手本质上是在建立“可靠、有序、双向”的会话基础，而不是简单打一声招呼。

**架构扩展**  
线上高并发服务里，握手阶段的问题常体现为 SYN 洪泛、连接堆积、短连接成本过高。比如网关服务如果大量短连接突发建立，光握手成本就会显著抬高系统负载。更合理的做法通常是连接复用、长连接、连接池和内核参数调优配合使用。

**工程指标**  
- 新建连接速率：看系统是不是被连接建立成本拖住。  
- SYN backlog 使用率：判断握手队列是否接近打满。  
- 握手失败率：看网络或防护策略是否有问题。  
- 连接建立耗时：反映网络和系统负载状态。  
- 短连接比例：辅助判断是否需要连接复用优化。

**易错点**  
- 只会背 SYN / ACK 流程  
- 不知道两次为什么不够  
- 不会把握手和线上连接成本联系起来

## 7. TCP 四次挥手为什么是四次？

**核心结论**  
四次挥手的本质是 TCP 连接关闭时，双方的发送方向要分别独立关闭，所以通常需要两个方向各自确认。

**标准答案**  
TCP 是全双工通信，所以断开连接时，客户端和服务端两个方向的发送通道都要分别关闭。通常一方先发 FIN，表示自己没有数据发了；对方先回 ACK，表示知道了；等对方自己也没有数据可发时，再发送 FIN；最后再由最初发起方回 ACK，这样就形成四次挥手。之所以不像握手那样可以合并成三次，是因为收到 FIN 的一方未必立刻就能关闭自己的发送方向，它可能还有剩余数据要发。工程上，这种设计保证了连接关闭过程尽量完整，不会粗暴截断还没发完的数据。面试里如果你能补一句“ACK 和 FIN 有时可能合并，但协议语义上是两个动作”，会更扎实。

**追问点答案**  
- 为什么收到 FIN 不一定马上发 FIN？  
因为对方可能还有数据没发完，只是先确认对端的关闭请求。  
- 为什么最后一个 ACK 这么重要？  
因为它确保对方知道自己的关闭请求已被确认。  
- 四次挥手有没有可能表现成三次？  
在某些实现和时序下 ACK 与 FIN 可能合并发送，但语义还是四步。

**原理补充**  
挥手和握手最大的区别在于：关闭连接时双方状态不一定同步，而建立连接时双方都在共同进入一个新状态。

**架构扩展**  
线上服务如果大量短连接频繁关闭，TIME_WAIT 和连接回收就会成为重要问题。比如调用方每次请求都新建短连接，服务量一大，系统里会堆很多关闭中的连接，最终消耗端口和内核资源。更合理的做法通常还是连接复用和长连接优先。

**工程指标**  
- 连接关闭速率：看短连接关闭是否过于频繁。  
- TIME_WAIT 数量：判断挥手后的连接残留是否过多。  
- 端口占用率：看本机可用端口是否被大量挥手后连接占据。  
- 连接回收耗时：辅助分析内核网络栈压力。  
- 下游连接复用率：验证是否有必要继续优化长连接策略。

**易错点**  
- 把四次挥手机械背流程  
- 不知道全双工关闭的本质  
- 不会把挥手和短连接成本联系起来

## 8. TIME_WAIT 为什么存在？

**核心结论**  
TIME_WAIT 的核心作用是确保最后一个 ACK 有机会重传，并避免旧连接里的延迟报文污染新连接。

**标准答案**  
TCP 连接主动关闭的一方在发送最后一个 ACK 之后，通常不会立刻释放连接，而是进入 TIME_WAIT 状态，持续一段时间。这样做有两个主要原因：第一，如果最后一个 ACK 丢了，对方还能重发 FIN，这时本端还可以重新确认；第二，可以让网络中可能残留的旧报文在这段时间里自然消失，避免后续相同四元组的新连接误收到旧数据。之所以看起来“很浪费”，是因为可靠性和资源利用率之间需要平衡。工程上，TIME_WAIT 多本身不一定是故障，但如果短连接量很大，它会占用本地端口和内核连接资源。面试里最好强调：TIME_WAIT 不是 bug，而是可靠关闭机制的一部分。

**追问点答案**  
- 为什么主动关闭方更容易进入 TIME_WAIT？  
因为它负责发送最后一个 ACK，需要承担重传保障责任。  
- TIME_WAIT 太多一定是坏事吗？  
不一定，要看端口资源、连接速率和业务模型。  
- 为什么长连接能缓解 TIME_WAIT 问题？  
因为它减少了连接创建和关闭频率。

**原理补充**  
TIME_WAIT 的本质是在用时间换协议正确性，确保连接关闭语义完整、旧报文不会串到新连接里。

**架构扩展**  
最典型的线上问题是服务作为客户端高频调用下游，每次都短连，结果本机 TIME_WAIT 飙升，端口很快被占满，接着新连接建立失败。很多团队第一反应是调内核参数，但根因通常是连接模型设计不合理。更合理的顺序通常是先做连接复用，再评估参数调优。

**工程指标**  
- TIME_WAIT 数量：看关闭连接残留是否异常。  
- 本地可用端口余量：判断是否有端口耗尽风险。  
- 短连接比例：辅助确认问题是不是业务模型导致。  
- 新建连接失败率：看 TIME_WAIT 是否已经伤害可用性。  
- 下游连接池命中率：验证连接复用是否足够。

**易错点**  
- 把 TIME_WAIT 当成纯负担  
- 不知道它保护了什么  
- 一上来就想靠调内核参数硬压

## 9. TCP 如何保证可靠传输？

**核心结论**  
TCP 的可靠性来自一整套机制协同工作，包括序列号、确认应答、超时重传、滑动窗口、流量控制和拥塞控制。

**标准答案**  
TCP 能保证可靠传输，不是靠单一手段，而是多种机制组合。序列号负责标识数据顺序，确认应答让发送方知道哪些数据已经到达，超时重传负责兜底丢包，滑动窗口让双方不必每发一包都停下来等确认，流量控制避免发送方把接收方压垮，拥塞控制则避免网络整体被打爆。这样一来，TCP 才能在不可靠网络之上提供“有序、尽力可靠”的字节流服务。工程上，可靠传输意味着更高协议开销，所以在某些对实时性更敏感的场景，才会转向 UDP 或应用层自定义协议。面试里可以总结一句：TCP 的可靠不是绝对不丢，而是通过确认和重传机制尽量把数据送达并按序重组。

**追问点答案**  
- 丢包后 TCP 怎么知道要重传？  
可以靠超时，也可以靠重复 ACK 等快速重传机制。  
- 为什么有了可靠传输还会有超时？  
因为可靠是有代价和边界的，网络拥塞或对方处理慢仍会影响延迟。  
- 滑动窗口为什么重要？  
它让发送效率更高，不必每个包都单独等待确认。

**原理补充**  
TCP 的可靠性本质上是“在不可靠网络上用状态机和反馈机制重建相对可靠的传输语义”。

**架构扩展**  
线上很多“网络抖动”问题，最后表现成重传增多、RT 上升、吞吐下降。比如机房网络抖动或下游服务压力大时，TCP 连接虽然没断，但因为重传和窗口收缩，业务响应会明显变慢。更合理的监控方式不是只看连不连得上，还要看重传、RTT、拥塞窗口变化等指标。

**工程指标**  
- 重传率：判断网络质量或对端处理是否异常。  
- RTT / 延迟抖动：看传输链路是否稳定。  
- 滑动窗口大小：判断吞吐受限在发送端还是接收端。  
- 丢包率：衡量链路质量。  
- 吞吐量：看可靠机制是否已经被频繁触发到影响业务。

**易错点**  
- 只会说 ACK 和重传  
- 忽略流量控制和拥塞控制  
- 不会把 TCP 指标和业务响应慢联系起来

## 10. HTTP1.1 和 HTTP2 差异？

**核心结论**  
HTTP/2 的核心升级不只是“更快”，而是通过二进制分帧、多路复用、头部压缩和服务器推送，改善了 HTTP/1.1 的队头阻塞和连接效率问题。

**标准答案**  
HTTP/1.1 相比早期版本已经支持长连接和管线化，但在真实使用中，浏览器和服务端通常还是会建立多个连接来并发请求，因为单连接上的请求和响应管理效率不高，还会受到应用层队头阻塞影响。HTTP/2 的主要升级包括：把文本协议改成二进制分帧、更容易高效解析；在一个连接上支持多路复用，让多个请求并发传输；通过 HPACK 做头部压缩，减少重复头开销；还支持服务器推送。这样做的本质，是减少连接数量、降低头部冗余、提高并发请求效率。工程上，HTTP/2 对高并发 Web 页面和 API 聚合场景很有价值，但并不自动解决所有性能问题，尤其后端慢查询和业务处理慢，换协议也救不了。

**追问点答案**  
- HTTP/2 为什么不是完全没有阻塞？  
它解决了应用层队头阻塞，但底层 TCP 丢包仍会影响整个连接。  
- 为什么 HTTP/2 更适合前端页面加载？  
因为一个页面会并发请求很多资源，多路复用收益明显。  
- 服务器推送为什么后来用得没那么多？  
因为收益不稳定、缓存协同复杂，很多场景下效果一般。

**原理补充**  
HTTP/2 的关键价值在于“让连接成为真正可并发复用的通道”，而不是像 HTTP/1.1 那样更多靠多连接来凑并发。

**架构扩展**  
网关、API 聚合层、前后端分离系统，在跨多个资源并发请求时，HTTP/2 很有价值；但如果后端真正瓶颈在数据库、RPC 调用或序列化，协议升级只是锦上添花。成熟团队通常会把协议优化和后端性能治理一起看，而不是只做“协议升级冲 KPI”。

**工程指标**  
- 单连接并发流数：看 HTTP/2 多路复用是否真正被利用。  
- 连接数：判断是否从多连接模式收敛到了高效复用。  
- 头部大小占比：评估压缩收益。  
- 页面资源加载耗时：前端场景下最直接的业务指标。  
- TCP 重传率：判断底层链路是否抵消了协议层收益。

**易错点**  
- 只会背“二进制、多路复用、压缩”  
- 以为 HTTP/2 没有任何队头阻塞  
- 不会判断它适合什么场景

## 11. HTTPS 解决了什么问题？

**核心结论**  
HTTPS 主要解决了 HTTP 明文传输下的窃听、篡改和身份伪造问题。

**标准答案**  
HTTP 本身是明文协议，数据在网络中传输时可能被中间人看到、篡改，客户端也无法确认自己连的是不是目标服务。HTTPS 在 HTTP 之下加了 TLS 层，通过加密保证传输内容不被轻易窃听，通过完整性校验防止数据被篡改，通过证书体系帮助客户端验证服务端身份。之所以要这么设计，是因为互联网环境下传输链路并不可信，单靠应用层约定无法保证安全。工程上，HTTPS 不只是登录和支付接口要用，几乎所有公网业务都应该默认启用。面试里可以总结一句：HTTPS 不是“更安全的 HTTP”，而是给 HTTP 加上一整套传输安全能力。

**追问点答案**  
- HTTPS 能防止什么？  
主要是窃听、篡改和伪装。  
- 为什么有了加密还要证书？  
因为光加密还不能证明对方身份，证书用于身份认证。  
- HTTPS 会不会更慢？  
有握手和加解密成本，但现代硬件和协议优化后通常可接受。

**原理补充**  
HTTPS 的核心价值不只是“内容加密”，而是“机密性 + 完整性 + 身份认证”三件事一起成立。

**架构扩展**  
真实系统里，HTTPS 还会影响网关、证书续期、负载均衡、内部服务通信策略。很多团队早期只给公网入口加 HTTPS，内部链路全明文，后面一做零信任或跨机房访问就得补课。更成熟的体系通常会把传输安全作为默认基线，而不是特殊接口例外。

**工程指标**  
- TLS 握手耗时：看安全链路的建立成本。  
- 证书过期预警：避免因为证书问题引发服务不可用。  
- TLS 握手失败率：判断证书、协议版本或配置是否有问题。  
- CPU 在加解密上的占比：高吞吐场景下很重要。  
- HTTPS 请求成功率：验证安全配置是否影响业务可用性。

**易错点**  
- 只会说“HTTP 加密版”  
- 不知道证书解决的是身份问题  
- 忽略运维和续期层面的工程成本

## 12. DNS 解析流程是什么？

**核心结论**  
DNS 解析本质上是把域名逐级翻译成 IP 的过程，通常会经过本地缓存、递归解析器和权威服务器多层协作。

**标准答案**  
当客户端访问一个域名时，通常会先查浏览器缓存、系统缓存、本地 hosts，如果没有，再把请求发给本地配置的递归 DNS 解析器。递归解析器如果也没有缓存，会按层级去问根域名服务器、顶级域服务器、权威域名服务器，最终拿到目标域名对应的记录，再返回给客户端并缓存一段时间。这样设计的目的是兼顾解析效率和全球域名管理的可扩展性，不可能让所有域名信息都集中在一个地方。工程上，DNS 解析问题常常表现成“服务没挂但访问不到”，所以排障时要分清是业务服务问题，还是解析、缓存、权威配置问题。面试里如果能补一句“DNS 本身不是即时一致的，TTL 会影响切换速度”，会更完整。

**追问点答案**  
- 为什么改 DNS 记录后不会立刻生效？  
因为各级缓存会按 TTL 保留旧结果。  
- 递归解析器和权威服务器有什么区别？  
前者帮客户端一路问到底，后者持有域名最终权威记录。  
- 为什么 DNS 故障影响会很大？  
因为服务即使健康，只要名字解析不到，流量就过不去。

**原理补充**  
DNS 的核心不是“查表”这么简单，而是一个分层、分布式、带缓存的名字解析系统。

**架构扩展**  
线上常见问题包括 DNS 缓存污染、权威配置错误、TTL 设置不合理、切流不生效等。比如一次紧急切机房，业务方以为改了记录就会立刻生效，结果客户端和递归解析器还在缓存旧 IP，流量久久不过来。更合理的做法通常是提前调低 TTL，并配合健康检查和流量治理一起做。

**工程指标**  
- DNS 解析耗时：看名字解析是否正在拖慢请求。  
- DNS 失败率：判断是不是解析链路问题而不是应用问题。  
- TTL 配置：决定变更传播速度和缓存收益。  
- 权威解析成功率：验证配置是否正确。  
- 切流后流量命中新旧 IP 的比例：判断 DNS 变更是否真的生效。

**易错点**  
- 只会背“根服务器、顶级域、权威服务器”  
- 不知道缓存和 TTL 的影响  
- 把解析失败误判成服务挂了

## 13. Cookie、Session、Token 区别？

**核心结论**  
Cookie 是浏览器存储机制，Session 是服务端会话状态，Token 是可携带认证信息的凭证，它们不是同一层概念。

**标准答案**  
Cookie 本质上是浏览器保存的一小段客户端数据，浏览器后续请求时会自动带上；Session 则是服务端保存的会话状态，通常通过 Cookie 里的 SessionId 关联到具体会话；Token 是一种显式凭证，客户端持有后在请求里主动携带，服务端据此识别身份。之所以会有这几种机制，是因为 Web 体系天然是无状态的，需要额外手段来表达“你是谁、你之前做过什么”。工程上，传统单体系统常见 Cookie + Session；前后端分离、移动端、微服务体系里，Token 更常见，因为扩展性和跨服务传播更方便。面试时最好强调：Cookie 是载体，Session 是状态，Token 是凭证，这三个词经常被误混。

**追问点答案**  
- Token 一定就是 JWT 吗？  
不一定，JWT 只是 Token 的一种实现形式。  
- Session 为什么在分布式下更麻烦？  
因为状态存在服务端，多实例间要共享或粘滞。  
- Cookie 为什么不适合存太多敏感信息？  
因为它在客户端，容易暴露、被篡改或被窃取。

**原理补充**  
这三者本质上分别对应“客户端存储”“服务端状态”“跨请求凭证”三个不同层面的设计。

**架构扩展**  
线上鉴权设计里，最常见的问题不是机制本身，而是边界没讲清。比如把大量权限状态塞进 JWT，后面权限一变更就难以及时失效；或者继续依赖 Session，却没有统一会话共享方案，导致分布式下频繁丢登录。更合理的做法通常是根据系统规模和变更频率权衡状态放哪一层。

**工程指标**  
- 登录态失效率：判断认证机制是否稳定。  
- Token 刷新成功率：看凭证续期是否可靠。  
- Session 共享延迟：分布式 Session 场景很关键。  
- 鉴权失败率：判断是不是状态传递和校验有问题。  
- 会话存储容量：看 Session 模型是否带来过高资源成本。

**易错点**  
- 把 Cookie、Session、Token 当同义词  
- 以为 Token 就没有状态问题  
- 不会分析分布式下的权衡

## 14. 常见状态码 301/302/401/403/500 含义？

**核心结论**  
这些状态码不只是背定义，而是要能区分“跳转、认证、授权、服务异常”这几类不同语义。

**标准答案**  
301 表示永久重定向，通常意味着资源地址已经长期迁移；302 表示临时重定向，更适合短期跳转或流程性跳转；401 表示未认证，意思是当前请求还没有合法身份；403 表示已认证但无权访问；500 表示服务端处理请求时出现了未预期异常。之所以这些状态码重要，是因为它们是客户端、网关、服务端之间表达语义的统一语言。工程上，用错状态码会直接影响客户端重试、SEO、缓存行为和前后端错误处理。面试里如果能补一句“401 是先登录，403 是登录了也不行”，会很清晰。

**追问点答案**  
- 为什么 301 和 302 不能乱用？  
因为浏览器、搜索引擎和缓存系统对它们的处理策略不同。  
- 401 和 403 最大区别是什么？  
401 是身份问题，403 是权限问题。  
- 500 应该直接返回给用户吗？  
通常返回统一错误语义，详细异常要打到日志而不是直接暴露。

**原理补充**  
HTTP 状态码本质上是协议层语义表达，不是“后端随便选个数字”。选对状态码，客户端和中间层才能做对行为。

**架构扩展**  
线上 API 网关和前后端协作中，状态码治理很重要。比如把所有异常都包成 200，会让监控和重试逻辑全部失真；把权限问题全打成 500，又会误导运维和客户端。成熟团队通常会统一错误码和 HTTP 状态码映射规则。

**工程指标**  
- 各状态码分布：快速看服务当前主要问题类型。  
- 5xx 比例：判断系统内部异常是否上升。  
- 401 / 403 比例：辅助区分认证体系和权限体系问题。  
- 3xx 跳转比例：看是否存在异常跳转或路由配置问题。  
- 客户端重试率：很多状态码设计问题会直接放大重试。

**易错点**  
- 只会背字面意思  
- 把 401 和 403 混淆  
- 把所有业务错误都包成 200

## 15. 什么是幂等 HTTP 方法？

**核心结论**  
幂等的核心是“同一个请求执行多次，结果和执行一次一致”，它解决的是重试安全问题，不是“只执行一次”。

**标准答案**  
在 HTTP 语义里，幂等通常指相同请求重复执行多次，资源最终状态不应发生额外变化。像 GET、PUT、DELETE 往往被设计成幂等，而 POST 通常不是天然幂等。之所以这个概念重要，是因为网络重试、超时重放、客户端重复提交在真实系统里非常常见，如果接口不具备幂等能力，重复请求就可能造成重复扣款、重复下单、库存重复扣减。工程上，幂等不只是 HTTP 方法语义，还要落到业务实现，比如幂等键、去重表、状态机约束。面试里最好强调：协议层幂等和业务层幂等不是一回事，真正关键的是业务语义。

**追问点答案**  
- 为什么 POST 也可以做成幂等？  
因为幂等性最终是业务实现决定的，比如通过幂等键去重。  
- DELETE 为什么通常被认为幂等？  
因为删一次和删多次，目标资源最终都处于“已删除”状态。  
- 幂等和事务是一个东西吗？  
不是，一个解决重复请求结果一致，一个解决一组操作一致完成。

**原理补充**  
幂等本质上是重试容错能力。只要系统可能重试，就必须认真设计幂等。

**架构扩展**  
支付、订单、消息消费、库存操作这些场景最怕没有幂等。很多线上事故不是业务逻辑错，而是超时后客户端和网关重试，把原本成功的请求又打了一遍。成熟系统通常会把幂等当成核心接口默认要求，而不是事后补丁。

**工程指标**  
- 重复请求率：看幂等设计是不是高频真实需求。  
- 幂等冲突命中率：判断去重机制是否有效工作。  
- 重试成功率：看幂等支持是否真正提升了可恢复性。  
- 重复下单 / 重复扣款事件数：这是最直接的业务后果。  
- 幂等存储查询耗时：避免去重机制本身成为新瓶颈。

**易错点**  
- 把幂等理解成“只执行一次”  
- 只会背 GET/POST 差异  
- 不会把幂等落到业务实现

## 16. Linux 下怎么查 CPU 飙高进程？

**核心结论**  
查 CPU 飙高不是停在“哪个进程高”，而是要继续往下定位到“哪个线程、哪段调用栈、为什么忙”。

**标准答案**  
定位 CPU 飙高时，通常先用 `top`、`htop` 或 `ps` 看哪个进程占用异常，再进入进程级分析。如果是 Java 服务，下一步常常要看进程内哪个线程最热，比如用 `top -H -p <pid>` 找线程，再把线程 ID 转成十六进制，结合 `jstack` 看对应调用栈。这样做的原因是 CPU 高只是结果，真正的根因可能是死循环、热点锁自旋、JSON 序列化过重、GC、自旋重试或下游失败后的忙等。工程上，找到热点线程之后，还要回到业务代码判断它是在“正常高算力执行”还是“异常空转”。面试里如果能讲到“进程 -> 线程 -> 调用栈”这个链路，基本就是实战型回答。

**追问点答案**  
- 为什么不能只看进程 CPU？  
因为一个进程高不代表你知道是哪个线程、哪段逻辑导致的。  
- Java 里为什么要把线程 ID 转十六进制？  
因为 `jstack` 线程标识常按十六进制展示。  
- CPU 高一定是业务代码有问题吗？  
不一定，也可能是 GC、自旋锁、系统调用或第三方库热点。

**原理补充**  
CPU 飙高排查的本质是从“资源异常”一路下钻到“执行路径异常”。不下钻到线程和栈，排查就停在表面。

**架构扩展**  
线上最常见的几类原因包括无限重试、错误的 while 轮询、自旋 CAS 失败、热点 JSON 序列化、日志刷盘、GC 过重。很多团队查到进程高就结束了，结果下一次还是同样的问题。更成熟的做法是把热点线程分析、火焰图和 GC 日志一起纳入日常排障工具链。

**工程指标**  
- 进程 CPU 使用率：先判断问题范围。  
- 热线程 CPU 占比：看是否集中在少数线程。  
- GC CPU 占比：排除是不是 GC 造成的假象。  
- 重试率 / 循环任务频率：辅助识别忙等或重试风暴。  
- 火焰图热点函数：看 CPU 时间到底烧在什么路径上。

**易错点**  
- 只停留在进程层面  
- 不会继续定位到线程栈  
- 把所有 CPU 高都归结为业务逻辑慢

## 17. Linux 下怎么查端口占用？

**核心结论**  
查端口占用的目标不只是知道“哪个端口被占”，而是搞清楚“哪个进程、什么协议状态、是不是预期监听”。

**标准答案**  
在 Linux 下查端口占用，常见做法是用 `ss -lntp`、`netstat -lntp` 或 `lsof -i :<port>`。如果只是看监听关系，重点是找到哪个进程在监听这个端口；如果还涉及连接异常，就要进一步看连接状态，比如 `ESTABLISHED`、`TIME_WAIT`、`CLOSE_WAIT` 等。之所以要区分，是因为“端口占用”可能是正常监听，也可能是僵尸进程残留、连接状态异常、服务重复启动失败。工程上，排障时一定要从“端口被谁占了”继续往下问“这个进程该不该存在、状态是不是正常、为什么没有释放”。面试时最好别停在命令本身，而要讲清楚判断逻辑。

**追问点答案**  
- `LISTEN` 和 `ESTABLISHED` 区别？  
前者表示进程在监听端口，后者表示已有连接建立。  
- 为什么端口明明没监听却还显示很多连接？  
可能是历史连接残留，比如 `TIME_WAIT`、`CLOSE_WAIT`。  
- `CLOSE_WAIT` 多说明什么？  
通常说明本端应用层没有及时关闭连接。

**原理补充**  
端口本质上是传输层资源入口，排查端口问题时必须把“监听关系”和“连接状态”分开看。

**架构扩展**  
线上最常见的不是单纯“端口被占”，而是发布失败、服务重复启动、连接状态泄漏。比如网关进程重启不干净导致端口还被旧进程占着，或者下游连接没及时释放导致 `CLOSE_WAIT` 堆积。成熟团队通常会在发布和健康检查里把端口状态纳入标准流程。

**工程指标**  
- 监听端口数量：看服务实例是否符合预期暴露。  
- 各连接状态数量：判断是不是存在连接泄漏。  
- `TIME_WAIT` / `CLOSE_WAIT` 占比：识别典型网络栈问题。  
- 发布失败次数：有些端口问题会直接体现在这里。  
- 连接建立失败率：判断端口占用是否已经影响可用性。

**易错点**  
- 只会说 `netstat` 或 `lsof`  
- 不区分监听和连接状态  
- 不知道 `CLOSE_WAIT` / `TIME_WAIT` 的排查意义

## 18. Linux 下怎么查内存异常？

**核心结论**  
查内存异常要先分清是“进程内存高、系统缓存高、交换分区高，还是 Java 堆高”，不能只看一个数字。

**标准答案**  
在 Linux 下看内存异常，通常先用 `free -m`、`top`、`vmstat` 了解整体内存和 swap 情况，再进一步看哪个进程占用高，比如 `ps aux --sort=-%mem`。如果是 Java 服务，还要继续区分进程 RSS、JVM Heap、Non-Heap、线程栈和直接内存，不能只看系统层面的总内存。之所以要这样拆，是因为 Linux 的内存高不一定是坏事，页缓存也会吃内存；真正危险的是业务进程持续膨胀、swap 抖动、OOM Killer 活跃。工程上，排障时最好建立一个分层视角：机器层、进程层、JVM 层分别看。面试里如果能讲到“缓存占用高不等于内存泄漏”，会比较成熟。

**追问点答案**  
- 为什么 Linux 内存经常看起来“快满了”？  
因为系统会积极用空闲内存做文件缓存，这不一定是问题。  
- swap 高为什么危险？  
因为说明内存紧张到已经开始把数据换到磁盘，性能会明显抖动。  
- RSS 和 Heap 为什么都要看？  
因为 RSS 是进程真实物理内存占用，Heap 只是 Java 堆的一部分。

**原理补充**  
内存问题排查的关键不是“有没有占满”，而是“占用的是谁、会不会回、会不会伤害性能和稳定性”。

**架构扩展**  
线上最常见的误判是看到机器内存高就说“内存泄漏”，最后发现只是 page cache；或者只看 Heap 正常，就忽略了 Direct Memory 和线程栈导致的 RSS 膨胀。更合理的排查方式一定是系统层和应用层结合看，尤其是 Java 进程。

**工程指标**  
- 机器可用内存：看系统是否接近资源边界。  
- swap 使用率：判断是否已经发生性能风险。  
- 进程 RSS：看进程真实总占用是否在持续增长。  
- JVM Heap / Non-Heap：区分问题是在堆还是其他区域。  
- OOM Killer 事件数：这是系统层面非常直接的风险信号。

**易错点**  
- 只看 free 命令就下结论  
- 把 page cache 当泄漏  
- 不分系统内存和 JVM 内存

## 19. `top`、`vmstat`、`iostat`、`netstat/ss` 怎么用？

**核心结论**  
这些命令不是孤立背诵，它们分别对应 CPU / 内存调度、磁盘 IO、网络连接几个不同观察视角，组合起来才有排障价值。

**标准答案**  
`top` 更适合先看整体资源概况和热点进程；`vmstat` 更适合看系统级运行状态，比如 runnable 队列、上下文切换、swap、IO wait；`iostat` 聚焦磁盘设备利用率、吞吐和 await 等指标，适合判断磁盘是不是瓶颈；`netstat` 或 `ss` 更适合看端口监听、连接状态和网络连接分布。之所以这些命令要组合看，是因为线上问题往往不是单一维度的，比如接口慢可能是 CPU 高，也可能是磁盘抖动、连接堆积或 swap 导致。工程上，排障最怕只盯一个命令下结论。面试里如果能说出“先全局、再局部、再回到具体进程/线程”，会非常像实战。

**追问点答案**  
- 为什么 `ss` 现在更常用？  
因为它通常比老的 `netstat` 更快、信息也更丰富。  
- `iostat` 看哪个值最关键？  
要结合 `%util`、await、svctm、吞吐一起看，别只看一个值。  
- `vmstat` 的 runnable 队列说明什么？  
可以帮助判断 CPU 是否供不应求。

**原理补充**  
这些命令其实对应了系统资源的不同维度：算力、内存/调度、磁盘、网络。线上问题常常是在这些维度之间串起来的。

**架构扩展**  
成熟排障流程一般是先用 `top` 或监控发现症状，再用 `vmstat` 看系统压力类型，用 `iostat` 看磁盘是否拖后腿，用 `ss` 看连接状态，最后再下钻到进程和线程。这样能避免“CPU 高就盯代码”“网络慢就盯下游”这种单点思维。

**工程指标**  
- load / CPU 使用率：看是不是算力瓶颈。  
- runnable 队列长度：判断 CPU 调度压力。  
- 磁盘 await：看 IO 等待是否异常。  
- 网络连接状态分布：判断连接是否健康。  
- 上下文切换和 IO wait：帮助识别系统层瓶颈类型。

**易错点**  
- 只会背命令，不会说看什么  
- 用单一指标下结论  
- 不会把系统层观察和应用层问题串起来

## 20. 线上网络抖动排障思路是什么？

**核心结论**  
网络抖动排障的核心不是一上来认定“网络有问题”，而是分层确认：客户端、服务端、链路、中间件、DNS、连接池、重试策略到底哪一层在放大问题。

**标准答案**  
线上遇到网络抖动时，第一步通常不是立刻怪网络，而是先看现象：是连接建立慢、RT 突增、丢包重传增多，还是只有某个下游服务异常。然后按层排查：先看客户端和服务端资源是否正常，再看 DNS 解析、TCP 建连、连接池、负载均衡、机房链路、交换机或云网络层是否有异常。之所以要这么做，是因为“网络抖动”常常是一个最终表象，真正根因可能是对端服务打满、连接池耗尽、重试风暴、长尾 GC 或链路丢包。工程上，成熟排障一般会结合应用日志、重传率、RTT、连接状态、超时分布、跨机房流量情况一起看，而不是只 ping 一下。面试里如果能说出“先分层、再缩范围、最后做对比验证”，会非常好。

**追问点答案**  
- 为什么 ping 正常不代表业务网络正常？  
因为业务协议、连接复用、重传、DNS、LB 路径都可能和 ping 不同。  
- 网络抖动为什么常和重试风暴一起出现？  
因为轻微延迟抖动会触发超时，超时又触发重试，最后把问题放大。  
- 怎么快速判断是单点问题还是链路问题？  
看是不是只影响某个服务实例、某个机房、某条调用链，还是全局普遍异常。

**原理补充**  
网络抖动排查本质上是在找“延迟和失败发生在哪一层”，它既是网络问题，也是分布式系统问题。

**架构扩展**  
很多线上事故看起来是“网络抖了”，其实是应用把小问题放大了。比如超时设置过短、重试次数过高、连接池太小、没有熔断降级，这些都会把链路轻微波动放大成大面积故障。更成熟的架构设计会在网络不稳定是常态这个前提下，提前做好超时、重试、限流、熔断和降级。

**工程指标**  
- RTT 与延迟抖动：看链路是否真的不稳定。  
- 重传率：判断丢包或链路质量是否异常。  
- 建连耗时：区分是连接建立问题还是传输问题。  
- DNS 解析耗时与失败率：排除名字解析层问题。  
- 请求超时率与重试率：判断应用层是否正在放大网络波动。  
- 机房 / 可用区维度错误率：快速识别问题是否集中在某一段链路。

**易错点**  
- 一上来就认定是网络团队的问题  
- 只用 ping 做判断  
- 不会把重试、连接池、超时策略一起考虑
