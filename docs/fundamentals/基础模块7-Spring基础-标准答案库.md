# 基础模块7 - Spring基础 - 标准答案库

## 1. IOC 和 DI 是什么？

**核心结论**  
IOC 是控制反转，强调对象创建和依赖关系的控制权交给容器；DI 是依赖注入，是 IOC 最常见的实现方式。

**标准答案**  
IOC 可以理解成：对象不再自己主动去创建依赖，而是把对象创建和组装的控制权交给 Spring 容器。DI 则是 IOC 的具体落地方式，也就是容器在对象创建好之后，把它依赖的其他对象注入进来。之所以这么设计，是因为传统代码里对象之间常常自己 `new` 依赖，导致耦合高、替换难、测试成本高。Spring 把依赖关系集中到容器层管理后，业务类更专注于职责本身。工程上，IOC 和 DI 的价值不只是“少写 `new`”，而是让对象生命周期、依赖关系、扩展点和测试替身都更容易统一管理。面试里如果能补一句“IOC 是思想，DI 是实现手段”，会很清晰。

**高分回答**
IoC 改变的是依赖决策的位置：业务对象只声明所需协作者及契约，容器负责选择实现、构造顺序、作用域和生命周期；DI 是把已解析依赖通过构造器、工厂方法或属性传入对象的机制。它降低的是“创建与选择实现”的耦合，不会自动消除业务接口之间的语义耦合，错误拆分仍会得到一张更复杂的 Bean 图。

构造器注入适合必需依赖：对象创建后即满足不变量，字段可 final，测试无需启动容器，也能在循环依赖出现时尽早暴露设计问题。Setter 只用于真正可选或可重配置依赖；字段注入隐藏依赖并让普通单元测试困难。容器边界应留在应用装配层，领域对象不应到处主动查 `ApplicationContext`，否则又回到 Service Locator 耦合。

**追问点答案**  
- 为什么 IOC 能降低耦合？  
因为业务类不再关心依赖怎么创建，只关心自己需要什么。  
- DI 常见方式有哪些？  
构造器、Setter/方法和字段注入。必需依赖优先构造器，能表达完整性并便于测试；可选依赖才用 Setter，字段注入通常不利于显式依赖和非容器测试。
- 没有 Spring 能做 IOC 吗？  
可以，IOC 是设计思想，不是 Spring 专属能力。

**原理补充**  
IOC 的本质，是把“对象之间如何连接”从业务代码里抽离出来，变成一个可统一治理的系统问题。

**架构扩展**  
真实项目里，IOC 带来的收益通常体现在扩展性和测试性上。比如你想把本地实现替换成远程实现，或者在测试里换成 mock，只要容器配置或装配规则变一下就行；如果所有类都在内部直接 `new`，这种替换会非常痛苦。成熟团队通常会把依赖边界和容器装配一起设计，而不是把 Spring 仅仅当成注解工具。

**工程指标**  
- Bean 装配失败率：看依赖关系是否清晰稳定。  
- 单元测试替换依赖成本：衡量 IOC 是否真正降低耦合。  
- 启动期 Bean 创建耗时：IOC 容器管理太重会先反映在这里。  
- 模块间依赖复杂度：看 Spring 是否真的在帮你解耦。  
- 配置变更影响范围：评估依赖管理是否集中可控。

**证据链**  
- Bean 定义方式  
- 依赖关系图  
- 测试替身注入方式  
- 启动失败或装配异常日志

**低风险优化路径**  
1. 先理清依赖边界  
2. 再优先采用构造器注入  
3. 最后收敛配置和装配规则

**易错点**  
- 把 IOC 和 DI 当同义词  
- 只会说“降低耦合”但不会解释怎么降  
- 认为 Spring 只是帮你省掉 `new`

## 2. Bean 生命周期有哪些关键阶段？

**核心结论**  
Bean 生命周期通常包括实例化、属性注入、初始化回调、可用阶段和销毁回调，Spring 会在这些阶段插入大量扩展点。

**标准答案**  
一个 Spring Bean 从被容器管理开始，通常会经历实例化、依赖注入、Aware 接口回调、BeanPostProcessor 前后处理、初始化方法执行、进入可用状态，最后在容器关闭时执行销毁逻辑。之所以生命周期这么丰富，是因为 Spring 不只是帮你创建对象，它还要支持代理、增强、属性填充、初始化校验、资源管理等一整套扩展能力。工程上，理解 Bean 生命周期非常重要，因为很多“为什么这个注解没生效”“为什么代理没织进去”“为什么初始化太慢”这类问题，都和生命周期阶段有关。面试里如果能顺手提到 `BeanPostProcessor` 和初始化回调，就已经比较扎实了。

**高分回答**
典型单例 Bean 经历定义解析、实例化、属性填充、Aware 回调、初始化前 BPP、`@PostConstruct`/初始化方法、初始化后 BPP，最终放入可用单例；代理通常在后处理阶段返回的可能已不是原始对象。容器关闭时再执行销毁回调，但 prototype Bean 的完整销毁通常不由容器自动跟踪，不能把所有作用域一概而论。

还要区分 `BeanFactoryPostProcessor` 与 `BeanPostProcessor`：前者在实例化前修改 BeanDefinition，后者围绕具体 Bean 实例扩展。初始化回调中做远程调用会把外部 TP99、重试和失败直接变成启动风险；应记录每个 Bean 创建耗时、关键路径与外部调用，把大预热拆成可观测、可降级的 readiness 阶段，并保证关闭回调有总超时。

**追问点答案**  
- 为什么 `BeanPostProcessor` 很重要？  
因为很多 Spring 扩展能力，比如代理和注解增强，都依赖它。  
- 初始化方法有哪些形式？  
常见有 `@PostConstruct`、`InitializingBean`、自定义 init-method。  
- 销毁回调什么时候触发？  
受容器管理的 singleton 在正常关闭时触发，适合停止接流、等待有界任务并释放资源；进程强杀不保证回调执行，prototype 的销毁也通常需调用方负责，所以外部正确性不能只依赖 destroy。

**原理补充**  
Bean 生命周期的本质，是让 Spring 在“对象创建完成之前、之后、销毁之前”都拥有可插入能力，从而把框架功能织进去。

**架构扩展**  
线上常见问题之一就是启动慢，排查时常会发现不是类加载慢，而是 Bean 初始化阶段做了太多重逻辑，比如预热缓存、连数据库、远程拉配置、注册线程池。成熟团队通常会让 Bean 初始化尽量轻，重逻辑下放到显式启动流程或后台异步预热。

**工程指标**  
- Bean 初始化耗时分布：看启动瓶颈集中在哪些 Bean。  
- Bean 创建失败率：判断装配或初始化逻辑是否不稳定。  
- 启动总耗时：生命周期链路太重时最先受影响。  
- 容器关闭耗时：销毁逻辑过重也会拖慢发布和重启。  
- 初始化阶段外部依赖调用次数：看是否把太多业务逻辑塞进了 Bean 生命周期。

**证据链**  
- 启动日志  
- Bean 创建顺序和耗时  
- `BeanPostProcessor` 执行路径  
- 初始化 / 销毁回调配置

**低风险优化路径**  
1. 先找重初始化 Bean  
2. 再把重逻辑从生命周期回调中剥离  
3. 最后再优化启动顺序和并行预热

**易错点**  
- 只会背实例化和注入  
- 不知道初始化前后还有扩展点  
- 不会把生命周期和启动问题联系起来

## 3. Spring 里单例 Bean 是线程安全吗？

**核心结论**  
Spring 的单例 Bean 默认只是“容器里只有一份实例”，并不天然等于线程安全。

**标准答案**  
Spring 的单例 Bean 指的是在同一个容器里通常只创建一个实例，但这和线程安全不是一回事。线程安全取决于这个 Bean 内部有没有共享可变状态，以及这些状态有没有被正确同步。之所以很多人会误解，是因为“单例”听起来像一种特殊安全语义，但 Spring 的单例只是作用域概念。工程上，如果一个单例 Bean 是无状态的，比如只做参数处理和下游调用，那通常没问题；但如果它把请求级数据放到成员变量里，或者持有可变集合而没有同步保护，就会很危险。面试里最好强调：Spring 单例 Bean 最推荐的设计方式是无状态化。

**高分回答**
Spring singleton 只保证一个 `ApplicationContext` 中该 Bean 定义通常复用同一实例，不为其方法建立互斥或内存协议。无状态 Service 的调用状态在参数和局部变量中，每个线程有自己的栈，因此通常可并发；但它引用的客户端、缓存、格式化器和回调也必须满足各自线程安全契约，字段 final 只保证引用不重新赋值，不保证对象内部安全。

若确需共享状态，先判断是否应移到数据库、缓存或专门并发组件，再写出不变量并选择锁/原子更新。把 Bean 改成 prototype 往往无效：它被注入 singleton 时通常只解析一次，而且同一实例仍可能跨线程传递。应通过静态扫描可变字段、并发压力与业务不变量校验发现问题，而不是给所有方法盲目加 synchronized。

**追问点答案**  
- 为什么无状态单例通常没问题？  
因为没有共享可变状态，就不存在并发修改冲突。  
- 把请求参数放成员变量为什么危险？  
因为多个线程会同时访问和覆盖同一份状态。  
- prototype Bean 就一定线程安全吗？  
不一定；作用域决定创建/获取方式，不决定实例所有权。prototype 注入 singleton 时不会自动每次调用新建，需要 Provider/作用域代理等显式机制，而且实例若被共享仍有竞态。

**原理补充**  
线程安全的本质永远是“共享可变状态管理”，和 Spring 单例作用域本身没有直接因果关系。

**架构扩展**  
真实项目里，这类问题常发生在工具类、Service、缓存包装器里。比如开发图方便把用户上下文、SQL 结果、分页状态挂到成员变量上，本地测不出来，一上并发就错。成熟团队一般会明确约束：Spring 单例 Bean 默认必须无状态，请求态数据只能放方法参数、局部变量或上下文容器里。

**工程指标**  
- 并发下数据串用事件数：这是最直接的业务信号。  
- 脏数据 / 错用户上下文日志：典型无状态约束失守表现。  
- 单例 Bean 可变字段扫描结果：辅助静态识别风险。  
- 并发测试异常率：用于提前暴露单例线程安全问题。  
- 线程安全缺陷回归次数：看规范是否真正落地。

**证据链**  
- Bean 作用域  
- 成员变量定义  
- 并发压测样本  
- 错乱上下文日志

**低风险优化路径**  
1. 先把单例 Bean 无状态化  
2. 再把请求级状态移到局部变量  
3. 必要时再加同步或调整作用域

**易错点**  
- 把单例和线程安全画等号  
- 在单例 Bean 里保存请求级变量  
- 遇到并发错乱只会怪 Spring

## 4. AOP 的核心概念有哪些？

**核心结论**  
AOP 的核心是把横切关注点从业务主流程里抽出来，通过切面在指定连接点织入增强逻辑。

**标准答案**  
AOP 可以理解成一种“把通用逻辑从业务代码里抽离”的机制。它的几个核心概念通常包括：切面（Aspect），表示一组横切逻辑；连接点（Join Point），表示程序执行过程中可被增强的位置；通知（Advice），表示在什么时候做什么增强，比如前置、后置、环绕；切点（Pointcut），表示到底拦哪些方法或位置。之所以需要 AOP，是因为日志、鉴权、事务、监控、审计这类逻辑如果散落在业务代码里，会让主流程很乱，也很难统一治理。工程上，AOP 最大的价值是把横切逻辑集中治理，但也容易带来“调用看不见真实执行链”的问题。面试里如果能说出“AOP 是解耦横切关注点，不是让所有逻辑都变魔法”，就比较成熟。

**高分回答**
在 Spring AOP 中 Join Point 实际主要是方法执行；Pointcut 选择哪些 Bean/方法，Advice 定义前后/异常/环绕行为，Advisor 把切点与通知组合，代理在调用边界串起拦截器链。它是运行时代理模型，不等于 AspectJ 能织入构造器、字段访问等所有连接点。目标内部 `this.method()` 不经过代理，是事务、缓存和自定义切面失效的高频原因。

切面只适合语义稳定且可组合的横切策略。环绕通知必须正确传播返回值、异常和上下文，并明确多切面的顺序；在通知里做阻塞 IO 或吞异常会改变所有命中方法。上线应暴露切面命中、耗时和异常，检查切点是否过宽；业务状态迁移若藏进 AOP，会让静态调用图与实际副作用分裂，应改回显式服务编排。

**追问点答案**  
- 什么叫横切关注点？  
就是很多业务都需要，但又不属于业务核心职责的通用逻辑。  
- 环绕通知为什么最强？  
它能决定是否、何时、调用几次目标并转换结果/异常，因此也最危险；除重试等明确语义外通常只调用一次 `proceed`，并保持异常契约与 finally 清理。
- AOP 会不会让排障更难？  
会，如果切面太多、链路不透明，排查时很容易迷路。

**原理补充**  
AOP 的本质，是把“业务做什么”和“系统如何治理这个过程”拆成两个层面，分别实现。

**架构扩展**  
线上很常见的情况是团队前期很喜欢 AOP，后来切面越来越多，调用链越来越不透明，最后事务、日志、埋点、鉴权全都织在一起，排障非常痛苦。成熟团队通常会严格约束 AOP 使用范围，只用于真正稳定的横切逻辑，而不是把业务分支也塞进去。

**工程指标**  
- 切面执行耗时：看横切逻辑是否开始拖慢主链路。  
- 切面命中次数：判断 AOP 覆盖范围是否合理。  
- 调用链复杂度：切面过多时会明显上升。  
- 事务 / 日志 / 埋点一致性问题数：判断切面治理是否真正统一。  
- 排障平均耗时：AOP 滥用常会在这里反映出来。

**证据链**  
- 切面定义  
- 切点表达式  
- 代理链路日志  
- 调用链追踪数据

**低风险优化路径**  
1. 先限制 AOP 只做稳定横切逻辑  
2. 再审查切面是否过多过重  
3. 必要时把复杂逻辑回归显式代码

**易错点**  
- 只会背术语  
- 把业务主流程逻辑也塞进切面  
- 不知道 AOP 过度使用会伤害可维护性

## 5. 动态代理 JDK 和 CGLIB 区别？

**核心结论**  
JDK 动态代理基于接口，CGLIB 基于继承生成子类；两者都是 Spring AOP 常见代理手段。

**标准答案**  
JDK 动态代理要求目标对象至少实现一个接口，代理对象本质上也是围绕接口方法做转发；CGLIB 则通过生成目标类的子类来实现增强，因此不依赖接口。之所以 Spring 同时支持这两套方式，是因为实际项目里并不是所有类都面向接口编程。工程上，JDK 代理更轻量、更符合接口抽象；CGLIB 更灵活，但受限于不能代理 final 类或 final 方法。面试里最好强调：Spring 选哪种代理方式，不只是技术实现差异，更和代码设计风格强相关。

**高分回答**
JDK 代理生成实现目标接口的代理类，调用经 `InvocationHandler` 转发；Spring 的类代理基于 CGLIB/字节码生成目标子类并覆写可代理方法。接口代理暴露的是接口类型，调用未声明在接口上的目标方法会受限；类代理不能覆写 final/private 方法，构造与可见性也有边界。两者的业务性能差异通常不是选型主因，拦截器内容和调用是否经过代理更重要。

代理方式要带环境边界回答：Spring Framework 可根据接口与 `proxyTargetClass` 选择；Spring Boot 的 AOP 自动配置通常默认启用类代理，可通过配置切换。无论哪种代理，self-invocation 都绕过外层代理；排障应打印运行时 Bean 类型、检查是否从容器取代理、方法是否可代理以及切点是否命中，而不是只看类上有没有注解。

**追问点答案**  
- 为什么 final 类不能被 CGLIB 代理？  
因为 CGLIB 依赖继承生成子类，final 类无法被继承。  
- JDK 代理为什么必须有接口？  
因为它本质上是基于接口生成代理类。  
- Spring Boot 项目里常见哪种？  
Spring Framework 的默认选择会考虑是否有接口；但 Spring Boot AOP 自动配置通常默认 `proxy-target-class=true`，即类代理。应以项目配置和运行时代理类型为准，不能用一条规则概括所有 Spring 环境。

**原理补充**  
这两种代理方式本质上都是“在目标调用前后插入增强逻辑”，区别只是代理对象生成路径不同：一个靠接口，一个靠继承。

**架构扩展**  
线上最常见的问题不是“选哪个更快”，而是某个增强为什么没生效，最后才发现目标类没有接口、方法是 final，或者代理边界理解错了。成熟团队通常不会死记代理方式，而是会从“这个 Bean 最终有没有被代理、调用是不是走代理对象”这个角度去排障。

**工程指标**  
- 代理 Bean 数量：看代理机制覆盖范围。  
- 增强命中率：判断代理是否实际生效。  
- 启动期代理生成耗时：代理很多时会影响启动。  
- final 类 / 方法命中数：辅助识别 CGLIB 代理风险。  
- AOP 失效问题数：这通常才是项目里的真实痛点。

**证据链**  
- 目标类是否实现接口  
- 代理方式配置  
- 代理对象类型  
- 方法增强执行日志

**低风险优化路径**  
1. 先确认目标对象实际代理方式  
2. 再检查 final、接口和调用边界  
3. 最后才考虑统一代理策略配置

**易错点**  
- 只会背“一个接口一个继承”  
- 不知道 final 约束  
- 排障时只盯注解，不看实际代理对象

## 6. `@Autowired` 和 `@Resource` 区别？

**核心结论**  
`@Autowired` 更偏 Spring 生态，通常按类型注入；`@Resource` 更偏 JSR 标准，通常优先按名称再回退类型。

**标准答案**  
`@Autowired` 是 Spring 提供的依赖注入注解，默认更偏向按类型注入；`@Resource` 来自 JSR 规范，通常优先按名称匹配，再考虑类型。之所以这两个注解都常见，是因为 Spring 既支持自己的生态能力，也兼容标准注解。工程上，`@Autowired` 常常会配合 `@Qualifier` 一起解决多实现类注入问题，而 `@Resource(name = "...")` 在明确指定 Bean 名称时也很直接。面试里最好强调：它们不只是“两个长得像的注解”，底层匹配策略和生态归属都有差异。

**高分回答**
`@Autowired` 由 Spring 的类型解析体系驱动：先按可赋值类型找候选，再结合 `@Qualifier`、`@Primary`、优先级以及注入点名称等规则消歧，还能注入 `Optional<T>`、集合或 `ObjectProvider<T>`。`@Resource` 遵循 Jakarta Common Annotations 的资源语义，显式 name 时按名称查找；未显式指定时通常先用字段/属性名，再按类型回退，规则与 `@Autowired` 不是简单对称。

版本要说清：Spring 6 / Boot 3 使用 `jakarta.annotation.Resource`，旧项目常见 `javax.annotation.Resource`。工程上必需依赖仍优先构造器注入，多个实现用领域含义明确的 qualifier，不要依赖字段名偶然匹配 Bean 名；可选或延迟依赖用 Provider 明确表达，避免把 `required=false` 变成隐藏的 null 分支。

**追问点答案**  
- 多个同类型 Bean 时 `@Autowired` 怎么办？  
通常要配合 `@Qualifier` 或 `@Primary`。  
- `@Resource` 为什么有时更稳定？  
显式指定稳定资源名时解析意图明确，但重命名 Bean 仍会破坏运行时装配；它不是天然更稳定，类型安全与命名耦合需要按接口演进方式取舍。
- 两者能不能混用？  
可以，但团队最好形成统一约定，减少装配歧义。

**原理补充**  
依赖注入的关键不是注解长什么样，而是容器最终按什么规则去找那个 Bean。装配规则不清楚，问题就会频发。

**架构扩展**  
线上常见的问题是项目越来越大、实现类越来越多，结果开发默认一个 `@Autowired` 就想解决所有情况，最后启动失败、注入歧义一堆。成熟团队通常会在基础组件层统一约定：是按类型装配为主，还是按名称约束为主，再配套 `@Primary`、`@Qualifier` 的规范。

**工程指标**  
- Bean 装配失败次数：看注入规则是否清晰。  
- 多实现类歧义告警数：反映依赖边界设计是否混乱。  
- 启动失败率：很多 Spring 注入问题最终都体现在这里。  
- 配置变更影响范围：装配规则越模糊，影响范围越大。  
- Bean 名称冲突数：辅助识别 `@Resource` 风险点。

**证据链**  
- Bean 定义与命名  
- 注入点类型  
- `@Qualifier` / `@Primary` 配置  
- 启动装配异常日志

**低风险优化路径**  
1. 先统一团队装配约定  
2. 多实现类显式加限定  
3. 最后清理隐式依赖和歧义命名

**易错点**  
- 只记住一个按类型一个按名称  
- 不知道多实现类冲突怎么解  
- 团队里混用但没有统一规范

## 7. `@Component`、`@Service`、`@Repository` 区别？

**核心结论**  
这几个注解本质上都是组件注册注解，但语义层次不同，更多是在表达职责和为框架扩展预留入口。

**标准答案**  
从 Spring 容器注册角度看，`@Component`、`@Service`、`@Repository` 最终都可以让类被扫描成 Bean。它们的主要区别不在“能不能注册”，而在语义表达：`@Component` 更通用，`@Service` 更偏业务服务层，`@Repository` 更偏数据访问层。之所以要区分这些注解，是为了让代码结构更清晰，也便于框架在某些层做额外处理，比如 `@Repository` 与数据访问异常转换更相关。工程上，成熟项目通常不会乱用，而是让注解本身就能帮助别人快速理解类职责。面试里最好强调：它们在大多数场景下注册能力类似，但分层语义和扩展意图不同。

**高分回答**
`@Service`、`@Repository` 都是以 `@Component` 为元注解的 stereotype，因此默认扫描注册路径相同，但它们给架构与框架提供层级元数据。`@Repository` 的持久化异常转换不是注解单独完成的：需要相应后处理器识别 Repository，并由具体持久化技术的 translator 把底层异常转换为 Spring `DataAccessException` 层次。

`@Service` 当前主要表达业务服务语义，不应宣称自带事务；事务仍由 `@Transactional` 与基础设施配置决定。团队可基于 stereotype 做 ArchUnit/静态规则，禁止 Controller 直接依赖 Repository、限制模块扫描边界。若类职责与标记不一致，注解只会制造虚假分层，指标应看跨层依赖违规和异常转换覆盖，而非各注解数量。

**追问点答案**  
- 为什么不用全都写 `@Component`？  
可以，但分层语义会变弱，代码可读性和治理性变差。  
- `@Repository` 有什么额外价值？  
它表达持久层角色，并可被异常转换后处理器识别；是否真正转换取决于后处理器与持久化技术适配，不能把所有运行时异常都认为会自动变成 `DataAccessException`。
- `@Service` 真有特殊底层能力吗？  
通常更多是语义和分层表达，不是完全不同的注册机制。

**原理补充**  
这类注解的核心价值，是把“容器扫描注册”和“代码职责表达”合在一起，方便框架和团队协作。

**架构扩展**  
在大型项目里，分层语义非常重要。比如看到 `@Repository`，排障时自然会先联想到数据库、ORM、SQL；看到 `@Service`，就会认为它更接近业务编排层。成熟团队会利用这些语义来做代码治理，而不是随便标一层。

**工程指标**  
- 分层约定违规数：判断代码结构是否逐渐失控。  
- Bean 分类统计：看系统职责边界是否清晰。  
- 数据访问异常映射问题数：辅助识别持久层边界问题。  
- 模块依赖复杂度：分层语义清晰时通常会更可控。  
- 代码评审发现率：看注解语义是否真的帮助团队协作。

**证据链**  
- Bean 注解分布  
- 分层依赖图  
- 异常映射与日志分类  
- 代码规范检查结果

**低风险优化路径**  
1. 先统一分层语义  
2. 再让注解与实际职责一致  
3. 最后清理跨层依赖和混乱命名

**易错点**  
- 只会说“都能注册 Bean”  
- 不知道分层语义价值  
- 在持久层、服务层、工具层乱贴注解

## 8. `@Configuration` 和 `@Component` 区别？

**核心结论**  
`@Configuration` 本质上也是组件，但它更强调“配置类语义”和 `@Bean` 方法增强能力，不只是普通组件注册。

**标准答案**  
`@Configuration` 可以看成是更特殊的 `@Component`，它本身也会被 Spring 扫描为 Bean，但它主要用于声明配置类，尤其是里面会定义多个 `@Bean` 方法时。之所以它和普通 `@Component` 不一样，是因为 Spring 通常会对 `@Configuration` 类做额外增强，比如通过代理确保类中多个 `@Bean` 方法相互调用时仍然返回容器里的单例 Bean，而不是每次都新建对象。工程上，如果只是普通工具组件，用 `@Component` 就够了；如果你是在集中定义 Bean 装配关系、第三方组件初始化、条件化配置，通常更适合 `@Configuration`。面试里如果能补一句“`@Configuration` 不只是语义更强，还有配置类增强语义”，会很扎实。

**高分回答**
完整模式的 `@Configuration` 默认 `proxyBeanMethods=true`，Spring 为配置类生成子类，拦截同类内 `@Bean` 方法调用并从容器取现有 Bean，从而保持作用域与生命周期语义。`proxyBeanMethods=false` 或普通 `@Component` 中的 `@Bean` 属于 lite 模式，方法调用只是普通 Java 调用，直接调用会新建对象，不应依赖方法互调表达 Bean 引用。

如果每个 `@Bean` 通过方法参数声明依赖，就不需要配置类方法互调，使用 lite 模式可减少增强限制和启动开销，也更适合 AOT；若遗留配置依赖直接调用，贸然关闭代理会产生未托管实例、重复资源和销毁遗漏。变更前应扫描方法调用关系，以 Bean 实例数、运行时类型、启动时间和资源关闭结果回归。

**追问点答案**  
- 为什么配置类需要增强？  
因为 `@Bean` 方法之间可能相互调用，需要保证返回的仍是容器托管对象。  
- `@Configuration` 能不能当普通 Bean 用？  
可以，但通常更推荐它专注于配置职责。  
- 为什么有时 `@Component + @Bean` 也能跑？  
因为 lite 模式仍会把各 `@Bean` 方法作为工厂注册；区别出现在配置类内部直接调用这些方法时，此时不会被增强为容器查找。使用参数注入依赖则通常无需 full 模式。

**原理补充**  
`@Configuration` 的核心价值，是把“Bean 定义工厂”和“普通组件”区分开来，让容器能对配置类做特别处理。

**架构扩展**  
真实项目里，很多第三方组件接入、线程池注册、数据源配置、序列化器配置都适合放到 `@Configuration`。如果把这些装配逻辑散落在普通组件里，后面查 Bean 来源和启动链路会越来越乱。成熟团队通常会把配置类当成“系统装配层”，而不是随便写几个工具方法的地方。

**工程指标**  
- 配置类数量：看系统装配是否过度分散。  
- `@Bean` 注册耗时：配置过重时会影响启动。  
- Bean 来源可追踪性：判断装配逻辑是否清晰。  
- 配置冲突次数：复杂项目里很常见。  
- 启动路径复杂度：很多装配问题都会体现在这里。

**证据链**  
- 配置类定义  
- `@Bean` 方法调用关系  
- 启动期 Bean 注册日志  
- 代理增强路径

**低风险优化路径**  
1. 先把装配逻辑集中到配置类  
2. 再避免把业务逻辑塞进配置  
3. 最后梳理配置冲突和条件装配

**易错点**  
- 把 `@Configuration` 当成只是更正式的 `@Component`  
- 不知道它和 `@Bean` 方法增强的关系  
- 配置类里混入大量业务逻辑

## 9. `@Bean` 和 `@Component` 区别？

**核心结论**  
`@Component` 适合让类被扫描自动注册，`@Bean` 适合在配置类里显式定义和装配第三方或复杂对象。

**标准答案**  
`@Component` 更像“告诉 Spring 扫描到这个类就把它注册成 Bean”；`@Bean` 则是“由配置类方法显式返回一个对象给容器管理”。之所以两者都存在，是因为有些对象是你自己写的类，适合组件扫描；有些对象来自第三方库、构造复杂、初始化参数依赖环境，更适合显式创建。工程上，`@Bean` 非常适合线程池、数据源、客户端 SDK、序列化器、策略组合对象这类装配逻辑；而业务类通常更适合 `@Component`、`@Service` 这类注解。面试里最好强调：二者不是谁高谁低，而是“自动发现”和“显式装配”两种风格。

**高分回答**
`@Component` 把注册元数据放在类型自身，适合拥有源码且角色稳定的应用组件；`@Bean` 把创建决策放在装配层，适合第三方类型、多个命名实例、条件化构造和需要显式配置的基础设施。两者产出的对象进入同一容器生命周期，`@Bean` 并不比扫描 Bean 更“高级”或天然单例，作用域仍由定义决定。

工厂方法应保持确定、短小且无隐藏副作用。连接池、客户端等 Bean 要显式声明配置校验、destroy 方法和观测标签；重复配置类或过宽扫描可能注册同类型多实例，不能靠允许 Bean 覆盖静默解决。使用 Actuator/启动报告追踪来源，测试中验证候选数量与关键属性，而不是只断言上下文能启动。

**追问点答案**  
- 为什么第三方类常用 `@Bean`？  
因为你不能给第三方类源码加 `@Component`。  
- `@Bean` 适合复杂初始化吗？  
适合显式表达构造参数、条件和销毁，但不应在工厂方法内执行不可控远程预热；复杂运行准备应拆成可观测、可超时的生命周期阶段。
- 业务 Service 为什么更少手写 `@Bean`？  
因为组件扫描已经足够，手写会增加装配噪声。

**原理补充**  
这两种方式本质上是在回答同一个问题：Bean 从哪里来。一个靠扫描发现，一个靠配置工厂显式提供。

**架构扩展**  
大型系统里，Bean 注册方式本身也体现架构风格。业务域组件通常走扫描，基础设施层和第三方接入通常走 `@Bean` 装配，这样边界更清晰。成熟团队会避免两种方式混乱交叉，防止 Bean 来源难追踪。

**工程指标**  
- Bean 注册来源分布：看装配风格是否清晰。  
- 第三方组件接入失败率：判断 `@Bean` 配置是否稳定。  
- 业务 Bean 显式装配比例：太高时通常意味着装配过度复杂。  
- 启动失败与重复注册次数：常见装配混乱信号。  
- 配置变更影响范围：显式装配和扫描混用时容易变大。

**证据链**  
- Bean 注册清单  
- 配置类与扫描路径  
- 重复 Bean 或冲突日志  
- 第三方组件初始化代码

**低风险优化路径**  
1. 先明确哪些 Bean 走扫描、哪些走显式装配  
2. 再统一团队约定  
3. 最后清理重复注册和来源不清问题

**易错点**  
- 只会说一个是注解一个是方法  
- 不知道第三方类为什么适合 `@Bean`  
- 业务 Bean 注册风格混乱

## 10. Spring 事务的实现原理？

**核心结论**  
Spring 事务本质上依赖 AOP 代理，在方法调用前后织入事务开启、提交、回滚等逻辑。

**标准答案**  
Spring 的声明式事务通常不是业务方法自己直接操作事务，而是由 Spring 在目标方法外层包上一层代理逻辑。调用进来时，代理先根据事务配置决定是否开启事务、绑定连接、设置隔离级别和传播行为；目标方法执行完成后再决定提交还是回滚。之所以这样设计，是因为事务本身就是典型横切逻辑，不应该污染业务代码。工程上，Spring 事务的关键不只是 `@Transactional` 这个注解，而是“这个调用有没有真的走到代理对象上”。面试里如果能补一句“事务是代理织进去的，不是注解贴上就自动有”，会非常像真实排障经验。

**高分回答**
声明式事务由 `TransactionInterceptor` 通过代理拦截方法：根据方法和类上的事务属性选择 `PlatformTransactionManager`，按传播行为获取/创建事务，将资源绑定到当前执行上下文，调用目标后依据返回或异常规则提交/回滚，并在 finally 清理。对 JDBC 常见的是线程绑定 Connection，因此普通事务上下文不会自动跨 `@Async`、手建线程或另一个线程池传播。

代理只是控制入口，真正原子边界仍由同一事务管理器覆盖的资源决定。一个方法同时写数据库和发 MQ，不会因 `@Transactional` 获得跨系统原子性，需要 outbox 等协议。Reactive 事务使用 Reactor Context 而非传统 ThreadLocal，不能用阻塞模型解释。排障应验证实际代理、事务管理器、传播、线程切换、异常规则和数据库连接状态，并监控事务 TP99、连接占用、锁等待及回滚原因。

**追问点答案**  
- 为什么事务和 AOP 常一起讲？  
因为 Spring 声明式事务通常就是 AOP 的一个典型应用。  
- 事务开启时通常绑定什么？  
具体由事务管理器决定；传统 JDBC 通常把连接/资源 holder 绑定线程上下文，Reactive 则绑定 Reactor Context。跨线程不会自动继承，需重新定义事务边界。
- 为什么代理边界这么重要？  
因为只有经过代理，事务增强逻辑才会执行。

**原理补充**  
Spring 事务的本质，是把数据库事务控制从业务代码里抽出来，用代理在调用边界统一处理。

**架构扩展**  
线上很多事务问题，根因根本不是数据库，而是调用没走代理、自调用绕过代理、异常被吞掉或者方法可见性不对。成熟团队在设计事务边界时，会同时考虑服务拆分、调用链、异常语义和代理边界，而不是只会贴注解。

**工程指标**  
- 事务成功 / 回滚率：看事务边界是否稳定。  
- 事务平均耗时：判断事务是否过重。  
- 事务失效问题数：这是 Spring 项目里很真实的指标。  
- 数据库连接占用时长：看事务是否把连接绑太久。  
- 锁等待时间：事务边界不合理时常被拉高。

**证据链**  
- `@Transactional` 配置  
- 代理对象类型  
- 调用链日志  
- 数据库事务与连接状态

**低风险优化路径**  
1. 先确认调用是否走代理  
2. 再检查异常和传播配置  
3. 最后优化事务边界和持有时长

**易错点**  
- 以为 `@Transactional` 自带魔法  
- 不知道事务其实靠代理  
- 排查事务问题只盯数据库不看调用边界

## 11. 事务传播行为有哪些？

**核心结论**  
事务传播行为决定的是“当前方法进来时，已有事务要不要加入、挂起、独立开启或强制要求”。

**标准答案**  
Spring 事务传播行为最常见的有 `REQUIRED`、`REQUIRES_NEW`、`SUPPORTS`、`MANDATORY`、`NOT_SUPPORTED`、`NEVER`、`NESTED`。其中最常用的是 `REQUIRED`，表示有事务就加入，没有就新开；`REQUIRES_NEW` 表示总是新开一个事务，必要时挂起外层事务；`SUPPORTS` 表示有就加入，没有也能跑；`MANDATORY` 表示必须在事务里；`NOT_SUPPORTED` 表示不要事务；`NEVER` 表示不能在事务里；`NESTED` 则更偏保存点语义。之所以传播行为这么多，是因为业务方法之间的事务边界关系很复杂，并不是所有子调用都应该和外层事务共命运。工程上，最常见的误用是到处乱用 `REQUIRES_NEW`，结果事务更碎、连接占用更重。面试里最好强调：传播行为是在描述“事务之间怎么相处”，不是在描述隔离级别。

**高分回答**
传播行为只在“通过事务拦截器进入方法”时计算。`REQUIRED` 复用现有物理事务，内层把事务标记 rollback-only 后，即使外层捕获异常仍可能在提交时得到 `UnexpectedRollbackException`；`REQUIRES_NEW` 挂起外层资源并借另一连接开启独立事务，内层提交不随外层回滚；`NESTED` 通常在同一物理事务用 savepoint 局部回滚，最终仍依赖外层提交，且需要事务管理器/驱动支持。

`REQUIRES_NEW` 不是可靠审计的通用答案：并发请求各自持有外层连接又等待内层连接，连接池不足会互相阻塞；外层回滚后审计已提交也可能记录一个最终不存在的业务结果。应先画物理事务、连接和失败语义，监控嵌套次数、连接等待与 rollback-only；跨系统审计优先 outbox，而不是靠新事务假装原子。

**追问点答案**  
- `REQUIRED` 为什么最常用？  
因为大多数业务方法都希望自然加入当前事务上下文。  
- `REQUIRES_NEW` 适合什么场景？  
像审计日志、补偿记录这类希望独立提交的场景。  
- `NESTED` 和 `REQUIRES_NEW` 一样吗？  
不一样。NESTED 通常复用外层物理事务并建立 savepoint，可局部回滚但不能独立于外层最终提交；REQUIRES_NEW 使用独立物理事务并挂起外层，额外占连接。实际支持取决于事务管理器。

**原理补充**  
传播行为本质上是在定义“方法嵌套调用时，事务上下文如何传递和切换”。这是事务边界设计问题，而不是数据库配置问题。

**架构扩展**  
真实系统里，传播行为选错最常见的后果是：你以为日志一定落库，结果跟主事务一起回滚了；或者你以为只是做个独立记录，结果 `REQUIRES_NEW` 开太多新事务把连接池打满。成熟团队通常只在少数明确场景下使用特殊传播行为，默认还是让事务边界尽量简单。

**工程指标**  
- 新事务创建次数：看 `REQUIRES_NEW` 是否被滥用。  
- 事务嵌套深度：辅助识别边界是否过于复杂。  
- 连接池等待时间：传播行为不当时常会放大这里。  
- 审计 / 补偿记录提交成功率：验证独立事务设计是否达到目的。  
- 回滚传播异常数：判断事务边界是否符合预期。

**证据链**  
- 事务传播配置  
- 调用链和事务日志  
- 连接池状态  
- 独立事务业务样本

**低风险优化路径**  
1. 默认优先 `REQUIRED`  
2. 只有明确独立提交需求才用 `REQUIRES_NEW`  
3. 复杂嵌套场景先画清调用与事务边界

**易错点**  
- 把传播行为和隔离级别混淆  
- `REQUIRES_NEW` 到处乱用  
- 不知道它会影响连接池和回滚关系

## 12. 哪些异常会触发回滚？

**核心结论**  
Spring 默认通常对运行时异常和 Error 回滚，对受检异常不自动回滚，除非显式指定。

**标准答案**  
在 Spring 默认事务规则里，运行时异常（`RuntimeException`）和 `Error` 通常会触发回滚，而受检异常（`Exception` 但不是运行时异常）默认不一定自动回滚。之所以这样设计，是因为 Spring 倾向于把运行时异常视为不可恢复错误，而受检异常更偏向业务层显式处理。工程上，这也是很多事务问题的来源：开发者明明抛了异常，却发现事务没回滚，最后才发现抛的是受检异常，或者异常已经被 catch 掉了。面试里最好强调：事务回滚既受异常类型影响，也受异常传播路径影响。

**高分回答**
代理根据“从目标方法传播出来的 Throwable”匹配 rollback 规则：默认 RuntimeException 和 Error 回滚，checked exception 默认提交；`rollbackFor`/`noRollbackFor` 可覆盖。规则不应按“技术异常还是业务异常”的模糊印象配置，而要问发生该异常时数据库状态能否合法提交。例如库存不足可能是正常业务分支无需抛异常，部分写入后遇到受检 IOException 则通常必须回滚。

catch 后可重新抛出满足规则的异常，或在确需转换为返回结果时显式 `setRollbackOnly`，但后者隐藏控制流，应谨慎使用。还要注意方法返回失败对象、异步任务稍后失败或事务提交后事件监听异常，都不会自动回滚已经结束的事务。测试应同时断言异常和数据库最终状态，并覆盖提交阶段异常、内部 catch 与外部代理边界。

**追问点答案**  
- 受检异常怎么让它也回滚？  
通常可以在 `@Transactional` 里显式配置 `rollbackFor`。  
- 异常被 catch 掉还会回滚吗？  
若异常没有越过事务拦截器且事务未被标记 rollback-only，代理会按正常返回提交。可以重新抛出、显式标记回滚，或重构让事务边界获得明确失败信号。
- 为什么默认不对所有异常都回滚？  
因为有些异常在设计上可能就是业务可处理分支。

**原理补充**  
回滚规则的本质，是 Spring 在代理层根据异常类型和传播结果做事务决策，而不是数据库自己判断“该不该回滚”。

**架构扩展**  
真实项目里，最常见的事务回滚事故不是规则不会背，而是 catch 了异常只打日志不继续抛，结果数据部分提交。成熟团队通常会统一异常处理和事务规范，明确哪些异常是业务分支，哪些必须让事务失败。

**工程指标**  
- 事务回滚率：看事务失败是否符合预期。  
- 未回滚脏数据事件数：最直接的风险信号。  
- 异常吞掉次数：排查事务不回滚的重要线索。  
- rollbackFor 显式配置覆盖率：看关键事务是否明确规则。  
- 业务补偿触发率：回滚失效常会在这里补出来。

**证据链**  
- 异常类型  
- `@Transactional` 配置  
- catch / throw 路径  
- 回滚与提交日志

**低风险优化路径**  
1. 先明确异常分类  
2. 关键场景显式配置回滚规则  
3. 统一异常处理规范，避免吞异常

**易错点**  
- 以为抛任何异常都会回滚  
- catch 后不再抛  
- 不知道受检异常默认规则

## 13. `@Transactional` 为什么有时不生效？

**核心结论**  
`@Transactional` 不生效，最常见原因不是 Spring 坏了，而是调用没走代理、方法边界不对、异常规则不对或者 Bean 根本没被代理。

**标准答案**  
`@Transactional` 不生效的常见原因很多，但核心通常可以归成几类：第一，自调用问题，也就是类内部方法直接互相调用，绕过了代理；第二，目标方法不是 public，或者代理方式和方法特征不匹配；第三，异常被吞掉了，Spring 感知不到失败；第四，Bean 本身没有被 Spring 托管或没有走到事务代理；第五，事务传播行为和预期不一致。之所以它这么容易出问题，是因为事务本质上不是注解魔法，而是代理 + 规则 + 调用边界三者共同生效。工程上，排查时不要一上来怀疑数据库，先确认“有没有代理、有没有进代理、异常有没有出去”。面试里如果能把排查顺序讲清楚，会非常有说服力。

**高分回答**
排查用一条可证伪链：Bean 是否由当前容器管理并实际形成事务 Advisor；调用方拿到的是代理还是 `new` 出来的目标；本次调用是否从代理外部进入；方法在当前代理类型和 Spring 版本下是否可拦截；选中了哪个事务管理器；传播是否复用了意外事务；异常是否穿过拦截器并匹配回滚规则。最后用事务日志或数据库连接的 auto-commit/事务 ID 证明，而不是看到注解就推断。

其他高频边界包括：`@Async`/线程池切走线程上下文，private/final 方法无法按常规代理增强，测试或配置未启用事务管理，多数据源选错 manager，以及同类 self-invocation。非 public 方法支持随代理方式和 Spring 版本变化，面试应避免绝对化。修复优先把事务入口提取到独立 Bean 或显式 TransactionTemplate，不建议通过暴露代理或自注入长期维持隐蔽调用。

**追问点答案**  
- 自调用为什么绕过事务？  
因为内部直接 `this.xxx()` 调用不会经过代理对象。  
- 方法不是 public 就一定不行吗？  
不应绝对化：JDK 接口代理受接口 public 方法边界约束，较新 Spring 的类代理可支持部分 protected/package-visible 方法，但 private/final 仍不能被覆写拦截；公开事务入口最清晰，并应以运行时代理验证。
- 为什么异常被吞掉事务就不回滚？  
因为代理层没有看到异常传播出来，就会认为方法执行成功。

**原理补充**  
事务失效问题的本质，几乎都能回到“代理有没有介入”这个核心点上。没有代理，就没有声明式事务。

**架构扩展**  
线上最典型的情况是开发以为注解贴上就完了，结果某次内部方法调用没回滚，数据已经部分落库。成熟团队通常会在事务重场景里把边界拉到独立 Service，尽量避免内部自调用和复杂代理路径。

**工程指标**  
- 事务失效故障数：这是团队事务治理成熟度的直接信号。  
- 代理命中率：判断目标调用是否真的走了代理。  
- 异常吞掉次数：事务失效排查的重要线索。  
- 脏数据补偿次数：很多事务失效最终会落到这里。  
- Service 内部自调用扫描结果：可做静态治理。

**证据链**  
- 调用路径  
- 代理对象类型  
- 异常传播日志  
- `@Transactional` 与传播配置

**低风险优化路径**  
1. 先确认 Bean 是否被代理  
2. 再检查调用边界和异常路径  
3. 最后收敛事务职责到明确 Service 层

**易错点**  
- 只会说“可能没加注解”  
- 不知道自调用问题  
- 一出问题就怪数据库不回滚

## 14. Spring 循环依赖怎么解决？

**核心结论**  
Spring 对部分单例循环依赖可以通过提前暴露对象引用来解决，但这不是推荐设计，而是一种兼容机制。

**标准答案**  
Spring 在处理单例 Bean 时，对某些循环依赖场景可以通过三级缓存和提前暴露对象引用来解决，也就是 Bean 还没完全初始化好时，先把一个早期引用暴露出去，供另一个 Bean 注入，从而打破死循环。这样做主要是为了兼容历史项目和复杂装配场景，而不是鼓励大家写循环依赖。工程上，它通常只能解决部分单例字段注入类型的循环依赖，对于构造器循环依赖就没那么容易了。更重要的是，循环依赖往往说明设计边界有问题：职责没拆清、方向反了、耦合太重。面试里最好强调：Spring 能帮你兜住一部分，但最好的方案通常还是重构设计。

**高分回答**
三级结构的关键不是背缓存名，而是代理一致性：实例化后的 singleton 可注册一个 ObjectFactory，需要早期引用时再生成原对象或早期代理；另一个 Bean 注入该引用后，原 Bean 完成属性填充和初始化，最终放入完整单例缓存。这样只覆盖允许早期暴露的 singleton 属性依赖，并要保证最终注入引用与后处理代理兼容。

构造器 A→B→A 在 A 尚未实例化时就要求完整 A，无法靠早期实例解决；prototype 也没有单例缓存生命周期。并且 Spring Boot 近年默认禁止循环引用，不能把“Spring 总能三级缓存解决”当当前默认。治理先画依赖方向，拆出共同职责、改成事件/回调或重新定义端口；`@Lazy`/Provider 只是延后解析，必须记录为临时债务并测试首次调用路径。

**追问点答案**  
- 为什么构造器循环依赖更难处理？  
因为对象连实例都还没完整创建出来，没法提前给出引用。  
- 为什么说循环依赖本身是设计味道？  
因为它意味着两个组件彼此强依赖，职责边界通常不清晰。  
- 哪些手段能缓解循环依赖？  
优先拆共同职责或反转依赖，通过领域事件解除同步双向调用；Provider/`@Lazy` 可临时延迟一端，但会把启动失败推迟到运行时，需有移除计划。不要为兼容而全局重新允许循环引用。

**原理补充**  
Spring 解决循环依赖的本质不是“魔法修好设计”，而是通过对象生命周期技巧让一部分场景先跑起来。

**架构扩展**  
真实系统里，如果一个模块里到处循环依赖，说明不是 Spring 太强，而是架构边界已经开始腐化。很多团队一开始靠 `@Lazy` 和字段注入顶住，后面越来越难维护。成熟团队通常会把循环依赖当成重构信号，而不是框架特性。

**工程指标**  
- 循环依赖 Bean 数量：看系统耦合是否正在恶化。  
- `@Lazy` 使用频次：有时能反映是不是在用技巧掩盖设计问题。  
- 启动失败次数：复杂循环依赖场景最容易先暴露在这里。  
- 模块互相依赖度：判断边界是否混乱。  
- 重构后依赖收敛情况：验证治理是否有效。

**证据链**  
- Bean 依赖图  
- 字段注入 / 构造器注入方式  
- 启动异常日志  
- `@Lazy` 与循环引用配置

**低风险优化路径**  
1. 先识别循环依赖链  
2. 优先通过职责拆分消除  
3. 临时场景再考虑 `@Lazy` 或延迟注入

**易错点**  
- 以为 Spring 既然能处理就说明没问题  
- 不知道构造器循环依赖更难解  
- 用技巧掩盖架构边界问题

## 15. Spring MVC 请求处理流程？

**核心结论**  
Spring MVC 的核心流程是：请求先进入 `DispatcherServlet`，再经由处理器映射、适配器、参数绑定、控制器执行、视图或响应体渲染完成整个请求。

**标准答案**  
一个 HTTP 请求进入 Spring MVC 后，通常先到前端控制器 `DispatcherServlet`。它会根据请求路径去找匹配的 Handler，也就是目标控制器方法；找到之后，通过对应的 HandlerAdapter 调用控制器；调用前会做参数解析、数据绑定、消息转换；方法执行完成后，再根据返回值类型决定是走视图解析，还是直接序列化成响应体返回。之所以 Spring MVC 要设计这么多层，是因为它希望把“请求路由、参数绑定、控制器调用、返回结果处理”拆成可扩展链条。工程上，理解这个流程最有价值的地方，是能帮你排查参数绑定失败、拦截器顺序、异常处理和消息转换问题。面试里如果能补一句“`DispatcherServlet` 是总调度入口，后面全是可扩展策略链”，会很稳。

**高分回答**
请求首先经过 Servlet Filter 链，再由 `DispatcherServlet` 查询 HandlerMapping 得到 HandlerExecutionChain；拦截器 `preHandle` 通过后，HandlerAdapter 调用目标方法。期间各 `HandlerMethodArgumentResolver` 解析参数，DataBinder/Validator 做绑定校验；返回值由 `HandlerMethodReturnValueHandler` 处理，`@ResponseBody` 通常委托 HttpMessageConverter 按内容协商写响应，模板场景才进入 ViewResolver。

异常可在 HandlerAdapter 调用阶段交给 HandlerExceptionResolver 链，如 `@ExceptionHandler`/`@ControllerAdvice`；若响应已提交，统一异常处理可能无法改写。排障应先按阶段分类 404 路由、400 绑定校验、415 请求媒体类型、406 响应协商、500 业务异常，并关联 filter/interceptor/controller/serialization 分段耗时，避免把所有失败归给 Controller。

**追问点答案**  
- 为什么要有 HandlerAdapter？  
因为 Spring 希望不同类型的处理器都能被统一调度，而不是强耦合一种调用方式。  
- JSON 返回是在哪一步完成的？  
通常是在返回值处理和消息转换阶段完成。  
- 参数绑定失败一般查哪？  
先看目标参数使用的 ArgumentResolver，再查消息转换、类型 Converter、DataBinder、校验注解和 BindingResult；结合状态码区分请求体媒体类型、格式错误与业务校验失败。

**原理补充**  
Spring MVC 的本质，是把一次 Web 请求拆成多段策略化处理链，每段都可扩展、可替换、可插拦截逻辑。

**架构扩展**  
线上很多接口问题看似是 Controller 逻辑错，其实根因在更前面，比如请求根本没匹配到 Handler，参数转换失败，或者消息转换器没配对。成熟团队会把 MVC 请求链路理解成一条完整处理链，而不是只盯 Controller 方法。

**工程指标**  
- 请求路由命中率：看请求是不是正确进入目标 Handler。  
- 参数绑定失败率：识别 Converter / 校验问题。  
- 消息转换失败次数：JSON/XML 返回问题常见信号。  
- Controller 执行耗时：区分路由问题和业务逻辑问题。  
- 全局异常处理命中率：看异常链路是否统一收敛。

**证据链**  
- 请求日志  
- Handler 映射日志  
- 参数绑定与消息转换异常  
- 返回值处理链日志

**低风险优化路径**  
1. 先定位问题在路由、绑定、执行还是返回  
2. 再针对对应策略链调试  
3. 最后优化统一异常和拦截治理

**易错点**  
- 只会说“请求进 Controller”  
- 不知道中间还有映射、适配和转换链  
- 排障时只盯业务方法

## 16. `DispatcherServlet` 作用是什么？

**核心结论**  
`DispatcherServlet` 是 Spring MVC 的前端控制器，负责把所有请求统一调度到后续处理链。

**标准答案**  
`DispatcherServlet` 可以理解成 Spring MVC 的总入口。所有符合映射规则的请求先进入它，然后由它去协调 HandlerMapping 找目标处理器、HandlerAdapter 调用控制器、ViewResolver 解析视图、异常解析器处理异常、消息转换器写出响应。之所以 Spring MVC 要有这样一个前端控制器，是为了把请求处理过程统一收口，方便后面接各种策略和扩展点，而不是让每个 Controller 各自处理所有事情。工程上，理解 `DispatcherServlet` 的价值在于你能知道 Web 问题为什么往往不是“Controller 不执行”这么简单，而是可能卡在它分发链的任意环节。面试里可以总结一句：`DispatcherServlet` 不是业务处理器，而是请求调度中枢。

**高分回答**
`DispatcherServlet` 是 Servlet 容器中的一个 Servlet，只处理映射到它的请求；在它之前仍有容器、Filter 链和可能的 Spring Security FilterChain。进入后，它从已初始化的策略组件中选择 HandlerMapping、HandlerAdapter、异常解析器、视图/返回值处理等完成调度。REST 返回并非 DispatcherServlet 自己直接“转 JSON”，而是 HandlerAdapter 内的返回值处理器调用 HttpMessageConverter。

统一入口带来可治理性，也意味着入口扩展必须轻量。Interceptor 适合 Handler 级前后处理，Filter 适合 Servlet 请求/响应与安全链，`@ControllerAdvice` 适合 MVC 异常/绑定；三者作用域与异常可见范围不同。排障先确认请求是否进入 Servlet mapping，再看 Filter 是否短路、HandlerMapping 是否命中、Adapter/Converter 和异常解析，配合各阶段计时定位。

**追问点答案**  
- 为什么叫前端控制器？  
因为它站在所有请求入口前面，统一控制后续分发。  
- 它自己处理业务吗？  
不执行领域业务，但负责调度、异常解析、multipart、locale、视图等框架流程；实际参数解析和控制器调用多由选中的 HandlerAdapter 及其组件完成。
- 为什么这层设计很重要？  
因为有了统一入口，拦截、异常、路由、视图等能力才容易集中治理。

**原理补充**  
`DispatcherServlet` 的本质价值是“统一入口 + 策略调度”，这使 MVC 成为一个高度可插拔的处理框架，而不是若干 Controller 的散装集合。

**架构扩展**  
线上很多问题，比如某类请求没命中、统一异常没生效、拦截器顺序异常，本质上都要回到 `DispatcherServlet` 调度链去看。成熟团队在做网关、Web 框架扩展或全局治理时，通常都会围绕这一层理解系统行为。

**工程指标**  
- 入口请求总数：看调度层总体压力。  
- 分发失败率：判断请求是否卡在路由阶段。  
- 全局异常命中率：看统一异常治理是否通过这层收口。  
- 拦截器执行耗时：很多非业务逻辑也会卡在入口链。  
- 路由未匹配比例：辅助识别配置和映射错误。

**证据链**  
- `DispatcherServlet` 日志  
- HandlerMapping / HandlerAdapter 路径  
- 未命中和异常处理日志  
- 拦截器链执行顺序

**低风险优化路径**  
1. 先把入口链路可观测化  
2. 再区分调度失败和业务失败  
3. 最后治理拦截器和异常统一出口

**易错点**  
- 以为 `DispatcherServlet` 就是 Controller  
- 不知道它负责统一调度  
- 排障时完全跳过入口链路

## 17. 参数绑定和消息转换器是怎么工作的？

**核心结论**  
参数绑定负责把请求里的各种输入映射成方法参数，消息转换器负责把请求体和响应体在对象与文本/二进制之间来回转换。

**标准答案**  
当 Spring MVC 调用控制器方法时，会先根据参数类型、注解和请求内容决定如何解析参数。像路径变量、请求参数、Header、Cookie、请求体，这些都会由不同的参数解析器处理；如果参数是 `@RequestBody`，通常就会交给消息转换器把 JSON、XML 等内容转成 Java 对象。方法返回值如果是对象，也通常会再交给消息转换器序列化成响应体。之所以需要这样分层，是因为请求输入来源很多、返回格式也很多，不可能靠 Controller 手写解析。工程上，很多“前端明明传了值后端却是 null”“返回值怎么变成这副样子”的问题，本质都在参数绑定和消息转换器这层。面试里如果能说出“一个解决方法参数，一个解决请求体/响应体格式转换”，就很清晰。

**高分回答**
ArgumentResolver 先根据参数类型与注解决定来源：路径、query/form、header、cookie、认证主体或 request body。简单属性绑定由 WebDataBinder、ConversionService 与 Validator 协作；`@RequestBody` 则根据请求 `Content-Type` 选择可读 HttpMessageConverter，返回体再结合 `Accept` 做内容协商。415 表示输入媒体类型不受支持，406 表示无法产生客户端接受的响应，不能都归为 JSON 错误。

DTO 边界要安全：避免把持久化实体直接做宽泛绑定，限制可绑定字段，区分缺失与 null，统一时间、枚举、数值精度和未知字段策略。反序列化成功不代表业务合法，仍需 Bean Validation 和跨字段校验；错误响应要保留字段路径与稳定错误码但不回显敏感原文。契约变更应用 consumer contract/兼容测试覆盖旧客户端。

**追问点答案**  
- `@RequestParam` 和 `@RequestBody` 最大区别？  
一个更偏 URL 参数或表单参数，一个更偏请求体内容。  
- 为什么有时字段名对不上就绑定失败？  
因为参数绑定和反序列化都依赖字段映射规则。  
- 消息转换器为什么和 Content-Type 有关？  
请求 Content-Type 表明 body 的编码格式，用于选择 readable converter；响应还要结合控制器可生产类型与 Accept 协商 writable converter。配置错误分别常表现为 415 或 406。

**原理补充**  
参数绑定和消息转换器本质上分别解决“控制器怎么拿到输入”和“输入输出怎么编码/解码”两类问题。

**架构扩展**  
线上接口联调最常见的问题之一就是绑定和转换不一致：前端传 JSON，后端按表单收；或者返回对象里日期格式、枚举序列化规则不统一，导致客户端解析异常。成熟团队通常会统一 DTO 规范、JSON 序列化规则和 Content-Type 约定，减少这类低级但高频的问题。

**工程指标**  
- 参数绑定失败率：反映接口输入规范是否混乱。  
- 消息转换异常数：定位 JSON / XML 解析问题。  
- 400 错误比例：很多绑定问题最终会体现在这里。  
- 接口联调失败次数：这是非常真实的工程指标。  
- DTO 兼容性问题数：反映消息格式治理是否成熟。

**证据链**  
- 请求报文  
- Controller 参数定义  
- Converter 配置  
- 绑定和转换异常日志

**低风险优化路径**  
1. 先统一输入输出协议  
2. 再收敛 DTO 和字段命名规范  
3. 最后定制转换器或全局序列化规则

**易错点**  
- 把参数绑定和消息转换器混为一谈  
- 不关注 Content-Type  
- 接口联调出问题只怪前端或后端其中一边

## 18. Spring Boot 自动配置原理？

**核心结论**  
Spring Boot 自动配置的核心是“按条件装配”，让框架根据类路径、配置项、Bean 状态自动决定该注册哪些组件。

**标准答案**  
Spring Boot 的自动配置本质上是在启动时加载一批预定义配置类，然后根据条件判断是否生效。常见条件包括：某个类是否在类路径上、某个配置项是否存在、容器里是否已经有同类型 Bean、当前环境是否满足某种要求。之所以这种机制强大，是因为它让框架能在“默认可用”和“可被覆盖”之间取得平衡：大多数时候你几乎不用手配，但需要自定义时也能接管。工程上，自动配置的关键不是“神奇自动生效”，而是“条件装配 + 默认约定 + 用户可覆盖”。面试里如果能说出“Spring Boot 不是帮你省掉配置，而是把常见配置模式模板化了”，会很成熟。

**高分回答**
`@SpringBootApplication` 包含 `@EnableAutoConfiguration`，选择器导入 classpath 中声明的自动配置候选；Boot 3 主要从 `META-INF/spring/...AutoConfiguration.imports` 发现，旧版本常见 `spring.factories`。候选类按顺序和条件评估，`@ConditionalOnClass` 看依赖，`@ConditionalOnProperty` 看配置，`@ConditionalOnMissingBean` 让用户定义后退，最终只有匹配的 BeanDefinition 进入容器。

条件判断有时序和搜索范围，不能认为“我之后声明 Bean 就一定让之前条件回退”。排障用 Condition Evaluation Report 查看 positive/negative matches、配置属性绑定和 Bean 来源，先精确覆盖单个 Bean 或 exclude 单个自动配置，避免删除整个 starter。自定义自动配置要给出属性元数据、明确 back-off、限定条件、顺序和 ApplicationContextRunner 测试，防止因类路径偶然存在而污染所有应用。

**追问点答案**  
- 为什么有时自动配置没生效？  
可能是条件不满足，或者你自己提供了更高优先级的 Bean。  
- 自动配置为什么不怕冲突？  
它通过 missing-bean、property 等条件主动 back off，但并非不会冲突；候选同名、类型过宽或条件顺序错误仍会失败，Boot 默认也不会鼓励静默 Bean 覆盖，应看条件报告。
- 自动配置是不是全自动？  
不是，它是“基于约定和条件的半自动”。

**原理补充**  
自动配置的核心不是“自动”，而是“条件化装配”。它通过一整套判断规则，把配置行为做成了可推导过程。

**架构扩展**  
线上很多 Spring Boot 启动问题，本质上都和自动配置有关，比如某个 starter 引入后 Bean 多了一堆、条件配置没生效、覆盖关系不清。成熟团队在排查启动问题时，都会把自动配置报告和条件判断一起看，而不是只盯表面 Bean 异常。

**工程指标**  
- 自动配置生效数量：看启动时引入了多少框架能力。  
- 条件装配失败次数：辅助定位“为什么没生效”。  
- 启动耗时：自动配置越重，启动越容易被拖慢。  
- Bean 覆盖冲突次数：判断自定义和默认装配是否冲突。  
- starter 引入后变更影响范围：评估自动配置的副作用。

**证据链**  
- 自动配置报告  
- 条件判断日志  
- starter 依赖  
- Bean 定义冲突日志

**低风险优化路径**  
1. 先看条件装配报告  
2. 再明确是默认没生效还是被覆盖  
3. 最后做精确排除或自定义覆盖

**易错点**  
- 把自动配置理解成完全黑盒  
- 不会看条件装配  
- 一遇到启动异常就全删 starter

## 19. `starter` 机制原理是什么？

**核心结论**  
`starter` 的本质是把某一类能力所需依赖和自动配置打包成一个可复用入口，让用户通过一个依赖就获得整套能力。

**标准答案**  
Spring Boot 的 starter 可以理解成“能力入口包”。它通常自己代码不多，核心作用是把一组相关依赖、自动配置和约定默认值组合起来。比如你引入一个 Web starter，本质上是把 Web 所需的基础依赖和自动配置打包好，让你不用自己一项项拼。之所以 starter 很有价值，是因为它把“依赖收集”和“配置装配”标准化了，让框架能力更容易复用和传播。工程上，starter 的真正价值不是省几行依赖，而是统一能力接入方式、减少版本错配和接入成本。面试里如果能补一句“starter 本质上是依赖聚合 + 自动配置入口”，答案会很完整。

**高分回答**
Starter 通常是依赖描述符，解决“使用这项能力需要哪些兼容依赖”；auto-configure 模块才包含条件配置、属性类和 Bean 定义，两者可以合并但职责应区分。应用引入 starter 后，自动配置发现机制依据 classpath 和 properties 决定是否实例化；仅聚合依赖而没有自动配置，也可以叫 starter，但没有完成开箱即用的产品化闭环。

内部 starter 要避免强行接管应用：依赖应最小化并使用依赖管理控制兼容矩阵，所有外部连接提供 enable 开关、超时与 back-off，默认 Bean 可被用户替换，暴露健康、指标和诊断信息。发布要做多 Boot/JDK 版本契约测试和升级说明；退出机制同样重要，业务应能排除自动配置或替换实现，而不是被组织 starter 锁死。

**追问点答案**  
- starter 自己一定有很多代码吗？  
不一定，很多 starter 主要是依赖聚合和自动配置入口。  
- 为什么 starter 容易形成生态？  
因为它把能力接入标准化了，团队和社区都更容易复用。  
- 自定义 starter 有什么价值？  
可以沉淀兼容依赖、默认安全配置、自动装配、观测和升级策略；价值要用接入时间、配置缺陷、升级成功率证明，而不是把所有组织规范硬编码进不可退出的依赖。

**原理补充**  
starter 的核心，不是简单的 Maven 依赖包，而是“把一类技术能力变成可复制、可约定、可统一维护的产品化入口”。

**架构扩展**  
在大型公司或平台团队里，自定义 starter 非常常见，比如统一日志、链路追踪、配置中心、鉴权 SDK、数据库中间件接入。成熟团队会通过 starter 把组织经验沉淀成工程标准，而不是让每个项目都手工拼接。

**工程指标**  
- starter 接入耗时：衡量标准化是否真的降低成本。  
- 版本冲突次数：看 starter 是否帮助统一依赖体系。  
- 重复样板配置减少量：评估复用收益。  
- 接入失败率：判断 starter 质量和文档成熟度。  
- 组织内复用率：这是内部 starter 非常重要的指标。

**证据链**  
- starter 依赖树  
- 自动配置内容  
- 使用项目接入情况  
- 版本冲突与接入问题记录

**低风险优化路径**  
1. 先识别可标准化能力  
2. 再封装成 starter  
3. 最后用版本治理和文档支持推广

**易错点**  
- 以为 starter 只是省事的依赖包  
- 不知道它和自动配置强绑定  
- 自定义 starter 只做依赖聚合不做治理规范

## 20. 如何排查 Spring 启动慢问题？

**核心结论**  
Spring 启动慢排查的关键不是盲猜，而是先拆阶段：类加载慢、自动配置多、Bean 初始化重、外部依赖慢，还是扫描范围太大。

**标准答案**  
排查 Spring 启动慢时，首先要把启动过程拆开来看：是不是类路径扫描太重、自动配置加载太多、某些 Bean 初始化做了重逻辑、外部依赖连接太慢，还是代理和反射处理过于复杂。常见排查方式包括开启启动报告、看 Bean 初始化耗时、看自动配置条件报告、定位慢初始化 Bean、检查配置中心、数据库、缓存、消息中间件在启动阶段的连接行为。之所以不能笼统说“Spring 慢”，是因为 Spring 只是启动容器，真正慢的往往是具体阶段。工程上，最常见的慢点是 Bean 初始化过重和自动配置过多，而不是框架本身。面试里如果能说出“先拆阶段，再定位最慢 Bean 或最慢外部依赖”，会非常像线上排障经验。

**高分回答**
先定义测量边界：进程启动、ApplicationContext refreshed、readiness 可接流、首个请求预热完成是不同时间点。用 ApplicationStartup/启动步骤、JFR、Bean 创建耗时和线程栈建立 critical path，再把时间归入类扫描/条件评估、Bean 实例化后处理、外部连接、数据迁移和缓存预热。总耗时不是所有 Bean 耗时简单相加，还要找串行依赖链与并行阻塞。

优化按风险排序：移除无用 starter/扫描，缓存或减少重复元数据工作，给外部连接明确短超时，把非必需预热移到 readiness 后的受控后台任务。全局 lazy initialization 会缩短表面启动，却把配置错误和延迟推到首个请求，必须预热关键路径并比较首请求 TP99。每次只改一类因素，以冷/热多次启动的 P50/P95、readiness、首请求、内存和失败率验收并保留配置回退。

**追问点答案**  
- Bean 初始化为什么经常是罪魁祸首？  
因为很多项目会在启动阶段连数据库、拉配置、预热缓存、建线程池。  
- 自动配置怎么会拖慢启动？  
因为引入的 starter 越多，条件装配和相关 Bean 也越多。  
- 为什么扫描范围过大会变慢？  
候选类越多，资源扫描、元数据读取、条件判断和后处理越多，也可能意外注册组件；但应以启动步骤证明占比，现代索引/AOT环境下未必是首要瓶颈。

**原理补充**  
Spring 启动本质上是“扫描 + 判断 + 注册 + 初始化”这几段组合而成，任何一段变重都可能拖慢整体。

**架构扩展**  
线上很多团队把启动慢归咎于 Spring，最后发现是自己把几十个中间件、缓存预热、远程配置和任务初始化都塞进了启动流程。成熟团队通常会区分“必须启动期完成”和“可以后台异步完成”的工作，把真正重的动作从主启动链路中拆出去。

**工程指标**  
- 启动总耗时：全局观测入口。  
- 最慢 Bean 初始化 Top N：最直接的定位手段。  
- 自动配置生效数量：看是不是框架层过重。  
- 外部依赖连接耗时：识别是不是环境或中间件拖慢。  
- 类扫描数量：判断包扫描范围是否过大。  
- 重启恢复时长：这是部署效率和稳定性都关心的指标。

**证据链**  
- 启动报告  
- Bean 初始化耗时  
- 自动配置条件报告  
- 外部依赖连接日志  
- 扫描路径配置

**低风险优化路径**  
1. 先识别最慢阶段  
2. 再定位最慢 Bean 或最慢外部依赖  
3. 优先把重初始化异步化或延后  
4. 最后收敛 starter 和扫描范围

**易错点**  
- 把“Spring 启动慢”当统一结论  
- 不拆阶段直接乱优化  
- 重初始化全塞进启动流程
