# 基础模块1 - Java基础与集合 - 标准答案库

## 1. `==` 和 `equals` 区别是什么？

**核心结论**  
`==` 比较的是值或引用地址，`equals` 比较的是逻辑相等性，真正语义由类自己定义。

**标准答案**  
在 Java 里，`==` 如果用于基本类型，比较的是实际值；如果用于对象，比较的是两个引用是否指向同一个对象。`equals` 是 `Object` 提供的方法，默认实现也是比较地址，但很多类会重写它来比较内容。  
因为对象相等有两层语义，一层是“是不是同一个对象”，另一层是“业务上是不是同一个值”。Java 把这两件事拆开，是为了兼顾底层引用判断和上层业务建模。  
如果你关心的是身份，比如单例对象、枚举实例、缓存中的同一引用，就用 `==`；如果你关心的是内容或业务意义，就看类有没有正确重写 `equals`。像 `String`、包装类、很多值对象都依赖 `equals` 做逻辑判断。  
再往深一点说，集合去重、Map 查找、缓存命中都不是单独靠 `equals`，而是经常和 `hashCode` 一起工作，所以面试里最好把两者串起来讲。  
一句收尾：`==` 偏底层语义，`equals` 偏业务语义。

**高分回答**
`==` 的语义由操作数类型固定：基本类型比较数值，对象引用比较身份；`equals` 则是类对“逻辑相等”的建模。设计 `equals` 时必须满足自反、对称、传递、一致以及对 null 返回 false，真正困难的不是写字段比较，而是保证契约在继承、代理对象和对象生命周期变化后仍成立。

值对象通常适合按全部稳定属性判断相等；具有数据库身份的实体要先决定相等性是基于业务自然键还是持久化 ID。若 ID 在入库后才生成，把它直接用于 `equals` 会让对象在生命周期中改变身份，也会破坏 Set 和 Map 行为。

**追问点答案**  
- 为什么重写 `equals` 后通常必须重写 `hashCode`？  
因为哈希容器先用 `hashCode` 定位桶，再用 `equals` 判断逻辑相等。如果两个对象 `equals` 为真但 `hashCode` 不同，就会落到不同桶里，导致查找、去重异常。  
- `String` 的 `equals` 为什么这么常见？  
因为 `String` 是典型的值对象，我们几乎总是在比较“内容一样不一样”，而不是“是不是同一个字符串对象”。  
- `Integer` 的 `==` 为什么有时是 `true`？  
`Integer.valueOf` 至少缓存 `-128` 到 `127`，自动装箱可能复用对象；更大范围是否缓存属于实现配置，不能依赖。包装类型比较数值应使用 `equals`，并留意拆箱 null 会抛出异常。

**易错点**  
- 以为对象上的 `==` 比较的是内容  
- 以为所有类的 `equals` 都已经比较内容  
- 不知道包装类缓存会导致面试陷阱

## 2. `hashCode` 为什么要和 `equals` 一起重写？

**核心结论**  
因为哈希容器依赖“先 `hashCode`，再 `equals`”的两段式判断，两者必须保持一致。

**标准答案**  
`hashCode` 是对象在哈希结构里的散列值，`equals` 是对象逻辑相等性的判断规则。  
像 `HashMap`、`HashSet` 这类容器不会直接遍历全表找对象，而是先根据 `hashCode` 找桶，再在桶里用 `equals` 做最终确认。所以 Java 规定：如果两个对象 `equals` 为真，那么它们的 `hashCode` 必须相同。  
只要一个类需要按“内容”判断相等，比如值对象、DTO、作为 map key 的对象，就应该成对重写 `equals` 和 `hashCode`。  
如果只重写 `equals` 不重写 `hashCode`，对象逻辑相等但桶位置不同，可能出现 put 进去却 get 不到，或者 set 去重失败。  
反过来要补一句：`hashCode` 相同不代表一定相等，因为哈希冲突是允许的，最终仍然要靠 `equals` 决定。

**高分回答**
核心契约只有单向保证：`equals` 相等的对象必须有相同 `hashCode`，hash 相同的对象可以不相等。哈希质量影响桶分布和性能，契约一致性决定正确性。作为哈希键的对象还需要“驻留期间参与相等性计算的字段不变”，否则对象仍在旧桶中，却按新 hash 查找，表现为 key 明明存在却取不到。

继承体系是常见陷阱：父类用 `instanceof`、子类又增加相等字段，容易破坏对称性或传递性。值类型优先不可变并限制继承，或使用能够明确表达同类型比较的设计；不要靠 IDE 生成代码替代领域身份决策。

**追问点答案**  
- 为什么 `hashCode` 相同不代表 `equals` 一定为 `true`？  
因为哈希函数本来就可能冲突，不同对象散列到同一个值很正常，所以 `hashCode` 只是粗筛，不是最终判断。  
- Lombok 的 `@EqualsAndHashCode` 有什么风险？  
它只生成机制，不能替你决定身份。继承时是否调用父类、ORM 代理类型、懒加载字段和入库后才生成的 ID，都可能破坏契约或触发额外查询，应先写清相等性语义再配置生成策略。
- 可变对象当 `HashMap` key 会出什么问题？  
如果 key 参与 `hashCode` 计算的字段发生变化，对象原本所在桶就和当前 hash 不匹配，后续可能找不到。

**易错点**  
- 只重写 `equals`  
- 把会变化的字段放进 `hashCode`  
- 不理解哈希冲突和逻辑相等的区别

## 3. `String` 为什么不可变？

**核心结论**  
`String` 不可变是为了安全性、共享复用、哈希稳定和天然线程安全。

**标准答案**  
不可变的意思是字符串对象一旦创建，内部字符内容就不能被修改，任何“修改”本质上都是创建新对象。  
第一，安全，比如类名、文件路径、网络地址、SQL 片段这类基础信息如果可变会带来风险；第二，常量池可以安全复用，节省内存；第三，`hashCode` 可以缓存，不需要重复计算；第四，不可变对象天然线程安全。  
开发里把 `String` 当值对象使用即可，频繁拼接时不要反复生成新字符串，应改用 `StringBuilder`。  
面试里如果能再补一句“不可变的是对象，不是引用”，会更完整。引用变量可以重新指向别的字符串，但原字符串对象内容不会变。  
所以 `String` 的不可变，本质上是 JVM 和 Java 类库为了性能与安全做的一次非常成功的设计取舍。

**高分回答**
不可变性的关键不是某一个 `final` 字段，而是类不暴露任何能改变字符序列的路径：状态在构造完成后固定，操作返回新字符串。由此常量池才能安全共享，hash 可以缓存，字符串作为 Map key 不会因内容变化失联，多线程共享也不需要为内容读取加锁。

“为了安全”需要说具体：类加载名、路径、URL、权限键等字符串若能在校验后被原地修改，会出现检查时与使用时不一致。但不可变不代表敏感数据自动安全，密码放进 String 仍无法主动擦除内存，长期敏感内容更适合可控生命周期的字符数组。

**追问点答案**  
- `String` 常量池是什么？  
常量池是 JVM 对字符串字面量和部分计算结果做的共享缓存，相同内容可以复用同一对象，减少内存占用。  
- 为什么 `String` 线程安全？  
因为对象创建后状态不再变化，多线程只读不写，自然不需要同步。  
- `new String("abc")` 会创建几个对象？  
不能脱离执行上下文固定回答。“abc”字面量由类加载过程确保池中有对应对象，执行 `new String("abc")` 会再创建一个 String；这次观察到的新对象数取决于字面量是否已加载，不应把“永远两个”当规范结论。

**易错点**  
- 把不可变理解成“变量不能重新赋值”  
- 只会答线程安全，答不出常量池和哈希缓存  
- 不知道对象不可变和引用可变的区别

## 4. `StringBuilder` 和 `StringBuffer` 怎么选？

**核心结论**  
单线程高频拼接优先 `StringBuilder`，多线程共享同一缓冲区且必须同步时才考虑 `StringBuffer`。

**标准答案**  
这两个类都用于可变字符串操作，解决 `String` 多次拼接产生大量临时对象的问题。  
`StringBuffer` 的大多数方法带同步，线程安全但有锁开销；`StringBuilder` 不加锁，单线程性能更好。  
如果是方法内部局部变量、日志拼接、SQL 拼接、普通业务字符串处理，几乎都优先 `StringBuilder`。只有多个线程真的共享同一个缓冲对象并同时修改时，才有必要考虑 `StringBuffer`。  
另外可以补一句：源码层面很多编译器优化会把简单的 `+` 拼接在编译期或运行期转成 `StringBuilder`。  
所以这个题别只背“线程安全 vs 不安全”，要结合真实场景回答。

**高分回答**
选型首先看所有权，而不是看到“多线程项目”就选 `StringBuffer`。方法内部创建并完成拼接的 builder 属于线程封闭，即使应用整体并发也应使用 `StringBuilder`；如果多个线程共同构造一个字符串，给每个线程局部 builder 后归并，通常比共享 `StringBuffer` 更清晰、竞争更小。

性能还取决于拼接形态。编译期常量会直接折叠；现代 JDK 的普通 `+` 表达式可能通过 `invokedynamic` 优化；循环累加仍应复用 builder，并在可估算时预设容量，避免反复扩容。日志场景优先参数化日志，避免日志级别关闭时仍提前完成字符串构造。

**追问点答案**  
- `+` 拼接一定比 `StringBuilder` 慢吗？  
不一定，简单场景编译器会优化；但循环里大量拼接通常还是显式 `StringBuilder` 更稳。  
- 编译器会不会优化字符串拼接？  
会，但实现随 JDK 版本演进，不能把“必定翻译成 StringBuilder”当规范。常量表达式可在编译期折叠，普通表达式由编译器与运行时选择拼接策略，循环中的可变累加仍需关注临时对象。
- 为什么说大多数时候不需要 `StringBuffer`？  
因为大部分字符串拼接都发生在单线程局部作用域里，不存在共享修改问题。

**易错点**  
- 机械回答线程安全差异，不讲使用场景  
- 认为所有 `+` 拼接都很差  
- 不了解编译器优化

## 5. `final` 在变量/方法/类上的含义？

**核心结论**  
`final` 表示不能再被改变，但在变量、方法、类上含义不同。

**标准答案**  
`final` 用在变量上表示只能赋值一次；用在方法上表示不能被子类重写；用在类上表示不能被继承。  
这个关键字本质是在约束变化，帮助编译器、框架和开发者更明确地理解代码边界。  
对于局部变量和参数，如果不希望被重新赋值，可以用 `final` 表达设计意图；对关键方法或基础类，如果希望行为稳定不被篡改，也可以用 `final`。  
要注意一个非常高频的面试点：`final` 引用不能改指向，但引用对象内部状态未必不能改，所以 `final` 不等于不可变对象。  
如果你能补充“并发语义上 final 字段初始化安全发布”这一点，面试表现会更好。

**高分回答**
`final` 约束的是一次赋值、覆盖或继承边界，不自动带来深度不可变。`final List` 不能重新指向另一列表，但仍可修改列表内容；真正不可变对象还需状态私有、构造后不变、不泄漏可变引用，并对输入输出做防御性复制。

Java 内存模型为构造期间正确写入的 final 字段提供特殊初始化安全性：对象在构造完成后被其他线程获得时，final 字段具有更强的可见保证。但前提是构造函数中没有让 `this` 逸出；该保证也不会让 final 字段指向的可变对象后续修改自动线程安全，普通字段仍需要正确发布和同步。

**追问点答案**  
- `final` 和不可变对象是什么关系？  
`final` 只是限制引用或定义层面的再变化，不自动保证对象内部状态不可变；不可变对象需要类设计本身配合。  
- `final` 方法为什么有助于设计稳定性？  
因为它阻止子类重写，避免基础行为在继承体系里被意外改掉。  
- `final` 变量在并发场景下有什么额外意义？  
正确构造且构造期间没有 `this` 逸出时，其他线程看到该对象后能获得 final 字段的初始化保证；它不是通用内存屏障，也不覆盖引用对象之后发生的可变写入。

**易错点**  
- 把 `final` 和不可变对象混为一谈  
- 不会解释引用不变、对象可变  
- 忽略并发语义

## 6. `try-catch-finally` 执行顺序是什么？

**核心结论**  
先执行 `try`，异常匹配时进 `catch`，最后一般都会执行 `finally`。

**标准答案**  
`try` 负责包住可能出错的代码，`catch` 用来处理异常，`finally` 用来做清理动作。  
Java 需要把“正常逻辑”“异常处理”“资源释放”分层表达，这样代码结构更清晰。  
先在 `try` 中写主逻辑，异常交给最合适的 `catch`，资源关闭或清理动作放在 `finally`。  
执行时如果 `try` 没异常，就跳过 `catch` 直接执行 `finally`；如果 `try` 抛出并被匹配到，就执行对应 `catch`，再执行 `finally`。  
面试里如果补一句“返回前仍会执行 finally，除非进程直接退出等极端情况”，就比较完整。

**高分回答**
控制流要说到“返回表达式先求值，方法真正返回前执行 finally”。如果 `try` 中 `return x`，返回值会先被保存；`finally` 重新给局部变量 x 赋值通常不改变已保存的值，但若 x 是可变对象，finally 修改对象内容，调用方可能观察到变化。finally 自己的 return 或异常会覆盖先前返回与异常，因此应只做不会掩盖根因的清理。

资源管理优先 try-with-resources：资源按声明的逆序关闭；主逻辑和 close 同时失败时，主异常保留，关闭异常进入 `getSuppressed()`。手写 finally 若关闭资源又抛异常，很容易把真正的业务失败覆盖掉。

**追问点答案**  
- `finally` 一定会执行吗？  
大多数情况下会，但像 `System.exit()`、JVM 崩溃、线程被强杀等极端情况不保证。  
- `finally` 里写 `return` 有什么风险？  
会覆盖前面 `try` 或 `catch` 的返回值，也会吞掉异常语义，所以非常不推荐。  
- `try-with-resources` 为什么更推荐？  
它保证 `AutoCloseable` 资源逆序关闭，并用 suppressed exceptions 保留“主异常 + 关闭异常”的证据链；仍需注意 `close()` 的幂等性和声明顺序。

**易错点**  
- 说成 `return` 后不执行 `finally`  
- 不知道 `finally` 可能覆盖返回值  
- 不会联系资源管理

## 7. `throw` 和 `throws` 区别？

**核心结论**  
`throw` 是动作，`throws` 是声明。

**标准答案**  
`throw` 用在方法体内部，抛出一个具体异常对象；`throws` 写在方法签名上，声明当前方法可能抛出的异常类型。  
Java 需要把“方法会不会出错”暴露给调用方，同时也要允许当前代码主动抛出异常。  
如果当前层知道怎么恢复或兜底，就 `catch` 处理；如果当前层无法合理恢复，就用 `throw` 抛出异常，并在必要时通过 `throws` 告诉调用者。  
受检异常通常需要显式声明或捕获，运行时异常不强制，但并不代表可以随便忽略。  
回答这题时，最好再补一句“异常处理要结合职责分层”，会更像实际工程回答。

**高分回答**
`throw` 决定本次控制流抛出哪个异常对象，`throws` 是方法契约的一部分，约束调用方如何处理受检异常。工程重点是保留失败语义：包装底层异常时应把原异常作为 cause，补充稳定的业务上下文，但不要重复记录后再层层抛出造成日志噪声。

捕获的标准不是“能不能 catch”，而是当前层能否采取有效动作：恢复、补偿、转换边界语义或补充上下文。重试也不等于处理，只有错误可重试、操作幂等且受统一超时和次数预算约束时才合理；参数非法、权限失败等确定性异常重试只会放大流量。

**追问点答案**  
- 受检异常和非受检异常区别？  
受检异常编译期强制处理，非受检异常通常是运行期错误，不强制声明。  
- 为什么很多项目更少用受检异常？  
因为它会让调用链签名膨胀，很多业务场景里收益不如成本高。  
- 什么时候该捕获，什么时候该继续抛？  
当前层能恢复、补偿、转换协议或补充关键上下文时捕获；否则保持 cause 向上传播。禁止空 catch，也不要为了打印日志而捕获后原样抛出，避免同一失败被多层重复记录。

**易错点**  
- 把动作和声明说反  
- 说不清受检/非受检异常  
- 不会结合分层职责回答

## 8. `ArrayList` 和 `LinkedList` 怎么选？

**核心结论**  
大多数业务优先 `ArrayList`，只有在明确的链表操作场景才考虑 `LinkedList`。

**标准答案**  
`ArrayList` 底层是动态数组，`LinkedList` 底层是双向链表。  
数组支持高效随机访问、缓存局部性也好，所以现实中遍历和按下标访问通常更快；链表虽然插入删除理论复杂度漂亮，但前提是你已经定位到了节点，否则遍历成本很高。  
读多写少、频繁遍历、按索引访问，选 `ArrayList`；如果你是队列、双端队列或者已知节点位置频繁插删，才考虑 `LinkedList`。  
面试里最好补一句：真实系统性能不只看大 O，还要看 CPU 缓存命中、对象分配和内存占用。  
所以工程上 `ArrayList` 的使用频率远高于 `LinkedList`。

**高分回答**
复杂度必须带操作前提：`LinkedList` 在“已经拿到目标节点”时插删是 O(1)，但它的公开索引访问通常先走 O(n) 定位；通过迭代器在当前位置插删才接近该优势。`ArrayList` 尾部追加摊销 O(1)，中间插删要搬移数组，却因连续内存、低对象开销和 CPU cache 友好，在多数遍历型负载中更快。

队列场景通常选 `ArrayDeque`，它同样用数组但支持两端操作，避免每个元素一个链表节点。只有基准测试证明节点稳定定位或特殊接口需求时才考虑 LinkedList；容量可预估的 ArrayList 应预设初始容量以减少扩容复制。

**追问点答案**  
- 为什么 `ArrayList` 实际常常比 `LinkedList` 更快？  
因为数组连续存储，CPU 缓存友好，遍历和随机访问都更高效；链表节点分散，指针跳转成本高。  
- `ArrayList` 扩容机制是什么？  
容量不足时创建更大数组并复制元素。OpenJDK 常见实现按约 1.5 倍增长，但这是实现细节而非 List 接口保证；面试应先声明目标 JDK，工程上关注预估容量和扩容峰值内存。
- `LinkedList` 作为队列是不是合适？  
可以，但现代 Java 更常直接用 `ArrayDeque` 作为队列或双端队列，性能通常更好。

**易错点**  
- 只背理论复杂度  
- 一说插入删除就选链表  
- 不会讲实际性能因素

## 9. `HashMap` 底层结构是什么？

**核心结论**  
`HashMap` 底层是数组 + 链表/红黑树，通过哈希分桶管理数据。

**标准答案**  
`HashMap` 的主体是一个桶数组，元素通过 key 的 hash 值映射到某个桶中；同桶冲突时，桶内会用链表或红黑树存储多个节点。  
这种设计是为了在平均情况下把查找、插入的复杂度控制得比较低，同时又允许哈希冲突存在。  
查找时先通过 hash 定位桶，再在桶内通过 `equals` 找目标节点；插入时也类似，如果冲突多到一定程度并满足容量条件，JDK8 会树化来提升最坏情况性能。  
面试里最好补充负载因子、扩容和树化阈值，这样答案会更完整。  
一句总结：`HashMap` 的性能核心不只是结构，还和 key 设计、容量初始化、冲突分布直接相关。

**高分回答**
JDK 8 HashMap 先扰动 key 的 hash，再利用容量为 2 的幂这一条件通过位运算定位桶；桶内先比 hash，再按身份或 `equals` 判断 key。put 使元素数超过 `capacity × loadFactor` 时扩容，迁移后旧桶节点只可能留在原索引或移动到“原索引 + 旧容量”，不必完整重新取模。

树化不是到 8 个节点就必然发生：桶节点达到阈值时，如果表容量还小于 64，优先扩容分散冲突；容量足够才树化。即便平均 O(1)，糟糕或恶意 hash、频繁扩容、可变 key 都能让性能或正确性退化，所以容量和 key 契约比背结构名更重要。

**追问点答案**  
- 为什么要树化？  
因为当链表太长时，查找会退化成线性复杂度，树化可以把最坏情况优化到对数级。  
- 负载因子默认为什么是 `0.75`？  
这是时间和空间之间的经验折中，过小浪费内存，过大冲突增多。  
- `HashMap` 为什么线程不安全？  
因为复合写入、size/threshold 更新和扩容迁移没有并发协议，会丢更新或产生不可预测视图。不能把风险只归结为 JDK 7 的“环”；即使某版本避开该现象，也不提供线程安全语义。

**易错点**  
- 只说数组 + 链表，忘了红黑树  
- 不会描述查找流程  
- 不知道负载因子和扩容的关系

## 10. `HashMap` 在 JDK7 和 JDK8 的差异？

**核心结论**  
JDK8 引入红黑树，并优化了扩容迁移逻辑，重点提升高冲突情况下的性能和结构稳定性。

**标准答案**  
JDK7 的冲突桶主要是链表；JDK8 在链表过长时会转成红黑树。  
链表在冲突严重时查找性能会明显下降，而红黑树能降低最坏情况复杂度。  
回答时先讲树化，再讲扩容迁移。JDK8 迁移元素时不需要重新完整计算所有位置，而是根据新增的一位判断落在原索引还是原索引加旧容量的位置。  
面试里还常提到 JDK7 在并发扩容时可能出现链表成环，JDK8 规避了这个典型问题，但这不代表 `HashMap` 就线程安全了。  
一句总结：JDK8 的重点是“结构优化”和“迁移优化”，不是“并发安全升级”。

**高分回答**
版本比较要区分三个点：JDK 7 的桶冲突结构主要是链表，迁移常用头插；JDK 8 增加红黑树以限制严重冲突下的查找退化，并利用扩容新增的高位把旧桶拆成 low/high 两条链，保持各链相对顺序。树化还受桶长度和最小表容量共同约束。

“JDK 8 解决并发扩容死循环”只能作为历史实现差异，不能推导出可并发使用。HashMap 从未承诺并发读写的 happens-before、原子复合操作或一致迭代；多线程共享写应使用 ConcurrentHashMap 或外部同步，而不是依赖某版源码恰好不成环。

**追问点答案**  
- 为什么 JDK7 可能出现成环？  
因为并发迁移链表节点时，头插法和指针重连在竞争下可能形成环。  
- JDK8 的树化阈值是多少？  
常见实现的 `TREEIFY_THRESHOLD` 为 8；达到条件时还会检查表容量，低于 `MIN_TREEIFY_CAPACITY` 64 通常先扩容。阈值属于具体实现，回答时应带版本边界。
- JDK8 就能在并发场景下安全使用吗？  
不能，JDK8 只是减少了一些典型结构问题，但依旧不是并发容器。

**易错点**  
- 把“规避成环”说成“线程安全”  
- 只答红黑树，不答迁移逻辑  
- 不会区分结构优化和并发语义

## 11. `ConcurrentHashMap` 为什么线程安全？

**核心结论**  
它通过更细粒度的并发控制实现线程安全，JDK8 主要是 CAS + 桶级同步，而不是整表大锁。

**标准答案**  
`ConcurrentHashMap` 是专门面向多线程场景设计的哈希容器。  
如果像 `HashMap` 那样完全不加保护，多线程读写会破坏结构；如果整张表都加一把大锁，并发性能又太差，所以它采用更细粒度的控制方式。  
JDK7 用 segment 分段锁，JDK8 取消 segment，改成数组桶级别的控制。常见无竞争路径通过 CAS 做原子更新，冲突更新时再结合 `synchronized` 保护桶节点。  
这样设计的目标是让不同桶可以尽量并发执行，提高吞吐量。  
面试里最好强调：它不是“完全无锁”，而是“锁更细、无锁优化更多”。

**高分回答**
线程安全包含结构安全、可见性和操作语义三层。JDK 8 的读路径依靠 volatile/CAS 建立可见性，空桶可 CAS 插入，桶冲突时对桶首节点同步，扩容时多个线程还能协助迁移。不同桶通常可以并发更新，但热点 key 仍会形成单桶竞争，因此它不是无限扩展的无锁容器。

单个方法线程安全不代表复合业务原子性。`get` 后 `put`、遍历后删除仍可能竞态，应使用 `putIfAbsent`、`compute`、`merge` 等原子复合 API，并遵守回调短小、避免递归更新和外部阻塞的约束。它的遍历是弱一致性的，不能拿来做要求同一时刻精确快照的结算。

**追问点答案**  
- JDK7 和 JDK8 的 `ConcurrentHashMap` 差异？  
JDK7 核心是 segment 分段锁；JDK8 核心是桶级控制、CAS 和局部同步。  
- 为什么它不允许 `null` key/value？  
因为并发环境下 `null` 会让“没找到”和“值就是 null”语义混淆，增加实现复杂度。  
- `size()` 在高并发下是不是绝对精确？  
调用会返回某个统计结果，但并发修改持续发生时，“返回瞬间的绝对快照”本身没有稳定业务意义，也不应先 size 再据此执行复合决策；需要强一致配额时应使用独立原子协议。

**易错点**  
- 误答成“全部方法都加锁”  
- 不知道 segment 历史  
- 讲不清并发性能为什么更高

## 12. `HashSet` 和 `TreeSet` 区别？

**核心结论**  
`HashSet` 关注去重和快速查找，`TreeSet` 关注有序去重和范围能力。

**标准答案**  
`HashSet` 通常基于 `HashMap` 实现，元素无序；`TreeSet` 基于红黑树实现，元素有序。  
两者分别服务不同目标，前者追求哈希性能，后者追求顺序和区间能力。  
如果只需要去重且不关心顺序，优先 `HashSet`；如果需要自然排序、比较器排序、最值、范围查询，则用 `TreeSet`。  
面试里可以再补充 `LinkedHashSet`，表示既要去重又想保留插入顺序时的折中方案。  
回答时一定要把“底层结构”一起带上，这样更完整。

**高分回答**
二者的“唯一”判定依据不同：HashSet 使用 `hashCode + equals`，TreeSet 使用自然顺序或 Comparator 的比较结果，比较为 0 就视为同一元素。因此 TreeSet 的比较规则若与 equals 不一致，会出现 `contains`、去重结果与其他集合不同，这不是树结构错误，而是排序契约决定的语义。

TreeSet 除了有序遍历，还提供 `floor/ceiling`、区间视图等 NavigableSet 能力，代价是 O(log n) 更新和比较器成本。只需去重选 HashSet；要插入顺序选 LinkedHashSet；要排序和范围查询才选 TreeSet，不要为了最终展示排序让所有写入持续付树维护成本。

**追问点答案**  
- `TreeSet` 为什么有序？  
因为它底层是红黑树，插入时会按比较规则维护有序结构。  
- 比较器和 `equals` 不一致会怎样？  
TreeSet 以 `compare(a,b)==0` 判断集合意义上的重复，即使 `a.equals(b)` 为 false 也只保留一个；比较器还必须满足反对称、传递和稳定性，否则树的查找行为会不可预测。
- `LinkedHashSet` 又是干什么的？  
它在哈希结构基础上维护插入顺序，适合既要去重又要稳定遍历顺序的场景。

**易错点**  
- 只会答有序无序  
- 不知道底层实现  
- 忽略比较器语义问题

## 13. `Collections.synchronizedMap` 和 `ConcurrentHashMap` 差异？

**核心结论**  
前者是同步包装，后者是专门的并发容器设计，二者目标和并发性能差异都很大。

**标准答案**  
`Collections.synchronizedMap` 是对普通 `Map` 的方法做同步包装；`ConcurrentHashMap` 是从底层结构到并发控制都专门为多线程场景设计的容器。  
同步包装实现简单，但很多操作会串行化；而并发容器希望在保证安全的同时提升多线程吞吐。  
如果只是简单共享、并发度不高、改动成本小，同步包装可能够用；如果是高并发读写、服务端共享缓存、热点访问场景，优先 `ConcurrentHashMap`。  
另一个关键点是迭代行为：前者遍历时通常还需要外部同步，后者提供的是更适合并发访问的弱一致性迭代。  
面试里可以总结成一句：一个是“给老 map 套锁”，一个是“从设计上支持并发”。

**高分回答**
`synchronizedMap` 用同一个 mutex 串行保护包装器的方法，适合低并发且需要兼容可为 null 的普通 Map 语义；但迭代、`keySet` 视图和多步操作必须按文档在同一 mutex 上外部同步，否则“方法安全”无法覆盖完整过程。泄漏底层 Map 引用也会绕过所有保护。

ConcurrentHashMap 通过细粒度协调提高并发吞吐，禁止 null，并提供 `compute/merge` 等原子复合操作和弱一致迭代。选择不只看速度：若业务需要遍历期间阻止所有修改，同步包装器外部锁可以表达这种强边界；若允许并发更新且无需瞬时快照，ConcurrentHashMap 更合适。

**追问点答案**  
- 为什么遍历 `synchronizedMap` 时还要手动加锁？  
因为单个方法同步不等于整个迭代过程同步，遍历期间别的线程仍可能修改集合。  
- 什么是弱一致性迭代？  
迭代器反映创建时或之后的部分状态，不保证包含全部并发更新，也不会因并发修改而 fail-fast；它适合监控和扫描，不适合要求原子快照的对账逻辑。
- 什么时候同步包装就够用了？  
并发不高、代码量小、场景简单时可以接受，但不适合高并发核心链路。

**易错点**  
- 把两者都说成“线程安全 map”就结束  
- 不会讲迭代差异  
- 不理解同步包装和并发设计的区别

## 14. 泛型擦除是什么？

**核心结论**  
Java 泛型主要发生在编译期，运行时大部分类型参数信息会被擦除。

**标准答案**  
泛型擦除指的是编译器在生成字节码时，会把类型参数替换成原始类型或上界类型，并插入必要的类型检查和强转。  
Java 当初这样设计主要是为了兼容历史字节码和旧版本类库，不破坏语言演进。  
开发者层面不需要手动做擦除，但要理解它带来的限制，比如不能直接 `new T()`、不能直接创建泛型数组、运行时很难通过普通方式区分 `List<String>` 和 `List<Integer>`。  
它的好处是编译期更安全，缺点是运行时泛型能力受限。  
如果面试追问到反射，可以补一句：某些声明层面的泛型信息还能通过反射元数据部分读取。

**高分回答**
擦除不是“泛型完全消失”。编译器把类型变量替换为上界、在使用点插入 cast，并在覆盖擦除后签名不同的泛型方法时生成 bridge method 维持多态。类文件还可在 Signature 等元数据中保存声明泛型，反射能读到 `Field.getGenericType()`，但普通 `new ArrayList<String>()` 对象通常不携带可查询的 String 实参。

限制都能从“运行时没有可具体化的 T”推导：不能 `new T()`、不能 `T.class`、不能用参数化类型做 `instanceof`，也不能安全创建 `new T[]`。需要运行时类型时应显式传入 `Class<T>`、构造函数或保留类型令牌，而不是靠未经检查强转。

**追问点答案**  
- 为什么不能 `new T()`？  
因为运行时并不知道 `T` 的具体实际类型，没法直接安全构造。  
- 为什么不能直接创建泛型数组？  
数组是协变且运行时检查元素类型，参数化泛型则被擦除；若允许 `new List<String>[n]`，可经数组协变写入 `List<Integer>`，读取时才延迟失败，编译器因此禁止这种创建。
- 反射里还能不能看到泛型信息？  
有些声明层信息能看到，比如字段、方法签名上的泛型参数，但普通对象实例运行时通常看不到完整实参。

**易错点**  
- 以为泛型运行时什么都没有  
- 不理解编译期和运行期边界  
- 不会把各种限制和擦除对应起来

## 15. `<? extends T>` 和 `<? super T>` 区别？

**核心结论**  
`extends` 适合读，`super` 适合写，口诀是 PECS。

**标准答案**  
`<? extends T>` 表示某个未知类型，它是 `T` 的子类型；`<? super T>` 表示某个未知类型，它是 `T` 的父类型。  
因为 Java 泛型默认是不变的，`List<Dog>` 不是 `List<Animal>`，通配符是为了解决“只读”或“只写”这种更细的类型关系需求。  
如果集合主要负责“提供数据”，也就是 producer，就用 `extends`；如果集合主要负责“接收数据”，也就是 consumer，就用 `super`。  
这就是 PECS：Producer Extends，Consumer Super。  
回答时最好加一句：`extends` 里通常不能安全写入具体对象，`super` 里读出来往往只能按 `Object` 看待。

**高分回答**
PECS 描述的是某个参数在这次 API 中的角色，不是集合永久只读或只写。`List<? extends Number>` 的真实元素类型可能是 Integer，因此不能安全 add 任意 Number，但可以读取为 Number；`List<? super Integer>` 的真实类型可能是 Integer、Number 或 Object，可以安全写 Integer，读取只能保证 Object。

API 设计时若类型既输入又输出，优先用具名类型参数建立关系，例如 `<T> T transform(T value)`，不要强套通配符。通配符适合扩大调用兼容性，类型参数适合表达多个参数和返回值之间必须相同或受约束的关系。

**追问点答案**  
- 为什么 `List<Dog>` 不能赋值给 `List<Animal>`？  
因为如果允许这样做，就可能往 `Dog` 列表里塞入其他 `Animal`，破坏类型安全。  
- `Collections.copy` 为什么一个参数用 `extends`，一个用 `super`？  
源集合负责生产元素，所以用 `extends`；目标集合负责接收元素，所以用 `super`。  
- 有没有同时可读可写的通配符写法？  
没有能对未知具体类型自由读写的通配符。若 API 需要同一种类型双向流动，应声明 `<T>` 并让参数共享 T；若集合仅生产或消费，再分别使用 extends 或 super。

**易错点**  
- 只会背 PECS，不会解释  
- 读写边界说反  
- 说不清为什么 Java 要这么设计

## 16. 反射的优缺点是什么？

**核心结论**  
反射的优点是灵活和解耦，缺点是性能、类型安全、可维护性和封装性变差。

**标准答案**  
反射允许程序在运行时读取类信息、构造对象、调用方法、访问字段。  
框架层经常不知道具体业务类是什么，但又要通用地装配对象、做依赖注入、序列化、ORM 映射，所以反射非常重要。  
在框架、中间件、底层基础能力里使用反射很合理，但在普通业务代码中应谨慎，避免让代码变得难读、难测、难维护。  
它的主要问题是性能不如直接调用、错误从编译期推迟到运行期，而且可能破坏封装。  
更成熟的框架通常会在反射基础上加缓存、方法句柄或字节码增强，减少性能损耗。

**高分回答**
反射的核心价值是把“编译期写死的类型关系”推迟到运行时发现，适合依赖注入、ORM、测试和插件框架。代价不应只说慢：它削弱静态类型检查和 IDE 重构能力，模块系统下还受可访问性约束，Native Image/AOT 场景需要提前声明反射元数据，错误会更晚暴露。

性能要按调用频率判断。一次启动扫描通常不是瓶颈，高频逐字段反射才需要缓存元数据、MethodHandle、代码生成或批量绑定；未经基准测试就全面字节码增强会增加类加载、调试和兼容成本。反射入口还应白名单化，不能让外部输入任意指定类名或可调用方法。

**追问点答案**  
- Spring 为什么离不开反射？  
因为它要在运行时扫描类、创建 bean、注入依赖、执行 AOP 代理，这些都依赖动态能力。  
- 反射性能差在哪里？  
查找、访问检查、参数装箱与动态调用可能增加成本且限制部分优化，但 JVM 会优化热点路径。应区分元数据查找和已缓存成员调用，并用目标 JDK 的基准测试决定是否优化。
- 什么时候更适合字节码增强而不是纯反射？  
当框架高频调用、性能敏感、代理逻辑复杂时，字节码增强通常更高效。

**易错点**  
- 一味说反射不好  
- 不会讲真实框架场景  
- 不区分框架层和业务层使用边界

## 17. 注解的生命周期有哪些？

**核心结论**  
常见生命周期有 `SOURCE`、`CLASS`、`RUNTIME`，决定注解能保留到哪个阶段。

**标准答案**  
Java 通过 `RetentionPolicy` 定义注解在源码、字节码、运行时哪个阶段仍然保留。  
不同注解服务的对象不同，有的只给编译器看，有的给字节码工具看，有的要给运行时框架读取。  
如果只是编译期检查或代码生成提示，可用 `SOURCE`；如果只需要进字节码，选 `CLASS`；如果要让 Spring 这类框架在运行时读取，就必须用 `RUNTIME`。  
面试里可以顺手区分一下 `@Retention` 和 `@Target`，前者管生命周期，后者管能标在哪。  
这个题最核心的原理点就是：运行时框架识别注解，本质上依赖 `RUNTIME` 保留策略。

**高分回答**
Retention 决定信息最晚保留到哪里：SOURCE 只供编译阶段，CLASS 写入 class 文件但默认不供运行时反射读取，RUNTIME 可由反射读取。它不决定注解是否“自动生效”；注解只是元数据，必须有编译器、注解处理器、字节码工具或运行时框架解释它。

设计自定义注解还要明确 `@Target`、`@Inherited`、`@Repeatable` 和组合注解语义。`@Inherited` 只影响类级注解通过 Class 查询的继承，不会让方法注解自动继承；框架是否搜索接口、父方法和元注解仍由框架自己的查找规则决定。

**追问点答案**  
- `@Target` 是做什么的？  
它定义注解可以标注在哪些程序元素上，比如类、方法、字段、参数。  
- 自定义注解让 Spring 识别需要什么条件？  
若依赖运行时反射通常要 RUNTIME，同时必须注册扫描、切面、BeanPostProcessor 等处理逻辑；仅定义注解不会产生行为，Spring 的组合注解规则也不能直接等同于 JDK 原生反射规则。
- 为什么很多框架依赖 `RUNTIME` 注解？  
因为它们需要在程序运行时通过反射读取注解信息并执行相应行为。

**易错点**  
- 把 `Target` 和 `Retention` 混淆  
- 不知道运行时读取必须是 `RUNTIME`  
- 只会背名字，不会讲用途

## 18. 序列化为什么要 `serialVersionUID`？

**核心结论**  
它是类序列化兼容性的版本标识，用来控制反序列化时类结构是否匹配。

**标准答案**  
`serialVersionUID` 是 Java 原生序列化机制里类的版本号。  
对象被序列化到文件、网络或缓存后，后续反序列化时 JVM 需要确认当前类定义和当时写出去的版本是否兼容，否则可能恢复出错误对象。  
对于实现了 `Serializable` 的类，通常建议显式声明 `serialVersionUID`，这样类结构有轻微变化时，兼容策略可控，而不是依赖编译器自动生成。  
如果版本号不一致，常见结果是抛出 `InvalidClassException`。  
现代项目虽然不一定大量使用 Java 原生序列化，但这个知识点依然体现你是否理解兼容性问题。

**高分回答**
`serialVersionUID` 是接收端判断“流中的类版本是否允许交给当前类恢复”的门槛，不是完整 schema 兼容机制。显式固定 UID 可以避免编译器根据类细节计算出的默认值因无关改动而变化，但 UID 相同也不保证反序列化一定语义正确；字段类型、继承层次、自定义读写逻辑等不兼容变化仍会失败或产生错误状态。

Java 原生反序列化还具有安全边界：它能实例化对象图并触发特定回调，处理不可信数据可能形成 gadget 攻击。跨服务协议优先使用带显式 schema、白名单和版本演进规则的格式；若遗留系统必须使用，应启用对象输入过滤、限制类型与图规模，并避免长期存储无法治理的字节流。

**追问点答案**  
- 不写会怎样？  
运行时会根据类结构计算默认 UID（不是编译器简单写一个固定值），源码或编译细节变化可能使结果改变。显式声明能稳定版本选择，但仍需测试实际字段演进兼容性。
- 哪些字段变化会影响兼容性？  
字段增删改、继承结构变化、类定义调整都可能影响序列化兼容。  
- 为什么很多分布式系统不用 Java 原生序列化？  
因为它性能、可读性、跨语言能力、版本治理都不如 JSON、Protobuf、Avro 这类方案。

**易错点**  
- 以为这是性能优化参数  
- 只知道要写，不知道它控制什么  
- 不会把它和版本兼容联系起来

## 19. `transient` 的作用？

**核心结论**  
`transient` 表示该字段不参与 Java 原生序列化。

**标准答案**  
被 `transient` 修饰的字段，在对象做 Java 原生序列化时不会被写出去。  
有些字段是敏感信息、临时状态、缓存值，或者可以在恢复后重新计算，没有必要或不应该被持久化。  
像密码、token、运行时缓存、临时上下文之类的字段，可以评估是否用 `transient`。  
但要注意，它只直接影响 Java 原生序列化，不一定影响 Jackson、Gson 等 JSON 框架，因为这些框架各自有自己的规则。  
所以回答时最好补一句：这是“Java 序列化语义”的关键字，不是“所有序列化通杀”的关键字。

**高分回答**
`transient` 表达“该实例字段不进入默认 Java 序列化字段流”，反序列化后它通常是类型默认值，除非 `readObject`、`readResolve` 或后续初始化显式恢复。适合派生缓存和进程本地句柄，但不能把它当安全擦除：敏感值仍存在于运行中对象内，也可能被日志、JSON 框架或自定义 `writeObject` 输出。

若 transient 字段维持类不变量，恢复策略必须与构造逻辑同等严格。例如反序列化绕过普通构造流程，仅靠“下次用到再初始化”可能在并发下出现竞态，应在恢复钩子中校验数据并建立完整状态，或使用序列化代理模式缩小可恢复表面。

**追问点答案**  
- `static` 字段会被序列化吗？  
通常不会，因为序列化的是对象状态，而 `static` 属于类，不属于对象实例。  
- `transient` 对 JSON 序列化一定有效吗？  
不一定，要看具体 JSON 框架是否遵循它，很多框架有自己的注解和配置。  
- 如果字段必须反序列化后恢复，怎么处理？  
可在 `readObject` 中先调用 `defaultReadObject`，再校验并重建派生状态；复杂类型更推荐序列化代理。自定义钩子同样属于反序列化攻击面，不能信任输入字段。

**易错点**  
- 误以为它影响所有框架  
- 不知道它只针对 Java 原生序列化最直接  
- 把它和 `static` 的行为混在一起

## 20. Java 里深拷贝和浅拷贝怎么实现？

**核心结论**  
浅拷贝只复制对象本身和引用地址，深拷贝会把引用链上的对象也复制出来。

**标准答案**  
浅拷贝之后，新旧对象的基本字段互不影响，但内部引用对象通常仍然共享；深拷贝则会把引用对象也完整复制一份，形成真正独立的新对象图。  
有些业务场景只需要复制外层结构，浅拷贝成本低；但如果对象内部状态会被修改且彼此不能相互影响，就需要深拷贝。  
浅拷贝可以通过拷贝构造、对象复制方法、`clone` 等实现；深拷贝通常更推荐显式手写复制逻辑，或者在简单场景下用序列化再反序列化。  
面试里最好补一句：很多团队不喜欢直接依赖 `clone`，因为语义不够清晰、可控性差。  
如果对象本身设计成不可变，深拷贝需求就会大幅降低。

**高分回答**
深浅不是由“是否新建外层对象”决定，而是由对象图的共享策略决定。深拷贝也不一定要复制图中一切：不可变值、共享字典和显式 flyweight 可以安全复用；需要复制的是后续可能独立修改的可变节点。还必须定义循环引用、别名保持、父子反向引用和外部资源句柄如何处理，否则“递归复制所有字段”可能无限递归或破坏图结构。

工程上优先显式拷贝构造或 mapper，因为它能清楚选择字段、验证不变量并适配版本。Java/JSON 序列化做拷贝成本高、受可序列化范围和安全问题限制，还可能丢类型与共享引用关系；`clone` 默认仅浅拷贝且绕开构造语义。更根本的方案是不可变对象、持久化数据结构或所有权转移，减少复制需求。

**追问点答案**  
- 为什么很多项目不推荐直接用 `clone`？  
因为它语义历史包袱重、实现细节容易出错，而且对子类和深拷贝控制不够直观。  
- JSON 序列化做深拷贝有什么问题？  
除性能外，还可能丢多态类型、精度、对象别名和循环关系，并受忽略字段、构造器与框架配置影响；它适合明确 DTO 边界，不应作为通用对象图复制器。
- 不可变对象是不是就不太需要深拷贝？  
是的，因为不可变对象可以安全共享，不需要担心被修改导致相互影响。

**易错点**  
- 把“新建对象”误认为深拷贝  
- 不会举引用对象共享的例子  
- 不知道 `clone` 在工程上常被谨慎对待
