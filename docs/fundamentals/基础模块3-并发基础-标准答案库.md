# 基础模块3 - 并发基础 - 标准答案库

## 1. 进程和线程区别是什么？

**核心结论**  
进程是资源分配的基本单位，线程是 CPU 调度的基本单位；进程之间隔离更强，线程之间共享更紧密。

**标准答案**  
进程可以理解成一个程序运行起来后的独立资源容器，它有自己的地址空间、文件句柄、内存和系统资源；线程则是在进程内部真正执行代码的最小调度单元，同一个进程里的多个线程共享进程的大部分资源。这样设计的原因是操作系统既要做资源隔离，又要支持更轻量的并发执行，所以把“资源拥有者”和“执行单元”拆开了。工程上，进程更适合做强隔离，比如不同服务、不同容器；线程更适合做同一服务内部的并发处理，比如 Web 请求处理、任务执行、异步回调。面试里如果再补一句“线程切换成本通常低于进程切换，但线程之间共享资源也更容易产生并发问题”，会更完整。

**追问点答案**  
- 为什么进程切换通常更重？  
因为它涉及地址空间切换、页表切换、更多上下文保存与恢复。  
- 线程共享了什么？  
通常共享堆、方法区、文件描述符等进程级资源，但栈和程序计数器一般是线程私有。  
- 为什么微服务更常按进程隔离，而不是按线程隔离？  
因为进程级隔离更强，故障边界更清晰，资源治理也更独立。

**原理补充**  
这道题的关键不是背“谁大谁小”，而是理解操作系统为什么既需要隔离，又需要高效调度。进程解决资源隔离，线程解决执行并发。

**架构扩展**  
在线上系统里，这个区别会直接影响部署和稳定性设计。比如一个 Java 服务里线程打满，通常影响的是当前实例；但如果一个进程把内存吃光，可能整个容器都会被 OOM Killer 杀掉。再比如多租户平台如果用线程隔离不同租户逻辑，故障扩散风险会明显大于进程隔离。更合理的做法通常是：服务间优先用进程或容器隔离，服务内再用线程池做并发。

**工程指标**  
- 进程 RSS：看单个服务实例是否已经把总内存吃高。  
- 线程总数：判断服务内部并发模型是否失控。  
- 上下文切换次数：看 CPU 是否花了太多时间在线程切换而不是业务执行。  
- 容器 OOM / 重启次数：判断进程级资源问题是否已经影响可用性。  
- 单实例吞吐：看线程增加后是否真的带来收益，而不是只是增加切换成本。

**易错点**  
- 只会说“进程大线程小”  
- 不知道线程共享哪些资源  
- 不会把隔离和调度成本联系起来

## 2. 线程的生命周期有哪些状态？

**核心结论**  
Java 线程常见状态包括新建、就绪/运行、阻塞、等待、超时等待、终止，重点不是背名字，而是知道线程为什么卡在某个状态。

**标准答案**  
从 Java 角度看，线程通常会经历 `NEW`、`RUNNABLE`、`BLOCKED`、`WAITING`、`TIMED_WAITING` 和 `TERMINATED` 这些状态。`NEW` 是刚创建还没启动，`RUNNABLE` 表示可运行或正在运行，`BLOCKED` 通常是在等锁，`WAITING` 是无限等待其他线程唤醒，`TIMED_WAITING` 是带超时的等待，比如 `sleep`、带超时的 `wait` 或 `join`，`TERMINATED` 则是执行结束。这样划分的意义是让 JVM 和开发者都能更准确地描述线程当前在干什么。工程上，排查线程问题时，最关键的不是记住全部状态，而是看线程为什么大量停留在某个状态，比如都卡在 `BLOCKED` 往往说明锁竞争严重，大量 `WAITING` 可能说明线程池空转或任务调度异常。

**追问点答案**  
- `RUNNABLE` 一定在真正运行吗？  
不一定，它在 Java 语义里也可能表示已经就绪、等待 CPU 调度。  
- `BLOCKED` 和 `WAITING` 有什么核心区别？  
`BLOCKED` 通常是等进入同步块的监视器锁，`WAITING` 是线程主动进入等待状态，等别人唤醒。  
- 为什么线上排障要看线程状态分布？  
因为状态分布能快速告诉你系统更像是锁问题、IO 问题还是任务调度问题。

**原理补充**  
线程状态本质上是 JVM 对线程执行现场的抽象标签。它让我们能把“线程在等 CPU”“线程在等锁”“线程在等通知”这些不同情况区分开。

**架构扩展**  
真实系统里，线程状态分布经常是故障的第一现场。比如接口大量超时时，如果你看到线程池里的线程都在 `BLOCKED`，就要怀疑锁争用；如果都在 `WAITING`，可能是任务生产出了问题；如果大量线程在 `TIMED_WAITING`，则要看是不是 sleep、轮询或超时策略设计得不合理。成熟团队通常会把线程状态采样接入可观测系统，而不是等故障来了才手查。

**工程指标**  
- 各线程状态数量分布：快速判断系统主要卡在哪一类问题上。  
- 线程池活跃线程数：看线程是否真的在干活。  
- 线程池队列长度：辅助判断是不是任务堆积。  
- 热线程调用栈：看高 CPU 线程是在忙等还是正常计算。  
- 接口超时率：判断线程状态异常是否已经传导到业务层。

**易错点**  
- 机械背状态，不会解释原因  
- 把 `BLOCKED` 和 `WAITING` 混淆  
- 不会把线程状态和故障类型对应起来

## 3. `sleep` 和 `wait` 区别？

**核心结论**  
`sleep` 是线程类的方法，主要用于让当前线程暂停一段时间；`wait` 是对象监视器的方法，主要用于线程间协作，并且会释放锁。

**标准答案**  
`sleep` 来自 `Thread`，作用是让当前线程暂停指定时间，时间到了自动恢复可运行状态；`wait` 来自 `Object`，前提是线程已经持有该对象的监视器锁，它调用后会进入等待状态，并释放当前持有的对象锁，直到被 `notify` / `notifyAll` 唤醒或者超时。两者最根本的区别在于用途不同，`sleep` 更像线程调度层面的暂停，`wait` 是线程协作模型的一部分。工程上，如果你只是想简单延迟、退避重试，通常用 `sleep`；如果你想表达“条件不满足，等别人改变条件后再继续”，才应该用 `wait` / `notify` 这类协作机制。面试里一定要强调一句：`sleep` 不会释放已经持有的锁，`wait` 会释放对应对象锁。

**追问点答案**  
- 为什么 `wait` 必须在同步块里调用？  
因为它依赖对象监视器语义，调用时必须持有该对象的锁。  
- `sleep` 被打断会怎样？  
会抛出 `InterruptedException`，通常需要根据业务决定重试、退出还是恢复中断标记。  
- 为什么很多项目里更少直接用 `wait` / `notify`？  
因为更高层的并发工具，如 `CountDownLatch`、`BlockingQueue`、`Condition` 通常更安全、语义更清晰。

**原理补充**  
这道题的底层差异本质上是“调度暂停”和“条件协作”的差异。一个是线程层操作，一个是监视器协作模型。

**架构扩展**  
线上最常见的误用是把 `sleep` 当成同步手段，比如轮询等待下游状态更新，结果线程大量空转，占着线程池不做事；或者错误使用 `wait` / `notify`，导致唤醒丢失、线程永久等待。更合理的做法是，简单退避用 `sleep`，有条件等待用并发工具类，跨服务协调则交给消息、回调或状态机。

**工程指标**  
- 线程 `TIMED_WAITING` 数量：看是否存在大量 `sleep` 式轮询。  
- 线程 `WAITING` 数量：判断是否有条件协作卡住。  
- 线程池活跃度：看线程是不是都被等待操作占住。  
- 任务平均等待时长：辅助判断等待模型是否设计过重。  
- 重试率：很多错误的 `sleep` 方案最终会放大成重试风暴。

**易错点**  
- 只记住“一个是静态方法一个是实例方法”  
- 忘记 `wait` 会释放锁  
- 把延迟和线程协作混为一谈

## 4. `notify` 和 `notifyAll` 区别？

**核心结论**  
`notify` 只随机唤醒一个等待线程，`notifyAll` 会唤醒所有等待线程；选错会带来唤醒丢失或无效竞争。

**标准答案**  
当多个线程在同一个对象监视器上等待时，`notify` 只会唤醒其中一个等待线程，而 `notifyAll` 会把所有等待线程都唤醒，让它们重新竞争锁并检查条件。这样设计是为了兼顾效率和正确性，因为有些场景只需要唤醒一个消费者，而有些场景如果只唤醒一个线程，可能唤醒到“条件不匹配”的线程，导致真正能继续执行的线程没有被唤醒。工程上，如果等待队列里只有一种条件、语义明确，`notify` 可以减少无谓竞争；但如果多个条件混在一起，或者你不确定唤醒哪个线程才对，`notifyAll` 更安全。面试里通常建议强调：并发编程里首先要保证正确性，其次才是效率。

**追问点答案**  
- 为什么 `notify` 可能有风险？  
因为你无法精确控制唤醒的是哪个线程，可能唤醒了条件仍然不满足的线程。  
- `notifyAll` 会不会太重？  
会，它可能带来“惊群效应”，让很多线程被唤醒后又重新阻塞。  
- 为什么被唤醒后还要重新检查条件？  
因为可能是虚假唤醒，也可能是别的线程先一步修改了条件。

**原理补充**  
唤醒不是“直接继续执行”，而是“从等待队列进入锁竞争队列”。真正能继续往下走，还要重新拿到锁并重新校验条件。

**架构扩展**  
真实系统里，如果自己手写生产者消费者模型，最容易出的问题之一就是误用 `notify`，导致线程偶发卡死，线上只在高并发偶现，排查非常痛苦。也正因为如此，很多团队更倾向于直接使用 `BlockingQueue`、`Condition` 或其他更高层并发工具，减少这种底层协作错误。

**工程指标**  
- 等待线程数量：判断唤醒策略是否会影响很多线程。  
- 锁竞争时间：看 `notifyAll` 是否引发过度竞争。  
- 线程唤醒后成功执行比例：辅助判断唤醒是否精准。  
- 任务堆积长度：看错误唤醒策略是否已经影响吞吐。  
- 线程卡死告警次数：判断协作模型是否存在偶发死锁或挂起。

**易错点**  
- 以为 `notify` 更高级、一定更好  
- 忘记唤醒后还要竞争锁  
- 不知道虚假唤醒和条件重检

## 5. 什么是线程安全？

**核心结论**  
线程安全不是“用了锁就安全”，而是多个线程同时访问共享资源时，程序行为仍然正确且结果可预期。

**标准答案**  
线程安全的本质，是在并发场景下多个线程同时操作共享状态时，程序不会因为竞态条件而出现错误结果、脏数据、状态错乱或不可预期行为。之所以需要这个概念，是因为线程之间共享堆内数据，而 CPU 缓存、指令重排和执行交错都会让程序行为变复杂。工程上，判断一个组件是不是线程安全，关键不是看它有没有 `synchronized`，而是看它有没有共享状态、有没有完整的同步策略、有没有把原子性、可见性和有序性都考虑进去。比如不可变对象通常天然线程安全，局部变量一般也线程安全，但共享可变集合如果没有保护就很危险。面试里最好补一句：线程安全讲的是“结果正确”，性能和实现方式是下一层问题。

**追问点答案**  
- 无状态类为什么通常线程安全？  
因为没有共享可变状态，自然不存在并发修改问题。  
- 线程安全和高性能一定冲突吗？  
不一定，好的设计可以通过不可变、分段、无锁、限域等方式兼顾。  
- 用了 `ConcurrentHashMap` 就一定线程安全吗？  
不一定，容器线程安全不等于复合业务操作线程安全。

**原理补充**  
线程安全本质上是“共享可变状态管理问题”。没有共享、没有可变，问题就会小很多。

**架构扩展**  
很多线上并发 Bug 不是因为开发者不会加锁，而是因为组件边界设计得不好。比如一个服务把请求上下文缓存成成员变量、把可变对象跨线程共享、把多个步骤拆成非原子复合操作，最后即使局部用了线程安全容器，也救不了整体逻辑。更合理的做法是优先减少共享，再决定是否同步。

**工程指标**  
- 并发错误日志数：看线程安全问题是否已经暴露成线上异常。  
- 数据不一致事件数：比如重复扣款、库存错乱，这类是业务层表现。  
- 锁竞争时间：看你为保证线程安全付出了多少代价。  
- 重试率 / 幂等冲突率：辅助判断并发写入是否失控。  
- 线程安全测试覆盖率：确保关键组件不是靠“感觉安全”。

**易错点**  
- 认为线程安全等于加锁  
- 不会区分容器安全和业务逻辑安全  
- 忽略无状态和不可变的价值

## 6. 原子性、可见性、有序性分别是什么？

**核心结论**  
这是并发正确性的三个核心维度：原子性保证操作不可被中途打断，可见性保证线程间修改能被及时看到，有序性保证执行顺序符合约束。

**标准答案**  
原子性指的是一个操作或一组操作执行时不会被中途打断，别人看不到“做了一半”的状态；可见性指的是一个线程修改了共享变量之后，其他线程能够及时看到这个新值；有序性指的是程序执行结果不会因为编译器优化、CPU 重排而偏离正确约束。之所以要拆成这三个维度，是因为并发问题并不都来自同一种原因，有的是丢更新，有的是读到旧值，有的是顺序错乱。工程上，`synchronized` 可以同时在一定程度上解决这三件事，`volatile` 更偏向可见性和有序性，CAS 更偏向原子更新。面试时如果能用一个具体例子，比如双重检查单例为什么需要 `volatile`，会很有说服力。

**追问点答案**  
- `i++` 为什么不是原子操作？  
因为它底层包含读、改、写三个步骤。  
- 可见性问题为什么会发生？  
因为线程可能读的是自己工作内存或 CPU 缓存里的旧值。  
- 有序性问题为什么通常不容易直观看到？  
因为它依赖重排和并发交错，很多时候只在极端时序下暴露。

**原理补充**  
这三个概念本质上是并发内存模型对“状态更新”提出的三类约束，没有它们，程序员写的顺序不等于机器执行和其他线程观察到的顺序。

**架构扩展**  
真实系统里，很多“偶现 Bug”都能归到这三类之一。比如配置热更新后部分线程看不到新值，是可见性；库存扣减丢失，是原子性；双重检查单例拿到半初始化对象，是有序性。能把问题归类，排查就会快很多。

**工程指标**  
- 并发写冲突率：看原子性问题是否频繁发生。  
- 配置生效延迟：辅助判断可见性问题。  
- 偶发脏数据事件数：很多有序性问题最终会落到这里。  
- 压测下异常复现率：并发问题通常需要靠高压场景触发。  
- 线程间状态不一致告警：用于发现共享变量可见性缺陷。

**易错点**  
- 背概念但不会举例  
- 以为 `volatile` 能解决所有三类问题  
- 说不清三者在实际问题中的区别

## 7. `volatile` 能保证什么？

**核心结论**  
`volatile` 主要保证可见性和一定程度上的有序性，但不能保证复合操作的原子性。

**标准答案**  
当一个变量被 `volatile` 修饰后，一个线程对它的写操作会及时刷新到主内存，其他线程读这个变量时也会直接从主内存获取最新值，从而保证可见性。同时，`volatile` 还会通过内存屏障约束指令重排，所以在一些场景下也能保证必要的有序性，比如双重检查单例。但它不能保证复合操作的原子性，像 `count++` 这类读改写操作即使变量是 `volatile`，多个线程同时执行仍然会丢更新。工程上，`volatile` 适合状态开关、配置刷新、停止标志这类“一写多读”的轻量场景，不适合计数器、库存扣减这类竞争更新场景。

**追问点答案**  
- 为什么 `volatile` 不能保证 `i++` 正确？  
因为 `i++` 不是单次写，而是读、改、写三个步骤。  
- 双重检查单例为什么要加 `volatile`？  
为了防止对象引用先发布、初始化后完成这种重排问题。  
- `volatile` 会不会影响性能？  
会有一定内存屏障开销，但通常比加锁轻很多。

**原理补充**  
`volatile` 的底层关键点是内存屏障。它不让线程随意缓存变量旧值，也约束某些重排，从而让共享状态更新更可预测。

**架构扩展**  
线上最常见的正确用法是控制类状态，比如服务降级开关、停止消费标记、配置版本号；最常见的误用则是拿它做高并发计数，最后数据怎么都对不上。更合理的做法是：状态传播用 `volatile`，竞争更新用原子类或锁。

**工程指标**  
- 配置刷新生效延迟：看状态传播是否及时。  
- 状态开关一致性检查：判断多个线程看到的值是否一致。  
- 并发计数误差率：帮助发现把 `volatile` 用错成计数器。  
- 热点变量读写频率：看是否有必要进一步优化成原子类或分段设计。  
- CPU 缓存 miss / 屏障敏感热点：在极高频场景下辅助分析性能影响。

**易错点**  
- 认为 `volatile` 等于轻量锁  
- 把可见性和原子性混为一谈  
- 不知道它为什么能禁止部分重排

## 8. `synchronized` 的底层原理是什么？

**核心结论**  
`synchronized` 底层依赖对象监视器和字节码级别的进入/退出锁语义，在不同竞争程度下会经历偏向、轻量级到重量级等状态演进。

**标准答案**  
从语言层面看，`synchronized` 用来保证同一时刻只有一个线程进入临界区；从底层看，它依赖对象头里的监视器信息以及 JVM 对 `monitorenter` / `monitorexit` 指令的支持。这样设计的目的，是让开发者以相对简单的语法获得原子性、可见性和有序性保障。JVM 为了避免“一上来就是重锁”，会在不同竞争场景下做锁优化，比如偏向锁、轻量级锁、自旋和锁膨胀，让低竞争场景尽量便宜，高竞争场景再升级。工程上，`synchronized` 最大的优点是语义清晰、出错概率低，很多普通业务临界区其实已经够用；但如果锁粒度太大、热点竞争太强，吞吐和延迟都会明显受影响。

**追问点答案**  
- 为什么说 `synchronized` 也有性能优化？  
因为 JVM 不会所有情况都直接走重量级锁，会根据竞争程度优化。  
- `synchronized` 能保证可见性吗？  
能，进入和退出同步块时会建立内存语义。  
- 为什么锁竞争激烈时性能会变差？  
因为线程会阻塞、唤醒、上下文切换，系统成本迅速上升。

**原理补充**  
`synchronized` 本质上不是“加个标记”，而是 JVM 在对象监视器上建立了一套完整的互斥和内存语义协议。

**架构扩展**  
很多线上锁问题不是因为 `synchronized` 本身不好，而是临界区设计过大。比如把整个下单流程都包进同步块，里面还有远程调用和数据库操作，那锁就会被长时间持有，系统很容易雪崩。更合理的做法通常是缩小锁范围、拆分共享状态、用更高层并发控制替代粗粒度互斥。

**工程指标**  
- 锁竞争时间：看线程是不是大量卡在临界区外。  
- 线程 `BLOCKED` 数量：快速判断 `synchronized` 是否已成为瓶颈。  
- 临界区平均耗时：看锁住的代码是否过重。  
- 请求超时率：重锁场景最终常会放大到业务超时。  
- CPU 上下文切换次数：判断锁竞争是否已经转化为系统级损耗。

**易错点**  
- 只会说“悲观锁”  
- 不知道对象监视器和字节码指令  
- 忽略临界区大小对性能的影响

## 9. `ReentrantLock` 比 `synchronized` 多了什么能力？

**核心结论**  
`ReentrantLock` 在互斥之外提供了更丰富的控制能力，比如可中断获取锁、可超时、可公平锁、多个条件队列等。

**标准答案**  
`ReentrantLock` 和 `synchronized` 都能做互斥，但 `ReentrantLock` 提供了更多显式控制能力。它支持可中断加锁、超时尝试加锁、公平锁策略，以及通过 `Condition` 创建多个等待队列，这些都是 `synchronized` 很难优雅表达的。之所以会有它，是因为很多复杂并发场景不只是“互斥进入临界区”，还需要更精细的等待、唤醒和失败处理。工程上，如果只是简单同步，`synchronized` 语义更直观、出错更少；如果需要可中断获取锁、限时等待、多个条件队列，`ReentrantLock` 更适合。面试里可以总结成一句：`synchronized` 更像内置基础能力，`ReentrantLock` 更像可编排的并发工具。

**追问点答案**  
- 什么叫可中断获取锁？  
线程在等待锁时可以响应中断，不必一直死等。  
- 公平锁一定更好吗？  
不一定，更公平通常意味着更低吞吐。  
- `Condition` 为什么比 `wait` / `notify` 更灵活？  
因为它可以基于同一把锁拆分多个等待队列，语义更清晰。

**原理补充**  
`ReentrantLock` 的价值本质上不在“比 `synchronized` 更高级”，而在于它把锁管理从 JVM 隐式语义扩展成了应用层可编排能力。

**架构扩展**  
真实项目里，复杂调度、线程池协作、有多个等待条件的任务系统，经常更适合 `ReentrantLock`。但它的风险也更高，比如忘记 `unlock`、异常路径没释放锁，就会留下更隐蔽的问题。所以很多团队的策略是：简单场景优先 `synchronized`，复杂场景再用 `ReentrantLock`。

**工程指标**  
- 加锁等待时长：看是否需要可超时或可中断能力。  
- 锁获取失败率：帮助判断 `tryLock` 是否更适合当前场景。  
- 各条件队列等待长度：在复杂协作场景下很关键。  
- 锁竞争下吞吐变化：看公平/非公平策略的实际代价。  
- 死锁告警次数：显式锁用得越多，越要关注释放路径问题。

**易错点**  
- 以为 `ReentrantLock` 一定比 `synchronized` 快  
- 不知道它多出来的能力具体是什么  
- 忘记 finally 里释放锁

## 10. CAS 是什么？ABA 问题怎么解？

**核心结论**  
CAS 是一种基于比较并交换的无锁原子更新机制，ABA 问题则是值看起来没变，但中间可能已经被别人改过。

**标准答案**  
CAS 的全称是 Compare-And-Swap，它会先比较某个内存位置的当前值是否等于预期值，如果相等就原子地更新成新值，否则更新失败，调用方再决定重试。这样做的好处是很多场景不需要加锁就能完成原子更新，因此吞吐通常比重锁更好。问题在于，CAS 只比较“当前值是否相同”，如果一个值从 A 改成 B 又改回 A，CAS 会认为它没变，这就是 ABA 问题。工程上，如果场景只关心最终值，ABA 可能不是问题；但如果场景关心中间有没有人改过，比如链表节点、栈顶指针等，就要用带版本号的方案，比如 `AtomicStampedReference`。面试里最好强调：CAS 不是银弹，竞争激烈时自旋重试也可能很耗 CPU。

**追问点答案**  
- CAS 为什么能无锁？  
因为原子比较交换由底层 CPU 指令保证，不需要线程阻塞互斥。  
- ABA 在什么场景下更危险？  
在链表、栈、队列这类依赖节点身份变化的结构里尤其危险。  
- CAS 失败后一定马上重试吗？  
不一定，要看竞争程度和业务语义，有时可以退避、让步或直接失败返回。

**原理补充**  
CAS 的本质是乐观并发控制。它假设冲突不多，先不加锁，只有发现数据变了才重试。

**架构扩展**  
线上高并发计数器、限流器、无锁队列都经常依赖 CAS。但如果把它用在高冲突热点上，比如全局热点计数，失败重试会变成 CPU 空转，最后看起来“没锁却很慢”。更合理的做法是热点分片、局部聚合，别把所有线程都压到同一个原子变量上。

**工程指标**  
- CAS 失败重试次数：判断竞争是否已经太激烈。  
- CPU 自旋占比：看无锁方案是否反而浪费了 CPU。  
- 热点原子变量访问频率：识别是否需要分片。  
- 吞吐量变化：判断 CAS 相比锁到底有没有收益。  
- 版本冲突率：在带版本号场景下辅助识别 ABA 风险。

**易错点**  
- 只会说 CAS 快  
- 不知道 ABA 为什么危险  
- 忽略高冲突下自旋重试成本

## 11. AQS 是什么？

**核心结论**  
AQS 是 Java 并发包里很多同步器的基础框架，它用一个状态值和一个 FIFO 等待队列来统一描述获取/释放同步资源。

**标准答案**  
AQS 全称是 AbstractQueuedSynchronizer，可以把它理解成并发同步器的“骨架框架”。它内部核心有两个东西：一个是表示同步状态的 `state`，另一个是维护等待线程的队列。当线程获取同步资源失败时，会进入队列等待；资源释放后，再按规则唤醒后继线程。之所以设计 AQS，是为了让锁、信号量、倒计时器这类同步器共享同一套排队、唤醒、状态管理机制，而不是每个都自己从零实现。工程上，`ReentrantLock`、`Semaphore`、`CountDownLatch` 等都建立在 AQS 思想之上。面试里如果你能说出“独占模式”和“共享模式”，会更完整。

**追问点答案**  
- `state` 一定表示锁次数吗？  
不一定，它只是同步状态，具体含义由实现类决定。  
- 为什么要有队列？  
因为拿不到资源的线程需要有序等待，避免混乱竞争。  
- AQS 只服务锁吗？  
不是，它服务很多同步器，不限于互斥锁。

**原理补充**  
AQS 的本质价值是把“状态 + 排队 + 唤醒”抽象成统一框架，让同步器实现者只关注如何定义状态语义。

**架构扩展**  
大多数业务开发不需要自己实现 AQS，但理解它能帮助你更好地选择并发工具。比如知道 `CountDownLatch` 和 `Semaphore` 本质上不是神奇工具，而是共享同一套同步框架的不同状态语义。做高并发组件时，这种抽象能力很重要。

**工程指标**  
- 等待队列长度：看同步器是否已经成为热点。  
- 线程阻塞时长：判断资源获取是否过慢。  
- 资源获取成功率：看同步器设计是否合理。  
- 唤醒延迟：辅助分析排队与调度效率。  
- 线程状态分布：看 AQS 同步器是否导致大量 `WAITING` / `BLOCKED`。

**易错点**  
- 把 AQS 当成某一把具体锁  
- 不知道它抽象了什么  
- 只记名字，不会和并发工具联系起来

## 12. `CountDownLatch` 和 `CyclicBarrier` 区别？

**核心结论**  
`CountDownLatch` 更像一次性的倒计时门闩，`CyclicBarrier` 更像可复用的阶段性集合点。

**标准答案**  
`CountDownLatch` 的核心语义是“等计数归零再继续”，通常一个或多个线程等待，另外一些线程完成任务后调用 `countDown`，计数到 0 后等待线程继续执行；`CyclicBarrier` 的核心语义是“一组线程互相等待，直到都到齐再同时往下走”，而且它可以循环复用。之所以要区分这两个工具，是因为并发协作场景并不一样，有的是主线程等子任务完成，有的是一组工作线程要分阶段对齐。工程上，批量初始化、并行加载后统一汇总，常用 `CountDownLatch`；多阶段计算、并行分治、模拟同步步进，常用 `CyclicBarrier`。面试里可以总结成一句：前者是“等别人做完”，后者是“大家都到齐再一起走”。

**追问点答案**  
- `CountDownLatch` 为什么通常不能重用？  
因为它的计数归零后就完成使命了。  
- `CyclicBarrier` 为什么适合分阶段任务？  
因为一轮到齐后可以重置，进入下一轮。  
- 谁更适合主线程等待子线程完成？  
通常是 `CountDownLatch`。

**原理补充**  
这两个工具都是“协调多个线程节奏”的手段，但一个偏单向通知，一个偏双向汇合。

**架构扩展**  
真实系统里，这类工具常见于启动流程和批量并行任务。比如服务启动时并行加载配置、字典、缓存，最后统一继续，适合 `CountDownLatch`；如果是多阶段并行计算，比如图计算或模拟任务，每一轮都要等所有 worker 同步对齐，`CyclicBarrier` 更自然。选错工具通常会让代码语义怪异，后续维护成本很高。

**工程指标**  
- 子任务完成耗时分布：看是不是某个慢任务拖住整体。  
- barrier 等待时长：判断阶段同步是否过重。  
- 超时次数：看协作工具是否经常等不到预期线程。  
- 启动阶段并行任务成功率：在初始化场景下很重要。  
- 批处理总耗时：辅助判断协调方式是否合理。

**易错点**  
- 只记 API，不会说场景  
- 不知道 `CyclicBarrier` 可复用  
- 说不清谁是等待别人、谁是相互等待

## 13. `Semaphore` 适合什么场景？

**核心结论**  
`Semaphore` 用来控制并发访问数量，本质上是“许可证”模型，适合限流、资源池和并发窗口控制。

**标准答案**  
`Semaphore` 可以理解成一个许可证池，线程在进入某段逻辑前先申请许可证，执行完再释放，只有拿到许可证的线程才能继续执行。这样设计的目的是限制同一时刻并发进入的数量，而不是像互斥锁那样只允许一个线程进入。工程上，它很适合做连接池访问控制、下游调用并发上限、批处理任务并发度控制，或者某些本地资源的限流。面试里最好强调：`Semaphore` 管的是“并发度”，不是“资源互斥”。

**追问点答案**  
- `Semaphore(1)` 和锁一样吗？  
效果上都能限制单线程进入，但语义和使用目的不同，`Semaphore` 更像许可证控制。  
- 为什么它适合做限流？  
因为你可以很直观地限制同时执行中的任务数量。  
- 忘记 release 会怎样？  
许可证会被耗尽，后续线程可能一直卡住。

**原理补充**  
`Semaphore` 的核心思想不是加锁，而是容量治理。它把“允许多少个线程同时通过”变成了一个显式可控制的状态。

**架构扩展**  
线上很常见的用法是保护下游，比如一个服务调用第三方接口时，不能让本地线程无限并发打出去，否则本地线程池和对方接口都会被拖垮。此时在本地先用 `Semaphore` 卡住并发窗口，通常比出事后再降级更主动。更合理的做法是把并发度限制、超时和熔断一起设计。

**工程指标**  
- 可用许可证数量：判断并发窗口是否经常打满。  
- 获取许可证等待时长：看是否已经出现拥塞。  
- 下游调用超时率：判断并发控制是否有效保护了依赖。  
- 本地线程池活跃度：辅助看任务是否被无界放大。  
- 拒绝 / 限流次数：衡量容量控制的触发频率。

**易错点**  
- 把 `Semaphore` 当成锁  
- 不知道它更适合控制并发度  
- 忘记释放许可证

## 14. `ThreadLocal` 原理和内存泄漏风险？

**核心结论**  
`ThreadLocal` 通过线程内部的 `ThreadLocalMap` 为每个线程保存独立副本，但在线程池场景下如果不清理，容易形成脏数据和内存泄漏。

**标准答案**  
`ThreadLocal` 的核心语义是“同一个变量在线程内独享一份副本”。底层上，每个线程对象里都有一个 `ThreadLocalMap`，真正的数据不是存在线程局部变量对象本身，而是存在当前线程持有的 Map 里。这样设计的好处是避免多线程共享同一份上下文，比如用户信息、链路 traceId、数据库连接、格式化器等。风险在于，`ThreadLocalMap` 的 key 是弱引用，但 value 不是，如果 `ThreadLocal` 对象本身被回收了，而线程又长期存活，value 可能残留在线程里，直到下次清理机会出现；在线程池这种线程长寿场景下，如果不及时 `remove`，就容易出现上下文串用和内存泄漏。工程上，一定要记住：用完及时清理，尤其是在过滤器、拦截器和线程池任务边界上。

**追问点答案**  
- 为什么线程池下风险更大？  
因为线程不会很快结束，旧数据可能长期挂在线程上。  
- 为什么 key 弱引用还会泄漏？  
因为 value 仍然是强引用，只要线程活着，残留 value 就可能留在 Map 里。  
- `InheritableThreadLocal` 有什么额外风险？  
它会把父线程值传给子线程，更容易让上下文边界变得混乱。

**原理补充**  
`ThreadLocal` 不是把变量存在某个“神秘线程空间”，而是把值挂在线程对象内部，因此线程生命周期直接决定了数据生命周期。

**架构扩展**  
线上最典型的问题是请求上下文串用。比如一个线程池线程上次处理请求时留下了用户信息，下次复用线程处理新请求时如果没清理干净，就可能读到上一次的上下文，严重时会造成越权和脏数据。很多团队把 `ThreadLocal` 当成方便工具，但真正成熟的做法是把它只用在明确边界里，并在 filter、interceptor、task wrapper 中统一清理。

**工程指标**  
- 线程池线程寿命：线程越长寿，`ThreadLocal` 残留风险越大。  
- 单线程残留上下文数量：判断是否存在长期未清理的数据。  
- 请求上下文串用事件数：这是最危险的业务层信号。  
- 堆中 `ThreadLocalMap.Entry` 占比：辅助定位内存残留。  
- 线程池任务完成后清理成功率：确保治理动作真正落地。

**易错点**  
- 只知道“线程隔离”  
- 不知道 value 为什么会残留  
- 在线程池里用完不清理

## 15. 线程池核心参数如何理解？

**核心结论**  
线程池的核心参数不是孤立的配置项，而是一套关于吞吐、排队、资源上限和失败策略的平衡方案。

**标准答案**  
线程池常见核心参数包括核心线程数、最大线程数、空闲线程存活时间、阻塞队列、线程工厂和拒绝策略。理解它们时，不要单独看某一个参数，而要把它们放在一起看：核心线程数决定平时维持多少工作线程，最大线程数决定突发时能扩到多大，队列决定任务是先排队还是先扩线程，空闲存活时间决定额外线程多久回收，拒绝策略则决定系统扛不住时怎么办。这样设计的本质，是把任务处理能力、资源占用和故障边界显式化。工程上，线程池调优最常见的误区就是只盯线程数，却忽略队列和拒绝策略，最后问题从“执行不过来”变成“无限排队、延迟失控”。

**追问点答案**  
- 为什么不能只调大最大线程数？  
因为线程太多会带来上下文切换、内存占用和下游放大。  
- 队列越大越好吗？  
不一定，队列太大常常只是把问题从拒绝变成延迟。  
- 为什么线程池一定要有拒绝策略？  
因为系统能力是有限的，扛不住时必须有明确边界。

**原理补充**  
线程池的本质价值不是“复用线程”这么简单，更重要的是把并发执行能力变成可治理的资源模型。

**架构扩展**  
很多线上事故都是线程池参数不合理放大的。比如核心线程少、队列无界，峰值时任务都排队，用户看到的是响应越来越慢；或者最大线程数开太大，请求一来线程暴涨，结果数据库连接池和下游服务一起被冲垮。更合理的设计通常是先按业务类型分池，再根据 CPU 型、IO 型任务分别定线程和队列策略。

**工程指标**  
- 活跃线程数：看线程池是不是长期满载。  
- 队列长度：判断任务是否在持续堆积。  
- 任务等待时长：比队列长度更能反映用户感知。  
- 拒绝次数：看线程池是否已经到达能力边界。  
- 下游超时率：判断线程池扩张是否把压力传给了依赖。

**易错点**  
- 只会背参数名  
- 队列和线程数分开理解  
- 不知道线程池配置本质上是在做容量治理

## 16. 线程池拒绝策略如何选？

**核心结论**  
拒绝策略的本质是在系统过载时决定“牺牲什么”，不是随便选一个默认值就行。

**标准答案**  
当线程池和队列都到达上限时，拒绝策略决定系统如何处理新增任务。常见策略有直接抛异常、由调用线程自己执行、丢弃当前任务、丢弃队列里最旧任务等。选择时不能只看 API 名字，而要结合业务语义：如果任务绝不能悄悄丢失，通常不能选无感知丢弃；如果你希望通过反压让上游慢下来，`CallerRunsPolicy` 往往更合适；如果是可降级、可容忍损失的日志、监控、异步通知类任务，丢弃策略才可能接受。工程上，拒绝策略其实是过载保护的一部分，核心问题是“系统扛不住时，是让调用方变慢、失败，还是允许一定损失”。

**追问点答案**  
- 为什么 `CallerRunsPolicy` 常被看成一种反压？  
因为它会让提交任务的线程自己执行任务，从而拖慢上游。  
- 为什么日志异步线程池有时可以丢弃？  
因为日志在某些系统里不是主链路，可用性优先级低于核心业务。  
- 为什么默认抛异常未必不好？  
因为它至少让问题显式暴露，不会悄悄把任务吞掉。

**原理补充**  
拒绝策略本质上是在系统能力边界之外定义行为。没有明确拒绝语义，系统过载时就只能随机坏掉。

**架构扩展**  
线上最危险的不是“拒绝了任务”，而是“明明处理不过来还在假装处理”。比如一个无界排队的异步线程池，表面上没有拒绝，实际上任务已经排到几十秒后才执行，这对用户来说比立刻失败更糟。成熟系统通常会把拒绝策略和超时、熔断、限流统一设计。

**工程指标**  
- 拒绝任务数：看过载是否已经发生。  
- 调用线程执行比例：在 `CallerRuns` 场景下判断反压强度。  
- 队列排队时长：看是否已经“没拒绝但快不可用”。  
- 任务丢弃率：在可降级任务中必须明确衡量。  
- 用户请求失败率：判断拒绝策略是否已经影响核心业务。

**易错点**  
- 拒绝策略按感觉选  
- 认为不拒绝就是更稳定  
- 不把拒绝语义和业务优先级绑定

## 17. `Runnable` 和 `Callable` 区别？

**核心结论**  
`Runnable` 更适合无返回值任务，`Callable` 更适合有结果、可抛异常并需要结合 `Future` 使用的任务。

**标准答案**  
`Runnable` 的 `run` 方法没有返回值，也不能显式抛出受检异常，更适合执行型任务；`Callable` 的 `call` 方法可以返回结果，也可以抛异常，通常会和 `Future` 或 `FutureTask` 一起使用，用于异步计算和结果回收。之所以会有这两套接口，是因为并发任务并不都只是“把事做了”，很多时候还要拿到结果、判断成功失败、支持超时控制。工程上，如果任务只是简单异步执行，比如日志、清理、通知，`Runnable` 就够了；如果要并行查询、聚合多个结果、回传异常，`Callable` 更自然。面试里可以补一句：`Callable` 让异步任务更像同步函数调用的抽象。

**追问点答案**  
- 为什么线程池的 `submit` 可以接收两者？  
因为线程池既支持纯执行型任务，也支持需要返回结果的任务。  
- `Runnable` 里的异常去哪了？  
如果不手动处理，通常只能通过线程池或线程的异常处理机制感知。  
- `Future` 最大的问题是什么？  
调用 `get` 容易阻塞，如果用不好会把异步又写回同步。

**原理补充**  
两者差异的本质是“任务抽象层级不同”：一个只表达动作，一个同时表达动作、结果和失败。

**架构扩展**  
很多业务代码虽然用了异步线程池，但最后马上 `future.get()`，本质上还是同步等待，只是多了一层复杂度。更合理的做法是明确区分“真异步处理”和“并行后统一汇总”，不要为了异步而异步。

**工程指标**  
- Future 等待时长：判断异步是否被同步阻塞抵消。  
- 异步任务成功率：看 `Callable` 结果链路是否稳定。  
- 线程池阻塞时间：判断 `get()` 是否把线程占死。  
- 并行任务聚合总耗时：看异步拆分是否真的带来收益。  
- 异常回收率：确保异步异常没有悄悄丢掉。

**易错点**  
- 只记住“一个有返回值一个没有”  
- 不知道异常处理差异  
- 异步任务立即 `get` 导致设计失真

## 18. 死锁的四个必要条件？

**核心结论**  
死锁常见的四个必要条件是互斥、请求并持有、不可剥夺、循环等待，破掉其中任意一个，死锁就不成立。

**标准答案**  
经典死锁理论里，通常认为死锁成立需要四个条件：资源互斥使用、线程已经持有资源但还在请求其他资源、资源不能被强行剥夺，以及多个线程之间形成循环等待链。之所以要记这四个条件，不是为了背书，而是因为它们给了我们设计和排障的切入点。工程上，如果你想预防死锁，通常会从统一加锁顺序、减少锁嵌套、可超时获取锁、资源分级这些角度入手，本质上就是在破坏其中某个条件。面试时如果能再补一句“死锁不是只有锁才会有，数据库连接、线程池、分布式资源都可能形成类似等待环”，会更像实际经验。

**追问点答案**  
- 为什么统一加锁顺序能避免死锁？  
因为它破坏了循环等待条件。  
- 可中断或超时获取锁破坏了哪个条件？  
通常是在弱化不可剥夺和无限等待。  
- 数据库和线程池也会形成死锁吗？  
会，本质上是资源等待环，不限于 Java 对象锁。

**原理补充**  
死锁的关键不在“锁住了”，而在“资源等待形成闭环且没人能退出”。这是一个资源图问题。

**架构扩展**  
线上死锁不一定表现成所有线程全挂，有时只是少数核心线程互相等，导致关键接口逐渐卡死。比如订单服务里一个线程先拿本地锁再等数据库连接，另一个线程先拿连接再等本地锁，量大时就可能卡成死局。更合理的设计是统一资源获取顺序，并尽量避免在持锁区里做远程调用和阻塞 IO。

**工程指标**  
- 死锁检测次数：看是否已经出现明确等待环。  
- 锁等待超时数：很多死锁前兆会先表现为长时间等待。  
- 线程 `BLOCKED` 持续时间：判断是否有线程长期拿不到锁。  
- 数据库连接池耗尽率：辅助识别跨资源死锁。  
- 核心接口超时率：看局部死锁是否已经传导到业务层。

**易错点**  
- 只会背四个条件  
- 不知道如何在设计上破坏条件  
- 以为死锁只发生在 `synchronized`

## 19. 如何定位死锁？

**核心结论**  
定位死锁的关键是找到“谁持有资源、谁在等谁”，本质上是在恢复等待环。

**标准答案**  
定位死锁时，第一步通常是看线程栈和锁等待关系，最常见的工具就是 `jstack`，因为它能直接告诉你哪些线程持有哪些监视器、又在等待哪些锁。然后要把线程调用栈和业务逻辑对应起来，看是不是有锁嵌套、锁顺序不一致、持锁期间做阻塞操作等模式。之所以这么做，是因为死锁不是单个线程的问题，而是多个线程之间资源依赖形成了环。工程上，真正有效的定位通常要结合线程栈、日志、线程池状态和业务上下文一起看，不能只看某一条线程。面试里如果你能说出“先恢复等待关系图，再回到代码路径看资源获取顺序”，会比较成熟。

**追问点答案**  
- `jstack` 为什么是首选？  
因为它能直接展示线程状态、持锁关系和调用栈。  
- 如果不是对象锁死锁怎么办？  
那就要把数据库连接、线程池、外部资源也当成等待资源一起分析。  
- 为什么线上死锁有时很难复现？  
因为它通常依赖特定时序，高并发下偶发出现。

**原理补充**  
死锁定位的核心不是“找一把坏锁”，而是“还原资源依赖图”。只有看到等待环，问题才真正被定位。

**架构扩展**  
真实项目里，有些“伪死锁”其实是长时间阻塞，比如持锁代码里做远程调用，线程看起来都卡住了，但本质不是闭环等待而是锁持有太久。所以定位时一定要区分真死锁和长阻塞。更合理的治理通常是缩短持锁区、禁止持锁远程调用，并在核心资源上加超时保护。

**工程指标**  
- 死锁线程数：判断影响范围。  
- 锁等待超时数：帮助识别死锁前兆和长阻塞。  
- 线程栈相似度：很多死锁线程会集中卡在同一段路径。  
- 持锁时长：看问题是闭环死锁还是临界区过长。  
- 故障恢复时长：衡量排障和治理动作是否有效。

**易错点**  
- 只会说 `jstack`，不会解释怎么看  
- 把长阻塞误判成死锁  
- 不会回到代码看资源获取顺序

## 20. 如何设计一个高并发下安全又高性能的组件？

**核心结论**  
设计高并发组件的核心，不是先选锁还是无锁，而是先减少共享、明确一致性要求，再用最合适的同步策略控制热点。

**标准答案**  
设计一个高并发下既安全又高性能的组件，第一步不是直接上并发工具，而是先定义清楚共享状态是什么、并发下哪些结果必须绝对正确、哪些可以最终一致。然后优先从设计上减少共享，比如做无状态化、不可变对象、分片、局部聚合、读写分离；只有共享不可避免时，再选择具体同步手段，比如锁、CAS、原子类、队列或线程隔离。这样做的原因是并发性能问题很多时候并不是同步原语选错了，而是共享设计本身就太重。工程上，一个成熟答案通常还会包括失败模式，比如热点 key、锁竞争、重试风暴、队列堆积，以及配套的监控和降级策略。面试里如果你能从“先建模，再控共享，最后选并发原语”的顺序来回答，会很像架构思维。

**追问点答案**  
- 为什么要先减少共享？  
因为共享越少，并发控制越简单，性能也越容易做高。  
- 什么时候该选锁，什么时候该选 CAS？  
低冲突短临界区可以考虑 CAS，高冲突复杂逻辑通常锁更稳。  
- 高性能是不是一定要无锁？  
不一定，真正关键是热点有没有被控制住，而不是形式上有没有锁。

**原理补充**  
并发设计本质上是在正确性、吞吐、延迟和复杂度之间做取舍。没有任何一种工具能自动同时最优。

**架构扩展**  
真实系统里，高并发组件设计最怕的不是慢，而是“局部热点把全局打崩”。比如全局计数器、单热点缓存键、单队列串行写入，都会在流量上来后迅速变成瓶颈。成熟设计通常会先做分片、批量、异步化和限流，把热点摊开；然后再配合线程池、队列、锁或 CAS 做细节控制。更进一步，还要预留降级能力，比如热点失败时允许局部丢弃、退化成近似值或异步补偿。

**工程指标**  
- 吞吐量：看组件在高并发下能处理多少请求。  
- TP99 延迟：判断尾部性能是否可接受。  
- 锁竞争或 CAS 失败率：看热点控制是否到位。  
- 队列堆积长度：判断异步化是否变成积压问题。  
- 错误率 / 降级触发率：看组件在压力下是否还能优雅退化。  
- 资源利用率（CPU/内存/线程）：衡量性能是不是靠“硬扛资源”换来的。

**易错点**  
- 一上来就选具体工具  
- 只谈性能，不谈一致性和失败模式  
- 不考虑监控、限流和降级
