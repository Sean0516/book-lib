## 1. 进程和线程区别是什么？

**核心结论**  
进程是资源分配的基本单位，线程是 CPU 调度的基本单位；进程之间隔离更强，线程之间共享更紧密。

**标准答案**  
进程可以理解成一个程序运行起来后的独立资源容器，它有自己的地址空间、文件句柄、内存和系统资源；线程则是在进程内部真正执行代码的最小调度单元，同一个进程里的多个线程共享进程的大部分资源。这样设计的原因是操作系统既要做资源隔离，又要支持更轻量的并发执行，所以把“资源拥有者”和“执行单元”拆开了。工程上，进程更适合做强隔离，比如不同服务、不同容器；线程更适合做同一服务内部的并发处理，比如 Web 请求处理、任务执行、异步回调。面试里如果再补一句“线程切换成本通常低于进程切换，但线程之间共享资源也更容易产生并发问题”，会更完整。

**高分回答**
这道题应该从“资源边界、调度边界、故障边界”三个层次回答。进程拥有独立虚拟地址空间，是操作系统进行资源治理和故障隔离的重要边界；线程共享进程资源，是更轻量的执行流。共享让线程间传递对象很便宜，却也让数据竞争、内存泄漏和进程崩溃具有共同命运。进程或容器隔离更强，但需要 IPC、序列化和独立内存，不能简单地说线程永远更快。

设计时要问的不是“用进程还是线程”，而是“故障允许扩散到哪里”。同一可信服务内部的并行任务适合线程池；不可信插件、不同租户或资源消耗不可控的任务，仅分线程池并不能形成可靠隔离，还需要进程或容器边界、配额和独立依赖池。

**追问点答案**  
- 为什么进程切换通常更重？  
通常涉及地址空间、页表以及更多内核上下文切换；但线程切换也可能破坏 CPU cache、跨 NUMA 节点迁移，所以不能只按“进程重、线程轻”估算性能。
- 线程共享了什么？  
通常共享堆、方法区、文件描述符等进程级资源，栈和程序计数器是线程私有。注意“引用在线程栈上”不代表对象私有：引用指向的堆对象仍可能被共享。
- 为什么微服务更常按进程隔离，而不是按线程隔离？  
因为进程级故障边界更清晰，CPU、内存和权限也更容易独立治理。线程池只能限制并发数，无法阻止同一 JVM 内的 OOM、死循环或进程级崩溃互相影响。

**原理补充**  
这道题的关键不是背“谁大谁小”，而是理解操作系统为什么既需要隔离，又需要高效调度。进程解决资源隔离，线程解决执行并发。

**架构扩展**  
在线上系统里，这个区别会直接影响部署和稳定性设计。比如一个 Java 服务里线程打满，通常影响的是当前实例；但如果一个进程把内存吃光，可能整个容器都会被 OOM Killer 杀掉。再比如多租户平台如果用线程隔离不同租户逻辑，故障扩散风险会明显大于进程隔离。更合理的做法通常是：服务间优先用进程或容器隔离，服务内再用线程池做并发。

**工程指标**  
- 进程 RSS：看单个服务实例是否已经把总内存吃高。  
- 线程总数：判断服务内部并发模型是否失控。  
- 上下文切换次数：看 CPU 是否花了太多时间在线程切换而不是业务执行。  
- 容器 OOM / 重启次数：判断进程级资源问题是否已经影响可用性。  
- 单实例吞吐：看线程增加后是否真的带来收益，而不是只是增加切换成本。

**易错点**  
- 只会说“进程大线程小”  
- 不知道线程共享哪些资源  
- 不会把隔离和调度成本联系起来

## 2. 线程的生命周期有哪些状态？

**核心结论**  
Java 线程常见状态包括新建、就绪/运行、阻塞、等待、超时等待、终止，重点不是背名字，而是知道线程为什么卡在某个状态。

**标准答案**  
从 Java 角度看，线程通常会经历 `NEW`、`RUNNABLE`、`BLOCKED`、`WAITING`、`TIMED_WAITING` 和 `TERMINATED` 这些状态。`NEW` 是刚创建还没启动，`RUNNABLE` 表示可运行或正在运行，`BLOCKED` 通常是在等锁，`WAITING` 是无限等待其他线程唤醒，`TIMED_WAITING` 是带超时的等待，比如 `sleep`、带超时的 `wait` 或 `join`，`TERMINATED` 则是执行结束。这样划分的意义是让 JVM 和开发者都能更准确地描述线程当前在干什么。工程上，排查线程问题时，最关键的不是记住全部状态，而是看线程为什么大量停留在某个状态，比如都卡在 `BLOCKED` 往往说明锁竞争严重，大量 `WAITING` 可能说明线程池空转或任务调度异常。

**高分回答**
`Thread.State` 是 JVM 提供的诊断抽象，不等于操作系统调度状态。尤其 `RUNNABLE` 同时覆盖正在使用 CPU、等待 CPU，以及部分执行 native I/O 的线程，因此看到大量 `RUNNABLE` 不能直接断言 CPU 不够。`BLOCKED` 专指等待进入 `synchronized` 监视器；等待 AQS 锁、条件队列、`park` 或 `join`，通常会呈现 `WAITING` 或 `TIMED_WAITING`。

线上判断必须把“状态、调用栈、数量变化、业务指标”放在一起，并连续采样。单张线程快照只能说明某一瞬间在哪里，无法区分正常等待、持续阻塞和周期性抖动。

**追问点答案**  
- `RUNNABLE` 一定在真正运行吗？  
不一定。它还可能在等待调度或执行 JVM 视角下仍属于可运行态的 native 调用；例如部分 socket read 在不同实现和栈帧下也可能显示为 `RUNNABLE`。
- `BLOCKED` 和 `WAITING` 有什么核心区别？  
`BLOCKED` 是等待 `synchronized` 监视器；AQS 的 `ReentrantLock` 竞争通常通过 `park` 等待，表现为 `WAITING`。诊断时必须同时看锁类型和栈顶方法。
- 为什么线上排障要看线程状态分布？  
它只能提供方向，不能单独定因。即使有 500 个 `WAITING`，也可能只是空闲线程池；要与正常基线、队列长度、活跃线程数和多次快照中的同栈持续时间一起判断。

**原理补充**  
线程状态本质上是 JVM 对线程执行现场的抽象标签。它让我们能把“线程在等 CPU”“线程在等锁”“线程在等通知”这些不同情况区分开。

**架构扩展**  
真实系统里，线程状态分布经常是故障的第一现场。比如接口大量超时时，如果你看到线程池里的线程都在 `BLOCKED`，就要怀疑锁争用；如果都在 `WAITING`，可能是任务生产出了问题；如果大量线程在 `TIMED_WAITING`，则要看是不是 sleep、轮询或超时策略设计得不合理。成熟团队通常会把线程状态采样接入可观测系统，而不是等故障来了才手查。

**工程指标**  
- 各线程状态数量分布：快速判断系统主要卡在哪一类问题上。  
- 线程池活跃线程数：看线程是否真的在干活。  
- 线程池队列长度：辅助判断是不是任务堆积。  
- 热线程调用栈：看高 CPU 线程是在忙等还是正常计算。  
- 接口超时率：判断线程状态异常是否已经传导到业务层。

**易错点**  
- 机械背状态，不会解释原因  
- 把 `BLOCKED` 和 `WAITING` 混淆  
- 不会把线程状态和故障类型对应起来

## 3. `sleep` 和 `wait` 区别？

**核心结论**  
`sleep` 是线程类的方法，主要用于让当前线程暂停一段时间；`wait` 是对象监视器的方法，主要用于线程间协作，并且会释放锁。

**标准答案**  
`sleep` 来自 `Thread`，作用是让当前线程暂停指定时间，时间到了自动恢复可运行状态；`wait` 来自 `Object`，前提是线程已经持有该对象的监视器锁，它调用后会进入等待状态，并释放当前持有的对象锁，直到被 `notify` / `notifyAll` 唤醒或者超时。两者最根本的区别在于用途不同，`sleep` 更像线程调度层面的暂停，`wait` 是线程协作模型的一部分。工程上，如果你只是想简单延迟、退避重试，通常用 `sleep`；如果你想表达“条件不满足，等别人改变条件后再继续”，才应该用 `wait` / `notify` 这类协作机制。面试里一定要强调一句：`sleep` 不会释放已经持有的锁，`wait` 会释放对应对象锁。

**高分回答**
二者不是“都会暂停线程”的同类 API，而是两套协议。`sleep` 只改变当前线程的调度时机，不参与共享条件协作，也不会释放已经持有的任何锁；`wait` 必须在持有目标监视器时调用，它会原子地释放该监视器并进入等待集合，醒来后还要重新竞争同一把锁。

正确的条件等待模板是 `synchronized` 内用 `while (!condition) wait()`：条件检查与进入等待必须受同一把锁保护，醒来后也必须循环检查。这样同时应对虚假唤醒、超时、中断，以及被唤醒后条件已被其他线程消费的情况。

**追问点答案**  
- 为什么 `wait` 必须在同步块里调用？  
因为“检查条件—释放锁—进入等待”必须形成监视器保护下的协议；否则通知可能发生在检查之后、真正等待之前，形成永久丢失通知。
- `sleep` 被打断会怎样？  
会抛出 `InterruptedException` 并清除中断标记。若当前层不能完成取消语义，通常应恢复中断 `Thread.currentThread().interrupt()`，交给上层处理，而不是静默吞掉。
- 为什么很多项目里更少直接用 `wait` / `notify`？  
因为 `BlockingQueue`、`CountDownLatch`、`Condition` 等工具把状态、等待队列和唤醒协议封装在一起。还要注意，在 `synchronized` 中调用 `sleep` 会一直占锁，可能把一次退避变成整个临界区停顿。

**原理补充**  
这道题的底层差异本质上是“调度暂停”和“条件协作”的差异。一个是线程层操作，一个是监视器协作模型。

**架构扩展**  
线上最常见的误用是把 `sleep` 当成同步手段，比如轮询等待下游状态更新，结果线程大量空转，占着线程池不做事；或者错误使用 `wait` / `notify`，导致唤醒丢失、线程永久等待。更合理的做法是，简单退避用 `sleep`，有条件等待用并发工具类，跨服务协调则交给消息、回调或状态机。

**工程指标**  
- 线程 `TIMED_WAITING` 数量：看是否存在大量 `sleep` 式轮询。  
- 线程 `WAITING` 数量：判断是否有条件协作卡住。  
- 线程池活跃度：看线程是不是都被等待操作占住。  
- 任务平均等待时长：辅助判断等待模型是否设计过重。  
- 重试率：很多错误的 `sleep` 方案最终会放大成重试风暴。

**易错点**  
- 只记住“一个是静态方法一个是实例方法”  
- 忘记 `wait` 会释放锁  
- 把延迟和线程协作混为一谈

## 4. `notify` 和 `notifyAll` 区别？

**核心结论**  
`notify` 只随机唤醒一个等待线程，`notifyAll` 会唤醒所有等待线程；选错会带来唤醒丢失或无效竞争。

**标准答案**  
当多个线程在同一个对象监视器上等待时，`notify` 只会唤醒其中一个等待线程，而 `notifyAll` 会把所有等待线程都唤醒，让它们重新竞争锁并检查条件。这样设计是为了兼顾效率和正确性，因为有些场景只需要唤醒一个消费者，而有些场景如果只唤醒一个线程，可能唤醒到“条件不匹配”的线程，导致真正能继续执行的线程没有被唤醒。工程上，如果等待队列里只有一种条件、语义明确，`notify` 可以减少无谓竞争；但如果多个条件混在一起，或者你不确定唤醒哪个线程才对，`notifyAll` 更安全。面试里通常建议强调：并发编程里首先要保证正确性，其次才是效率。

**高分回答**
通知不会把锁或执行权直接交给等待线程，只会把线程从监视器的等待集合移入锁竞争过程。`notify` 是否正确，不取决于“通常只需一个消费者”，而取决于任意被选中的等待者是否都能消费这次状态变化；只要同一监视器承载多个等待谓词，就可能唤醒错误角色而导致活性问题。

`notifyAll` 用额外竞争换取更稳妥的活性，但大量等待者会产生惊群。更可维护的方案是用多个 `Condition` 分离不同谓词，或者直接使用 `BlockingQueue` 等成熟组件，让通知对象和业务条件一一对应。

**追问点答案**  
- 为什么 `notify` 可能有风险？  
因为 JVM 不保证选择哪个等待者。即使所有线程都在等“队列非空”，也要证明任意线程都可消费、一次变化只需推进一个等待者；复杂生产消费优先使用 `BlockingQueue`。
- `notifyAll` 会不会太重？  
会。大量线程同时从 wait set 醒来、竞争同一监视器、发现条件不满足再睡回去，会带来调度和锁竞争成本，这就是惊群。
- 为什么被唤醒后还要重新检查条件？  
因为通知只是“状态可能变化”的提示，不是条件成立的承诺。此外，`notify` 发生时若没有等待者，通知不会被保存；需要保存的是共享状态、队列元素或 `Future` 结果，而不是通知本身。

**原理补充**  
唤醒不是“直接继续执行”，而是“从等待队列进入锁竞争队列”。真正能继续往下走，还要重新拿到锁并重新校验条件。

**架构扩展**  
真实系统里，如果自己手写生产者消费者模型，最容易出的问题之一就是误用 `notify`，导致线程偶发卡死，线上只在高并发偶现，排查非常痛苦。也正因为如此，很多团队更倾向于直接使用 `BlockingQueue`、`Condition` 或其他更高层并发工具，减少这种底层协作错误。

**工程指标**  
- 等待线程数量：判断唤醒策略是否会影响很多线程。  
- 锁竞争时间：看 `notifyAll` 是否引发过度竞争。  
- 线程唤醒后成功执行比例：辅助判断唤醒是否精准。  
- 任务堆积长度：看错误唤醒策略是否已经影响吞吐。  
- 线程卡死告警次数：判断协作模型是否存在偶发死锁或挂起。

**易错点**  
- 以为 `notify` 更高级、一定更好  
- 忘记唤醒后还要竞争锁  
- 不知道虚假唤醒和条件重检

## 5. 什么是线程安全？

**核心结论**  
线程安全不是“用了锁就安全”，而是多个线程同时访问共享资源时，程序行为仍然正确且结果可预期。

**标准答案**  
线程安全的本质，是在并发场景下多个线程同时操作共享状态时，程序不会因为竞态条件而出现错误结果、脏数据、状态错乱或不可预期行为。之所以需要这个概念，是因为线程之间共享堆内数据，而 CPU 缓存、指令重排和执行交错都会让程序行为变复杂。工程上，判断一个组件是不是线程安全，关键不是看它有没有 `synchronized`，而是看它有没有共享状态、有没有完整的同步策略、有没有把原子性、可见性和有序性都考虑进去。比如不可变对象通常天然线程安全，局部变量一般也线程安全，但共享可变集合如果没有保护就很危险。面试里最好补一句：线程安全讲的是“结果正确”，性能和实现方式是下一层问题。

**高分回答**
线程安全必须相对于组件的可观察契约来定义：对所有允许的并发调用，状态不变量持续成立，结果能用该契约允许的顺序解释，并且不会因错误发布而看到半初始化状态。因而“每个方法都加锁”或“字段都放进并发容器”都不是充分条件，锁是否覆盖完整不变量、复合操作是否原子才是关键。

例如 `ConcurrentHashMap` 的 `get` 和 `put` 各自安全，但“若不存在则创建并产生一次外部副作用”是跨步骤协议，仍可能重复执行。审查还应覆盖对象发布、迭代视图、回调重入、异常退出和取消路径，而不只是正常方法体。

**追问点答案**  
- 无状态类为什么通常线程安全？  
因为调用之间没有共享可变状态；但它调用的依赖、静态缓存或回调若共享状态，整个组件仍未必安全。
- 线程安全和高性能一定冲突吗？  
不一定。优先通过不可变、线程封闭和所有权转移减少共享，再按竞争特征选择锁、分段或原子变量；无锁也可能在高冲突下因 CAS 重试而退化。
- 用了 `ConcurrentHashMap` 就一定线程安全吗？  
不一定。`get` 后 `put` 仍可能重复创建；`computeIfAbsent` 能原子关联键值，但映射函数应短小，不能递归更新同一映射，也不能把外部副作用的“恰好一次”误认为由容器保证。测试时应直接断言业务不变量，并用并发起跑、随机交错、压力工具或 jcstress 扩大竞态窗口。

**原理补充**  
线程安全本质上是“共享可变状态管理问题”。没有共享、没有可变，问题就会小很多。

**架构扩展**  
很多线上并发 Bug 不是因为开发者不会加锁，而是因为组件边界设计得不好。比如一个服务把请求上下文缓存成成员变量、把可变对象跨线程共享、把多个步骤拆成非原子复合操作，最后即使局部用了线程安全容器，也救不了整体逻辑。更合理的做法是优先减少共享，再决定是否同步。

**工程指标**  
- 并发错误日志数：看线程安全问题是否已经暴露成线上异常。  
- 数据不一致事件数：比如重复扣款、库存错乱，这类是业务层表现。  
- 锁竞争时间：看你为保证线程安全付出了多少代价。  
- 重试率 / 幂等冲突率：辅助判断并发写入是否失控。  
- 线程安全测试覆盖率：确保关键组件不是靠“感觉安全”。

**易错点**  
- 认为线程安全等于加锁  
- 不会区分容器安全和业务逻辑安全  
- 忽略无状态和不可变的价值

## 6. 原子性、可见性、有序性分别是什么？

**核心结论**  
这是并发正确性的三个核心维度：原子性保证操作不可被中途打断，可见性保证线程间修改能被及时看到，有序性保证执行顺序符合约束。

**标准答案**  
原子性描述其他线程能否观察到操作的中间状态，关键是先声明操作边界：单次引用赋值可以是原子的，但“检查余额—扣减—记录流水”这个业务不变量不是。可见性描述一个线程的写何时保证能被另一个线程观察到；它依赖 happens-before，而不是模糊的“过一会总能看到”。有序性描述编译器和 CPU 可以重排，但重排不能破坏单线程语义和已建立的跨线程顺序约束。
三者相互关联但不能替代。`volatile` 能建立可见性和顺序边界，却不能把 `count++` 变成原子操作；CAS 可完成单变量原子更新，但多个 CAS 不会自动维护跨字段不变量；锁在正确使用同一把锁的前提下，可以同时提供互斥和内存语义。工程上应先写出共享状态的不变量，再选择能覆盖整个不变量的同步边界。

**高分回答**
用“配置发布”和“库存扣减”对比最清楚：写线程构造不可变配置后，通过 volatile 引用整体替换，解决的是安全发布和可见性；库存扣减同时要求读取、校验和更新不可分割，仅有可见性仍会超卖，必须使用锁、数据库条件更新或等价原子事务。回答时不只给定义，还要说明每种工具覆盖了哪一类保证、没有覆盖什么。

**追问点答案**  
- `AtomicInteger` 能保证两个计数器之和始终不变吗？
不能。每个原子变量只保证自己的更新，读取和修改两个变量之间仍可能被其他线程穿插。要维护 `a + b = 常量`，应把状态放进一个不可变对象整体 CAS，或用同一把锁覆盖完整不变量。
- 为什么“最终能看到新值”不等于 JMM 可见性保证？
缓存一致性可能让某次运行碰巧观察到更新，但程序没有 happens-before 就仍存在数据竞争，编译器可做合法优化，行为也不可移植。正确性不能建立在“机器通常会刷新”上。
- 锁为什么还能解决有序性？
对同一监视器的解锁 happens-before 后续加锁。临界区内的写在释放锁前完成，后来获得锁的线程按该同步顺序观察状态；前提是所有访问都遵守同一协议。

**原理补充**  
这三个概念本质上是并发内存模型对“状态更新”提出的三类约束，没有它们，程序员写的顺序不等于机器执行和其他线程观察到的顺序。

**架构扩展**  
真实系统里，很多“偶现 Bug”都能归到这三类之一。比如配置热更新后部分线程看不到新值，是可见性；库存扣减丢失，是原子性；双重检查单例拿到半初始化对象，是有序性。能把问题归类，排查就会快很多。

**工程指标**  
- 并发写冲突率：看原子性问题是否频繁发生。  
- 配置生效延迟：辅助判断可见性问题。  
- 偶发脏数据事件数：很多有序性问题最终会落到这里。  
- 压测下异常复现率：并发问题通常需要靠高压场景触发。  
- 线程间状态不一致告警：用于发现共享变量可见性缺陷。

**保证与工具边界**

| 工具 | 能直接提供 | 不能自动提供 |
| --- | --- | --- |
| `volatile` | 单次读写、可见性、顺序边界 | 复合读改写、跨字段不变量 |
| 原子类 / CAS | 单变量原子更新与相应内存语义 | 任意业务事务、公平性、低冲突保证 |
| 锁 | 临界区互斥与 happens-before | 自动避免死锁、自动缩短临界区 |
| 不可变对象整体替换 | 一致快照和安全发布 | 高频大对象复制成本、外部副作用原子性 |

**易错点**  
- 背概念但不会举例  
- 以为 `volatile` 能解决所有三类问题  
- 说不清三者在实际问题中的区别

## 7. `volatile` 能保证什么？

**核心结论**  
`volatile` 主要保证可见性和一定程度上的有序性，但不能保证复合操作的原子性。

**标准答案**  
从 Java 内存模型看，对某个 `volatile` 变量的写 happens-before 后续对该变量的读。写线程在写入前完成的普通写，对读到新值的线程可见；同时，编译器和 CPU 不能把相关操作任意重排到这条同步边界之外。相比“刷新主内存、绕过线程缓存”的口语说法，用 happens-before 描述更准确，也不会把具体 CPU 缓存实现误当成 Java 规范。
`volatile` 单次读写具有原子性，但“读取—计算—写回”的整体没有原子性，因此 `count++`、先检查再执行、同时更新多个字段都可能竞态。它适合独立状态标志、不可变快照引用和单写多读的版本发布；涉及复合不变量时，应使用锁、原子类或不可变状态整体替换。

**高分回答**
先说明语义：`volatile` 建立可见性与有序性边界，但不提供互斥。再说明成立条件：共享状态能由一个独立变量完整表达，或者写线程构造好不可变对象后只替换引用。最后说失效场景：多个写线程竞争、跨字段不变量、读后再写和“检查后执行”都不能仅靠 `volatile`。例如降级开关可以用它，库存扣减不能；配置快照可以整体替换，分别修改 `host`、`port`、`version` 三个 volatile 字段仍可能被读到混合状态。

**追问点答案**  
- 双重检查单例为什么既需要锁又需要 `volatile`？
第一次判空用于快速路径，锁保证只有一个线程创建实例，锁内第二次判空处理并发进入；`volatile` 则保证构造完成与引用发布之间的顺序和可见性。缺少其中任何一个，都不能完整满足单例的并发语义。
- `volatile Config config` 能让 `Config` 内部的可变集合线程安全吗？
不能。它只保证引用的发布和读取语义，不会自动保护对象之后的原地修改。更稳的方案是构造不可变快照并整体替换引用，或者为内部可变状态单独建立同步协议。
- 为什么一个高频 `volatile` 写可能拖慢多核吞吐？
写入会引发缓存一致性通信；多个核心持续写同一缓存行时，所有权来回迁移。若相邻热点字段共享缓存行，还可能出现伪共享，应通过分片、批量聚合或减少共享写解决，而不是简单换成锁。

**原理补充**  
`volatile` 的底层关键点是内存屏障。它不让线程随意缓存变量旧值，也约束某些重排，从而让共享状态更新更可预测。

**架构扩展**  
线上最常见的正确用法是控制类状态，比如服务降级开关、停止消费标记、配置版本号；最常见的误用则是拿它做高并发计数，最后数据怎么都对不上。更合理的做法是：状态传播用 `volatile`，竞争更新用原子类或锁。

**工程指标**  
- 配置刷新生效延迟：看状态传播是否及时。  
- 状态开关一致性检查：判断多个线程看到的值是否一致。  
- 并发计数误差率：帮助发现把 `volatile` 用错成计数器。  
- 热点变量读写频率：看是否有必要进一步优化成原子类或分段设计。  
- CPU 缓存 miss / 屏障敏感热点：在极高频场景下辅助分析性能影响。

**适用边界**

| 场景 | 是否适合 | 原因 |
| --- | --- | --- |
| 停止标志、降级开关 | 适合 | 单变量即可表达完整状态 |
| 不可变配置快照整体替换 | 适合 | 安全发布后只读，不破坏内部不变量 |
| `count++`、库存扣减 | 不适合 | 复合读改写需要原子更新 |
| 多字段必须同时变化 | 不适合 | 独立 volatile 读写无法保证整体一致快照 |

**易错点**  
- 认为 `volatile` 等于轻量锁  
- 把可见性和原子性混为一谈  
- 不知道它为什么能禁止部分重排

## 8. `synchronized` 的底层原理是什么？

**核心结论**  
`synchronized` 提供互斥和 happens-before 语义；HotSpot 会根据竞争情况选择快速路径、自旋或监视器阻塞，但具体优化不能脱离 JDK 版本回答。

**标准答案**  
对同步代码块，字节码通常通过 `monitorenter` / `monitorexit` 表达进入和退出；同步方法通过方法访问标志表达。Java 内存模型还规定：一次解锁 happens-before 后续对同一监视器的加锁，所以退出临界区前的写入能被后来获得锁的线程看到。\
HotSpot 不会把所有竞争都立即处理成操作系统级阻塞。无竞争或低竞争时可以走较轻的获取路径，短暂竞争可能自旋，竞争持续或线程需要挂起时再进入监视器阻塞。面试里常背的“偏向锁 → 轻量级锁 → 重量级锁”是特定时期 HotSpot 的经典实现，不是 JVM 规范，也不是所有现代 JDK 的固定路径。工程决策应回到临界区长度、竞争度和是否包含阻塞操作，而不是背锁状态名称。

**高分回答**
回答 `synchronized` 要分规范语义和 HotSpot 实现。规范层保证同一监视器的互斥，并通过“解锁 happens-before 后续加锁”提供可见性；实现层才涉及对象头、快速路径、自旋和监视器阻塞，而且会随 JDK 演进。性能判断应围绕锁持有时间、竞争概率和线程挂起成本，不能拿某一代 JVM 的锁升级图当永久规范。

最危险的工程用法不是关键字本身，而是在临界区内执行远程调用、数据库访问或不可控回调。此时持锁时间由外部长尾决定，少量慢请求就能把等待队列和尾延迟一起放大；治理顺序应是缩小不变量边界、移出阻塞操作、按状态分片，最后才是替换锁实现。

**追问点答案**  
- 为什么说 `synchronized` 也有性能优化？  
因为 HotSpot 对无竞争、短暂竞争和持续竞争采用的执行路径不同，避免所有请求都直接进入昂贵阻塞。\
- `synchronized` 能保证可见性吗？  
能，进入和退出同步块时会建立内存语义。  
- 为什么锁竞争激烈时性能会变差？  
因为线程会阻塞、唤醒、上下文切换，系统成本迅速上升。
- `synchronized` 和 `ReentrantLock` 谁更快？\
没有脱离负载的固定答案。简单临界区优先选语义清晰的方案；只有需要超时、可中断、公平或多条件队列时，才应因能力选择显式锁。

**原理补充**  
`synchronized` 本质上不是“加个标记”，而是 JVM 在对象监视器上建立了一套完整的互斥和内存语义协议。

**版本边界**

OpenJDK 从 JDK 15 起默认禁用并弃用偏向锁相关选项（[JEP 374](https://openjdk.org/jeps/374)）。因此面向 JDK 17/21 的回答，应把偏向锁作为历史实现补充，而不是当前默认路径。锁实现仍可能继续演进，面试时应先声明目标 JDK。

**架构扩展**  
很多线上锁问题不是因为 `synchronized` 本身不好，而是临界区设计过大。比如把整个下单流程都包进同步块，里面还有远程调用和数据库操作，那锁就会被长时间持有，系统很容易雪崩。更合理的做法通常是缩小锁范围、拆分共享状态、用更高层并发控制替代粗粒度互斥。

**工程指标**  
- 锁竞争时间：看线程是不是大量卡在临界区外。  
- 线程 `BLOCKED` 数量：快速判断 `synchronized` 是否已成为瓶颈。  
- 临界区平均耗时：看锁住的代码是否过重。  
- JFR `JavaMonitorEnter` / 锁分析事件：定位具体竞争对象和调用栈。
- 请求超时率：重锁场景最终常会放大到业务超时。  
- CPU 上下文切换次数：判断锁竞争是否已经转化为系统级损耗。

**易错点**  
- 只会说“悲观锁”  
- 不知道对象监视器和字节码指令  
- 忽略临界区大小对性能的影响
- 不声明 JDK 版本就机械背锁升级路径

## 9. `ReentrantLock` 比 `synchronized` 多了什么能力？

**核心结论**  
`ReentrantLock` 在互斥之外提供了更丰富的控制能力，比如可中断获取锁、可超时、可公平锁、多个条件队列等。

**标准答案**  
`ReentrantLock` 和 `synchronized` 都提供可重入互斥与内存可见性，差异主要在控制能力，而不是固定性能排名。显式锁支持 `lockInterruptibly`、带超时的 `tryLock`、可选公平策略，以及一个锁对应多个 `Condition` 等待集合；这些能力适合需要取消、超时退出或多条件协作的组件。
代价是生命周期完全由代码负责：成功获得锁后必须在 `finally` 释放，条件等待必须放在 `while` 循环里重新检查谓词，超时和中断也要有明确业务语义。简单临界区优先 `synchronized`，减少漏解锁和异常路径；只有需求确实需要额外能力时才选择 `ReentrantLock`。

**高分回答**
选型从需求反推：请求线程不能无限等锁，用 `tryLock(timeout)` 快速降级；任务取消必须能打断等待，用 `lockInterruptibly`；生产者/消费者存在“非空”和“未满”两个条件，用两个 `Condition` 精确唤醒。若没有这些需求，显式锁只是增加出错面。公平锁也不是“更正确”，它减少插队但增加调度与吞吐成本，只有饥饿不可接受且已被指标证明时才考虑。

**追问点答案**  
- `tryLock(100ms)` 失败后直接重试，算解决死锁了吗？
不算。无退避重试可能形成活锁和 CPU/流量放大。失败后应释放已持有资源，并根据业务选择退避、降级或返回；统一资源顺序仍是更基础的预防手段。
- 捕获 `InterruptedException` 后什么都不做有什么风险？
上层取消信号被吞掉，线程可能继续执行已经不需要的任务。若当前层不能完成取消，应清理状态并恢复中断标记，或把中断转换成明确的业务失败。
- `Condition.await()` 为什么必须写在 `while` 而不是 `if` 中？
线程可能被虚假唤醒，也可能在重新拿到锁前条件又被其他线程改变。醒来只代表“可以再次竞争锁”，不代表业务谓词仍成立。

**原理补充**  
`ReentrantLock` 的价值本质上不在“比 `synchronized` 更高级”，而在于它把锁管理从 JVM 隐式语义扩展成了应用层可编排能力。

**架构扩展**  
真实项目里，复杂调度、线程池协作、有多个等待条件的任务系统，经常更适合 `ReentrantLock`。但它的风险也更高，比如忘记 `unlock`、异常路径没释放锁，就会留下更隐蔽的问题。所以很多团队的策略是：简单场景优先 `synchronized`，复杂场景再用 `ReentrantLock`。

**工程指标**  
- 加锁等待时长：看是否需要可超时或可中断能力。  
- 锁获取失败率：帮助判断 `tryLock` 是否更适合当前场景。  
- 各条件队列等待长度：在复杂协作场景下很关键。  
- 锁竞争下吞吐变化：看公平/非公平策略的实际代价。  
- 死锁告警次数：显式锁用得越多，越要关注释放路径问题。

**失败模式**

| 错误使用 | 后果 | 防护 |
| --- | --- | --- |
| 未在 `finally` 解锁 | 异常后永久占锁 | 获取成功后立即进入 try/finally |
| 吞掉中断 | 任务无法取消、停机变慢 | 清理后传播或恢复中断 |
| `if` 判断条件后 await | 虚假唤醒或条件变化导致越界 | `while` 循环检查谓词 |
| 默认使用公平锁 | 吞吐下降、调度开销增大 | 用饥饿指标证明需求后再启用 |

**易错点**  
- 以为 `ReentrantLock` 一定比 `synchronized` 快  
- 不知道它多出来的能力具体是什么  
- 忘记 finally 里释放锁

## 10. CAS 是什么？ABA 问题怎么解？

**核心结论**  
CAS 是一种基于比较并交换的无锁原子更新机制，ABA 问题则是值看起来没变，但中间可能已经被别人改过。

**标准答案**  
CAS 的全称是 Compare-And-Set/Swap：只有当内存位置当前值仍等于预期值时，才原子地写入新值，并返回成功；否则失败。Java 原子类通常把 CAS 放在重试循环里完成读改写，底层由 JVM 映射到目标平台的原子指令并提供相应内存语义。它避免了等待线程挂起，但不等于“没有成本”：失败重试、缓存行争用和活锁风险都可能在热点竞争下吞掉 CPU。
ABA 指值或引用从 A 变成 B 又回到 A，纯值比较无法发现中间变化。它是否有害取决于业务语义：只关心最终数值的计数可能无所谓；无锁栈、对象状态机或资源代次若依赖“期间从未变化”，就必须把版本一起比较，可用 `AtomicStampedReference`、版本字段或不可复用的状态标识解决。

**高分回答**
CAS 适合冲突较低、临界逻辑短、失败可安全重试的单变量更新。高冲突热点计数不应让所有线程争抢一个原子变量，可用 `LongAdder` 分散写热点，但要接受 `sum()` 不是线性一致快照；复杂的多字段不变量也不应硬拼多个 CAS，应考虑锁、不可变状态对象整体 CAS 或串行化处理。选型时比较的不是“有锁/无锁”标签，而是冲突率、重试成本、正确性语义和尾延迟。

**追问点答案**  
- CAS 是无锁，是否也等于 wait-free？
不是。系统整体可能持续有线程成功，因此常称 lock-free；但某个线程可能长期失败并饥饿，不能保证它在有限步内完成。是否满足更强进展保证，要看完整算法而不是一条 CAS 指令。
- `AtomicStampedReference` 为什么能解决 ABA？
比较条件从“引用是否仍为 A”扩展成“引用和版本是否都未变化”。每次成功更新同时推进版本，即使引用回到 A，旧版本的 CAS 也会失败。版本回绕和版本生成规则仍要纳入设计。
- CAS 失败率很高时如何治理？
先确认是不是单热点，再根据业务选择指数退避、分片计数、局部聚合、队列串行化或显式锁。必须用失败重试次数、CPU、吞吐和 TP99 对比，不能仅凭“无锁理论更快”坚持 CAS。

**原理补充**  
CAS 的本质是乐观并发控制。它假设冲突不多，先不加锁，只有发现数据变了才重试。

**架构扩展**  
线上高并发计数器、限流器、无锁队列都经常依赖 CAS。但如果把它用在高冲突热点上，比如全局热点计数，失败重试会变成 CPU 空转，最后看起来“没锁却很慢”。更合理的做法是热点分片、局部聚合，别把所有线程都压到同一个原子变量上。

**工程指标**  
- CAS 失败重试次数：判断竞争是否已经太激烈。  
- CPU 自旋占比：看无锁方案是否反而浪费了 CPU。  
- 热点原子变量访问频率：识别是否需要分片。  
- 吞吐量变化：判断 CAS 相比锁到底有没有收益。  
- 版本冲突率：在带版本号场景下辅助识别 ABA 风险。

**方案边界**

| 方案 | 优点 | 主要限制 |
| --- | --- | --- |
| `AtomicLong` | 读取结果具有清晰单值语义 | 单热点高竞争时重试与缓存争用明显 |
| `LongAdder` | 多写线程下分散竞争 | `sum()` 不适合要求线性一致快照的结算语义 |
| 带版本 CAS | 能识别中间变化 | 版本管理更复杂，仍是单状态原子更新 |
| 显式锁 | 容易保护复杂不变量 | 可能阻塞，需要控制临界区和超时 |

**易错点**  
- 只会说 CAS 快  
- 不知道 ABA 为什么危险  
- 忽略高冲突下自旋重试成本

## 11. AQS 是什么？

**核心结论**  
AQS 是 Java 并发包里很多同步器的基础框架，它用一个状态值和一个 FIFO 等待队列来统一描述获取/释放同步资源。

**标准答案**  
AQS 全称 `AbstractQueuedSynchronizer`，它把同步器的通用部分抽象为：一个用 volatile/CAS 管理的 `state`、一条等待队列，以及获取失败后的入队、阻塞、唤醒和取消协议。子类通过 `tryAcquire`、`tryRelease` 等方法定义“状态何时算获取成功”，AQS 负责线程排队和调度协作。
独占模式一次只允许一个持有者，例如 `ReentrantLock`；共享模式一次状态变化可能允许多个等待者通过，例如 `Semaphore` 和 `CountDownLatch`。获取失败的线程不是立即反复空转，而是在队列中通过 `LockSupport.park` 挂起，在前驱状态和资源释放允许时被唤醒后重新竞争。队列近似 FIFO，但公平性仍取决于具体同步器策略，不能把“有队列”直接等同于严格公平。

**高分回答**
理解 AQS 的重点不是背字段，而是解释职责分离：实现者定义 `state` 的业务语义，框架处理竞争线程的生命周期。以 `ReentrantLock` 为例，state 可表示重入次数；以 `Semaphore` 为例，它表示剩余许可。自定义同步器还必须处理取消、中断、超时、溢出和释放后的传播，因此业务代码通常应该组合现成同步器，而不是轻易继承 AQS。

**追问点答案**  
- AQS 队列里排在前面，为什么仍不保证一定先拿到锁？
非公平锁允许新线程在队列线程恢复调度前直接尝试 CAS；即使公平模式也受取消、调度和重新检查状态影响。FIFO 主要描述等待组织，不是操作系统级严格到达顺序承诺。
- 为什么 AQS 既需要 CAS 又需要 park/unpark？
CAS 负责短临界状态竞争，失败线程若持续自旋会浪费 CPU；队列和 park 把长期等待转成阻塞，释放方再 unpark 合适的后继。二者组合覆盖快速路径和慢速等待路径。
- 自定义 AQS 最容易错在哪里？
状态语义与重入/共享传播不一致、释放时未正确唤醒、取消节点处理错误，以及中断语义含糊。错误往往只在超时取消和高竞争交错下出现，需要专门并发测试。

**原理补充**  
AQS 的本质价值是把“状态 + 排队 + 唤醒”抽象成统一框架，让同步器实现者只关注如何定义状态语义。

**架构扩展**  
大多数业务开发不需要自己实现 AQS，但理解它能帮助你更好地选择并发工具。比如知道 `CountDownLatch` 和 `Semaphore` 本质上不是神奇工具，而是共享同一套同步框架的不同状态语义。做高并发组件时，这种抽象能力很重要。

**工程指标**  
- 等待队列长度：看同步器是否已经成为热点。  
- 线程阻塞时长：判断资源获取是否过慢。  
- 资源获取成功率：看同步器设计是否合理。  
- 唤醒延迟：辅助分析排队与调度效率。  
- 线程状态分布：看 AQS 同步器是否导致大量 `WAITING` / `BLOCKED`。

**实现边界**

- AQS 是同步器框架，不是业务任务队列，也不替代线程池。
- 等待队列解决组织问题，不自动保证业务公平或无饥饿。
- `state` 只能表达有限状态；跨资源事务和外部副作用仍需更高层协议。
- 自定义实现必须覆盖超时、中断、取消和异常路径，不能只验证成功获取。

**易错点**  
- 把 AQS 当成某一把具体锁  
- 不知道它抽象了什么  
- 只记名字，不会和并发工具联系起来

## 12. `CountDownLatch` 和 `CyclicBarrier` 区别？

**核心结论**  
`CountDownLatch` 更像一次性的倒计时门闩，`CyclicBarrier` 更像可复用的阶段性集合点。

**标准答案**  
`CountDownLatch` 的核心语义是“等计数归零再继续”，通常一个或多个线程等待，另外一些线程完成任务后调用 `countDown`，计数到 0 后等待线程继续执行；`CyclicBarrier` 的核心语义是“一组线程互相等待，直到都到齐再同时往下走”，而且它可以循环复用。之所以要区分这两个工具，是因为并发协作场景并不一样，有的是主线程等子任务完成，有的是一组工作线程要分阶段对齐。工程上，批量初始化、并行加载后统一汇总，常用 `CountDownLatch`；多阶段计算、并行分治、模拟同步步进，常用 `CyclicBarrier`。面试里可以总结成一句：前者是“等别人做完”，后者是“大家都到齐再一起走”。

**高分回答**
两者真正的差异是参与者关系和失败传播。`CountDownLatch` 是事件计数器：执行 `countDown` 的线程不需要等待，计数也不会自动复位；`CyclicBarrier` 是一组固定参与者的阶段栅栏，每个参与者都要 `await`，最后到达者可触发 barrier action，然后进入下一代。

两者都不会自动传播子任务的业务异常。Latch 若有任务异常退出但未在 `finally` 中倒数，等待方可能永久挂起；Barrier 中断、超时或重置会使当前代进入 broken 状态，其他等待者收到 `BrokenBarrierException`。生产代码必须设置超时、收集失败结果并定义取消剩余任务的策略。

**追问点答案**  
- `CountDownLatch` 为什么通常不能重用？  
因为 AQS state 只会递减到零，没有重新设置计数的公开协议；新一轮任务应创建新实例，避免旧等待者与新任务混代。
- `CyclicBarrier` 为什么适合分阶段任务？  
因为所有参与者到齐后会自动进入下一代，同一组 worker 可重复执行“计算—汇合—再计算”；但任一参与者缺席都会拖住整组，所以必须有超时和失败联动。
- 谁更适合主线程等待子线程完成？  
通常是 `CountDownLatch`，但若还要聚合返回值和异常，`CompletableFuture.allOf` 或结构化并发往往比“门闩 + 外部结果容器”表达得更完整。

**原理补充**  
这两个工具都是“协调多个线程节奏”的手段，但一个偏单向通知，一个偏双向汇合。

**架构扩展**  
真实系统里，这类工具常见于启动流程和批量并行任务。比如服务启动时并行加载配置、字典、缓存，最后统一继续，适合 `CountDownLatch`；如果是多阶段并行计算，比如图计算或模拟任务，每一轮都要等所有 worker 同步对齐，`CyclicBarrier` 更自然。选错工具通常会让代码语义怪异，后续维护成本很高。

**工程指标**  
- 子任务完成耗时分布：看是不是某个慢任务拖住整体。  
- barrier 等待时长：判断阶段同步是否过重。  
- 超时次数：看协作工具是否经常等不到预期线程。  
- 启动阶段并行任务成功率：在初始化场景下很重要。  
- 批处理总耗时：辅助判断协调方式是否合理。

**易错点**  
- 只记 API，不会说场景  
- 不知道 `CyclicBarrier` 可复用  
- 说不清谁是等待别人、谁是相互等待

## 13. `Semaphore` 适合什么场景？

**核心结论**  
`Semaphore` 用来控制并发访问数量，本质上是“许可证”模型，适合限流、资源池和并发窗口控制。

**标准答案**  
`Semaphore` 可以理解成一个许可证池，线程在进入某段逻辑前先申请许可证，执行完再释放，只有拿到许可证的线程才能继续执行。这样设计的目的是限制同一时刻并发进入的数量，而不是像互斥锁那样只允许一个线程进入。工程上，它很适合做连接池访问控制、下游调用并发上限、批处理任务并发度控制，或者某些本地资源的限流。面试里最好强调：`Semaphore` 管的是“并发度”，不是“资源互斥”。

**高分回答**
`Semaphore` 是单实例内的在途并发门控，不是按时间窗口计算 QPS 的分布式限流器。它适合约束“同时有多少任务正在占用稀缺资源”，例如下游连接或内存昂贵的解析任务；如果请求 1 ms 就完成，20 个许可可能允许很高 QPS，如果每个请求卡 5 秒，同样 20 个许可则很快耗尽。

获取许可必须带等待预算，成功后才能在 `finally` 释放，防止未获取成功却多释放造成许可膨胀。许可数也不能只按本机线程数配置，应小于下游总容量分配给当前实例的份额，并随实例数变化重新核算。

**追问点答案**  
- `Semaphore(1)` 和锁一样吗？  
都可限制一个并发，但 Semaphore 没有锁所有权，许可可由另一线程释放，也不提供可重入语义；若要保护共享不变量，优先使用具有明确所有权的锁。
- 为什么它适合做限流？  
它适合并发限流而非速率限流，直接约束在途请求对连接、线程和下游容量的占用。按每秒请求数限流还需要令牌桶等时间维度算法。
- 忘记 release 会怎样？  
许可逐步泄漏，最终所有调用都在获取阶段等待。相反，无条件 release 或重复 release 会把容量上限悄悄放大；应记录 `acquired` 状态并只释放成功获得的许可。

**原理补充**  
`Semaphore` 的核心思想不是加锁，而是容量治理。它把“允许多少个线程同时通过”变成了一个显式可控制的状态。

**架构扩展**  
线上很常见的用法是保护下游，比如一个服务调用第三方接口时，不能让本地线程无限并发打出去，否则本地线程池和对方接口都会被拖垮。此时在本地先用 `Semaphore` 卡住并发窗口，通常比出事后再降级更主动。更合理的做法是把并发度限制、超时和熔断一起设计。

**工程指标**  
- 可用许可证数量：判断并发窗口是否经常打满。  
- 获取许可证等待时长：看是否已经出现拥塞。  
- 下游调用超时率：判断并发控制是否有效保护了依赖。  
- 本地线程池活跃度：辅助看任务是否被无界放大。  
- 拒绝 / 限流次数：衡量容量控制的触发频率。

**易错点**  
- 把 `Semaphore` 当成锁  
- 不知道它更适合控制并发度  
- 忘记释放许可证

## 14. `ThreadLocal` 原理和内存泄漏风险？

**核心结论**  
`ThreadLocal` 通过线程内部的 `ThreadLocalMap` 为每个线程保存独立副本，但在线程池场景下如果不清理，容易形成脏数据和内存泄漏。

**标准答案**  
`ThreadLocal` 不是把数据存进变量本身，而是以当前线程为所有者：线程内部的 `ThreadLocalMap` 用 `ThreadLocal` 作为 key、上下文对象作为 value。隔离来自“每个线程查自己的 Map”，不是 value 被复制或自动线程安全。
Map 的 key 是弱引用，value 仍是强引用。外部不再引用某个 ThreadLocal 后，key 可能变成 null，但 value 会继续被长寿命线程持有，直到 Map 后续操作触发清理或线程终止。线程池复用还会带来比泄漏更直接的风险：下一个请求可能读到上一个请求的用户、租户或 Trace 上下文。因此设置和清理必须位于同一个任务边界，并放在 `try/finally` 中。

**高分回答**
ThreadLocal 适合线程内同步调用链传递少量上下文，不适合当通用缓存、连接生命周期管理器或跨异步任务的隐式参数。提交到另一个线程池后，上下文不会天然跟随；盲目使用可继承 ThreadLocal 也可能复制过期或敏感数据。更稳的工程方案是显式参数优先，确需透传时由 Executor 装饰器在提交侧捕获、执行侧安装，并在 finally 中恢复原值，避免污染复用线程。

**追问点答案**  
- 只在请求结束时调用 `remove()`，能处理嵌套调用吗？
若内层代码覆盖了外层已有值，直接 remove 会破坏外层上下文。通用组件应先保存旧值，结束时恢复旧值；只有确认当前层拥有完整生命周期时才删除。
- 为什么 ThreadLocal 不适合保存数据库连接？
连接是稀缺资源，需要明确借还、超时和事务边界。线程池线程长寿且任务会切换，把连接绑定在线程上容易长期占用、跨请求污染，并绕开连接池治理。
- 异步任务怎样安全传递 Trace 或租户上下文？
在提交时复制允许传播的不可变快照，执行时安装，完成后恢复；需要白名单避免 Token 等敏感数据无边界传播，并监控清理失败。不能假设 ThreadLocal 会自动跨池。

**原理补充**  
`ThreadLocal` 不是把变量存在某个“神秘线程空间”，而是把值挂在线程对象内部，因此线程生命周期直接决定了数据生命周期。

**架构扩展**  
线上最典型的问题是请求上下文串用。比如一个线程池线程上次处理请求时留下了用户信息，下次复用线程处理新请求时如果没清理干净，就可能读到上一次的上下文，严重时会造成越权和脏数据。很多团队把 `ThreadLocal` 当成方便工具，但真正成熟的做法是把它只用在明确边界里，并在 filter、interceptor、task wrapper 中统一清理。

**工程指标**  
- 线程池线程寿命：线程越长寿，`ThreadLocal` 残留风险越大。  
- 单线程残留上下文数量：判断是否存在长期未清理的数据。  
- 请求上下文串用事件数：这是最危险的业务层信号。  
- 堆中 `ThreadLocalMap.Entry` 占比：辅助定位内存残留。  
- 线程池任务完成后清理成功率：确保治理动作真正落地。

**风险分型**

| 风险 | 表现 | 治理 |
| --- | --- | --- |
| stale entry 持有大对象 | Old Gen 随线程池运行时间增长 | finally 清理、限制 value 大小、堆引用链检查 |
| 请求上下文串用 | 用户或租户数据偶发错乱 | 任务入口覆盖、出口恢复/清理、并发隔离测试 |
| 跨线程上下文丢失 | Trace 断链、审计字段为空 | 受控 Executor 装饰器或显式参数 |
| 嵌套调用破坏外层值 | 调用返回后上下文消失 | 保存旧值并恢复，而非无条件 remove |

**易错点**  
- 只知道“线程隔离”  
- 不知道 value 为什么会残留  
- 在线程池里用完不清理

## 15. 线程池核心参数如何理解？

**核心结论**  
线程池参数要从到达率、任务服务时间、下游并发上限和可接受排队时间共同推导；线程数、队列和拒绝策略必须作为一个容量模型设计。

**标准答案**  
先用 Little's Law 建立量级：稳态并发数约等于到达率乘以平均服务时间。例如任务到达率 200/s、平均耗时 100 ms，理论在途并发约为 20；若 TP99 为 400 ms，还要为长尾、波动和故障态留出空间。\
但线程数不能只按公式得出。CPU 密集任务受到核数约束；I/O 密集任务可以有更多等待线程，却不能超过数据库连接池、下游限流和本机内存能承受的范围。队列也不是越大越好：若允许最多排队 200 ms，200/s 的稳态流量对应的队列预算量级约为 40 个任务，再结合峰值系数验证，而不是默认使用无界队列。\
最终要把 `corePoolSize`、`maximumPoolSize`、有界队列、任务超时和拒绝策略一起压测。达到边界时应显式拒绝、反压或降级，不能让排队时间无限增长后再由用户超时。

**高分回答**
线程池是一个带服务台、等候区和过载出口的排队系统。先用到达率和服务时间估算稳态在途量，再受 CPU、连接池和下游配额约束线程上限，用允许排队时间反推有界队列量级，最后为容量耗尽定义拒绝或反压。平均值只用于初值，真正验收必须覆盖长尾、突发和依赖变慢。

还要理解 `ThreadPoolExecutor` 的执行顺序：核心线程满后先入队，队列满后才扩到最大线程。因此“大队列 + 很大的 maximumPoolSize”可能让最大线程配置几乎永远不生效。虚拟线程降低等待期间的平台线程成本，但不会增加数据库连接、内存或下游吞吐，容量门控仍不可省略。

**追问点答案**  
- 为什么不能只调大最大线程数？  
因为线程太多会增加上下文切换和内存占用，还可能把并发直接放大到数据库与下游。若数据库连接池只有 30，把工作线程调到 200 通常只会制造更多等待。\
- 队列越大越好吗？  
不一定，队列太大常常只是把问题从拒绝变成延迟。  
- 为什么线程池一定要有拒绝策略？  
因为系统能力是有限的，扛不住时必须有明确边界。
- 公式算出 20 个线程，为什么不能直接上线？\
因为平均值隐藏了 TP99、突发流量、任务类型和下游约束；公式只给初值，最终要用同负载压测和故障注入校准。

**原理补充**  
线程池的本质价值不是“复用线程”这么简单，更重要的是把并发执行能力变成可治理的资源模型。

**微型容量推导**

假设一个文档解析服务的教学负载为：峰值 300 请求/s，单任务 CPU 20 ms、远程存储等待 80 ms，目标排队 TP99 小于 200 ms，下游允许单实例最多 40 个并发请求。

1. 平均服务时间约 100 ms，理论在途并发约 `300 × 0.1 = 30`。
2. 初始线程数可从 30 附近压测，但最大并发必须小于等于下游 40 的保护线。
3. 200 ms 排队预算对应的任务量级约 `300 × 0.2 = 60`，因此从 60 左右的有界队列开始验证，而不是上无界队列。
4. 当活跃线程达到 40 且队列接近 60 时，非核心请求应快速失败或降级；核心请求可进入独立资源池。

这些数字是推导示例，不是生产默认值。若任务耗时分布、突发系数或下游限制变化，参数必须重算。

**方案取舍**

| 方案 | 收益 | 主要风险 | 适用条件 |
| --- | --- | --- | --- |
| 小线程池 + 小队列 | 快速暴露过载，尾延迟可控 | 峰值拒绝较早 | 低延迟同步接口 |
| 较大有界队列 | 吸收短突发 | 排队掩盖容量不足 | 可接受短时延迟的异步任务 |
| `CallerRunsPolicy` | 给提交方反压 | 可能阻塞关键调用线程 | 调用方可安全变慢 |
| 持久队列 / MQ | 跨实例削峰、可重放 | 最终一致和积压治理 | 可异步化且任务不能丢 |

**架构扩展**  
很多线上事故都是线程池参数不合理放大的。比如核心线程少、队列无界，峰值时任务都排队；或者最大线程数开太大，把数据库连接池和下游一起冲垮。应按业务优先级和依赖拆池，并让线程池上限不超过最脆弱依赖能承受的并发。虚拟线程可以降低“等待时占用平台线程”的成本，但不会扩大数据库连接、下游 QPS、CPU 和内存容量，因此仍然需要限流与并发门控。

**工程指标**  
- 活跃线程数：看线程池是不是长期满载。  
- 队列长度：判断任务是否在持续堆积。  
- 任务等待时长：比队列长度更能反映用户感知。  
- 拒绝次数：看线程池是否已经到达能力边界。  
- 下游超时率：判断线程池扩张是否把压力传给了依赖。
- 到达率、完成率与服务时间分布：验证容量模型输入是否仍成立。

**验收与回退**

- 用正常峰值、两倍突发和下游延迟放大三组场景压测。
- 观察排队 TP99、拒绝率、下游并发和整体请求 TP99，而不是只看线程池利用率。
- 如果新参数使下游超时或请求 TP99 越过 SLO，立即回到上一组配置；不要同时修改线程数、队列、超时和重试而导致无法归因。

**易错点**  
- 只会背参数名  
- 队列和线程数分开理解  
- 不知道线程池配置本质上是在做容量治理
- 用平均耗时套公式后跳过长尾、下游限制和压测验证

## 16. 线程池拒绝策略如何选？

**核心结论**  
拒绝策略的本质是在系统过载时决定“牺牲什么”，不是随便选一个默认值就行。

**标准答案**  
在线程池达到执行边界时，拒绝策略决定新增任务由谁承担、是否丢失以及调用方如何感知。需要先看 `execute` 路径：核心线程满后通常先入队；有界队列也满时才继续扩到最大线程；线程数和队列同时到上限，或线程池已关闭，任务才进入拒绝处理。忽略这条路径，就容易出现“配置了很大的 maximumPoolSize，实际从未扩容”的误判。
`AbortPolicy` 显式失败，便于上游按超时预算降级；`CallerRunsPolicy` 让提交线程执行，可形成同步反压，但如果提交者是 Netty EventLoop、调度线程或持锁线程，可能阻塞关键链路甚至形成死锁；静默丢弃只适合明确可损失且有丢弃指标的任务。绝不能丢的任务不应只依赖内存线程池，应该先进入可持久化队列，再由消费者按容量处理。

**高分回答**
先声明业务语义：任务能否丢、能否重复、调用方能否变慢、最晚何时完成。再选择过载行为：同步主链路优先快速失败并返回明确错误；允许反压且提交线程安全时才考虑 Caller Runs；遥测类任务可采样或丢弃；支付、账务等任务要用持久化队列、幂等和补偿。拒绝处理器自身必须常数时间、不可阻塞，且拒绝率、排队时长和调用方耗时必须可观测。

**追问点答案**  
- `CallerRunsPolicy` 为什么可能把局部过载扩大成全站超时？
提交线程开始执行慢任务后，原本负责接收请求、调度或持有锁的关键线程被占用；上游积压和超时重试随之增加。只有调用方允许同步变慢、没有关键线程语义且任务可在当前线程安全执行时，它才是有效反压。
- 丢弃最旧任务适合消息顺序处理吗？
通常不适合。它会破坏顺序和因果关系，还可能丢掉等待最久、最接近超时但业务价值最高的任务。若必须按优先级淘汰，应使用显式优先级和业务过期时间，而不是按入队时间盲目删除。
- 任务不能丢，改成无界队列可以吗？
不可以。无界内存队列只是把显式拒绝改成不可控延迟和 OOM。需要持久化、幂等消费、重试上限、死信和积压治理，线程池只负责限制单实例执行并发。

**原理补充**  
拒绝策略本质上是在系统能力边界之外定义行为。没有明确拒绝语义，系统过载时就只能随机坏掉。

**架构扩展**  
线上最危险的不是“拒绝了任务”，而是“明明处理不过来还在假装处理”。比如一个无界排队的异步线程池，表面上没有拒绝，实际上任务已经排到几十秒后才执行，这对用户来说比立刻失败更糟。成熟系统通常会把拒绝策略和超时、熔断、限流统一设计。

**工程指标**  
- 拒绝任务数：看过载是否已经发生。  
- 调用线程执行比例：在 `CallerRuns` 场景下判断反压强度。  
- 队列排队时长：看是否已经“没拒绝但快不可用”。  
- 任务丢弃率：在可降级任务中必须明确衡量。  
- 用户请求失败率：判断拒绝策略是否已经影响核心业务。

**策略选择表**

| 业务语义 | 推荐行为 | 必须配套 |
| --- | --- | --- |
| 同步核心请求 | 快速失败或降级 | 错误码、超时预算、限流与告警 |
| 可安全反压的生产者 | 谨慎使用 Caller Runs | 禁止关键线程提交、监控调用方耗时 |
| 可损失遥测任务 | 采样或显式丢弃 | 丢弃计数、采样率与最低保留策略 |
| 不可丢业务任务 | 持久化队列后异步执行 | 幂等、重试、DLQ、积压水位与补偿 |

**易错点**  
- 拒绝策略按感觉选  
- 认为不拒绝就是更稳定  
- 不把拒绝语义和业务优先级绑定

## 17. `Runnable` 和 `Callable` 区别？

**核心结论**  
`Runnable` 更适合无返回值任务，`Callable` 更适合有结果、可抛异常并需要结合 `Future` 使用的任务。

**标准答案**  
`Runnable` 的 `run` 方法没有返回值，也不能显式抛出受检异常，更适合执行型任务；`Callable` 的 `call` 方法可以返回结果，也可以抛异常，通常会和 `Future` 或 `FutureTask` 一起使用，用于异步计算和结果回收。之所以会有这两套接口，是因为并发任务并不都只是“把事做了”，很多时候还要拿到结果、判断成功失败、支持超时控制。工程上，如果任务只是简单异步执行，比如日志、清理、通知，`Runnable` 就够了；如果要并行查询、聚合多个结果、回传异常，`Callable` 更自然。面试里可以补一句：`Callable` 让异步任务更像同步函数调用的抽象。

**高分回答**
接口差异只是表面，更重要的是任务的结果、失败和取消由谁负责。`execute(Runnable)` 的未捕获异常通常沿工作线程异常处理路径暴露；`submit` 会把 `Runnable` 或 `Callable` 包装成 `FutureTask`，异常被保存到 Future 中，若调用方从不 `get`，失败可能悄无声息。因此不能用“提交成功”代表“执行成功”，异步任务必须有结果消费、超时和异常观测闭环。

`Future.cancel(true)` 也只是请求中断，不保证任务停止；任务需要正确响应中断并清理资源。多个任务聚合时逐个按相同超时 `get` 会累加等待预算，应该使用统一截止时间、完成队列或组合式 API，并在整体失败后取消仍无价值的任务。

**追问点答案**  
- 为什么线程池的 `submit` 可以接收两者？  
因为两者最终都可适配成带完成状态的 `RunnableFuture`；`submit(Runnable)` 的 Future 结果通常为 null，但仍可表示完成、异常和取消。
- `Runnable` 里的异常去哪了？  
取决于提交方式。`execute` 的未捕获运行时异常可到达线程的 `UncaughtExceptionHandler`；`submit` 包装后异常存入 Future，需要 `get`、`afterExecute` 检查或统一任务装饰器主动上报。
- `Future` 最大的问题是什么？  
它提供的是单结果句柄，组合与错误传播能力有限；无超时 `get` 会无限占用调用线程，按顺序获取还会形成队头阻塞。需要复杂编排时应选择 `CompletionService`、`CompletableFuture` 或结构化并发，并保持统一超时预算。

**原理补充**  
两者差异的本质是“任务抽象层级不同”：一个只表达动作，一个同时表达动作、结果和失败。

**架构扩展**  
很多业务代码虽然用了异步线程池，但最后马上 `future.get()`，本质上还是同步等待，只是多了一层复杂度。更合理的做法是明确区分“真异步处理”和“并行后统一汇总”，不要为了异步而异步。

**工程指标**  
- Future 等待时长：判断异步是否被同步阻塞抵消。  
- 异步任务成功率：看 `Callable` 结果链路是否稳定。  
- 线程池阻塞时间：判断 `get()` 是否把线程占死。  
- 并行任务聚合总耗时：看异步拆分是否真的带来收益。  
- 异常回收率：确保异步异常没有悄悄丢掉。

**易错点**  
- 只记住“一个有返回值一个没有”  
- 不知道异常处理差异  
- 异步任务立即 `get` 导致设计失真

## 18. 死锁的四个必要条件？

**核心结论**  
死锁常见的四个必要条件是互斥、请求并持有、不可剥夺、循环等待，破掉其中任意一个，死锁就不成立。

**标准答案**  
经典 Coffman 条件包括：资源具有互斥性；参与者持有已有资源同时继续请求其他资源；资源不能被系统强行剥夺；等待关系形成有向环。四者同时存在才构成这类资源死锁，因此预防策略应明确破坏哪一项，而不是泛泛地说“加超时”。
统一资源编号并按全局顺序获取，直接破坏循环等待；一次申请完整资源或拿不到就释放，破坏请求并持有；超时获取本身只是提供退出机会，只有失败路径真正释放已持有资源并终止或重试事务，等待环才会被解除。数据库锁、连接池、线程池任务和远程配额都可以成为资源节点，分析范围不能局限于 `synchronized`。

**高分回答**
预防与检测要分开：可静态排序的本地锁优先统一顺序；资源集合动态时使用超时、可取消和失败后全量释放；无法完全预防时，持续采集等待图并在检测到环后选择受害者回滚。任何方案都要评估饥饿、活锁和回滚副作用，避免“没有死锁但系统一直重试不前进”。

**追问点答案**  
- 所有锁都加 1 秒超时，为什么仍可能不可用？
线程可能同时超时、同时重试，形成活锁或重试风暴；已完成的外部副作用也未必能自动回滚。需要随机退避、重试预算、幂等和明确的资源释放顺序。
- 统一本地锁顺序后，为什么数据库仍可能死锁？
数据库实际加锁顺序由访问行、索引路径和执行计划决定，且多个服务可能以不同顺序更新数据。还需统一业务访问顺序、缩短事务，并依赖数据库死锁检测与事务重试。
- 线程池为什么也可能参与死锁？
池内所有工作线程若同步等待同一池中尚未执行的子任务，执行槽位被完全占住，形成资源等待环。拆池或改为非阻塞编排，比单纯增大线程数更可靠。

**原理补充**  
死锁的关键不在“锁住了”，而在“资源等待形成闭环且没人能退出”。这是一个资源图问题。

**架构扩展**  
线上死锁不一定表现成所有线程全挂，有时只是少数核心线程互相等，导致关键接口逐渐卡死。比如订单服务里一个线程先拿本地锁再等数据库连接，另一个线程先拿连接再等本地锁，量大时就可能卡成死局。更合理的设计是统一资源获取顺序，并尽量避免在持锁区里做远程调用和阻塞 IO。

**工程指标**  
- 死锁检测次数：看是否已经出现明确等待环。  
- 锁等待超时数：很多死锁前兆会先表现为长时间等待。  
- 线程 `BLOCKED` 持续时间：判断是否有线程长期拿不到锁。  
- 数据库连接池耗尽率：辅助识别跨资源死锁。  
- 核心接口超时率：看局部死锁是否已经传导到业务层。

**条件与破坏方式**

| 条件 | 可选破坏方式 | 新风险 |
| --- | --- | --- |
| 互斥 | 无锁、多版本或资源复制 | 一致性与合并复杂度 |
| 请求并持有 | 一次申请、失败即释放 | 资源利用率下降、饥饿 |
| 不可剥夺 | 取消、租约、事务回滚 | 补偿和幂等成本 |
| 循环等待 | 全局资源顺序、分层获取 | 动态资源排序与改造成本 |

**易错点**  
- 只会背四个条件  
- 不知道如何在设计上破坏条件  
- 以为死锁只发生在 `synchronized`

## 19. 如何定位死锁？

**核心结论**  
定位死锁的关键是找到“谁持有资源、谁在等谁”，本质上是在恢复等待环。

**标准答案**  
定位死锁先区分“JVM 能直接识别的锁环”和“跨资源等待环”。对 Java 监视器与部分 `OwnableSynchronizer`，使用 `jcmd <pid> Thread.print -l` 或 `jstack -l` 获取线程、持锁和等待关系，工具通常会在末尾报告检测到的 Java-level deadlock。要保留完整调用栈、锁对象标识和时间戳，把每条边还原为“线程 A 持有 X 等待 Y”。
如果工具未报告死锁，不能立刻排除：数据库锁、连接池、线程池、文件锁和远程调用可能跨越 JVM 可见边界。此时需要多份间隔线程快照、数据库锁等待图、连接池指标和超时日志共同恢复资源图，并区分真正的环、锁饥饿、线程池耗尽和单个下游长期阻塞。止血通常是摘流量、隔离实例或重启受影响实例；长期修复必须统一锁顺序、减少嵌套和持锁 I/O，并补并发回归测试。

**高分回答**
回答应形成证据链：先确认业务影响和发生窗口；再采集两到三份线程栈，识别稳定等待关系；随后补齐 JVM 外资源，画出有向等待图并找到环；回到代码检查锁顺序与异常路径；最后给止血、修复和复现方案。单说“用 jstack”只能算知道工具，不能算完成定位。

**追问点答案**  
- Thread Dump 里大量线程 `BLOCKED`，为什么不一定是死锁？
它只说明线程在等待监视器。若锁持有者仍在推进，可能只是严重竞争；真正死锁需要闭环且参与者都无法自行前进。应比较多份快照中的栈和锁关系，而不是按状态数量下结论。
- 两个线程池互相提交任务并同步等待，jstack 会自动报告吗？
未必。它们可能都在等待 `Future` 或队列结果，并不存在 JVM 监视器层可识别的锁环。需要把“任务占用了哪个池、又等待哪个池的新任务”纳入资源图，这类现象也常被称为线程饥饿死锁。
- 线上重启恢复后，如何证明修复有效？
构造可控并发测试放大冲突窗口，验证全局锁顺序或去嵌套改造；同时观察线程池、锁等待、超时率和死锁检测。仅凭一段时间未复现不能证明根因消失。

**原理补充**  
死锁定位的核心不是“找一把坏锁”，而是“还原资源依赖图”。只有看到等待环，问题才真正被定位。

**架构扩展**  
真实项目里，有些“伪死锁”其实是长时间阻塞，比如持锁代码里做远程调用，线程看起来都卡住了，但本质不是闭环等待而是锁持有太久。所以定位时一定要区分真死锁和长阻塞。更合理的治理通常是缩短持锁区、禁止持锁远程调用，并在核心资源上加超时保护。

**工程指标**  
- 死锁线程数：判断影响范围。  
- 锁等待超时数：帮助识别死锁前兆和长阻塞。  
- 线程栈相似度：很多死锁线程会集中卡在同一段路径。  
- 持锁时长：看问题是闭环死锁还是临界区过长。  
- 故障恢复时长：衡量排障和治理动作是否有效。

**诊断分型**

| 现象 | 关键证据 | 典型治理 |
| --- | --- | --- |
| Java 锁死锁 | Dump 中存在闭环持有/等待关系 | 统一锁顺序、减少嵌套、超时获取 |
| 长临界区竞争 | 多份 Dump 中持有者仍变化但等待很长 | 缩小临界区、移出阻塞 I/O |
| 线程池饥饿 | 池内任务同步等待同池待执行任务 | 拆池、异步编排、禁止池内同步回等 |
| 数据库/外部资源环 | DB 锁图或跨系统调用图形成闭环 | 统一资源顺序、事务缩短、超时与回滚 |

**易错点**  
- 只会说 `jstack`，不会解释怎么看  
- 把长阻塞误判成死锁  
- 不会回到代码看资源获取顺序

## 20. 如何设计一个高并发下安全又高性能的组件？

**核心结论**  
设计高并发组件的核心，不是先选锁还是无锁，而是先减少共享、明确一致性要求，再用最合适的同步策略控制热点。

**标准答案**  
设计高并发组件先定义正确性契约：操作是否要求线性一致、是否允许最终一致、重复请求如何处理、失败后能否重试。第二步识别共享状态和热点键，优先使用不可变快照、分片、线程封闭、按 key 串行或局部聚合减少竞争。第三步才选择锁、CAS、原子类、队列或消息化，并为超时、取消、容量上限和关闭过程定义明确语义。
性能验证必须覆盖稳态、突发和依赖变慢，而不是只测无竞争平均吞吐。组件达到上限时要有有界队列、拒绝或降级；重试必须幂等并带退避。最后用正确性压力测试、线性化历史检查或不变量校验验证“快但没算错”，再用吞吐、TP99、竞争率、队列等待和资源成本验证性能。

**高分回答**
以“热点计数组件”为例：若结算要求精确单值，可按业务 key 分片并在每片内使用原子更新或串行批处理；若监控统计允许短暂误差，可用 `LongAdder` 做本地聚合并周期汇总。对单热点不能只增加线程，应设每 key 并发门控和过载降级。实例故障时，明确状态是否可丢、从日志/MQ 重放还是依赖外部存储恢复。这个回答同时覆盖正确性、竞争拆分、容量边界和故障恢复，才算组件设计，而不是并发 API 罗列。

**追问点答案**  
- 如何证明组件“线程安全”，而不只是压测没报错？
先写出不变量和并发历史允许的结果，再用屏障同时启动、随机调度、故障注入和长时间压力测试验证；关键算法可使用 jcstress 或线性一致性检查。吞吐测试只能证明性能，不能证明所有交错下都正确。
- 分片能提升吞吐，会付出什么代价？
跨分片读取难以获得瞬时一致快照，扩缩分片涉及迁移，热点 key 仍可能集中，内存和管理成本也会上升。必须声明聚合一致性和再均衡方案。
- 组件关闭时为什么也要设计并发语义？
关闭可能与提交、执行和回调并发发生。要明确是否接受新任务、在途任务是等待、取消还是转存，以及超时后如何释放资源；否则停机和发布阶段最容易丢任务或永久阻塞。

**原理补充**  
并发设计本质上是在正确性、吞吐、延迟和复杂度之间做取舍。没有任何一种工具能自动同时最优。

**架构扩展**  
真实系统里，高并发组件设计最怕的不是慢，而是“局部热点把全局打崩”。比如全局计数器、单热点缓存键、单队列串行写入，都会在流量上来后迅速变成瓶颈。成熟设计通常会先做分片、批量、异步化和限流，把热点摊开；然后再配合线程池、队列、锁或 CAS 做细节控制。更进一步，还要预留降级能力，比如热点失败时允许局部丢弃、退化成近似值或异步补偿。

**工程指标**  
- 吞吐量：看组件在高并发下能处理多少请求。  
- TP99 延迟：判断尾部性能是否可接受。  
- 锁竞争或 CAS 失败率：看热点控制是否到位。  
- 队列堆积长度：判断异步化是否变成积压问题。  
- 错误率 / 降级触发率：看组件在压力下是否还能优雅退化。  
- 资源利用率（CPU/内存/线程）：衡量性能是不是靠“硬扛资源”换来的。

**验收矩阵**

| 维度 | 必测内容 |
| --- | --- |
| 正确性 | 不变量、重复请求、乱序、取消、超时和实例故障 |
| 性能 | 稳态、突发、热点 key、长尾任务和依赖变慢 |
| 过载 | 队列上限、拒绝语义、降级效果和重试放大 |
| 生命周期 | 启动、扩缩容、优雅关闭、强制终止和状态恢复 |

**易错点**  
- 一上来就选具体工具  
- 只谈性能，不谈一致性和失败模式  
- 不考虑监控、限流和降级
