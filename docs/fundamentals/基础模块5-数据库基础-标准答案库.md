## 1. 事务四大特性 ACID 是什么？

**核心结论**  
ACID 是数据库事务的四个核心保证：原子性、一致性、隔离性、持久性，它们共同保证一组操作要么正确完成，要么像没发生过一样。

**标准答案**  
ACID 可以理解成数据库事务的基本契约。原子性表示一组操作要么全部成功，要么全部回滚；一致性表示事务执行前后，数据必须满足业务和约束规则；隔离性表示并发事务之间彼此尽量不互相干扰；持久性表示事务一旦提交，结果就不能轻易丢失。之所以需要这四个特性，是因为数据库面对的不只是“写一条数据”，而是高并发、多步骤、可能失败的真实业务场景。工程上，最容易被误解的是“一致性”，它不是单靠数据库自己神奇保证的，很多时候还依赖业务约束、外键、唯一索引、应用逻辑共同配合。面试里如果能补一句“ACID 是目标，具体隔离级别和性能之间要做权衡”，会更成熟。

**追问点答案**  
- 一致性和原子性有什么区别？  
原子性强调一组操作不能只做一半，一致性强调做完之后数据仍然满足规则。  
- 持久性靠什么保证？  
通常依赖 redo log、刷盘策略和存储系统保证。  
- 事务一定越强越好吗？  
不一定，越强的事务语义通常意味着更高成本和更低并发。

**原理补充**  
ACID 本质上是在定义数据库面对并发和故障时，应该给应用什么样的可靠承诺。它不是抽象口号，而是数据库内核设计和业务建模共同作用的结果。

**架构扩展**  
在单库单表时代，ACID 很像理所当然的基础能力；但到了微服务、分布式事务、消息异步链路里，很多团队会误以为“用了数据库事务就全局一致了”。其实数据库事务只能保护它自己的那部分资源边界，跨服务之后，一致性就变成架构设计问题了。更合理的回答通常会从“单库事务没问题，跨边界就要靠补偿、幂等和状态机”这个角度展开。

**工程指标**  
- 事务成功率：看事务语义是否稳定兑现。  
- 回滚率：判断业务逻辑或竞争冲突是否过多。  
- 事务平均耗时：事务过长会直接影响并发能力。  
- 锁等待时间：看 ACID 保证是否已经转化成明显竞争成本。  
- 死锁次数：帮助识别事务边界设计是否不合理。

**证据链**  
- 事务耗时分布  
- 回滚日志或错误码分布  
- 锁等待 / 死锁日志  
- 提交成功后数据落盘与恢复验证

**低风险优化路径**  
1. 先缩短事务范围  
2. 再减少事务内远程调用和大批量操作  
3. 最后再考虑隔离级别和架构层拆分

**易错点**  
- 把一致性理解成“数据库自动帮你保证所有业务正确”  
- 不知道 ACID 之间也有成本权衡  
- 以为数据库事务能天然覆盖跨服务场景

## 2. 隔离级别有哪些？

**核心结论**  
事务隔离级别通常从低到高分为读未提交、读已提交、可重复读、串行化，隔离越强，一般并发成本也越高。

**标准答案**  
数据库常见的事务隔离级别有四种：读未提交、读已提交、可重复读和串行化。读未提交最弱，事务能读到别的事务还没提交的数据；读已提交要求只能读到已提交的数据；可重复读要求同一事务里多次读同一条件结果尽量保持一致；串行化最强，基本把并发事务按串行顺序执行。之所以设计成分层，是因为真实业务对一致性要求不同，数据库不可能用一种最强策略覆盖所有场景。工程上，隔离级别选得越高，锁冲突、等待和吞吐损耗往往越明显，所以通常不会默认追求最高隔离。面试里如果能顺手提一句“不同数据库实现细节不完全一样，尤其是可重复读和幻读处理”，会比较加分。

**追问点答案**  
- 为什么不默认都用串行化？  
因为代价太高，会严重影响并发能力。  
- 读已提交和可重复读最大的体感差异是什么？  
同一事务里重复读取同一数据时，结果是否允许变化。  
- 隔离级别是不是越低越危险？  
不绝对，要看业务是否真的依赖更强一致性。

**原理补充**  
隔离级别本质上是在“并发性能”和“读写一致性”之间做分层承诺。数据库不是不知道更强更安全，而是必须平衡代价。

**架构扩展**  
线上很多系统默认用数据库默认隔离级别，但真正遇到问题时才发现业务语义和隔离级别不匹配。比如报表系统可以接受读到稍旧数据，而扣库存、转账类场景通常更敏感。成熟系统不会只背隔离级别，而是会把业务类型和隔离要求一一对齐。

**工程指标**  
- 锁等待时间：隔离级别提高后通常先反映在这里。  
- 事务冲突率：看并发下代价是不是明显升高。  
- TP99 延迟：判断更强隔离是否已伤害请求时延。  
- 死锁次数：高隔离或长事务场景下经常上升。  
- 读写 QPS 变化：评估并发能力是否下降。

**证据链**  
- 当前数据库隔离级别配置  
- 锁等待与死锁日志  
- 高并发压测前后吞吐与延迟  
- 关键业务是否出现脏读/重复读相关异常

**低风险优化路径**  
1. 先确认业务是否真的需要更强隔离  
2. 优先缩事务和控访问模式  
3. 最后再调整隔离级别

**易错点**  
- 只会背四个名字  
- 不知道隔离级别和性能强相关  
- 不会把业务场景映射到隔离级别

## 3. 脏读、不可重复读、幻读区别？

**核心结论**  
这三者都是并发读异常，但关注点不同：脏读是读到了未提交数据，不可重复读是同一行前后读到不同结果，幻读是同一条件范围前后读到不同集合。

**标准答案**  
脏读指的是一个事务读到了另一个事务尚未提交的数据，如果对方回滚，这次读取就成了“脏”的；不可重复读指的是同一事务里，两次读取同一行数据，结果不同，通常是因为别人中途提交了更新；幻读则更偏向范围查询，比如第一次查某个条件下有 10 行，第二次再查变成了 11 行，多出来的像“幻影”一样。之所以要区分这三种现象，是因为它们对应的并发风险和解决机制并不完全一样。工程上，很多人把不可重复读和幻读混在一起，但一个偏“同一行值变了”，一个偏“结果集数量变了”。面试时如果能举库存、报表、范围扫描的例子，通常更容易说明白。

**追问点答案**  
- 为什么幻读更像范围问题？  
因为它通常发生在条件查询或范围扫描中，而不是单行主键读取。  
- 不可重复读和幻读能同时出现吗？  
可以，它们本来就是不同维度的并发异常。  
- MySQL InnoDB 为什么谈幻读时经常会提间隙锁？  
因为它要通过范围锁控制插入，减少范围结果集变化。

**原理补充**  
这三类异常本质上是在描述并发事务交错执行时，读取视角可能出现的不同不一致形式。它们是定义隔离级别必要性的基础。

**架构扩展**  
线上业务真正会踩坑的，往往不是“定义不会背”，而是对现象没有感觉。比如风控系统做范围统计时，前后查询结果不一致可能就是幻读；库存系统看见行值变化则更像不可重复读。成熟系统不会只依赖数据库隔离，还会配合唯一约束、状态机和幂等逻辑减少业务层异常。

**工程指标**  
- 并发事务冲突事件数：看这类异常是否高频出现。  
- 读写交错压测异常率：验证业务对并发异常的敏感度。  
- 范围查询结果不一致率：辅助识别幻读类风险。  
- 单行更新冲突率：更贴近不可重复读相关场景。  
- 业务补偿触发率：看数据库隔离不足是否已经传导到应用层。

**证据链**  
- 事务并发时序复现脚本  
- 隔离级别配置  
- 锁行为和查询结果对比  
- 业务日志中前后读取差异记录

**低风险优化路径**  
1. 先确认是单行问题还是范围问题  
2. 再看能否用更小事务或加约束解决  
3. 最后再考虑提高隔离级别或加锁读

**易错点**  
- 把不可重复读和幻读混为一谈  
- 只会背名词不会举例  
- 以为所有读异常都靠调隔离级别解决

## 4. MySQL 默认隔离级别是什么？

**核心结论**  
MySQL InnoDB 默认隔离级别通常是可重复读，这个选择是在一致性和并发能力之间的一种工程折中。

**标准答案**  
在 MySQL 的 InnoDB 引擎里，默认隔离级别通常是可重复读。之所以默认选这个级别，而不是更低的读已提交或更高的串行化，是因为它在大部分 OLTP 场景下能提供比较稳的读取一致性，同时又保留相对可接受的并发性能。InnoDB 通过 MVCC、Undo Log 和部分锁机制，让可重复读在很多情况下既不至于太重，也能避免常见的脏读和不可重复读问题。工程上，这个默认值并不意味着所有业务都天然适合它，比如某些追求更高并发的读多写少场景可能会更偏向读已提交。面试里如果能补一句“默认值不是最佳值，要结合业务模型判断”，会比较像真正做过数据库设计的人。

**追问点答案**  
- 为什么很多人说 Oracle 更常见读已提交？  
因为不同数据库对默认隔离级别和实现策略的工程取舍不同。  
- 可重复读默认是不是就彻底没有幻读？  
不能简单这么说，要结合 InnoDB 的具体实现和访问方式理解。  
- 默认隔离级别能随便改吗？  
不能，改之前要评估业务语义、锁冲突和回归风险。

**原理补充**  
默认隔离级别本质上是数据库厂商对最常见场景的一种“经验默认”，它不是万能设置，而是一个大多数时候可接受的平衡点。

**架构扩展**  
真实项目里，很多团队根本不知道自己数据库默认隔离级别是什么，出了并发读问题才开始补课。成熟团队在做事务设计时，通常会先确认引擎、默认隔离级别、读写模式，再决定需不需要特意调整，而不是完全依赖默认配置。

**工程指标**  
- 事务冲突率：看默认隔离是否带来过多竞争。  
- 锁等待时间：判断默认隔离的并发成本。  
- TP99 延迟：验证事务语义是否已经影响请求时延。  
- 读一致性异常数：辅助判断业务是否适配当前隔离级别。  
- 压测回归结果：隔离级别调整前后必须对比。

**证据链**  
- 当前数据库隔离级别配置  
- 核心事务链路读写模式  
- 锁等待与死锁日志  
- 业务压测数据

**低风险优化路径**  
1. 先确认默认配置和业务是否一致  
2. 优先从事务边界和 SQL 模式优化  
3. 最后再讨论是否改默认隔离

**易错点**  
- 只背“可重复读”  
- 不知道默认值的工程含义  
- 以为默认设置一定适合自己业务

## 5. InnoDB 为什么用 B+Tree？

**核心结论**  
InnoDB 选 B+Tree 的核心原因，是它更适合磁盘/页式存储环境下的范围查询、顺序访问和高扇出索引组织。

**标准答案**  
InnoDB 使用 B+Tree 作为主要索引结构，核心原因是它非常适合数据库页式存储和磁盘访问模式。B+Tree 的非叶子节点只存键和指针，不存整行数据，因此一个页里可以容纳更多索引项，树的高度更低；真正的数据都在叶子节点，而且叶子节点之间通常按顺序串联，非常适合范围查询和排序扫描。之所以不用普通二叉树，是因为层数会太高，磁盘随机访问成本太大；也不选哈希作为主索引结构，是因为哈希不擅长范围查询和排序。工程上，数据库索引设计不仅要考虑等值查询，还要考虑范围、排序、分页和回表路径，所以 B+Tree 是非常均衡的选择。面试里如果能补一句“数据库索引设计首先是 IO 问题，不是算法教科书问题”，会很加分。

**追问点答案**  
- B+Tree 为什么比 B-Tree 更适合数据库？  
因为数据集中在叶子节点，非叶子节点更轻，扇出更高。  
- 哈希为什么不适合主流数据库通用索引？  
因为它不支持高效范围查询和排序。  
- 为什么树高度低很重要？  
因为每多一层通常就可能多一次页访问，延迟和 IO 都会上升。

**原理补充**  
数据库索引结构的核心优化对象，不是 CPU 比较次数，而是页访问次数和顺序访问能力。B+Tree 正好契合这类存储模型。

**架构扩展**  
线上很多索引问题，本质上不是“有没有索引”，而是访问模式和 B+Tree 的能力是否对齐。比如大量范围查询、排序分页、本地缓存失效后回源，都会放大 B+Tree 路径质量的重要性。成熟团队做索引设计时，会把查询模式和存储结构一起考虑，而不是凭感觉加索引。

**工程指标**  
- 查询平均扫描页数：看索引树高度和命中路径是否合理。  
- rows scanned / rows returned：判断 B+Tree 路径选择是否有效。  
- 范围查询耗时：这是 B+Tree 优势最典型的验证场景。  
- buffer pool 命中率：看索引页是否大多能留在内存中。  
- 排序 / 临时表次数：判断查询是否没走到理想索引路径。

**证据链**  
- 查询 SQL  
- `EXPLAIN`  
- 索引定义  
- 数据量与基数

**低风险优化路径**  
1. 先看查询模式是否匹配现有 B+Tree 索引  
2. 再做联合索引或 SQL 重写  
3. 最后再考虑数据模型调整

**易错点**  
- 从算法角度空谈复杂度  
- 不知道叶子节点链表对范围查询的价值  
- 忽略页访问和磁盘 IO 这个本质

## 6. 聚簇索引和二级索引区别？

**核心结论**  
聚簇索引的叶子节点直接存整行数据，二级索引的叶子节点通常存索引列和主键值，因此二级索引查询常常还要回到主键索引再找一次。

**标准答案**  
在 InnoDB 里，聚簇索引可以理解成“数据本身按主键顺序组织”的索引结构，它的叶子节点直接存整行记录；而二级索引则是额外建立的辅助索引，叶子节点通常存索引列值和对应主键值，而不是整行数据。这样设计的好处是主键索引能天然兼顾索引和数据组织，二级索引则更灵活，能覆盖不同查询路径。代价是，很多走二级索引的查询最终还要再通过主键回到聚簇索引里取整行数据，这就是常说的回表。工程上，主键设计会直接影响聚簇索引布局，所以主键不能只看唯一，还要看长度、顺序性和写入分布。面试里最好强调：聚簇索引和主键强相关，但本质上说的是“数据怎么组织”，不是单纯一个字段类型问题。

**追问点答案**  
- 为什么主键过长不好？  
因为二级索引叶子节点会带主键值，主键越长，所有二级索引越膨胀。  
- 没有主键时 InnoDB 怎么办？  
会选一个唯一非空索引，实在没有再生成隐藏主键。  
- 聚簇索引是不是一个表只能有一个？  
是的，因为数据物理组织方式只能有一种。

**原理补充**  
聚簇索引和二级索引的本质差异是“叶子节点存什么”和“数据最终按什么顺序组织”。这是 InnoDB 查询和写入成本分析的基础。

**架构扩展**  
线上很多索引设计问题根上在主键。比如用业务长字符串当主键，结果二级索引体积暴涨；或者用随机 UUID 做主键，导致页分裂和写放大明显。成熟设计通常会优先选短、稳定、尽量顺序的主键，再让二级索引围绕查询模式展开。

**工程指标**  
- 二级索引体积：反映主键设计是否过重。  
- 回表比例：判断二级索引路径是否高效。  
- 页分裂频率：辅助识别随机主键写入问题。  
- 查询 TP99：看索引路径是否已经伤害时延。  
- buffer pool 命中率：索引越大越容易影响缓存效果。

**证据链**  
- 主键定义  
- 二级索引定义  
- `EXPLAIN` 和回表场景  
- 索引大小与表大小

**低风险优化路径**  
1. 先确认主键设计是否合理  
2. 再评估二级索引是否能覆盖主要查询  
3. 最后考虑主键或表结构重构

**易错点**  
- 只说“主键索引和普通索引不一样”  
- 不知道二级索引叶子节点存主键  
- 忽略主键长度对全表索引体积的影响

## 7. 什么是回表？

**核心结论**  
回表指的是先通过二级索引找到主键，再回到聚簇索引取完整行数据，这会带来额外 IO 和延迟成本。

**标准答案**  
在 InnoDB 中，如果一个查询先命中了二级索引，但查询结果需要的列没有全部包含在这个索引里，那么数据库就只能先从二级索引叶子节点拿到主键值，再去聚簇索引里找完整记录，这个过程就叫回表。之所以会发生回表，是因为二级索引通常不存整行数据，只存索引列和主键。工程上，回表本身不是错误，很多查询天然就会回表；真正的问题在于高频查询、大结果集或随机 IO 场景下，回表代价会明显放大。面试里可以总结一句：回表不是概念题，而是索引覆盖程度不够带来的真实代价。

**追问点答案**  
- 怎么减少回表？  
最直接的方式是做覆盖索引，让查询所需列都在索引里。  
- 回表为什么在高并发下更明显？  
因为额外索引跳转会放大 IO 和缓存压力。  
- 主键查询会回表吗？  
聚簇索引主键路径本身就拿到整行数据，通常不叫回表。

**原理补充**  
回表的本质是“二级索引只给了你定位信息，没有给你完整结果”，所以你必须再走一次主键路径。

**架构扩展**  
很多线上慢 SQL，看 `EXPLAIN` 好像已经用索引了，但还是慢，根因往往就是回表太多。比如推荐列表、订单列表、分页查询，如果每次命中几万条索引再去逐条回表，延迟会迅速抬高。成熟团队做热点查询优化时，经常先问一句：这个索引有没有覆盖住。

**工程指标**  
- rows scanned / rows returned：看命中后二次取数是否过重。  
- 随机读 IO：回表多时通常会上升。  
- 查询 TP95 / TP99：高并发下回表代价会在尾延迟里体现。  
- buffer pool 命中率：回表路径越多，对缓存越敏感。  
- 热点 SQL 调用频率：判断是否值得专门做覆盖优化。

**证据链**  
- `EXPLAIN` 中选中的索引  
- 实际 select 列  
- 索引定义  
- 扫描行数与返回行数

**低风险优化路径**  
1. 先缩减查询列  
2. 再评估覆盖索引  
3. 最后再考虑查询分页和模型拆分

**易错点**  
- 以为“用了索引就一定很快”  
- 不知道为什么会回表  
- 不会把回表和查询列设计联系起来

## 8. 覆盖索引为什么快？

**核心结论**  
覆盖索引快的核心原因是查询所需数据全部在索引里，避免了回表带来的额外跳转和 IO。

**标准答案**  
如果一个查询需要的列全部都已经包含在某个索引里，数据库就不需要再回到聚簇索引去取整行数据，这种情况就叫覆盖索引。它之所以快，主要有两个原因：第一，少了一次回表路径，减少随机 IO；第二，索引通常比整行数据更紧凑，一个页里能容纳更多记录，缓存命中率也更高。工程上，覆盖索引特别适合高频列表查询、分页查询、只查少量列的接口。面试里如果能补一句“覆盖索引不是列越多越好，因为索引本身也有维护和写入成本”，答案会更平衡。

**追问点答案**  
- 覆盖索引是不是要把所有列都建进去？  
不是，只覆盖高频查询真正需要的列即可。  
- 为什么索引页通常比数据页更高效？  
因为索引列更少、更紧凑，同样页大小能容纳更多记录。  
- 覆盖索引有代价吗？  
有，索引越宽，写入成本、存储成本和维护成本都会增加。

**原理补充**  
覆盖索引的本质是“把查询变成纯索引访问”，让数据库不必走第二跳。这在数据库里通常是很大的优化。

**架构扩展**  
线上热点列表接口往往很适合覆盖索引，比如订单列表只展示状态、时间、金额、用户 ID，这类查询如果每次都回表拿整行，很容易拖垮热点表。成熟团队通常会围绕高频只读查询做专门覆盖索引，而不是给所有查询都堆索引。

**工程指标**  
- rows scanned / rows returned：看覆盖后扫描是否更接近结果集大小。  
- 查询 TP99：验证覆盖索引是否真的降低尾延迟。  
- 随机 IO 次数：判断回表减少是否带来收益。  
- 索引大小：防止覆盖索引过宽导致新问题。  
- 写入延迟：评估新增索引对写路径的副作用。

**证据链**  
- 查询列列表  
- 现有索引定义  
- `EXPLAIN` 中 `Using index` 等信号  
- 写入压测对比

**低风险优化路径**  
1. 先减少 select 列  
2. 再围绕热点查询补覆盖索引  
3. 上线后同时验证读收益和写代价

**易错点**  
- 以为覆盖索引就是万能解  
- 索引列堆得过宽  
- 不同时验证写入代价

## 9. 最左前缀原则是什么？

**核心结论**  
联合索引通常只能从最左边开始连续使用，不能随意跳列，否则索引利用率会明显下降。

**标准答案**  
对于联合索引，比如 `(a, b, c)`，数据库在很多情况下只能从最左边开始连续利用，也就是先看 `a`，然后是 `a,b`，再是 `a,b,c`。如果查询条件跳过了最左列，或者中间列断掉了，索引通常就无法按理想路径继续利用，这就是常说的最左前缀原则。之所以这样，是因为 B+Tree 的有序性是按联合键从左到右组织的，后面的列排序依赖前面列的值分组。工程上，这个原则会直接影响联合索引列顺序设计，不是把最常用字段随便拼在一起就行。面试时最好补一句：最左前缀是常见经验规则，但具体还要结合谓词、范围条件和执行计划看。

**追问点答案**  
- `(a,b,c)` 能不能只用 `a,c`？  
通常不能完整高效利用，中间断掉会影响后续列使用。  
- 范围查询为什么经常让后续列利用受限？  
因为范围条件打破了继续有序定位的能力。  
- 联合索引列顺序怎么选？  
要结合过滤性、排序需求、最常见访问路径一起定。

**原理补充**  
最左前缀的底层原因，是联合索引在 B+Tree 中按多列字典序组织。后列的可用性建立在前列先被约束住的前提上。

**架构扩展**  
很多线上索引“看起来建了联合索引，实际没用上”，根因就是列顺序和查询模式不匹配。比如电商后台经常用 `(status, create_time)` 做筛选排序，如果顺序反了，查询成本可能完全不同。成熟团队做联合索引时会基于热点 SQL 列表反推列顺序，而不是凭经验排序。

**工程指标**  
- 索引命中率：看联合索引是否真的按预期被使用。  
- `EXPLAIN` 的 key / key_len：判断最左列利用程度。  
- rows scanned：看列顺序错误是否导致扫描过多。  
- 排序 / 临时表次数：判断后续排序是否失去索引支持。  
- 热点 SQL TP99：验证索引顺序调整是否有效。

**证据链**  
- 热点 SQL  
- 联合索引定义  
- `EXPLAIN` 结果  
- 数据基数与过滤性

**低风险优化路径**  
1. 先按热点 SQL 梳理访问模式  
2. 再优化联合索引列顺序  
3. 最后清理冗余或无效索引

**易错点**  
- 只会背最左前缀  
- 不知道中间断列和范围条件的影响  
- 不会结合执行计划看是否真的命中

## 10. 什么情况下索引会失效？

**核心结论**  
索引失效通常不是“数据库不聪明”，而是 SQL 写法、列使用方式或数据分布让优化器认为走索引不划算或根本用不上。

**标准答案**  
索引失效常见原因包括：对索引列做函数或运算、隐式类型转换、最左前缀被破坏、范围条件后续列利用受限、前导模糊查询、选择性太差导致优化器放弃、排序和查询路径不匹配等。之所以会发生这些问题，是因为索引的有序结构和优化器成本模型都有前提条件，SQL 写法一旦破坏这些前提，数据库就只能退化成更差的执行路径。工程上，很多慢 SQL 不是因为没索引，而是索引“理论存在、实际没用”。面试里最好把这题回答成“索引结构 + SQL 写法 + 优化器选择”三件事共同作用，而不是只背几条口诀。

**追问点答案**  
- 为什么对列做函数容易让索引失效？  
因为原始有序值被转换后，索引结构无法直接按原路径利用。  
- 为什么有时索引存在，优化器还是不走？  
因为它估算全表扫描或其他路径更便宜。  
- like '%xx' 为什么常失效？  
因为前导通配符破坏了索引的前缀有序定位。

**原理补充**  
索引失效本质上是“访问路径不再和索引组织方式匹配”或者“优化器判断别的路径更便宜”。

**架构扩展**  
线上最常见的是开发觉得“我已经建了索引”，DBA 觉得“你 SQL 把索引写废了”。比如日期函数、字符串转换、前导模糊、类型不一致，在高频业务接口里非常常见。成熟团队会在代码评审和慢 SQL 平台里把这类模式自动识别出来。

**工程指标**  
- `EXPLAIN` 走索引比例：看热点 SQL 是否经常失效。  
- 全表扫描次数：是最直接的异常信号之一。  
- rows scanned / rows returned：评估优化器路径质量。  
- 慢 SQL 数量：判断索引失效是否已经形成普遍问题。  
- 排序 / 临时表次数：很多索引失效会顺带放大这两类开销。

**证据链**  
- SQL 文本  
- `EXPLAIN`  
- 列类型与索引定义  
- 慢日志样本

**低风险优化路径**  
1. 先改 SQL 写法  
2. 再补或调整索引  
3. 最后才考虑模型重构

**易错点**  
- 以为建了索引就万事大吉  
- 只背几条口诀，不会看执行计划  
- 不知道优化器也会主动放弃索引

## 11. `EXPLAIN` 重点看哪些字段？

**核心结论**  
`EXPLAIN` 不是看一眼有没有索引，而是要通过 type、key、rows、Extra 等字段恢复优化器的真实执行路径。

**标准答案**  
看 `EXPLAIN` 时，最常关注的字段包括：`type`，它大致反映访问方式；`key` 和 `possible_keys`，看实际使用了哪个索引；`rows`，看优化器预估要扫描多少行；`key_len`，帮助判断联合索引利用到哪一段；`Extra`，则经常暴露重要信号，比如 `Using filesort`、`Using temporary`、`Using index` 等。之所以这些字段重要，是因为慢 SQL 问题通常不是“有没有索引”这么简单，而是“走没走对索引、扫了多少行、有没有额外排序和临时表”。工程上，`EXPLAIN` 是证据链的一部分，但不是全部，还要结合实际 SQL、数据分布和慢日志看。面试里最好强调：`EXPLAIN` 看的不是结果好不好看，而是执行路径合不合理。

**追问点答案**  
- `rows` 一定准确吗？  
不一定，它是优化器估算值，但很有参考意义。  
- `Using index` 一定代表很好吗？  
通常表示覆盖索引，是好信号，但仍要结合扫描行数看。  
- `Using filesort` 一定很差吗？  
不绝对，但在大数据量高频查询下通常要重点关注。

**原理补充**  
`EXPLAIN` 本质上是在暴露优化器怎么“想”的。慢 SQL 调优很多时候就是在修正优化器的选择空间。

**架构扩展**  
成熟团队做数据库调优时，不会只让开发口头说“查得慢”，而是要求必须带 `EXPLAIN`、带 SQL、带慢日志、带数据量。没有证据链，优化就很容易变成拍脑袋。数据库模块用 `EXPLAIN` 思维回答，面试官通常会感觉你做过实战。

**工程指标**  
- `rows` 估算值：看扫描量是不是离谱。  
- 走索引比例：判断优化器路径是否稳定。  
- 临时表 / filesort 出现次数：看额外代价是否高频。  
- 热点 SQL TP99：验证执行计划问题是否真的伤害业务。  
- 慢日志命中频次：看是否值得优先优化。

**证据链**  
- SQL  
- `EXPLAIN`  
- 慢日志  
- 行数和索引定义

**低风险优化路径**  
1. 先看执行计划  
2. 再改 SQL 或索引  
3. 优化后回看 `EXPLAIN` 与实际耗时

**易错点**  
- 只看有没有 key  
- 忽略 `rows` 和 `Extra`  
- 把 `EXPLAIN` 当成唯一真相，不结合实际数据

## 12. MVCC 是什么？

**核心结论**  
MVCC 是多版本并发控制，通过让不同事务读到不同版本的数据，减少读写冲突，提高并发能力。

**标准答案**  
MVCC 可以理解成数据库为了减少读写互斥，引入的“多版本快照读取”机制。它不会总是让读操作去等写锁释放，而是让事务根据自己的视图读取某个合适版本的数据，这样读和写就能在很多场景下并发进行。之所以要这么设计，是因为如果所有读都要严格等写完成，数据库并发能力会明显下降。以 InnoDB 为例，MVCC 常常依赖隐藏字段、Undo Log 和 Read View 共同实现。工程上，MVCC 的价值主要体现在一致性读、减少锁冲突和提升吞吐，但它也带来版本链长度、Undo 膨胀和长事务副作用。面试里如果能补一句“MVCC 不是没有锁，而是尽量让读走版本、写走锁”，会更到位。

**追问点答案**  
- MVCC 就等于快照读吗？  
快照读是它最典型的表现之一，但实现还依赖版本链和可见性规则。  
- 为什么长事务会伤害 MVCC？  
因为旧版本需要保留更久，Undo 和版本链会变长。  
- 当前读为什么还要加锁？  
因为它要读最新值并参与修改，不能只看历史快照。

**原理补充**  
MVCC 的本质是用“版本空间”换“并发时间”。它不是让冲突消失，而是把很多读写冲突转移成版本选择问题。

**架构扩展**  
线上很多“数据库并发还不错”的体验，背后其实是 MVCC 在默默发挥作用。但它也不是没有代价，比如长事务、批量更新、慢消费会让 Undo 版本保留过久，最终导致 purge 压力和存储膨胀。成熟团队不会只知道 MVCC 提高并发，还会关注它的维护成本。

**工程指标**  
- Undo 空间增长：看版本保留是否过久。  
- 长事务数量：这是 MVCC 压力的重要来源。  
- 锁等待时间：看 MVCC 是否真的减少了读写冲突。  
- 查询 TP99：验证快照读是否带来收益。  
- purge 延迟：辅助判断历史版本清理压力。

**证据链**  
- 长事务列表  
- Undo / purge 状态  
- 锁等待日志  
- 并发压测结果

**低风险优化路径**  
1. 先减少长事务  
2. 再控制批量更新与慢消费  
3. 最后再看隔离级别和 SQL 模式

**易错点**  
- 只会说“多版本”  
- 不知道 Undo Log 和 Read View 的角色  
- 以为 MVCC 就完全没有锁

## 13. undo log / redo log / binlog 区别？

**核心结论**  
undo log 用来回滚和支持 MVCC，redo log 用来保证崩溃恢复，binlog 用来记录逻辑变更以支持复制和审计。

**标准答案**  
这三个日志看起来都叫 log，但职责完全不同。undo log 主要服务于事务回滚和 MVCC，保存的是修改前的信息；redo log 主要服务于崩溃恢复，记录的是页级物理或逻辑恢复所需信息，核心目标是让已提交事务在崩溃后能恢复；binlog 是 MySQL Server 层的逻辑日志，记录的是语句或行变更，主要用于主从复制、审计和数据同步。之所以同时存在，是因为“事务回滚”“崩溃恢复”“复制同步”本来就是三类不同问题。工程上，很多人只会背一句“redo 保持持久性”，但真正理解这题的关键是看清它们服务的边界不一样。面试里如果能补一句“redo 属于引擎层，binlog 属于 Server 层”，会很加分。

**追问点答案**  
- 为什么已经有 redo 还要 binlog？  
因为 redo 主要服务恢复，不适合直接承担复制和逻辑同步职责。  
- undo log 为什么和 MVCC 相关？  
因为旧版本数据通常要从 undo 链里找。  
- 两阶段提交为什么常和 redo、binlog 一起讲？  
因为要保证引擎层提交和 Server 层日志写入的一致性。

**原理补充**  
这道题本质上是在区分“回滚语义”“恢复语义”“复制语义”。如果不把这三层分开，很容易全混。

**架构扩展**  
线上主从复制延迟、数据回滚、崩溃恢复、CDC 同步平台，这些问题最终都会落到这几类日志上。比如做增量订阅平台时，真正依赖的是 binlog；而事务回滚能不能成立，靠的是 undo；崩机后恢复提交结果，靠的是 redo。成熟团队回答这题通常会自然把它带到主从复制和恢复机制上。

**工程指标**  
- binlog 生成速率：看写入和同步压力。  
- redo 刷盘延迟：判断持久化链路是否健康。  
- undo 膨胀程度：辅助识别长事务问题。  
- 主从复制延迟：binlog 下游消费是否跟得上。  
- 崩溃恢复耗时：验证日志体系的恢复效果。

**证据链**  
- 日志配置  
- 主从延迟数据  
- 长事务与 Undo 状态  
- 崩溃恢复演练结果

**低风险优化路径**  
1. 先确认问题在回滚、恢复还是复制  
2. 再定位对应日志链路  
3. 最后做刷盘、事务或复制策略调整

**易错点**  
- 把三个日志混成一类  
- 不知道 undo 和 MVCC 的关系  
- 不会区分引擎层和 Server 层

## 14. 乐观锁和悲观锁区别？

**核心结论**  
悲观锁假设冲突经常发生，所以先锁住再操作；乐观锁假设冲突不多，所以先尝试更新，失败再重试或报错。

**标准答案**  
悲观锁的思路是“先保护，再修改”，通常会在读或改之前就加锁，防止别人同时修改；乐观锁的思路是“先试试看”，读取时不加重锁，更新时通过版本号、时间戳或条件判断确认数据有没有被别人改过。之所以会有这两种方案，是因为不同业务的冲突概率和成本完全不同。工程上，冲突高、修改代价大、必须强保护的场景更适合悲观锁；读多写少、冲突低、允许失败重试的场景更适合乐观锁。面试里最好强调：乐观锁不是“没有锁”，而是把冲突检测推迟到提交或更新时。

**追问点答案**  
- 乐观锁通常怎么实现？  
常见做法是版本号字段或更新时间字段比对。  
- 悲观锁一定就是数据库行锁吗？  
不一定，也可能是应用层锁或分布式锁。  
- 乐观锁失败后怎么办？  
视业务而定，可以重试、提示冲突或走补偿逻辑。

**原理补充**  
这两者的核心差异，本质上是对冲突概率的不同假设。一个提前付成本，一个延后处理冲突。

**架构扩展**  
线上最典型的场景是库存、余额、表单编辑冲突。比如后台编辑页面通常更适合乐观锁，避免长时间持有数据库锁；而高并发扣库存如果直接悲观锁全压在数据库上，热点行很快就会成为瓶颈。成熟设计往往会按冲突模型选方案，而不是固定崇拜某一种锁。

**工程指标**  
- 锁等待时长：看悲观锁代价是否过高。  
- 乐观锁冲突率：判断版本冲突是否频繁。  
- 重试成功率：看乐观锁失败后能否平滑恢复。  
- 热点行更新 TPS：识别高冲突热点。  
- 用户冲突提示率：评估业务体验影响。

**证据链**  
- 更新 SQL  
- 冲突日志  
- 锁等待 / 死锁日志  
- 热点数据访问统计

**低风险优化路径**  
1. 先看冲突概率  
2. 再选乐观还是悲观  
3. 高热点场景再配合削峰、分片或异步化

**易错点**  
- 把乐观锁理解成完全无锁  
- 不看冲突模型就选方案  
- 忽略失败后的补偿与重试策略

## 15. 行锁、表锁、间隙锁区别？

**核心结论**  
行锁控制单行并发，表锁控制整表访问，间隙锁则主要用于保护范围区间，防止并发插入带来的结果集变化。

**标准答案**  
行锁顾名思义是锁住具体记录，适合细粒度并发控制；表锁粒度更粗，通常会影响整张表的访问，并发能力明显更差；间隙锁则不是锁已有记录本身，而是锁住记录之间的区间，主要为了在范围查询和更新场景下防止别人插入新记录。之所以有这些不同粒度的锁，是因为数据库既要保证一致性，又要尽量保留并发能力。工程上，开发最容易感知的是“明明只查几条数据，为什么别人插入也被影响”，很多时候就是范围查询触发了间隙锁。面试里最好强调：锁粒度越细，通常并发越好，但实现和冲突分析也更复杂。

**追问点答案**  
- 间隙锁为什么会影响插入？  
因为它锁的是区间，插入新值也属于进入这个区间。  
- 行锁是不是一定只锁一行？  
理论上取决于索引命中路径，不合理 SQL 也可能放大锁范围。  
- 为什么表锁并发差？  
因为它直接扩大了互斥范围，很多不相关操作也会互相影响。

**原理补充**  
数据库锁不只是“锁对象”，更是在定义“哪个并发访问范围不能同时进行”。行锁、表锁、间隙锁差异就在这个范围定义上。

**架构扩展**  
线上锁问题很多不是数据库自己太重，而是 SQL 和索引路径把锁放大了。比如范围更新没走好索引、批量操作覆盖面太大，很容易让间隙锁或大范围行锁扩散。成熟团队做热点事务设计时，会同时看 SQL、索引和锁范围，而不是只看表面业务逻辑。

**工程指标**  
- 锁等待时间：判断锁冲突是否严重。  
- 死锁次数：大范围锁和热点冲突常见表现。  
- 受影响事务数：看锁扩散范围是否超预期。  
- 插入延迟：间隙锁场景尤其关键。  
- 热点 SQL 执行频率：帮助识别谁在制造锁压力。

**证据链**  
- 锁等待日志  
- 死锁样本  
- SQL 与索引定义  
- 事务执行时长

**低风险优化路径**  
1. 先优化 SQL 和索引，缩小锁范围  
2. 再缩短事务  
3. 最后再考虑架构层拆热点

**易错点**  
- 只会背三个名字  
- 不知道间隙锁为什么影响插入  
- 忽略索引路径对锁范围的决定作用

## 16. 大事务有什么风险？

**核心结论**  
大事务的风险不只是“执行慢”，更严重的是会拖长锁持有时间、放大回滚成本、膨胀日志和版本链，最终伤害整个系统并发能力。

**标准答案**  
大事务通常指一次事务里修改行数过多、持续时间过长、日志量过大，或者把很多原本可以拆分的操作绑在一起。它的风险主要有几个：锁持有时间变长，别人更容易被阻塞；Undo / Redo / Binlog 都会膨胀，回滚和恢复成本上升；MVCC 旧版本保留时间变长，清理压力增大；主从复制也可能被一个大事务拖慢。之所以大事务危险，是因为数据库事务的很多成本都和事务规模近似正相关甚至放大。工程上，最典型的大事务来源包括批量更新、一次性删除大表数据、事务里做远程调用、任务型 SQL 长时间不提交。面试里如果能说出“大事务本质上是在把很多风险打包成一个大故障点”，会很有说服力。

**追问点答案**  
- 为什么大事务会影响主从复制？  
因为复制通常要等事务整体提交，单个大事务会造成明显延迟。  
- 为什么事务里不能做远程调用？  
因为外部慢点会直接拉长锁持有时间和日志保留时间。  
- 批量删除为什么危险？  
因为它会长时间持锁、产生日志膨胀，还可能拖慢整个表。

**原理补充**  
大事务的问题本质上不是“数据多”，而是“数据库很多代价项同时被放大”：锁、日志、版本、复制、恢复都会一起恶化。

**架构扩展**  
线上最典型的大事务事故就是运维或业务脚本“顺手跑个批量更新”，结果整个核心表锁冲突、复制延迟、接口超时一起上来。成熟团队通常会把批量任务拆批、限流、异步执行，并明确禁止事务里做远程调用和长耗时逻辑。

**工程指标**  
- 长事务数量：判断是否有持续性风险源。  
- 事务平均 / 最大耗时：看是不是有明显长尾。  
- 锁等待时间：大事务通常会把它拉高。  
- Undo / Binlog 增长速率：看日志成本是否被放大。  
- 主从复制延迟：识别大事务对下游影响。

**证据链**  
- 长事务列表  
- 锁等待与死锁日志  
- Binlog / Undo 状态  
- 主从延迟数据

**低风险优化路径**  
1. 先拆批  
2. 再缩事务边界  
3. 最后做批任务异步化和限流

**易错点**  
- 只把大事务理解成“慢 SQL”  
- 不知道它会拖累 MVCC 和复制  
- 事务里还做远程调用或大批量操作

## 17. 慢 SQL 的定位和优化步骤？

**核心结论**  
慢 SQL 优化不能一上来就“加索引”，而要先建立证据链：它到底慢在扫描、排序、回表、锁等待，还是事务和资源争用。

**标准答案**  
定位慢 SQL 时，第一步通常是拿到真实 SQL、慢日志样本、调用频率和 TP95 / TP99；第二步看 `EXPLAIN`，确认走没走索引、扫描多少行、有没有 filesort、临时表、回表；第三步再结合表数据量、索引定义、锁等待、事务时长，判断瓶颈是在 SQL 写法、索引设计、数据模型还是并发竞争。真正优化时，低风险顺序通常是：先改 SQL 写法，再调索引，再缩事务或热点访问模式，最后才考虑分库分表或架构重构。之所以这样，是因为很多慢 SQL 问题其实在单库内就能解决，贸然上大改造成本太高。面试里如果能明确说出“证据链驱动、低风险优先”，基本就是数据库实战型回答。

**追问点答案**  
- 为什么不建议一上来就分库分表？  
因为很多问题根本没到那个阶段，先做大改动风险太高。  
- 什么叫证据链？  
SQL、`EXPLAIN`、慢日志、扫描行数、锁等待、数据量这些证据要能互相印证。  
- 慢 SQL 一定是索引问题吗？  
不一定，也可能是锁、事务、热点、模型或资源瓶颈。

**原理补充**  
慢 SQL 优化本质上是恢复数据库的真实执行代价。没有证据链，优化就只是猜。

**架构扩展**  
线上最危险的不是慢 SQL 本身，而是热点慢 SQL 被高频调用，最后把数据库 CPU、IO、连接池一起压满。成熟团队通常会有慢 SQL 平台，把 SQL、执行计划、频次、调用链一起展示，而不是靠人肉排查。

**工程指标**  
- SQL TP95 / TP99：看延迟到底有多严重。  
- rows scanned / rows returned：判断执行路径是否粗糙。  
- 锁等待时间：排除是不是被别人堵住。  
- filesort / temporary 次数：识别额外代价。  
- 调用频率：判断优化优先级。

**证据链**  
- SQL  
- `EXPLAIN`  
- 慢日志  
- 锁等待 / 事务日志  
- 数据量与索引定义

**低风险优化路径**  
1. SQL 改写  
2. 索引优化  
3. 缩事务与控热点  
4. 读写隔离或缓存  
5. 最后才是分库分表

**易错点**  
- 一上来就加索引  
- 不区分查询慢还是等待慢  
- 不按调用频率和收益做优先级

## 18. 主从复制原理是什么？

**核心结论**  
主从复制的核心是主库把变更写入 binlog，从库按顺序拉取并重放这些变更，从而实现数据同步。

**标准答案**  
在 MySQL 主从复制体系里，主库事务提交后会把变更写入 binlog；从库通过 IO 线程或类似机制把主库的 binlog 拉到本地中继日志，再由 SQL 线程或重放线程按顺序执行，从而把主库变更同步过来。这样设计的原因是主从之间不能直接共享存储，只能通过日志复制数据变化。工程上，主从复制最大的价值是读扩展、备份容灾和故障切换，但它不是免费午餐，因为复制通常有延迟，强一致读场景必须谨慎。面试里如果能补一句“主从复制本质上是异步或半同步的日志传播过程，不是强一致镜像”，会比较完整。

**追问点答案**  
- binlog 为什么能支持复制？  
因为它记录了主库已提交的逻辑变更。  
- 主从复制一定实时吗？  
不一定，通常会有复制延迟。  
- 从库为什么不能简单理解成主库镜像？  
因为它可能落后，状态不是严格实时同步。

**原理补充**  
主从复制的本质是“日志驱动的数据同步”，它把主库提交结果传播给从库，而不是共享一套内存或磁盘状态。

**架构扩展**  
线上很多团队上读写分离之后才第一次认真面对复制延迟问题。比如订单刚写完就去从库读，读不到最新结果，业务就误以为写失败。成熟设计通常会把主从复制和读一致性策略一起讨论，而不是只说“主库写、从库读”。

**工程指标**  
- replication lag：这是最核心的从库健康指标。  
- binlog 生成速率：看主库写入压力。  
- 从库重放延迟：判断消费是否跟得上。  
- 主从差异校验结果：验证数据是否一致。  
- 读流量占比：评估主从体系带来的实际价值。

**证据链**  
- 复制模式配置  
- binlog 状态  
- 从库延迟数据  
- 主从校验结果

**低风险优化路径**  
1. 先确认复制链路稳定  
2. 再决定是否做读写分离  
3. 上线后必须同时监控延迟和读一致性异常

**易错点**  
- 把主从复制当强一致  
- 只会背主库写 binlog 从库回放  
- 忽略读一致性策略

## 19. 主从延迟怎么治理？

**核心结论**  
主从延迟治理不能只怪从库慢，而要从主库写入形态、复制链路、从库资源和业务读策略一起看。

**标准答案**  
主从延迟常见原因包括：主库大事务或高写入峰值、binlog 生成过快、从库单线程或重放能力不足、从库资源紧张、DDL / 批量任务拖慢复制等。治理时不能只盯从库，要先确认延迟是“持续性跟不上”还是“偶发性被大事务打爆”。低风险手段通常包括：拆小大事务、限制批量任务、优化 SQL、提升从库硬件和并行复制能力、区分关键读和可接受延迟的读、必要时做读路由回主。之所以要分层治理，是因为主从延迟既是数据库问题，也是业务读一致性问题。面试里如果能补一句“主从延迟的真正风险不只是数据慢一点，而是业务读到了错误时间点的数据”，会更像线上经验。

**追问点答案**  
- 为什么大事务会放大延迟？  
因为从库往往要整体跟着重放，单个大事务会卡住后面的复制进度。  
- 为什么不能简单让所有读都走从库？  
因为有些业务要求读刚写入的数据，延迟会直接造成错误判断。  
- 延迟治理和读写分离是什么关系？  
读写分离的可行性本身就建立在延迟可控之上。

**原理补充**  
主从延迟的本质是“写传播速度赶不上主库变更速度”，所以它既有链路问题，也有业务负载问题。

**架构扩展**  
线上最常见的坑是上了从库后，读多写少场景很好，但到了订单、支付、库存这种写后立刻读的链路，业务开始随机出错。更成熟的做法是分级：强一致读回主，弱一致读走从，并给复制延迟设明确阈值和熔断策略。

**工程指标**  
- replication lag：最直接的治理目标。  
- 大事务数量：延迟经常由它们触发。  
- 从库 CPU / IO：看从库是否资源不足。  
- 读回主比例：判断从库延迟是否迫使业务降级。  
- 读一致性异常数：比如写后读不到、状态错判等业务信号。

**证据链**  
- 延迟曲线  
- binlog 生成峰值  
- 大事务日志  
- 从库资源监控  
- 读一致性异常样本

**低风险优化路径**  
1. 先拆大事务和批量任务  
2. 再提升从库消费能力  
3. 最后做业务读路由和架构调整

**易错点**  
- 以为主从延迟只是 DBA 的事  
- 不把业务读一致性纳入治理  
- 只会说“加从库”不会说“怎么读”

## 20. 分库分表前必须评估哪些点？

**核心结论**  
分库分表不是性能优化起手式，而是最后阶段的大改造，必须先证明当前瓶颈真的是单库单表容量或协调成本，而不是 SQL、索引、热点和架构设计问题。

**标准答案**  
在决定分库分表之前，首先要评估当前瓶颈到底是什么：是单表数据量太大、写入 TPS 顶到上限、热点冲突严重，还是其实只是慢 SQL、索引没设计好、缓存和读写隔离还没做好。然后要评估分片键能不能选得稳定、是否会产生热点、跨分片查询和事务怎么处理、历史数据回填怎么做、如何双写或校验、切换和回滚窗口能不能控制。之所以要这么谨慎，是因为分库分表会把几乎所有简单问题都变成复杂问题：分页更难、聚合更贵、事务更弱、运维更重、迁移风险更高。工程上，成熟团队通常把它当成最后手段，而不是“数据库一慢就上分片”。面试里如果能强调“先证明值得拆，再设计怎么拆”，基本就很稳。

**追问点答案**  
- 什么情况下还不该分库分表？  
如果 SQL、索引、缓存、读写分离、冷热分层都还没做到位，通常还不到那一步。  
- 分片键为什么这么关键？  
因为它决定数据分布、热点、扩容难度和查询代价。  
- 为什么分库分表后很多问题反而更多？  
因为原本数据库帮你兜住的事务、聚合、排序、唯一约束，很多都要自己补。

**原理补充**  
分库分表本质上是在用“数据分布式化”换“单点容量提升”，但代价是把本来数据库内部解决的问题暴露到应用层。

**架构扩展**  
线上最常见的失败分片，不是技术做不到，而是拆得太早。比如热点其实只在一个状态字段、一个大事务、几个慢 SQL，上来就分片，最后业务复杂度暴涨但收益有限。成熟团队通常会先跑完低风险优化链路，再基于增长阈值、容量预测和迁移窗口来决定是否真的拆分。

**工程指标**  
- 峰值 QPS / TPS：判断单库是不是接近上限。  
- 热点表写入延迟：看瓶颈是不是集中在单表写压力。  
- rows scanned 与慢 SQL 分布：排除低层优化还没做完。  
- replication lag / 读写压力：判断读写隔离是否还够用。  
- 迁移窗口时长与回滚成功率：评估分片改造可操作性。  
- 热点分片占比预测：判断分片键是否会制造新热点。

**证据链**  
- 容量趋势  
- 热点 SQL / 表统计  
- 当前拓扑瓶颈  
- 分片键候选分布  
- 迁移和回滚方案

**低风险优化路径**  
1. SQL 改写  
2. 索引和事务优化  
3. 缓存与读写隔离  
4. 热点治理与模型优化  
5. 最后才是分库分表

**易错点**  
- 把分库分表当默认答案  
- 不评估分片键和迁移风险  
- 只会说“数据量大就拆”，不会说“为什么现在必须拆”
