## 1. JVM 运行时数据区有哪些？

**核心结论**\
回答这题要区分 JVM 规范里的运行时数据区、HotSpot 的具体实现，以及整个 Java 进程的本地内存，不能把三者画成一张图后混为一谈。

**标准答案**\
JVM 规范定义的主要区域包括程序计数器、Java 虚拟机栈、本地方法栈、堆和方法区；运行时常量池属于方法区的一部分。程序计数器、Java 栈和本地方法栈是线程私有的，堆和方法区是线程共享的。\
堆主要承载对象实例，是 GC 的主要工作区；Java 栈由栈帧组成，保存局部变量、操作数栈和返回信息；程序计数器记录当前线程的字节码执行位置；方法区保存类结构、方法数据、运行时常量池等类级信息。\
在 HotSpot 中，JDK 8 以后通常用本地内存中的 Metaspace 实现方法区的主要职责，但这不意味着所有非堆内存都属于 Metaspace。直接内存、线程栈、JIT Code Cache 和 GC 本地结构都可能增加进程 RSS，它们不是 Java Heap，也不能一概叫方法区。\
线上排障时先看错误类型和进程内存构成：`Java heap space` 关注对象及引用链，`Metaspace` 关注类加载器和动态类，`unable to create new native thread` 关注线程数与系统限制，容器被 OOM Kill 则要同时核对 Heap、Direct Memory、线程栈和容器上限。

**回答层级对照**

| 层级 | 典型回答 | 面试评价 |
| --- | --- | --- |
| 普通回答 | 背出程序计数器、栈、堆和方法区，并区分线程私有与共享。 | 基础正确，但无法解释真实进程为什么超内存。 |
| 高分回答 | 先分 JVM 规范、HotSpot 实现和 OS/容器进程三个层次，再把 Heap、Metaspace、线程栈、Direct Memory、Code Cache 与 native 结构纳入内存账本；最后根据错误类型选择 GC 日志、NMT、线程和容器指标形成证据链。 | 概念边界清晰，并能直接用于生产诊断。 |

**三级追问链**

- 一级：为什么程序计数器通常不讨论 OOM？\
  JVM 规范没有为该区域规定 `OutOfMemoryError` 场景；回答应强调规范边界，而不是仅用“它很小”推导绝对结论。
- 二级：2 GiB 容器里把 `Xmx` 设为 2 GiB，风险在哪里？\
  容器限制约束进程总占用，Heap 之外还需给 Metaspace、线程栈、Direct Memory、Code Cache、GC/native 结构和安全余量留预算。把 limit 全给 Heap 会让进程在 Heap 未满时先被 OOMKilled。
- 三级：Heap、Metaspace 和 Direct Buffer 指标合计仍小于 RSS，如何继续定位？\
  对齐线程数与 `Xss`、Code Cache、内存映射、JNI/native 库和 allocator 行为；在可接受开销下使用 NMT 看 JVM 可归类 native memory，再结合 OS 映射与 cgroup 指标。NMT 也不是完整 RSS 账本，未被 JVM 跟踪的 native 分配仍需系统侧证据。

**原理补充**\
这道题的关键不只是背区域名字，而是先声明讨论层次。JVM 规范规定语义，HotSpot 决定实现，操作系统和容器看到的是整个进程内存。很多错误答案正是把规范术语、HotSpot 内存池和 Linux RSS 混在一起。

**版本与实现边界**

- “方法区”是 JVM 规范概念，Metaspace 是 HotSpot 的实现选择。
- JIT Code Cache 是 HotSpot 的独立本地内存区域，不应直接等同于 Metaspace。
- 静态字段最终如何落到内存属于实现细节；面试时说“它是类级状态”比机械断言“静态变量都在方法区”更稳。

**架构扩展**\
在大型服务里，这几个区域的压力来源通常完全不同。比如一个典型的订单服务，在大促前为了提升吞吐把线程池开得很大，结果请求一多，单机线程数迅速上升，先出问题的往往不是堆，而是线程栈和进程总内存；再比如推荐服务或配置中心，会在应用启动后把大量热点数据预加载到本地缓存里，这时候堆很容易被顶高，GC 频率也会跟着上来；如果系统大量使用 Spring AOP、动态代理、脚本引擎或者插件机制，长期运行后元空间也可能持续增长。很多团队早期只盯着 Heap，最后发现真正把实例拖垮的是线程数、Metaspace 或进程 RSS。更合理的演进方式通常是先按服务类型拆问题：IO 型服务先控制线程模型和连接池规模，缓存型服务先限制本地缓存大小和对象体积，平台型服务则重点管理类加载器和动态类生成。

**工程指标**

- Heap / Old Gen 水位及 Full GC 后回落幅度：判断对象压力和存活对象规模。
- Metaspace Used、已加载/卸载类数量：判断类加载器或动态类是否持续增长。
- 线程数 × `-Xss`：估算线程栈的理论上限，同时结合实际 RSS 验证。
- Direct Buffer Pool、Code Cache 和进程 RSS：补齐 Heap 之外的主要内存视角。
- 容器 working set / limit：判断进程是否接近 cgroup 上限。

**微型场景**

容器限制 2 GiB，`-Xmx` 为 1 GiB，监控显示 Heap 只有 700 MiB，但进程仍被 OOM Kill。此时不能得出“堆泄漏”的结论，应继续核对 600 个线程的栈空间、Netty Direct Memory、Metaspace 和本地库占用。止血动作也不应只加大 `-Xmx`，否则可能进一步挤压非堆空间。

**评分锚点（25 分）**

| 维度 | 5 分要求 |
| --- | --- |
| 正确性 | 完整区分规范区域、HotSpot 内存池和进程总内存。 |
| 深度 | 能解释线程栈、Direct Memory、Code Cache 与 native 结构。 |
| 取舍 | 能为容器建立内存预算并保留安全余量。 |
| 表达 | 先分层，再映射故障类型和证据。 |
| 可运维性 | 能使用 GC、NMT、线程、RSS 与 cgroup 指标交叉验证。 |

**常见错误回答与扣分点**

- “非堆内存就是 Metaspace”，混淆实现区域，扣正确性。
- “`Xmx` 可以等于容器 limit”，没有进程内存账本，扣工程能力。
- 只看 Heap 判断 OOMKilled，忽略 native 与系统限制，扣诊断深度。

**复述任务**

请在 90 秒内为一个 2 GiB 容器建立 JVM 内存账本，并解释 Heap 只有 700 MiB 时进程仍可能被杀的三类原因和证据。

## 2. 堆和栈的区别？

**核心结论**\
堆主要管理对象实例，栈主要管理方法调用过程和局部变量，两者职责、生命周期和性能特征都不同。

**标准答案**\
堆和栈是 JVM 里最常被拿来对比的两个区域。堆是线程共享的，主要存放对象实例和数组，也是垃圾回收的核心区域；栈是线程私有的，方法每次调用都会创建一个栈帧，里面放局部变量、操作数栈、返回地址等信息。之所以这样设计，是因为对象通常要跨方法、跨线程被访问，而方法执行现场天然属于某个线程。工程上，堆问题更容易表现为 OOM、GC 频繁、响应变慢；栈问题更常见的是 `StackOverflowError`，比如递归太深或者调用链异常。不要把二者简单归结为“栈一定更快”：真实性能还受逃逸分析、缓存局部性、JIT 优化和访问模式影响。

**回答层级对照**

| 层级 | 典型回答 | 面试评价 |
| --- | --- | --- |
| 普通回答 | 堆存对象、线程共享；栈存局部变量、线程私有，堆由 GC 管理。 | 基础正确，但容易落入“基本类型都在栈”的口诀。 |
| 高分回答 | 从职责、生命周期、共享性和故障模式比较，并说明变量的存储位置由上下文与 JIT 优化决定；生产上把 Heap 压力与线程数 × `Xss` 的 native 预算分开治理。 | 能从语言概念推导资源模型。 |

**三级追问链**

- 一级：引用变量在栈上，引用的对象也在栈上吗？\
  不能这样推导。局部引用可作为栈帧中的变量，而对象通常按堆语义管理；字段、数组元素和被捕获变量的位置又不同，要区分“引用值”和“被引用对象”。
- 二级：逃逸分析是否意味着 HotSpot 一定把对象物理分配到栈上？\
  不一定。更常见的优化结果可能是标量替换，使对象根本不以完整形态分配；优化依赖热点编译和具体代码，不能把它当语言保证。
- 三级：把 `Xss` 从 1 MiB 降到 256 KiB 能安全支持四倍线程吗？\
  只能降低单线程理论栈预算，不能消除 PID、调度、上下文切换和 native 结构成本，还会降低深调用链余量。必须用真实调用深度、线程模型、RSS 和压测验证，优先治理无界线程而不是只压栈。

**原理补充**\
堆和栈的核心差异，本质上是“动态共享对象管理”和“线程执行上下文管理”的差异。堆解决的是对象存活和共享问题，栈解决的是调用语义和线程隔离问题。

**架构扩展**\
这个区别在真实系统里特别容易演变成两种完全不同的故障。第一种是线程模型问题，比如一个同步阻塞型接口服务，线程池、数据库连接池、下游调用都堆在一起，高峰期线程数暴涨，最后并不是业务对象太多，而是每个线程都要占一段栈内存，系统先被线程资源和上下文切换拖垮。第二种是对象模型问题，比如报表服务一次性加载大量数据到内存，或者本地缓存没有上限，结果堆被长时间占满，GC 频繁，延迟一路抖高。很多排障一开始都会被一句“内存不够”带偏，但其实栈问题更像线程和调用深度问题，堆问题更像对象生命周期和数据规模问题。更合理的做法是分开治理：前者控制线程池、降低同步阻塞、压缩调用链；后者控制对象大小、限制缓存、拆批处理和减少回源放大。

**工程指标**

- 线程总数：判断当前压力更像是线程模型失控还是对象模型失控。
- 单线程栈大小：结合线程数估算线程栈总占用，避免线程一多就把进程内存顶满。
- Heap Usage：看对象实例是否正在快速堆积，这是堆侧问题最直接的水位。
- GC Pause Time：看堆压力是否已经转化为业务可感知的停顿。
- `StackOverflowError` / `OutOfMemoryError` 次数：一个偏调用深度异常，一个偏整体内存资源耗尽，能帮助快速分型。
- 请求超时率：很多 JVM 问题最终先反映在接口超时而不是错误日志里。

**边界与取舍**

- “基本类型在栈、对象在堆”不是完整规则，变量上下文与优化结果才决定实际形态。
- `Xss` 更小可容纳更多线程，但会增加深递归或复杂调用链触发 `StackOverflowError` 的风险。
- 减少堆分配有利于 GC，但过度对象池化可能增加共享状态、锁竞争和内存常驻。

**评分锚点（25 分）**

满分回答需覆盖职责、共享性、生命周期、JIT 边界、故障分型和线程内存预算。只背“堆存对象、栈存变量”通常不超过 14 分。

**常见错误回答与扣分点**

- “栈一定比堆快”，缺少访问模式和 JIT 条件，扣准确性。
- “逃逸分析一定会栈上分配”，把优化可能性说成保证，扣实现边界。
- 线程暴涨时只调小 `Xss`，不治理线程模型和系统限制，扣工程取舍。

**复述任务**

请在 90 秒内解释基本类型和对象为何不能用一句口诀定位，并评估“调小 `Xss` 以支持更多线程”的收益、风险和验证指标。

## 3. 方法区（元空间）存什么？

**核心结论**\
方法区是 JVM 规范定义的类级数据区域；Metaspace 是 HotSpot 在 JDK 8 以后对其主要职责的实现，二者不能不加边界地当成同义词。

**标准答案**\
方法区保存类的结构信息、字段和方法数据、运行时常量池等类级信息。HotSpot 在 JDK 8 之后移除了永久代，并使用本地内存中的 Metaspace 存放类元数据。JIT 编译后的机器码主要进入独立的 Code Cache，不应笼统地算进 Metaspace；静态字段的具体存放位置也属于实现细节。\
发生 `OutOfMemoryError: Metaspace` 时，先区分“业务确实需要加载大量类”和“旧类无法卸载”。前者可能需要容量调整，后者常见于插件、脚本、热部署或自定义类加载器形成引用链。只增大 `MaxMetaspaceSize` 会推迟故障，却不会修复类加载器泄漏。

**回答层级对照**

| 层级 | 典型回答 | 面试评价 |
| --- | --- | --- |
| 普通回答 | 方法区存类信息，JDK 8 后由 Metaspace 实现，使用本地内存。 | 基础正确，但不知道容量为何增长和类如何卸载。 |
| 高分回答 | 区分规范方法区、HotSpot Metaspace 和独立 Code Cache；排障时比较类加载/卸载趋势和 ClassLoader 实例，以加载器为生命周期单位沿引用链定位，并把 Heap、Metaspace 与容器 RSS 一起预算。 | 能从定义走到类加载器泄漏治理。 |

**三级追问链**

- 一级：类加载数量持续增加就一定是泄漏吗？\
  不一定。启动、按需加载和业务扩展都可能合法增长；要看达到稳定期后是否仍只加载不卸载、Metaspace 基线是否持续抬升，以及对应 ClassLoader 是否已超出业务生命周期。
- 二级：为什么排查单位应是 ClassLoader，而不是单个 `Class`？\
  类由其定义加载器参与确定身份，同一加载器通常持有一组类及相关元数据。只要加载器仍可达，这组类往往无法卸载，因此要沿线程上下文、静态字段、缓存或框架注册表找到加载器的持有链。
- 三级：调大 `MaxMetaspaceSize` 什么时候合理，什么时候危险？\
  合法类规模确实增加且增长已稳定、容器有预算时可扩容；若类和加载器数量持续线性增长，扩容只延迟故障。任何调整都要同时设告警、增长斜率和回滚/重启保护。

**原理补充**\
类是否能够卸载，关键不只在 `Class` 对象，而在定义它的类加载器是否仍可达。同一个类加载器加载的类通常一起存活，因此排查 Metaspace 增长时要把观察单位从“某个类”提升到“类加载器及其生命周期”。

**架构扩展**\
元空间问题特别常见于平台型和框架型系统。比如一个低代码平台支持脚本热更新，或者一个中台系统支持插件动态加载，短期内功能很灵活，但如果类加载器生命周期没管理好，旧类和旧加载器就可能一直挂着，最后元空间慢慢涨满；再比如大量使用 Spring AOP、CGLIB、Groovy、Janino 这类动态生成类的系统，测试环境跑几个小时可能看不出来，生产跑几天后才开始出现 `Metaspace OOM`。这类问题的危险在于它不像普通堆问题那么直观，很多团队一开始会误判成“堆没配够”。更合理的演进通常是限制动态类生成规模、复用类加载器、减少无边界脚本执行，并给热更新和插件系统设计明确的卸载路径。

**工程指标**

- Metaspace Used / Committed：先看类元数据占用是不是持续增长，判断元空间压力的基本盘。
- 类加载总数和卸载总数：看系统是不是只加载不卸载，很多类加载器泄漏会体现在这里。
- 动态代理类数量：判断 AOP、字节码增强、脚本引擎是不是在不断制造新类。
- Full GC 与 Metaspace 回收频率：看 JVM 是否已经因为元空间压力被迫做更重的回收动作。
- 类加载器实例数量：判断插件、热部署、脚本执行链路里是否存在加载器堆积。
- 进程运行时长与元空间曲线：很多元空间问题是“随时间缓慢增长”，要结合时间维度看，而不是只看某个瞬时值。

**版本与实现边界**

- 方法区是规范概念，Metaspace 是 HotSpot 实现；其他 JVM 不必采用相同布局。
- Code Cache、Direct Memory 和线程栈不属于 Metaspace。
- 类卸载通常要求定义它的 ClassLoader 不再可达，并受收集器与 JVM 配置影响，不能靠一次 Full GC 保证发生。

**评分锚点（25 分）**

满分回答需区分概念层次、解释 ClassLoader 生命周期、给出增长趋势证据，并能判断扩容与修泄漏的边界。只说“JDK 8 后在本地内存”通常不超过 15 分。

**常见错误回答与扣分点**

- “Metaspace 默认无限，所以不会 OOM”，忽略 native 与容器上限，扣正确性。
- 只统计类数量，不统计加载器和卸载趋势，扣定位深度。
- 元空间上涨后直接扩容，没有判断增长是否收敛，扣工程取舍。

**复述任务**

请在 90 秒内回答：如何区分合法类增长和 ClassLoader 泄漏、用什么证据找到持有链、什么条件下允许调大元空间上限。

## 4. 类加载过程有哪些阶段？

**核心结论**\
类加载过程通常分为加载、验证、准备、解析、初始化，使用时还涉及卸载，真正让静态代码块执行的是初始化阶段。

**标准答案**\
一个类从字节码到可运行状态，通常会经历加载、验证、准备、解析、初始化这几个阶段。加载是把字节码读进来并生成类对象；验证是确保字节码格式和语义安全；准备阶段会给类变量分配内存并赋默认值；解析是把符号引用转换成直接引用；初始化才真正执行类构造逻辑，比如静态变量显式赋值和静态代码块。这样拆阶段的目的是让 JVM 在安全性、兼容性和执行效率之间取得平衡。面试时如果能补一句“准备阶段是默认值，初始化阶段才是程序员写的赋值逻辑”，很多人会觉得你理解得比较细。

**回答层级对照**

| 层级 | 典型回答 | 面试评价 |
| --- | --- | --- |
| 普通回答 | 背出加载、验证、准备、解析、初始化，并说准备给默认值、初始化执行静态代码。 | 基础正确，但忽略 Linking、延迟解析和初始化并发语义。 |
| 高分回答 | 将生命周期拆为 Loading、Linking（验证、准备、解析）和 Initialization，说明编译期常量、延迟解析、`<clinit>` 锁与失败状态；再把启动慢、首次请求抖动和类卸载问题映射到可观测证据。 | 既懂规范阶段，也能分析启动和热更新故障。 |

**三级追问链**

- 一级：`static int a = 1` 和 `static final int B = 2` 在准备阶段分别是什么？\
  `a` 通常先得到默认值 0，显式赋值在初始化阶段执行；带 `ConstantValue` 属性的编译期常量 `B` 可在准备阶段设为 2，并可能被调用方内联。不要把所有 `static final` 都视为编译期常量。
- 二级：两个线程同时首次使用一个类，会执行两次静态初始化吗？\
  JVM 保证类初始化同步，同一个 Class 对象的初始化逻辑只由一个线程执行，其他线程等待结果。若初始化失败，该类会进入错误状态，后续主动使用可能看到 `NoClassDefFoundError`，而首次失败常表现为 `ExceptionInInitializerError`。
- 三级：应用启动慢，怎么证明瓶颈在类加载还是 `<clinit>`/框架初始化？\
  使用类加载日志、JFR/启动剖析和阶段计时对齐“加载了多少类、从哪里加载、类初始化或 Bean 初始化耗时”；同时区分磁盘/类路径扫描、字节码验证转换和静态初始化中的外部调用，避免只凭 loaded class 数量下结论。

**原理补充**\
类加载过程的核心价值在于把“把文件读进来”和“让类真正可用”拆开。这样 JVM 才能在运行前先做安全检查，在真正使用前再执行初始化逻辑。

**架构扩展**\
类加载流程在真实系统里最典型的两个场景，一个是“启动慢”，一个是“热更新难”。比如 Spring Boot 应用在启动阶段会做大量类路径扫描、注解解析、反射注册，这时候如果依赖很多、自动配置很重，类加载和元数据解析就会变成明显耗时；另一个场景是插件化平台或 IDE 类产品，业务希望模块能动态装载和卸载，但一旦类加载器边界没设计清楚，类可能加载了却很难卸掉，最后要么内存越来越高，要么热更新不生效。更合理的演进通常是把启动期重活拆分出来，减少无意义扫描，同时给插件、脚本、热部署链路设计清晰的类加载器生命周期。

**工程指标**

- 应用启动耗时：先看类加载相关工作是否已经成为启动主瓶颈。
- 类加载数量：判断是不是引入了过多无效依赖或扫描范围过大。
- 类加载失败次数：看是否存在依赖冲突、缺类、版本错配这类问题。
- 类加载器层级数量：看容器、插件或脚本环境是否过于复杂，复杂层级往往意味着后续排障成本更高。
- 启动阶段 CPU 峰值：辅助判断启动慢是 I/O、类解析还是初始化逻辑导致。
- 热更新时间 / 卸载成功率：在插件化系统里用来判断类加载体系是否真的可演进。

**规范与实现边界**

- 验证、准备、解析合称 Linking；解析可按实现选择延迟，不必机械认为所有符号引用在初始化前一次性完成。
- “准备阶段只赋默认值”有编译期常量例外；非编译期 `static final` 仍可能在 `<clinit>` 中赋值。
- 类卸载不是生命周期中必然发生的一步，通常依赖定义加载器不可达及 JVM/收集器支持。

**评分锚点（25 分）**

满分回答需说清 Linking、常量例外、延迟解析、初始化同步与失败状态，并能给出启动诊断证据。只背五个阶段通常不超过 14 分。

**常见错误回答与扣分点**

- “所有静态变量准备阶段都只赋零值”，忽略 `ConstantValue`，扣准确性。
- “解析一定完整发生在初始化前”，把允许延迟的过程绝对化，扣规范边界。
- 启动慢只归因于“加载类太多”，不分扫描、转换和初始化耗时，扣诊断能力。

**复述任务**

请在 90 秒内解释 `static` 与编译期常量的赋值时机、并发初始化及失败后的行为，并给出启动慢的三类证据。

## 5. 双亲委派模型是什么？

**核心结论**\
双亲委派的核心不是“有两个父类”，而是类加载请求优先向上委托，只有父加载器加载不了，子加载器才自己尝试。

**标准答案**\
双亲委派模型指的是一个类加载器收到加载请求时，不会立刻自己去加载，而是先把请求往上交给父加载器，层层向上，直到启动类加载器。只有上层都找不到时，当前加载器才会自己尝试加载。这样设计最重要的原因是保证类的唯一性和核心类库安全性，比如 `java.lang.String` 这类核心类不能被业务代码随便伪造覆盖。工程上，这意味着同一个类如果由不同类加载器加载，JVM 可能会把它们视为完全不同的类型。很多插件化、容器隔离、Tomcat WebApp 隔离问题，本质上都和类加载器边界有关。

**回答层级对照**

| 层级 | 典型回答 | 面试评价 |
| --- | --- | --- |
| 普通回答 | 子加载器先委托父加载器，父加载不了才自己加载，用于避免核心类被重复加载。 | 基础正确，但不知道何时需要反向发现或子优先隔离。 |
| 高分回答 | 把它解释为 `loadClass` 的委托策略，而非类加载器继承关系；说明类身份由二进制名和定义加载器共同决定，再比较父优先、子优先与线程上下文类加载器在核心共享、应用隔离和 SPI 发现中的取舍。 | 能设计插件/容器边界，并诊断类型冲突。 |

**三级追问链**

- 一级：同名同字节码的两个类，为什么仍可能不能互相强转？\
  JVM 类型身份不仅包含二进制名，还包含定义它的 ClassLoader。由不同加载器定义的同名类属于不同类型，常见表现是看似“自己转自己”的 `ClassCastException`。
- 二级：父加载器加载的 SPI 接口如何发现子加载器路径中的实现？\
  严格父委派下父加载器看不到子级类路径，常借助线程上下文类加载器把“选择实现”的能力传给调用线程；这是一种受控的反向发现机制，使用后要恢复上下文并防止线程池持有旧加载器。
- 三级：插件平台采用 child-first，有哪些隔离与泄漏风险？\
  插件可携带独立依赖，但共享 API 若被重复加载会造成类型分裂；线程、ThreadLocal、驱动注册、日志框架和静态缓存还可能持有插件加载器导致卸载失败。设计上应将稳定 SPI/API 放父加载器，只让插件实现与私有依赖子优先，并提供显式清理生命周期。

**原理补充**\
类的身份并不只是“包名 + 类名”，而是“类加载器 + 包名 + 类名”。这也是为什么很多 `ClassCastException` 看上去类型一样，实际上还是不能转。

**架构扩展**\
双亲委派在业务代码里看着像底层细节，但在容器、插件、脚本和中间件平台里就是架构设计点。比如一个多租户插件平台，平台希望共享核心框架类，但每个租户又想带自己的依赖版本，这时候如果类加载边界没切清，最直接的后果就是依赖冲突和 `ClassCastException`；再比如 Tomcat 要让不同 WebApp 互相隔离，就不能完全照搬严格双亲委派。很多“线上明明包名类名都一样，为什么还转型失败”的问题，本质上都是类加载器隔离的副作用。更合理的做法是先明确哪些类必须全局唯一，哪些类允许租户级隔离，再去设计加载器层级。

**工程指标**

- 自定义类加载器数量：看系统是否已经进入复杂加载器治理阶段。
- 类重复加载数量：判断不同加载器是否在重复加载同一批类，容易带来冲突和内存浪费。
- 类加载失败与 `ClassNotFoundException` 次数：看依赖解析和隔离策略是否已经失效。
- `ClassCastException` 异常数：很多类加载器边界问题最终都会表现成这个异常。
- 插件加载成功率：用来验证类隔离设计是否真正可落地。
- 卸载后内存回收情况：判断类加载器是否真的被释放，没有形成泄漏。

**安全与架构边界**

- 双亲委派有助于核心类一致性，但不是 Java 安全的唯一保证；包限制、模块边界和类验证等机制也参与约束。
- 父优先适合共享核心 API，子优先适合依赖隔离；没有声明共享边界的 child-first 很容易制造 LinkageError 或类型分裂。
- 线程上下文类加载器解决可见性，也可能让线程池长期持有已卸载应用的 ClassLoader。

**评分锚点（25 分）**

| 维度 | 5 分要求 |
| --- | --- |
| 正确性 | 能说明委托流程和“类名 + 定义加载器”的类型身份。 |
| 深度 | 能解释 SPI/TCCL、父优先、子优先和卸载引用链。 |
| 取舍 | 能划分共享 API 与插件私有依赖边界。 |
| 表达 | 从目标、策略、例外、风险、验证展开。 |
| 可运维性 | 能用加载来源、加载器层级、异常和卸载指标诊断。 |

**常见错误回答与扣分点**

- “双亲”指两个父加载器，属于概念错误。
- “child-first 能彻底解决依赖冲突”，忽略共享 API 类型分裂，扣架构取舍。
- 只会说 Tomcat 打破委派，不会解释 SPI/TCCL 和卸载风险，扣深度。

**复述任务**

请在 90 秒内设计一个插件类加载边界：哪些类父优先、哪些类子优先、SPI 如何发现实现，以及卸载时必须清理哪些引用。

## 6. 什么情况下会触发类初始化？

**核心结论**\
真正触发类初始化的，是“首次主动使用”这个类，而不是单纯被加载到 JVM 里。

**标准答案**\
类初始化通常发生在第一次主动使用这个类的时候。常见触发场景包括：创建类实例、访问或赋值类的静态变量、调用类的静态方法、通过反射主动使用类、初始化子类时先初始化父类，以及 JVM 启动时执行主类。之所以这样设计，是为了延迟开销，避免所有类一启动就都执行初始化逻辑。面试里容易混淆的一点是：加载、验证、准备不等于初始化，真正执行静态代码块和程序员写的静态赋值逻辑是在初始化阶段。工程上，这会影响单例初始化时机、配置加载时机，以及某些类首次访问特别慢的问题。

**回答层级对照**

| 层级 | 典型回答 | 面试评价 |
| --- | --- | --- |
| 普通回答 | `new`、访问静态字段、调用静态方法和反射会触发初始化。 | 大方向正确，但容易把所有静态字段、反射和子类引用绝对化。 |
| 高分回答 | 以“首次主动使用声明者”为主线，区分编译期常量、父类静态字段、类字面量、数组和可控反射等被动引用；再说明初始化同步、失败状态以及重静态初始化对启动和首请求的风险。 | 语义准确，并能指导冷启动治理。 |

**三级追问链**

- 一级：通过 `Child.PARENT_FIELD` 访问父类声明的静态字段，会初始化谁？\
  通常只初始化真正声明该字段的父类，不因语法上写了 `Child` 就初始化子类。这说明判断关键是“主动使用的声明者”，而不是源码中出现了哪个类名。
- 二级：`SomeClass.class`、创建 `SomeClass[]`、读取编译期常量都会初始化类吗？\
  通常不会。类字面量和数组类型创建不执行目标类 `<clinit>`；编译期常量可能被内联。反射也要看具体 API，例如 `Class.forName` 可以选择是否初始化，不能笼统说“反射必然触发”。
- 三级：静态初始化调用远程配置，启动时卡死，如何治理？\
  先用启动阶段计时、线程栈和 JFR 找到等待在 `<clinit>` 的线程及外部依赖；止血可绕过非关键预加载或延迟启流。长期应把远程 I/O 移出静态初始化，进入有超时、重试边界、降级和健康检查的显式生命周期。

**原理补充**\
这里的关键词是“主动使用”和“被动引用”。JVM 尽量把初始化推迟到真正要执行类逻辑的时刻，来减少无效成本。

**架构扩展**\
类初始化问题最典型的线上表现就是“第一次访问特别慢”或者“应用启动时莫名卡住”。比如有些单例类在静态代码块里直接连数据库、读取远程配置、初始化线程池，开发环境看着没问题，但生产一旦依赖变慢，就会把初始化路径放大成启动事故；再比如某些工具类在首次被反射使用时触发了大量资源准备，导致首个请求延迟明显抖高。很多团队一开始会把这个问题误判成网络慢或框架慢，实际上是类初始化做了太多不该做的事。更合理的演进是让类初始化尽量轻量，把重逻辑移到显式启动流程或懒加载逻辑里。

**工程指标**

- 首次访问延迟：判断是否存在类初始化导致的冷启动抖动。
- 应用启动阶段耗时分布：看启动慢究竟卡在类初始化、Bean 创建还是外部依赖。
- 反射触发类初始化次数：判断是否有“无意中激活大量类”的问题。
- 静态初始化失败次数：看是否存在类初始化异常导致的连锁故障。
- 冷启动请求 TP99：很多初始化问题会优先拉高冷启动阶段的尾延迟。
- 依赖准备耗时：辅助区分类初始化里的本地逻辑和外部依赖调用。

**语义与工程边界**

- 初始化类前通常先初始化其父类；接口及带默认方法父接口的细节更复杂，不宜用“所有父接口都先初始化”概括。
- 初始化由 JVM 同步保护，但在 `<clinit>` 中执行阻塞 I/O 仍可能让其他线程长期等待。
- 初始化失败后不能靠下一次访问简单重试，常需修复依赖并重新加载类或重启相应加载器/进程。

**评分锚点（25 分）**

满分回答要能列出主动使用、识别三类被动引用、解释声明者与初始化同步，并给出冷启动故障治理。只背触发清单通常不超过 15 分。

**常见错误回答与扣分点**

- “源码出现类名就会初始化”，混淆加载、解析与主动使用，扣正确性。
- “所有反射都会初始化类”，忽略 API 选项，扣边界意识。
- 在静态代码块执行无超时远程 I/O，且认为 JVM 同步能保证安全，扣工程能力。

**复述任务**

请在 90 秒内比较访问父类静态字段、类字面量、数组、编译期常量和反射五种场景，并说明如何治理阻塞型 `<clinit>`。

## 7. 对象创建流程是什么？

**核心结论**\
对象创建不是一句 `new` 就结束了，背后包含类检查、内存分配、零值初始化、对象头设置和构造方法执行等步骤。

**标准答案**\
当 JVM 执行 `new` 指令时，首先会检查对应类是否已经加载、解析和初始化；如果还没有，就先完成类准备工作。然后 JVM 会在堆上为对象分配内存，分配方式可能是指针碰撞，也可能是空闲列表，取决于堆是否规整。分配完成后，会先把对象内存置为零值，再设置对象头，包括哈希、锁状态、类元信息指针等，最后才执行构造方法，把程序员定义的初始化逻辑跑完。这样设计是为了保证对象在真正暴露给程序前，至少处于一个可识别、字段有默认值的合法状态。实际工程里，如果对象创建特别频繁，就要考虑对象分配速率、逃逸分析、TLAB 和 GC 压力。

**回答层级对照**

| 层级 | 典型回答 | 面试评价 |
| --- | --- | --- |
| 普通回答 | 类检查、分配内存、零值初始化、设置对象头、执行构造方法。 | 顺序基本正确，但不知道分配并发、JIT 消除和安全发布。 |
| 高分回答 | 从类初始化检查进入 TLAB/共享分配路径，说明指针碰撞或空闲结构、零值与对象头，再解释构造器链和引用发布；最后用每请求分配、TLAB、JFR 和 GC 指标判断优化对象模型还是调整堆。 | 能把分配机制连接到吞吐与并发正确性。 |

**三级追问链**

- 一级：有 TLAB 后，所有对象分配都不需要同步吗？\
  不是。能放入当前 TLAB 的常见小对象可走快速路径；TLAB 申请、超大对象、TLAB 放不下或特定收集器路径仍可能进入共享分配与慢路径。
- 二级：源码里执行了 `new`，为什么分配指标可能没有增加？\
  热点编译器可能通过逃逸分析和标量替换消除完整对象分配；也可能因死代码消除而不存在实际工作。必须以 JIT/分配剖析证据为准，不能从源码次数直接推算堆对象数。
- 三级：构造器执行完就能保证其他线程看到完整对象吗？\
  不能只靠“构造完成”推导安全发布。若 `this` 在构造期间逸出，或引用通过没有 happens-before 的方式共享，其他线程可能观察到不安全状态。应通过 final 字段语义、锁、volatile、线程安全容器或启动阶段封闭完成发布。

**原理补充**\
对象创建流程的核心是“先保证对象合法，再执行业务初始化”。这也是为什么默认值、对象头、类元数据指针这些底层内容先于构造方法出现。

**架构扩展**\
对象创建流程在性能问题里经常不是“知识点”，而是“流量放大器”。比如网关服务在每个请求里都构造大量临时 DTO、日志对象和 JSON 中间对象，单次看不明显，但高 QPS 下对象分配速率会非常惊人，Young GC 频率也会被直接拉高；再比如推荐链路里每个请求都要新建大量短命集合和包装对象，系统吞吐上不去，根因其实不是 CPU 不够，而是对象创建太多。更合理的演进通常不是先调 GC，而是减少中间对象、复用缓冲区、控制序列化层级，先从对象模型本身减压。

**工程指标**

- 对象分配速率：看业务是不是在持续制造大量短命对象。
- Young GC 频率：判断分配压力是否已经转化为回收压力。
- TLAB 命中率：看线程本地分配是否高效，间接反映对象创建热点。
- Allocation Failure 次数：看对象分配是否已经开始遇到内存布局压力。
- 每请求对象数：比单看总量更适合判断某条链路是否设计过重。
- CPU 在 GC 上的占比：判断对象创建问题是否已经伤害吞吐。

**实现与优化边界**

- 指针碰撞、空闲列表和 TLAB 是 HotSpot 常见实现路径，实际行为取决于收集器、对象大小和 JIT。
- 零值语义是语言/JVM 可观察保证，但实现可以通过 TLAB 预清零等方式优化，并非每次 `new` 都执行一段显式清零循环。
- 减少对象分配有价值，但无界对象池可能增加常驻内存、锁竞争和生命周期错误。

| --- | --- |

| 取舍 | 能比较短命分配、复用池和常驻内存成本。 |
| 表达 | 从创建步骤讲到并发与性能结果。 |
| 可运维性 | 使用 Allocation Rate、每请求字节、TLAB、GC CPU 和 JFR 验证。 |

- “有 TLAB 就完全没有分配竞争”，忽略慢路径，扣机制深度。

## 8. 对象头里有什么？

**核心结论**\
对象头主要承载运行时管理信息，包括 Mark Word 和类型指针，数组对象还会额外记录长度。

**标准答案**\
Java 对象在堆里并不只是字段数据，前面还有对象头。对象头通常包含两块核心内容，一块是 Mark Word，用来保存哈希码、GC 年龄、锁状态等运行时信息；另一块是类型指针，用来指向对象所属类的元数据。对于数组对象，还会额外记录数组长度。这样设计的原因是 JVM 需要在不侵入业务字段的前提下，对对象做锁管理、类型识别、GC 管理等底层操作。面试里如果把对象头和锁升级、偏向锁、轻量级锁这些知识串起来，会显得理解更深入。

**回答层级对照**

| 层级 | 典型回答 | 面试评价 |
| --- | --- | --- |
| 普通回答 | 对象头包含 Mark Word、类型指针，数组还包含长度。 | 基础正确，但容易把某个 JDK 的位布局背成永久规则。 |
| 高分回答 | 先声明对象头属于 HotSpot 实现细节，再解释 Mark Word 的状态复用、Klass Pointer、数组长度、对齐和压缩指针；最后把布局成本关联到大规模缓存、锁竞争与 JDK 版本。 | 既懂布局，也有版本和容量意识。 |

**三级追问链**

- 一级：Mark Word 是否同时永久保存哈希码、年龄和所有锁信息？\
  不是简单并排字段。它会按对象状态复用有限位空间，具体编码取决于 JDK、位宽和锁实现；计算 identity hashCode 还可能影响某些锁优化路径。
- 二级：一个只有 `boolean` 字段的对象为什么远大于 1 字节？\
  还要计入对象头、字段布局和对象对齐填充；是否启用压缩类指针、对象对齐粒度和字段重排都会影响最终大小，应通过目标 JDK 的布局工具验证。
- 三级：对象头每个只多几字节，为什么架构上值得关注？\
  对数千万缓存条目，固定头部、包装对象、引用和对齐会被规模放大，可能显著抬高 Heap、GC 扫描和缓存成本。治理优先考虑减少对象数量与包装层级，再评估紧凑数据结构，而不是只压缩单个字段。

**原理补充**\
对象头是 JVM 运行时能力附着在对象上的“管理入口”。很多看上去是语言层特性，比如 `synchronized`、`hashCode()`、类型判断，底层都要借助对象头信息。

**架构扩展**\
对象头本身不是面试里最常落地的点，但在高性能场景里很实用。比如一个中间件团队在优化内存占用时，会发现真正撑大对象的可能不只是字段本身，还有对象头、对齐填充和包装层级；再比如并发热点对象经常加锁，锁状态变化和对象头直接相关，分析锁膨胀、偏向锁撤销时就绕不过这块。很多人只会把对象当“字段集合”，但做缓存、中间件、序列化优化时，对象布局本身就是成本。更合理的演进方式是少用无意义包装对象，控制对象层级，并在高并发热点上减少锁竞争。

**工程指标**

- 锁膨胀次数：看轻量级锁是否频繁升级成重量级锁。
- 对象平均大小：判断对象布局是否过重，是否存在过度包装。
- 哈希冲突与对象分布特征：辅助分析对象头里的哈希与集合行为是否匹配。
- 热点锁竞争时间：看对象头相关锁状态变化是否已经影响吞吐。
- 缓存对象数量与总大小：判断对象头和布局在大规模缓存下的放大效应。
- 序列化前后对象体积：辅助评估对象模型是不是过于臃肿。

**版本与实现边界**

- Mark Word 和 Klass Pointer 是 HotSpot 常见术语，不是 Java 语言规范承诺的固定物理布局。
- 偏向锁相关描述有明显 JDK 版本差异，不能把旧版锁状态图原样套到所有现代 JDK。
- JOL 等工具适合在目标 JDK/参数下验证布局；估算生产容量时还要加入引用图、集合桶和对齐放大。

**评分锚点（25 分）**

满分回答需覆盖对象头组成、状态复用、数组差异、压缩指针、对齐和规模化成本，并主动声明版本边界。只背“三部分”通常不超过 15 分。

**常见错误回答与扣分点**

- 背固定 64 位布局但不说明 JDK 和压缩指针参数，扣准确性。
- 认为一个 `boolean` 对象只占 1 字节，忽略头部与对齐，扣深度。
- 只优化字段位宽，不评估对象数量、引用和集合结构，扣工程能力。

## 9. 什么是逃逸分析？

**核心结论**\
逃逸分析是 JVM 判断对象是否会逃出当前作用域的一种优化基础，用来支撑栈上分配、标量替换和锁消除等优化。

**标准答案**\
逃逸分析的核心问题是：一个对象创建出来之后，会不会被方法外部、线程外部拿到。如果不会，JVM 就有机会对它做更激进的优化，比如不一定真的在堆上分配，或者把对象拆成若干标量变量来处理。这样做的目的，是减少堆分配、降低 GC 压力、减少锁开销。工程上，这个知识点最重要的不是去手动控制 JVM，而是理解“有些代码虽然写了 `new`，并不一定真的产生一个标准堆对象”。如果你的服务对象分配很频繁，这类优化可能会带来可观收益。

**回答层级对照**

| 层级 | 典型回答 | 面试评价 |
| --- | --- | --- |
| 普通回答 | JVM 判断对象是否逃出方法或线程，据此做栈上分配、标量替换和锁消除。 | 知道作用，但容易把优化能力说成执行保证。 |
| 高分回答 | 将逃逸分析视为 JIT 编译期证明基础，区分不逃逸、方法逃逸和线程逃逸，说明内联与编译边界会影响结论；优化结果更常关注分配消除、标量替换和锁消除，并用 JFR/JMH 与分配指标验证。 | 能正确面对 JIT 的条件性与不可观测性。 |

**三级追问链**

- 一级：对象作为方法返回值，就一定发生实际堆分配吗？\
  源码层面具备逃逸形式，但调用点若被内联，JIT 可能在更大编译范围重新证明对象不逃逸并消除分配。因此要区分源码边界和最终编译边界。
- 二级：开启逃逸分析后，为什么 Allocation Rate 仍很高？\
  对象可能真实逃逸、调用未内联、代码尚未热编译、存在反射/复杂控制流，或优化受 JIT 能力限制。逃逸分析不是“打开即消除所有短命对象”的开关。
- 三级：如何证明一次重构真的减少了分配，而不是被基准测试骗了？\
  使用 JMH 做充分预热并消费结果，避免常量折叠和死代码消除；对比每操作分配字节、吞吐和 GC，同时用 JFR/async-profiler 等分配剖析定位对象类型，并在接近生产的 JDK、参数和流量下复验。

**原理补充**\
逃逸分析本质上是编译器对对象作用域和可见性的静态推理。它不是单独的“内存技术”，而是一整套优化决策的前提。

**架构扩展**\
逃逸分析最典型的价值不是让我们手动“玩黑科技”，而是帮助理解为什么一些代码结构天然更容易被 JVM 优化。比如一个高频工具方法里只做局部计算、对象不返回不共享，JVM 更有机会把对象优化掉；但如果同样逻辑被设计成层层返回对象、共享到外部、再跨线程传递，优化空间就小很多。很多高吞吐链路里，减少共享可变状态、缩小对象作用域，不只是代码更清晰，也意味着 JVM 更容易做锁消除和分配优化。更合理的做法不是去赌编译器，而是写出本来就作用域明确、共享少的代码。

**工程指标**

- 对象分配速率：看逃逸分析没有帮你兜住的真实分配成本。
- GC 次数：对象优化不足时，最终常表现为回收更频繁。
- 锁消除相关 JIT 现象：看编译器是否真的做了预期优化。
- 方法热点排行：判断哪些高频调用最值得优化对象逃逸。
- CPU 使用率：对象和锁优化往往最后会体现在 CPU 成本下降。
- 每请求分配字节数：更直观地看设计是否让对象不断逃逸出去。

**优化边界与取舍**

- HotSpot 的逃逸分析和标量替换依赖 JIT、内联和具体版本，不是 Java 语义保证。
- “栈上分配”是常见教学表达，实际优化可能是对象完全被标量化，并不存在完整对象。
- 为了不逃逸而强行改 API 可能破坏可读性和封装；只有热点证据充分时才值得做针对性重构。

| 维度 | 5 分要求 |
| --- | --- |
| 正确性 | 能解释逃逸层级和三类主要优化，且不承诺必然发生。 |
| 深度 | 能说明内联、编译边界、预热和标量替换。 |
| 取舍 | 能权衡分配收益与 API、可读性及维护成本。 |
| 表达 | 按分析条件、优化结果、失效原因、验证方法展开。 |
| 可运维性 | 使用每操作/请求分配、JFR、GC 和基准数据交叉验证。 |

**常见错误回答与扣分点**

- “对象不逃逸就一定分配到栈”，把可能优化说成保证，扣正确性。
- 只看微基准耗时，不检查分配字节和死代码消除，扣验证能力。
- 为迎合 JIT 大幅破坏接口设计，却没有热点数据，扣工程取舍。

## 10. 哪些对象会进入老年代？

**核心结论**\
对象进入老年代通常和年龄、大小、分配担保失败以及空间分配策略有关，不是只有“活得久”这一种情况。

**标准答案**\
大多数对象会先进入新生代，但如果对象在多次 Young GC 后仍然存活，年龄达到阈值，就可能晋升到老年代。除此之外，大对象有时会直接进入老年代，避免在新生代频繁复制；还有一些情况下，如果 Survivor 区放不下，或者分配担保判断认为新生代回收后仍不安全，也会提前晋升。这样设计的原因是不同对象生命周期差异很大，JVM 希望把“朝生夕死”和“长时间存活”的对象分区管理，提高回收效率。工程里，缓存对象、会话对象、长生命周期集合、线程本地变量残留，都很容易把老年代顶高。

**回答层级对照**

| 层级 | 典型回答 | 面试评价 |
| --- | --- | --- |
| 普通回答 | 对象年龄达到阈值或对象太大，就会进入老年代。 | 记住了结论，但容易把某个收集器的策略说成 JVM 统一规则。 |
| 高分回答 | 先声明晋升策略取决于收集器和参数，再解释年龄阈值、Survivor 容量/动态年龄、晋升担保及大对象特殊路径；最后用分配速率、年龄分布和晋升速率判断是短期冲击还是对象生命周期设计失衡。 | 既懂机制，也知道如何用日志验证。 |

**三级追问链**

- 一级：对象年龄达到 `MaxTenuringThreshold` 才会晋升吗？\
  不一定。它是上限类参数，不是所有对象都必须熬到该年龄；Survivor 容量、年龄分布和收集器策略都可能让对象更早晋升。
- 二级：晋升速率突然升高，如何区分 Survivor 太小和业务对象活得更久？\
  对齐同流量下的年龄分布、Survivor 占用、对象分配速率和晋升对象类型。调大 Survivor 只改变容纳能力；如果业务批次、缓存或请求上下文的存活期变长，回收后的老年代基线仍会继续抬升。
- 三级：看到“大对象直接进老年代”，为什么不能立刻给统一参数方案？\
  不同收集器的大对象路径不同，例如 G1 使用 Humongous Region 管理超大对象，不能照搬传统分代收集器的结论。必须先确认 JDK、收集器、对象大小分布和日志证据，再决定拆对象、流式处理还是调整区域/堆布局。

**原理补充**\
分代回收的前提是“绝大多数对象都很快死亡”。进入老年代的对象，代表 JVM 判断它们更可能继续活很久，所以采用更不同的回收策略。

**架构扩展**\
进入老年代的问题在服务端特别典型。比如营销系统把活动配置、用户标签、本地缓存都长期挂在内存里，这些对象自然会一路晋升；又比如一个报表任务把几十万条数据放进集合中反复加工，短时间内虽然只是“执行一个任务”，但对象存活时间足够长，也会把老年代很快顶起来。很多 Full GC 问题表面看像 JVM 配置不合理，根上其实是缓存无上限、会话对象太重、批量任务不分段。更合理的演进是把对象按生命周期治理：能短命就短命，能分页就分页，能限流和上限控制就不要无限堆在内存里。

**工程指标**

- Old Gen 使用率：看长生命周期对象是否已经开始占据主导。
- 晋升速率：判断新生代对象是不是过快流入老年代。
- Survivor 使用率：看对象在年轻代的过渡是否健康，是否存在挤压。
- Full GC 次数：判断老年代压力是否已经演变成重回收问题。
- Full GC 后 Old Gen 回落幅度：看老年代里到底是暂时堆积还是异常常驻。
- 缓存对象占比：辅助确认是不是缓存和会话类对象在持续晋升。

**版本与实现边界**

- “对象都先进入 Eden”和“大对象直接进入老年代”是常见 HotSpot 分代模型的概括，不是 JVM 规范对所有收集器的硬性承诺。
- G1 的 Humongous Object、传统收集器的晋升和其他低延迟收集器的布局不能混用一套术语解释。
- 降低晋升不等于越晚晋升越好；Survivor 复制成本、Young GC 停顿和老年代压力需要一起权衡。

**评分锚点（25 分）**

满分回答应覆盖年龄与容量机制、收集器边界、日志证据、业务对象来源和治理取舍。只回答“年龄大或对象大”属于基础正确，通常不超过 15 分。

**常见错误回答与扣分点**

- 把晋升阈值说成固定次数，扣机制准确性。
- 把所有收集器的大对象处理都说成“直接进老年代”，扣版本边界。
- 只调 Survivor/年龄参数，不确认缓存、批任务和请求上下文的真实生命周期，扣工程能力。

**复述任务**

请在 90 秒内解释一次晋升速率突增：列出至少三种原因、需要看的 GC 证据，以及为什么不能直接调大 Survivor 区。

## 11. Minor GC / Full GC 区别？

**核心结论**\
Minor GC 主要回收新生代，频率高但通常较快；Full GC 通常涉及老年代甚至更大范围，停顿更重，对服务影响更明显。

**标准答案**\
Minor GC 主要发生在新生代，目标是快速回收大量短命对象；Full GC 则通常会回收老年代，并可能连带更大范围的内存区域。两者差异的根本原因是对象分布和回收成本不同，新生代对象大多朝生夕死，适合高频快回收；老年代对象存活率高，扫描和整理成本更大。实际系统里，Minor GC 频繁未必可怕，只要暂停时间稳定、吞吐可接受；真正要警惕的是 Full GC 频繁，因为它往往意味着老年代压力、内存泄漏、对象晋升异常或者参数配置不合理。面试时最好别把 GC 次数本身当问题，要把“暂停时间和业务影响”一起讲。

**回答层级对照**

| 层级 | 典型回答 | 面试评价 |
| --- | --- | --- |
| 普通回答 | Minor GC 回收新生代、快而频繁；Full GC 回收整个堆、慢而危险。 | 适合入门，但“整个堆”与“必然更慢”过度绝对。 |
| 高分回答 | 先按当前收集器和 GC 日志事件定义 Young、Mixed、Major/Full，再比较回收范围、触发原因、并发阶段和 STW 成本；最终用 Pause 分位数、吞吐和业务超时判断影响，而不是只数次数。 | 避免跨收集器混淆，并能落到 SLO。 |

**三级追问链**

- 一级：Young GC 每分钟 20 次，能直接判定异常吗？\
  不能。还要知道每次停顿、总 GC CPU 占比、分配速率、吞吐和请求延迟；高分配型服务可能频繁但稳定，低 QPS 服务同样频率则可能异常。
- 二级：G1 的 Mixed GC 是 Minor GC 还是 Full GC？\
  它会在回收年轻代 Region 的同时选择部分老年代 Region，不能简单塞进传统二分法。面试时应以日志事件、回收集合和停顿阶段解释，而不是只套名称。
- 三级：GC 次数下降一半，但 TP99 更差，可能发生了什么？\
  单次回收集合可能更大、停顿更长，或堆增大后最坏回收成本上升；也可能 GC 不是主要延迟来源。需要对齐 Pause 分布、Safepoint、CPU、在途请求和下游延迟重新归因。

**原理补充**\
GC 不只是“回收内存”，而是吞吐、延迟、内存占用三者的平衡过程。Minor GC 和 Full GC 的差异，正是这个平衡在不同代上的体现。

**架构扩展**\
Minor GC 和 Full GC 的差异，在线上最直接体现在“影响面完全不同”。比如一个网关服务，Minor GC 即使一天很多次，只要每次停顿很短，用户通常感知不强；但如果 Full GC 一次就停几百毫秒甚至更久，接口超时和线程堆积马上就会传导出来。相反，离线批处理更关注总体吞吐，对短暂停顿容忍更高。很多团队一看到 GC 次数多就紧张，其实应该先区分是哪类 GC、停顿多长、业务是否真的受影响。更合理的治理思路是让 GC 策略服务业务目标，而不是追求一个抽象的“GC 次数越少越好”。

**工程指标**

- Young GC 次数与平均耗时：判断新生代回收是否在正常高频运转。
- Full GC 次数与平均耗时：看重回收是否已经成为稳定性风险。
- GC Pause TP99：比平均值更能反映用户实际感知。
- 晋升速率：看 Minor GC 压力是否正在向老年代问题演化。
- 请求超时率：判断 GC 停顿是否已经传导到业务层。
- 吞吐变化：辅助区分是 GC 本身有问题，还是业务负载变化导致的表象。

**术语边界与取舍**

- Minor、Major、Full GC 并非所有收集器都以完全相同的方式定义，分析时以实际 GC 日志为准。
- G1 Mixed GC 涉及部分老年代 Region，但通常不等同于 Full GC。
- 优化目标不是让 GC 次数最少，而是在延迟、吞吐、CPU 与内存成本之间满足业务 SLO。

**评分锚点（25 分）**

满分回答要能识别收集器术语、解释回收范围和触发机制、量化业务影响，并给出次数下降但延迟变差的反例。绝对化地背“Minor 快、Full 慢”通常不超过 16 分。

**常见错误回答与扣分点**

- “Full GC 一定回收整个 JVM 所有内存”，忽略实现与日志语义，扣正确性。
- 只比较次数，不比较 Pause 分位数和业务超时，扣可运维性。
- 把 G1 Mixed GC 直接叫 Minor 或 Full GC，且无法解释回收集合，扣深度。

**复述任务**

请在 90 秒内回答：如何判断 Young GC 每分钟 20 次是否健康、G1 Mixed GC 应如何归类，以及你会用哪四项指标关联业务影响。

## 12. 常见垃圾回收算法有哪些？

**核心结论**\
常见 GC 算法包括标记-清除、复制、标记-整理和分代收集，不同算法本质上是在空间利用率、停顿时间和实现复杂度之间做权衡。

**标准答案**\
垃圾回收算法最常见的有几种。标记-清除先标记垃圾，再直接回收，优点是实现简单，但会产生内存碎片；复制算法把存活对象复制到另一块区域，回收效率高，适合对象存活率低的新生代；标记-整理会在标记后把存活对象往一侧压缩，减少碎片，适合老年代；分代收集则不是单一算法，而是根据对象年龄分区后组合使用这些算法。之所以会有这么多算法，是因为没有哪一种能同时在所有场景下最优。面试时最好补一句：现代收集器本质上是在这些基础算法上做组合和工程实现。

**回答层级对照**

| 层级 | 典型回答 | 面试评价 |
| --- | --- | --- |
| 普通回答 | 标记清除有碎片，复制浪费空间，标记整理没有碎片，分代组合使用。 | 概念正确，但没有解释成本由什么决定。 |
| 高分回答 | 从存活对象比例、移动成本、额外空间、碎片和停顿预算比较算法；再说明并发标记、增量回收和分代只是执行/组织策略，现代收集器会组合基础机制并用写屏障维护并发正确性。 | 能从基础算法推导现代收集器设计。 |

**三级追问链**

- 一级：复制算法的成本取决于存活对象还是垃圾对象？\
  主要取决于需要复制的存活对象数量，因此在死亡率高的区域更合适；代价是预留目标空间、更新引用和对象移动带来的停顿。
- 二级：并发标记为什么需要写屏障？\
  用户线程仍在修改对象图，收集器需要记录引用变化以维持可达性判断的正确性。写屏障会带来运行时开销，不是“并发就免费”。
- 三级：低延迟收集器为什么还要做对象转移或重定位？\
  长期不整理会积累碎片并限制分配。关键不是永不移动，而是借助 Region、转发表、着色指针或读/写屏障等机制，把更多移动工作并发化并控制暂停窗口。

**原理补充**\
GC 算法本质上是在“回收快不快”“内存浪不浪费”“停顿能不能接受”之间做平衡，不存在绝对完美方案。

**架构扩展**\
这些算法看起来像 JVM 内部实现，但其实会反过来影响架构选择。比如低延迟交易服务会更关注停顿上限，因此对能否控制整理和暂停特别敏感；而日志处理、批计算服务更看重总体吞吐，对偶发停顿容忍更高。还有一些大对象很多的系统，比如图片处理、搜索索引构建，如果内存碎片控制不好，即使总内存够也可能频繁分配失败。真正理解这些算法的价值，不是去背教科书，而是知道它们为什么会在不同业务形态里产生不同后果。

**工程指标**

- 停顿时间：判断回收算法带来的直接业务感知成本。
- 吞吐量：看系统是不是为了回收付出了过多 CPU 和时间。
- 内存碎片情况：判断标记清除类问题是否已经影响大对象分配。
- 回收前后存活对象比例：看对象生命周期分布是否符合预期。
- 大对象分配失败次数：在碎片敏感场景下非常关键。
- GC CPU 占比：判断算法成本是否已经伤害业务处理能力。

**概念边界与取舍**

- 标记-清除、复制、标记-整理描述基础回收机制；分代、并发、增量描述不同维度，不能放在同一层级互斥比较。
- “复制算法浪费一半空间”是某些经典模型的教学简化，现代实现会按 Region、Survivor 比例等方式组织空间。
- 并发降低部分 STW，不代表总 CPU 成本更低，也不代表没有屏障和重定位开销。

**评分锚点（25 分）**

满分回答应能按存活率、碎片、额外空间、停顿和并发开销比较算法，并把基础机制映射到现代收集器。只背优缺点通常不超过 17 分。

**常见错误回答与扣分点**

- 把“分代”和“标记整理”当成同一层级的互斥算法，扣概念结构。
- 认为并发回收没有 STW、没有额外 CPU 成本，扣正确性。
- 只说碎片浪费空间，不知道它会导致连续空间分配失败，扣工程深度。

**复述任务**

请在 90 秒内用“存活率、额外空间、碎片、停顿、并发成本”五个维度比较三种基础算法，并说明现代收集器如何组合它们。

## 13. CMS 和 G1 的核心差异？

**核心结论**\
CMS 更强调低停顿的并发标记清除，G1 则通过 Region 化内存和收益优先回收，在可预测停顿和大堆场景上更均衡。

**标准答案**\
CMS 的设计目标是尽量缩短老年代回收停顿时间，它通过并发标记和并发清除来减少 Stop-The-World 时间，但缺点是会产生内存碎片，而且对 CPU 资源比较敏感。G1 则把堆拆成很多 Region，不再严格区分物理上的新生代和老年代，通过维护回收收益模型来优先回收价值更高的区域。这样做的结果是，G1 在大堆场景下更容易做停顿目标控制，也更适合现代服务端应用。工程上，如果回答这题，最好不要只说“G1 更新”，而是明确 CMS 的碎片问题和 G1 的 Region + 预测停顿模型。

**回答层级对照**

| 层级 | 典型回答 | 面试评价 |
| --- | --- | --- |
| 普通回答 | CMS 低停顿但有碎片，G1 用 Region、适合大堆，所以 G1 更好。 | 知道标签，但“更好”缺少目标和版本条件。 |
| 高分回答 | CMS 以老年代并发标记清除降低停顿，风险是碎片、浮动垃圾和并发失败；G1 用 Region、并发标记、回收集合与收益预测统一管理堆，并通过 Young/Mixed 周期追求停顿目标。选型还要结合 JDK 支持状态、堆规模、CPU 余量、对象分布和 SLO，通过可比压测验证。 | 能解释设计差异、退化路径和迁移决策。 |

**三级追问链**

- 一级：G1 设置 `MaxGCPauseMillis` 后能保证不超过目标吗？\
  不能。它是软目标，G1 会据此预测和选择回收集合；对象分配突增、Humongous Object、CPU 竞争或并发周期启动过晚都可能让实际停顿超标。
- 二级：CMS 的 concurrent mode failure 意味着什么？\
  并发回收尚未完成，老年代已没有足够空间承接分配或晋升，JVM 可能退化到更重的 STW 回收。它提示启动时机、分配/晋升速率和内存余量之间失衡，而不只是“CMS 参数不对”。
- 三级：从 CMS 迁移 G1，如何证明收益不是流量波动？\
  固定 JDK、堆和业务版本，复现相同流量及对象分布；比较 Pause TP99/最大值、吞吐、GC CPU、Old Gen 基线、Humongous 分配和 Full GC 退化，再做小比例交叉灰度并保留回滚路径。

**原理补充**\
这两者差异的本质是：CMS 更像在老年代上做低停顿优化，G1 更像把整个堆管理方式重构成一种可预测、分区化的回收模型。

**架构扩展**\
CMS 和 G1 的选择，典型地取决于你的业务更怕什么。比如早期低延迟交易系统可能会偏向 CMS，看重老年代低停顿；但如果堆越来越大、对象越来越复杂，碎片和不可预测停顿会慢慢变成负担。相反，大缓存服务、搜索服务、推荐服务往往更适合 G1，因为它们更需要在大堆下维持相对稳定的停顿目标。很多团队不是一开始就选错，而是业务规模变了、堆规模变了，却还沿用旧的收集器策略。更合理的做法是把收集器选择和业务目标、堆大小、停顿预算绑在一起持续复盘。

**工程指标**

- GC Pause TP99：看收集器是否真的满足延迟预算。
- Heap 使用率：判断当前堆规模是否已经逼近收集器的舒适区。
- 碎片导致的分配失败：CMS 场景下尤其关键。
- Mixed GC / Full GC 次数：判断 G1 是否在稳定工作，还是已经频繁退化。
- CPU 在 GC 上的占比：看并发回收是否吃掉了过多业务资源。
- 大堆下停顿波动：判断可预测性是否真的比旧方案更好。

**版本与迁移边界**

- CMS 在 JDK 9 被标记废弃，并在 JDK 14 移除；新版本 HotSpot 不能把它当作可选生产方案。
- G1 自 JDK 9 起成为 HotSpot 服务端常见默认收集器，但“默认”不等于对所有延迟和堆规模都最优。
- G1 的停顿目标是调度依据，不是实时系统承诺；迁移时不能只比较平均 Pause。

**评分锚点（25 分）**

| 维度 | 5 分要求 |
| --- | --- |
| 正确性 | 能解释 CMS 与 G1 的堆布局、回收过程和退化路径。 |
| 深度 | 能说清浮动垃圾、碎片、Mixed GC、Humongous 与软停顿目标。 |
| 取舍 | 结合 JDK、堆规模、CPU、延迟和吞吐选择，而非只说新旧。 |
| 表达 | 按目标、机制、风险、迁移、验证展开。 |
| 可运维性 | 有可比压测、灰度指标、停止条件和回滚方案。 |

**常见错误回答与扣分点**

- “G1 保证停顿不超过 200 ms”，把软目标说成 SLA，严重扣正确性。
- 在 JDK 17 选型中仍建议启用 CMS，忽略移除事实，扣版本意识。
- 迁移只看平均 Pause，不看最大停顿、吞吐和 Full GC 退化，扣工程判断。

**复述任务**

请在 90 秒内完成一次 CMS 到 G1 的迁移答辩：为什么迁、担心什么、比较哪些指标、何时停止灰度并回滚。

## 14. 什么是 STW？

**核心结论**\
STW 就是 Stop-The-World，指 JVM 为了做某些关键操作，暂停所有用户线程的一段时间。

**标准答案**\
STW 指的是 JVM 在执行某些内存管理或运行时关键操作时，需要暂停所有 Java 用户线程，只保留必要的 JVM 工作线程。最典型的场景就是垃圾回收的某些阶段，比如根扫描、对象移动、重新标记等。之所以要暂停，是因为 JVM 需要拿到一个一致的对象图视图，否则对象关系在变动过程中，回收结果就可能不正确。工程上真正要关注的不是 STW 是否存在，因为几乎无法完全避免，而是停顿时间能不能控制在业务可接受范围内。面试里如果你能说出“追求的是可接受暂停，而不是神话式零暂停”，会更成熟。

**回答层级对照**

| 层级 | 典型回答 | 面试评价 |
| --- | --- | --- |
| 普通回答 | STW 是 GC 时暂停所有用户线程，会让接口延迟变高。 | 结论正确，但把 STW 等同于 GC Pause。 |
| 高分回答 | STW 是 JVM 让相关 Java 线程到达安全状态后执行一致性敏感工作的窗口；总影响既包含到达 Safepoint 的同步等待，也包含暂停中的 VM 操作。分析时要拆分 Time to Safepoint、GC/VM Operation 时间和请求排队放大，并用尾延迟而不是平均值判断业务影响。 | 能区分暂停来源，并解释故障传播。 |

**三级追问链**

- 一级：GC 日志显示 GC 工作只用了 20 ms，请求为什么可能停了 300 ms？\
  可能在进入 Safepoint 前等待了较长时间，或暂停前后发生 CPU 饥饿、队列堆积。要把 Safepoint 同步时间、VM Operation 和业务排队分别对齐，不能只看 GC 核心阶段。
- 二级：并发收集器为什么仍不能做到绝对零停顿？\
  根处理、重定位或其他需要一致状态的阶段仍可能暂停；并发阶段还依赖屏障并与业务线程争用 CPU。目标是缩短并控制暂停，不是消灭所有协调点。
- 三级：一次 300 ms STW 如何演化成数秒故障？\
  暂停期间在途请求和连接继续积压，上游超时后重试，恢复瞬间形成突发流量；线程池、连接池或下游容量被二次打满。因此治理还要包括超时预算、重试退避、限流和队列上限。

**原理补充**\
STW 的根源是“拿到一致性快照”。JVM 不是为了偷懒暂停线程，而是为了在内存关系复杂变化时仍然保证运行时正确性。

**架构扩展**\
STW 最大的工程价值在于解释“为什么平均值看起来还行，但用户还是觉得卡”。比如交易系统平时 RT 很稳，但偶发一两次 STW 就能把 TP999 拉得很难看；消息中枢和网关这种链路型服务，一次暂停可能同时拖慢大量在途请求，形成瞬时堆积。很多团队只看平均响应时间，会低估 STW 的破坏力，直到高峰期出现雪崩式超时。更合理的做法是把 STW 当成尾延迟治理的一部分，而不是只把它当 GC 的附属指标。

**工程指标**

- GC Pause TP99 / TP999：直接看 STW 对尾延迟的伤害程度。
- 最大停顿时间：判断是否存在偶发但足以打穿 SLA 的长暂停。
- 请求超时率：看 STW 是否已经影响到业务成功率。
- Safepoint 停顿时间：判断不只是 GC，连安全点同步本身是否也在拖慢系统。
- 在途请求数峰值：看一次停顿是否会放大成请求堆积。
- 重试率：有些 STW 问题最终表现成上游超时后的重试风暴。

**诊断边界与取舍**

- STW 不只来自 GC；Safepoint 相关 VM 操作、诊断命令等也可能暂停线程，具体事件以日志和 JFR 为准。
- “暂停时间”与“业务增加的延迟”不是一比一关系，排队和重试可能放大影响。
- 追求更短暂停通常要付出更多并发 CPU、内存余量或屏障成本，需要结合 SLO 权衡。

**评分锚点（25 分）**

满分回答要区分 Time to Safepoint 与 VM Operation，解释并发收集器的剩余暂停，并能从一次暂停推导请求堆积和重试放大。只回答“GC 暂停线程”通常不超过 15 分。

**常见错误回答与扣分点**

- “只有 Full GC 才 STW”，属于事实错误。
- “ZGC/G1 没有 STW”，把低停顿说成零停顿，扣正确性。
- 只看平均 GC Pause，不看最大值、Safepoint 和业务 TP99/TP999，扣可运维性。

**复述任务**

请在 90 秒内解释：20 ms GC 为什么可能对应 300 ms 请求停顿、并发收集器为何仍需协调，以及如何防止一次 STW 被重试放大。

## 15. 如何定位内存泄漏？

**核心结论**\
定位内存泄漏的核心不是只看 OOM，而是找到“对象为什么一直不能被回收”的引用链。

**标准答案**\
内存泄漏在 Java 里通常不是“内存真的丢了”，而是对象明明没用了，但仍然被引用着，导致 GC 回收不掉。定位时一般会先看现象，比如堆持续增长、Full GC 后内存回不去、服务运行时间越长越慢。然后会结合堆 dump 分析大对象、可疑集合、缓存、`ThreadLocal`、静态 Map、监听器、连接池等典型泄漏点，沿着 GC Root 引用链往上找，看看是谁把这些对象一直挂住了。真正的难点不在工具，而在判断哪些对象“应该活着”，哪些是“异常存活”。工程上，一旦确定泄漏点，除了修代码，还要补监控和上限保护，避免问题再次拖垮整个服务。

**回答层级对照**

| 层级 | 典型回答 | 面试评价 |
| --- | --- | --- |
| 普通回答 | 看堆持续上涨，导出 Heap Dump，用 MAT 找大对象。 | 方向正确，但没有证明对象为何存活，也没有线上处置方案。 |
| 高分回答 | 先用 Full GC 后基线和多实例对照确认异常，再比较两份 Dump 的对象增量，通过 Dominator Tree、Retained Heap 和 GC Roots 找到持有链；最后回到代码验证生命周期，并补上限、降级和回归监控。 | 形成“现象—对象—引用链—代码—验证”的完整证据链。 |

**三级追问链**

- 一级：Java 有 GC，为什么还会泄漏？\
  GC 只按可达性判断对象能否回收，不理解业务上的“已经无用”。对象只要仍被 GC Root 间接引用，就会继续存活。
- 二级：拿到 20 GiB Dump 后怎么缩小范围？\
  先看类直方图和增量，不按 Shallow Heap 草率下结论；再看 Dominator Tree 与 Retained Heap 找控制大量对象生命周期的支配者，最后沿到 GC Root 的最短路径确认是静态集合、线程、类加载器还是 JNI 引用在持有。
- 三级：生产实例已接近 OOM，导 Dump 可能加重停顿，怎么办？\
  先摘流量或扩容保容量，确认磁盘和实例余量后再选择副本取证；风险过高时先保留 GC 日志、类直方图、NMT/JFR 等轻量证据。取证动作要限时、可中止，不能为了 Dump 把最后一个健康实例拖死。

**原理补充**\
Java 内存泄漏本质上是“对象可达性错误”，不是“手工 malloc/free 失配”。所以排查重点永远是引用链，而不是单纯看分配量。

**架构扩展**\
内存泄漏在生产上很少只是“某个实例 OOM 了”这么简单。比如一个在线服务里 `ThreadLocal` 没清理干净，实例开始长时间持有请求上下文，先出现的是 Full GC 变慢、延迟升高，然后实例探活失败被摘流量，流量又被打到剩下实例上，形成连锁放大；再比如缓存没有上限，某次活动后热点数据不降反升，堆越来越高，最终演化成多实例重启。很多团队只在单机层面看泄漏，实际上它是可用性问题。更好的治理方式是除了修引用链，还要加容量上限、实例保护和泄漏预警。

**工程指标**

- Full GC 后堆占用：看回收后是不是还保持高水位，这是泄漏排查的第一信号。
- Old Gen 趋势：判断长生命周期对象是否在持续增长。
- Top N 大对象类型：帮助快速定位最值得怀疑的对象族群。
- 实例摘除率 / OOM 重启次数：看问题是否已经从单机上升到服务稳定性层面。
- Heap Dump 保留周期：确保问题出现时有证据，不是只靠回忆排障。
- 缓存命中率与缓存体积：辅助判断“缓存有效增长”和“缓存失控堆积”的区别。

**评分锚点（25 分）**

| 维度 | 5 分要求 |
| --- | --- |
| 正确性 | 能区分高分配、容量不足和真正的异常存活。 |
| 深度 | 能解释 Retained Heap、Dominator Tree 与 GC Root 引用链。 |
| 取舍 | 知道 Dump 的停顿、磁盘和敏感数据风险，并给出替代证据。 |
| 表达 | 按确认现象、保存证据、定位持有者、代码验证、回归观测展开。 |
| 可运维性 | 同时覆盖止血、上限保护、多实例隔离和修复后的基线验证。 |

**常见错误回答与扣分点**

- “堆上涨就是泄漏”：没有排除流量增长、预热、批任务和容量不足，扣正确性。
- “导 Dump 看最大对象”：Shallow Heap 大不等于真正持有者，扣分析深度。
- “先在所有实例导 Dump”：忽略生产取证风险和服务容量，扣工程判断。

## 16. 如何分析频繁 Full GC？

**核心结论**\
频繁 Full GC 不是一个参数问题标签，而是一个现象，要从对象生命周期、老年代压力和回收触发条件倒推原因。

**标准答案**\
分析频繁 Full GC 时，第一步不是立刻调参数，而是先确认 Full GC 为什么被触发。常见原因包括老年代空间不足、元空间不足、大对象分配失败、晋升失败、显式调用 GC，或者整体堆配置和业务负载不匹配。接着要看几个关键事实：Full GC 前后内存变化、老年代占用趋势、对象晋升速率、有没有大对象、有没有异常缓存增长。如果 Full GC 后内存明显下降，说明主要是对象太多或配置不匹配；如果回收后仍然很高，就要重点怀疑泄漏或长生命周期对象过多。工程上，真正有效的处理手段通常包括减少对象创建、优化缓存策略、拆大对象、限制批量、修复泄漏，再结合 GC 参数微调。

**回答层级对照**

| 层级 | 典型回答 | 面试评价 |
| --- | --- | --- |
| 普通回答 | 看老年代和 GC 日志，堆不够就加大内存。 | 能看到现象，但把 Full GC 默认归因于容量不足。 |
| 高分回答 | 先从 GC 日志确认收集器、事件类型和触发原因，再用回收前后水位、分配/晋升速率和停顿阶段判断是泄漏、大对象、晋升压力、元空间还是显式 GC；先做业务止血，再以单变量实验验证代码或参数改动。 | 能从触发机制定位根因，并兼顾业务恢复和变更风险。 |

**三级追问链**

- 一级：Full GC 后 Old Gen 明显回落，能排除泄漏吗？\
  不能，只能说明本次有较多对象可回收。仍要结合回落后的长期基线、增长斜率和业务周期判断；周期性大批对象也可能造成频繁停顿。
- 二级：如何从证据区分晋升压力和长期存活对象过多？\
  对齐 GC 日志中的分配速率、Survivor/年龄分布、晋升量和 Old Gen 回落基线。晋升压力更强调短时间进入老年代的速率，长期存活问题则表现为回收后基线持续抬升。
- 三级：线上 TP99 已恶化，修代码来不及，怎么止血？\
  先限流、暂停重批任务或降低无界缓存压力，必要时滚动重启并扩容承接流量；参数调整只能选有证据、可灰度、可回滚的最小变更。止血期间持续观察错误率、TP99、Old Gen 基线和实例可用数，避免重启风暴。

**原理补充**\
Full GC 是结果，不是原因。真正要找的是“为什么老年代一直被顶住”以及“为什么 JVM 认为必须做一次重回收”。

**架构扩展**\
频繁 Full GC 往往是业务设计问题在 JVM 层面的投影。比如订单批处理一次把几百万记录拉到内存里排序聚合，推荐预计算把大量候选集常驻内存，报表服务把中间结果塞进一个超大 Map，这些都会让老年代和大对象压力迅速失控。很多人一开始会先加堆、先改参数，但如果数据处理模型没变，问题只是从“10 分钟爆”变成“30 分钟爆”。更合理的演进通常是流式处理、分页处理、异步拆分、缓存上限控制，再配合参数微调，而不是反过来。

**工程指标**

- Full GC 次数：看问题是偶发还是已经进入持续恶化状态。
- Full GC 平均耗时：判断它对业务造成的实际停顿成本。
- Old Gen 使用率：看老年代是否长期处在高压区。
- 晋升速率：判断新生代对象是否正在快速把压力转移到老年代。
- Metaspace 使用率：排除是否是非堆问题伪装成 Full GC 异常。
- Full GC 后回落幅度：看是“对象多但可回收”，还是“对象根本回不掉”。

**边界与取舍**

- “Full GC”在不同收集器和日志版本中的事件语义并不完全相同，先识别 JDK、收集器和日志标签，再解释原因。
- 增大堆可能降低回收频率，也可能拉长最坏停顿和实例恢复时间；低延迟目标不能只看 GC 次数。
- 回收后水位是重要信号，但单个时间点不能证明泄漏，必须结合趋势和业务周期。

**评分锚点（25 分）**

满分回答应覆盖五点：识别触发原因、解释回收前后水位、关联对象来源、给出线上止血、通过灰度和指标验证。每缺一类关键证据或处置环节扣 3–5 分；直接给参数结论通常不超过 15 分。

**常见错误回答与扣分点**

- “Full GC 多就把 `Xmx` 调大”：没有触发原因和容量约束，扣正确性与取舍。
- “回收后下降就不是问题”：忽略频率、停顿和业务周期，扣深度。
- 只讲 GC 参数，不追踪缓存、批任务和大对象来源，扣工程能力。

**复述任务**

请用 90 秒回答：如何从一段 GC 日志确认 Full GC 类型、如何分辨泄漏与分配/晋升压力，以及生产环境先做哪三个止血动作。

[Full GC 导致 TP99 飙升：完整教学案例](/deep-dives/jvm-concurrency/case-full-gc-latency)

## 17. OOM 常见类型有哪些？

**核心结论**\
OOM 不是一个统一错误，常见类型包括堆内存溢出、元空间溢出、栈溢出相关资源耗尽、直接内存溢出等，不同类型定位路径完全不同。

**标准答案**\
最常见的 OOM 类型包括 `Java heap space`，表示堆内存不足；`Metaspace`，通常和类元数据或类加载器问题有关；`Direct buffer memory`，常见于 NIO、Netty 这类大量使用堆外内存的场景；还有和线程相关的内存不足，比如线程太多导致系统无法再创建本地线程。虽然 `StackOverflowError` 严格说不一定属于 OOM，但它也属于 JVM 内存相关问题，面试里常一起讨论。不同 OOM 的本质差异在于“哪个内存区域被耗尽了”，所以排查路径也完全不同。真正成熟的回答应该是先区分类型，再给出对应排查方向，而不是一股脑都说成堆不够。

**回答层级对照**

| 层级 | 典型回答 | 面试评价 |
| --- | --- | --- |
| 普通回答 | OOM 有 Heap、Metaspace 和 Direct Memory，分别查堆、类和堆外内存。 | 会分类，但没有区分 JVM 抛错、进程被杀和资源上限。 |
| 高分回答 | 先按错误消息和退出原因判断耗尽区域，再把 Heap、Metaspace、Direct Memory、线程栈、Code Cache 与进程 RSS 对账；同时检查容器 limit、系统线程/PID 限制和 native memory，最后为不同类型选择不同证据与止血方式。 | 从 JVM 区域提升到了进程和容器资源模型。 |

**三级追问链**

- 一级：Heap 没到 `Xmx`，Pod 为什么仍会 OOMKilled？\
  OOMKilled 是操作系统或容器层面的结果，约束的是进程总内存。Direct Memory、Metaspace、线程栈、Code Cache、JVM/native 库和页缓存相关占用都可能把 working set 推过 limit。
- 二级：`unable to create new native thread` 一定是内存不够吗？\
  不一定。还要检查线程数、`-Xss`、用户进程数、容器 PID 限制和系统资源。它表达的是 native thread 创建失败，不能直接等价为 Java Heap OOM。
- 三级：同一服务偶发 OOMKilled、没有 Java OOM 日志，如何补证据？\
  对齐容器退出码与事件、进程 RSS/working set、Heap/Non-Heap、线程数和 Direct Buffer 指标；必要时开启受控的 Native Memory Tracking 或持续 JFR。证据要在进程被杀前外送，不能依赖本地临终日志。

**原理补充**\
JVM 进程的内存远不只是 Java Heap，真正的进程内存包含堆、元空间、线程栈、本地内存、直接内存等多个部分，所以 OOM 必须分区看。

**架构扩展**\
OOM 分类题最容易拉开差距的地方，是你能不能把“不同系统为什么会撞上不同类型 OOM”讲清楚。比如网关和 Netty 服务更容易遇到 Direct Memory 问题，因为大量连接和 ByteBuf 都在堆外；插件平台和低代码平台更容易遇到 Metaspace 问题；阻塞型老系统更容易在线程暴涨后撞上 `unable to create new native thread`；而缓存型服务和报表型服务则更容易堆 OOM。很多线上事故的误区是只盯 Heap，结果明明堆还没满，进程 RSS 已经快把机器吃空了。更合理的做法是从服务类型出发，分区治理内存。

**工程指标**

- Heap / Non-Heap 使用率：先区分问题到底在堆还是非堆。
- Metaspace 使用率：判断是不是类元数据侧的内存压力。
- Direct Memory 使用率：网关、消息、NIO 服务里尤其关键。
- 线程数：看是不是线程资源耗尽型问题。
- 进程 RSS 内存：反映真实进程总占用，避免被单一 Heap 指标误导。
- 容器内存余量：在 K8s 或容器环境里，用来判断是否会先被 OOM Killer 干掉。

**边界与取舍**

- `StackOverflowError` 属于 `Error`，但不是 `OutOfMemoryError`，可以作为内存相关故障对比，不能混为同一种异常。
- 容器 OOMKilled 不是 JVM OOM 类型；没有 Java 异常并不表示没有内存压力。
- Direct Memory 上限和统计口径受 JDK、框架及分配方式影响，Netty 指标不能代表全部 native memory。

**评分锚点（25 分）**

满分回答要完成“错误类型—资源区域—关键证据—止血动作—长期治理”的映射，并能解释 Heap 正常但进程被杀的场景。只罗列异常名称最多属于基础正确，通常不超过 17 分。

**常见错误回答与扣分点**

- “OOM 就加大 `Xmx`”：可能进一步挤压非堆空间，严重扣取舍能力。
- “Heap 没满，所以不是内存问题”：混淆堆和进程总内存，扣正确性。
- 把 `StackOverflowError`、Java OOM 和 OOMKilled 当成同一类型，扣概念边界。

**复述任务**

给定“2 GiB Pod、`Xmx=1 GiB`、Heap 峰值 700 MiB、退出原因为 OOMKilled”，请在 90 秒内列出内存账本、前三项证据和第一轮止血方案。

## 18. `jstack`、`jmap`、`jstat` 分别做什么？

**核心结论**\
这三个工具分别偏线程、内存快照和 GC 统计，是 JVM 线上排障最基础的组合。

**标准答案**\
`jstack` 主要看线程栈信息，适合排查死锁、线程阻塞、CPU 高线程卡在哪里；`jmap` 更偏内存分析，可以看堆信息、导出 heap dump；`jstat` 主要看 JVM 运行时统计，尤其是 GC 相关指标，比如各代使用率和回收次数。它们之所以重要，是因为 JVM 问题通常不会只体现在一个维度上，线程问题、内存问题、GC 问题经常互相关联。实际排障里，CPU 高先看 `jstack`，GC 异常先看 `jstat`，怀疑内存泄漏就配合 `jmap`。面试时如果能把“现象 -&gt; 工具 -&gt; 定位路径”一起讲，就会很像真实做过排障的人。

**回答层级对照**

| 层级 | 典型回答 | 面试评价 |
| --- | --- | --- |
| 普通回答 | `jstack` 看线程，`jmap` 看堆，`jstat` 看 GC。 | 工具对应正确，但没有证据关联和线上风险控制。 |
| 高分回答 | 先按 CPU、线程阻塞、GC 或存活对象问题选择入口，再通过时间戳和线程 ID 关联 OS、线程栈、GC 日志与 Dump；执行前评估 Attach 权限、停顿、磁盘和敏感数据风险，并优先使用持续采集与低风险证据。 | 能形成跨层证据链，也不会让诊断动作扩大事故。 |

**三级追问链**

- 一级：CPU 高时只取一份线程栈够吗？\
  通常不够。应先从 OS 找到持续高 CPU 的线程，将十进制线程 ID 转为 Dump 中的十六进制 `nid`，再间隔采集多份栈，确认调用路径持续存在，而不是一次瞬时采样。
- 二级：`jstat` 显示老年代持续上涨，能直接证明泄漏吗？\
  不能。它只能提供容量和 GC 趋势，还需结合 Full GC 后基线、业务周期、类直方图或 Heap Dump 的对象增量和 GC Root 引用链证明异常存活。
- 三级：线上 32 GiB 堆接近 OOM，是否立即 `jmap -dump`？\
  先评估剩余实例容量、磁盘、停顿和敏感数据风险，优先摘流量或选择可牺牲副本。风险不可接受时先保存 GC 日志、连续线程栈、类直方图或 JFR；Dump 动作必须限时、可中止，并有磁盘清理和加密传输流程。

**原理补充**\
排障工具的价值不在于会背命令，而在于知道它们分别切入 JVM 的哪个视角：线程、内存对象、运行时统计。

**架构扩展**\
这几个工具的真正价值，不是你会临时敲命令，而是你能不能把它们对应的视角沉淀成长期排障能力。比如 CPU 飙高时，团队如果已经有热线程分析和线程池告警，就不必每次线上手敲 `jstack`；GC 异常时，如果平台已经自动采集 `jstat` 类指标和 GC 日志，定位会快很多；而 `jmap` 导 dump 这种高风险动作，更适合进入标准化应急流程而不是凭经验操作。更成熟的工程体系里，工具和流程是绑定的。

**工程指标**

- 线程阻塞数：对应 `jstack` 视角，判断是不是锁、IO 或下游阻塞。
- GC 次数与耗时：对应 `jstat` 视角，判断回收是否异常。
- Heap Dump 大小：对应 `jmap` 视角，评估对象问题的规模和导出成本。
- 热线程占比：判断 CPU 高是否集中在少数线程上。
- Dump 成功率与耗时：确保线上排障动作本身可控。
- GC 日志覆盖率：没有持续日志，很多 JVM 问题只能靠猜。

**工具与生产边界**

- 工具可用性、Attach 权限和命令选项随 JDK、容器及运行用户而变化，生产流程应预演而不是事故时临时试。
- Thread Dump 是时间切片，多份连续样本比单份更能证明持续热点或阻塞。
- Heap Dump 可能包含账号、Token 和业务数据，除性能风险外还要纳入数据分级、访问审计与销毁策略。

**评分锚点（25 分）**

满分回答需完成“现象—工具—跨层关联—风险控制—长期平台化”闭环，并能解释单份栈、`jstat` 趋势和 Dump 各自不能证明什么。只背工具用途通常不超过 14 分。

**常见错误回答与扣分点**

- CPU 高只截一份 `jstack` 就下结论，缺少持续性证据，扣诊断深度。
- 看到 Old Gen 上涨就断言泄漏，混淆趋势与引用链证据，扣正确性。
- 在最后一个健康实例直接导大堆 Dump，忽略容量与数据风险，严重扣工程判断。

**复述任务**

请在 90 秒内设计一次 CPU 高与内存上涨并发事故的取证顺序，说明每个工具能证明什么、不能证明什么，以及何时禁止在线导 Dump。

## 19. JVM 调优的一般步骤是什么？

**核心结论**\
JVM 调优应该是“先明确目标，再观测现状，找到瓶颈后最小化调整并验证”，而不是直接改参数碰运气。

**标准答案**\
真正靠谱的 JVM 调优，一般先要定义目标，比如是要降低 TP99 延迟、减少 Full GC、提升吞吐，还是解决 OOM。然后看现状数据，包括 GC 日志、堆使用率、老年代趋势、停顿时间、对象分配速率、线程数等，先确认问题到底在内存、对象模型还是参数本身。接下来再结合业务行为定位根因，比如缓存过大、批处理太重、对象创建过多、线程模型不合理，最后才是做参数调整，并用压测或线上观测验证结果。这个顺序很重要，因为很多“JVM 问题”最后根因其实在代码和架构。面试时如果你能明确说“调优先调业务和对象模型，再调参数”，会很加分。

**回答层级对照**

| 层级 | 典型回答 | 面试评价 |
| --- | --- | --- |
| 普通回答 | 收集 GC 日志，根据停顿和堆大小调整参数，再压测。 | 有基本流程，但目标、约束和归因仍不清楚。 |
| 高分回答 | 先把业务 SLO 转成可验证的 JVM 目标和护栏，建立同流量基线；再按 CPU、分配、晋升、存活集和停顿阶段建立假设，优先消除业务根因；最后通过单变量压测、灰度和回滚验证收益，并检查是否把成本转移到吞吐、内存或恢复时间。 | 把调优视为受约束的性能实验，而非参数经验。 |

**三级追问链**

- 一级：吞吐和低延迟目标冲突时怎么选？\
  先回到业务 SLO、资源预算和最坏延迟容忍度。不能只优化平均停顿；需要同时设 TP99、吞吐、CPU、内存余量和错误率护栏，由业务优先级决定接受哪种交换。
- 二级：压测后 GC Pause 降了，能证明调优成功吗？\
  不能。要保证流量模型、数据规模和预热条件可比，并确认吞吐、CPU、Allocation Rate、Old Gen 基线和实例成本没有恶化；否则可能只是把停顿成本转移到了别处。
- 三级：如何设计一次可上线的 JVM 调优实验？\
  明确假设与成功阈值，固定代码和流量模型，仅改变一个主变量；先离线压测，再小比例灰度，观察完整业务周期。达到停止条件立即回滚，最后记录参数、环境、结果和适用边界。

**原理补充**\
JVM 调优本质上是系统性能调优的一部分，不是独立玄学。它必须建立在可观测性和实验验证之上。

**架构扩展**\
真正成熟的 JVM 调优，不是事故来了才救火，而是在架构设计阶段就减少后续调优空间。比如缓存提前设上限、批处理提前拆段、线程池边界提前控住、对象模型尽量轻量，这样 JVM 本来就在更健康的工作区间里。相反，如果一个系统长期靠“上线后再慢慢调参数”来稳定，其实说明前面的对象生命周期设计和流量控制已经不够稳。优秀团队会把 JVM 调优和容量规划、压测、可观测性放在一起看，而不是把它当成某个资深工程师的个人手艺。

**工程指标**

- GC Pause TP99：看调优是否真正改善了用户可感知延迟。
- 吞吐量：判断优化有没有伤害整体处理能力。
- Allocation Rate：看根因是不是高分配速率而不是参数本身。
- Old Gen 水位：看长生命周期对象是否仍然过重。
- Full GC 次数：衡量调优后系统是否更稳定。
- 压测结果对比：避免把线上流量波动误判成调优成果。

**适用边界与取舍**

- 收集器选择取决于 JDK 版本、堆规模、延迟目标和 CPU 预算，不存在脱离场景的“最佳 GC”。
- 更大的堆可能换来更低频率，也可能增加最坏停顿、容器成本和故障恢复时间。
- 低分配率不是唯一目标；过度对象复用可能引入并发安全、脏数据和代码复杂度。

**评分锚点（25 分）**

| 维度 | 5 分要求 |
| --- | --- |
| 正确性 | 能从业务 SLO 推导 JVM 目标，不把 GC 次数当最终目标。 |
| 深度 | 能分析分配、晋升、存活集、停顿阶段与收集器行为。 |
| 取舍 | 同时评估延迟、吞吐、CPU、内存成本和恢复时间。 |
| 表达 | 用基线、假设、实验、灰度、结论组织回答。 |
| 可运维性 | 有成功阈值、停止条件、回滚方案和变更记录。 |

**常见错误回答与扣分点**

- “换成 G1/ZGC 就能解决”：缺少版本、堆规模和目标约束，扣正确性。
- 只展示 Pause 下降，不检查吞吐和资源成本，扣取舍能力。
- 同时改堆、收集器和业务代码后宣称参数有效，因不可归因扣实验能力。

**复述任务**

请为“TP99 从 200 ms 上升到 800 ms，GC Pause 占其中一部分”的服务设计一个 90 秒调优方案，必须包含目标、证据、实验、护栏和回滚。

## 20. 如何避免“拍脑袋调参数”？

**核心结论**\
避免拍脑袋调参数的核心是建立证据链：先观测，再假设，再实验，再验证。

**标准答案**\
所谓“拍脑袋调参数”，本质上是没有证据就修改 JVM 配置，希望碰巧改善问题。要避免这种情况，第一步是把现象量化，比如 Full GC 频繁、暂停超 500ms、OOM、吞吐下降。第二步是拿数据支撑判断，比如 GC 日志、堆 dump、监控曲线、线程和对象分布，先形成一个明确假设。第三步是做小范围实验，比如只调整一两个参数，或者先在压测环境验证，再上线观察效果。最后一定要对结果复盘，看是参数有效，还是业务流量波动导致的假象。真正成熟的工程实践里，JVM 参数应该是有基线、有说明、有回滚方案的，而不是谁感觉不对就随手一改。

**与上一题的区别**

第 19 题考查一次调优如何完成技术闭环；本题考查团队如何长期治理参数变更。高分回答不能只重复“观察—调参—验证”，还要覆盖基线模板、变更评审、灰度护栏、配置漂移和知识沉淀。

**回答层级对照**

| 层级 | 典型回答 | 面试评价 |
| --- | --- | --- |
| 普通回答 | 调参前看日志，一次少改几个参数，压测后再上线。 | 个人操作基本正确，但无法阻止团队重复犯错。 |
| 高分回答 | 按服务类型维护版本化基线和参数说明；每次变更绑定问题证据、假设、负责人、灰度指标、停止条件与回滚；平台持续检测配置漂移，并把实验结果沉淀为有适用边界的决策记录。 | 从个人经验升级为可审计、可复制的工程治理。 |

**三级追问链**

- 一级：为什么不能把一套“黄金参数”复制到所有服务？\
  服务的 JDK、容器额度、堆规模、对象生命周期、流量模型和 SLO 不同。基线只能作为起点，必须声明适用条件，不能替代验证。
- 二级：如何发现某个实例的参数已经和基线漂移？\
  将启动参数、JDK 和容器资源作为可查询配置上报，与版本化基线持续比对；发布门禁检查高风险参数，运行时看同集群实例行为是否异常分叉。
- 三级：一次灰度看起来有效，如何避免把流量波动当收益？\
  采用同时间窗口对照或交叉灰度，校准请求量、数据分布、机器规格和预热状态；覆盖完整业务周期并预先定义显著改善阈值。无法控制混杂变量时，只能记为弱证据，不能直接推广。

**原理补充**\
参数调优是控制系统行为的手段，不是替代问题分析的方法。没有观测和假设，参数本身就没有意义。

**架构扩展**\
避免拍脑袋调参数，最终不是靠某个人更谨慎，而是靠团队把方法固化下来。比如给不同业务形态建立 JVM 基线，网关类服务一套、批处理类服务一套、缓存类服务一套；每次调参都要求有问题描述、假设、压测结果、回滚方案和线上验证窗口。这样 JVM 参数不再是“网上抄一份最优解”，而是团队自己的经验资产。很多团队一出问题就改参数，下一次又忘了为什么改过，最终环境越来越不可解释，这其实比参数本身更危险。

**工程指标**

- GC 基线对比：看当前行为偏离历史正常值有多远。
- 调参前后 Pause / Throughput：验证优化是不是在目标维度上真的生效。
- OOM / Full GC 事件数：看事故级问题是否显著下降。
- 回滚成功率：确保调参失败时能快速回到安全状态。
- 压测覆盖率：没有压测验证，很多参数变化都不值得直接上线。
- 参数变更审计记录：保证每次修改都可追溯，而不是靠记忆管理 JVM。

**参数变更最小记录**

| 字段 | 必须回答的问题 |
| --- | --- |
| 问题与基线 | 哪个业务指标异常，正常区间是多少？ |
| 证据与假设 | 哪些日志或曲线支持哪个因果判断？ |
| 变更与边界 | 改什么、适用于哪些 JDK/服务、哪些明确不适用？ |
| 验证与护栏 | 成功阈值、观察周期和停止条件是什么？ |
| 回滚与负责人 | 如何恢复、由谁决策、预计多久完成？ |

**评分锚点（25 分）**

满分回答不仅有技术证据，还要覆盖版本化基线、变更审计、灰度对照、自动回滚和配置漂移检测。只说“看监控、做压测”属于个人经验层，通常不超过 17 分。

**常见错误回答与扣分点**

- “公司统一一套最优参数”：忽略服务差异和版本边界，扣取舍。
- “上线后看一天没报警就成功”：没有对照组和成功阈值，扣验证能力。
- 只记录最终参数，不记录为什么改、如何回滚，扣治理与可运维性。

**复述任务**

请在 90 秒内设计一份 JVM 参数变更单，并说明如何发现配置漂移、如何做灰度对照、什么情况下必须自动回滚。
