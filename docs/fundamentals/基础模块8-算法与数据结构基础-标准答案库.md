# 基础模块8 - 算法与数据结构基础 - 标准答案库

## 1. 时间复杂度和空间复杂度怎么分析？

**核心结论**  
时间复杂度关注输入规模变大后算法执行次数如何增长，空间复杂度关注额外占用的存储如何增长，两者本质上都是衡量算法可扩展性的方式。

**标准答案**  
分析时间复杂度，本质上是在看核心操作会随着输入规模 `n` 怎么增长，比如单层循环常见是 `O(n)`，双层嵌套循环常见是 `O(n^2)`，二分查找是 `O(log n)`。空间复杂度则是看算法除了输入本身之外，还额外申请了多少存储，比如几个变量通常算 `O(1)`，一个长度为 `n` 的辅助数组就是 `O(n)`。之所以要分析这两个维度，是因为面试和工程里都不只是关心“能不能做出来”，还关心数据规模一上来会不会直接失控。实际回答时，最好先说明主导操作，再忽略低阶项和常数项，这样会显得更专业。工程上，如果数据规模大、延迟要求严，就要优先关注时间复杂度；如果是在内存敏感场景，比如嵌入式、缓存服务或者批处理任务，空间复杂度也同样关键。

**高分回答**
先定义输入规模，不能默认所有变量都叫 n：遍历矩阵可能是 `O(rows×cols)`，图遍历是 `O(V+E)`，字符串匹配还涉及文本和模式长度。再选核心操作并统计上界：串行阶段相加取主导项，嵌套是否相乘取决于内层总执行次数；双指针即使写成 while 套 while，只要两个指针各单调移动不回退，总次数仍可能 O(n)。

说明分析口径：最坏、平均、期望、摊还或输出敏感。Hash 查找平均 O(1) 依赖散列假设，动态数组追加单次扩容 O(n) 但摊还 O(1)，枚举全部答案至少要付输出规模。空间要计辅助容器、递归栈和是否允许修改输入；工程决策再带常数、局部性、输入上限和基准，因为 `O(n log n)` 在小数据上未必胜过简单 O(n²)。

**追问点答案**  
- 为什么可以忽略常数项和低阶项？  
因为复杂度分析更关注增长趋势，而不是某个固定规模下的具体常数。  
- 递归算法怎么分析空间复杂度？  
除了看额外数据结构，还要把递归调用栈占用算进去。  
- 最坏、平均、最好复杂度都要背吗？  
不应只背一个值；至少声明本题承诺的是最坏还是期望/摊还，说明成立条件。线上容量通常更关注可控上界和高分位，而不是没有分布依据的“平均”。

**原理补充**  
复杂度分析的本质，是用输入规模去衡量算法的增长曲线，而不是只看一次运行快不快。

**架构扩展**  
真实系统里，很多性能问题本质上就是算法复杂度选错了。比如一个接口在测试环境里处理 100 条数据没问题，但线上一到 10 万条就超时，最后发现用了 `O(n^2)` 的比对逻辑。成熟团队做性能评审时，不会只看机器配置和中间件，而是会追到核心处理逻辑的复杂度是否适配目标规模。

**工程指标**  
- 单次请求处理时长：直接反映复杂度在实际流量下的影响。  
- 数据规模增长后的 TP99：看复杂度是否会随着输入放大导致尾延迟恶化。  
- 内存占用曲线：帮助识别空间复杂度过高的问题。  
- CPU 使用率：复杂度偏高时往往先体现在这里。  
- 批处理吞吐量：适合观察算法优化前后的收益。

**证据链**  
- 核心循环或递归结构  
- 输入规模统计  
- 压测或基准测试结果  
- CPU 和内存剖析数据

**低风险优化路径**  
1. 先确认主导复杂度来自哪里  
2. 再判断是否可以换数据结构或算法  
3. 最后再做并行化或工程层优化

**易错点**  
- 只背公式，不会结合代码分析  
- 把输入数据本身也算进额外空间  
- 面试里只说结论，不说明为什么

## 2. 数组和链表核心差异？

**核心结论**  
数组更适合随机访问，链表更适合频繁插入删除，差异根源在于内存布局和访问方式不同。

**标准答案**  
数组的元素在内存中通常是连续存放的，所以它支持通过下标直接定位，随机访问效率高；链表的元素则是通过指针把节点串起来，不要求连续存储，因此插入和删除更灵活。之所以会有这种差异，是因为数组一旦中间插入或删除元素，通常需要整体搬移；而链表只要改几个指针就可以完成结构调整。面试里如果只说“数组查找快，链表增删快”还不够，最好补一句：数组快在连续内存和下标寻址，链表灵活在节点重连。工程上，很多人会高估链表的实用性，因为现代 CPU 对连续内存更友好，所以在很多真实系统里，数组或动态数组结构用得比链表更多。

**高分回答**
数组以等宽元素和基址计算实现 O(1) 索引，动态数组尾部追加摊还 O(1)，中间插删 O(n) 搬移；链表随机访问 O(n)，只有已经持有目标节点/前驱引用时，插删指针才是 O(1)。因此“链表增删快”缺少定位前提，在公开 `add(index)` 场景仍先付线性遍历。

现实性能还受布局影响：数组连续、每元素元数据少，可利用 cache line、预取和批量复制；链表节点分散，指针、对象头与 allocator/GC 成本高。队列优先环形数组/ArrayDeque 常比 LinkedList 更好；若需稳定引用、O(1) 中间摘除或频繁拼接已知节点，链表才有明确优势。用实际元素大小、操作比例和 JMH/硬件计数验证，避免只按 Big-O 选型。

**追问点答案**  
- 链表查找为什么慢？  
因为它不能随机定位，只能顺着指针一个个往后走。  
- 数组扩容为什么贵？  
单次要申请更大连续区域并复制引用/元素，产生 O(n) 时间和短时双份内存；几何扩容使追加摊还 O(1)，容量可预估时应预分配但也避免过度闲置。
- 为什么工程里 `ArrayList` 比 `LinkedList` 常见？  
因为遍历和随机访问更频繁，而且连续内存对缓存友好。

**原理补充**  
这道题真正的关键，不是背增删查复杂度，而是理解数据结构的物理组织方式会直接决定操作成本。

**架构扩展**  
真实服务里，如果一个热点逻辑需要高频遍历，数组类结构往往比链表更稳定，因为 CPU 缓存命中率更好。反过来，如果你做的是一个频繁拼接、拆分节点的场景，比如某些编辑器缓冲区或特定队列实现，链表思路才更有价值。成熟工程师不会机械地按教科书选结构，而是会结合访问模式和硬件特性一起判断。

**工程指标**  
- 遍历耗时：衡量数组和链表在真实负载下的访问差异。  
- CPU cache miss：常能解释为什么链表理论复杂度不差但跑得慢。  
- 插入删除耗时：适合验证场景是否真的更适合链表。  
- GC 压力：链表节点对象多时会增加内存管理成本。  
- 内存占用：链表额外指针开销通常更大。

**证据链**  
- 数据访问模式  
- 基准测试结果  
- CPU 性能分析  
- 内存对象分布

**低风险优化路径**  
1. 先确认是读多还是改多  
2. 再看是否真的需要链表级别的结构灵活性  
3. 最后通过基准测试验证选择

**易错点**  
- 把理论复杂度直接等同于真实性能  
- 不提连续内存和缓存友好性  
- 觉得链表在工程里天然更高级

## 3. 栈和队列常见应用？

**核心结论**  
栈强调后进先出，队列强调先进先出，它们分别适合处理“最近状态回退”和“按顺序调度”的问题。

**标准答案**  
栈的特点是后进先出，所以它很适合处理递归调用、括号匹配、表达式求值、函数回退、浏览器回退这类“最近进入的状态先处理”的场景。队列则是先进先出，适合任务排队、消息消费、广度优先搜索、线程池任务调度这类“按进入顺序处理”的场景。之所以这两个结构这么基础，是因为很多复杂系统底层其实都能抽象成“状态压栈”和“任务排队”两类模型。面试里回答时，最好不要只报定义，要顺手给出一个算法场景和一个工程场景。这样既说明你会题，也说明你知道它们在真实系统里怎么用。

**高分回答**
栈/队列首先是访问约束，不是某个固定实现。栈把最近未完成状态放顶端，适合解析嵌套结构、撤销和显式模拟 DFS；队列按到达顺序推进，适合 BFS 与工作调度。实际工程队列还必须定义容量、满时行为、优先级、公平性、取消和失败重试，FIFO 只说明顺序，不提供可靠性或处理时限。

无界队列会把过载隐藏成内存增长和陈旧任务，bounded queue 才能建立反压/拒绝边界；调用栈深度也有限，递归处理攻击者可控的深树可能 StackOverflow，应用显式栈并设置节点/深度预算。需要两端操作用 deque，优先级调度用 heap，不能因名称里有“队列”就忽略语义差异。

**追问点答案**  
- 单调栈是什么？  
本质上是利用栈维护一个有序状态，常用来解决“下一个更大元素”这类问题。  
- 阻塞队列常见在哪？  
常用于生产者消费者和线程池，但选型必须带容量与满策略；阻塞本身不等于反压能传到最上游，关键线程提交时还需防止死锁或全链路停顿。
- 递归为什么和栈有关？  
因为递归调用本质上依赖函数调用栈保存现场。

**原理补充**  
栈和队列的价值，不在于它们结构多复杂，而在于它们为“访问顺序”建立了明确约束。

**架构扩展**  
很多线上问题其实可以映射到队列模型，比如任务积压、消费延迟、异步削峰；而很多算法问题可以映射到栈模型，比如状态恢复、解析器实现、表达式计算。成熟工程师会习惯先把问题抽象成访问顺序模型，这样方案会更快落下来。

**工程指标**  
- 队列长度：看任务是否积压。  
- 出队延迟：判断 FIFO 处理是否开始变慢。  
- 栈深度：递归或调用链太深时很关键。  
- 任务吞吐量：验证队列调度能力。  
- 失败重试堆积量：说明队列模型是否失衡。

**证据链**  
- 数据进入和处理顺序  
- 任务堆积日志  
- 调用栈信息  
- 队列监控面板

**低风险优化路径**  
1. 先判断问题更像状态回退还是顺序调度  
2. 再选栈或队列模型  
3. 最后补容量、限流或异常处理机制

**易错点**  
- 只会背 LIFO / FIFO  
- 不会把数据结构映射到真实问题  
- 把栈和递归、队列和调度割裂开看

## 4. 哈希表适合解决哪些问题？

**核心结论**  
哈希表适合解决需要高效查找、去重、计数、映射关系维护的问题，它的核心价值是把查找平均复杂度降到接近 `O(1)`。

**标准答案**  
哈希表最适合处理“根据 key 快速找到 value”这类问题，比如去重、词频统计、两数之和、缓存索引、对象映射、状态记录等。它之所以高效，是因为通过哈希函数把 key 映射到桶位置，大多数情况下不需要像数组遍历或树结构那样逐步比较。面试里回答时，最好强调它擅长的是“高频查找和映射”，而不是所有场景都无脑上哈希。工程上，哈希表还常作为缓存、索引、路由表、本地状态机的底层结构，但要注意冲突、扩容和内存占用带来的实际成本。

**高分回答**
哈希表适合按精确 key 做 membership、映射与聚合，平均插入/查询 O(1) 来自均匀散列和受控负载因子，不是接口保证。拉链、开放寻址等策略在冲突、删除和缓存局部性上取舍不同；恶意或低质量 hash 会把操作退化，扩容还会产生 O(n) 搬迁与内存峰值。

key 必须有稳定的相等与 hash 契约，放入后改变参与 hash 的字段会“失联”。哈希结构通常不支持范围、最值、有序迭代或前缀查询，这些更适合有序树/trie；数据很密且键域小可用数组/bitmap 节省元数据。并发场景还要选择原子复合 API，普通 HashMap 的 get-then-put 不能维护业务不变量。

**追问点答案**  
- 哈希冲突怎么处理？  
常见方法有拉链法、开放寻址法，Java `HashMap` 里还会在高冲突时树化。  
- 为什么哈希表查找不是绝对 `O(1)`？  
平均 O(1) 依赖 hash 分布和负载控制，最坏可因大量冲突退化；扩容单次也是 O(n)。回答应声明平均/期望还是最坏，并说明具体实现的冲突治理。
- 哈希表为什么常用来做去重？  
因为能快速判断某个元素是否已经出现过。

**原理补充**  
哈希表真正厉害的地方，是用空间换时间，把“比较很多次”变成“定位一次再局部处理”。

**架构扩展**  
线上很多热点组件本质上都依赖哈希结构，比如本地缓存、用户会话表、限流计数器、服务路由映射。真正的工程问题往往不在“会不会用 HashMap”，而在于高并发下怎么控冲突、控扩容、控内存。成熟团队会把哈希结构和容量规划一起看，而不是只把它当一个算法题答案。

**工程指标**  
- 哈希桶冲突率：衡量分布是否均匀。  
- 扩容次数：频繁扩容会影响性能稳定性。  
- 查询命中耗时：看哈希结构是否真的发挥价值。  
- 内存占用：哈希表常用空间换时间，需要重点关注。  
- 热 key 分布：帮助判断是否存在不均匀访问问题。

**证据链**  
- key 分布情况  
- 冲突统计  
- 扩容日志  
- 查询和更新延迟

**低风险优化路径**  
1. 先确认问题是否适合映射模型  
2. 再评估 key 分布和容量  
3. 最后再考虑并发容器或分片策略

**易错点**  
- 把哈希表当万能结构  
- 不知道冲突会影响复杂度  
- 忽略空间成本和扩容抖动

## 5. 二叉树前中后序遍历怎么写？

**核心结论**  
前中后序遍历的差异在于“访问根节点”的时机不同，写法上既可以递归，也可以用栈模拟。

**标准答案**  
二叉树前序遍历是先访问根节点，再访问左子树和右子树；中序遍历是先左再根再右；后序遍历是先左再右最后根。之所以要区分这三种顺序，是因为它们对应的使用场景不同，比如前序适合序列化结构，中序在二叉搜索树里会得到有序结果，后序适合自底向上的处理。面试里如果时间够，最好先给递归写法，因为最直观；如果面试官继续追问，再补充用栈模拟递归的迭代写法。工程上，递归版本更清晰，但树很深时要小心栈溢出风险。

**高分回答**
先用统一不变量回答：进入节点时选择在“左子树之前、左右之间、右子树之后”处理根，分别得到前、中、后序，每个节点访问一次所以时间 O(n)。递归额外空间是树高 O(h)：平衡树 O(log n)，退化链 O(n) 并可能栈溢出；迭代用显式栈仍是 O(h)，但能控制深度和暂停恢复。

实现时空节点立即返回；迭代中序保持“栈中是尚未输出的祖先”，一路压左，弹出访问后转右；后序可用 `(node, visited)` 标记或维护上次访问节点，不能只把前序结果随意翻转而忽略左右顺序。BST 中序非递减还依赖树确实满足比较器不变量，重复键策略会影响“严格递增”。测试覆盖空树、单节点、仅左/右、退化深树、重复值和结果长度等于节点数。

**追问点答案**  
- 为什么中序遍历 BST 会有序？  
因为 BST 左子树小于根，右子树大于根，中序顺序刚好符合这个结构。  
- 后序遍历适合什么问题？  
适合先处理子问题再处理父问题，比如删除树、计算高度。  
- 递归和迭代怎么选？  
已知深度受控时递归更直观；深度来自外部输入、可能退化，或需要暂停/流式产出时用显式栈，并设置深度/节点预算。

**原理补充**  
树遍历的本质，是定义“在处理一个节点时，根与左右子树的处理顺序”。

**架构扩展**  
树遍历思路在工程里其实很常见，比如解析 AST、处理组织架构、菜单树、评论树、权限树。很多业务代码复杂，不是因为树结构难，而是没先想清“我要在父节点之前处理，还是在子节点完成之后处理”。这和前中后序的思想是一致的。

**工程指标**  
- 遍历总耗时：看树处理是否随节点数线性增长。  
- 递归深度：帮助判断是否有栈溢出风险。  
- 内存峰值：迭代解法常通过显式栈换控制力。  
- 节点处理失败率：树状批处理任务里很常见。  
- 单节点平均处理耗时：定位慢节点逻辑。

**证据链**  
- 树结构样本  
- 遍历代码执行顺序  
- 递归深度统计  
- 节点处理日志

**低风险优化路径**  
1. 先明确遍历目标  
2. 再选最自然的访问顺序  
3. 树深较大时再切换迭代方案

**易错点**  
- 只会背定义，不会写代码  
- 分不清访问根节点的时机  
- 深树场景不考虑递归风险

## 6. BFS 和 DFS 的适用场景？

**核心结论**  
BFS 适合按层推进和找最短步数，DFS 适合深度探索、回溯和路径枚举，两者本质上是不同的搜索顺序。

**标准答案**  
BFS 会从起点一层一层向外扩展，所以它特别适合处理最短路径、层序遍历、最少步数这类问题；DFS 则会优先沿一条路径走到底，再回退继续搜索，适合做全路径枚举、连通性判断、回溯组合、树和图的深度处理。之所以它们适用场景不同，是因为 BFS 天然保留了“距离”概念，而 DFS 天然更适合表达“先试到底，不行再回退”的过程。面试里回答时，如果能同时说出一个树题和一个图题场景，会更完整。工程上，这种差别也很常见，比如工作流状态探索、依赖图检查、路径搜索、推荐图扩散都会用到类似思想。

**高分回答**
BFS 的队列不变量是按非递减层数出队，因此在无权或等权图中，节点第一次被发现时得到最少边数；有 0/1 权重可用 0-1 BFS，任意非负权通常用 Dijkstra，不能把 BFS 的最短路结论推广到有权图。visited 应在入队时标记，否则同一节点可能被多个父节点重复入队，空间和时间被放大。

DFS 适合连通分量、拓扑/环检测、树形 DP 与枚举，但图上必须用 visited 或三色状态区分“已完成”和“当前递归栈”，否则遇环无限递归。两者时间通常 O(V+E)，空间分别受最大 frontier 与深度影响，BFS 不一定总比 DFS 占内存。大图还需深度/节点/时间预算、双向 BFS 或启发式搜索，避免搜索空间爆炸。

**追问点答案**  
- 无权图最短路径为什么常用 BFS？  
因为 BFS 按层扩展，第一次到达目标通常就是最短步数。  
- DFS 为什么适合回溯？  
因为它天然是深入一条路径再撤回，方便恢复状态。  
- BFS 一定更耗内存吗？  
不一定。BFS 空间与最大前沿宽度相关，DFS 与最大深度及回溯状态相关；宽浅图 BFS 更大，极深图递归 DFS 也可能溢栈，应按结构上界比较。

**原理补充**  
搜索算法的核心区别，不在于会不会访问到节点，而在于“先访问谁”和“保留什么状态”。

**架构扩展**  
真实系统里，BFS 和 DFS 的思想不只出现在题目里。比如依赖分析、服务拓扑遍历、菜单权限展开更像 BFS；配置传播、规则递归处理、图谱探索更像 DFS。成熟工程师碰到图状问题时，往往先想清楚自己要的是最短层级关系，还是完整深度探索。

**工程指标**  
- 搜索节点总数：反映算法规模。  
- 内存峰值：BFS 常在这里更敏感。  
- 路径发现时延：最短路径类问题尤其关注。  
- 回溯次数：DFS 类问题的重要特征。  
- 超时率：复杂搜索空间过大时会明显升高。

**证据链**  
- 搜索队列 / 栈大小变化  
- 节点访问顺序  
- 搜索路径日志  
- 输入图规模统计

**低风险优化路径**  
1. 先确认目标是最短步数还是完整搜索  
2. 再选 BFS 或 DFS  
3. 状态空间过大时再补剪枝或去重

**易错点**  
- 不结合题目目标就直接套模板  
- 说不出 BFS 为什么适合最短路径  
- DFS 回溯时忘记恢复状态

## 7. 二分查找的边界怎么处理？

**核心结论**  
二分查找最容易出错的地方不是思路，而是边界定义、循环条件和收缩规则没有统一。

**标准答案**  
二分查找的核心是每次通过中点把搜索区间砍掉一半，但真正难点在于你要先统一“区间是闭区间还是左闭右开”，然后整个代码都按同一套规则走。比如闭区间写法通常是 `while (left <= right)`，命中后根据条件调整左右边界；左闭右开则通常是 `while (left < right)`。之所以很多人明明会二分却老写错，是因为脑子里区间模型没固定，导致 mid 处理、边界更新和最终返回值经常互相打架。面试里建议你先明确：我是找某个值、找第一个大于等于、还是找最后一个小于等于，因为不同问题返回条件不一样。工程上，二分查找最重要的是模板稳定，而不是现场硬想。

**高分回答**
先写区间不变量再写代码。找第一个 `>= target` 可用半开区间 `[lo, hi)`：循环中答案始终在区间内，若 `a[mid] >= target` 则 `hi=mid`，否则 `lo=mid+1`，终止时 `lo==hi` 即插入位置；最后再判断是否真的等于 target。每次更新必须严格缩小区间，否则在两元素区间可能死循环。

答案二分要求判定函数单调，例如容量 x 能完成，则更大容量也能完成；先证明单调方向和搜索上下界，再防止判定中的乘加溢出。`mid = lo + (hi-lo)/2` 只在 `hi-lo` 本身可表示时安全，极端整型域可用无符号位移/更宽类型。测试空、单元素、目标在两端外、重复值全等和最大整数，并用线性 oracle 随机对拍。

**追问点答案**  
- 为什么 `mid` 常写成 `left + (right - left) / 2`？  
为了避免整型溢出。  
- 查找左边界和普通查找有什么区别？  
左边界求的是满足谓词的第一个位置，命中仍保留 mid 并收右界；普通查找可返回任一匹配。把它写成 lower_bound 语义，空结果返回插入点，再单独验证相等更稳定。
- 二分只能用于有序数组吗？  
本质是用于“答案具有单调性”的问题，不限于数组。

**原理补充**  
二分查找的本质，是利用单调性把线性搜索压缩成对数级搜索。

**架构扩展**  
真实项目里，二分思想常用于阈值调优、容量上线探测、参数搜索和版本回退定位，不只是查数组。比如你要找系统在什么并发下开始超时，本质上就可以做二分实验。成熟工程师通常会把二分当成“单调性问题求边界”的通用工具。

**工程指标**  
- 搜索轮次：应接近 `log n`。  
- 边界错误回归次数：说明模板是否稳定。  
- 线上阈值定位耗时：体现二分思想的工程价值。  
- 参数搜索实验次数：优化中常用。  
- 误判率：看边界条件是否写对。

**证据链**  
- 区间定义  
- 单调性证明  
- 边界测试样例  
- 搜索轮次日志

**低风险优化路径**  
1. 先固定一种区间模板  
2. 再分别练普通查找、左边界、右边界  
3. 最后扩展到抽象单调性问题

**易错点**  
- 边界模型混用  
- 命中后直接返回导致边界题写错  
- 不验证空数组、单元素、全相等等极端样例

## 8. 双指针适合哪些题型？

**核心结论**  
双指针适合处理有序数组、区间收缩、快慢遍历和原地修改问题，它的核心是用两个位置关系替代额外嵌套遍历。

**标准答案**  
双指针常见有几类用法：一类是左右指针从两端逼近，常见于有序数组求和、回文判断、容器盛水这类区间收缩问题；一类是快慢指针，常见于链表找环、找中点、删除重复元素这类不同速度遍历问题；还有一类是同向双指针，常用于数组去重、原地覆盖、区间维护。之所以双指针这么高频，是因为它常常能把两层循环压成一层，从而把复杂度从 `O(n^2)` 降到 `O(n)`。面试里回答时，最好把“指针的语义”说清楚，比如左指针代表已处理边界，右指针代表待探索区域。这样会比只背模板更像真正理解了方法。

**高分回答**
双指针能线性化搜索的前提是每次移动都能证明一批候选永久不可能。两端求有序两数和时，和太小移动左端，因为固定左端与更小右端只会更小；没有有序/单调性就不能安全丢弃。快慢压缩则保持 `[0, slow)` 已是满足条件的输出，fast 扫描未处理输入，写入时需明确能否覆盖尚未读取的数据。

复杂度 O(n) 不是因“有两个指针”，而是每个指针单调且总移动次数受 n 限制。链表环检测的相遇证明来自模环长的相对速度；找入口需在相遇后让一指针回头并同速走。实现测试交叉/相遇、全重复、无解、空/单元素，以及原地结果的有效长度而非数组尾部残留。

**追问点答案**  
- 双指针为什么常和有序数组一起出现？  
因为有序性可以让你根据大小关系决定移动方向。  
- 快慢指针为什么能找环？  
因为如果有环，快指针最终会追上慢指针。  
- 原地去重为什么适合同向双指针？  
慢指针定义已写有效区边界，快指针只读尚未处理元素；只有证明写位置不越过读位置且覆盖不影响未来判断时才安全，并需返回有效长度。

**原理补充**  
双指针方法的关键，是把“两个维度的搜索”转化成“受控移动的两个状态”。

**架构扩展**  
工程里很多流式处理、本地批量清洗、日志窗口扫描，底层也会用到类似双指针思想。比如一个指针代表当前处理位置，另一个代表边界或回收位置，本质上就是用状态推进替代重复扫描。成熟工程师会把双指针看成一种数据流处理模型，而不只是 LeetCode 技巧。

**工程指标**  
- 单次扫描耗时：双指针优化后通常更稳定。  
- 内存占用：原地处理问题尤其关注。  
- 数据复制次数：反映是否真正做到原地修改。  
- 大输入下吞吐量：验证从双循环到单循环的收益。  
- 边界错误率：双指针题常见 bug 指标。

**证据链**  
- 指针移动规则  
- 输入是否有序  
- 覆盖写入逻辑  
- 边界样例执行结果

**低风险优化路径**  
1. 先明确两个指针各自语义  
2. 再写移动条件  
3. 最后补极端边界测试

**易错点**  
- 不清楚指针代表什么  
- 指针移动顺序写乱  
- 没有验证空数组和重复元素场景

## 9. 滑动窗口适合哪些题型？

**核心结论**  
滑动窗口适合处理连续区间问题，核心思想是用左右边界动态维护一个满足条件的子区间。

**标准答案**  
滑动窗口最适合处理“连续子数组”或“连续子串”这类题，比如最长不重复子串、最小覆盖子串、长度固定窗口最大和、满足某个约束的最长区间。之所以这类题适合滑动窗口，是因为题目关注的是连续范围，而不是任意组合，所以可以用两个边界把候选区间维护起来。面试里回答时，最好先说清窗口里维护的状态是什么，比如字符计数、元素和、种类数、合法性条件，然后再说什么时候扩、什么时候缩。工程上，这种思想也很像流式数据处理和实时统计，很多在线监控、限流、聚合分析都能看到窗口模型。

**高分回答**
可变窗口成立需要“右移加入元素后，若条件不合法，通过左移可以单调恢复；合法后无需回头考虑被丢前缀”。最长不重复子串满足这一点；但“和至少 K”在含负数数组中不满足，因为移除负数可能增大和、加入负数可能减小和，普通窗口会漏解，需要前缀和+单调队列等方法。

模板必须保持状态更新对称：扩右先加入计数，非法时循环移左并删除，合法时按最长/最短目标更新答案；最小窗口通常在合法期间尽量缩，最长窗口通常在修复后更新。固定窗口可在进入新元素同时移出旧元素。工程时间窗口还要区分处理时间/事件时间、乱序与水位线，不能把数组下标窗口直接等同于实时统计语义。

**追问点答案**  
- 固定窗口和可变窗口差别是什么？  
固定窗口长度不变，可变窗口根据条件动态扩缩。  
- 为什么滑动窗口通常是 `O(n)`？  
在适用单调条件下左右边界各最多前进 n 次，内层 while 的总次数也是 O(n)，不是每次外循环都重新跑 n 次；状态更新本身还应为 O(1) 或计入其成本。
- 最小覆盖子串难点在哪？  
难在窗口合法性判断和缩窗时状态维护。

**原理补充**  
滑动窗口的本质，是把大量重复计算的区间信息维护成可增量更新的状态。

**架构扩展**  
真实系统里，时间窗口、流量窗口、错误率窗口、监控聚合窗口都和这套思想高度相似。比如限流算法里的滑动窗口计数、本地日志异常检测、实时指标统计，底层都在做“区间进入一个元素、移出一个元素”的状态更新。会做题的人很多，能把窗口思想迁移到工程问题的人更少，这正是加分点。

**工程指标**  
- 窗口更新耗时：反映状态维护是否高效。  
- 内存占用：大窗口或复杂状态时很重要。  
- 状态修正次数：衡量窗口合法性调整成本。  
- 实时吞吐量：窗口算法常用于流式场景。  
- 误报或漏报率：工程里的窗口统计很关注这个。

**证据链**  
- 左右边界移动记录  
- 窗口状态数据  
- 输入连续性要求  
- 关键样例回放

**低风险优化路径**  
1. 先确认题目要求连续区间  
2. 再确定窗口维护的状态  
3. 最后设计扩窗和缩窗条件

**易错点**  
- 把非连续问题硬套窗口  
- 状态更新不对称  
- 不会解释为什么能做到线性复杂度

## 10. 单调栈典型场景是什么？

**核心结论**  
单调栈适合解决“下一个更大/更小元素”和“局部最值影响范围”这类问题，核心是维护一个有序候选集。

**标准答案**  
单调栈通常用来处理每个元素左边或右边第一个比它大、比它小的元素，或者求某个元素作为最小值/最大值时能影响的区间范围，比如柱状图最大矩形、每日温度、接雨水、下一个更大元素。之所以它有效，是因为栈里保持了一个单调有序的状态，当新元素进来时，可以一次性淘汰一批不再可能成为答案的元素，从而避免重复比较。面试里如果能说明“被弹出的元素，其答案就在当前时刻确定了”，通常就说明你真正理解了它。工程上，这类思想也适合处理流式峰值检测、价格波动边界、指标拐点分析等问题。

**高分回答**
栈中通常存索引而非值，因为答案需要距离/边界且重复值必须区分位置。以“右侧第一个更大”为例维护值单调递减的未决索引；新值大于栈顶时，当前索引就是被弹元素第一个更大位置，持续弹出后再压入自己。每个索引入栈、出栈至多一次，时间 O(n)，栈最坏 O(n)。

真正难点是严格与非严格比较。柱状图最大矩形需为每根柱找第一个严格更矮边界，重复高度采用统一一侧弹等号策略，否则宽度重复/遗漏；首尾加哨兵可统一清栈。接雨水等题还需在弹栈时确认左右边界与高度差。验证全升、全降、全等、谷峰、空输入，并用 O(n²) 暴力随机对拍。

**追问点答案**  
- 为什么叫单调栈？  
因为栈内元素始终保持单调递增或递减。  
- 和普通栈最大区别是什么？  
普通栈只关心后进先出，单调栈还对元素顺序有约束。  
- 为什么能做到线性时间？  
虽然有嵌套 while，但每个索引只入栈一次并最多被弹一次，总栈操作不超过线性；若弹出后又反复重新入栈，则需重新分析。

**原理补充**  
单调栈的核心不是“用栈”，而是“把未来还可能有用的候选保留下来，把已经不可能有用的状态立即淘汰”。

**架构扩展**  
真实业务里，很多告警边界、趋势峰谷、价格区间和资源水位分析，也可以借用单调结构思想。虽然不会直接写成“单调栈题”，但思维模型是一致的：只保留对未来决策还有价值的状态，尽快剔除无效历史。

**工程指标**  
- 状态处理总耗时：应接近线性。  
- 入栈出栈次数：验证算法是否正确收敛。  
- 峰值检测延迟：工程映射里很关键。  
- 内存峰值：观察栈大小是否异常。  
- 结果误差率：用于校验区间边界是否求对。

**证据链**  
- 栈状态变化过程  
- 元素进出栈记录  
- 关键样例推演  
- 结果与暴力解对比

**低风险优化路径**  
1. 先判断题目是否在找“最近更大/更小”  
2. 再决定维护递增还是递减栈  
3. 最后用样例验证弹栈时机

**易错点**  
- 不清楚栈里该放索引还是值  
- 单调方向选错  
- 弹栈后不会处理答案归属

## 11. 堆（优先队列）适合哪些 TopK 问题？

**核心结论**  
堆特别适合处理动态 TopK、最值维护和实时优先级调度问题，因为它能在较低成本下持续维护局部最优集合。

**标准答案**  
如果题目要求找前 K 大、前 K 小、数据流中位数、实时排行榜、定时任务优先执行，堆通常是非常合适的结构。之所以它适合，是因为堆可以在插入和弹出时保持根节点始终是当前最值，从而避免每次都全量排序。面试里回答 TopK 问题时，最好能说出：如果 `k` 远小于 `n`，用大小为 `k` 的堆通常比全量排序更划算。工程上，优先队列还常用于调度系统、定时任务、限流排队、资源抢占等场景，本质上都是“局部保持最重要的一批数据”。

**高分回答**
找前 K 大可维护容量 k 的小顶堆：堆顶是当前入选集合最小值，新元素更大时替换，扫描 n 个元素时间 O(n log k)、空间 O(k)，最终堆内无完整顺序，若要有序输出还需 O(k log k)。若离线只求第 k 大，quickselect 期望 O(n) 可能更合适；k 接近 n 或需要全序时直接排序更简单；已知小整数键域还能用计数法。

堆只保证根最优，不保证同优先级稳定顺序，也不擅长 O(1) 删除/修改任意元素。调度系统常用 `(priority, sequence)` 明确平局规则，取消/更新可配索引表或 lazy deletion，但陈旧项会增加内存与弹出成本。动态 TopK 还要定义窗口、并发一致性与分片合并，不能把单机 heap 直接当全局实时排行榜。

**追问点答案**  
- 为什么 TopK 常用小顶堆？  
找前 K 大时，小顶堆方便把当前第 K 大放在堆顶，用来淘汰更小元素。  
- 什么时候全量排序更合适？  
需要完整顺序、k 接近 n、数据已近有序或代码简单性更重要时；选择应比较 `n log k + k log k`、排序成本与实际库实现，而非固定套堆。
- 堆和有序数组比谁更适合动态场景？  
堆更适合，因为插入删除成本更平衡。

**原理补充**  
堆的价值不在于“排序”，而在于“用局部有序换取持续维护最值的能力”。

**架构扩展**  
在线服务里，TopK 和优先级问题非常常见，比如热门内容排行、最慢接口榜单、最大流量用户集合、调度队列。成熟工程师会先判断是不是需要“动态维护最值集合”，如果是，堆往往比排序和遍历都更自然。

**工程指标**  
- 插入和弹出耗时：验证堆维护成本。  
- TopK 刷新延迟：排行类系统常关注。  
- 堆大小波动：判断容量是否合理。  
- 内存占用：大规模流式场景要注意。  
- 结果准确率：近似 TopK 场景尤其关键。

**证据链**  
- `k` 与 `n` 的规模关系  
- 更新频率  
- 堆状态快照  
- 全量排序对比结果

**低风险优化路径**  
1. 先确认是否是动态最值问题  
2. 再根据目标选小顶堆或大顶堆  
3. 最后评估是否需要补最终排序

**易错点**  
- 不知道为什么找前 K 大要用小顶堆  
- 把堆和完整排序混为一谈  
- 忽略动态更新场景的优势

## 12. 并查集解决什么问题？

**核心结论**  
并查集适合解决连通性判断、集合合并和分组归属这类问题，核心是高效管理“哪些元素属于同一个集合”。

**标准答案**  
并查集最适合处理动态连通性问题，比如朋友圈数量、岛屿合并、图中是否连通、账号归并、网络节点归属等。它之所以高效，是因为通过父节点数组表示集合关系，再通过路径压缩和按秩合并，把多次查找和合并的成本降得很低。面试里如果你能说清楚 `find` 是查根节点、`union` 是合并两个集合，再补一句路径压缩为什么能优化，就已经比较完整了。工程上，并查集虽然不像哈希表那么常见，但在批量归并、图结构聚合、用户画像去重、资源分组场景里非常有价值。

**高分回答**
并查集维护集合分区：`parent[root]=root`，find 返回代表元，union 只在两个根不同时连接并减少连通分量计数。路径压缩把查找路径指向更高祖先，按 size/rank 让小树挂大树；组合后 m 次操作的摊还复杂度为 `O(m α(n))`，近似常数但不是严格 O(1)。

它擅长边逐步加入后的连通性，不保存实际路径，也不直接支持删除边、最短路或有向图环检测。若需要离线删除可反向处理或使用 rollback DSU/动态连通结构；账号合并还要先用哈希把业务标识映射成整数，并定义冲突数据的权威规则。测试重复 union、自环、孤立点、链式合并和分量计数。

**追问点答案**  
- 路径压缩的作用是什么？  
让查找时经过的节点直接挂到根上，后续查找更快。  
- 按秩合并是什么？  
尽量把小树挂到大树上，避免树太高。  
- 并查集能判断有环吗？  
在无向图逐边加入时，边两端已有同一根说明该边形成环；有向图不能直接套此结论，需要 DFS 三色或拓扑等方法。

**原理补充**  
并查集的本质，是把“集合归属”抽象成树结构，并不断压缩这棵树，让查询越来越快。

**架构扩展**  
在实际系统里，归并思路很常见，比如多个账号合并成一个用户、多个资源聚类到一个组、多个节点组成一个服务集群。虽然未必直接写成并查集代码，但“快速判断是否同属一个集合”的需求非常真实。理解并查集，有助于你在建模时更快看出这类问题。

**工程指标**  
- 合并耗时：观察批量归并性能。  
- 查找耗时：路径压缩收益会体现在这里。  
- 集合数量变化：分析归并结果。  
- 树高分布：辅助验证优化是否生效。  
- 批任务总耗时：并查集常出现在离线处理链路里。

**证据链**  
- 父节点数组  
- 合并记录  
- 连通分量统计  
- 性能对比样本

**低风险优化路径**  
1. 先判断问题是不是“同组归属”  
2. 再实现标准 `find/union`  
3. 最后加路径压缩和按秩合并

**易错点**  
- 不理解根节点代表集合  
- 写了并查集却没做压缩  
- 看不出题目本质是连通性问题

## 13. 回溯算法的通用模板是什么？

**核心结论**  
回溯的核心是“做选择、递归进入、撤销选择”，它适合求所有可能解、组合、排列和约束搜索问题。

**标准答案**  
回溯算法通常有一个很稳定的模板：先判断当前状态是否满足结束条件；如果没结束，就在当前层枚举所有可能选择；每做一个选择，就把状态更新后递归进入下一层；递归返回后，再撤销这个选择，恢复现场。之所以回溯高频，是因为很多题目要求的是“找出所有可能答案”或者“在一堆约束里试出可行解”，这类问题天然就适合试探和回退。面试里如果你能把“路径、选择列表、结束条件”这三个元素说清楚，再强调回溯一定要恢复状态，通常就比较稳。工程上，规则组合、权限组合、工作流路径穷举、策略试探也都能看到类似思路。

**高分回答**
模板背后要先定义搜索树：节点状态包含当前路径与剩余选择，边是一项决策，叶子是完整候选。进入分支前修改状态，返回后严格恢复；若用不可变状态可免手动撤销但增加分配。结果要复制当前路径，不能把同一个可变 list 引用反复加入答案。

剪枝必须证明被剪子树不可能含解或更优解：约束剪枝、排序后的上下界、branch-and-bound 都有不同前提。候选含重复值时通常先排序，并在“同一层”跳过相同选择，不能跨层一概去重。搜索空间和输出本身可能指数级，若只求一个解可尽早返回；若相同状态反复出现且后续结果只由状态决定，记忆化/DP 比裸回溯更合适。设置节点与时间预算并报告是否完整搜索。

**追问点答案**  
- 回溯和 DFS 什么关系？  
回溯通常建立在 DFS 之上，但强调“试探后撤销状态”。  
- 为什么回溯容易超时？  
因为搜索空间可能指数级爆炸。  
- 怎么优化回溯？  
先证明安全剪枝与重复状态定义，再用候选排序、上下界、同层去重、memo；若输出量本身指数级只能做延迟生成、限制结果或改变需求，不能宣称优化到多项式。

**原理补充**  
回溯算法的本质，是在搜索树上逐步构造候选解，并在发现无效时尽早停止继续扩展。

**架构扩展**  
真实系统里，很多“方案组合生成”问题也会走类似回溯路线，比如规则组合、AB 流程组合、任务编排路径枚举。只是工程上通常会加很多剪枝和上限控制，不会裸跑。真正的关键不是会不会套模板，而是能不能看出搜索空间什么时候会爆。

**工程指标**  
- 搜索节点数：直接反映状态空间大小。  
- 剪枝命中率：判断优化是否有效。  
- 超时率：回溯问题最常见风险。  
- 递归深度：衡量搜索层数。  
- 结果集大小：帮助预估内存和输出成本。

**证据链**  
- 搜索树结构  
- 状态恢复代码  
- 剪枝条件  
- 超时样例

**低风险优化路径**  
1. 先写正确模板  
2. 再补去重和剪枝  
3. 最后根据规模评估是否要换思路

**易错点**  
- 忘记撤销选择  
- 不知道什么是搜索树  
- 只会暴力枚举，不会剪枝

## 14. 动态规划五步法是什么？

**核心结论**  
动态规划的关键不在于背题，而在于把问题拆成状态、转移和边界，再利用重复子问题避免重复计算。

**标准答案**  
动态规划常见的五步法可以概括为：第一，定义 `dp` 数组或状态含义；第二，找出状态转移方程；第三，确定初始状态；第四，明确遍历顺序；第五，用样例验证状态含义和转移是否一致。之所以很多人觉得 DP 难，是因为它不像双指针那样直观，真正难点是“怎么抽象状态”。面试里如果你能先说“我先定义状态”，通常就能让整个回答更有条理。工程上，DP 的思路也很像把一个大问题拆成很多可复用的小决策，再逐步累计最优结果，这在资源分配、路径规划、成本优化里都很常见。

**高分回答**
DP 状态必须是“足够且最小”的子问题摘要：给定状态后，未来决策无需知道更早历史。明确状态维度与取值、转移从哪些已解状态来、不可达哨兵、初值和答案位置，再按依赖 DAG 的拓扑顺序计算。复杂度是状态数乘每个状态转移数，不能看到二维表就固定说 O(n²)。

空间压缩前要证明旧状态不再需要，并确定遍历方向：0/1 背包一维容量倒序防止同一物品被重复使用，完全背包常正序允许复用。最大/最小值转移还要防哨兵参与加法溢出。若面试要求具体方案而非最优值，保存 predecessor 或从完整表回溯；滚动压缩可能丢失重建信息，是明确取舍。

**追问点答案**  
- 为什么遍历顺序重要？  
因为当前状态要依赖之前已经算好的状态。  
- 滚动数组是什么？  
当状态只依赖前几项时，可以压缩空间。  
- 什么时候不用 DP？  
没有重叠子问题时 memo 无收益；状态无法概括未来、转移形成未处理循环，或状态空间超过资源预算时应换建模、贪心/图算法/近似，而不是强建大表。

**原理补充**  
DP 的本质，是把指数级的重复搜索变成有记忆的状态复用。

**架构扩展**  
工程里虽然不会天天手写 DP 题，但很多调度、成本最优、路径选择、预算分配问题本质上都很像动态规划。能理解 DP，本质上是在训练“把复杂问题拆成可递推状态”的能力，这对架构设计也有帮助。

**工程指标**  
- 状态数量：决定时间和空间成本。  
- 转移耗时：评估状态设计是否过重。  
- 内存峰值：二维 DP 等场景很关键。  
- 压缩后空间节省率：验证优化收益。  
- 错误率：DP 常因状态定义不清导致 bug。

**证据链**  
- 状态定义  
- 转移方程  
- 初始化条件  
- 小样例推导过程

**低风险优化路径**  
1. 先写出状态含义  
2. 再推转移  
3. 最后再考虑空间压缩

**易错点**  
- 一上来就想公式，不先定义状态  
- 转移方程和初始条件不配套  
- 不会用样例验证

## 15. 贪心和 DP 的区别？

**核心结论**  
贪心是在每一步都做局部最优选择，DP 则是通过全局状态转移找整体最优，区别在于局部最优能否推出全局最优。

**标准答案**  
贪心算法的特点是每一步都做当前看起来最优的选择，希望最终得到整体最优结果；DP 则会系统地比较不同状态转移，确保最终结果来自全局最优。之所以两者容易混淆，是因为很多题都在求“最优”，但关键区别在于这个问题是否满足贪心选择性质。面试里如果你能说清楚“贪心更像是直接做决策，DP 更像是先把所有合理状态都算出来再选最好”，就很清楚了。工程上，贪心通常实现更简单、性能更好，但如果证明不了局部最优一定能推出全局最优，就应该谨慎，宁可用更稳的 DP 或其他方法。

**高分回答**
贪心要证明存在一个最优解包含当前选择，常用交换论证：把任意最优解的第一步替换成贪心选择而不变差；或证明算法始终“领先”。只有最优子结构不够，DP 也需要它，贪心还需要无需回头的 greedy-choice property。活动选择按最早结束可交换证明，普通硬币系统的“每次取最大面额”则可能被反例击败。

DP 保留足够多的状态后再比较，通常更稳但成本更高；贪心状态少、速度快，错误也更隐蔽。答题时先提出策略，主动找最小反例与暴力/DP oracle 随机对拍，再写证明。工程若使用近似贪心，应明确 approximation gap 或基线差值、容量与公平约束，不能把快速可行方案称为全局最优。

**追问点答案**  
- 为什么有些题看起来能贪心，实际不行？  
因为局部最优选择可能会破坏后续决策空间。  
- 贪心一定比 DP 快吗？  
很多时候是，但前提是题目真的适用。  
- 怎么判断能不能贪心？  
需要交换论证、领先性或问题结构证明每次局部选择可嵌入某个全局最优解；先找反例，证明不了时使用 DP/搜索，不能仅凭样例通过。

**原理补充**  
这道题的核心，不是背概念，而是理解“最优”的求法有两种不同哲学：一种立即决策，一种系统枚举状态。

**架构扩展**  
工程决策里也常有类似区别。比如资源分配、重试策略、流量调度，如果只按眼前最优做选择，可能会伤害整体稳定性；这就像错误地套用了贪心思路。成熟架构师会分辨哪些问题可以局部决策，哪些必须做全局平衡。

**工程指标**  
- 方案收益波动：判断局部决策是否稳定。  
- 计算耗时：贪心通常更轻量。  
- 全局最优偏差：评估贪心近似误差。  
- 状态规模：DP 方案常重点关注。  
- 回退次数：错误贪心在工程里常表现为反复修正。

**证据链**  
- 局部选择依据  
- 反例样本  
- 状态对比过程  
- 最优结果验证

**低风险优化路径**  
1. 先尝试找贪心性质  
2. 找不到可靠证明时改用 DP  
3. 最后用样例和反例验证策略

**易错点**  
- 觉得简单就一定能贪心  
- 不会举反例  
- 只背术语，不会解释决策差异

## 16. 快排和归并排序区别？

**核心结论**  
快排通常原地、平均性能好，归并稳定且时间复杂度更稳定，但需要额外空间。

**标准答案**  
快速排序的核心是选一个基准，把数组分成左右两部分后递归排序；归并排序则是先分治到最小单元，再把两个有序部分合并起来。两者平均时间复杂度都常见为 `O(n log n)`，但快排在极端情况下可能退化到 `O(n^2)`，归并则更稳定。之所以工程里两者都重要，是因为它们代表了两类典型分治思想：一种偏原地交换，一种偏有序合并。面试里最好补一句：归并是稳定排序，快排通常不是，这在处理对象排序时很关键。工程上，如果你更在意缓存友好和原地性能，快排很常见；如果你更在意稳定性或处理链表、外部排序，归并思路更自然。

**高分回答**
快排把“分区”放在递归前，理想平衡时 `T(n)=2T(n/2)+O(n)` 得 O(n log n)，原地实现额外栈平均 O(log n)，但连续选到极端 pivot 会变 O(n²) 且栈 O(n)。随机 pivot、三数取中和小区间插入排序降低退化概率；需要最坏上界的库实现常用 introsort 在深度过大时切堆排序，不能只说“随机后永不退化”。

归并始终分半，数组版时间 O(n log n)、辅助空间 O(n)，若合并相等元素时先取左侧即可稳定；它顺序读写并适合外部排序和多路归并。链表归并可通过改指针减少元素复制，但递归栈/拆分仍有成本。选型看稳定性、最坏时延、内存、输入介质与现成库，不建议业务手写排序替代经过优化的标准库。

**追问点答案**  
- 为什么快排会退化？  
如果每次选的基准都很差，比如总是最小或最大，就会导致划分极不均匀。  
- 归并为什么稳定？  
稳定性来自实现选择：比较相等时优先取左半段，才能保留原相对次序；若优先右侧，归并也会失去稳定性。
- 链表为什么更适合归并？  
因为链表不适合随机交换，但很适合拆分和合并。

**原理补充**  
快排和归并的区别，本质上是“划分后各自独立解决”和“先拆开再按序合并”两种分治风格。

**架构扩展**  
工程里排序问题并不只是算法题。比如日志归并、分片结果合并、排行榜更新，本质上都涉及“局部有序结果如何形成全局有序结果”，这是归并思想；而批量本地排序、原地重排更像快排思路。理解这两种模式，有助于你在系统层面做数据处理设计。

**工程指标**  
- 排序耗时：核心性能指标。  
- 额外内存占用：归并尤其关注。  
- 数据分布敏感度：快排退化风险看这里。  
- 稳定性需求满足率：业务对象排序时很重要。  
- 分片合并吞吐量：工程里常对应归并场景。

**证据链**  
- 输入分布  
- 基准选择策略  
- 合并逻辑  
- 排序性能测试

**低风险优化路径**  
1. 先看是否要求稳定排序  
2. 再看内存和数据规模约束  
3. 最后选快排、归并或混合实现

**易错点**  
- 只会背复杂度  
- 不知道稳定性差异  
- 不会把排序特性和业务场景联系起来

## 17. 如何判断链表有环？

**核心结论**  
判断链表有环最经典的方法是快慢指针，核心依据是如果存在环，快指针最终一定会追上慢指针。

**标准答案**  
判断链表是否有环，最常用的方法是快慢指针。慢指针每次走一步，快指针每次走两步；如果链表无环，快指针最终会走到空；如果有环，快指针会在环里不断追赶慢指针，最终两者相遇。之所以这个方法好，是因为它不需要额外哈希表，空间复杂度能做到 `O(1)`。面试里如果面试官继续追问，你还可以补一句：相遇后再做进一步推导，还能找到入环点。工程上，这种“不同速度状态最终相遇”的思路也很适合分析循环依赖和重复路径问题。

**高分回答**
Floyd 第一阶段先安全检查 `fast` 与 `fast.next`，slow 走 1、fast 走 2。进入长度 L 的环后，相对位置每轮前进 1（模 L），有限步内必相遇；比较的是节点身份，不是 value，相同值节点不代表同一节点。无环时 fast 到 null，时间 O(n)、额外空间 O(1)。

设头到入口距离 μ，相遇点距入口 x，slow 走 `μ+x`，fast 比它多走若干整环且速度双倍，可推出 `μ ≡ L-x (mod L)`；所以一指针回头、两者各走 1，首次相遇在入口。还可固定一指针绕一圈求环长。测试空、单节点无环/自环、两节点、入口在头/尾部后，并避免为了检测改写 next 破坏输入。

**追问点答案**  
- 为什么快指针一定会追上慢指针？  
因为在环里它们的相对速度是固定的，快指针每轮都会缩短距离。  
- 用哈希表能做吗？  
可以，遍历时记录访问过的节点，重复出现就说明有环。  
- 怎么找入环点？  
利用头到入口距离与相遇点到入口剩余距离模环长相等；一指针回头后同速走，首次节点身份相同处为入口。回答最好给出距离等式而非只背步骤。

**原理补充**  
这道题最核心的原理，是把结构问题转化成速度差问题。

**架构扩展**  
工程里虽然不常直接判断链表有环，但“检测结构闭环或重复依赖”的思路非常常见，比如配置依赖、自定义流程编排、对象引用关系分析。理解快慢指针，不只是会一道题，而是在训练你把结构问题转成状态推进问题。

**工程指标**  
- 检测耗时：应接近线性。  
- 额外内存占用：快慢指针的优势就在这里。  
- 循环依赖发现率：在工程映射问题里很关键。  
- 错误路径重试次数：有时是环路存在的外在表现。  
- 结构异常日志数：辅助定位问题规模。

**证据链**  
- 指针移动记录  
- 节点访问轨迹  
- 相遇位置  
- 结构样例图

**低风险优化路径**  
1. 先实现快慢指针判环  
2. 再补入环点推导  
3. 最后验证空链表和单节点等边界

**易错点**  
- 快指针空判断不完整  
- 只会判环，不会解释原理  
- 不理解为什么可以找到入环点

## 18. LRU 缓存怎么实现？

**核心结论**  
LRU 需要同时支持“快速查找”和“快速更新最近使用顺序”，最经典实现是哈希表加双向链表。

**标准答案**  
LRU 的目标是：当缓存满了以后，淘汰最近最少使用的数据；同时，任何一次访问或更新，都要把对应元素变成“最近刚用过”。之所以常用哈希表加双向链表，是因为哈希表负责 `O(1)` 找到节点，双向链表负责 `O(1)` 把节点移动到头部或者删除尾部节点。面试里这题的关键不是背“哈希表 + 链表”，而是要说清楚为什么单独用其中一个结构不够。工程上，LRU 是本地缓存、页面缓存、连接池回收、热点对象淘汰的常见基础思路，但实际系统里还会结合过期时间、权重和并发控制一起设计。

**高分回答**
Map 保存 `key -> node`，双链表按最近使用顺序排列；`get` 命中和 `put` 更新都先 detach 再插到头，超容量删除尾前节点并同步移除 Map。头尾哨兵把空表、单节点和端点操作统一。容量为 0、更新已有 key 不增加 size、淘汰后 map/list 一致，是最常漏的边界；O(1) 依赖哈希平均复杂度。

这只是按条目数的非并发 LRU。生产缓存通常按权重/字节容量，处理 TTL、加载去重、统计、弱引用与并发；全局锁虽正确但高并发下会让每次读都写链表形成热点，分段/近似 LRU 或成熟缓存库更现实。LRU 假设近期使用预测未来，扫描流量会污染缓存；需以命中率、eviction、加载 TP99、锁等待和实际字节验证，而不是只看 get/put O(1)。

**追问点答案**  
- 为什么要双向链表？  
因为需要在 `O(1)` 时间内把任意节点删除和移动。  
- 只用链表行不行？  
不行，查找节点会退化成 `O(n)`。  
- 只用哈希表行不行？  
普通无序哈希表不够；带访问顺序的 LinkedHashMap 等实现可以封装哈希+双链职责。关键是同时拥有 O(1) 定位和 O(1) 顺序更新，而非必须手写两个对象。

**原理补充**  
LRU 的本质，是在“访问局部性”假设下，用有序淘汰策略提升缓存命中率。

**架构扩展**  
真实系统里，缓存淘汰策略从来不是单独存在的。比如热点服务的本地缓存，除了 LRU 之外，还要考虑 TTL、热点 key 抖动、并发更新、内存上限和淘汰风暴。成熟架构师回答这题时，往往会顺手补一句：LRU 是基础策略，真正上线还要结合业务访问模式做调优。

**工程指标**  
- 缓存命中率：衡量 LRU 是否有效。  
- 淘汰次数：反映容量和访问模式是否匹配。  
- 单次 get/put 延迟：验证 `O(1)` 实现是否成立。  
- 内存占用：缓存类问题必须盯。  
- 热点 key 失效率：帮助判断 LRU 是否误伤热点。

**证据链**  
- 缓存访问轨迹  
- 链表顺序变化  
- 命中 / 淘汰统计  
- 容量配置

**低风险优化路径**  
1. 先实现标准哈希表 + 双向链表  
2. 再补容量控制和边界处理  
3. 最后根据场景扩展 TTL 或并发控制

**易错点**  
- 只会背组合结构，不会解释职责分工  
- 链表节点删除和移动边界写错  
- 不知道工程里还要关注命中率和热点行为

## 19. 二叉树最近公共祖先怎么求？

**核心结论**  
最近公共祖先问题通常利用递归从子树向上汇总结果，谁第一次同时覆盖目标节点，谁就是答案。

**标准答案**  
在普通二叉树里，求最近公共祖先最经典的做法是递归：如果当前节点为空，或者等于 `p`、`q` 中的任意一个，就直接返回当前节点；然后分别去左右子树找。如果左右子树都找到了结果，说明当前节点就是最近公共祖先；如果只有一边找到，就把那一边继续往上返回。之所以这个方法好，是因为它自然符合“祖先关系从底向上汇总”的结构特征。面试里如果是二叉搜索树，还可以补充利用大小关系剪枝，写法会更高效。工程上，这类“从局部子树汇总信息再向上决策”的思路，在权限树、组织树、目录树处理里都很常见。

**高分回答**
递归返回值要先声明契约。若题目保证 p、q 都存在，函数返回“该子树中找到的 p/q/LCA”：当前即目标就返回；左右都非空表示目标分居两侧，当前是最低汇合点；仅一侧非空向上传递。时间 O(n)，空间 O(h)。但若节点可能不存在，这个简版会在只找到 p 时错误返回 p，需同时返回 found count/位掩码，只有累计找到两个才确认答案。

BST 可按比较器判断：p、q 都小于当前走左，都大于走右，否则当前为分叉点；前提是节点确实属于该树、键与比较器满足 BST 不变量，重复键时仅按 value 可能无法识别目标身份。多次 LCA 查询可预处理 parent/depth、binary lifting 或 Euler tour+RMQ，以预处理空间换查询时间。

**追问点答案**  
- 为什么左右子树都找到时当前节点就是答案？  
因为这说明 `p` 和 `q` 分别位于当前节点两侧，当前节点是最低同时覆盖它们的点。  
- BST 为什么可以更快？  
因为可以根据节点值大小直接决定往左还是往右。  
- 如果一个节点本身就是另一个节点祖先呢？  
在允许节点作为自身祖先的常见定义下是；递归遇到该节点返回，并由另一目标的存在性保证确认。若输入不保证存在，仍需计数验证两个节点都找到。

**原理补充**  
这道题的本质，是把“祖先”问题转化成“子树里是否包含目标节点”的递归汇总问题。

**架构扩展**  
在业务系统里，最近公共祖先的思想常出现在组织架构权限继承、类目路径合并、目录冲突定位等场景。你不一定直接写这道题，但“从树的下层信息向上汇总出一个最小共同边界”的思路非常有价值。

**工程指标**  
- 查询耗时：反映树深对性能的影响。  
- 递归深度：树过深时要关注。  
- 命中率：看输入节点是否都在树中。  
- 路径长度：帮助分析祖先位置。  
- 错误结果率：边界处理不当时常见。

**证据链**  
- 目标节点路径  
- 左右子树返回值  
- 递归执行轨迹  
- 树结构快照

**低风险优化路径**  
1. 先写普通二叉树递归解  
2. 再单独总结 BST 优化版  
3. 最后补不存在节点时的边界处理

**易错点**  
- 把“最近”理解错成“任意共同祖先”  
- 递归返回语义不清  
- 忽略一个节点就是祖先的情况

## 20. 面试手撕代码如何自测边界？

**核心结论**  
手撕代码自测的关键不是随便举例，而是系统性覆盖空、单个、重复、极值、顺逆序和非法输入这些高风险边界。

**标准答案**  
面试手撕代码时，自测最重要的是按边界类型系统检查，而不是想到一个例子就跑一个例子。通常我会先测空输入和最小规模输入，确保不会空指针或越界；再测单元素、双元素、全相同、严格递增、严格递减，看看逻辑在简单结构下是否稳定；如果题目和数值有关，还会测负数、0、极大值、重复值。之所以要这么做，是因为大多数手撕 bug 并不是思路错，而是边界没守住。面试里如果你能一边写代码一边主动说“我现在补几个边界样例”，会显得非常稳。工程上，这其实也是单元测试思维的缩小版。

**高分回答**
写前先复述契约：输入能否为空、是否允许修改、重复值、返回何种哨兵、溢出范围和复杂度目标。写时给循环/递归状态一句不变量，每次更新检查是否保持且一定前进；写后先做最小反例，再按结构分区测试，而不是只跑题目样例。边界应由算法分支导出：二分测区间外和重复，链表测自环，DP 测不可达，图测自环/多边/非连通。

高效自测组合三层：手工 trace 一个能走遍关键分支的小例；写简单但慢的 brute-force oracle，对小随机输入与优化解对拍；再做性质测试，如排序结果有序且多重集合不变、LCA 覆盖两节点、LRU map/list size 一致。最后复算时间/空间、检查输入是否被意外修改、递归深度和整数溢出，并口述失败时返回什么。

**追问点答案**  
- 为什么很多人思路对但代码错？  
因为边界条件、索引和状态初始化最容易出问题。  
- 自测时优先测什么？  
优先测空输入、最小输入和最容易越界的情况。  
- 没时间全测怎么办？  
优先测能改变控制流的最小反例：空/一项、边界命中、重复/无解，再手工走一遍最复杂分支并验证不变量；比重复跑多个正常样例更有信息量。

**原理补充**  
自测的本质，是用最少的关键样例去覆盖最容易出错的控制分支。

**架构扩展**  
这套习惯对工程开发非常有帮助。很多线上事故，本质上不是功能逻辑不会写，而是没人验证空值、极端输入、配置缺失、流量突增这些边界。会自测的人，通常也更容易写出稳的业务代码。

**工程指标**  
- 边界缺陷命中率：衡量自测是否有效。  
- 回归 bug 数量：说明代码稳定性。  
- 单测覆盖率：工程里的延伸指标。  
- 线上异常率：边界没守住时常会抬头。  
- 修复返工次数：好的自测会降低这里。

**证据链**  
- 样例清单  
- 边界分类  
- 代码分支覆盖情况  
- 手工推演记录

**低风险优化路径**  
1. 先列边界类型  
2. 再为每类挑一个代表样例  
3. 最后把最危险路径手工走一遍

**易错点**  
- 只测正常样例  
- 不会系统分类边界  
- 写完代码不回头验证索引和初始化
