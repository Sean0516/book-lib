# 高频100题-抽认卡版

使用方式：
- `正面`：先自己口述 30 到 90 秒
- `背面`：只看关键词，不要一上来就读全文
- `复习顺序`：基础 -> 架构 -> AI

---

## 第一部分：基础高频40题

### 基础模块1 Java基础与集合

**卡片001**
- 正面：`==` 和 `equals` 有什么区别？
- 背面：`==` 比较地址或基本类型值；`equals` 比较逻辑相等；包装类和字符串题最容易混淆。

**卡片002**
- 正面：为什么 `hashCode` 要和 `equals` 一起重写？
- 背面：相等对象必须有相同 hash；否则 HashMap/HashSet 命中异常；口诀是“先 hash 分桶，再 equals 判等”。

**卡片003**
- 正面：`String` 为什么不可变？
- 背面：便于缓存和字符串池、天然线程安全、适合作为 HashMap key；底层值不变，避免 hash 漂移。

**卡片004**
- 正面：`HashMap` 为什么线程不安全？
- 背面：并发 `put/resize` 会破坏结构；问题核心是结构修改无同步保护；并发场景改用 `ConcurrentHashMap`。

**卡片005**
- 正面：`ConcurrentHashMap` 为什么比 `synchronizedMap` 更适合高并发？
- 背面：前者锁粒度更细，JDK8 主要靠 `CAS + synchronized` 桶级控制；后者更像整表大锁。

### 基础模块2 JVM基础

**卡片006**
- 正面：JVM 运行时数据区有哪些？
- 背面：程序计数器、虚拟机栈、本地方法栈、堆、方法区/元空间；面试常追问“谁线程私有、谁线程共享”。

**卡片007**
- 正面：堆和栈的区别？
- 背面：栈存方法帧和局部变量，线程私有；堆存对象实例，线程共享；一个偏执行现场，一个偏对象存储。

**卡片008**
- 正面：双亲委派模型解决什么问题？
- 背面：避免类重复加载和核心类被篡改；先向上委托，再自己加载；常结合 Tomcat、自定义类加载器追问。

**卡片009**
- 正面：Minor GC 和 Full GC 区别？
- 背面：Minor GC 主要回收新生代，频繁但相对快；Full GC 涉及老年代甚至元空间，停顿更重。

**卡片010**
- 正面：频繁 Full GC 怎么分析？
- 背面：先看 GC 日志和 Old 区回落，再判断是大对象、泄漏、缓存失控还是参数不合理；先修代码再调参数。

**卡片011**
- 正面：内存泄漏怎么定位？
- 背面：看 Full GC 后内存是否不回落；抓 dump，查大对象和引用链；高频泄漏点是缓存、ThreadLocal、连接未释放。

### 基础模块3 并发基础

**卡片012**
- 正面：原子性、可见性、有序性分别是什么？
- 背面：原子性是不可分割，可见性是一个线程修改对别的线程可见，有序性是执行顺序不被错误重排。

**卡片013**
- 正面：`volatile` 能保证什么，不能保证什么？
- 背面：能保证可见性和一定的有序性，不能保证复合操作原子性；经常和 `synchronized` 对比问。

**卡片014**
- 正面：`synchronized` 底层原理是什么？
- 背面：对象监视器 monitor；JDK 有偏向锁、轻量级锁、重量级锁升级路径；本质是互斥和内存语义。

**卡片015**
- 正面：CAS 的优缺点是什么？
- 背面：优点是无锁、高并发下吞吐好；缺点是自旋开销、ABA 问题、只能保证单变量原子更新。

**卡片016**
- 正面：AQS 是什么？
- 背面：同步器框架；核心是 `state + CLH队列 + 模板方法`；`ReentrantLock`、`CountDownLatch` 都建立在它上面。

**卡片017**
- 正面：线程池核心参数怎么理解？
- 背面：核心线程数、最大线程数、队列、存活时间、拒绝策略；本质是在“线程数”和“排队”之间做背压取舍。

**卡片018**
- 正面：ThreadLocal 为什么会内存泄漏？
- 背面：key 是弱引用，value 不是；线程池线程长期存活时，未清理 value 会挂在线程上；用完 `remove`。

### 基础模块4 计算机基础

**卡片019**
- 正面：阻塞 IO 和非阻塞 IO 的区别？
- 背面：核心区别是系统调用返回方式；阻塞会等数据准备好，非阻塞会立即返回；常继续追问 IO 多路复用。

**卡片020**
- 正面：TCP 为什么三次握手而不是两次？
- 背面：双方都要确认“自己能发、对方能收”；两次不能可靠确认客户端接收能力；三次才完整建立连接状态。

**卡片021**
- 正面：TIME_WAIT 为什么存在？
- 背面：保证最后一个 ACK 可重发，避免旧连接报文污染新连接；面试常延伸到端口耗尽和连接复用。

**卡片022**
- 正面：线上 CPU 飙高怎么排查？
- 背面：`top` 找进程，`top -Hp` 找线程，`jstack` 看栈，再结合 GC、锁、热点方法定位；先止血再根因分析。

### 基础模块5 数据库基础

**卡片023**
- 正面：InnoDB 为什么选择 B+Tree？
- 背面：磁盘友好、层数低、范围查询强；叶子节点有序，适合数据库索引；比红黑树更适合页式存储。

**卡片024**
- 正面：聚簇索引和二级索引有什么区别？
- 背面：聚簇索引叶子节点存整行数据，二级索引叶子节点存主键值；二级索引查整行通常要回表。

**卡片025**
- 正面：覆盖索引为什么快？
- 背面：查询字段都在索引里，不用回表；减少随机 IO；是慢 SQL 优化里性价比很高的一招。

**卡片026**
- 正面：什么情况下索引会失效？
- 背面：最左前缀被破坏、函数/计算包裹、隐式类型转换、范围后列利用受限；优化前先看 `EXPLAIN`。

**卡片027**
- 正面：MVCC 解决了什么问题？
- 背面：让读写冲突减少；通过版本链和 Read View 实现一致性读；重点不是“完全无锁”，而是“减少锁竞争”。

**卡片028**
- 正面：慢 SQL 怎么定位和优化？
- 背面：先看慢日志与业务峰值，再看 `EXPLAIN`、索引、扫描行数、锁等待；先小改 SQL/索引，再评估架构升级。

### 基础模块6 缓存与消息基础

**卡片029**
- 正面：Redis 为什么快？
- 背面：内存操作、单线程避免锁竞争、高效数据结构、IO 多路复用；注意“快”不代表不会被大 key 打爆。

**卡片030**
- 正面：缓存穿透、击穿、雪崩分别是什么？
- 背面：穿透是查不存在数据；击穿是热点 key 失效瞬间打到 DB；雪崩是大量 key 同时失效；答案要配治理手段。

**卡片031**
- 正面：缓存和数据库一致性怎么做？
- 背面：常见是旁路缓存：读缓存、写 DB 后删缓存；关键不是绝对一致，而是控制不一致窗口和补偿机制。

**卡片032**
- 正面：如何保证消息不丢？
- 背面：生产端确认、Broker 持久化、消费端成功后再提交位点；中间任一环节都要能重试和审计。

**卡片033**
- 正面：如何处理重复消费？
- 背面：业务幂等是核心；用唯一业务键、去重表、去重缓存；不要幻想 MQ 本身帮你完全解决。

### 基础模块7 Spring基础

**卡片034**
- 正面：IOC 和 DI 是什么？
- 背面：IOC 是控制反转，DI 是依赖注入的实现方式；重点不是名词，而是对象创建和依赖装配交给容器。

**卡片035**
- 正面：AOP 的核心价值是什么？
- 背面：把横切逻辑从业务中抽出来；常见在事务、日志、权限；面试常追问代理机制和事务实现。

**卡片036**
- 正面：Spring 事务是怎么实现的？
- 背面：基于 AOP 代理拦截方法调用，在调用前后做事务开启、提交、回滚；关键点是“必须经过代理”。

**卡片037**
- 正面：`@Transactional` 为什么会失效？
- 背面：同类内部调用、非 public、异常被吞、代理没生效；回答时最好顺手提“自调用”这个经典坑。

### 基础模块8 算法与数据结构基础

**卡片038**
- 正面：二分查找最容易错在哪里？
- 背面：边界和终止条件；`mid` 计算、防死循环、找左边界/右边界写法不同；面试时一定手动测边界。

**卡片039**
- 正面：滑动窗口适合哪类题？
- 背面：连续区间最值、满足条件的最短/最长子串；关键是“右扩张、左收缩、维护窗口状态”。

**卡片040**
- 正面：LRU 怎么实现？
- 背面：`HashMap + 双向链表`；Map 负责 O(1) 定位，链表负责 O(1) 移动和淘汰；这是面试经典手撕题。

## 第二部分：架构高频45题

### 架构模块1 系统设计与容量规划

**卡片041**
- 正面：系统设计题为什么要先讲目标和约束？
- 背面：因为架构不是堆组件；QPS、TP99、可用性、成本、团队能力决定方案边界；先定边界再谈方案。

**卡片042**
- 正面：峰值 QPS 一般怎么估？
- 背面：从 DAU、日请求量、峰值系数、活动放大倍数推；关键不是算得绝对准，而是建立容量模型和冗余。

**卡片043**
- 正面：容量规划最容易漏什么？
- 背面：异步链路、下游依赖、回源放大、缓存失效后的瞬时峰值；只算应用层通常会失真。

**卡片044**
- 正面：大促前为什么一定要做压测和演练？
- 背面：因为纸面容量模型和真实瓶颈常不同；压测找热点，演练验证回滚、限流、降级和告警是否真能工作。

### 架构模块2 分布式架构基础

**卡片045**
- 正面：CAP 在真实业务里怎么理解？
- 背面：不是三选一，而是网络分区发生时在一致性和可用性间取舍；大多数业务是按场景分级一致性。

**卡片046**
- 正面：为什么分布式系统必须做幂等？
- 背面：重试、超时、消息重复、回调重复都会发生；没有幂等，系统越努力恢复，业务越容易出错。

**卡片047**
- 正面：最终一致性落地时必须补什么？
- 背面：补偿、对账、审计、告警；如果只会说“最终一致”，但说不出发现和修复机制，答案就不完整。

**卡片048**
- 正面：分布式锁最容易踩什么坑？
- 背面：锁过期误释放、业务执行时间超时、时钟漂移、把锁当万能药；回答时最好提“锁只保护临界区，不解决业务设计问题”。

**卡片049**
- 正面：为什么不建议轻易做跨服务强事务？
- 背面：耦合高、可用性差、失败链路长；更稳的做法通常是本地事务 + 消息/事件 + 补偿。

### 架构模块3 微服务治理

**卡片050**
- 正面：服务拆分粒度怎么定？
- 背面：按业务边界、变更频率、团队协作来定；太粗变更耦合，太细治理成本爆炸；避免“为了微服务而微服务”。

**卡片051**
- 正面：网关层职责边界是什么？
- 背面：统一鉴权、路由、限流、灰度、协议适配；不要把过重业务逻辑塞进网关，避免新单点。

**卡片052**
- 正面：限流和熔断有什么区别？
- 背面：限流是主动控流量，熔断是依赖异常时快速失败；两者通常配合使用，不是替代关系。

**卡片053**
- 正面：重试为什么容易放大故障？
- 背面：下游本来就慢，再叠加重试会把请求放大几倍；所以重试必须配超时、退避、幂等和限流。

**卡片054**
- 正面：灰度发布怎么回答才完整？
- 背面：流量切分、指标对比、回滚开关、版本兼容、发布观察窗口；别只说“先发一部分机器”。

### 架构模块4 数据架构

**卡片055**
- 正面：什么时候才该分库分表？
- 背面：单库单表容量、索引和写入热点已经逼近极限，且 SQL/索引/缓存优化收益见顶；不要过早复杂化。

**卡片056**
- 正面：分片键怎么选？
- 背面：要兼顾写入均匀、查询路由、热点控制、扩容迁移；选错分片键比不上分库分表更可怕。

**卡片057**
- 正面：读写分离为什么会带来一致性问题？
- 背面：因为主从复制有延迟；刚写完去从库查可能读旧数据；常见解决是强制读主、会话一致性、延迟感知路由。

**卡片058**
- 正面：ES 什么时候比 MySQL 更合适？
- 背面：复杂搜索、倒排检索、多条件组合和相关性排序；但 ES 不是交易库，强一致场景仍以 MySQL 为主。

**卡片059**
- 正面：数据迁移不停机怎么做？
- 背面：双写/双读、增量同步、灰度校验、回滚预案；关键不是“迁过去”，而是“迁的过程可控可回退”。

**卡片060**
- 正面：主从延迟怎么治理？
- 背面：先看复制链路和慢 SQL，再控大事务、批量写入和 IO 瓶颈；业务上补读主兜底和延迟感知。

### 架构模块5 消息与异步架构

**卡片061**
- 正面：什么时候必须引入 MQ？
- 背面：削峰、解耦、异步化、可靠投递；如果只是为了“看起来高级”而上 MQ，复杂度通常不值。

**卡片062**
- 正面：如何保证顺序消息？
- 背面：让同一业务键路由到同一分区，消费端单线程或受控并发处理；顺序通常要用吞吐换。

**卡片063**
- 正面：死信队列最常见用途是什么？
- 背面：隔离无法成功消费的消息，避免主消费链路被拖死；同时给补偿、人工处理、问题分析留下入口。

**卡片064**
- 正面：消息积压怎么回答才像做过系统？
- 背面：先分辨是生产过快、消费变慢还是下游故障；再说扩容、跳过非核心、限流、降级、补消费和数据修复。

**卡片065**
- 正面：事务消息适合什么场景？
- 背面：本地事务成功后必须可靠地触发异步动作，比如订单创建后发库存扣减事件；它不是万用分布式事务方案。

### 架构模块6 稳定性与高可用

**卡片066**
- 正面：99.95% 可用性意味着什么？
- 背面：年不可用时间约 4.38 小时；可用性目标要能分解到架构、发布、值班和故障演练，不只是口号。

**卡片067**
- 正面：故障隔离通常从哪些层做？
- 背面：线程池、连接池、机房、集群、租户、业务域；核心思想是把故障限制在局部，而不是全站共振。

**卡片068**
- 正面：降级策略怎么按业务分级？
- 背面：先保交易主链路，再保关键查询，最后放弃非核心功能；降级不是简单返回错误，而是保核心价值。

**卡片069**
- 正面：RTO 和 RPO 分别是什么？
- 背面：RTO 是恢复时间目标，RPO 是可接受数据损失窗口；容灾题不说这两个，答案会偏空。

**卡片070**
- 正面：故障复盘最应该沉淀什么？
- 背面：事实时间线、根因、止血动作、长期治理项、责任边界和验证结果；复盘不是找人背锅，是减少复发。

### 架构模块7 可观测性与运维体系

**卡片071**
- 正面：SLI、SLO、SLA 分别是什么？
- 背面：SLI 是指标，SLO 是目标，SLA 是对外承诺；三者层级不同，别混说。

**卡片072**
- 正面：为什么日志、指标、链路追踪要联动？
- 背面：指标告诉你“出问题了”，链路告诉你“哪段慢”，日志告诉你“具体发生了什么”；单看一种往往不够。

**卡片073**
- 正面：怎么避免告警噪音？
- 背面：分级、聚合、抑制、去重，关注“可行动告警”；真正好的告警是能直接驱动处理动作。

**卡片074**
- 正面：MTTR 为什么重要？
- 背面：平均恢复时间直接体现故障处理效率；提升 MTTR 往往比单纯减少报警次数更贴近业务价值。

### 架构模块8 性能工程

**卡片075**
- 正面：性能优化第一步为什么不是改代码？
- 背面：因为不定位瓶颈就优化，常常优化错地方；先定目标、建基线、找证据，再动刀。

**卡片076**
- 正面：TPS 和 TP99 为什么要一起看？
- 背面：只看吞吐可能掩盖长尾延迟，只看延迟可能忽略整体处理能力；性能题一定要兼顾平均和尾部。

**卡片077**
- 正面：锁竞争怎么定位？
- 背面：看线程栈、阻塞时间、热点锁对象、上下游耗时；很多所谓 CPU 高，其实是锁竞争和自旋浪费。

**卡片078**
- 正面：性能回归怎么防？
- 背面：保留基线、建立压测用例、关键指标门禁、版本对比；性能不是一次优化，是持续治理。

### 架构模块9 安全与合规

**卡片079**
- 正面：认证和鉴权有什么区别？
- 背面：认证回答“你是谁”，鉴权回答“你能做什么”；面试经常先问概念，再问怎么分层落地。

**卡片080**
- 正面：JWT 最大的工程风险是什么？
- 背面：一旦签发出去难以主动失效，密钥泄露风险大；适合无状态场景，但要配刷新、吊销和短期有效期。

**卡片081**
- 正面：为什么审计日志很重要？
- 背面：高危操作追责、合规、事故还原都靠它；没有审计，很多安全设计就缺闭环。

### 架构模块10 云原生与平台化

**卡片082**
- 正面：Kubernetes 的核心价值是什么？
- 背面：统一部署、调度、弹性、服务发现和自愈；它的价值不只是“容器编排”，而是平台化标准能力。

**卡片083**
- 正面：滚动发布和蓝绿发布怎么选？
- 背面：滚动省资源，蓝绿回滚快、隔离好；选型要看系统规模、成本和风险承受度。

**卡片084**
- 正面：IaC 为什么重要？
- 背面：把环境配置代码化、可审计、可复现；减少“我机器上没问题”这类环境漂移。

### 架构模块11 业务建模与领域拆分

**卡片085**
- 正面：限界上下文为什么重要？
- 背面：它定义了模型和语言的边界；没有上下文边界，系统会在概念和职责上越来越混乱。

**卡片086**
- 正面：怎么识别核心域？
- 背面：看业务差异化、竞争壁垒、长期投入价值；核心域值得最强工程和建模资源。

**卡片087**
- 正面：单体拆微服务第一刀怎么切？
- 背面：优先切边界清晰、收益明显、对外依赖较少的模块；不要一开始就动最核心最复杂部分。

### 架构模块12 架构治理与技术管理

**卡片088**
- 正面：技术债为什么不能只靠“以后再说”？
- 背面：技术债会持续吞噬交付速度和稳定性；要把偿还节奏纳入规划，而不是靠激情式重构。

**卡片089**
- 正面：架构评审到底在评什么？
- 背面：评业务目标、边界条件、核心方案、风险、指标和落地路径；不是只看图画得漂不漂亮。

**卡片090**
- 正面：向管理层汇报技术方案要强调什么？
- 背面：业务价值、风险、成本、时间、预期收益；少讲框架名，多讲决策依据和结果。

## 第三部分：AI高频15题

### AI模块1 LLM基础与模型选型

**卡片091**
- 正面：模型选型最核心的维度有哪些？
- 背面：质量、延迟、成本、安全、可控性；别只谈模型聪明不聪明，要谈业务目标和 SLA。

**卡片092**
- 正面：什么时候要做多模型路由？
- 背面：任务复杂度差异大、成本压力大、需要高可用或多供应商兜底时；简单任务上大模型往往浪费。

**卡片093**
- 正面：模型升级为什么必须有回滚？
- 背面：升级可能带来风格漂移、工具调用退化、成本飙升；没有回滚，线上试错代价太高。

### AI模块2 RAG架构设计与检索工程

**卡片094**
- 正面：什么时候必须上 RAG？
- 背面：知识时效性强、私有数据多、必须可引用溯源时；如果问题都靠通识回答，直接问模型更简单。

**卡片095**
- 正面：切块策略为什么会直接影响答案质量？
- 背面：切太碎上下文断裂，切太大召回噪音高、成本高；切块本质是在召回率和精度间平衡。

**卡片096**
- 正面：混合检索为什么常比纯向量检索更稳？
- 背面：关键词匹配和语义检索互补；对专有名词、编号、精确术语类问题尤其明显。

**卡片097**
- 正面：RAG 如何降低幻觉？
- 背面：提高召回质量、加重排、强制引用、无答案拒答；不要把“幻觉率”只归咎给模型本身。

### AI模块3 Agent系统与工具调用

**卡片098**
- 正面：Agent 和固定工作流怎么区分？
- 背面：流程稳定、分支可枚举时用工作流；任务开放、需要动态决策和工具选择时用 Agent；别什么都 Agent 化。

**卡片099**
- 正面：Agent 最大的工程风险是什么？
- 背面：权限失控、错误放大、循环调用、成本失控；所以要加工具白名单、预算、审计和人工兜底。

**卡片100**
- 正面：AI 项目怎么证明 ROI？
- 背面：看质量提升、人效提升、响应时长下降、人工替代率和单位请求成本；没有业务指标，AI 项目很难长期成立。

## 刷卡建议

1. `第一轮`：只刷 100 张正面，训练你“看到题就能开口”。
2. `第二轮`：对答不顺的卡片，回到对应模块文档看 `标准答案 + 追问点答案`。
3. `第三轮`：把卡片按“场景题、原理题、治理题、AI题”重新分组刷。
4. `冲刺轮`：每天抽 20 张，要求自己在 20 到 40 秒内给出结构化回答。
