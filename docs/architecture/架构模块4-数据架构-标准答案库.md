## 1. 分库分表的触发阈值如何判断？

::: tip L2 深度训练
先区分业务边界与物理扩展，避免用分片代替领域所有权：[领域发现与限界上下文](/deep-dives/business-evolution/01-domain-discovery)
:::

**核心结论**  
分库分表的触发点不该只看数据量，而要综合看单机资源瓶颈、写入压力、查询模式和增长趋势。

**标准答案**  
我判断分库分表时，通常不会只看“表是不是过亿”这种单一数字，而是先看单机还能不能稳定扛住当前和未来一段时间的写入、查询和索引维护压力。之所以很多团队会过早拆分，是因为把“数据量大”和“必须分片”画了等号，但真实瓶颈往往来自热点写、索引膨胀、慢查询比例升高、主从延迟、DDL 成本和运维窗口变短。工程上更稳的做法是先把单库优化空间压榨到位，比如索引、冷热分层、归档、读写分离，再判断增长趋势是否会继续把系统推过边界。面试里如果能补一句“分库分表是单机治理空间接近天花板后的架构动作”，会比较成熟。

**高分回答**
我不会设“5000 万行必拆”的固定线，而会证明单节点在迁移提前期内将越过安全边界。证据包括写 TPS 与锁等待、查询 TP99 和扫描行数、buffer pool 命中、主从延迟、存储增长、备份恢复/DDL 窗口，以及未来 6～12 个月业务预测；例如预计 8 个月后写入达到单主 N-1 安全容量，而方案设计、回填和演练需 4 个月，就应在剩余 4 个月前启动。先验证索引/SQL、批量写、归档冷热数据、读副本、垂直拆分和实例升级是否还能以较低复杂度满足 SLO。确需分片时还要确认访问模式能提供稳定路由键，并把跨片事务、全局唯一、聚合分页、扩容重分片和运维工具成本计入决策；否则只是把数据库瓶颈转成应用故障。迁移采用影子路由、分段回填、CDC 增量、业务不变量校验和灰度切读写，保留权威源与回滚窗口。上线阈值以容量拐点前、单分片故障后仍满足 SLO 为准。最终触发条件是“现有治理空间小于增长与迁移所需余量”，不是当前行数看起来很大。

**追问点答案**  
- 单表千万级一定要拆吗？  
不一定，关键要看 SQL 模式、索引设计和写入压力。  
- 为什么增长趋势比当前量更重要？  
因为需要用预计耗尽时间减去设计、回填、演练和回滚周期来确定启动日期；等资源已经饱和才拆通常没有安全迁移窗口。
- 拆分前为什么要先做归档？  
因为很多所谓“大表”其实是冷热数据混在一起导致的。

**原理补充**  
分库分表本质上是把单节点瓶颈转化成分布式复杂度，所以前提一定是单节点收益已经逼近上限。

**架构扩展**  
真实系统里，订单流水、行为日志、支付明细最容易走到分片阶段；但用户表、配置表、商品主表很多时候并不需要急着拆。成熟架构师会先区分“容量型问题”和“模型型问题”，避免为了拆而拆。

**工程指标**  
- 单表写入 TPS：衡量写压。  
- 慢 SQL 占比：判断单机治理空间。  
- 主从延迟：反映扩展压力。  
- 索引大小和增长率：看维护成本。  
- 归档后数据回落比例：评估是否真需要分片。

**证据链**  
- 表增长曲线  
- SQL 分布  
- 资源水位数据  
- 慢日志和复制延迟记录

**低风险优化路径**  
1. 先做单库优化和归档  
2. 再评估增长趋势和热点模式  
3. 最后设计分片策略和迁移方案

**易错点**  
- 只看数据量不看压力模式  
- 过早分片  
- 不先做单机治理

## 2. 分片键怎么选？

::: tip L2 深度训练
让访问模式、聚合标识和迁移路由共同决定分片键：[聚合、不变量与领域事件](/deep-dives/business-evolution/02-aggregate-invariants)
:::

**核心结论**  
分片键的核心不是“随机均匀”四个字，而是既要尽量打散流量，又要尽量贴合主查询路径。

**标准答案**  
选择分片键时，我一般会先看最核心的写入路径和最高频的查询路径，优先选既能保证分布相对均匀，又能覆盖主业务访问模式的字段。之所以这件事这么难，是因为纯粹均匀的键不一定适合查询，而纯粹贴合查询的键又可能形成热点。工程上常见的候选有用户 ID、订单 ID、租户 ID、时间维度或它们的组合，有时还会做 hash 映射来平衡分布。面试里如果能补一句“分片键选错，后面很多复杂度都在为它还债”，会很有说服力。

**高分回答**
我先从真实访问日志列出读写路径及占比，要求主路径能从请求直接得到路由键，避免先查映射表再定位分片。例如订单主要按用户查看且写入均匀，可按 `hash(userId)` 路由；按订单号查询则让订单号编码或可查到分片信息。候选键用真实数据模拟高基数、最大/平均分片比、Top 租户占比和未来增长，既看行数也看 QPS、字节和写放大。纯时间键会把新写集中到当前分片，纯租户键会被头部客户压垮；可用 `tenantId + bucket`、时间分区内 hash，或将超大租户独立路由，但这会增加跨桶查询与迁移复杂度。分片键应稳定且不可轻易变更，最好与聚合所有权一致，使同一事务和高频查询局部化；跨分片的唯一约束、排序分页、统计和 join 必须有明确替代方案。还要预留逻辑分片到物理节点的映射层，以便扩容时迁移少量逻辑桶而不改变业务键。验收通过流量回放比较跨片率、倾斜比、热点故障和重分片时间；分片键不是追求绝对均匀，而是在局部性、均衡和可演进之间选择可量化的最低总成本。

**追问点答案**  
- 为什么时间字段通常不是好分片键？  
因为写入往往集中在最新时间段，容易形成热点。  
- 业务主键一定适合分片吗？  
不一定，要看查询是不是也主要按这个键走。  
- 组合分片键有什么价值？  
它可保留租户或时间局部性，同时用 hash/bucket 打散热点；代价是按前缀查询可能跨多个桶，需要汇总与路由能力。

**原理补充**  
分片键设计本质上是在“数据分布均匀性”和“业务访问局部性”之间做取舍。

**架构扩展**  
订单系统常按用户维度或订单号 hash，SaaS 系统常按租户，日志系统常按时间分桶再 hash。成熟架构师不会只说“选主键”，而是会先解释主路径查询和热点分布。

**工程指标**  
- 分片负载均衡度：核心指标。  
- Top 分片 QPS：判断是否有热点。  
- 跨分片查询占比：评估查询成本。  
- 数据倾斜比例：衡量长期风险。  
- 分片扩容复杂度：看未来演进压力。

**证据链**  
- 查询模式统计  
- 写入分布数据  
- 分片模拟结果  
- 热点访问分析

**低风险优化路径**  
1. 先找主写入和主查询路径  
2. 再做分布模拟  
3. 最后为热点和扩容预留演进空间

**易错点**  
- 只追求均匀不看查询  
- 只看当前流量不看增长  
- 不验证热点风险

## 3. 如何避免热点分片？

**核心结论**  
热点分片治理的核心不是让所有分片绝对平均，而是尽量让单点压力不先成为系统瓶颈。

**标准答案**  
避免热点分片时，我通常先区分热点是业务天然集中，还是分片键选型导致的。如果是键设计问题，就要考虑重新 hash、加桶、组合键或者逻辑分片再映射；如果是业务天然热点，比如头部租户、爆款商品、明星主播，就要进一步结合缓存、读写分离、请求合并和热点隔离。之所以这件事重要，是因为分片系统最容易出现“总体容量没满，但某个分片先爆”的情况。面试里如果能补一句“热点分片问题本质上是局部流量分布失衡”，会比较准确。

**高分回答**
我先按分片、租户、key 和 SQL 建立 P95/最大值监控，因为平均 CPU 40% 可能掩盖某分片已 95%。用 `max shard load / median load`、Top 1% key QPS 和队列/锁等待判断是数据量倾斜、读热点还是写热点。读热点优先用多级缓存、请求合并、热点副本和过期抖动吸收，但要限制回源并定义陈旧窗口；写热点不能简单复制，可把计数拆桶后异步聚合、按业务子键分散，或把超级租户迁到专属分片并独立限流。若根因是顺序 ID 或时间路由导致新写集中，则增加 hash/bucket 与逻辑分片映射；迁移采用双读校验、限速复制和路由版本，避免重分布流量本身打垮源库。容量调度为热点保留独立预算，单租户配额防止邻居受损，并准备热点突发时关闭非核心查询。方案要说明一致性代价，例如拆分计数会牺牲实时精确聚合。用真实 Top key 回放、缓存失效和头部租户突发演练，观察最大/中位负载比、热点 TP99、回源量、迁移时间和业务正确性；治理目标是任何单点都不先越过安全水位，而不是所有分片数值完全相等。

**追问点答案**  
- 加机器能解决热点分片吗？  
只有路由能把热点进一步拆分或热点分片可独立迁移时才有效；单个写 key 仍落在一处时，增加普通节点不会改变瓶颈。
- 热点隔离是什么意思？  
把超大租户或超热点对象单独迁到专属分片。  
- 缓存为什么能缓解热点分片？  
因为它能把最集中的读流量挡在数据库前面。

**原理补充**  
热点分片并不是总容量不够，而是局部承载不均导致最先失稳。

**架构扩展**  
很多系统拆完分片后会发现“容量是有了，热点更明显了”。成熟架构师面对这类问题，不会只看平均指标，而是重点看 top 分片、top key、top 租户。

**工程指标**  
- Top 分片 CPU/QPS：最关键监控。  
- 分片倾斜比：看均衡性。  
- 热点租户占比：识别业务集中度。  
- 缓存命中率：缓解效果指标。  
- 分片迁移次数：反映长期稳定性。

**证据链**  
- 分片监控  
- 业务租户分布  
- 热点 key 统计  
- 迁移和扩容记录

**低风险优化路径**  
1. 先识别热点类型  
2. 再做缓存和读流量吸收  
3. 最后考虑逻辑重分布或热点隔离

**易错点**  
- 只看平均分片负载  
- 热点出现后只扩容不重分布  
- 不区分设计问题和业务天然热点

## 4. 读写分离的一致性问题怎么解决？

**核心结论**  
读写分离的一致性问题，本质是主库写入和从库可见之间存在复制延迟窗口，需要按业务重要性分级处理。

**标准答案**  
做读写分离时，我一般会先明确哪些读请求必须读到最新结果，哪些可以接受短暂延迟。之所以会有一致性问题，是因为主库提交成功不代表从库瞬间可见，复制延迟会让“刚写完就读”读到旧数据。工程上常见的解决手段包括关键链路强制读主、写后短时间内读主、基于会话或请求上下文做读路由、延迟敏感操作绕过从库，以及把从库只用于报表和弱一致查询。面试里如果能补一句“读写分离不是白送的性能，它是拿一致性窗口换吞吐”，会比较成熟。

**高分回答**
我会先给每类读定义语义：商品详情允许秒级陈旧，订单写后读要求 read-your-writes，资金判断必须读取权威状态。写成功后数据库返回提交位点，例如 GTID/LSN；同一会话后续读可以选择已追上该位点的副本，等待上限内未追上则回主库，而不是粗暴固定“5 秒内读主”。无法传位点时可用短期 session 标记，但要根据复制延迟动态调整。关键决策和条件更新直接在主库事务内完成，不能先从副本读库存再据此扣减。路由持续监控每个副本的 apply lag、错误和回放状态，超过业务窗口自动摘除；主库故障或复制异常时，低价值读可返回带版本的缓存，高一致读安全失败或降级查询。强制读主比例必须受容量预算约束，否则高峰时一致性策略会把主库打满。演练复制暂停、网络抖动、主从切换与旧副本误路由，统计写后旧读事件、位点等待 P99、回主比例、主库水位和关键业务误判数。读写分离的正确做法是让每次读明确知道允许多旧，并用提交位点证明是否满足，而不是统一“读从库”。

**追问点答案**  
- 为什么不能所有读都走从库？  
因为很多关键业务对最新状态敏感。  
- 写后读主怎么实现？  
优先传播写事务的 GTID/LSN，选择已回放到该位点的副本，等待超时则读主；退化方案才是按会话设置与实际复制延迟匹配的短期读主标记。
- 主从延迟高时最危险的是什么？  
订单、支付、库存这类状态判断错误。

**原理补充**  
读写分离的核心代价，就是把单机读取压力转嫁为“可见性延迟”的管理问题。

**架构扩展**  
商品详情、资讯内容、报表查询通常更适合从库；订单状态、支付结果、库存确认则常要读主。成熟架构师会用业务等级做路由，而不是统一策略。

**工程指标**  
- 主从复制延迟：核心指标。  
- 强制读主比例：衡量一致性成本。  
- 写后读旧值事件数：真实质量信号。  
- 从库查询占比：吞吐收益指标。  
- 关键链路判断错误率：业务风险指标。

**证据链**  
- 复制延迟监控  
- 路由策略配置  
- 写后读日志样本  
- 业务异常案例

**低风险优化路径**  
1. 先给读请求按一致性分级  
2. 再设计强制读主和弱一致读从规则  
3. 最后监控复制延迟和误判事件

**易错点**  
- 所有读统一走从库  
- 不做一致性分级  
- 不监控写后读问题

## 5. 慢 SQL 治理闭环是什么？

**核心结论**  
慢 SQL 治理不是改一条语句，而是“发现、定位、优化、验证、固化”的持续闭环。

**标准答案**  
做慢 SQL 治理时，我通常先通过慢日志、APM、数据库监控把高频慢 SQL 找出来，再结合 `EXPLAIN`、执行时间分布、扫描行数、返回行数和锁等待分析根因。之所以不能直接“先加索引”，是因为慢 SQL 可能来自错误索引、SQL 写法、数据分布变化、锁竞争、回表、函数计算或业务查询模型本身不合理。工程上更稳的闭环是：先识别问题 SQL，再做最小改动优化，最后压测验证和线上观察，必要时把规范沉淀成 SQL 评审规则。面试里如果能补一句“慢 SQL 治理的目标不是把一条 SQL 变快，而是把同类问题减少”，会比较有治理思维。

**高分回答**
我会先按“总资源消耗”而不只是单次最慢排序：对 SQL 归一化指纹，结合 QPS、P95/P99、扫描/返回行比、CPU/IO、锁等待和业务等级计算优先级，一条 200ms 但每秒 2000 次的 SQL 往往比偶发 5 秒报表更危险。定位时用真实参数执行 `EXPLAIN ANALYZE` 或等价运行证据，核对估算与实际行数、索引选择、回表、排序临时表、锁和数据倾斜；同时查看调用 trace，排除 N+1、无界分页和事务过长等上层模型问题。优化从风险最小处开始：改查询边界、批量化、调整联合索引与列顺序，必要时重构数据模型或异步报表；新增索引要计算写放大、空间、buffer pool 和在线 DDL 风险，并灰度观察。验证使用生产规模与冷热分布，比较结果正确性、TP99、扫描行、数据库水位和写入副作用，保留回滚。上线后监控计划漂移和数据增长，绑定 SQL owner 与性能预算；将全表扫、无界查询、危险 DDL 和 N+1 检查纳入 CI/评审。闭环以慢 SQL 总消耗、SLO 和复发率改善为准，而不是某次执行计划看起来用了索引。

**追问点答案**  
- 为什么 `EXPLAIN` 不是全部真相？  
普通 `EXPLAIN` 主要给优化器估算与选择，可能受统计信息和参数分布误导；还需真实参数的运行行数、耗时、锁等待和资源数据验证。
- 慢 SQL 一定是索引问题吗？  
不一定，锁等待和数据模型问题也很常见。  
- 治理后为什么还要持续观察？  
因为数据量和访问模式变化后计划可能再次恶化。

**原理补充**  
慢 SQL 治理的本质，是让数据库执行成本与业务查询价值重新对齐。

**架构扩展**  
很多系统的问题不是单条 SQL 写得差，而是接口一次请求发十几条 SQL、分页方式错误、模糊查询无边界。成熟架构师会把 SQL 治理和接口模型、缓存策略、索引设计一起看。

**工程指标**  
- 慢 SQL 数量：治理规模。  
- TP95/TP99 查询耗时：性能结果。  
- 扫描行数：执行成本核心指标。  
- 锁等待时间：识别并发瓶颈。  
- 优化后回归率：治理稳定性。

**证据链**  
- 慢日志  
- `EXPLAIN` 结果  
- 执行统计  
- 接口 tracing 和数据库监控

**低风险优化路径**  
1. 先找高频高影响 SQL  
2. 再最小改动优化  
3. 最后把经验沉淀进规范和评审

**易错点**  
- 上来就加索引  
- 不看真实频次和业务价值  
- 优化完不验证回归

## 6. 二级索引与覆盖索引如何取舍？

**核心结论**  
二级索引的价值在于缩小检索范围，覆盖索引的价值在于避免回表，取舍关键看查询模式和写入成本。

**标准答案**  
我做索引设计时，一般先看查询到底需要哪些字段。如果一个查询只需要索引中已有的列，那做成覆盖索引就能避免回表，通常会明显减少 IO；如果查询列很多，强行做覆盖索引会让索引非常宽，写入成本和维护成本又会升高。之所以这件事要权衡，是因为索引不是越多越全越好，每加一个索引都在给写入、存储和变更成本加码。面试里如果能补一句“覆盖索引是用更多索引空间换更少回表 IO”，会很准确。

**高分回答**
我会从具体谓词、排序和返回列推导联合索引，而不是抽象地问“要不要覆盖”。例如高频查询 `WHERE tenant_id=? AND status=? ORDER BY created_at DESC LIMIT 20`，候选索引可按等值列、排序列组织为 `(tenant_id, status, created_at, id)`；若列表只返回这些字段即可覆盖，避免二级索引找到主键后逐行回聚簇索引。是否值得覆盖要用 QPS、实际扫描/返回行数、回表次数和 buffer pool 命中证明：低频查询或一次只回表 1 行，收益可能不足以抵消成本。把大文本、低稳定性字段塞进索引会降低每页记录数、扩大内存与磁盘、增加页分裂和每次 INSERT/UPDATE 的写放大；多个前缀相同的冗余索引也会重复维护。还要验证最左前缀、范围列后的利用限制、排序方向和选择性，使用真实参数的 `EXPLAIN ANALYZE`，而不是只看到 `Using index` 就宣告成功。上线通过在线 DDL/灰度建立，观察查询 TP99、rows examined、索引大小、写入延迟和 buffer pool，收益不达标即可删除新索引。覆盖索引是为稳定高频访问路径做的有预算冗余，不是让每条查询都“零回表”。

**追问点答案**  
- 为什么回表会慢？  
因为先查二级索引，再回主键索引取完整数据，会多一次访问。  
- 覆盖索引适合所有查询吗？  
不适合；只有高频、返回列稳定且回表成本有证据时才值得，单行主键回表或宽字段查询通常收益有限。
- 为什么索引太宽不好？  
会增加存储、写放大和缓存压力。

**原理补充**  
索引设计的本质，是在读性能和写维护成本之间持续做平衡。

**架构扩展**  
高频接口、列表页、分页查询特别适合考虑覆盖索引；而写多场景、字段变化快的表则要更保守。成熟架构师不会只为了一个查询快一点就把索引设计成巨无霸。

**工程指标**  
- 回表比例：衡量优化空间。  
- 索引大小：评估成本。  
- 查询 TP99：看读收益。  
- 写入延迟：观察副作用。  
- Buffer pool 命中率：索引宽度会影响缓存效果。

**证据链**  
- `EXPLAIN` 执行计划  
- 查询字段列表  
- 索引定义  
- 读写性能对比

**低风险优化路径**  
1. 先锁定高频查询  
2. 再决定是否值得做覆盖  
3. 最后评估写成本和索引体积

**易错点**  
- 为了避免回表盲目加宽索引  
- 不看写入副作用  
- 不了解查询字段稳定性

## 7. MySQL 事务隔离级别如何选？

**核心结论**  
事务隔离级别不是越高越安全越好，而是要在并发性能和一致性要求之间做匹配。

**标准答案**  
选择事务隔离级别时，我一般先看业务是否允许脏读、不可重复读或幻读，再评估更高隔离级别带来的锁竞争和吞吐损失。之所以不能默认所有场景都上最高级别，是因为隔离级别越高，通常并发性能成本越大。工程上大多数 MySQL InnoDB 场景默认 `REPEATABLE READ` 已经能兼顾一致性和性能，但在某些读写模型清晰、并发高且可接受一定读变化的场景，也可能采用 `READ COMMITTED`。面试里如果能补一句“隔离级别选择本质上是异常类型容忍度的业务决策”，会比较成熟。

**高分回答**
我会先写业务不变量和事务模式，再选隔离级别。报表分页允许每条语句看到较新快照，可在 READ COMMITTED 下减少某些间隙锁影响；一次结算中多次读取必须基于同一视图，可用 REPEATABLE READ；“值班医生至少一人”这类跨行约束即使快照读可重复，也可能发生 write skew，需要显式锁定约束范围、唯一/检查约束或 SERIALIZABLE。MySQL InnoDB 下还要区分快照读与 `SELECT ... FOR UPDATE` 当前读：RR 通过 MVCC 提供一致性快照，并对范围当前读使用 next-key lock 防幻写，但这会扩大锁范围；RC 每条语句新快照、通常减少 gap lock，却不能自动保证业务逻辑。任何级别都不能替代正确的条件更新，例如扣库存用 `UPDATE ... SET stock=stock-1 WHERE id=? AND stock>0`，以受影响行数守住不变量。评估真实并发下的锁等待、死锁、事务时长、TPS 和异常结果，长事务先缩短再考虑降级隔离。变更隔离级别需灰度并验证复制/应用假设与重试语义。结论不是默认 RR 或一味 SERIALIZABLE，而是用最小协调成本阻止业务不可接受的并发异常。

**追问点答案**  
- 为什么不是都用串行化？  
因为性能代价太高，很多业务不值得。  
- `READ COMMITTED` 常见在哪？  
一些高并发 OLTP 场景会用它减少 gap lock 影响。  
- 默认级别安全吗？  
不能笼统说安全；RR 仍需区分快照读、当前读和跨行不变量，必须用并发测试及数据库约束证明具体业务不会出现丢失更新或 write skew。

**原理补充**  
事务隔离级别本质上是在读一致性和并发吞吐之间设定不同强度的约束。

**架构扩展**  
成熟架构师谈隔离级别时，不会只背四个英文名，而是会继续说明自己的业务最怕哪类异常，以及为什么愿意为避免它付出多大性能代价。

**工程指标**  
- 锁等待时间：隔离级别副作用核心指标。  
- 死锁数：高隔离并发下常见。  
- 事务平均耗时：性能结果。  
- 异常读事件数：业务质量指标。  
- TPS：整体吞吐变化。

**证据链**  
- 业务一致性要求  
- 锁监控  
- 死锁日志  
- 事务性能对比

**低风险优化路径**  
1. 先确认业务容忍的异常类型  
2. 再选隔离级别  
3. 最后压测验证锁冲突和吞吐

**易错点**  
- 默认最高隔离最安全  
- 不理解异常类型  
- 不验证性能副作用

## 8. 缓存与数据库一致性怎么做？

**核心结论**  
缓存与数据库一致性的核心，不是追求绝对零延迟一致，而是根据业务级别控制不一致窗口并提供补偿手段。

**标准答案**  
做缓存与数据库一致性时，我通常先问这条数据到底有多敏感。如果是普通详情、列表或配置读取，通常采用旁路缓存模式，也就是读缓存、缓存 miss 回源、写数据库后删缓存；如果是一致性要求更高的链路，就要进一步补延迟双删、消息通知、binlog 订阅或强制绕过缓存。之所以这件事难，是因为缓存和数据库本来就是两个独立系统，不可能天然原子一致。工程上真正重要的是定义不一致窗口、回源保护、补偿机制和监控，而不是迷信某个固定模板。面试里如果能补一句“缓存一致性问题的本质是两个存储副本的同步问题”，会更有深度。

**高分回答**
我先按数据语义分级：价格展示可允许秒级陈旧，库存扣减和余额判断必须以数据库/账本原子操作为准，缓存只做展示或前置削峰。普通读采用 cache-aside：miss 后读库并带 TTL 写缓存，更新在数据库事务提交后失效缓存；若先删缓存再写库，间隙中的读会回填旧值。即使“写库后删缓存”，删除失败仍会留下旧值，因此通过 outbox 或 binlog/CDC 可靠发布失效事件，消费者按数据版本幂等删除/刷新，失败可重试并进入死信；TTL 是最终自愈上界而不是唯一保障。并发旧读可能在删除后晚到并覆盖新值，缓存值携带数据库版本，写入时只接受更高版本，或对中低风险场景用延迟双删缩短窗口。热点 miss 使用 singleflight、回源限流和旧值降级，防一致性修复反而打垮数据库。监控不能只看命中率，要抽样比较缓存与事实源版本，记录失效事件延迟、删除失败、脏值 P99 与用户影响；演练 CDC 停机、Redis 故障、乱序事件和缓存重建。目标是明确事实源、允许多旧、如何检测与收敛，而不是声称两个独立存储能绝对同步。

**追问点答案**  
- 为什么通常是更新 DB 后删缓存？  
数据库先提交可保证缓存 miss 回源看到新事实；但删除仍可能失败或被慢旧读回填，所以还需可靠失效事件、版本校验和 TTL 兜底。
- 延迟双删解决什么？  
解决并发读把旧值重新回填进缓存的问题。  
- 强一致场景还用缓存吗？  
可以用，但关键判断常要绕过缓存或缩小缓存职责。

**原理补充**  
缓存一致性的关键不是有没有一种完美方案，而是有没有让不一致可控、可观测、可补救。

**架构扩展**  
商品详情、价格展示、用户画像、风控标签都用缓存，但敏感程度完全不同。成熟架构师会按业务重要性给缓存链路分层，而不是一套旁路缓存走天下。

**工程指标**  
- 缓存命中率：读收益指标。  
- 回源比例：稳定性指标。  
- 不一致窗口时长：核心一致性指标。  
- 删除缓存失败率：高风险信号。  
- 用户可见错误数：业务感知结果。

**证据链**  
- 写链路日志  
- 缓存删除记录  
- 回源监控  
- 对账和异常案例

**低风险优化路径**  
1. 先给数据按一致性分级  
2. 再选删除缓存或订阅刷新策略  
3. 最后补对账和监控

**易错点**  
- 追求“绝对一致”口号化  
- 一套模板套所有场景  
- 不监控不一致窗口

## 9. 延迟双删什么时候有效？

**核心结论**  
延迟双删的价值在于降低并发读把旧值重新写回缓存的概率，但它不是万能一致性方案。

**标准答案**  
我通常在这种场景考虑延迟双删：写请求先更新数据库并删一次缓存，之后等待一个足以覆盖潜在并发读回填窗口的时间，再删第二次缓存。之所以要这么做，是因为第一次删缓存之后，可能正好有一个并发读从旧库或旧事务视图里读到了旧值，又把它重新写回缓存。延迟双删能降低这个问题发生的持续时间，但它并不能从根上解决所有一致性问题。工程上如果业务更敏感，还是应该考虑消息驱动刷新、binlog 订阅或关键链路读主。面试里如果能补一句“延迟双删是降低风险，不是提供强一致承诺”，会很准确。

**高分回答**
延迟双删只针对一个具体竞态：读请求在更新提交前已读到旧值，但在更新提交并第一次删缓存后才把旧值写回缓存；第二次删除若晚于这次最慢旧回填，就能缩短脏值存活。延迟应根据读库 P99、网络、序列化、线程排队和副本延迟估算并留余量，不能固定抄 500ms；延迟越长脏值窗口越大，越短又覆盖不了慢请求。第二次删除必须进入持久化延迟任务或可靠消息，带业务键、版本、有限重试和告警，若只在应用线程 sleep，进程重启就丢失。它仍无法消除删除两次都失败、超长暂停、乱序刷新或读副本严重落后，也不能给资金/库存提供强一致。因此中低风险展示数据可把它作为 cache-aside 的增强，高风险链路应由权威存储判断，并用 CDC/outbox 失效、版本化写入和对账闭环。验证要构造慢读与并发写的竞态，统计二删成功率、旧值回填数、脏值持续 P99、任务积压与数据库回源，而不是看到代码执行两次 delete 就认为有效。

**追问点答案**  
- 第二次删除延迟多久合适？  
至少覆盖旧读从取数到完成回填的 P99/P99.9，包括副本延迟、线程排队和网络，并用竞态压测校准；业务无法容忍该窗口时就不该只用双删。
- 为什么它不是万能方案？  
因为仍然可能存在删除失败、延迟选错或复杂并发场景。  
- 哪类业务不建议只靠它？  
库存、余额、支付状态这类高敏感链路。

**原理补充**  
延迟双删本质上是在用第二次机会缩短脏缓存停留窗口。

**架构扩展**  
很多团队把延迟双删背得很熟，但真正上线时会发现：时间窗口不好选、删除可能失败、消息链路也可能抖。成熟架构师会把它当成中低风险链路的增强策略，而不是顶级方案。

**工程指标**  
- 二次删除成功率：执行质量。  
- 脏缓存持续时长：核心效果指标。  
- 回填旧值事件数：观察问题是否仍存在。  
- 用户查询异常率：业务感知结果。  
- 删除任务积压量：运维风险。

**证据链**  
- 双删执行日志  
- 脏读样本  
- 写后读 tracing  
- 删除失败告警

**低风险优化路径**  
1. 先确认链路容忍度  
2. 再用延迟双删缩小窗口  
3. 最后给高风险链路补更强方案

**易错点**  
- 把延迟双删当强一致  
- 延迟时间拍脑袋  
- 不监控二次删除效果

## 10. Redis 大 Key 如何治理？

**核心结论**  
大 Key 治理的重点不只是找出来，而是避免它在读写、删除和迁移时形成长尾抖动。

**标准答案**  
治理 Redis 大 Key 时，我通常先识别它的类型，是大字符串、大 hash、大 set 还是大 list，然后再看它是业务建模不当、长期堆积还是热点聚合造成的。之所以大 Key 危险，是因为单次读取、序列化、网络传输和删除都可能变慢，严重时还会阻塞主线程。工程上常见治理手段包括拆分 key、分页读取、异步删除、限制单 key 元素规模和对热点聚合改建模。面试里如果能补一句“大 Key 问题本质是把本应分散的数据操作集中成了单点大开销”，会比较准确。

**高分回答**
我先定义与实例规格匹配的阈值，例如 string 字节数、集合元素数、单命令返回字节和预计删除耗时，再用内存采样、`SCAN`/渐进式检查、slowlog、latency monitor 与网络流量定位，禁止生产上用阻塞式全库 `KEYS` 或一次性导出。风险不只占内存：`HGETALL/SMEMBERS` 会长时间占用单线程并产生大网络包，复制与 AOF 重写放大，迁移或故障切换时也会阻塞。按类型治理：大 string 拆成压缩分块或转对象存储；大 hash/set 按业务 ID hash 分桶并用 `HSCAN/SSCAN` 分页；时间序列按时间窗口滚动；聚合结果只保存必要字段。迁移用新旧 key 版本、双读校验和渐进回填，避免一次拆分造成缓存雪崩；删除先 `UNLINK` 或分批移除，确认内存释放后台任务不会冲击延迟。为 key schema 设置 owner、TTL、最大元素和 CI/运行时守卫，超过阈值拒绝继续聚合或报警。演练读取、过期、删除、复制和节点迁移，观察主线程延迟 P99、最大响应字节、复制 lag、内存碎片和清理时长；真正闭环是改建模并防止大 key 再生，不是扫描后手工删一次。

**追问点答案**  
- 为什么删除大 Key 会卡？  
因为释放大量元素和内存可能阻塞 Redis 主线程。  
- 大 Key 一定是坏设计吗？  
很多时候是，也有少数业务天然聚合，但仍要控制规模。  
- 怎么发现大 Key？  
先设按类型阈值，再用渐进式采样、内存分析、slowlog/latency 和网络响应大小交叉确认；避免在线执行 `KEYS` 或全量重命令造成二次故障。

**原理补充**  
Redis 大 Key 问题本质上是单线程模型下的“单次操作过重”。

**架构扩展**  
排行榜、用户全量行为、群成员列表、超大配置集合都容易出现大 Key。成熟架构师做缓存建模时，会很早限制单 key 规模，而不是等 Redis 变慢后再查。

**工程指标**  
- 大 Key 数量：问题规模。  
- 单次命令耗时：直接性能指标。  
- 网络包大小：传输成本。  
- Redis 主线程阻塞时间：高风险指标。  
- 异步删除完成时长：治理效果。

**证据链**  
- 内存扫描结果  
- 慢命令日志  
- key 类型分布  
- 业务建模说明

**低风险优化路径**  
1. 先识别大 Key 类型和来源  
2. 再拆分或分页化访问  
3. 最后用异步删除和建模约束防回潮

**易错点**  
- 只删不改建模  
- 不区分类型  
- 忽略删除阻塞风险

## 11. 缓存雪崩如何防护？

**核心结论**  
缓存雪崩的核心风险，不是某个 key 失效，而是大量 key 同时失效导致回源洪峰。

**标准答案**  
防缓存雪崩时，我通常会先在过期时间上打散，避免一批 key 同时失效；同时对热点链路配本地缓存、回源限流、请求合并和降级兜底。之所以雪崩危险，是因为它表面看是缓存问题，实际打崩的往往是数据库和下游依赖。工程上如果业务特别关键，还会做缓存预热、永不过期加主动刷新，或者对关键结果保留静态兜底。面试里如果能补一句“雪崩问题本质上是缓存层失去削峰作用后，系统瞬间暴露在真实流量下”，会比较到位。

**高分回答**
我会先量化最坏回源：正常 10 万 QPS、命中率 95% 时数据库约承受 5000 QPS，若一批 key 同时失效，回源可能瞬间放大 20 倍，必须让数据库安全容量而非 Redis 容量决定保护线。预防分三层：TTL 使用业务基础时间加随机抖动，避免同批发布或预热产生相关过期；头部 key 采用逻辑过期和后台提前刷新，本地缓存保留带最大陈旧度的旧值；回源端使用 singleflight、全局/分片并发上限和有界队列，数据库接近水位时直接降级，不能让所有 miss 排队。缓存集群整体故障还需熔断 Redis 快速失败、防客户端重连风暴，并按业务等级返回静态/旧数据或拒绝高风险写；恢复时先预热 Top key，再阶梯放流，避免冷缓存二次雪崩。批量失效、配置发布和大促前预热都要有速率控制与回滚。演练同一时间过期 30%、Redis 分片故障和全缓存冷启动，观察回源 QPS 上限、数据库 TP99/连接池、旧值命中、降级正确率与恢复时间。随机 TTL 只是降低相关性，真正的防线是保证缓存失效时下游仍被限在安全容量内。

**追问点答案**  
- 为什么过期时间随机化有效？  
因为它能把同一时刻的失效请求分散开。  
- 只有 Redis 会雪崩吗？  
不是；任何承担削峰作用的 CDN、本地缓存、配置或查询副本若相关失效，都可能把隐藏流量同步传给下游，需按端到端回源预算治理。
- 雪崩和击穿的区别是什么？  
雪崩是大量 key 同时失效，击穿更偏单个热点 key 失效。

**原理补充**  
缓存雪崩本质上是失效时间高度相关，导致回源流量同步爆发。

**架构扩展**  
活动页、首页推荐、统一配置、热点列表最容易出现雪崩。成熟架构师面对雪崩不会只给 Redis 加机器，而是会多层次做“打散、挡流、回源保护、降级”。

**工程指标**  
- 同时失效 key 数：风险规模。  
- 回源 QPS：核心冲击指标。  
- 数据库错误率：下游承压结果。  
- 请求合并命中率：缓解效果。  
- 降级命中率：兜底能力。

**证据链**  
- key 过期分布  
- 回源曲线  
- 缓存命中率变化  
- 数据库压力监控

**低风险优化路径**  
1. 先打散过期时间  
2. 再做回源限流和多级缓存  
3. 最后给关键链路配降级兜底

**易错点**  
- 只做随机过期不做回源保护  
- 不看下游冲击  
- 误把雪崩和击穿混为一谈

## 12. 缓存击穿如何防护？

**核心结论**  
缓存击穿的重点是保护单个高热点 key 失效后不会被大量并发请求同时打穿。

**标准答案**  
应对缓存击穿时，我通常会先识别是不是热点 key，然后针对热点 key 做永不过期加异步刷新、互斥回源、逻辑过期或请求合并。之所以击穿和雪崩不同，是因为它往往只有一个或少数几个 key，但这些 key 的并发量特别高，一旦失效就会瞬间把数据库打满。工程上最有效的往往不是简单缩短 TTL，而是把“谁去回源”“其余人怎么等”“失败时怎么兜底”设计清楚。面试里如果能补一句“击穿问题本质是热点读流量集中冲击单个缺失点”，会比较完整。

**高分回答**
我会先用 Top key QPS 与 miss 突增识别真正热点，例如单商品 2 万 QPS，而数据库同查询安全能力只有 300 QPS，则一次过期不能让 2 万请求同时回源。可接受旧值的详情页采用逻辑过期：缓存保存数据版本和过期时间，首个请求获得短租约后异步刷新，其余请求继续读带陈旧标识的旧值；严格要求新值时使用进程/分布式 singleflight，只允许一个回源，其余等待有 deadline，超时降级而非无限堆积。锁必须带 owner、TTL 和续约边界，回源失败不能缓存错误结果，且要限制全局回源并发，防多个热点同时失效。热点 key 提前刷新并加入 TTL 抖动，但“永不过期”仍需可靠刷新、版本校验和内存治理。若数据库或刷新任务故障，按业务语义返回最后已知值、部分字段或安全失败。演练热点过期、刷新线程宕机、锁持有者暂停与数据库变慢，验证单 key 回源并发上限、等待 P99、旧值年龄、回源失败率和数据库水位。击穿治理不是简单加锁，而是明确谁刷新、其余请求如何有界等待、刷新失败如何继续服务。

**追问点答案**  
- 互斥锁回源有什么问题？  
锁持有者变慢或崩溃会让大量请求排队，锁本身也可能过期双持有；必须设置 owner/TTL、等待 deadline、全局并发上限和旧值降级。
- 永不过期是不是最好？  
不一定，还要解决数据刷新和一致性问题。  
- 逻辑过期适合什么场景？  
适合可接受短暂旧值但需要高可用的热点数据。

**原理补充**  
缓存击穿的核心是“热点集中 + 缺失瞬间”，而不是普通缓存 miss。

**架构扩展**  
商品详情、直播状态、热门配置、首页榜单都可能是热点 key。成熟架构师会提前识别热点对象，而不是等数据库被打穿后才追 Redis miss。

**工程指标**  
- 热点 key QPS：识别风险核心指标。  
- 单 key miss 次数：击穿信号。  
- 互斥回源等待时长：治理成本指标。  
- 回源成功率：保护效果。  
- 数据库单点负载：下游结果指标。

**证据链**  
- Top key 访问统计  
- miss 日志  
- 回源锁监控  
- 数据库热点查询记录

**低风险优化路径**  
1. 先识别热点 key  
2. 再做互斥刷新或逻辑过期  
3. 最后补失败兜底和多层缓存

**易错点**  
- 把击穿当普通 miss  
- 只加锁不做超时和降级  
- 不提前识别热点对象

## 13. 缓存穿透如何防护？

**核心结论**  
缓存穿透的本质是请求绕过缓存不断访问本不存在的数据，防护重点是尽早识别和拦截无效请求。

**标准答案**  
应对缓存穿透时，我通常会从两个方向下手。第一是入口防护，比如参数校验、黑白名单、布隆过滤器；第二是回源结果缓存，比如对明确不存在的数据做短 TTL 空值缓存。之所以穿透危险，是因为它绕过了缓存系统的保护，把大量无效请求直接打到数据库。工程上要区分这是恶意攻击还是业务脏请求，因为前者更偏安全防护，后者更偏数据治理。面试里如果能补一句“穿透问题不是缓存没命中，而是缓存根本没机会发挥作用”，会更准确。

**高分回答**
我会先按来源、参数形态和无结果 SQL 指纹区分恶意枚举、客户端 bug 与正常查询长尾。入口做格式、范围、租户权限和速率校验，让明显非法 ID 不进入缓存；对巨大存在集合可用布隆过滤器快速拒绝“确定不存在”的请求，但要正确理解它无假阴性、可能假阳性，位图大小和 hash 数根据元素数及目标误判率计算，并处理新增、重建与删除语义。数据库确认不存在后缓存空值，key 必须包含租户与查询条件，TTL 较短并加抖动；新建数据时主动删除对应空值，避免短时间“幽灵不存在”。布隆过滤器误判或空值 miss 仍会回源，因此数据库前保留每租户限流、singleflight 和熔断；攻击流量联动 WAF/账号风控，不能让海量随机 key 把空值缓存撑爆。定期分析无结果高频来源并修复上游脏数据。演练随机 ID 扫描、合法数据刚创建、过滤器重建和缓存故障，观察数据库无结果 QPS、过滤器拦截/假阳性、空值命中、内存增长与正常请求误杀率。防穿透的核心是用比数据库查询更便宜且有容量边界的“不存在”判定链。

**追问点答案**  
- 布隆过滤器为什么适合？  
因为它能快速判断“大概率不存在”，挡住很多无效请求。  
- 空值缓存有什么风险？  
除新建数据短暂不可见外，随机恶意 key 还会撑大缓存；需要短 TTL、容量限制，并在数据创建后主动失效对应空值。
- 为什么参数校验也算治理？  
因为很多穿透是明显无效请求，本该在更前面拦下。

**原理补充**  
缓存穿透的核心，是请求路径没有对“不存在”建立足够便宜的判定机制。

**架构扩展**  
恶意爬虫、伪造 ID、错误参数、脏链路数据都可能造成穿透。成熟架构师面对这类问题，会同时拉安全、业务和缓存团队一起治理。

**工程指标**  
- 不存在 key 请求占比：核心识别指标。  
- 空值缓存命中率：治理效果。  
- 布隆过滤器拦截率：前置防护能力。  
- 数据库无结果查询 QPS：直接压力指标。  
- 异常来源 IP/用户分布：帮助区分攻击与脏流量。

**证据链**  
- 参数样本  
- 无结果查询日志  
- 布隆过滤器统计  
- 安全访问日志

**低风险优化路径**  
1. 先在入口做参数和来源校验  
2. 再补布隆过滤器或空值缓存  
3. 最后联动安全策略和数据治理

**易错点**  
- 把穿透和击穿混淆  
- 只做空值缓存不做前置拦截  
- 不区分恶意流量和脏流量

## 14. ES 适合什么查询场景？

**核心结论**  
ES 更适合复杂检索、全文搜索、聚合分析和多字段组合查询，不适合承担高一致性核心事务存储。

**标准答案**  
我通常会在这些场景考虑 ES：全文搜索、多字段组合过滤、复杂排序、聚合统计、模糊匹配和近实时检索。之所以它适合，是因为倒排索引能很好支持文本和组合查询，而如果把这类需求硬压在 MySQL 上，索引设计和查询性能都会越来越吃力。工程上 ES 更像查询加速层和检索引擎，而不是主交易库，因此关键业务状态通常仍以 MySQL 为准。面试里如果能补一句“ES 擅长的是找信息，不是保真相”，会很有记忆点。

**高分回答**
我会从查询形态而不是数据量决定是否上 ES。它适合分词全文检索、相关性排序、模糊/同义词、多字段过滤和近实时聚合，例如商品搜索既要关键词匹配又要品牌、类目、价格筛选；简单主键查询、强事务更新、跨记录不变量和立即一致读仍由 MySQL 等事实源负责。设计前明确文档模型：为查询反范式聚合字段，`text` 与 `keyword` 区分全文和精确过滤，避免动态 mapping 爆炸；高基数深度聚合、前导通配符和深分页可能消耗巨大，应改用预聚合、search_after、字段限制与查询配额。写入经 CDC 异步构建索引，因此承诺的是 refresh 加同步延迟后的近实时可见性，用户关键判断不能依赖刚写入 ES。分片数按数据量、查询并发和恢复时间估算，过多小分片同样浪费资源；索引模板、alias 与版本化重建支持 mapping 变更和回滚。用真实语料评估 relevance（如 NDCG/零结果率）以及 P95/P99、索引延迟、拒绝率、heap/segment 和恢复时间。只有复杂检索收益明显超过同步、运维和一致性成本时才引入，ES 是可重建的查询投影，不是万能主库。

**追问点答案**  
- 为什么不能把 ES 当主库？  
因为它在事务一致性和写入语义上不如关系库适合核心交易。  
- ES 适合实时性很强的场景吗？  
通常只能承诺 refresh 与同步链路共同决定的近实时窗口；若写后必须立即参与业务判断，应查事实源或设计读主兜底。
- 什么时候 MySQL 比 ES 更合适？  
高一致性写入、简单主键查询、事务性 OLTP 场景。

**原理补充**  
ES 的价值在于用倒排和分布式检索模型提升复杂查询能力，而不是替代所有数据存储。

**架构扩展**  
商品搜索、内容检索、日志分析、用户画像筛选特别适合 ES；订单扣减、账户余额、支付状态则不适合把 ES 放在真相位。成熟架构师通常把 ES 设计成查询副本而非主事实源。

**工程指标**  
- 查询延迟：核心体验指标。  
- 索引刷新延迟：近实时能力指标。  
- 聚合耗时：分析场景重点。  
- 写入吞吐：索引构建压力。  
- 查询命中率与错误率：稳定性结果。

**证据链**  
- 查询模式统计  
- ES 索引配置  
- 查询慢日志  
- 写入与刷新监控

**低风险优化路径**  
1. 先确认查询确实复杂到关系库吃力  
2. 再把 ES 定位为检索层  
3. 最后设计主库和索引同步关系

**易错点**  
- 把 ES 当万能数据库  
- 忽略近实时延迟  
- 不区分查询副本和主事实源

**核心结论**  
MySQL 与 ES 双写一致性的重点不是追求同步双写零误差，而是让主库写入成功后，索引侧可追踪、可补偿地收敛。

**标准答案**  
我一般不会直接在业务代码里同步双写 MySQL 和 ES，因为这样既放大主链路时延，也会把两个系统的失败处理纠缠在一起。更稳的做法通常是以 MySQL 为主事实源，通过 binlog 订阅、消息队列或变更事件把数据异步同步到 ES。之所以这样做，是因为 ES 主要承担读路径和检索能力，允许存在一个可控的近实时窗口。工程上关键不是“有没有延迟”，而是“同步失败能不能发现、重放和修复”。面试里如果能补一句“主库保真相，ES 保可查，双写一致性的核心是可恢复”，会比较成熟。

**高分回答**
我会明确 MySQL 是唯一事实源，业务事务只写 MySQL；通过 binlog CDC 或事务内 outbox 产生带主键、操作类型、提交版本和 schema 版本的事件，异步更新 ES。消费者按文档版本做外部版本控制或条件更新，重复事件幂等、旧事件不能覆盖新状态；删除使用 tombstone，避免漏删后文档永久残留。同步链路有分区顺序、有限重试、死信与可从位点重放，ES 不可用时不阻塞交易写入，但 lag 超过业务 SLA 时搜索页面标记延迟、降级到主库的有限查询或暂停依赖新状态的功能。mapping 变更采用新索引全量回填，在固定快照位点接增量，持续影子查询/对账后原子切 alias，旧索引保留回滚窗口。对账既比较行数和校验和，也抽样业务字段、删除状态与搜索结果；差异可从主库重新构建，不以 ES 反写真相。演练 CDC 中断、乱序重复、死信积压、mapping 拒绝和 alias 回退，监控端到端可见延迟 P99、位点 lag、失败/死信、差异率与重建 RTO。完整方案追求可检测、可重放、可重建的收敛，而不是无法原子保证的同步双写。

**追问点答案**  
- 为什么不建议同步双写？  
因为会增加主链路耦合和失败处理复杂度。  
- binlog 订阅有什么优势？  
能把数据库变更当成统一事实源。  
- 如果同步失败怎么办？  
按错误分类有限重试，永久失败进带 owner/SLA 的死信；从已提交位点重放并以版本幂等，独立对账从 MySQL 重建差异，且同步积压不阻塞主交易。

**原理补充**  
MySQL 到 ES 的一致性本质上是“主事实存储到查询副本”的异步复制问题。

**架构扩展**  
成熟架构师不会把“双写一致性”理解成“每次写两边都成功才算成功”，而会先收缩主链路，再把查询索引的可靠同步单独治理成一套通道。

**工程指标**  
- 同步延迟：核心一致性窗口指标。  
- 同步失败率：稳定性指标。  
- 死信积压量：恢复风险。  
- 对账差异数：最终质量指标。  
- 回放成功率：补偿能力指标。

**证据链**  
- binlog / 消息同步日志  
- ES 写入结果  
- 死信队列  
- 对账报表

**低风险优化路径**  
1. 先明确 MySQL 为主事实源  
2. 再用异步通道同步 ES  
3. 最后补对账和回放机制

**易错点**  
- 业务线程直接同步双写  
- 不做失败重放  
- 没有明确谁是真相源

## 16. 冷热数据分层如何设计？

**核心结论**  
冷热分层的核心不是把旧数据扔走，而是让不同价值和访问频率的数据走不同成本的存储路径。

**标准答案**  
设计冷热分层时，我通常先根据访问频率、写入变化频率、业务价值和合规要求把数据分成热、温、冷三层。热数据放高性能存储和缓存，温数据保留在普通在线库，冷数据归档到低成本存储或历史库。之所以要这么做，是因为不是所有数据都值得长期占用最贵资源。工程上关键还包括查询回流策略、冷热边界时间点、归档恢复流程和对业务透明度。面试里如果能补一句“冷热分层本质上是成本和访问价值匹配”，会比较完整。

**高分回答**
我会先用 30/90/180 天访问日志、更新频率、查询 P99、数据增长和合规保留期画出生命周期，而不是仅按“超过一年”判断冷数据。例如近 30 天订单占 95% 查询留在线主库，31～365 天进入成本较低的历史库，超过一年转列式/对象存储；支付审计即使很少访问也必须保留并校验不可篡改要求。每层定义可接受的查询延迟、可用性、成本/GB、恢复时间和更新语义，冷层通常只追加或不可变。迁移按分区小批执行，先复制、校验行数/校验和与金额等业务不变量，再切查询路由并从热层删除，全过程限速避免抢占在线 IO。查询入口根据时间范围拆分并合并结果，普通请求默认限制跨层扫描，审计查询异步化并展示进度；需要回热时有审批、容量检查和 TTL，避免冷数据永久回流。删除与保留策略由合规规则控制，备份也必须同步生命周期。用热层容量下降、命中率、跨层查询 P99、恢复成功率、存储成本和归档差异衡量；边界随着访问分布复核。冷热分层是带服务等级和可逆迁移的数据产品，不是把旧表搬到便宜磁盘后失去可查性。

**追问点答案**  
- 冷数据是不是就不能查了？  
不是，只是查询路径可能更慢或更特殊。  
- 怎么定义冷热边界？  
用访问频率/长尾、更新概率、查询延迟要求、恢复成本和合规保留共同确定，并定期根据真实分布调整，不能只按固定年份。
- 归档后为什么还要保恢复能力？  
因为审计、投诉、对账可能还需要历史数据。

**原理补充**  
冷热分层的核心，是让访问模式决定资源级别，而不是让所有数据享受同样高成本待遇。

**架构扩展**  
订单、日志、支付流水、埋点数据都非常适合分层。成熟架构师谈分层时，不只说“归档节约成本”，还会顺手讲查询路径和恢复流程。

**工程指标**  
- 热数据命中率：看分层是否合理。  
- 在线库容量回落比例：直接收益。  
- 归档查询占比：评估冷数据可访问性。  
- 存储成本下降幅度：成本结果。  
- 冷数据恢复时长：运维保障指标。

**证据链**  
- 访问频率分布  
- 数据生命周期规则  
- 存储成本账单  
- 归档作业记录

**低风险优化路径**  
1. 先按访问频率分层  
2. 再设计透明查询或特殊查询路径  
3. 最后补恢复和审计能力

**易错点**  
- 只归档不考虑查询  
- 冷热边界拍脑袋  
- 不设计恢复路径

## 17. 数据归档策略怎么定？

**核心结论**  
归档策略的关键不是“多久搬一次”，而是同时满足业务可查、成本下降、系统稳定和合规要求。

**标准答案**  
定数据归档策略时，我通常会先回答四个问题：哪些数据多久之后访问极低、业务还需要查多久、法规要求保留多久、归档后谁来查。之所以归档不能只按时间一刀切，是因为不同业务的数据生命周期差异很大。工程上常见做法是按时间分段、按状态归档或按业务域拆历史库，同时保证归档作业本身对在线系统影响可控。面试里如果能补一句“归档策略本质上是数据生命周期治理”，会更有架构感。

**高分回答**
我会建立按数据类别可审计的生命周期表：事实源、归档条件、最短/最长保留、legal hold、目标存储、查询 owner、恢复 RTO 和最终销毁方式。订单不能仅按创建时间归档，还要确认已完成、无退款/争议且关联状态闭环；受诉讼保全的数据即使到期也不能删除，隐私数据则可能必须按期匿名化或销毁。作业使用主键/时间分区小批复制，限制每批行数、事务时长、IO 和复制 lag；先写归档清单，校验行数、hash 与金额等业务不变量，再标记路由并延迟删除源数据，保留可回退窗口。历史查询通过专用只读服务或异步导出，权限、脱敏与审计不因进入冷层而降低。任务可断点续跑、幂等，失败不会产生半归档状态；备份、索引和下游派生数据同步执行保留/删除策略。上线先影子归档和恢复演练，监控吞吐、锁等待、主从延迟、差异、历史查询成功率、恢复 RTO 与合规覆盖。归档完成的证明不是在线表变小，而是数据可按承诺查询、恢复、保留并在到期后可证明销毁。

**追问点答案**  
- 为什么合规要求要提前考虑？  
因为有些数据必须保留，有些数据反而需要按期删除。  
- 归档作业为什么也会出问题？  
大事务会持锁、产生日志和复制积压，批量删除还会造成页回收抖动；应按主键小批、限速、监控在线 SLO 和复制 lag，并支持暂停续跑。
- 历史查询要不要透明？  
取决于业务场景，关键链路常需要统一查询体验。

**原理补充**  
归档不是简单搬数据，而是重新定义数据在不同生命周期阶段的存储位置和访问方式。

**架构扩展**  
成熟架构师会把归档和冷热分层、索引治理、历史查询、合规销毁放在同一个数据生命周期框架里，而不是孤立做一次迁移脚本。

**工程指标**  
- 归档成功率：作业质量。  
- 在线库容量下降比例：收益指标。  
- 归档作业对主库影响：稳定性指标。  
- 历史查询成功率：业务可用性指标。  
- 合规保留达标率：治理底线指标。

**证据链**  
- 生命周期规则  
- 归档作业日志  
- 在线库监控  
- 合规策略文档

**低风险优化路径**  
1. 先梳理业务和合规生命周期  
2. 再按低峰小批量归档  
3. 最后补查询和恢复通道

**易错点**  
- 只看存储成本  
- 忽略归档作业对在线影响  
- 不考虑合规和历史查询

## 18. 数据迁移如何不停机？

::: tip L2 深度训练
按快照、增量、影子校验、写栅栏和退场完成迁移：[遗留系统与模型腐化治理](/deep-dives/business-evolution/04-legacy-modernization)
:::

**核心结论**  
不停机迁移的核心不是完全没有切换，而是把切换做成短、可控、可回退的最后一步。

**标准答案**  
做不停机迁移时，我一般会把过程拆成全量搬迁、增量同步、双写或校验期、读流量切换和最终写切换几步。之所以不能直接停机搬，是因为大表和核心系统停机窗口往往不现实。工程上真正关键的是增量同步链路稳定、主从差异可核对、切换时有回退方案，并且业务在双写或双读期间不会被状态差异搞乱。面试里如果能补一句“不停机迁移的本质是把大风险拆成很多小风险”，会很加分。

**高分回答**
我先定义事实源、RPO/RTO、可接受增量 lag、写入冻结上限和回滚条件。迁移从一致性快照及对应 binlog/LSN 位点开始，按主键范围限速回填历史数据，同时 CDC 捕获之后的增删改；目标端按业务主键和版本幂等应用，删除有 tombstone，避免“先增量后全量”被旧快照覆盖。全量追平后做影子读：同一请求同时查询新旧库但只返回旧结果，比较行级字段、聚合值和性能；独立对账覆盖行数、校验和及金额/库存等业务不变量。切换先小比例读，再把写路由通过单一控制点和 fencing token 切到新库，短暂冻结只用于排空并确认 CDC 位点，而不是承担全量搬迁。迁移期若使用双写，必须有 outbox/补偿和明确权威源，不能在业务代码盲写两边。回滚窗口内旧库持续接收反向增量或保持可重放，禁止提前执行不可逆 schema 删除；回退同样要校验数据位点。演练同步中断、乱序、目标限流、切换失败和回滚，监控回填进度、lag、差异率、双边写冲突、SLO 与切换时长。所谓不停机，是把不可避免的权威切换缩成经过证据验证的短步骤。

**追问点答案**  
- 为什么切换前一定要对账？  
因为数据迁移最怕悄悄丢和悄悄错。  
- 双写是不是必须？  
不是；数据库 CDC 往往比应用双写更易保持单一事实源。只有 CDC 无法覆盖的写入才考虑双写，并必须有幂等、失败补偿、对账和权威侧。
- 回退为什么重要？  
因为真正危险的是切换后发现问题却回不去。

**原理补充**  
不停机迁移的本质，是把数据复制、状态追平和流量切换做成三个相互独立但可验证的阶段。

**架构扩展**  
表扩容、换库、换分片、上云迁移、本地机房迁云都离不开这套思路。成熟架构师会特别关注迁移可观测性和回退路径，而不是只关心同步速度。

**工程指标**  
- 全量迁移完成率：进度指标。  
- 增量同步延迟：切换前关键指标。  
- 对账差异率：数据质量核心指标。  
- 切换耗时：业务中断风险。  
- 回退成功率：安全兜底能力。

**证据链**  
- 迁移任务日志  
- 同步延迟监控  
- 对账报表
- 切换与回退记录

**低风险优化路径**  
1. 先做全量和增量同步分离  
2. 再长时间对账观察  
3. 最后在低峰做短切换并保留回退

**易错点**  
- 没有对账就切  
- 只设计迁移不设计回退  
- 把迁移当一次性动作

## 19. 主从延迟怎么监控和缓解？

**核心结论**  
主从延迟的问题不是“有延迟”，而是“延迟多久开始影响业务判断”，所以监控和路由要和业务一起设计。

**标准答案**  
监控主从延迟时，我通常不只看数据库内部复制秒数，还会结合业务层的写后读异常、读旧值案例和关键查询误判情况一起看。之所以不能只看一个秒级指标，是因为有些业务对几十毫秒都敏感，有些业务几秒也能接受。工程上缓解方式常见有：关键链路读主、复制线程和 IO 优化、拆热点写入、减轻大事务、控制从库压力，以及在高延迟时自动切换读策略。面试里如果能补一句“主从延迟治理本质上是把复制状态翻译成业务路由策略”，会比较成熟。

**高分回答**
我会同时监控传输与回放：主库当前 GTID/LSN 到副本已接收位点的差距反映网络/IO，已接收到已应用的差距反映 SQL 回放瓶颈；秒级 lag 容易受时钟和无写入场景误导，因此再写入周期性心跳事务测端到端可见延迟。业务层按提交位点记录 write token，读副本前确认其已回放到该位点，统计写后旧读和等待时间。告警阈值按业务分级，例如商品允许 3 秒，订单状态 200ms，超过后动态摘除该副本或关键查询回主，但回主比例受主库容量保护。根因治理先看大事务、DDL、热点单线程回放、从库慢查询、磁盘 IO 和网络，拆小事务、隔离报表、优化索引和提高并行回放；盲目增加从库会增加复制负担而不解决源头。故障恢复或积压追赶时限速业务读，避免回放与查询争抢资源。演练大事务、网络限速、从库报表和主从切换，观察 receive/apply lag、最老未应用事务、写后旧读、回主 QPS、主库水位和追平时间。延迟不是一个 DBA 数字，而是决定哪些副本有资格回答哪类读请求的实时能力信号。

**追问点答案**  
- 大事务为什么会拉高主从延迟？  
因为会占用复制和回放时间，导致后续事务排队。  
- 延迟高时为什么不直接扩从库？  
新增副本还会增加复制与运维成本；应先区分网络接收、回放、慢查询或大事务瓶颈，只有读负载挤占回放资源时扩容/隔离读才可能有效。
- 业务怎么感知延迟问题？  
最典型的是写后读旧、状态判断错误和展示异常。

**原理补充**  
主从延迟本质上是主库状态传播到从库的时间差，它只有映射到业务可见性时才真正成为风险。

**架构扩展**  
成熟架构师不会把复制延迟只当 DBA 指标，而是会把它直接接到业务路由和高风险查询策略里，让系统知道“延迟大时谁还能读从库，谁必须读主库”。

**工程指标**  
- 复制延迟：基础指标。  
- 写后读旧值事件数：业务质量指标。  
- 大事务占比：常见根因线索。  
- 从库回放吞吐：定位瓶颈。  
- 强制读主比例：缓解成本指标。

**证据链**  
- 主从复制监控  
- 慢事务日志  
- 业务异常样本  
- 读路由策略记录

**低风险优化路径**  
1. 先把复制延迟和业务影响挂钩  
2. 再治理大事务和热点写  
3. 最后用动态路由降低风险

**易错点**  
- 只看 DBA 指标不看业务效果  
- 延迟高了只扩机器  
- 不做读路由兜底

## 20. 数据库故障切换怎么做？

**核心结论**  
数据库故障切换的核心不是切得最快，而是切得对、切得稳、切完之后业务能继续跑且数据风险可控。

**标准答案**  
设计数据库故障切换时，我通常会先明确切换触发条件、选主策略、数据一致性边界、连接切换方式和回退路径。之所以不能只追求秒级切换，是因为误判、脑裂和数据落后往往比短暂中断更危险。工程上成熟的切换方案通常会结合健康检查、复制状态、仲裁、连接池刷新、读写路由更新和事后对账。面试里如果能补一句“数据库切换的难点不是主从角色变化，而是业务如何安全跨过这次变化”，会很有架构感。

**高分回答**
我先定义故障判定、RTO/RPO 和允许的数据损失，再设计切换。健康检查由多个故障域交叉确认连接、提交能力和复制状态，单点网络抖动不能直接提升新主；候选副本必须是最新可用位点，并通过多数仲裁获得新 term。提升前对旧主执行 fencing/隔离，数据面拒绝旧 term 写入，宁可短暂不可写也不能双主。路由层以版本化主地址切换，应用连接池缩短失效连接、重新解析并传播 deadline；写超时结果未知时按业务键查询，幂等重试，不能生成新流水重复执行。切换后先小流量验证写入、复制链、业务不变量和延迟，再逐步恢复读副本与批任务；对故障窗口内的确认/未确认事务依据日志、外部流水和对账修复。旧主恢复只能作为副本重新全量/增量同步，不能自动抢回；计划回切同样走一次受控迁移，避免来回振荡。演练进程宕机、网络分区、存储慢、复制落后和应用 DNS/连接缓存，记录检测、fencing、提升、应用恢复各阶段耗时、RPO 差异和误切次数。切换成功的证据是唯一写主、业务恢复且数据风险闭环，不是 VIP 已漂移。

**追问点答案**  
- 为什么误切主比切慢更危险？  
因为会导致双主写入和数据分叉。  
- 切换后为什么还要对账？  
因为故障窗口内可能存在复制未完全追平或写入丢失风险。  
- 应用侧要配合什么？  
还要传播 deadline、清理旧主连接、按业务键幂等处理结果未知写、动态摘除落后读副本，并在切换后验证关键业务不变量。

**原理补充**  
数据库切换本质上是在高风险瞬间重新定义“谁是真相源”，所以必须同时控制判断、路由和数据一致性风险。

**架构扩展**  
成熟架构师谈数据库故障切换，不会只说 MHA、Orchestrator 或云厂商能力，而是会继续讲：谁来判故障、多久切、切完应用怎么恢复、差异数据怎么补。这才是真正可运营的切换方案。

**工程指标**  
- 故障检测时长：切换前半段指标。  
- 切换成功率：体系成熟度核心。  
- 业务恢复时长：最终结果指标。  
- 数据差异数：风险指标。  
- 误切换次数：高危治理指标。

**证据链**  
- 切换日志  
- 复制状态  
- 应用连接错误记录  
- 事后对账结果

**低风险优化路径**  
1. 先定义切换触发和仲裁机制  
2. 再让应用路由和连接可快速切换  
3. 最后定期演练和对账复盘

**易错点**  
- 只追求快不追求准  
- 不考虑应用侧恢复  
- 切换后不做对账
