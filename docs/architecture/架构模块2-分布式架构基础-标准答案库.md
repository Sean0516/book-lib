# 架构模块2 - 分布式架构基础 - 标准答案库

## 1. CAP 在真实业务中怎么取舍？

::: tip L2 深度训练
用业务不变量和一致性窗口做选择，而不是背 AP/CP：[分布式一致性选择树](/deep-dives/distributed-stability/01-distributed-consistency)
:::

**核心结论**  
CAP 不是让系统三选二的口号题，而是在发生网络分区时，必须在一致性和可用性之间明确业务优先级。

**标准答案**  
我理解 CAP 时，不会把它当成平时系统设计的全部，而是把重点放在“网络分区发生时，系统到底优先保证什么”。之所以会有这个取舍，是因为一旦节点之间通信异常，系统不可能同时做到强一致和持续可用。工程上，大多数互联网业务不会全局强一致，而是核心交易链路偏一致性，外围查询链路偏可用性。面试里如果只说“选 AP 或 CP”会显得比较空，最好补一句：不同业务子系统可以有不同选择，不应该全站一个答案。真正成熟的回答，是把取舍和具体业务影响绑在一起。

**高分回答**
我先澄清 CAP 只约束网络分区期间的行为：系统无法确定对端是宕机还是暂时不可达，此时要么拒绝一部分请求来避免读写分叉，要么继续服务并接受不同节点短期给出不同结果。取舍单位不是“整个电商系统”，而是一条业务不变量。账户不能凭空增减、库存不能超卖这类不变量通常在无法取得法定多数时停止写入，牺牲局部可用性；商品详情、推荐和物流轨迹可以返回带版本或时间戳的旧值，优先保持可读，并在分区恢复后合并或重建。还要明确“可用”的含义：CAP 中要求每个非故障节点最终响应，并不等于业务承诺的低延迟和成功率；同样，选择一致性也不代表所有读都线性一致。设计时逐接口记录分区行为、允许陈旧窗口、冲突规则、客户端重试语义和恢复后的校验方式，再通过隔离一个节点或可用区验证：核心不变量是否守住、降级读是否命中、分叉数据多久收敛。这样 CAP 才从 AP/CP 标签变成可测试的失败策略。

**追问点答案**  
- 为什么没有人说选 CA？  
因为分布式系统一旦出现网络分区，就必须面对 P，不能假装不存在。  
- 电商系统一般怎么取舍？  
要按不变量与操作拆分：库存扣减、支付记账的写入通常要求多数派或单一权威；商品详情和推荐可读旧版本，但后台纠错、对账等路径仍可能采用强一致协调。
- CAP 是平时也一直生效吗？  
重点是在分区场景下体现，不是所有时候都在做极端取舍。

**原理补充**  
CAP 真正讨论的是“分区容错前提下的行为边界”，不是日常架构设计的万能分类标签。

**架构扩展**  
真实系统里很少会说“我们系统就是 AP”或者“就是 CP”，更多是订单状态、库存、账户余额这些链路偏强一致，而查询、推荐、搜索、缓存展示这些链路偏可用优先。成熟架构师会按业务域切一致性等级，而不是拿 CAP 当统一口号。

**工程指标**  
- 分区期间错误率：衡量系统是否还能对外服务。  
- 数据不一致窗口：评估一致性代价。  
- 核心交易成功率：验证 CP 链路是否稳住。  
- 降级命中率：看 AP 路径是否有效。  
- 故障恢复时间：决定业务可接受度。

**证据链**  
- 网络故障演练记录  
- 业务链路一致性分级  
- 分区期间日志和监控  
- 失败补偿记录

**低风险优化路径**  
1. 先按业务域划一致性等级  
2. 再给关键链路设计 CP 策略  
3. 最后给外围链路补降级和补偿

**易错点**  
- 把 CAP 当静态标签  
- 不结合业务场景  
- 觉得一个系统只能有一种取舍

## 2. BASE 具体意味着什么？

::: tip L2 深度训练
补齐“最终会一致”的检测、补偿与对账证据：[分布式一致性选择树](/deep-dives/distributed-stability/01-distributed-consistency)
:::

**核心结论**  
BASE 的本质不是“不一致也没关系”，而是在接受短暂不一致的前提下，通过异步和补偿换更高可用性。

**标准答案**  
BASE 通常可以理解成 Basically Available、Soft State 和 Eventually Consistent，它强调的是系统在分布式环境下允许中间状态短暂不一致，但最终要能收敛到正确结果。之所以会采用 BASE，是因为很多业务如果一味追求全链路强一致，系统复杂度和可用性成本会非常高。工程上，BASE 常见在积分、消息通知、订单状态同步、搜索索引刷新这类链路里。面试里如果只背英文缩写不够，最好补一句：BASE 的关键不只是“最终一致”，而是你必须设计好补偿、重试、幂等和核对机制，否则最终就不一定能一致。

**高分回答**
我不会把 BASE 当成放弃一致性的许可证，而会先定义可验证的收敛契约。以“支付成功后发积分”为例，支付账本是事实源，积分可以异步到账，但必须声明 99.9% 在 30 秒内、全部自动任务在 10 分钟内收敛，超过窗口进入告警与人工队列。实现上用本地事务同时写业务事实与 outbox，发布器至少一次投递，消费者以业务键幂等推进状态机；暂时失败采用有上限的退避重试，永久失败进入死信和补偿。对账任务从事实源重新推导期望结果，比较差异并执行可审计修复，因此即使消息丢失、重复或消费者长期停机也有独立恢复路径。Soft State 表示副本或派生状态会随异步传播变化，并不表示状态没有约束；冲突时必须规定事实源、版本和合并规则。验收指标包括一致性延迟分布、超窗数量、补偿成功率、对账差异与人工介入率。只有能够说明“由谁、在多久内、依据什么事实、失败后如何证明收敛”，才真正实现了 BASE。

**追问点答案**  
- 最终一致是不是一定会一致？  
不是自然保证；必须有明确事实源、可重放事件、幂等消费、有限重试、补偿与独立对账，并用超窗告警证明它正在收敛。
- Soft State 是什么意思？  
中间状态允许短暂变化，不要求每一时刻都对所有节点完全同步。  
- 哪些业务不适合 BASE？  
余额扣减、支付记账、核心库存确认这类链路要非常谨慎。

**原理补充**  
BASE 本质上是在一致性成本和系统可用性之间，选择了“允许延迟一致，但必须可恢复”的路线。

**架构扩展**  
很多团队会把“最终一致”说得很轻松，但真正上线后经常是消息丢了、补偿没做、对账没有，最后一致性只是一个口头承诺。成熟架构师提 BASE 时，一定会顺带讲补偿闭环，否则这个方案是不完整的。

**工程指标**  
- 一致性延迟：看最终一致窗口有多长。  
- 补偿成功率：决定能不能真正收敛。  
- 对账差异数：衡量一致性质量。  
- 重试次数：反映恢复成本。  
- 用户可见错误数：最终要回到业务感知。

**证据链**  
- 消息流转日志  
- 补偿任务记录  
- 对账报表  
- 不一致案例复盘

**低风险优化路径**  
1. 先明确哪些链路接受最终一致  
2. 再补全异步、幂等、补偿和对账  
3. 最后定义一致性窗口和告警

**易错点**  
- 把 BASE 理解成可以随便不一致  
- 只设计异步，不设计补偿  
- 不定义最终收敛标准

## 3. 最终一致性如何向业务解释？

**核心结论**  
向业务解释最终一致性时，关键不是讲技术原理，而是讲清楚“不一致会持续多久、影响谁、怎么兜底”。

**标准答案**  
我向业务解释最终一致性时，一般不会先讲消息队列和分布式事务，而是先说明：这条链路为了提升可用性和解耦，会允许结果在短时间内不同步，但系统会在约定时间内自动收敛。之所以这样讲更有效，是因为业务关心的是影响范围和恢复时间，而不是底层中间件名字。工程上我会明确三件事：第一，不一致窗口大概多长；第二，用户会不会感知到；第三，失败后用什么补偿和对账兜底。面试里如果能说出“技术方案要翻译成业务可接受的延迟和风险”，会很加分。

**高分回答**
我会用用户旅程和服务承诺解释，而不是说“消息最终会到”。例如支付成功后订单页通常 3 秒内更新，极端情况下 10 分钟内自动修复；这段时间用户看到“支付处理中”，不会再次扣款，且能凭支付流水查询。随后把长尾说清楚：超过 10 分钟自动告警并进入人工工单，资金账本以支付渠道和内部记账为事实源，每日对账确保没有永久丢单。与业务一起确认四项契约：哪些状态允许延迟、P99 收敛窗口、窗口内的用户文案与重复操作保护、超窗赔付或人工流程。若余额、库存等不变量不允许暂时错误，就把强一致边界缩到关键写入，只让通知、索引等派生结果异步。上线后用超窗订单数、重复操作率、投诉率、自动修复率和人工处理时长复盘，并在产品页面暴露可查询状态。这样业务接受的是一个有时间、有体验、有责任人的风险方案，而不是不受约束的“以后会一致”。

**追问点答案**  
- 业务最担心什么？  
最担心关键结果长期不一致、用户重复操作或者资金类错误。  
- 怎么降低业务焦虑？  
给出 P99 与最长收敛窗口、窗口内用户看到什么、如何防重复操作、超窗后由谁自动或人工处理，并用历史指标证明承诺可达成。
- 所有最终一致都需要业务确认吗？  
不一定，但关键业务要有明确共识。

**原理补充**  
最终一致性的难点，不只是技术实现，而是把“技术允许的延迟”转换成“业务可接受的承诺”。

**架构扩展**  
比如积分到账、优惠券发放、站内消息、搜索索引刷新，很多业务其实能接受几秒甚至几分钟延迟；但支付成功后订单状态长时间不同步就不行。成熟架构师会先让业务接受一个清晰的时间窗口和失败路径，再落技术方案。

**工程指标**  
- 收敛时长：业务最关心的指标之一。  
- 用户投诉率：真实感知结果。  
- 补偿成功率：决定业务信任度。  
- 人工介入率：反映自动化兜底是否足够。  
- 对账差异修复时长：衡量尾部问题处理能力。

**证据链**  
- 业务 SLA 说明  
- 历史不一致案例  
- 补偿任务统计  
- 用户侧感知数据

**低风险优化路径**  
1. 先定义可接受窗口  
2. 再设计补偿和对账  
3. 最后把业务说明写进规则和预案

**易错点**  
- 只和业务说“最终会一致”  
- 不讲时间窗口  
- 不讲失败兜底

## 4. 幂等设计常见方法有哪些？

**核心结论**  
幂等的目标是让同一个业务请求无论被执行一次还是多次，最终结果都一致，常见实现依赖唯一键、状态机和去重表。

**标准答案**  
做幂等设计时，我通常先问这条链路的业务唯一标识是什么，比如订单号、支付流水号、请求号、消息 ID，然后再决定是用唯一索引、防重表、状态机校验还是分布式锁。之所以幂等这么重要，是因为分布式系统里重试、消息重复投递、超时回放几乎是常态。工程上最稳的方式通常不是加锁，而是让业务写入天然具备“重复执行也不会产生第二次副作用”的能力，比如唯一约束加状态判断。面试里如果能补一句“幂等不是防止请求重复到达，而是防止重复到达后产生重复副作用”，会比较准确。

**高分回答**
我先定义幂等作用域和结果语义：同一用户用同一幂等键重复“创建订单”，应返回同一订单及同一业务结果，而不是仅返回“重复请求”。键应绑定业务意图，例如 `tenantId + clientOrderNo + operation`，并保存请求摘要，防止调用方误用同一键提交不同金额。数据库内优先用唯一约束作为最终并发防线，在一个事务中插入幂等记录、校验摘要并执行副作用；首次请求处于处理中时，重复请求等待、轮询或返回明确状态，成功后缓存原响应。已有实体则通过带版本的条件更新推进单向状态机，例如只有 `INIT` 能扣款，受影响行数为 0 时读取并返回当前结果。跨外部系统不能靠本地去重证明端到端幂等，应把同一业务键传给支付方，或用 outbox、可查询结果和补偿处理“本地成功但响应丢失”。幂等记录的 TTL 至少覆盖客户端重试、消息回放和人工补偿窗口，不能过早清除。最终用并发重复、超时后重试、响应丢失和消息重放测试，指标关注重复副作用必须为零、键冲突、处理中超时与记录增长；分布式锁只能降低并发，不能替代业务约束。

**追问点答案**  
- 幂等和去重有什么区别？  
去重更像手段，幂等是业务最终效果要求。  
- 为什么唯一索引常见？  
因为唯一约束在并发竞争下仍是最终原子防线；但还需保存请求摘要与原结果，才能处理键复用和让重复请求得到一致响应。
- 所有接口都要幂等吗？  
对有副作用的写操作尤其重要，读接口通常天然幂等。

**原理补充**  
幂等设计的核心，是让系统把“重复请求”映射成“同一业务结果”，而不是重复执行副作用。

**架构扩展**  
支付回调、订单创建、优惠券发放、消息消费、库存扣减都很依赖幂等。成熟架构师不会把幂等只理解成“加个 token”，而是会从业务唯一性、状态推进和失败补偿一整套一起看。

**工程指标**  
- 重复请求命中率：衡量幂等需求强度。  
- 幂等冲突数：看重复场景有多频繁。  
- 重复副作用事件数：核心质量指标。  
- 状态回滚率：说明幂等设计是否不稳。  
- 人工修复次数：评估自动防重效果。

**证据链**  
- 业务唯一键设计  
- 唯一索引或防重表  
- 重试日志  
- 状态流转记录

**低风险优化路径**  
1. 先找业务唯一标识  
2. 再选唯一约束或状态机方案  
3. 最后补重试和补偿验证

**易错点**  
- 用技术请求 ID 替代业务唯一键  
- 只做接口防重，不做业务副作用防重  
- 把加锁当成唯一答案

## 5. 重试机制如何避免放大故障？

::: tip L2 深度训练
计算多层重试的乘法放大，并设计端到端重试预算：[流量保护与级联故障](/deep-dives/distributed-stability/02-traffic-protection)
:::

**核心结论**  
重试不是默认正确策略，必须配合限次、退避、熔断和幂等，否则很容易把局部故障放大成全链路雪崩。

**标准答案**  
设计重试机制时，我一般会先判断错误类型值不值得重试，比如短暂超时、网络抖动可以考虑重试，但参数错误、明确业务拒绝通常不应该重试。之所以重试危险，是因为在下游已经慢或已经挂的时候，继续快速重试只会把压力打得更高。工程上通常要配合指数退避、随机抖动、最大重试次数、熔断和幂等一起设计，必要时还要转成异步补偿。面试里如果能补一句“重试本质上是拿更多资源换更高成功率，所以一定要有边界”，会比较稳。

**高分回答**
我把重试视为全链路共享的额外负载预算。若网关、服务 A 和 SDK 各自最多尝试 3 次，一次用户请求最坏可能放大到 27 次，所以只允许最靠近失败点且了解错误语义的一层重试，并传播绝对 deadline 与剩余尝试预算。只重试明确的瞬时故障和幂等操作；参数错误、限额拒绝、业务冲突不重试，超时也要区分下游尚未执行还是“结果未知”，后者先查询结果而不是盲目再次扣款。间隔采用指数退避加随机抖动，服务端 `Retry-After` 优先，同时限制单请求次数、全局重试并发和重试流量占比；下游错误率或延迟越过阈值时熔断并快速失败，把可延迟业务转入有界队列或补偿任务。还要防止 deadline 只剩 20ms 却发起注定失败的重试，并让每次尝试复用业务幂等键。验证时注入超时、限流和部分故障，观察总放大倍数、重试成功收益、下游负载、队列增长和恢复时间；如果重试带来的新增成功率很低却明显增加压力，就应减少或关闭它。重试的目标不是隐藏故障，而是在不损害系统恢复的前提下处理短暂失败。

**追问点答案**  
- 为什么要指数退避？  
为了避免大量请求同时立即重试，形成重试风暴。  
- 所有超时都值得重试吗？  
不值得；超时可能表示结果未知或下游过载，应结合操作幂等性、剩余 deadline、错误类型和熔断状态决定，必要时先查询结果或转异步补偿。
- 重试为什么必须配幂等？  
因为重复执行可能造成副作用放大。

**原理补充**  
重试机制的核心，是只在“成功概率会上升”的情况下追加请求，而不是盲目重复。

**架构扩展**  
很多事故里，最开始只是数据库慢、Redis 抖或者某个服务超时，但真正把系统打崩的却是调用方成倍重试。成熟架构师设计重试时，会把它当“最后补救手段”，而不是基础默认动作。

**工程指标**  
- 重试率：识别系统是否进入异常态。  
- 重试成功率：衡量重试是否有价值。  
- 下游错误率：避免重试掩盖真实故障。  
- 熔断命中率：看保护是否生效。  
- 总请求放大倍数：关键风险指标。

**证据链**  
- 错误类型统计  
- 重试日志  
- 调用链 tracing  
- 熔断和限流记录

**低风险优化路径**  
1. 先按错误类型区分是否可重试  
2. 再加退避、限次和熔断  
3. 最后把高风险链路改成异步补偿

**易错点**  
- 所有失败都重试  
- 无退避直接猛打  
- 不做幂等校验

## 6. 分布式锁怎么避免误删锁？

**核心结论**  
避免误删锁的关键，不是“拿到锁就删”，而是保证只有锁的持有者才能释放它。

**标准答案**  
设计分布式锁时，我通常会在加锁时写入一个全局唯一 value，比如 UUID 或请求唯一标识，释放锁时先校验 value 是否还和自己持有的一致，再删除。之所以必须这样做，是因为锁可能已经过期被别人重新拿到，如果你不校验就删，就会把别人的锁误删掉。工程上如果用 Redis 做锁，释放动作通常要通过 Lua 脚本保证“校验 + 删除”原子执行。面试里如果能补一句“误删锁的本质是锁生命周期和业务执行时间错位”，就比较到位。

**高分回答**
Redis 锁至少要用带过期时间的原子 `SET key owner NX PX ttl`，owner 是每次获取生成的唯一令牌；释放时用 Lua 原子比较 owner 后删除，解决“客户端 A 暂停超过 TTL、锁被 B 获取、A 恢复后误删 B 的锁”。但这只保证不会误删，不能保证旧持有者不会继续操作：A 在锁过期后仍可能写数据库，所以高价值场景还要由锁服务返回单调递增的 fencing token，下游只接受比已见版本更大的写入令牌。TTL 应基于业务耗时长尾并留余量，续约只由仍持有 owner 且进程健康的任务执行；续约失败后业务应停止进入新的关键步骤，而不是假设锁仍有效。还要评估主从切换时锁写入尚未复制导致双持有者的风险，不能把单 Redis 锁宣称为绝对互斥；若无法让下游校验 fencing，应优先使用数据库唯一约束、条件更新或具备共识的协调服务。测试覆盖暂停、网络分区、锁服务切主、超时重获和释放重入，指标关注双持有者、过期后写入、续约失败及持有时长。分布式锁是协调手段，最终正确性仍应尽量落在业务状态约束上。

**追问点答案**  
- 为什么不能直接 `DEL`？  
因为锁可能已经超时并被其他线程重新占有。  
- 为什么要用 Lua 脚本？  
为了保证校验和删除是原子操作。  
- 锁超时时间怎么定？  
根据业务耗时 P99.9、暂停和网络抖动留余量，并设置最大持有上限；长任务用带所有权校验的续约，续约失败必须停止关键操作，不能简单设成无限长。

**原理补充**  
分布式锁真正要保护的，不只是互斥，而是“锁所有权”的正确释放。

**架构扩展**  
订单关闭、库存扣减、定时任务抢占、幂等补偿这类场景经常会用锁，但很多线上 bug 不是“没加锁”，而是“锁加了但释放错了”。成熟架构师设计锁时，会同时思考续约、超时、持有者校验和降级兜底。

**工程指标**  
- 锁获取成功率：看竞争是否过高。  
- 锁持有时长：评估 TTL 是否合理。  
- 误删锁事件数：核心风险指标。  
- 续约失败率：影响长任务稳定性。  
- 竞争重试次数：反映热点程度。

**证据链**  
- 锁 key/value 记录  
- Lua 脚本执行日志  
- 业务耗时分布  
- 锁竞争监控

**低风险优化路径**  
1. 先给锁绑定唯一持有者  
2. 再用原子脚本释放  
3. 最后补续约和超时告警

**易错点**  
- 直接删除锁  
- 不关心持有者身份  
- TTL 和业务耗时不匹配

## 7. 时钟漂移对分布式系统影响是什么？

**核心结论**  
时钟漂移会影响超时判断、分布式锁、ID 生成、日志排序和一致性判断，问题本质是多个节点对“当前时间”的认知不一致。

**标准答案**  
分布式系统里如果节点时钟不一致，会直接影响依赖时间语义的逻辑，比如锁超时、任务调度、订单超时关闭、雪花 ID 生成、日志顺序和消息延迟判断。之所以这个问题危险，是因为很多代码里默认“所有机器时间差不多”，但在跨机房、虚拟化和 NTP 抖动场景下，这个假设并不可靠。工程上一般会尽量避免把本地物理时间当作强一致判断依据，关键逻辑优先使用单调递增序列、数据库时间或中心化协调。面试里如果能补一句“时间在分布式里是一个近似值，不是绝对真理”，会比较成熟。

**高分回答**
我先区分墙上时钟与单调时钟：墙上时钟会被 NTP 校正或回拨，适合展示时间；进程内计算超时应使用单调时钟，避免时间跳变让 deadline 变长或变负。跨节点不能仅靠时间戳推导因果顺序，订单先后、选主任期和写入版本应使用数据库序列、逻辑时钟、共识日志或 fencing token。雪花 ID 遇到回拨可暂停等待、切换保留 worker/序列区间或报警拒绝生成，不能在未知风险下继续产生重复 ID；租约也必须把最大时钟偏差和网络延迟计入有效期，否则旧主可能误以为租约仍有效。日志排障则同时记录事件时间、接收时间、trace/span 因果关系与节点偏差，避免按一列时间戳错误还原事故。基础设施持续监控每节点 offset、同步状态、回拨次数和时间源切换，偏差超过安全界限时将节点摘出时间敏感服务。演练主动注入快慢时钟和回拨，验证调度、锁、ID、证书校验与监控是否安全失败。核心原则是：物理时间只能提供有误差的估计，凡涉及正确性的先后关系都需要额外的顺序或仲裁机制。

**追问点答案**  
- 雪花 ID 为什么怕时钟回拨？  
因为时间戳回退会导致 ID 重复或拒绝生成。  
- 日志时间错乱会带来什么问题？  
会让排障时的事件顺序判断失真。  
- 怎么降低时钟漂移影响？  
进程内超时用单调时钟，跨节点顺序用逻辑版本或共识序列；同时同步并监控墙上时钟，偏差超界时摘除时间敏感节点。

**原理补充**  
分布式系统里的时间不是天然全局一致，只能通过同步和设计约束尽量逼近。

**架构扩展**  
真实项目里，很多“偶发诡异 bug”最后根因是时间问题，比如任务提前执行、延迟消息乱序、ID 生成报错。成熟架构师会把时间相关逻辑单独当成风险点看待，而不是默认操作系统会帮你兜底。

**工程指标**  
- 节点时间偏差：直接风险指标。  
- NTP 同步失败率：基础健康度。  
- 时钟回拨事件数：高风险告警。  
- 时间相关异常日志数：帮助定位影响面。  
- ID 生成失败率：典型业务信号。

**证据链**  
- NTP 状态  
- 节点时间差监控  
- 时间敏感业务日志  
- 异常事件时间线

**低风险优化路径**  
1. 先监控时间偏差  
2. 再减少关键逻辑对本地时间的强依赖  
3. 最后给高风险模块补中心化时间策略

**易错点**  
- 默认机器时间绝对可靠  
- 用本地时间做强一致判断  
- 不监控回拨事件

## 8. 去重表和幂等键怎么选？

**核心结论**  
去重表和幂等键不是二选一，关键要看业务副作用位置、生命周期和一致性要求。

**标准答案**  
如果幂等目标比较简单，比如同一个业务请求只允许成功一次，而且天然有唯一业务号，那我通常优先使用幂等键配合唯一约束；如果链路更复杂，需要记录重复处理历史、失败状态或后续补偿，我会更倾向于去重表。之所以要这样区分，是因为幂等键更轻量，适合简单防重，而去重表更像完整的状态记录器。工程上还要看数据保留周期、查询成本和并发冲突频率，因为去重表如果设计不好，也会变成新热点。面试里如果能补一句“选型不只是看性能，还要看你是否需要可追溯状态”，会比较完整。

**高分回答**
我会先看防重记录能否和业务副作用放在同一原子边界。单库内创建订单时，业务表上的 `tenant_id + client_order_no` 唯一约束通常最可靠，重复请求直接读取并返回原订单；若需要保存请求摘要、原响应和处理中状态，则增加幂等记录。消息消费跨多个步骤、需要重放和人工补偿时，使用去重/收件箱表记录 `consumer + message_id`、业务键、状态、尝试次数和结果，并与本地业务写在同一事务提交，避免“先标记已处理后业务回滚”造成永久漏处理。只在 Redis 保存短期幂等键速度快，但缓存丢失或 TTL 到期后会再次产生副作用，因此只适合可容忍重复或数据库仍有最终约束的场景。保留周期按生产者最大重试、消息留存、人工回放和审计窗口取最大值，清理按时间分区批量执行；高吞吐场景避免单一热点键，并观察唯一冲突与表膨胀。两者并非互斥：业务唯一键保证最终效果，去重表提供处理状态和审计。选择标准是原子性、结果复用、生命周期与可追溯需求，而不是笼统地说哪个性能更高。

**追问点答案**  
- 幂等键适合什么场景？  
适合一次性写入、副作用单一、业务号天然唯一的场景。  
- 去重表适合什么场景？  
适合异步消费、补偿链路、需要状态记录和排查的场景。  
- 两者可以一起用吗？  
可以，业务唯一键或唯一约束负责最终副作用不重复，去重表记录消费状态、原结果和补偿审计；二者应尽量与业务写处于同一事务边界。

**原理补充**  
本质上是在“轻量唯一约束”和“完整处理状态”之间做取舍。

**架构扩展**  
支付回调、消息消费、补偿任务、券码发放这种链路，很多时候不仅要防重，还要知道“这次为什么失败、失败后做了什么、是否人工补过”。成熟架构师选幂等方案时，会把排障和审计一起考虑。

**工程指标**  
- 重复请求命中数：衡量防重需求。  
- 去重表写入 QPS：看是否形成热点。  
- 幂等冲突失败率：评估并发行为。  
- 排障定位时长：反映可追溯性价值。  
- 数据清理延迟：影响长期成本。

**证据链**  
- 业务唯一键设计  
- 去重表结构  
- 重复处理日志  
- 审计与补偿记录

**低风险优化路径**  
1. 先按副作用复杂度分层  
2. 简单场景优先唯一键  
3. 复杂场景再引入去重表和审计

**易错点**  
- 只追求快，不考虑可追溯  
- 去重表无限增长不治理  
- 把技术 requestId 当业务幂等核心

## 9. 事务消息适合哪些场景？

::: tip L2 深度训练
横向比较 Outbox、事务消息、Saga、TCC 与强一致边界：[分布式一致性选择树](/deep-dives/distributed-stability/01-distributed-consistency)
:::

**核心结论**  
事务消息适合“本地事务成功后必须可靠触发异步动作”的场景，本质是把数据库状态变化和消息发送做成同一业务结果。

**标准答案**  
我一般会在这种场景考虑事务消息：本地数据库写成功之后，必须可靠通知下游继续处理，但又不想做跨服务强事务。比如订单创建后通知库存、支付成功后通知发货、积分变更后通知账务同步。之所以事务消息适合这里，是因为它把“本地事务提交”和“消息可被后续确认发送”绑定在一起，降低了 DB 成功但消息丢失的风险。工程上仍然要配合幂等消费和回查机制，因为事务消息只解决发送阶段的可靠性，不解决整个链路的重复执行问题。面试里如果能补一句“事务消息是强一致事务和纯异步通知之间的折中方案”，会很加分。

**高分回答**
事务消息适合“本地事实一旦提交，某个事件必须至少一次传播，但下游允许在明确窗口内最终一致”的链路，例如支付账本提交后驱动订单和权益更新。实现可以是中间件半消息加本地事务回查，也可以是数据库事务内写 outbox，再由发布器转发；我会根据团队中间件能力选择，但两者都必须把消息资格与本地事实放在同一可靠边界。事件携带稳定业务 ID、事实版本和 schema 版本，消费者用业务键幂等并只允许合法状态跃迁。生产方崩溃导致状态未知时，回查必须查询权威业务状态，而不是凭内存判断；未决消息设置超时、重查上限和人工队列。它不适合要求跨服务立即可见、不可接受中间态的不变量，那应缩小强事务边界或评估 TCC；也不值得用于可丢弃的低价值通知。验收要覆盖本地提交前后宕机、重复投递、乱序、回查失败和消费者停机，监控未决消息、回查率、重复率、收敛时长与对账差异。事务消息只闭合“事实到事件”的缺口，端到端正确性仍由幂等、补偿和对账共同保证。

**追问点答案**  
- 事务消息能替代分布式事务吗？  
不能；它通常保证本地事实与消息发送资格可靠关联，无法让多个服务同时原子提交，也不自动解决消费副作用和业务回滚。
- 为什么还需要回查？  
因为生产方在本地事务执行后可能还没来得及确认消息状态。  
- 所有异步通知都要事务消息吗？  
不一定，低价值通知可能普通消息就够了。

**原理补充**  
事务消息的核心，是把“消息发送资格”与本地事务结果关联起来，而不是简单同时执行两件事。

**架构扩展**  
真实项目里，事务消息很适合订单、支付、积分、状态同步这些核心但可异步收敛的链路。成熟架构师会在强一致和松散异步之间找这类中间方案，而不是所有地方都上分布式事务。

**工程指标**  
- 事务消息提交成功率：基础可靠性指标。  
- 回查次数：反映不确定状态比例。  
- 消费幂等冲突率：判断后续链路稳定性。  
- 最终收敛时长：业务视角关键指标。  
- 未决消息数：高风险信号。

**证据链**  
- 事务消息日志  
- 本地事务记录  
- 回查结果  
- 消费状态表

**低风险优化路径**  
1. 先确认链路适合最终一致  
2. 再引入事务消息和回查  
3. 最后补消费幂等和对账

**易错点**  
- 觉得事务消息就万无一失  
- 不做消费幂等  
- 把低价值通知也复杂化

## 10. 补偿机制怎么设计？

**核心结论**  
补偿机制不是简单“失败了再来一次”，而是为分布式链路设计一个可恢复、可回滚、可核对的兜底闭环。

**标准答案**  
设计补偿机制时，我通常先明确业务目标是“重做成功”还是“撤销回退”，然后再按链路拆出每一步的状态记录、重试条件、幂等校验和人工兜底入口。之所以补偿机制难，是因为它不是技术重试，而是业务恢复。工程上最重要的是把每一步状态持久化，否则后面连补哪一步都不清楚。面试里如果能补一句“补偿不是异常处理代码，而是业务流程的一部分”，会很有架构感。

**高分回答**
我先定义业务可接受的终态和不可逆点，再决定向前恢复还是反向补偿。以“订单—支付—库存—优惠券”为例，支付结果未知时不能直接退款重做，而应先查询权威支付流水；库存预留可释放，已发券可作废，但已发货可能只能走售后流程，说明补偿不等于技术回滚。每一步持久化业务键、状态、版本、输入摘要、尝试次数和下一动作，用显式状态机限制合法转换；补偿命令本身必须幂等，并使用条件更新防止与正常流程并发互相覆盖。瞬时失败有界退避，超过次数进入待人工队列；定期对账从支付、库存等事实源重新计算期望状态，捕获消息永久遗漏和补偿程序自身失败。执行补偿要考虑逆序、依赖和隔离，例如先阻止继续履约再退款，并为批量积压设置速率限制，避免恢复流量压垮下游。审计记录原动作、补偿理由、操作者与结果，高价值动作支持审批。用步骤级成功率、超窗数量、积压年龄、人工介入率和业务损失衡量，演练在每个提交点宕机与重复执行。完整方案回答的是“现在处于什么状态、下一步谁做、失败如何发现、最终以何证据确认恢复”。

**追问点答案**  
- 补偿和重试区别是什么？  
重试更像技术动作，补偿是业务恢复动作。  
- 什么链路一定要有补偿？  
跨服务写操作、有副作用的异步链路、高价值最终一致链路。  
- 补偿失败怎么办？  
按错误类型有限重试，超限进入带优先级和 SLA 的人工队列；独立对账持续发现未收敛项，且补偿处理必须可审计、可再次安全执行。

**原理补充**  
补偿机制的核心，是把不可原子化的分布式操作拆成“可追踪、可修复”的业务状态机。

**架构扩展**  
支付成功但积分没到账、订单取消但库存没回补、优惠券已发但状态没同步，这类场景都依赖补偿。成熟架构师会优先把“失败后怎么回到可接受状态”设计清楚，而不是默认所有链路都会成功。

**工程指标**  
- 补偿成功率：衡量恢复能力。  
- 补偿时延：决定业务感知。  
- 待补偿积压量：高峰风险指标。  
- 人工介入率：反映自动恢复成熟度。  
- 对账差异修复率：看尾部质量。

**证据链**  
- 状态机记录  
- 补偿任务日志  
- 对账结果  
- 人工处理工单

**低风险优化路径**  
1. 先记录完整业务状态  
2. 再设计自动补偿和幂等  
3. 最后补人工处理和对账

**易错点**  
- 只做技术重试  
- 没有持久化状态  
- 补偿失败后没有兜底

## 11. 一致性哈希解决了什么问题？

**核心结论**  
一致性哈希主要解决节点增减时数据迁移范围过大的问题，让分布式路由更平滑。

**标准答案**  
传统哈希取模在节点数量变化时，数据映射关系会大面积重算，导致缓存失效、数据迁移和流量抖动都非常严重。一致性哈希的核心价值，就是把数据和节点映射到同一个环上，当节点增加或减少时，只影响相邻一小段数据。之所以这很重要，是因为分布式系统经常需要扩容和缩容，如果每次变更都导致全量重映射，系统代价会非常高。工程上通常还会配合虚拟节点来改善数据分布不均的问题。面试里如果能补一句“一致性哈希解决的是重映射成本，不是所有路由问题”，会比较准确。

**高分回答**
我会先说明它解决的是成员变化时的重映射成本。普通 `hash(key) % N` 从 N 台扩到 N+1 台时，大多数 key 的余数都会变化；一致性哈希把节点和 key 映射到同一环上，key 顺时针找到第一个节点，新增或删除节点通常只影响相邻区间，理想情况下扩一台约迁移 `1/(N+1)` 的数据。真实集群还必须使用足够且稳定映射的虚拟节点，让一个物理节点承接多个离散区间，并按机器容量配置权重，否则节点少或哈希分布偏斜时最大节点负载仍会远高于平均值。扩容不是更新路由就结束：先让新节点复制或预热所属区间，双读/校验后再渐进切流；客户端与服务端要有路由版本，避免不同版本同时把同一 key 写到不同节点。缓存场景可接受重建，但持久化存储还要处理副本放置、迁移期间读写一致性和故障恢复。它也不解决单个超级热点，可用 key 拆分、热点副本或请求合并单独治理。验收对比节点变化前后的重映射比例、最大/平均负载比、命中率跌幅、迁移带宽与 TP99，证明扩缩容扰动确实被局部化。

**追问点答案**  
- 为什么还需要虚拟节点？  
因为真实节点少时，哈希环上分布可能不均。  
- 一致性哈希常见在哪？  
缓存分片、网关路由、分布式存储和负载分布。  
- 它能解决热点问题吗？  
不能；它改善大量 key 的整体分布，但单个高频 key 仍落在一个区间，需要热点副本、key 拆分、请求合并或专门路由。

**原理补充**  
一致性哈希的核心是让“节点变化的影响范围局部化”。

**架构扩展**  
缓存集群扩容、对象存储节点变更、代理路由这些场景都很适合一致性哈希。成熟架构师会把它看成“降低扩缩容扰动”的工具，而不是一提分布式路由就默认要上。

**工程指标**  
- 重映射比例：一致性哈希价值核心指标。  
- 节点负载分布：看是否均匀。  
- 热点 key 集中度：辅助识别尾部风险。  
- 扩容后命中率波动：缓存场景很重要。  
- 数据迁移耗时：衡量扩容平滑性。

**证据链**  
- key 分布统计  
- 扩容前后路由映射  
- 虚拟节点配置  
- 命中率和负载曲线

**低风险优化路径**  
1. 先确认扩容扰动是核心问题  
2. 再引入一致性哈希和虚拟节点  
3. 最后补热点治理

**易错点**  
- 以为它能解决所有负载不均  
- 不配虚拟节点  
- 不验证扩容后真实命中波动

## 12. 脑裂场景如何治理？

**核心结论**  
脑裂治理的关键不是让两个节点都继续工作，而是尽快判断谁有资格继续对外提供写能力。

**标准答案**  
脑裂通常发生在集群节点之间失联后，多个节点都认为自己是主，从而同时对外服务，最终造成数据冲突。治理这类问题时，我更关注的是如何快速收敛写权限，比如通过仲裁机制、法定人数、租约、fencing token 或外部协调组件保证只有一方继续写。之所以脑裂危险，是因为它表面上看系统还在运行，但实际上数据已经分叉。工程上很多高可用方案的核心，其实不是自动切主，而是避免错误切主。面试里如果能补一句“脑裂的核心问题不是可用性，而是双主写入导致的数据不可恢复”，会很到位。

**高分回答**
我把治理目标定义为“任何时刻只有拥有最新任期的一方能产生可接受写入”。控制面用奇数仲裁成员和多数派选主，节点失去多数后立即停止写；仲裁成员跨故障域部署，避免两节点互相看不到时各自提升为主。仅靠心跳不能区分对端宕机与网络隔离，租约也受暂停和时钟误差影响，因此数据面还要校验单调递增的 term 或 fencing token：即使旧主恢复并继续发请求，数据库、存储或外部设备也拒绝旧令牌。自动切主前检查复制进度和数据损失预算，宁可暂时不可写，也不能把落后副本提升后覆盖新数据；STONITH 等场景可物理隔离旧主。恢复时先冻结双边写入，依据已提交日志、任期和业务不变量选权威分支，重放可证明的操作，冲突项进入人工审核，不能用“最后写入时间”粗暴覆盖资金数据。演练网络单向隔离、仲裁丢失、长 GC、旧主复活和切主失败，观察双写数必须为零、失去写权耗时、RTO/RPO 与冲突修复时长。高可用的首要条件是防止两个主都成功写，而不只是更快选出新主。

**追问点答案**  
- 为什么脑裂比单点宕机更危险？  
因为它会产生冲突数据，恢复成本更高。  
- 什么是 fencing token？  
可以理解成一个单调递增的写入令牌，用来拒绝过期主节点继续写。  
- 只靠心跳能解决脑裂吗？  
不能，心跳只能表明当前通信失败，无法证明对方已经停止；还需要多数仲裁决定任期，并让数据面以 fencing token 或物理隔离拒绝旧主。

**原理补充**  
脑裂治理的本质，是把“谁有资格继续写”变成可验证、可拒绝、不可双开的机制。

**架构扩展**  
Redis 主从、数据库主备、分布式锁服务、调度主节点都可能遇到脑裂。成熟架构师面对高可用设计时，会先问“如果误切主会怎样”，而不是只关心切换速度。

**工程指标**  
- 双主写入事件数：核心风险指标。  
- 切主成功率：看自动恢复能力。  
- 仲裁失败率：判断控制面稳定性。  
- 数据冲突数：恢复成本信号。  
- 恢复耗时：业务影响衡量。

**证据链**  
- 心跳日志  
- 选主记录  
- 写入冲突审计  
- 仲裁节点状态

**低风险优化路径**  
1. 先识别是否存在双主风险  
2. 再引入仲裁和 fencing 机制  
3. 最后演练误切主恢复流程

**易错点**  
- 只追求切换快  
- 不防双主  
- 不演练脑裂恢复

## 13. 如何做全局唯一 ID 设计？

**核心结论**  
全局唯一 ID 设计要同时兼顾唯一性、趋势有序、性能、可用性和可运维性，不是只要“不重复”就够了。

**标准答案**  
设计全局唯一 ID 时，我一般先看业务要求：是不是必须趋势递增、有没有长度限制、是否暴露给外部、吞吐量多大、跨机房是否要生成。之所以不能只看唯一性，是因为订单号、流水号、日志 ID、消息 ID 对可排序性和可追踪性的要求往往不同。工程上常见方案有数据库自增、号段模式、雪花算法、UUID、Redis 递增等，不同方案在性能、可用性和有序性上取舍差异很大。面试里如果能补一句“ID 方案本质上是在唯一性、有序性和中心化程度之间做平衡”，会比较成熟。

**高分回答**
我先把需求拆开：数据库主键需要紧凑、趋势有序和高 TPS；对外订单号还需要不可枚举、校验位和业务可读性；trace ID 更重视本地生成和随机分布，通常不应强行共用一种编号。单库小规模可直接自增，号段模式把中心数据库分配的区间缓存在实例内，能降低发号压力但会产生空洞；雪花类方案用时间、节点号和序列本地生成，吞吐高且大致有序，但必须保证节点号唯一并处理时钟回拨；UUID 简单高可用，但随机长键可能加剧 B+ 树页分裂与索引体积。若选择雪花，我会明确位宽和寿命，节点号由协调系统租赁且防止重复，回拨在小范围内等待，超过阈值摘除实例并报警，不能冒险继续发号。多地域还要决定是否编码 region，避免中心服务跨区成为单点；数据库仍保留唯一约束作为最后防线。对外编号不直接暴露递增主键，可采用独立随机 ID 或加密映射。压测生成 TPS、P99、号段耗尽与中心故障，验证重复 ID 为零、重启/回拨安全、索引写放大和下游语言精度兼容，最终按具体用途选择最小复杂度方案。

**追问点答案**  
- UUID 为什么不总是合适？  
虽然简单，但太长、无序，对索引和可读性不友好。  
- 雪花算法为什么常见？  
因为性能高、趋势递增且不完全依赖中心服务。  
- 数据库自增为什么不适合大规模分布式？  
单一序列会形成跨节点协调与可用性约束；但可通过号段批量分配降低频率，所以是否不适合仍取决于吞吐、跨区与容灾要求。

**原理补充**  
ID 设计的本质，是在“谁负责发号”和“下游如何使用这个号”之间做系统性平衡。

**架构扩展**  
订单系统、支付流水、消息平台、日志追踪都离不开全局 ID。成熟架构师不会只背雪花算法，而是会先问：这个 ID 是给数据库主键用，还是给用户看，还是给链路追踪用，不同答案会直接影响选型。

**工程指标**  
- ID 生成 TPS：性能基线。  
- 重复 ID 事件数：底线指标。  
- 生成失败率：高可用信号。  
- 时钟回拨告警数：雪花类方案重点关注。  
- 索引写入性能：评估有序性收益。

**证据链**  
- 业务约束说明  
- 发号服务日志  
- ID 样本分布  
- 下游索引性能数据

**低风险优化路径**  
1. 先明确业务约束  
2. 再选最小复杂度方案  
3. 最后补可用性和监控

**易错点**  
- 只追求唯一，不看排序和长度  
- 盲目上复杂发号系统  
- 不监控时钟和重复事件

## 14. 如何避免跨服务强事务？

::: tip L2 深度训练
先缩小事实源和强一致边界，再设计状态机与补偿：[分布式一致性选择树](/deep-dives/distributed-stability/01-distributed-consistency)
:::

**核心结论**  
避免跨服务强事务的关键，是把业务拆成边界清晰的本地事务，再用事件、补偿和状态机连接起来。

**标准答案**  
我一般不会一上来就设计跨服务强事务，而是先回到业务边界上看能不能把一个大流程拆成多个本地事务，每个服务只对自己的数据负责，再通过事件驱动、状态推进和补偿完成整体协同。之所以要避免跨服务强事务，是因为它会把多个服务强耦合在一个提交点上，带来性能、可用性和故障恢复复杂度。工程上很有效的做法是把业务流程显式建成状态机，让每一步都可追踪、可重试、可补偿。面试里如果能补一句“分布式事务问题很多时候不是技术没选对，而是边界没拆对”，会很亮眼。

**高分回答**
我先识别真正必须原子成立的业务不变量，并尽量把这些数据放到同一个聚合和单库事务内。例如订单创建与“订单已受理”状态本地提交，库存只做有期限的预留，支付独立记账，履约在两者确认后启动；并不要求四个服务同一时刻提交。每个本地事务同时写 outbox，事件驱动 Saga 状态机推进，步骤带业务幂等键、版本和 deadline；失败时按语义选择向前重试还是逆序补偿，例如释放预留库存，而不是假装数据库回滚能撤销已发生的外部支付。对用户暴露 `处理中/已确认/失败` 等真实中间态，并定义 P99 收敛窗口和查询入口。若某一步不可补偿且错误代价极高，可以先预授权、延迟不可逆动作，或在极小边界内采用 TCC；只有所有参与方、性能与可用性都能接受时才考虑同步强事务。设计必须处理事件丢失、重复、乱序、协调器宕机和补偿失败，并以事实源对账闭环。监控流程各状态年龄、补偿成功率、超窗数量和人工介入率，通过在每个提交点注入故障验证可恢复。目标不是牺牲正确性，而是把“同时成功”改写成“核心不变量立即成立、其余流程可证明地收敛”。

**追问点答案**  
- 什么情况下容易误上强事务？  
通常是业务边界拆分不清、希望一步成功的心态太强。  
- 状态机为什么有帮助？  
因为它把流程显式化，方便追踪和补偿。  
- 事件驱动一定比强事务好吗？  
不一定；它提高解耦和可用性，却引入中间态、重复、乱序与补偿成本。若业务不允许中间态且参与边界很小，强事务可能更简单。

**原理补充**  
避免跨服务强事务的核心，不是逃避一致性，而是把一致性从“同步提交”转化成“可恢复流程”。

**架构扩展**  
订单、支付、履约、积分、优惠券这类业务特别容易被设计成一个大事务。成熟架构师往往先问：“真的需要所有服务同时提交吗？哪些可以延后确认？”很多问题一旦换个业务视角，复杂度就下来了。

**工程指标**  
- 跨服务同步调用数：耦合度信号。  
- 流程补偿成功率：评估替代方案稳定性。  
- 状态流转时长：业务感知关键。  
- 人工介入率：看流程是否够稳。  
- 服务耦合变更次数：反映架构边界质量。

**证据链**  
- 业务流程图  
- 状态机定义  
- 服务调用拓扑  
- 补偿和对账日志

**低风险优化路径**  
1. 先重画业务边界  
2. 再把大事务拆成本地事务 + 事件  
3. 最后补偿、对账和状态机

**易错点**  
- 一步成功执念太强  
- 服务边界混乱  
- 拆了异步但没补状态管理

## 15. 什么情况下必须强一致？

**核心结论**  
必须强一致的场景通常都和不可逆副作用、资金安全、库存准确性或法律合规直接相关。

**标准答案**  
我判断是否必须强一致时，通常会先看这条链路一旦短暂不一致，代价是不是不可接受。比如账户余额、支付记账、核心库存确认、券码核销、防重扣费这类场景，短暂不一致就可能直接造成资金风险、超卖、重复发货或用户投诉，这时就要更偏强一致。之所以不能把所有链路都做成强一致，是因为成本和复杂度会非常高。工程上成熟做法通常是只把最核心的一小段链路做强一致，外围通知、积分、搜索、报表仍然走最终一致。面试里如果能补一句“强一致不是技术洁癖，而是对业务风险的响应”，会很到位。

**高分回答**
我不用“支付都要强一致”这种宽泛结论，而是逐条写业务不变量：同一支付流水只能入账一次、账户总账借贷平衡、同一库存单元确认量不能超过可售量、同一券码只能核销一次。若任何客户端在成功返回后都必须立即看到该写入，或两个并发写不能都成功，就需要在这段权威写路径上提供线性一致或可串行化语义；若只是页面展示稍晚、搜索索引未刷新，则可以读旧值并最终收敛。实现优先收缩边界：把余额与账务分录放在同一账本事务，以唯一约束和条件更新守不变量；库存先原子预留再异步履约，而不是让订单、支付、库存全都进入分布式强事务。选择强一致还要接受分区时拒绝写、跨区延迟上升和吞吐降低，并明确超时结果未知时用查询而非重试。审计时分别验证重复、并发、故障切换和旧副本读取，指标看不变量违规必须为零，同时观察交易成功率、TP99 和不可写时长。判断标准是“短暂错误能否被用户安全感知并低成本补偿”，而不是数据看起来重要；只对补偿代价高于协调代价的最小范围付出强一致成本。

**追问点答案**  
- 余额展示一定要强一致吗？  
不一定，展示层有时可以容忍延迟，但扣款记账必须强一致。  
- 库存为什么经常被单独拿出来？  
因为它直接影响是否超卖和履约承诺。  
- 强一致一定意味着分布式事务吗？  
不一定；优先把同一不变量的数据收进单一事实源，用本地事务、唯一约束和条件更新完成，跨服务流程再异步协调。

**原理补充**  
强一致要求的本质，是某些业务副作用一旦错了，后续补偿成本远高于前置约束成本。

**架构扩展**  
成熟架构师不会泛泛地说“支付要强一致”，而是会继续拆：支付流水入账和账户扣减要强一致，但支付后发短信、发积分就不用。真正关键的是收缩强一致范围，而不是扩大强一致范围。

**工程指标**  
- 资金或库存差错数：核心业务风险指标。  
- 重复扣减事件数：强一致设计成效体现。  
- 对账差异率：帮助验证收敛质量。  
- 交易成功率：看一致性方案是否拖垮可用性。  
- 人工修复成本：衡量是否值得前置强一致。

**证据链**  
- 业务风险评估  
- 历史事故案例  
- 对账报表  
- 核心事务链路日志

**低风险优化路径**  
1. 先识别不可逆副作用链路  
2. 再收缩强一致范围  
3. 最后把外围链路外移为最终一致

**易错点**  
- 所有链路都想强一致  
- 不分核心和外围  
- 不从业务风险判断

## 16. 分布式系统如何做降级？

**核心结论**  
分布式降级不是简单返回空结果，而是有策略地保核心、砍外围、控放大、留恢复空间。

**标准答案**  
做降级时，我一般先给业务链路分级，把核心交易、关键查询、边缘功能和装饰性能力分层，然后在高压或故障场景下优先保障核心链路，把推荐、评论、埋点、非核心异步通知、复杂聚合查询这些能力降掉。之所以降级很重要，是因为系统出问题时，不可能所有能力都同时保住。工程上降级常见方式包括返回缓存兜底、关闭非核心模块、跳过异步动作、切只读、减少联动调用和限制复杂请求。面试里如果能补一句“降级不是放弃功能，而是主动缩小战线”，会比较成熟。

**高分回答**
我先按业务损失划分 P0 核心写、关键读、可延迟任务和装饰功能，为每个依赖建立“正常—轻度—重度—恢复”矩阵，而不是事故时临时决定返回空。触发信号既看本服务 SLO，也看线程池、连接池、队列和下游错误率；例如推荐依赖 TP99 超过 300ms 时立即返回带版本与最大陈旧时间的缓存，数据库写水位危险时暂停报表和批任务，无法确认主库权威时只读或拒绝资金写。降级开关需要动态配置、权限审计、作用域、TTL 和默认安全值，自动动作有最大持续时间，人工可一键覆盖。关键是阻断放大：减少 fan-out、关闭重试、限制大查询并保留核心链路独立资源池，不能用一个昂贵兜底继续压垮同一依赖。用户侧给出明确的处理中、稍后重试或旧数据标识，不能把错误伪装成“库存为零”。恢复采用滞后窗口和小比例回切，先确认积压、缓存回源和下游容量，避免全量恢复造成第二次冲击。演练每个开关的触发、业务影响与回滚，监控核心成功率、降级命中、兜底正确率、用户投诉和恢复时长；降级成功的标准是以可接受的功能损失换取核心不变量和恢复空间。

**追问点答案**  
- 降级和限流有什么区别？  
限流是控制流量进入，降级是减少系统内部处理复杂度。  
- 降级一定要提前做开关吗？  
关键路径必须预置并演练；开关还要有权限、审计、作用域、TTL、默认值和回切策略，避免误开后长期遗留。
- 什么是只读降级？  
保查询、停写入，常用于依赖不稳定或主链路保护。

**原理补充**  
降级的核心，是在资源不足时优先把有限能力留给最重要的业务目标。

**架构扩展**  
大促、热点、依赖故障、机房抖动时，真正决定系统是不是能活下来的往往是降级设计。成熟架构师不会等事故来了再想“关哪个功能”，而是平时就把降级矩阵和开关准备好。

**工程指标**  
- 降级命中率：看保护是否启用。  
- 核心链路成功率：降级后的首要目标。  
- 非核心功能关闭时长：评估影响范围。  
- 用户投诉率：看降级是否可接受。  
- 恢复时间：判断降级后回切能力。

**证据链**  
- 降级开关配置  
- 降级执行日志  
- 核心/非核心链路分级文档  
- 事故复盘记录

**低风险优化路径**  
1. 先做业务分级  
2. 再为每层设计降级动作  
3. 最后演练触发和回切

**易错点**  
- 没有业务分级  
- 降级动作不提前准备  
- 只会关流量，不会简化系统内部负载

## 17. 如何处理重复请求？

**核心结论**  
处理重复请求的关键不是阻止请求到来，而是确保重复到来后不会重复产生业务副作用。

**标准答案**  
面对重复请求时，我一般会先区分来源：是客户端重试、网关重放、消息重复投递，还是用户多次点击。然后再按业务影响设计幂等键、唯一约束、状态机校验或前端防抖。之所以要这样分层，是因为不同来源的重复，最有效的治理点并不一样。工程上更重要的是后端业务幂等，而不是只依赖前端按钮置灰，因为真正的重复很多来自系统内部。面试里如果能补一句“重复请求问题最终是业务副作用控制问题”，会比只说 token 更完整。

**高分回答**
我先确定“相同业务意图”的稳定标识，而不是依赖每次都会变化的 traceId。客户端创建订单携带 `tenant + clientOrderNo`，服务端保存请求摘要、处理中状态和最终响应；并发首次请求通过数据库唯一约束获胜，其余请求若摘要不同则拒绝键复用，摘要相同则返回原订单或明确的处理中状态。更新类操作以版本和状态机做条件更新，例如退款只能从 `PAID` 进入 `REFUNDING`，受影响行数为零后读取现状，不能再次调用支付副作用。跨服务传递同一业务键，消息消费者在本地事务中同时写 inbox/去重记录和业务数据；调用外部支付超时表示结果未知时，先按商户流水查询，而不是生成新键再次扣款。幂等记录保留周期覆盖客户端重试、消息留存、灾备回放和人工补偿，过期后仍由账本唯一约束兜底。前端置灰与网关短期去重只用于降低噪声，不能承担正确性。测试并发双击、响应丢失、网关重放、消费者崩溃与历史消息回放，验收重复副作用为零，同时观察冲突率、处理中超时、原结果复用率和记录清理水位。

**追问点答案**  
- 前端防抖够不够？  
不够，系统内部重试和消息重投仍会重复触发。  
- 什么时候适合唯一索引？  
当业务天然有唯一键且副作用集中在数据库写入时。  
- 状态机校验适合什么场景？  
适合有明确合法跃迁的流程业务；应使用带当前状态或版本的条件更新，使两个并发请求最多一个成功，而不是先查后改。

**原理补充**  
重复请求的真正风险，不在“请求多了一次”，而在“副作用被做了两次”。

**架构扩展**  
创建订单、支付回调、发券、扣库存、消息消费，这些场景都必须正视重复问题。成熟架构师会从请求入口、业务核心、副作用落点三层一起看，而不是只堵入口。

**工程指标**  
- 重复请求率：识别来源规模。  
- 重复副作用事件数：核心质量指标。  
- 幂等冲突数：观察系统行为。  
- 前端重复点击率：辅助判断用户行为。  
- 重试成功率：看系统自恢复路径。

**证据链**  
- 请求唯一标识  
- 业务状态流转  
- 唯一索引或幂等表  
- 客户端重试日志

**低风险优化路径**  
1. 先识别重复来源  
2. 再在副作用落点做幂等  
3. 最后补入口防抖和监控

**易错点**  
- 只做前端防抖  
- 不在后端做副作用保护  
- 重复来源识别不清

## 18. 如何处理网络分区导致的数据不一致？

**核心结论**  
网络分区导致的不一致，核心不是“怎么完全避免”，而是“怎么缩小影响范围、阻止继续写坏、并可恢复收敛”。

**标准答案**  
面对网络分区引发的数据不一致，我通常先做三件事：第一，识别哪些节点或链路还允许继续写；第二，把高风险写能力收缩，避免继续分叉；第三，用日志、版本号、对账和补偿让数据后续收敛。之所以不能只等网络恢复再说，是因为分区期间错误写入会越积越多。工程上常用的手段包括选主仲裁、版本比较、冲突检测、重放日志、人工校验和业务对账。面试里如果能补一句“分区后的第一目标是止损，第二目标才是修复”，会显得非常像做过系统的人。

**高分回答**
我会在设计阶段就为每类数据定义分区策略。资金、库存等权威写只有取得多数派和最新任期的一侧继续，少数派拒绝写或只读，并由数据面 fencing 拒绝旧主；商品资料、购物车等可合并数据可在两侧继续，但写入携带版本向量、操作日志或业务版本，且明确冲突规则与不一致窗口。检测到分区后先冻结切主和高风险批任务，确认影响的节点、租户、时间范围与最后共同提交点，避免自动化在网络抖动中来回切换。网络恢复并不代表数据恢复：先隔离旧写入口，按已提交日志和任期确定权威历史；可交换、幂等的操作自动重放，单调状态取更高合法版本，集合类可按业务规则合并，资金冲突则对照账本和外部流水人工裁决，绝不能普遍用最后写入时间覆盖。修复过程限速、可暂停、全量留审计，并通过行数、校验和及金额/库存等业务不变量验证；随后再灰度恢复写入。演练单向隔离、跨区断链、旧主复活和修复中断，指标包括分区发现与止写时间、冲突规模、自动修复率、收敛 P99 和人工损失。核心是提前决定谁能写、如何记录因果、如何证明恢复，而不是故障后临时选一份“看起来更新”的数据。

**追问点答案**  
- 网络恢复后为什么还不算结束？  
因为分区期间可能已经产生冲突数据，需要后续对账和修复。  
- 怎么判断谁的数据是正确的？  
先依据法定写权限、任期和已提交日志确定权威历史，再用业务不变量与外部事实源裁决；墙上时钟的“最后写入”通常不足以证明正确。
- 什么场景能自动修复？  
幂等更新、单调状态推进、可重放事件链路更容易自动收敛。

**原理补充**  
网络分区问题的本质，是状态在多个副本之间失去同步后，谁还能继续代表真实业务结果。

**架构扩展**  
缓存集群、主从数据库、多机房服务、分布式锁服务都可能遇到这个问题。成熟架构师的思路通常是：先防止分区期间继续写坏，再准备恢复后的核对和补偿，而不是只强调“高可用切过去了”。

**工程指标**  
- 分区期间写入冲突数：止损效果指标。  
- 数据版本冲突率：修复难度信号。  
- 对账差异数：恢复阶段关键指标。  
- 修复时长：业务恢复能力衡量。  
- 人工介入率：自动化修复成熟度体现。

**证据链**  
- 版本号或日志序列  
- 分区事件记录  
- 对账结果  
- 冲突修复操作日志

**低风险优化路径**  
1. 先停高风险写入  
2. 再用版本和日志识别冲突  
3. 最后通过补偿或人工修复收敛

**易错点**  
- 只关注网络恢复  
- 不做止损  
- 没有对账和冲突处理能力

## 19. 如何定义一致性 SLA？

**核心结论**  
一致性 SLA 不是一句“最终一致”就完了，而是要把一致性要求量化成时间、范围、误差和恢复承诺。

**标准答案**  
定义一致性 SLA 时，我通常会先明确业务对这条链路的容忍度，比如允许多久内完成同步、允许多少比例的短暂延迟、出现不一致时多久必须自动修复、多久需要人工介入。之所以要量化，是因为不一致如果没有边界，就无法监控、无法验收，也无法和业务达成共识。工程上一致性 SLA 往往包含收敛时长、差异率、补偿成功率和人工修复时限。面试里如果能补一句“没有量化的一致性要求，本质上是不可治理的”，会很加分。

**高分回答**
我会按用户可见的业务事件定义 SLI，而不是从消息发送成功开始计时。例如“支付渠道确认成功”到“订单页显示已支付”的延迟，目标可定义为 99.9% 在 5 秒内、99.99% 在 1 分钟内、任何超过 10 分钟的记录进入人工处理；同时约束每百万订单的超窗数、重复副作用为零和每日对账未解决差异为零。测量起点来自权威事实源，终点来自用户读取或目标系统的业务状态，携带同一业务 ID 关联；只看 MQ lag 不能代表真实一致性。不同链路分级：资金账本的不变量立即成立，订单展示秒级，积分分钟级，报表小时级，并说明窗口内用户体验与重复操作保护。错误预算将 SLA 转为治理动作，例如 30 天允许 0.01% 超过 1 分钟；小时 burn rate 过快时暂停发布、扩消费者或启用补偿。还要分别监控未收敛数量与最老年龄，避免高流量下比例很好看却遗留少量永久孤儿。通过抽样读校验和全量对账验证监控本身，定期演练消息停机、重放和补偿失败。这样一致性 SLA 同时包含时间、比例、不可违反的不变量、超窗处置和证据来源。

**追问点答案**  
- 为什么一致性也要 SLA？  
因为它本质上也是一项服务承诺。  
- 什么链路适合定义秒级 SLA？  
用户感知强、业务价值高、但允许短暂异步的链路。  
- 差异率高但能很快修复，算不算达标？  
不能只凭修复快判断；必须同时满足差异比例、收敛分位数、不可违反的不变量和用户影响预算，资金重复等事件即使秒级修复也可能不可接受。

**原理补充**  
一致性 SLA 的核心，是把抽象的一致性讨论落成可监控、可验收的业务约束。

**架构扩展**  
搜索索引更新、消息通知、积分同步、订单状态展示，这些链路都非常适合明确一致性 SLA。成熟架构师会主动和业务约定“几秒内可见”“几分钟内自动补齐”，这样系统才有明确目标。

**工程指标**  
- 收敛时长：一致性 SLA 核心指标。  
- 差异率：看短暂不一致规模。  
- 补偿成功率：决定能否稳定达标。  
- 人工介入时长：尾部问题指标。  
- 用户侧可见异常数：最终业务结果。

**证据链**  
- SLA 文档  
- 对账和补偿数据  
- 用户投诉统计  
- 业务验收记录

**低风险优化路径**  
1. 先和业务约定量化目标  
2. 再把目标映射到监控  
3. 最后用对账和告警持续校准

**易错点**  
- 只说最终一致，不量化  
- 没监控无法验收  
- 技术和业务目标脱节

## 20. 你如何做分布式架构验收？

::: tip L2 深度训练
把一致性、故障传播和恢复指标组合成架构验收：[分布式系统与稳定性](/deep-dives/distributed-stability/)
:::

**核心结论**  
分布式架构验收不能只看功能跑通，而要同时验证一致性、可用性、故障恢复、扩展能力和可观测性。

**标准答案**  
做分布式架构验收时，我一般会分五层看：第一是功能正确性，确认业务流程闭环可跑通；第二是一致性和幂等，验证重试、重复消息、顺序错乱下结果是否稳定；第三是容量和性能，确认峰值吞吐、延迟和安全水位；第四是故障恢复，演练节点宕机、依赖超时、消息积压、网络抖动等场景；第五是可观测性，看日志、指标、链路和告警是否足够支撑排障。之所以这样验收，是因为分布式系统最危险的问题很多平时看不出来，必须靠演练和极端场景验证。面试里如果能补一句“分布式验收的目标不是证明系统没问题，而是证明问题来了我们也知道怎么处理”，会非常加分。

**高分回答**
我会先把架构承诺转成可判定的验收矩阵：业务不变量与一致性窗口、峰值和 N-1 容量、可用性与 TP99、RTO/RPO、安全边界、成本上限及责任人，每项都有场景、通过线和证据。正常态验证核心与长尾流程、并发和数据规模；异常态系统性注入请求重复/乱序/丢失、调用超时与重试风暴、缓存失效、数据库热点、消息积压、节点/可用区故障、网络分区和旧主复活，在每个本地提交点终止进程，检查状态机能否恢复。验收不仅看接口成功，还用唯一约束、账本平衡、库存不为负、全量对账和数据校验和证明结果正确；压测采用真实到达率和热点分布，给出容量拐点与故障态安全水位。可观测性要现场证明：告警能在目标时间命中，值班者能从业务 ID 串起日志、指标和 trace，运行手册可完成止血、回滚、积压清理与数据复核。所有演练记录实际 RTO/RPO、错误预算消耗和用户影响，失败项有负责人、截止时间与重新验收，重大缺口不得以“后续观察”放行。最后检查单点、供应商配额、权限、发布和成本等非功能风险。架构验收的产物是可重复的自动测试与演练证据，而不是一份只证明 happy path 的评审 PPT。

**追问点答案**  
- 为什么只跑 happy path 不够？  
因为分布式问题大多出在异常态和边界条件。  
- 验收一定要做故障演练吗？  
关键分布式链路必须做；没有节点、依赖、网络与恢复场景的实际证据，就不能声称高可用和最终一致已经验收。
- 可观测性为什么也是验收项？  
因为系统出问题后能不能定位，本身就是能力的一部分。

**原理补充**  
分布式架构验收的核心，是验证系统在“不理想世界”里是否依然可控。

**架构扩展**  
成熟团队的验收不会停留在 API 测通过，而是会把压测、重试、重复消费、超时、网络波动、节点切换和监控演练都串起来。真正有价值的架构，不是平时看起来漂亮，而是故障来时扛得住、看得见、拉得回来。

**工程指标**  
- 功能通过率：基础项。  
- 峰值吞吐与 TP99：性能项。  
- 故障恢复时长：高可用项。  
- 不一致差异率：一致性项。  
- 告警命中率与定位时长：可观测性项。

**证据链**  
- 验收清单  
- 压测报告  
- 故障演练记录  
- 监控和告警截图

**低风险优化路径**  
1. 先按能力维度列验收项  
2. 再做正常态和异常态验证  
3. 最后复盘缺口并补运行手册

**易错点**  
- 只验功能不验故障  
- 不把观测能力纳入标准  
- 没有异常场景回归
