# 架构模块1 - 系统设计与容量规划 - 标准答案库

## 1. 如何估算系统峰值 QPS？

**核心结论**  
峰值 QPS 不能拍脑袋估，通常要从业务量、时间分布、峰值系数和关键链路拆分一步步推出来。

**标准答案**  
我做峰值 QPS 估算时，一般不会直接报一个数字，而是先从业务量往下拆。比如先拿到 DAU、日订单量、活动参与人数或页面访问量，再结合流量分布判断高峰集中在多长时间窗口里，然后乘一个峰值放大系数，最后再拆到具体接口。之所以要这么做，是因为同样是 100 万 DAU，电商大促和普通资讯站的流量集中度完全不一样，平均值没有意义。工程上还要区分总入口 QPS、核心写接口 QPS、热点查询 QPS 和下游依赖 QPS，因为系统不是一个点扛住就算完事了。面试里如果能补一句“峰值估算必须按链路拆分，而不是只报网关数字”，会显得很成熟。

**追问点答案**  
- 为什么平均 QPS 没意义？  
因为系统真正出问题通常发生在短时峰值，而不是全天平均值。  
- 峰值系数怎么定？  
可以参考历史流量、活动经验、行业经验，再结合保守冗余。  
- 要不要考虑重试流量？  
一定要考虑，高峰期重试和失败补偿会放大真实流量。

**原理补充**  
容量估算的本质，是把“业务规模”映射成“系统在最坏时刻需要承受的技术负载”。

**架构扩展**  
真实系统里最常见的误判，就是业务方给一个日活数字，技术团队直接换算成平均 QPS，然后线上在活动开始的前几分钟被打穿。成熟团队做容量评估时，会把流量分成自然流量、活动流量、失败重试流量和异常放大量，再分别进模型，而不是只算一条曲线。

**工程指标**  
- 峰值入口 QPS：核心容量基线。  
- TP99：看峰值下延迟是否已劣化。  
- 错误率：高峰期最先抬头的稳定性信号。  
- 重试请求占比：帮助识别放大流量。  
- 下游依赖成功率：说明入口扛住不等于全链路扛住。

**证据链**  
- 历史访问日志  
- 活动流量预测  
- 接口访问分布  
- 重试和超时日志

**低风险优化路径**  
1. 先估入口峰值  
2. 再拆核心链路和热点接口  
3. 最后补冗余和异常流量放大项

**易错点**  
- 用平均值代替峰值  
- 只估入口不估链路  
- 忽略重试、回源和异常放大

## 2. 你如何从 DAU 推导吞吐量？

**核心结论**  
从 DAU 推导吞吐量，关键不是人数本身，而是人均行为频次、时间分布和请求链路放大倍数。

**标准答案**  
从 DAU 推吞吐量时，我通常会先把 DAU 变成人均行为模型，比如人均打开次数、人均查询次数、人均下单次数，再根据场景把这些行为映射到对应接口。然后还要判断这些行为是在全天均匀分布，还是集中在午高峰、晚高峰或活动时段。之所以不能直接拿 DAU 除以一天秒数，是因为用户行为天然是簇状分布的，而且一次业务动作背后往往会触发多次内部调用。工程上我还会补上缓存未命中回源、MQ 异步链路、数据库写入放大这些吞吐放大项，这样算出来的结果更接近真实系统负载。

**追问点答案**  
- 同样 100 万 DAU，为什么吞吐差异很大？  
因为人均行为频次、时段集中度和业务链路复杂度都可能完全不同。  
- 页面 PV 和接口 QPS 是什么关系？  
一个页面访问背后通常对应多个接口调用，不能简单一比一。  
- 为什么要算内部放大倍数？  
因为系统压力不仅来自用户请求，还来自链路里的派生调用。

**原理补充**  
DAU 是业务规模指标，吞吐量是系统负载指标，两者中间必须经过行为建模才能正确转换。

**架构扩展**  
电商首页、短视频 feed、IM 会话这些业务，表面上都是 DAU 很高，但真正打系统的点完全不同。首页更像高读流量，订单更像高一致性写入，IM 更像长连接和消息分发。成熟架构师不会被一个业务总量数字带偏，而是会追问“用户的关键动作是什么，系统的压力点在哪”。

**工程指标**  
- DAU/MAU：反映业务规模。  
- 人均请求次数：行为强度核心参数。  
- 单业务动作内部调用数：链路放大依据。  
- 峰值时段请求集中度：帮助估算吞吐峰值。  
- 缓存命中率：直接影响最终回源吞吐。

**证据链**  
- 用户行为埋点  
- PV/UV 统计  
- 接口调用拓扑  
- 链路追踪数据

**低风险优化路径**  
1. 先建人均行为模型  
2. 再映射接口吞吐  
3. 最后加入链路放大和缓存因素

**易错点**  
- 把 DAU 直接等于 QPS  
- 不区分读写行为  
- 不看内部调用链

## 3. 容量规划里最容易漏掉什么？

**核心结论**  
容量规划最容易漏掉的，不是主链路本身，而是重试、回源、补偿、后台任务和异常流量带来的放大效应。

**标准答案**  
做容量规划时，大家通常会先看入口流量和数据库容量，但最容易漏掉的其实是放大量，比如接口超时后的重试、缓存失效后的回源、消息积压后的补偿消费、批处理任务、索引构建、日志打点、异步通知这些附加负载。之所以这些东西危险，是因为它们平时不显山不露水，但一到高峰或故障状态，就会从“后台小流量”变成压垮系统的关键因素。工程上我通常会把系统流量分成自然流量、派生流量、故障流量和运维流量四类，再分别评估容量。这样比只看业务请求更稳。

**追问点答案**  
- 重试为什么危险？  
因为它会把一个失败问题迅速放大成全链路流量风暴。  
- 后台任务为什么也要进容量模型？  
因为很多系统不是白天只有用户流量，夜间批任务也可能打爆资源。  
- 日志和监控也会吃容量吗？  
会，高峰期同步日志和高频埋点有时能明显拖慢主链路。

**原理补充**  
容量问题真正难的地方，不在业务流量本身，而在系统进入异常态后会出现多层次自我放大。

**架构扩展**  
很多线上事故不是因为系统没扛住正常业务，而是缓存雪崩后回源把 DB 打穿，随后客户端重试、网关重试、消费补偿一起叠加，最后整个系统级联故障。成熟团队做容量规划时，会重点问一句：“系统出问题时，流量会怎么被放大？”这比正常态容量更有价值。

**工程指标**  
- 重试率：判断是否有放大风险。  
- 回源比例：缓存类系统关键指标。  
- 异步积压量：帮助识别补偿洪峰。  
- 后台任务 CPU/IO 占比：看资源争抢。  
- 异常态总请求放大倍数：衡量故障冲击面。

**证据链**  
- 重试日志  
- MQ backlog  
- 缓存命中率趋势  
- 任务调度执行记录

**低风险优化路径**  
1. 先补全所有派生流量清单  
2. 再按正常态和故障态分别建模  
3. 最后给高风险链路做限流和降级预案

**易错点**  
- 只看用户请求  
- 低估异常态流量  
- 忽略后台任务和日志链路

## 4. 读多写少系统怎么设计？

**核心结论**  
读多写少系统的核心目标是把读扩展做足，用缓存、只读副本和异步化降低主存储压力，同时控制一致性成本。

**标准答案**  
如果一个系统明显是读多写少，我会优先把读链路设计成多级加速结构，比如本地缓存、分布式缓存、只读副本、搜索索引，尽量让热点读取不要直接打主库。之所以这样做，是因为这类系统真正的瓶颈一般在读放大，而不是写冲突。工程上还要区分哪些数据允许短暂不一致，哪些必须从强一致来源读取，比如商品详情和库存就不该走同一策略。面试里回答这类问题时，最好补一句：读多写少不等于只堆缓存，还要考虑缓存失效、热点 key、回源保护和数据更新链路。

**追问点答案**  
- 一定要上本地缓存吗？  
不一定，要看热点集中度和节点数，否则会增加一致性复杂度。  
- 只读副本的主要价值是什么？  
把查询压力从主库上分离出去。  
- 搜索索引适合什么场景？  
适合复杂检索、多字段过滤和排序。

**原理补充**  
读扩展的核心思想，是让“昂贵的一致性写入”少做，而让“可复制的读取能力”尽量横向铺开。

**架构扩展**  
资讯、电商详情、配置中心、内容平台这些都是典型读多写少系统。真实设计时，问题往往不在“会不会加缓存”，而在“数据更新后允许多久可见”“热点发布时会不会击穿”“回源失败怎么兜底”。成熟团队会把一致性等级和缓存层级一起设计，而不是先上缓存再补洞。

**工程指标**  
- 缓存命中率：读扩展效果核心指标。  
- 主库读 QPS：看是否真的卸载成功。  
- 回源比例：判断缓存压力传导。  
- 热点 key QPS：高峰期重点观察。  
- 数据可见性延迟：评估一致性成本。

**证据链**  
- 读写比统计  
- 热点数据分布  
- 缓存层命中日志  
- 主从读负载数据

**低风险优化路径**  
1. 先区分一致性等级  
2. 再设计缓存和副本层  
3. 最后补回源保护和热点治理

**易错点**  
- 读多写少就无脑堆缓存  
- 不区分不同数据的一致性要求  
- 忽略热点击穿

## 5. 写多读少系统怎么设计？

**核心结论**  
写多读少系统的重点不是读优化，而是把写路径做稳，包括顺序写、异步削峰、幂等控制和分区扩展。

**标准答案**  
面对写多读少系统，我会优先关注写路径的承压能力，比如是否需要消息队列削峰、是否要按业务主键分区、是否可以批量写、是否要做顺序落盘或日志化处理。之所以写多场景更难，是因为写操作通常伴随锁竞争、一致性要求和存储放大，不像读那样容易水平复制。工程上还要看写入是不是可异步、是否允许最终一致，以及失败后的补偿方式。面试里这题不要只说“分库分表”，更重要的是说明写放大来自哪里，以及怎么把同步写压缩到最小。

**追问点答案**  
- 为什么写多系统更容易遇到瓶颈？  
因为写入通常要落主存储、持久化、加锁、刷日志。  
- MQ 削峰适合所有写场景吗？  
不适合，强一致核心交易链路要谨慎异步化。  
- 批量写为什么有效？  
因为可以降低网络、事务和日志刷盘的固定成本。

**原理补充**  
写多系统的瓶颈本质上是“持久化和一致性成本”叠加后的资源争用问题。

**架构扩展**  
日志平台、埋点采集、订单流水、IoT 上报、监控指标写入，这些系统都更像写多读少。成熟架构师会优先想“写能不能排队、能不能分区、能不能批量、能不能异步确认”，而不是先上复杂查询引擎，因为真正扛不住的常常是写路径。

**工程指标**  
- 写入 TPS：核心吞吐指标。  
- 持久化延迟：看刷盘或提交成本。  
- 锁等待时间：判断并发写冲突。  
- 分区热点分布：看扩展是否均衡。  
- 失败补偿积压量：评估写入稳态。

**证据链**  
- 写入链路 tracing  
- 事务日志  
- 锁等待和慢 SQL  
- 分区写入分布

**低风险优化路径**  
1. 先找写瓶颈点  
2. 再做批量化、异步化或分区  
3. 最后再评估分库分表和存储替换

**易错点**  
- 写多就直接分库分表  
- 不分析写放大来源  
- 忽略幂等和失败补偿

## 6. 热点流量如何识别和治理？

**核心结论**  
热点流量治理要先识别“热点在哪里、持续多久、是否可预测”，再按限流、隔离、缓存和降级分层处理。

**标准答案**  
识别热点流量时，我一般先从访问分布入手，看是不是少数 key、少数接口、少数用户或少数分区承担了大部分流量，然后再判断它是短时突发、周期性热点还是持续性热点。之所以这一步很重要，是因为不同热点的治理方式完全不同。工程上，热点治理常见手段包括本地缓存、热点 key 拆分、请求合并、限流、异步化、只读副本扩展和服务隔离。面试里如果能补一句“热点治理不是只压流量，而是要保护后端资源不被集中打穿”，回答会更完整。

**追问点答案**  
- 热点 key 为什么危险？  
因为它会让单点缓存、单分片、单数据库节点承受不成比例的压力。  
- 热点一定要拆 key 吗？  
不一定，很多时候先做本地缓存和请求合并就够了。  
- 请求合并适合什么场景？  
适合高频重复查询的场景，比如商品详情、配置读取。

**原理补充**  
热点问题本质上不是总流量太大，而是流量分布极不均匀导致局部资源先崩。

**架构扩展**  
秒杀商品、热门直播间、明星新闻、配置中心、爆款内容，这些都是典型热点场景。成熟团队面对热点，不会只加机器，而是先把热点流量从“直接打存储”改造成“尽量在更靠前的层被吸收”，比如 CDN、本地缓存、共享缓存、边缘限流。

**工程指标**  
- Top key 流量占比：核心热点识别指标。  
- 单分片 QPS：看是否局部过载。  
- 缓存命中率：热点治理有效性关键。  
- 限流命中率：衡量保护是否生效。  
- 回源失败率：热点失控时重要信号。

**证据链**  
- key 访问分布  
- 分片负载分布  
- 热点时段日志  
- 限流和缓存监控

**低风险优化路径**  
1. 先识别热点对象和范围  
2. 再把流量前移吸收  
3. 最后做拆分、隔离和降级

**易错点**  
- 只会说加机器  
- 不区分全局流量和局部热点  
- 没有后端保护意识

## 7. 如何做多级缓存容量设计？

**核心结论**  
多级缓存设计不能只看总容量，要看命中路径、热点分布、失效策略和每一层承担的职责。

**标准答案**  
设计多级缓存时，我通常会先明确各层职责，比如浏览器/CDN 负责边缘静态内容，本地缓存负责极热点短路径数据，分布式缓存负责共享热点和主回源保护。之所以要这么拆，是因为不同层的容量、延迟和一致性成本不一样。工程上做容量设计时，不只是算总内存，还要估 key 数量、平均 value 大小、热点集中度、淘汰策略和扩容阈值，否则缓存很容易在高峰期抖动。面试里如果能补一句“多级缓存设计的核心不是层数多，而是每层边界清晰”，会比较加分。

**追问点答案**  
- 本地缓存为什么不能无限加？  
因为会增加一致性复杂度和节点内存压力。  
- 分布式缓存容量怎么算？  
通常按 key 数、value 平均大小、冗余副本和安全余量一起算。  
- 为什么要看热点集中度？  
因为热点不均匀时，平均容量并不能代表真实压力。

**原理补充**  
多级缓存本质上是在不同延迟和一致性成本之间做分层命中优化。

**架构扩展**  
真实系统里，最常见的问题不是“有没有缓存”，而是本地缓存太小扛不住热点、本地缓存太大难同步、Redis 容量估错导致淘汰抖动。成熟架构师做缓存容量设计时，会把命中率目标、失效策略和故障回源一起纳入设计，而不是只给一个 GB 数字。

**工程指标**  
- 各层命中率：判断职责划分是否合理。  
- 淘汰次数：看容量是否临界。  
- 回源比例：衡量缓存是否真正挡住流量。  
- 热点 key 分布：评估局部压力。  
- 缓存节点内存水位：扩容阈值核心依据。

**证据链**  
- key 数量统计  
- value 大小分布  
- 命中率面板  
- 淘汰和回源日志

**低风险优化路径**  
1. 先定义各层职责  
2. 再估算 key/value 和热点分布  
3. 最后按水位阈值做弹性扩容

**易错点**  
- 把所有缓存层当成一层算  
- 忽略热点集中度  
- 容量设计不带安全余量

## 8. 如何规划数据库连接池大小？

**核心结论**  
连接池大小不是越大越好，要结合数据库并发承载、应用实例数、SQL 耗时和等待策略综合确定。

**标准答案**  
规划数据库连接池时，我一般先反推单实例需要支撑的并发查询量，再结合单 SQL 平均耗时和峰值耗时估算需要的活跃连接数，然后再和数据库最大连接数、实例数量一起做约束校验。之所以不能把连接池开得很大，是因为连接不是免费资源，过多连接会增加数据库上下文切换、锁竞争和内存占用，反而让系统更慢。工程上通常还要给非业务连接、运维连接和突发抖动留余量，并且要配合超时、等待队列和熔断策略一起看。面试里如果你能说出“连接池大小是系统级约束，不是单应用局部调优”，会很稳。

**追问点答案**  
- 为什么连接池大了反而可能变慢？  
因为数据库并发过高会增加争用和调度成本。  
- 单实例和全集群怎么换算？  
要用数据库总连接预算反推每个应用实例上限。  
- 只看 CPU 核数够吗？  
不够，还要看 SQL 耗时、锁冲突和 IO 模型。

**原理补充**  
连接池本质上是应用对数据库并发访问的节流器，而不是无限并发加速器。

**架构扩展**  
很多线上问题不是数据库本身慢，而是某次扩容后应用实例数翻倍，但连接池配置没重新预算，结果数据库总连接直接被打满。成熟团队会把连接池参数放在容量模型里统一管理，而不是每个服务各自随便配。

**工程指标**  
- 活跃连接数：看是否接近上限。  
- 连接等待时间：关键瓶颈信号。  
- 数据库 CPU/IO：判断是连接不足还是 DB 本身过载。  
- 慢 SQL 占比：辅助解释连接占用时间。  
- 拒绝获取连接次数：说明池配置或负载异常。

**证据链**  
- 连接池监控  
- DB 最大连接配置  
- SQL 耗时分布  
- 应用实例数量

**低风险优化路径**  
1. 先估活跃连接需求  
2. 再按 DB 总预算分配到实例  
3. 最后配等待超时和熔断保护

**易错点**  
- 觉得连接池越大越安全  
- 只看单实例，不看集群总量  
- 不结合 SQL 时延分析

## 9. 线程池参数如何按容量估算？

**核心结论**  
线程池参数要根据任务类型、目标并发、执行耗时和下游承载能力来定，不能只抄默认值。

**标准答案**  
估算线程池参数时，我会先区分任务是 CPU 密集还是 IO 密集，然后再看单任务平均耗时、峰值耗时以及下游依赖是否能承受更高并发。之所以线程池不能随便开，是因为线程数太少会排队，太多又会造成上下文切换和下游雪崩。工程上通常要一起看核心线程数、最大线程数、队列长度和拒绝策略，因为这四个参数本质上是在定义系统“吞吐、延迟和保护边界”的关系。面试里最好补一句：线程池不是用来无限接流量的，它同时也是后端资源保护器。

**追问点答案**  
- CPU 密集和 IO 密集线程池怎么区分？  
CPU 密集更接近核数，IO 密集可以适当高一些。  
- 队列为什么不能无限大？  
无限队列会把压力延迟暴露，最后变成超时堆积。  
- 拒绝策略怎么选？  
要结合业务是否允许丢弃、降级或回退到调用线程。

**原理补充**  
线程池参数本质上是在控制“并发执行、排队等待和资源保护”三者之间的平衡。

**架构扩展**  
很多系统看起来 CPU 不高但接口越来越慢，最后发现根因不是业务逻辑，而是线程池排队太深。成熟架构师做线程池规划时，不会只看本服务，而是会一起看下游 DB、Redis、HTTP 服务的承载上限，否则线程池扩得越大，下游死得越快。

**工程指标**  
- 活跃线程数：看并发使用情况。  
- 队列长度：判断是否排队过深。  
- 任务等待时间：接口延迟的重要组成。  
- 拒绝次数：说明系统已到保护边界。  
- 下游依赖错误率：看线程池是否把压力传递出去了。

**证据链**  
- 线程池监控  
- 任务执行时长分布  
- 队列堆积趋势  
- 下游依赖容量数据

**低风险优化路径**  
1. 先识别任务类型  
2. 再按目标并发和下游约束定参数  
3. 最后用压测验证拒绝和降级行为

**易错点**  
- 套固定公式不看场景  
- 只调线程数不看队列  
- 不把下游承载纳入模型

## 10. 如何估算消息堆积风险？

**核心结论**  
消息堆积风险本质上取决于生产速度、消费速度、峰值持续时间和恢复能力之间的差值。

**标准答案**  
估算消息堆积风险时，我一般先算生产峰值 TPS 和消费稳态 TPS 的差额，再看这个差额会持续多久，然后换算成 backlog 规模和恢复时间。之所以只看一时的 TPS 没意义，是因为很多系统在高峰期允许短暂积压，真正的问题在于高峰过去后能不能在可接受时间内追平。工程上还要考虑失败重试、消费者 rebalance、消息大小和顺序消费约束，因为这些都会影响真实消费能力。面试里如果能补一句“堆积不是绝对坏事，坏在不可恢复”，会显得很懂系统稳态。

**追问点答案**  
- 消费能力明明够，为什么还会堆积？  
可能因为分区不均、热点、失败重试或顺序消费限制。  
- backlog 多大算危险？  
要看业务 SLA 和可恢复时间。  
- 堆积恢复时间怎么估？  
通常用 backlog / (恢复期消费能力 - 生产能力) 来估。

**原理补充**  
消息堆积本质上是系统供需不平衡在时间维度上的积累。

**架构扩展**  
很多业务在活动高峰时允许消息堆一些，比如异步通知、行为埋点、积分发放，但订单状态同步、支付回调这类链路对恢复时长就非常敏感。成熟团队不会只盯“有没有堆积”，而是会先定义每条链路允许的最大延迟窗口。

**工程指标**  
- backlog 条数：最直观指标。  
- 消费延迟：比条数更贴近业务感知。  
- 生产/消费 TPS 差额：解释堆积速度。  
- 恢复时间估算：判断是否会拖垮 SLA。  
- 失败重试率：识别隐性放大来源。

**证据链**  
- topic backlog  
- 分区消费速率  
- 消费失败日志  
- 高峰期曲线对比

**低风险优化路径**  
1. 先算峰值差额  
2. 再算可接受恢复时间  
3. 最后补扩容、降级和消息分类处理

**易错点**  
- 只看条数，不看恢复时间  
- 不区分不同业务链路容忍度  
- 忽略失败重试放大

## 11. 如何设计弹性扩缩容阈值？

**核心结论**  
扩缩容阈值不能只盯 CPU，要结合业务吞吐、延迟、水位和冷启动成本一起设计。

**标准答案**  
设计扩缩容阈值时，我一般会同时看资源指标和业务指标，比如 CPU、内存、水位、队列长度、QPS、TP99、错误率，再根据扩容是否有冷启动成本和下游是否能承接决定阈值。之所以不能只用 CPU，是因为很多系统在 CPU 还不高时，已经因为连接池、线程池、数据库或缓存回源出现明显延迟。工程上扩容通常需要设上升和回落两个阈值，避免频繁抖动，同时要考虑实例预热和缓存预热时间。面试里如果补一句“扩容不是把机器加上就行，还要看新实例多久能真正接流量”，会比较加分。

**追问点答案**  
- 为什么要有滞后区间？  
为了防止反复扩缩容抖动。  
- 下游没扩，前端扩容有用吗？  
可能没用，甚至会把压力更快传到下游。  
- 什么叫冷启动成本？  
比如 JVM 预热、缓存加载、连接建立、模型加载等。

**原理补充**  
扩缩容本质上是在动态供需平衡中引入滞后控制，避免系统振荡。

**架构扩展**  
很多团队第一次做弹性扩容，只盯 CPU 70% 就加机器，结果新实例没预热好，老实例已经超时堆积，扩容效果非常差。成熟架构师会把扩容阈值设计成“能提前一点感知风险，但又不至于太敏感”的组合信号。

**工程指标**  
- CPU/内存水位：基础资源指标。  
- TP99：业务感知关键。  
- 队列长度：异步链路重要信号。  
- 扩容生效耗时：判断预热影响。  
- 实例抖动次数：衡量阈值是否稳定。

**证据链**  
- 资源监控面板  
- 扩缩容历史事件  
- 新实例预热时长  
- 高峰时段延迟曲线

**低风险优化路径**  
1. 先定义触发指标组合  
2. 再设置滞后和预热机制  
3. 最后通过压测和演练校准阈值

**易错点**  
- 只用 CPU 触发  
- 不考虑预热成本  
- 缺少防抖机制

## 12. 带宽和 IO 瓶颈怎么提前识别？

**核心结论**  
带宽和 IO 瓶颈不能等线上打满才发现，要通过链路画像、压测和资源水位趋势提前暴露。

**标准答案**  
提前识别带宽和 IO 瓶颈时，我通常会先画出系统的关键数据流，比如大对象传输、日志写盘、图片视频分发、数据库刷盘、消息落盘，再结合压测和历史水位看哪一层最接近上限。之所以这类瓶颈容易被忽略，是因为平时业务逻辑没问题，但一到大对象、高并发或批量任务时，网络和磁盘会先成为物理天花板。工程上要重点看网卡带宽利用率、磁盘队列长度、IO wait、吞吐抖动和落盘耗时，而不是只看 CPU。面试里如果能补一句“系统性能瓶颈很多时候不是代码，而是资源通道”，会很专业。

**追问点答案**  
- 为什么 CPU 不高系统也会慢？  
因为网络、磁盘和锁等待都可能让 CPU 闲着但请求仍然超时。  
- 带宽瓶颈常见在哪些场景？  
大文件传输、日志采集、图片视频分发、跨机房同步。  
- IO 瓶颈如何体现？  
常见是 IO wait 高、落盘慢、吞吐下降、队列堆积。

**原理补充**  
带宽和 IO 瓶颈本质上是物理资源通道容量不足导致的吞吐受限。

**架构扩展**  
做架构设计时，如果业务涉及大量静态资源、对象存储、报表导出、埋点落盘或跨机房复制，网络和磁盘几乎一定会进入容量模型。成熟团队会把它们当成一等公民，而不是等线上慢了才去看系统命令。

**工程指标**  
- 网卡带宽利用率：判断是否接近瓶颈。  
- 磁盘 IO wait：定位存储拥塞。  
- 磁盘队列长度：看请求是否排队。  
- 落盘耗时：直接影响写链路。  
- 吞吐抖动幅度：识别资源逼近上限。

**证据链**  
- 系统资源监控  
- `iostat` / `sar` / `iftop` 数据  
- 压测报告  
- 存储和网络拓扑

**低风险优化路径**  
1. 先找高吞吐链路  
2. 再做资源水位压测  
3. 最后按瓶颈点做分流、压缩或升级

**易错点**  
- 只看 CPU  
- 不把物理资源纳入容量模型  
- 没有在压测里覆盖大对象和高写入场景

## 13. 如何做成本-性能平衡？

**核心结论**  
成本-性能平衡不是单纯省钱，而是在满足 SLA 的前提下，把高成本能力只用在真正必要的地方。

**标准答案**  
做成本和性能平衡时，我通常会先明确哪些指标是不能妥协的，比如核心交易接口的 TP99、可用性、数据一致性；然后再看哪些部分可以通过降级、分层存储、冷热分流、异步化和资源复用来降低成本。之所以要这样拆，是因为不是所有请求都值得最贵的处理路径。工程上很常见的做法是把热点数据放快存储，冷数据归档；把高价值链路保留强保障，把边缘链路做降级或异步处理。面试里如果能补一句“成本治理的前提是先给业务分级”，会显得不是纯技术视角。

**追问点答案**  
- 成本优化为什么不能简单减机器？  
因为可能直接伤害 SLA 和稳定性。  
- 哪些地方最适合做分层？  
缓存、存储、计算优先级、流量等级都适合。  
- 如何向业务解释成本方案？  
用收益、风险和 SLA 影响来讲，而不是只讲机器数。

**原理补充**  
成本优化本质上是在不同服务等级之间做资源匹配，而不是追求所有路径同等豪华。

**架构扩展**  
真实系统里，最常见的浪费不是某台机器贵，而是把所有流量都走最高配置路径。成熟架构师更关注“哪些流量值得贵处理，哪些流量可以便宜处理”，比如热点数据保 SSD，冷数据进对象存储，非实时任务放异步批处理。

**工程指标**  
- 单请求成本：衡量优化结果。  
- 资源利用率：看是否过度冗余。  
- SLA 达成率：成本优化不能破坏底线。  
- 冷热数据占比：帮助做分层。  
- 降级命中率：评估低成本路径是否有效。

**证据链**  
- 资源账单  
- 访问冷热分布  
- SLA 监控  
- 容量利用率数据

**低风险优化路径**  
1. 先划分业务等级  
2. 再做冷热和实时性分层  
3. 最后按收益逐步下探成本

**易错点**  
- 把省钱等同于减配置  
- 不看业务等级  
- 没有量化成本收益

## 14. 什么情况下要做分库分表？

**核心结论**  
分库分表应该是单库优化接近天花板之后的架构动作，而不是早期的习惯性预埋。

**标准答案**  
我通常会在单库单表已经接近容量或性能瓶颈，并且通过索引优化、SQL 优化、读写分离、冷热拆分后仍然无法满足 SLA 时，才考虑分库分表。之所以要谨慎，是因为它虽然能扩展容量和吞吐，但会显著增加事务、查询、运维和数据迁移复杂度。工程上判断触发点时，除了看数据量，还要看单表写入 TPS、索引维护成本、热点分布、扩容空间和业务增长速度。面试里最好强调：分库分表不是“数据多就拆”，而是“单机治理空间快耗尽，且增长趋势会继续放大”时才值得做。

**追问点答案**  
- 单表千万级一定要拆吗？  
不一定，关键还是看索引、SQL 模式和写入压力。  
- 为什么很多团队拆完更痛苦？  
因为没有评估跨分片查询、事务和运维成本。  
- 先分库还是先分表？  
要看瓶颈是容量、并发还是热点，不能机械套用。

**原理补充**  
分库分表本质上是把单节点瓶颈转化成分布式复杂度，以换取更大的扩展空间。

**架构扩展**  
在订单、流水、行为日志这类增长快的表上，分库分表很常见；但很多业务其实先做归档、冷热分层和索引优化就够了。成熟架构师不会把“拆分”当成默认正确答案，而是会先算清楚继续留在单库还能走多远。

**工程指标**  
- 单表数据量：基础判断指标。  
- 单表写 TPS：看并发压力。  
- 慢 SQL 占比：评估单机治理空间。  
- 主从延迟：说明扩展是否已吃紧。  
- 热点分片集中度：拆分前后都需要关注。

**证据链**  
- 表增长曲线  
- SQL 模式分析  
- 写入分布  
- 单机资源水位

**低风险优化路径**  
1. 先把单库优化做到位  
2. 再评估增长趋势和治理边界  
3. 最后有计划地演进分片方案

**易错点**  
- 数据量大就直接拆  
- 不评估分布式复杂度  
- 不看增长趋势和热点结构

## 15. 如何设计压测方案验证容量？

**核心结论**  
压测不是单纯把流量打上去，而是要让流量结构、数据模型和依赖行为尽量接近真实生产。

**标准答案**  
设计容量压测时，我一般会先选定目标链路和验证目标，比如验证峰值 QPS、TP99、安全水位还是降级效果；然后再准备接近生产的数据分布、热点比例、读写比、异常请求比例和依赖仿真。之所以不能只做“空接口压测”，是因为真正打垮系统的常常是数据库、缓存、线程池和外部依赖。工程上压测最好分层做，先单链路基准，再全链路压测，最后做故障场景和降级演练。面试里如果能补一句“压测结果必须能映射回容量模型，否则只是热闹”，会很亮眼。

**追问点答案**  
- 为什么要用真实数据分布？  
因为热点、长尾和缓存命中率都会显著影响结果。  
- 压测要不要带下游？  
要，至少核心依赖要在模型里体现。  
- 只看平均耗时可以吗？  
不行，容量判断更要看 TP95/TP99 和错误率。

**原理补充**  
压测的本质，是用受控实验验证容量模型和故障边界，而不是简单制造大流量。

**架构扩展**  
成熟团队的压测，不只是上线前打一轮流量，而是和大促预案、扩容策略、降级方案一起联动。真正值钱的不是“系统扛住了”，而是知道“扛到哪开始抖、抖了怎么止血、恢复要多久”。

**工程指标**  
- QPS/TPS：基础吞吐指标。  
- TP95/TP99：关键容量边界。  
- 错误率：稳定性底线。  
- 资源水位：定位瓶颈所在。  
- 降级触发率：看保护策略是否起作用。

**证据链**  
- 压测脚本和场景定义  
- 数据样本分布  
- 压测曲线  
- 依赖监控数据

**低风险优化路径**  
1. 先定义验证目标  
2. 再设计接近真实的流量模型  
3. 最后补故障和降级场景压测

**易错点**  
- 压测场景太理想化  
- 只看吞吐不看尾延迟  
- 结果不能反推容量阈值

## 16. 如何定义系统的安全水位？

**核心结论**  
安全水位是系统在正常业务波动和小故障下仍能稳定运行的保守容量边界，不等于理论极限。

**标准答案**  
定义安全水位时，我通常不会用“系统最大能跑到多少”作为标准，而是看在目标 SLA 下，资源、延迟、错误率和依赖稳定性都还比较健康时的负载水平。之所以安全水位要保守，是因为线上一定会有突发流量、抖动、GC、网络波动和局部失败，真正可上线使用的容量必须给这些不确定性留空间。工程上安全水位通常会体现在 CPU、连接池、线程池、数据库延迟、缓存命中率和错误率的组合阈值上。面试里如果能说“极限容量是实验值，安全水位才是可运营值”，会很有架构感。

**追问点答案**  
- 为什么不能把系统跑到极限再上线？  
因为线上环境永远存在抖动和不确定性。  
- 安全水位一般留多少余量？  
取决于业务波动和扩容速度，常见会留 20%-50% 冗余。  
- 安全水位要按单指标定吗？  
最好按组合指标定，更贴近真实运行状态。

**原理补充**  
安全水位的本质，是把“系统极限”转换成“可长期运营的稳定边界”。

**架构扩展**  
大促系统、支付系统、核心交易链路往往不会追求资源压到最满，而是宁愿有点浪费也要稳。成熟架构师做容量规划时，都会把安全水位单独定义出来，因为这直接决定扩容节奏和预警时机。

**工程指标**  
- CPU/内存安全线：基础资源边界。  
- TP99 稳定区间：延迟安全线。  
- 错误率阈值：核心稳定性信号。  
- 下游饱和度：防止局部先爆。  
- 扩容响应时间：决定冗余要留多少。

**证据链**  
- 压测极限数据  
- 日常运行曲线  
- 历史故障案例  
- 扩容和恢复耗时记录

**低风险优化路径**  
1. 先测出极限  
2. 再结合波动和恢复能力折算安全线  
3. 最后把安全水位接入预警体系

**易错点**  
- 把极限值当上线容量  
- 不给波动留余量  
- 只看单一资源指标

## 17. 大促前容量演练怎么做？

**核心结论**  
大促前演练要从“证明系统能跑”升级成“证明系统在异常态也能稳住并恢复”。

**标准答案**  
大促前做容量演练时，我通常会分三层来做：先验证核心链路的目标峰值能力，再验证扩容、限流、降级和缓存预热是否生效，最后验证关键依赖异常时系统是否还能止血。之所以大促演练不能只做正常流量压测，是因为真正的风险往往来自缓存未热、热点集中、依赖超时、消息堆积和人工操作失误。工程上我会要求把入口、服务、缓存、数据库、消息和监控告警一起拉通，演练结果不仅要有吞吐结论，还要有应急动作清单。面试里如果能补一句“演练不是为了报告好看，而是为了提前暴露失败模式”，会很有说服力。

**追问点答案**  
- 大促演练为什么要做故障注入？  
因为正常压测很难暴露异常态联动问题。  
- 缓存预热重要吗？  
非常重要，很多大促事故都发生在热点数据未预热阶段。  
- 演练结束后最重要的产物是什么？  
不是峰值数字，而是风险清单和处置预案。

**原理补充**  
大促演练的本质，是把未来高风险场景提前转化成可控实验和可执行预案。

**架构扩展**  
真正成熟的大促演练会覆盖热点商品、异步链路、库存更新、回调通知、监控告警、熔断降级和发布冻结。能扛住流量只是第一步，知道哪一环最先出问题、出了问题谁处理、多久恢复，才是架构团队真正的价值。

**工程指标**  
- 核心链路峰值 QPS：基础目标。  
- 预热完成率：看准备是否到位。  
- 降级触发耗时：验证保护机制。  
- 故障恢复时间：应急能力关键指标。  
- 告警命中率：看监控是否有效。

**证据链**  
- 演练方案  
- 压测报告  
- 故障注入记录  
- 应急复盘文档

**低风险优化路径**  
1. 先压正常态  
2. 再演故障态  
3. 最后固化处置和回滚流程

**易错点**  
- 只演正常流量  
- 不拉通全链路  
- 演练后没有复盘落地

## 18. 如何做容量预警机制？

**核心结论**  
容量预警不是简单设阈值，而是要把“接近危险”提前转化成可执行动作。

**标准答案**  
设计容量预警时，我一般会先确定哪些指标最能反映系统逼近边界，比如 CPU、内存、连接池、线程池、队列长度、主从延迟、缓存回源比例、TP99 和错误率，然后再把它们分成关注、预警、紧急三个级别。之所以要分级，是因为不同阶段对应的动作不一样，早期可能是观察和预热扩容，到了紧急级才触发限流和人工介入。工程上预警一定要结合趋势，而不是只看瞬时值，否则很容易噪音太大。面试里如果你能补一句“好的预警要能驱动动作，而不是只会报红”，会很加分。

**追问点答案**  
- 为什么瞬时告警不够？  
因为短暂抖动不一定危险，趋势持续上升才更值得处理。  
- 预警和告警有什么区别？  
预警更偏提前感知，告警更偏明确异常。  
- 预警后要有什么动作？  
观察、扩容、预热、限流、人工确认、发布冻结等。

**原理补充**  
容量预警本质上是在系统真正崩之前，建立一套从信号到动作的决策链。

**架构扩展**  
很多团队有一堆监控图，但真正高峰来时还是手忙脚乱，原因往往不是没指标，而是没有预定义动作。成熟架构师会把容量预警和运行手册绑定，做到看到哪个指标升到哪个区间，就知道下一步该做什么。

**工程指标**  
- 预警提前量：看是否来得及处理。  
- 误报率：过高会让团队麻木。  
- 漏报率：过高说明体系失效。  
- 自动处置触发率：衡量可运营能力。  
- 响应耗时：从预警到动作的时间。

**证据链**  
- 告警规则  
- 趋势图  
- 运行手册  
- 历史预警处置记录

**低风险优化路径**  
1. 先确定关键容量指标  
2. 再分级绑定动作  
3. 最后用真实高峰和演练不断校准

**易错点**  
- 只设阈值不设动作  
- 告警过多没有分级  
- 不复盘误报和漏报

## 19. 容量模型偏差怎么修正？

**核心结论**  
容量模型一定会有偏差，关键不在于一次算准，而在于建立“预测-验证-修正”的闭环。

**标准答案**  
当容量模型和实际运行结果出现偏差时，我不会急着重做一版大而全模型，而是先定位偏差来自输入假设、流量分布、链路放大、资源瓶颈还是压测场景本身。之所以偏差不可避免，是因为业务会变、热点会变、依赖会变，模型本来就是一种近似。工程上比较有效的做法是把关键参数拆出来，比如人均请求数、峰值系数、回源比例、重试率、命中率、平均耗时，然后通过真实监控和压测结果逐项校准。面试里如果能补一句“容量模型不是文档，而是持续迭代的运行资产”，会很亮眼。

**追问点答案**  
- 偏差最大通常来自哪里？  
常见是峰值分布判断错误和异常态流量低估。  
- 修正是重算还是微调？  
先拆参数，能微调就不要整体推翻。  
- 多久校准一次？  
业务变动大时按周或按活动，稳定系统也要定期复盘。

**原理补充**  
容量模型本质上是一个动态近似系统，价值来自持续修正，而不是首次完美。

**架构扩展**  
很多团队做完一次容量报告就丢在那，半年后业务翻倍、缓存策略变了、实例结构也变了，结果模型完全失真。成熟团队会把容量参数和监控平台打通，让模型和真实运行数据形成闭环更新。

**工程指标**  
- 预测误差率：核心校准指标。  
- 峰值系数偏差：高峰场景重点关注。  
- 命中率偏差：缓存类系统影响很大。  
- 重试率偏差：故障态模型常由此失真。  
- 校准周期：反映治理成熟度。

**证据链**  
- 容量模型参数表  
- 实际运行曲线  
- 压测与线上对比  
- 业务变更记录

**低风险优化路径**  
1. 先找偏差最大的参数  
2. 再用真实数据逐项回填  
3. 最后把校准过程标准化

**易错点**  
- 模型偏差就全部推翻  
- 不知道误差来自哪里  
- 模型长期不更新

## 20. 如何向业务方解释容量预算？

**核心结论**  
向业务解释容量预算时，不能只讲技术资源，而要把预算换成风险、收益和可保障的业务目标。

**标准答案**  
我跟业务方解释容量预算时，通常不会直接说“需要多少台机器、多少核内存”，而是先说明预算对应的业务目标，比如可支撑的峰值流量、目标可用性、下单成功率、活动承接量和故障余量。之所以这样沟通更有效，是因为业务真正关心的是“花这笔钱能换来什么保障，不花会有什么风险”。工程上我会把预算拆成基础保障、活动增量、容灾冗余和风险兜底几部分，让业务能看懂哪些是必须花、哪些是扩展项。面试里如果能补一句“容量预算本质上是在买确定性”，会显得很有管理视角。

**追问点答案**  
- 业务觉得贵怎么办？  
就把不投入的风险量化出来，比如峰值丢单率、恢复时间和收入损失。  
- 如何区分必须预算和可选预算？  
按 SLA 底线、活动目标和容灾要求分层。  
- 为什么架构师也要会讲预算？  
因为容量决策最终是技术和业务一起承担结果。

**原理补充**  
容量预算的本质，是把技术冗余和资源成本转换成业务可理解的风险对价。

**架构扩展**  
成熟架构师不仅会做容量模型，还会把模型讲给业务听。真正推动资源投入的，不是技术术语，而是“这笔预算能换来什么成功率、减少多少事故概率、支撑多少增长”。这也是架构角色和纯开发角色很大的差别。

**工程指标**  
- 单位流量成本：预算沟通的重要基线。  
- SLA 达成率：预算投入的业务回报。  
- 峰值承接能力：资源投入的直接收益。  
- 风险暴露概率：帮助解释不投的代价。  
- 容灾冗余水平：体现稳态保障价值。

**证据链**  
- 容量模型  
- 历史高峰数据  
- 预算拆分表  
- 风险评估结论

**低风险优化路径**  
1. 先讲业务目标  
2. 再讲预算对应的保障能力  
3. 最后讲不投入的风险和备选方案

**易错点**  
- 只讲技术资源，不讲业务收益  
- 不区分必须项和扩展项  
- 沟通里没有风险量化
