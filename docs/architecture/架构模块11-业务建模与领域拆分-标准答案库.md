# 架构模块11 - 业务建模与领域拆分 - 标准答案库

## 1. 你如何做一个新业务的领域划分？

::: tip L2 深度训练
从业务事件、命令、规则、统一语言和变化原因发现边界：[领域发现与限界上下文](/deep-dives/business-evolution/01-domain-discovery)
:::

**核心结论**  
领域划分的关键，不是按数据库表或团队现状硬切，而是围绕核心业务流程、规则变化和职责边界，把高内聚的业务能力收敛到一起。

**标准答案**  
我做新业务的领域划分时，通常不会一开始就谈服务拆分，而是先梳理业务目标、核心流程和关键规则。因为很多系统一上来就按页面、表结构或者现有组织去切，最后边界看起来清楚，实际职责却高度耦合。比较稳的做法，是先识别主流程里哪些能力是核心竞争力，哪些是支撑性能力，再看数据变化是否围绕同一组业务规则演进。工程上我会把订单、支付、库存、营销这类具有独立规则和生命周期的能力分开建模，同时保留后续演进空间，而不是第一版就拆得很碎。面试里如果能补一句“领域划分是为了隔离变化，不是为了拆得更细”，会更成熟。

**高分回答**
我先明确业务目标、关键结果和约束，再与领域专家用事件风暴梳理“发生了什么—由什么命令触发—谁决策—哪些规则必须成立”，同时建立统一语言。边界候选来自语义、业务不变量、生命周期、变化原因和 owner，而不是页面或表：订单承诺交易意图，支付记录资金事实，库存保护可售不超卖，它们的规则、失败模式和节奏不同。随后画上下文关系，明确 upstream/downstream、同步命令、领域事件和反腐层，并用真实需求回放：一个营销变更是否只修改一个边界，跨域事务是否被迫共享数据库。第一版优先模块化单体，将强一致规则放同一边界，可异步信息用事件传播；只有独立发布、扩展、故障隔离或团队自治收益超过分布式成本时才拆服务。用单需求跨边界改动、共享数据写入、跨团队等待、契约变更与事故归属验证边界，定期根据新业务证据调整，而不追求一次完美。

**追问点答案**  
- 为什么不能按数据库表来划分领域？  
因为表结构反映的是存储，不一定反映真实业务边界。  
- 为什么不能直接按团队组织划分？  
组织会变，但业务边界应该更稳定。  
- 第一版要不要追求完美划分？  
不用。先通过模块、接口和数据 owner 保留边界，记录不确定假设；用后续需求和事故验证，再在有独立演进收益时拆服务，避免过早固化错误认知。

**原理补充**  
领域划分的本质，是把经常一起变化、一起决策的一组业务规则放进同一个边界里。

**架构扩展**  
真实项目里，最常见的问题是“看起来拆了很多服务，实际每个需求都要跨好几个团队改”。这往往不是技术问题，而是领域边界没划对。成熟架构师会先看需求变更链路，判断一个功能改动到底需要动几个边界，再反过来检验领域划分是否健康。

**工程指标**  
- 单需求跨服务改动数：看边界是否清晰。  
- 领域内接口调用比例：看内聚度。  
- 跨域变更评审次数：看协作成本。  
- 领域模型变更频率：看边界稳定性。  
- 需求交付时长：看划分是否拖累效率。

**证据链**  
- 业务流程图  
- 领域事件梳理结果  
- 需求变更链路  
- 服务调用关系图

**低风险优化路径**  
1. 先梳理核心业务流程  
2. 再按规则和生命周期划分能力  
3. 最后再决定服务和团队边界

**易错点**  
- 按表结构切领域  
- 把领域划分等同于微服务拆分  
- 第一版拆得过细

## 2. 什么情况下应该拆分限界上下文？

::: tip L2 深度训练
联合语言、规则、变化、所有权和规模信号判断：[领域发现与限界上下文](/deep-dives/business-evolution/01-domain-discovery)
:::

**核心结论**  
限界上下文应该在语言、规则、职责和演进节奏明显分化时拆开，而不是因为模块大了就机械切分。

**标准答案**  
我判断要不要拆限界上下文时，通常先看几个信号：同一个名词在不同流程里含义是否已经不同、规则是否冲突、团队协作是否频繁卡边界、一个改动是否总是波及另一块看似无关的能力。如果这些问题越来越明显，说明一个上下文里混进了多套业务语义。之所以不能只看代码量，是因为代码多并不一定代表边界错，真正危险的是概念混乱。工程上如果订单里的“状态”、营销里的“活动状态”、履约里的“处理状态”已经彼此独立，就应该考虑拆开，让每个上下文用自己的语言和模型演进。成熟拆分不是为了层次好看，而是为了降低语义污染和变更牵连。

**高分回答**
我把拆分看成语义与变化隔离决策。强信号包括：同一词在不同流程含义冲突；两组不变量需要不同事务和生命周期；需求总由不同 owner 以不同节奏修改；模型为了兼容另一场景堆满条件；一方故障或容量需要独立隔离。代码行数大不是证据，单一复杂核心模型可能仍应保持一个上下文。先列出拆分后各自拥有的数据、决策和语言，再定义上下文映射：上游发布稳定契约，下游用 ACL 翻译，禁止两边继续共写表。迁移可先在单体内分模块和仓储，影子读取/双写需有单一事实源、对账和退出计划，随后才独立部署。拆分代价包括跨域一致性、网络失败、契约治理与团队值班；若改动仍总需同步发布，说明边界或迁移未完成。通过变更耦合、独立发布、跨域调用、数据写 owner、认知负担和事故传播前后对照决定是否值得。

**追问点答案**  
- 为什么同一个词语含义变化是危险信号？  
因为说明不同业务语义被混在一起了。  
- 限界上下文和微服务是一一对应吗？  
不一定。上下文是模型语义边界，可先作为单体模块；一个上下文也可能因容量拆多个技术服务，但它们仍共享同一语言和 owner，不能把部署单元反推成业务边界。
- 拆分之后为什么协作可能反而更顺？  
因为双方边界和契约更清楚了。

**原理补充**  
限界上下文的核心价值，是保护模型语义不被别的领域概念污染。

**架构扩展**  
很多遗留系统的问题不是“代码老”，而是订单、结算、营销、客服全都共享一套混乱模型，任何新需求都像在泥潭里走。限界上下文拆分能显著降低这种认知负担，但前提是先认清真正冲突的是语义，不只是模块数量。

**工程指标**  
- 同名概念定义冲突数：看语义污染程度。  
- 跨团队需求拉齐耗时：看边界摩擦。  
- 单需求跨模块改动比例：看耦合度。  
- 模型说明文档更新频率：看语义变化速度。  
- 生产问题中的边界归因争议数：看职责清晰度。

**证据链**  
- 领域术语表  
- 需求评审记录  
- 跨模块改动统计  
- 事故复盘中职责争议记录

**低风险优化路径**  
1. 先识别语义冲突点  
2. 再梳理各自规则和责任  
3. 最后逐步收敛数据和接口边界

**易错点**  
- 只按代码量决定拆分  
- 把上下文拆分等同于立即分库分服务  
- 忽略语义冲突的根因

## 3. 聚合根怎么确定？

::: tip L2 深度训练
从强一致不变量反推最小事务边界：[聚合、不变量与领域事件](/deep-dives/business-evolution/02-aggregate-invariants)
:::

**核心结论**  
聚合根应该围绕一致性边界来确定，谁负责维护核心业务不变量，谁通常就是聚合根。

**标准答案**  
我确定聚合根时，通常先看一个业务对象周围有哪些数据必须在同一事务里保持一致。如果一组对象的约束必须一起成立，比如订单金额、订单项、状态流转这些核心规则，那就需要一个统一入口来维护它们，这个入口往往就是聚合根。之所以不能把所有相关对象都塞进一个大聚合，是因为会让事务边界太重，写入冲突变高，后续扩展也很难。工程上比较稳的做法，是只把真正强一致的规则放进聚合内部，把可异步传播的内容通过事件或查询模型解耦出去。面试里如果能把“聚合根是为了保护业务不变量”说清楚，会比背定义更有说服力。

**高分回答**
我从命令发生瞬间必须成立的不变量反推最小事务边界。例如订单确认时“订单项总额等于应付金额、已取消订单不能再确认”必须原子成立，Order 作为唯一修改入口；商品描述快照、支付结果和履约进度不需与订单同一事务，就只保存引用或快照并通过事件更新。外部不能直接持有并修改内部实体，仓储按聚合加载和保存，命令携带版本做乐观并发，防两个合法操作覆盖彼此。大集合若全装进聚合会造成长事务和热点，应重新判断不变量是否能用配额、预留或独立聚合表达；跨聚合规则使用 saga、幂等事件和最终对账，并明确暂态。聚合根不是 ORM 主表，也不是所有查询入口，读模型可绕过聚合优化展示，但写入必须守规则。用事务时长、版本冲突、聚合加载大小、规则绕过、补偿率和并发测试验证；若跨聚合补偿频繁且业务不能接受暂态，可能边界切得过小。

**追问点答案**  
- 为什么聚合不能太大？  
因为一致性边界越大，性能和并发成本越高。  
- 聚合根能直接暴露内部对象吗？  
可以返回只读快照或值，但不应让外部持有可变引用并绕过根修改；所有状态变化通过表达业务意图的方法执行并校验不变量。
- 为什么聚合根不等于数据库主表？  
因为它表达的是业务一致性边界，不是存储结构。

**原理补充**  
聚合根的本质，是一致性边界的唯一入口，用来保护一组业务规则不被外部绕开。

**架构扩展**  
真实线上里，很多“并发下状态错乱”的问题，本质上就是聚合边界没设计好。比如库存和预占逻辑散落在多个服务里，谁都能改状态，最后一致性全靠补救。成熟设计会先问：哪些规则必须同一时刻成立，再决定聚合边界。

**工程指标**  
- 聚合内并发冲突率：看边界是否过大。  
- 事务平均耗时：看一致性成本。  
- 规则绕过导致的问题数：看入口是否统一。  
- 聚合变更影响范围：看边界稳定性。  
- 异步补偿触发比例：看边界拆分是否合理。

**证据链**  
- 核心业务规则清单  
- 事务边界设计  
- 并发冲突监控  
- 问题复盘记录

**低风险优化路径**  
1. 先列出必须强一致的规则  
2. 再收敛统一修改入口  
3. 最后把非强一致内容外移

**易错点**  
- 把聚合做成大而全对象图  
- 只按数据库关系定聚合  
- 允许外部直接修改内部状态

## 4. 实体和值对象如何区分？

**核心结论**  
实体关注身份和值随时间变化的连续性，值对象关注属性组合本身和不可变语义。

**标准答案**  
我区分实体和值对象时，核心会看它是不是需要被持续追踪身份。像订单、用户、商品这种对象，即使属性变了，我们依然关心“还是不是同一个对象”，这类通常是实体；像地址、金额、时间区间、坐标这类更关注属性组合含义，本身不强调独立身份，更适合建成值对象。之所以这个区分重要，是因为值对象天然更适合不可变设计和组合复用，而实体则更关注生命周期和状态流转。工程上如果一个对象没有独立身份，却被建成实体，通常会引入很多没必要的复杂度。成熟的建模不是名词好听，而是让代码和业务语义更贴近。

**高分回答**
判断标准是业务是否关心跨时间的同一性。订单即使金额或状态变化仍是同一订单，需要稳定身份、生命周期和并发版本，是实体；Money 由币种与金额共同定义，两个值相同即可替换，适合不可变值对象，并在构造时保证精度、币种和非负等约束。地址不天然是哪一种：结算时保存的收货地址快照是值对象；地址簿条目若可独立编辑、引用和审计，则可能是实体。值对象按结构相等、无 setter，修改即创建新值，可安全共享并让非法状态无法构造；实体相等通常基于领域 ID，不能因可变属性变化而改变集合键。持久化可将值对象嵌入同表或序列化，但技术映射不应反向决定模型。过度实体化会产生无意义 ID、CRUD 和 join，过度值化又会丢失审计与生命周期。用业务语言、是否独立引用/修改、历史追踪和并发冲突来验证。

**追问点答案**  
- 值对象为什么常常设计成不可变？  
因为它表达的是一个值语义，改值通常等于替换。  
- 地址一定是值对象吗？  
不一定。订单中的地址快照通常是值对象；用户地址簿条目若有独立 ID、修改历史、默认状态和外部引用，则是实体。判断取决于上下文而非名词。
- 为什么滥用实体会让系统变复杂？  
因为会引入多余的身份、持久化和关系管理。

**原理补充**  
实体和值对象的区分，本质上是在区分“身份语义”和“值语义”。

**架构扩展**  
很多项目里，所有对象都做成“表 + id + CRUD”，结果模型很重却没有表达真实业务。把值对象建模好，往往能显著提升可读性，也能减少很多无意义的状态管理和持久化复杂度。

**工程指标**  
- 纯属性对象被建成实体的比例：看建模冗余。  
- 值对象复用率：看模型抽象质量。  
- 与身份无关的更新操作比例：看实体滥用程度。  
- 模型变更影响范围：看语义贴合度。  
- 领域对象平均复杂度：看设计负担。

**证据链**  
- 领域模型图  
- 核心对象生命周期说明  
- 代码中的对象职责分布  
- 变更案例

**低风险优化路径**  
1. 先识别不需要独立身份的对象  
2. 再把它们收敛为值对象  
3. 最后逐步清理多余关系和状态

**易错点**  
- 所有对象都建成实体  
- 值对象可变且被四处修改  
- 不结合业务生命周期判断

## 5. 领域服务和应用服务如何分工？

::: tip L2 深度训练
用命令编排、领域规则、聚合修改和外部副作用划分职责：[聚合、不变量与领域事件](/deep-dives/business-evolution/02-aggregate-invariants)
:::

**核心结论**  
应用服务负责编排流程，领域服务负责承载跨实体但仍属于业务规则本身的逻辑。

**标准答案**  
我区分应用服务和领域服务时，通常会先问这段逻辑到底是在“做业务决策”，还是在“串业务流程”。如果它更像调用仓储、校验权限、发消息、开事务、组合多个领域对象完成一个用例，那通常属于应用服务；如果它本身就是核心业务规则，而且又不自然地归属于某一个实体，那更适合放进领域服务。之所以很多系统会把两者混在一起，是因为流程编排代码里往往也掺着业务判断，久了应用层就会越来越胖。工程上比较稳的做法，是让应用服务保持薄，让真正的业务规则尽量回到领域模型内部或领域服务里。这样后续变化时，模型会更稳定，流程也更清晰。

**高分回答**
应用服务表达一个用例的流程边界：验证调用者、加载聚合、开启事务、调用领域行为、保存、写 outbox/发布结果，并将领域错误映射为接口响应；它不应决定折扣怎么算或订单能否取消。业务规则优先放拥有数据和不变量的实体/值对象，只有规则跨多个对象且不自然属于任何一个实体时才用无状态领域服务，例如由合同与客户等级计算授信。调用外部支付、邮件等副作用通过端口由应用层编排，领域层只表达决策或事件，避免基础设施依赖污染模型。一个“领域服务”若只是仓储和 RPC 的脚本包装，说明命名并未解决贫血模型；反之把跨聚合网络流程塞进实体会让测试和事务边界失真。用纯领域单测验证规则，用应用级测试验证事务、幂等和副作用。评审关注同一规则重复、应用层条件分支、领域对象 setter 和跨聚合事务，而不是机械限制方法行数。

**追问点答案**  
- 应用服务里能不能写业务规则？  
权限、幂等和流程前置属于应用编排；决定业务结果且需跨入口一致的规则应进入领域模型。临时放置也要有 owner 和收敛计划，避免同一规则在 API、任务和消费者中复制。
- 领域服务是不是越多越好？  
不是，过多说明实体边界可能没建好。  
- 为什么应用服务容易变胖？  
因为它最容易承接临时需求和流程拼接。

**原理补充**  
两者分工的核心，是把“业务规则”与“用例流程”分离，避免流程层吞掉领域语义。

**架构扩展**  
真实项目里，很多所谓“DDD 失败”，其实不是 DDD 本身难，而是所有东西都塞进 Service，最后连团队自己都分不清哪些逻辑是业务规则、哪些只是过程编排。成熟架构师会特别关注应用服务是否越来越像脚本层。

**工程指标**  
- 应用服务平均方法长度：看是否过胖。  
- 领域规则复用率：看是否沉淀在领域层。  
- 同一业务规则复制次数：看分工是否失衡。  
- 用例改动对领域对象影响范围：看边界清晰度。  
- 代码评审中“流程与规则混杂”问题数：看设计健康度。

**证据链**  
- Service 层代码结构  
- 领域规则清单  
- 用例流程图  
- 重复逻辑统计

**低风险优化路径**  
1. 先识别重复业务规则  
2. 再把跨实体规则抽回领域层  
3. 最后让应用服务只保留编排职责

**易错点**  
- 把应用服务写成万能业务层  
- 把所有不想放实体里的逻辑都塞进领域服务  
- 不区分规则和流程

## 6. 订单域和支付域边界如何划分？

::: tip L2 深度训练
区分资金事实与订单投影，用事件和对账跨域收敛：[聚合、不变量与领域事件](/deep-dives/business-evolution/02-aggregate-invariants)
:::

**核心结论**  
订单域负责交易意图和履约约束，支付域负责资金动作和支付状态，两者要协作，但不应互相吞掉核心职责。

**标准答案**  
我划分订单域和支付域边界时，通常先看它们分别对什么负责。订单域更关注用户买了什么、价格是多少、是否可以取消、是否进入履约；支付域更关注支付方式、支付渠道、交易流水、扣款结果和退款处理。之所以这两个域经常被混在一起，是因为它们在下单流程里靠得很近，但“下单成功”和“支付成功”其实不是一回事。工程上比较稳的设计，是订单维护自己的状态机，支付维护自己的交易状态，两边通过领域事件或明确契约协作，而不是共用一套状态和表结构。这样做的好处是后续引入组合支付、分账、退款或多渠道切换时，边界不会立刻崩掉。

**高分回答**
我按事实所有权划分：订单域拥有购买意图、商品/价格快照、取消与履约资格；支付域拥有支付意图、渠道尝试、资金流水、退款和对账。订单的“待支付/已确认”是业务投影，支付的“处理中/成功/失败”是资金事实，不能共用一个状态字段或互写数据库。订单创建支付请求时使用稳定 orderId 与 paymentRequestId，支付以幂等键防重复扣款；支付成功通过 outbox 可靠发布事实，订单消费后按自己的状态机确认，若订单已取消则触发退款/人工异常，而不是篡改支付结果。回调可能重复、乱序、伪造和延迟，支付先验签、查询渠道真相并落流水，对账是外部资金的最终兜底。组合支付、部分退款时一单可对应多笔交易，更证明状态不可一一映射。指标包括重复扣款、回调延迟、跨域处理中时长、退款/补偿成功、对账差异和人工单量，并演练“扣款成功但订单事件丢失”。

**追问点答案**  
- 为什么订单状态不能直接等于支付状态？  
因为订单还受库存、履约、取消等因素影响。  
- 支付失败为什么不一定意味着订单失败？  
订单可能仍在支付窗口内，允许换渠道或重试；只有订单自身超时/取消规则决定失败。反之支付成功也不自动保证履约，需要订单按库存和状态机处理。
- 为什么退款更适合在支付域处理？  
因为它本质上是资金逆向流转。

**原理补充**  
领域边界划分的关键，是按核心职责和业务不变量分开，而不是按流程先后顺序绑死。

**架构扩展**  
电商、会员订阅、票务这些系统里，订单和支付边界一旦混乱，后面做退款、补单、支付对账、异常补偿都会特别痛。成熟系统几乎都会把订单与支付解耦，因为两者的外部依赖、失败模式和演进节奏都不同。

**工程指标**  
- 订单与支付跨域同步失败率：看协作质量。  
- 支付回调导致的订单异常数：看边界稳定性。  
- 退款链路平均耗时：看支付域独立性。  
- 跨域状态映射复杂度：看职责是否混乱。  
- 对账异常数：看资金边界清晰度。

**证据链**  
- 订单状态机  
- 支付状态机  
- 回调处理链路  
- 对账和退款流程图

**低风险优化路径**  
1. 先拆清状态定义和职责  
2. 再用事件或契约同步状态  
3. 最后治理异常补偿和对账链路

**易错点**  
- 把订单和支付看成一个域  
- 共用状态字段  
- 退款职责挂在订单域里

## 7. 如何避免“伪微服务”拆分？

::: tip L2 深度训练
先验证业务边界和模块依赖，避免把代码耦合变成网络耦合：[领域发现与限界上下文](/deep-dives/business-evolution/01-domain-discovery)
:::

**核心结论**  
避免伪微服务的关键，不是少拆服务，而是保证每个服务都有清晰业务边界、独立决策权和合理演进空间。

**标准答案**  
我判断是不是“伪微服务”时，通常不会只看服务数量，而是看一个服务有没有自己的业务职责、数据边界和独立演进节奏。如果一个系统拆成了很多服务，但每个需求都要改多个服务、共享数据库、共用发布、共用一堆 DTO，那本质上只是把单体拆碎了。之所以会出现伪微服务，往往是把技术拆分当成目标，却没先把业务边界想清楚。工程上比较稳的做法，是先在单体里把模块边界收敛好，确认职责清晰后，再拆出真正需要独立扩展或独立治理的能力。面试里如果能说“服务数量不是成熟度指标，协作成本和独立性才是”，会很加分。

**高分回答**
我用自治性检查而不是服务数判断：是否有明确业务能力和 owner，是否唯一写自己的数据，能否在兼容契约下独立测试、发布、扩容和故障隔离。若多个服务共享表、一个需求固定改五个仓库、同步调用链像本地函数、版本必须锁步发布，就只是分布式单体，网络失败和运维成本增加却未隔离变化。治理先统计变更耦合、共享写、循环依赖、跨服务事务、独立发布率和事故传播，找出应该合并或重划边界的簇。低风险做法是在代码/模块层先收敛语言和数据 owner，通过 API/事件替换跨库写，契约版本化；只有容量、发布、故障或团队自治有量化收益时再物理拆分。某些过细服务应勇于合并，减少 hop 和 owner 空洞。平台能力是拆分前提但不是理由。成功不是服务变多，而是单需求改动面、跨团队等待和级联故障下降，同时端到端延迟与值班负担可控。

**追问点答案**  
- 为什么共享数据库很危险？  
更准确地说，多服务共写同一数据会绕过 owner、不变量和发布契约，任何 schema 变化都要求锁步协调。共享物理实例可以接受，但逻辑 schema/写权限必须有清晰归属。
- 每个需求都跨多个服务说明什么？  
说明拆分边界和业务变化路径不一致。  
- 单体先模块化为什么重要？  
因为这是验证边界最便宜的方式。

**原理补充**  
微服务的核心不是“分布式”，而是围绕业务边界实现独立演进和独立治理。

**架构扩展**  
很多团队在规模不大时就急着拆微服务，结果把部署、排障、链路、事务复杂度全引进来了，业务收益却不明显。成熟架构师会先问：拆完以后，是不是能减少协作摩擦，还是只是把摩擦搬到了网络上。

**工程指标**  
- 单需求跨服务改动数：看拆分有效性。  
- 服务共享库/共享表比例：看边界独立性。  
- 服务独立发布率：看自治程度。  
- 跨服务故障传播次数：看耦合度。  
- 服务间同步接口数量：看边界复杂度。

**证据链**  
- 服务依赖图  
- 数据库共享情况  
- 发布记录  
- 需求交付统计

**低风险优化路径**  
1. 先做单体模块化  
2. 再挑高价值边界试拆  
3. 最后持续观察独立性和协作成本

**易错点**  
- 为拆而拆  
- 多服务共用数据库  
- 用数量证明架构先进

## 8. 单体迁移微服务第一刀怎么切？

::: tip L2 深度训练
用薄切片、双轨迁移、兼容和旧路径退场降低改造风险：[ADR、技术债与架构守护](/deep-dives/cloud-governance/04-architecture-governance)
:::

**核心结论**  
单体拆微服务的第一刀，应该优先切出边界相对清楚、收益明显、对主链路冲击可控的能力。

**标准答案**  
我做单体拆微服务时，第一刀通常不会切最核心、最复杂的交易主链路，而是优先找边界清晰、独立性强、对主流程有价值但失败风险可控的模块。比如通知、搜索、营销规则、会员权益、报表这类能力，往往比订单主流程更适合先拆。之所以这样做，是因为第一刀更像组织和工程能力的验证，如果一开始就切最核心域，失败成本太高。工程上我会先补好监控、日志、追踪和发布回滚，再做流量切换和数据同步策略，确保新旧系统能平滑共存。成熟拆分不是“技术上能拆”，而是“拆了以后系统和团队都撑得住”。

**高分回答**
我先建立候选评分：边界清晰、业务收益、变更频率、数据耦合、调用扇入、失败影响、团队 owner 和可回滚性。第一刀选“有真实独立收益但风险可控”的纵向薄切片，不机械规定必须外围模块；通知适合验证平台，却未必验证交易一致性，若目标是解除库存容量瓶颈，也可选择一个隔离良好的库存子场景。采用 strangler：在单体内部先建接口与数据 owner，新服务通过 ACL 适配旧模型；用 CDC/事件迁移数据时明确单一写主、对账与延迟，避免无期限双写。先影子读比较结果，再按租户/场景灰度切流，异常回到旧路径；完成后删除旧代码、表写权限和同步任务，防止“双轨永久化”。拆分前补契约、trace、SLO、值班和发布能力。验收不仅看上线，还看独立发布、需求 lead time、数据差异、跨边界调用、故障隔离和旧路径退场，若收益不达标暂停下一刀并复盘边界。

**追问点答案**  
- 为什么不先拆最核心链路？  
因为风险高，容错空间小。  
- 第一刀最重要的收益是什么？  
验证边界、流程和团队协作方式。  
- 先拆报表、通知是不是没价值？  
有平台验证价值，但业务边界验证有限。第一刀应匹配改造目标：若要验证交付可选外围；若要解除核心瓶颈，应选风险可控的核心薄切片，并设置更强灰度与回退。

**原理补充**  
系统拆分的第一阶段，重点不是追求最优边界，而是低风险建立迁移路径和工程信心。

**架构扩展**  
很多单体拆分失败，不是方向错，而是第一刀切得太重，直接把最复杂的交易主链路搬出去，结果组织、工具、治理都没准备好。成熟团队通常会先用外围能力打样，跑通一整套分布式开发和运维流程。

**工程指标**  
- 首批拆分服务独立发布成功率：看工程基础是否到位。  
- 新旧系统协同故障数：看迁移平滑度。  
- 拆分后需求交付效率变化：看收益。  
- 数据同步异常数：看边界切分质量。  
- 首批服务回滚耗时：看风险可控性。

**证据链**  
- 模块依赖关系  
- 发布与回滚记录  
- 新旧链路流量分布  
- 迁移方案文档

**低风险优化路径**  
1. 先选边界清晰的外围模块  
2. 再补治理和发布能力  
3. 最后逐步切向核心链路

**易错点**  
- 第一刀直接切最核心交易链路  
- 没有灰度和回滚方案  
- 拆分前不补观测能力

## 9. DDD 在中小团队是否过度设计？

**核心结论**  
DDD 不是天然过度设计，关键在于用多深、多重；对中小团队来说，轻量化地用建模思想往往比完整套模板更有价值。

**标准答案**  
我看 DDD 是否过度设计，通常不会简单回答“是”或“不是”，而是看业务复杂度和团队承载能力。如果业务规则简单、团队小、需求快，硬上完整战术模式和一堆分层，确实容易把事情做重；但如果业务已经明显复杂，概念混乱、规则四处散落，那完全不用建模思想也会很痛。比较稳的做法，是先吸收 DDD 里最值钱的部分，比如统一语言、边界划分、核心规则回归模型，而不是一开始把所有术语和模式全套进去。成熟架构师谈 DDD，不会把它当宗教，而是当一套按复杂度取舍的方法。

**高分回答**
DDD 的投入应与业务规则复杂度而非公司规模匹配。简单 CRUD、生命周期短且没有语义冲突时，清晰分层和模块就够了；若同一术语多义、规则频繁变化、多个入口口径不一致，即使五人团队也需要统一语言和限界上下文。轻量起步：与业务维护术语表和事件流程，在模块化单体内明确数据 owner，把关键不变量放领域对象；只有聚合、领域事件或 CQRS 能解决已观察到的问题时才引入。每个模式都应回答“减少了哪类变更或错误”，禁止为了目录结构创建空洞 Repository、Factory 和 Service。通过需求跨模块数、规则重复、业务 bug、新人理解时间和交付 lead time比较收益；若抽象层增加却指标不改善，回退到简单设计。DDD 产物也要随业务维护，否则旧模型比无模型更危险。

**追问点答案**  
- 中小团队最值得用 DDD 的哪部分？  
统一语言、业务事件和模块边界回报最高；先让核心规则有唯一位置，再按实际并发、一致性和查询问题引入战术模式，不按清单全套落地。
- 为什么完整照搬容易过重？  
因为团队未必需要那么多模式和分层。  
- 不用 DDD 就一定简单吗？  
不一定，复杂业务不用建模往往会更乱。

**原理补充**  
DDD 的核心不是模式数量，而是让代码结构更忠实地反映业务结构。

**架构扩展**  
现实里很多团队不是被 DDD 害得复杂，而是被“半懂不懂的 DDD”害得复杂。比如名字都叫得很对，代码里却还是过程式堆在 Service。真正有效的是把建模思想和团队现状结合起来，而不是追求术语完整度。

**工程指标**  
- 业务术语歧义数：看统一语言收益。  
- 核心规则重复实现次数：看模型沉淀效果。  
- Service 层流程堆积比例：看是否仍然过程化。  
- 新成员理解核心业务耗时：看模型可读性。  
- 需求变更影响范围：看边界稳定性。

**证据链**  
- 领域术语文档  
- 代码结构分布  
- 规则重复统计  
- 新人 onboarding 反馈

**低风险优化路径**  
1. 先统一语言和边界  
2. 再收敛核心规则位置  
3. 最后按需引入更多模式

**易错点**  
- 把 DDD 当全量模板照搬  
- 只学术语不改建模方式  
- 业务并不复杂却做得很重

## 10. 如何识别核心域和支撑域？

::: tip L2 深度训练
结合业务差异化、变化频率、规则复杂度和投入策略判断：[领域发现与限界上下文](/deep-dives/business-evolution/01-domain-discovery)
:::

**核心结论**  
核心域通常承载差异化竞争力和关键业务规则，支撑域更多是重要但可替代、可平台化的能力。

**标准答案**  
我识别核心域和支撑域时，通常会先问一个问题：如果这块能力做得不好，会不会直接削弱公司的业务竞争力，或者让关键链路失去差异化。如果答案是会，那它更可能是核心域；如果它虽然重要，但市面上有成熟方案、规则相对标准化、替代成本可控，那更可能是支撑域。之所以这个区分很重要，是因为资源投入方式应该不同。核心域更适合投入更强的人和更多精力做深度建模，支撑域则更适合追求稳定、复用和平台化。成熟架构师不会对所有模块一视同仁，而会先按业务价值分层。

**高分回答**
我与业务负责人从差异化与经济价值判断，而非“是否关键运行”。支付通道很重要但可采购，可能是通用/支撑域；独有的定价、风控或履约优化若直接提高转化、毛利和壁垒，才更像核心域。再看规则独特性、变化频率、数据反馈飞轮、失败影响、市场替代和战略时限，形成域组合图。核心域投入最强领域专家和工程团队，保留自主模型、快速实验与高可靠；支撑域追求稳定接口、复用或购买，避免过度自研，但仍有明确 SLO 和退出方案。身份会随阶段变化：早期物流可能支撑，规模化后成为竞争力，因此季度结合业务指标复核。决策还需验证“自研优势能否持续”，否则所谓核心只是沉没成本。用核心域需求对收入/成本贡献、实验周期、规则复制难度、研发投入和支撑域复用/采购 TCO 检查资源是否匹配战略。

**追问点答案**  
- 支撑域是不是不重要？  
不是。认证、结算等支撑域可能是业务底线，需要高 SLO；区别在于是否值得用独特模型重投入，而非能否出故障。
- 核心域为什么要更多投入？  
因为它最影响业务差异化和长期演进。  
- 一个域会变化身份吗？  
会，业务阶段变化后重要性也可能变化。

**原理补充**  
识别核心域，本质上是在做架构资源配置，决定哪些地方值得重投入。

**架构扩展**  
很多团队架构做得辛苦，是因为把大量精力花在非核心领域的技术精巧上，核心链路反而建模粗糙。成熟架构师会先想清楚哪些地方值得“深做”，哪些地方应该“稳做”。

**工程指标**  
- 核心域需求占业务收益比例：看战略价值。  
- 核心域缺陷对业务影响度：看风险权重。  
- 研发投入分布：看资源是否匹配战略。  
- 支撑域平台复用率：看标准化程度。  
- 核心域迭代速度：看投入是否有效。

**证据链**  
- 业务目标拆解  
- 需求来源分布  
- 研发投入统计  
- 关键链路收益与故障影响分析

**低风险优化路径**  
1. 先按业务价值给域分层  
2. 再把关键人才和设计精力集中到核心域  
3. 最后把支撑域平台化和标准化

**易错点**  
- 把“重要”直接等同于“核心”  
- 核心域和支撑域都用同样设计力度  
- 不随着业务阶段调整判断

## 11. 业务规则放在哪一层最合适？

**核心结论**  
业务规则最适合放在最接近领域语义、又能被统一保护的地方，而不是分散在 Controller、Mapper 或脚本式 Service 里。

**标准答案**  
我放业务规则时，通常会先区分规则到底属于展示、流程还是领域决策。如果它只是接口格式、参数校验，那可以留在接口层；如果它是事务编排和用例组合，可以放在应用层；但如果它决定业务状态是否合法、动作能不能发生、约束是否满足，那更适合进入领域对象或领域服务。之所以规则位置这么重要，是因为规则一旦分散，后续很容易出现 A 接口能做、B 接口也能做，但判断条件不一样。工程上比较稳的做法，是让核心规则有唯一可信位置，然后其他层只做适配和编排。这样做不仅更容易复用，也更容易验证和 review。

**高分回答**
我按规则语义和一致性归位。格式、类型和协议校验在接口层；权限、幂等、事务与用例顺序在应用层；决定“订单能否取消、折扣是否成立”且所有入口必须一致的业务不变量放实体、值对象或领域服务。跨域流程由 saga/流程管理器编排，不应假装成单个聚合规则。领域方法接受业务概念而非 DTO，构造时阻止非法状态，避免 Controller、定时任务和消费者各复制一套 if。外部政策如税率可通过领域端口提供输入，但具体 RPC 和缓存位于基础设施层，保证领域测试纯净。规则版本化场景要记录适用时间和决策依据，历史订单不能被新规则无意重算。治理上从高频 bug 和重复判断开始收敛，以特征测试锁住旧行为，再逐入口替换；通过规则重复数、入口差异、领域测试覆盖、变更影响面和线上规则争议验证，而不是追求“领域层代码占比”。

**追问点答案**  
- 为什么很多规则最后堆在 Service？  
因为临时需求最容易在那里加判断。  
- Controller 能不能放业务规则？  
可以做协议格式、必填和认证上下文转换；凡是 API、消息和批任务都应一致执行的业务决策，必须进入统一领域入口，否则换入口就会绕过。
- 规则集中有什么好处？  
减少重复、避免口径不一致。

**原理补充**  
规则归位的本质，是让业务不变量有稳定、唯一、可验证的承载位置。

**架构扩展**  
线上很多“同一个业务不同接口结果不一致”的问题，根因不是接口多，而是规则散。成熟系统会逐步把规则从流程层抽回模型层，这样无论是 API、任务还是批处理，最终用的都是同一套判断。

**工程指标**  
- 核心规则重复实现次数：看规则分散度。  
- 不同入口规则不一致导致的问题数：看口径统一性。  
- 领域层规则覆盖率：看模型承载能力。  
- Service 层临时判断增长速度：看规则外溢程度。  
- 规则相关 bug 修复耗时：看维护成本。

**证据链**  
- 代码结构扫描  
- 规则重复案例  
- 接口对比测试  
- 事故复盘

**低风险优化路径**  
1. 先找高频重复规则  
2. 再统一沉到领域层  
3. 最后清理外围层散落逻辑

**易错点**  
- 哪层方便就放哪层  
- 一个规则多处复制  
- 把流程判断和业务规则混在一起

## 12. 领域事件什么时候引入？

::: tip L2 深度训练
区分命令与已发生事实，并设计提交后可靠发布：[聚合、不变量与领域事件](/deep-dives/business-evolution/02-aggregate-invariants)
:::

**核心结论**  
领域事件适合在一个领域内完成核心决策后，把已发生的重要业务事实向外传播，用于解耦后续动作。

**标准答案**  
我引入领域事件时，通常会先确认这个动作是不是已经形成了稳定业务事实，比如订单已创建、支付已完成、库存已预占。只有当这个事实本身值得被其他边界感知，而且不希望强耦合地同步调用，我才会考虑用领域事件。之所以不是所有动作都发事件，是因为事件一多很容易把系统变成难追踪的“广播场”。工程上比较稳的做法，是只为重要业务事实建事件，明确事件语义、幂等策略和消费边界，同时保留审计和补偿能力。成熟架构师讲事件时，不会只停留在“异步更优雅”，而会顺带讲一致性和可追踪性。

**高分回答**
领域事件描述聚合完成决策后已经发生、值得其他边界感知的事实，用过去式并携带稳定 eventId、aggregateId、version、occurredAt 与最小业务数据。例如 `PaymentSucceeded` 是事实，`CreateShipment` 是命令，二者不能混用。事件与聚合状态在同一本地事务写入 outbox，提交后异步发布；消费者以 eventId/业务键幂等，处理乱序、重复、重试和死信，并有对账回放。事件契约遵循只增字段、版本和弃用窗口，不能直接暴露内部表结构；敏感数据最小化。域内事件可同步驱动同一边界规则，跨上下文集成事件需经过 ACL 转换，避免模型泄漏。只有需要解耦多个后续动作、审计或读模型时引入，不为每次字段更新广播。验收发布延迟、消费成功、幂等冲突、死信年龄、事件到业务状态收敛与 trace 可追踪，并演练提交成功但 broker 不可用。

**追问点答案**  
- 为什么不是所有状态变更都发事件？  
因为会带来额外复杂度和噪音。  
- 领域事件和技术事件有什么区别？  
前者描述业务事实，后者更多描述系统动作。  
- 事件发出去就算完成了吗？  
不够。必须证明状态与 outbox 原子、最终发布、消费者幂等、乱序/死信可收敛，并能从事件定位最终业务结果；“MQ send 成功”不是端到端完成。

**原理补充**  
领域事件的核心，是把“已经发生的业务事实”作为解耦协作的基础，而不是把流程控制塞进消息里。

**架构扩展**  
很多团队把事件当作“异步万能胶”，结果系统越来越难排查。真正成熟的事件设计，会严格控制事件种类和语义，只在确实需要跨边界传播事实时使用，并配套事件表、重试、死信和追踪。

**工程指标**  
- 事件消费成功率：看传播稳定性。  
- 事件重复消费率：看幂等治理。  
- 跨域同步延迟：看协作时效。  
- 因事件链路导致的问题数：看复杂度成本。  
- 关键事件可追踪覆盖率：看审计能力。

**证据链**  
- 事件定义文档  
- 事件表或消息记录  
- 消费重试和死信数据  
- 链路追踪信息

**低风险优化路径**  
1. 先只为关键业务事实建事件  
2. 再补幂等和追踪  
3. 最后逐步解耦后续动作

**易错点**  
- 所有动作都发事件  
- 把流程控制写进事件语义  
- 没有幂等和审计能力

## 13. 读写模型分离的触发条件是什么？

**核心结论**  
读写模型分离适合在写模型规则复杂、读侧查询高度定制且性能压力明显时引入，而不是为了概念完整而提前拆。

**标准答案**  
我判断要不要做读写模型分离时，通常先看写侧是不是已经承载了复杂业务规则，而读侧又有大量列表、聚合、搜索、报表这类与写模型不一致的查询需求。如果一个模型既要保护复杂规则，又要满足各种读性能和展示需求，最后往往会变得很别扭。之所以不能太早做分离，是因为它会引入同步链路、一致性延迟和额外维护成本。工程上更稳的策略，是先确认查询压力和读模型差异已经足够明显，再逐步把高价值查询拆成独立读模型，而不是全量上 CQRS。成熟架构师会强调“先有问题，再引入分离”。

**高分回答**
触发条件是读写目标长期冲突：写侧需小聚合保护复杂不变量，而读侧需要跨域聚合、全文搜索、不同排序或数量远高于写入，且索引/只读副本已无法满足 P99 与成本。先从一个高价值查询建立投影，不必全系统 CQRS；写模型仍是事实源，通过 outbox/CDC 更新读模型，记录 source version/checkpoint，消费者幂等并支持全量重建。接口明确 read-your-write 窗口：提交后可返回写结果、短时查主或展示“处理中”，不能假装强一致。读模型故障时允许降级旧查询或稍旧数据，写链路不应被投影阻塞；Schema/事件版本需支持双跑和对账。成本包括额外存储、同步、删除传播和隐私治理。用读 P99/成本、同步 P95/P99、陈旧读投诉、重建时间、数据差异和运维工时验证；如果查询收益不足以覆盖一致性与维护复杂度，就保持单模型。

**追问点答案**  
- 为什么不是所有系统都做 CQRS？  
因为复杂度并不总能被收益抵消。  
- 读写分离后为什么会有一致性延迟？  
因为写事务提交和投影消费不是同一原子动作；需要量化延迟窗口、提供 read-your-write 策略，并用 checkpoint、重放和对账保证最终收敛。
- 什么查询最适合先拆出去？  
报表、搜索、复杂聚合、首页汇总等。

**原理补充**  
读写模型分离的本质，是让“保护规则”和“高效查询”各自使用更合适的数据形态。

**架构扩展**  
在交易、供应链、营销这类系统里，写模型往往必须严守规则，而读侧又需要给运营、客服、BI 提供完全不同的视图。成熟团队不会一开始就全面 CQRS，而是先从最痛的查询场景试点，逐步形成稳定同步链路。

**工程指标**  
- 读查询 TP99：看分离是否带来收益。  
- 读模型同步延迟：看一致性窗口。  
- 写链路事务复杂度：看写模型负担。  
- 复杂查询数量：看分离必要性。  
- 读模型故障对业务影响范围：看方案成熟度。

**证据链**  
- 慢查询与报表需求  
- 写侧规则复杂度清单  
- 读模型同步链路  
- 读写延迟监控

**低风险优化路径**  
1. 先找读写冲突最明显的场景  
2. 再局部引入独立读模型  
3. 最后完善同步和监控能力

**易错点**  
- 为了架构概念提前全面 CQRS  
- 不评估同步复杂度  
- 忽略最终一致窗口

## 14. 跨域事务怎么设计？

::: tip L2 深度训练
缩小强一致边界，显式设计处理中、补偿和对账：[聚合、不变量与领域事件](/deep-dives/business-evolution/02-aggregate-invariants)
:::

**核心结论**  
跨域事务设计的重点，不是强行拉一个全局大事务，而是按业务重要性选择本地事务、事件驱动、补偿和对账等组合方案。

**标准答案**  
我做跨域事务设计时，通常先分清哪些动作必须本地强一致，哪些可以通过最终一致完成。因为一旦跨域，就意味着不同模型、不同服务、不同故障模式，硬做强一致事务通常代价很高。比较稳的做法，是每个域先保证自己的本地事务成功，然后通过事件、消息、状态机和补偿机制推进整体业务完成。之所以不能只说“最终一致就好了”，是因为面试官更关注你怎么补偿、怎么对账、怎么发现异常。工程上我会特别强调失败路径设计，比如幂等、重试、死信、人工介入和对账闭环。成熟架构师谈跨域事务，重点不在名词，而在异常怎么收敛。

**高分回答**
我先定义业务不变量、允许的暂态、完成时限和补偿可逆性。例如下单可经历“库存已预留、支付处理中”，但不能重复扣款或无限占库存。每个域只在本地事务修改自身状态并写 outbox，由显式 saga 状态机推进步骤；每个命令有幂等键、超时与有限重试，事件带版本防乱序。编排式适合流程清晰和统一可见，协同式耦合低但事件链难理解，按复杂度选择。补偿是新的业务动作，不是数据库回滚：支付成功而库存失败要退款并审计，退款失败进入重试、对账和人工队列。不可逆步骤尽量后置，资源使用预留/确认/释放。状态机必须有终态、超时扫描和人工操作防重复，外部渠道以对账作为事实校准。量化处理中年龄、补偿触发/成功、重复副作用、不一致存量、对账发现和恢复 MTTR，并注入消息丢失、重复、乱序与依赖超时验证。

**追问点答案**  
- 为什么不建议跨服务大事务？  
因为耦合重、可用性差、扩展性差。  
- 最终一致最大挑战是什么？  
是让所有暂态最终可观测、可重试、可补偿或可人工收敛，同时不产生重复副作用；必须有状态机、超时、幂等、对账和明确业务 owner。
- 对账为什么重要？  
因为它是发现漏补偿和脏状态的最后防线。

**原理补充**  
跨域事务的核心，是在一致性和可用性之间做现实取舍，并设计可靠的异常收敛机制。

**架构扩展**  
支付、库存、营销权益这类链路最常见。很多事故不是“方案选错”，而是成功路径设计得很好，失败路径没人管。成熟系统在跨域事务上花的精力，往往一半以上都在异常治理上。

**工程指标**  
- 补偿触发率：看异常发生频度。  
- 补偿成功率：看收敛能力。  
- 跨域状态不一致数量：看事务质量。  
- 对账异常发现率：看兜底能力。  
- 异常恢复平均时长：看韧性。

**证据链**  
- 状态机设计  
- 事务消息和补偿记录  
- 对账报表  
- 异常工单和复盘

**低风险优化路径**  
1. 先做本地事务边界清晰化  
2. 再补消息和补偿机制  
3. 最后做对账和人工介入闭环

**易错点**  
- 迷信全局事务  
- 只讲成功路径  
- 没有对账和异常兜底

## 15. 组织结构对领域拆分有什么影响？

::: tip L2 深度训练
让平台降低跨团队认知负担，同时保留领域所有权：[平台工程与黄金路径](/deep-dives/cloud-governance/03-platform-engineering)
:::

**核心结论**  
组织结构会强烈影响领域拆分的落地方式，但不能反过来让短期组织边界永久绑死业务边界。

**标准答案**  
我看组织结构对领域拆分的影响时，会承认现实里团队分工一定会反过来影响系统边界，因为沟通成本、负责人归属、迭代节奏都和组织有关。但我不会简单按组织图来切领域，因为组织会变，业务边界相对更稳定。比较稳的做法，是先按业务语义划出建议边界，再看当前组织能否承接，如果一时承接不了，可以先在代码结构和模块层面落实，等团队成熟再逐步对齐组织。之所以这个问题重要，是因为很多“领域设计”最后失败，不是模型错，而是没有考虑谁来长期负责。成熟架构师会同时看业务合理性和组织可执行性。

**高分回答**
康威定律意味着沟通结构会复制进系统，因此领域边界必须有端到端 owner，能决定模型、数据、SLO 和路线图；若两个团队长期共同写一个聚合，职责争议和锁步发布几乎必然出现。但不能按当前汇报线机械切，因为季度组织调整不应迫使数据迁移。先以业务语言、不变量和变化流划上下文，再设计团队拓扑：流对齐团队拥有核心域，平台团队提供自助能力，复杂子系统团队负责稀缺专业性，跨域协作通过版本化契约而非共享库。组织暂时承接不了时，先同仓模块化、CODEOWNERS 和接口边界，统一发布但独立责任，逐步培养值班和产品能力后再物理拆分。衡量跨团队 handoff、需求触达数、锁步发布、owner 空洞、事故升级和认知负担；若服务边界造成持续高协调，应调整系统或团队，而不是把摩擦正常化。

**追问点答案**  
- 为什么不能完全按组织拆？  
因为组织调整时系统边界会跟着动，代价很大。  
- 为什么又不能完全忽略组织？  
因为没人负责的边界再漂亮也落不下去。  
- 组织还没准备好怎么办？  
先建立模块、数据 owner、契约和责任人，保留统一部署；补齐值班、发布与领域知识后再拆。不要创建无人能端到端维护的“孤儿微服务”。

**原理补充**  
系统边界与组织边界通常会相互塑造，理想设计需要兼顾长期稳定性与现实执行力。

**架构扩展**  
现实里最常见的矛盾是：业务边界应该拆开，但团队还没准备好独立维护。成熟做法通常不是妥协掉边界，而是用阶段性方案过渡，比如同仓不同模块、统一发布但分责任人，逐步把组织和系统往一致方向推。

**工程指标**  
- 领域负责人清晰度：看组织承接能力。  
- 跨团队协作工单数：看组织边界摩擦。  
- 因职责不清导致的事故数：看组织适配度。  
- 模块 ownership 覆盖率：看长期维护能力。  
- 团队调整对架构影响面：看边界独立性。

**证据链**  
- 团队职责图  
- 模块 owner 清单  
- 跨团队需求流转记录  
- 事故责任归因记录

**低风险优化路径**  
1. 先按业务划建议边界  
2. 再评估组织承接方式  
3. 最后分阶段把组织与边界逐步对齐

**易错点**  
- 完全按组织图切系统  
- 完全忽略团队现实  
- 边界设计没人负责落地

## 16. 你如何处理历史包袱系统的模型腐化？

::: tip L2 深度训练
用反腐层保护新语言，并为适配和双轨设置退场指标：[遗留系统与模型腐化治理](/deep-dives/business-evolution/04-legacy-modernization)
:::

**核心结论**  
处理模型腐化的关键，不是一次性重写，而是先识别混乱最集中的边界，用增量重构把规则、术语和责任慢慢收回来。

**标准答案**  
我面对历史包袱系统时，通常不会一上来就说“重构整个模型”，因为这种事大概率落不下来。更稳的做法，是先找最痛的区域，比如高频改动、事故多发、概念最混乱的模块，先把术语统一、规则收拢、接口收口。之所以模型会腐化，通常是因为多年需求叠加下，规则散在各层、同一个概念多种含义、没有稳定边界。工程上我会先通过代码扫描和需求回溯找到污染最重的点，再结合新需求顺手推进重构，而不是脱离业务独立搞大改造。成熟架构师处理遗留系统，往往更强调收敛路径而不是理想蓝图。

**高分回答**
我先用需求变更、事故、复杂度、规则重复和跨模块依赖建立腐化热力图，选业务价值高且变化频繁的“缝隙”，而非平均重构。通过特征测试和生产样本锁住现有行为，与专家建立旧词→新统一语言映射；在新边界前放反腐层，把遗留 DTO/状态翻译为新模型，禁止旧概念继续渗透。采用 strangler：新需求进入新模型，旧路径逐场景迁移，影子比较、对账和灰度切流；每个适配器、双写和开关都有 owner、退场条件和期限。核心规则先收敛唯一入口，再迁移数据，避免“新代码调用旧共享表”长期存在。大重写仅在旧系统不可测试、替换范围小且双轨成本更高时考虑，也需分段验证。指标看高风险模块 lead time、事故、规则重复、旧路径流量、ACL 数量与年龄、数据差异和退场完成率，证明模型在收敛而非只是增加一层包装。

**追问点答案**  
- 为什么不建议大规模一次性重写？  
因为风险高、收益兑现慢、容易中途失控。  
- 怎么判断先改哪一块？  
优先改事故多、变化频繁、边界最乱的地方。  
- 新需求为什么是重构机会？  
它同时提供业务预算、真实验收场景和增量迁移边界；但要设置结构改善与旧路径退场条件，避免新旧逻辑并存后复杂度更高。

**原理补充**  
遗留系统治理的本质，是把模型修复和业务演进绑定，让结构改善不断获得现实驱动力。

**架构扩展**  
真实团队里，最糟糕的两种做法是：一种完全不动，继续堆烂；另一种一口气全重写。成熟方式通常介于两者之间，用“边交付边收敛”的节奏逐步修复模型腐化。

**工程指标**  
- 高风险模块需求改动频率：看治理优先级。  
- 同义概念数量：看语义混乱程度。  
- 遗留模块事故率：看腐化代价。  
- 重构相关回归缺陷率：看治理节奏是否稳。  
- 收敛后代码复杂度变化：看重构效果。

**证据链**  
- 历史需求与事故数据  
- 代码复杂度分析  
- 模型术语对照表  
- 重构方案与阶段结果

**低风险优化路径**  
1. 先选最痛边界  
2. 再统一术语和规则入口  
3. 最后借新需求持续清理腐化区域

**易错点**  
- 想一次性重写全部模型  
- 重构脱离真实业务需求  
- 看不清最痛点就平均发力

## 17. 如何评估一次拆分是否成功？

::: tip L2 深度训练
把交付前置时间、变更失败率和跨团队等待纳入验收：[平台工程与黄金路径](/deep-dives/cloud-governance/03-platform-engineering)
:::

**核心结论**  
拆分成功与否，不是看服务数量或图画得多漂亮，而是看边界是否更清晰、协作是否更顺、变更是否更可控。

**标准答案**  
我评估一次拆分是否成功时，通常会看三类指标。第一类是业务和交付层面，比如单需求是否更少跨团队、发布是否更独立、改动是否更聚焦；第二类是系统质量层面，比如故障隔离是不是更好、依赖关系是不是更清楚；第三类是长期演进层面，比如新需求是否更容易落在清晰边界内。之所以不能只看“拆完运行正常”，是因为很多拆分短期看没问题，半年后才暴露出协作成本更高。工程上我会把拆分前后的交付路径、故障模式和依赖关系做对比，而不是只看技术指标。成熟拆分应该让系统和团队都更轻松，而不是只是架构图更现代。

**高分回答**
拆分立项前我就保存 8–12 周基线和成功门槛：目标可能是库存需求 lead time 降 30%、独立发布率超 80%、故障不再扩散到订单，同时端到端 P99 和运维成本增幅受限。上线后分四维对照：业务交付看需求触达团队/仓库、等待和发布频率；边界自治看数据唯一写 owner、契约兼容与锁步发布；可靠性看级联故障、SLO、恢复和补偿；经济性看基础设施、值班与认知负担。短期迁移期数据不代表稳态，要在多个迭代和事故周期复核，并区分业务流量变化。旧表写入、双写、转发接口和旧服务必须退场，否则只是复制系统。若跨服务调用和协调上升、交付反而变慢，允许合并或重划边界，不能因沉没成本宣布成功。最终用拆分目标的净收益而非服务数、代码仓库数或“运行正常”验收。

**追问点答案**  
- 为什么服务独立发布率很重要？  
因为它能直接反映自治程度。  
- 故障隔离效果怎么体现？  
看问题是否更容易局部止血，不再大面积传播。  
- 如果技术上拆了但协作更慢算成功吗？  
通常不算，除非这是有期限的迁移成本且可靠性/合规收益明确；应检查边界、团队 owner、契约和平台摩擦，并设恢复到目标值的期限。

**原理补充**  
架构拆分的最终目标，是优化变化成本和风险控制，而不是制造更多结构层次。

**架构扩展**  
不少团队拆完服务后会说“系统更先进了”，但真正该问的是：新需求是不是更快了，排障是不是更清楚了，故障是不是更容易挡住了。成熟架构师会把这些作为拆分成败的主要判断标准。

**工程指标**  
- 单需求跨团队数：看协作效率。  
- 服务独立发布率：看自治程度。  
- 拆分后故障影响面：看隔离效果。  
- 接口依赖复杂度：看边界清晰度。  
- 需求交付时长变化：看真实收益。

**证据链**  
- 拆分前后发布记录  
- 需求链路对比  
- 事故影响范围统计  
- 服务依赖图

**低风险优化路径**  
1. 先定义成功指标  
2. 再做拆分前后对比  
3. 最后针对反效果继续修边界

**易错点**  
- 用服务数量衡量成功  
- 不做拆分前后基线对比  
- 忽略团队协作体验

## 18. 领域模型如何与数据库模型解耦？

::: tip L2 深度训练
明确业务模型、事实源、存储映射和查询投影边界：[聚合、不变量与领域事件](/deep-dives/business-evolution/02-aggregate-invariants)
:::

**核心结论**  
领域模型和数据库模型不必一一对应，解耦的关键是让持久化适配业务模型，而不是让业务语义被表结构反向控制。

**标准答案**  
我做领域模型和数据库模型解耦时，通常会先承认两者关注点不同。领域模型关注业务规则、状态变化和语义表达，数据库模型更关注存储效率、查询方式和约束实现。如果强行一一对应，往往会让领域对象被表结构牵着走，最后业务代码里全是持久化细节。工程上比较稳的做法，是通过仓储、映射层或装配逻辑把两者隔开，让领域对象保持业务语义，而把 join、冗余字段、索引优化这些细节留在持久化侧。成熟架构师不会为了“纯粹”完全忽略数据库，也不会让数据库模型直接主导领域设计。

**高分回答**
领域模型负责行为、不变量和身份，持久化模型负责索引、范式、分区和查询，两者通过聚合级 Repository 与显式 Mapper 连接。领域对象不暴露 ORM lazy proxy、表字段或公共 setter；Repository 以业务意图查询并原子保存聚合，使用 version 乐观锁。Money 等值对象可映射多列，订单项可独立表，但这些物理结构不改变领域入口。复杂列表/报表使用专用读模型或 SQL，不强迫加载完整聚合；数据库约束仍作为最后防线，如唯一键保护幂等，但错误映射回领域语义。解耦也有成本，简单 CRUD 上不必复制完全相同模型；在核心域和表结构频繁优化处收益最大。Schema 迁移使用兼容双读/双写或回填、对账和退场，避免 Mapper 永久兼容所有历史形态。验收表结构变化对领域层影响、领域中的 ORM 泄漏、聚合加载大小、映射错误、并发冲突和查询 P99。

**追问点答案**  
- 为什么不能完全按表结构设计对象？  
因为会丢失业务语义，模型容易退化成 CRUD。  
- 解耦是不是一定会增加代码量？  
会，因此只在规则复杂、模型与存储形态明显不同或存储频繁演进处使用；简单 CRUD 可共用模型。目标是降低变更成本，不是追求形式纯洁。
- 查询优化会不会破坏解耦？  
不会，读模型和仓储策略可以单独优化。

**原理补充**  
解耦的核心，是避免存储结构直接决定业务模型边界和行为。

**架构扩展**  
在遗留系统里最常见的情况是“表一改，业务对象全变”。这种耦合会让任何数据库优化都变成业务风险。成熟团队通常会接受适度映射成本，换更稳定的业务抽象和更清晰的演进边界。

**工程指标**  
- 业务对象直接暴露持久化字段比例：看耦合度。  
- 表结构变更影响业务层范围：看解耦效果。  
- 仓储/映射复用率：看边界清晰度。  
- 因数据库改动触发的业务回归数：看风险。  
- 领域层代码中的 SQL/ORM 细节渗透率：看污染程度。

**证据链**  
- 模型类与表结构对比  
- 仓储接口设计  
- 表变更影响分析  
- 代码扫描结果

**低风险优化路径**  
1. 先隔离最核心对象的持久化细节  
2. 再收敛仓储和映射职责  
3. 最后逐步清理业务层表结构依赖

**易错点**  
- 把领域对象直接当 ORM 实体  
- 为追求纯粹而完全无视查询现实  
- 持久化细节渗透到业务规则里

## 19. 多团队协作下如何保证领域一致性？

::: tip L2 深度训练
用 ADR、契约测试、所有权和到期例外治理跨域决策：[ADR、技术债与架构守护](/deep-dives/cloud-governance/04-architecture-governance)
:::

**核心结论**  
多团队协作下的领域一致性，靠的不是大家“口头理解差不多”，而是统一语言、契约治理和持续评审机制。

**标准答案**  
我在多团队协作下保证领域一致性时，通常会先统一核心术语和边界定义，再把关键契约文档化、评审化。因为只要团队一多，同一个概念被不同人按不同理解实现几乎是必然的。比较稳的做法，是维护领域术语表、关键流程图、事件定义和接口契约，并把重大边界调整纳入架构评审，而不是靠聊天记录同步。工程上还要有 owner 机制，让每个核心域有人持续守边界。成熟架构师知道，领域一致性不是一次会议达成的，而是需要持续治理的。

**高分回答**
我建立“可执行的统一语言”。每个限界上下文有业务+技术 owner，术语、事件、状态机和不变量与代码/API 同仓版本化；跨域只共享契约，不强行共享一个全局模型，同一词在不同上下文可不同，但必须经上下文映射明确翻译。API/事件采用 schema registry、兼容规则、consumer-driven contract 和弃用窗口，CI 阻断破坏性变更；重大语义/边界决策用 ADR 记录背景、替代、owner 和复审日期。定期用真实需求和事故做领域评审，文档自动链接实现与仪表盘，避免静态百科。跨团队变更由上游发布迁移计划，下游可在窗口内自治升级，紧急例外有到期。指标包括契约破坏、术语歧义缺陷、锁步发布、评审等待、过期 ADR/契约和 owner 覆盖；治理若让交付变慢，应简化标准或提供平台工具，而不是依赖更多会议。

**追问点答案**  
- 为什么文档化很重要？  
因为口头共识很容易随时间走样。  
- 只有文档就够了吗？  
不够。还要把契约兼容、schema 校验和领域测试接入 CI，由 owner 对变更和退场负责；否则文档会与运行事实漂移。
- 接口契约为什么会影响领域一致性？  
因为很多语义偏差最终都会反映在接口上。

**原理补充**  
领域一致性的核心，是让团队共享同一套业务语言和边界约束，并持续校正偏差。

**架构扩展**  
大团队最怕的不是意见不一致，而是大家都以为自己理解一致。成熟组织会把领域治理做成持续机制，比如定期术语校准、事件变更评审、核心域 owner 审核，这些比单次宣讲更有效。

**工程指标**  
- 术语歧义问题数：看一致性程度。  
- 接口语义变更频率：看边界稳定性。  
- 跨团队评审通过时长：看协作效率。  
- 因语义理解差异导致的缺陷数：看治理效果。  
- 核心域 owner 覆盖率：看责任机制。

**证据链**  
- 领域术语表  
- API/事件契约文档  
- 评审记录  
- 缺陷复盘

**低风险优化路径**  
1. 先统一术语和契约  
2. 再建立 owner 和评审机制  
3. 最后做持续校准和变更管理

**易错点**  
- 靠口头同步维持一致性  
- 没有核心域 owner  
- 契约变更无治理

## 20. 你如何做领域建模评审？

::: tip L2 深度训练
用事件时间线、不变量、上下文映射与反例场景评审：[领域发现与限界上下文](/deep-dives/business-evolution/01-domain-discovery)
:::

**核心结论**  
领域建模评审的重点，不是看图画得好不好，而是验证边界、术语、规则入口和演进路径是否真正合理。

**标准答案**  
我做领域建模评审时，通常会围绕几个问题展开：核心流程是不是梳理清楚了，术语是不是一致，边界划分是否按规则和职责而不是按表结构，核心规则是否有唯一入口，跨域协作是不是有清晰契约。之所以评审不能只看类图，是因为很多模型在纸面上很好看，一落到真实需求和故障场景就立刻崩。工程上我还会特别看异常流程、补偿路径和未来变化点，因为这些地方最能暴露模型是否真的稳。成熟架构师做建模评审，不只是判断“现在能不能用”，还会判断“半年后改起来会不会很痛”。

**高分回答**
评审前要求提供业务目标、统一语言、事件时间线、上下文图、关键不变量和三类真实需求，不以类图为主。会议中让领域专家讲命令与事实，逐条追问：谁拥有决策和数据；哪些规则必须同事务；跨域暂态是否可接受；失败、重复、乱序、取消和人工介入如何收敛。再用反例压力测试模型，例如支付成功后订单取消、部分退款、新监管字段或流量增长十倍，看边界是否需要共享写、大聚合或锁步发布。检查聚合唯一写入口、事件过去式语义、契约版本、读模型一致窗口和组织 owner，并明确未选择方案及触发重新评审条件。问题按阻断/技术债分级，写入 ADR、owner、期限和验收测试；上线后用需求改动面、规则缺陷、补偿与事故反馈模型。好的评审输出是可执行决策和验证用例，不是“大家同意这张图”。

**追问点答案**  
- 为什么异常流程要进建模评审？  
因为很多边界问题都藏在失败路径里。  
- 只看类图为什么不够？  
因为图静态，真实问题往往出现在流程和变化里。  
- 评审时最该追问什么？  
再追问失败与并发下不变量是否仍成立、数据唯一 owner 是谁、跨域暂态如何收敛，以及未来哪类变化会迫使边界重划。

**原理补充**  
建模评审的本质，是在真实业务变化和异常场景下验证模型是否稳定。

**架构扩展**  
真实项目里，很多所谓“模型设计过了评审”，其实只是大家对着几张图点头。成熟评审会故意把高频变更、异常流程、跨域协作都拿出来跑一遍，看模型会不会立刻泄底。

**工程指标**  
- 评审后模型重大返工次数：看评审质量。  
- 规则入口重复发现数：看边界问题。  
- 异常流程覆盖率：看评审深度。  
- 建模变更对既有边界影响面：看稳定性。  
- 评审问题闭环完成率：看治理执行力。

**证据链**  
- 领域模型图  
- 关键流程与异常流程  
- 契约定义  
- 评审意见与闭环记录

**低风险优化路径**  
1. 先按流程和术语做评审  
2. 再补异常和演进场景验证  
3. 最后跟踪评审问题闭环

**易错点**  
- 评审只看静态图  
- 不追问异常和变化路径  
- 评审完没有闭环动作
