# 架构模块8 - 性能工程 - 标准答案库

## 1. 性能优化的第一步是什么？

::: tip L2 深度训练
先建立用户目标、变更时间线和可证伪假设，再开始调参：[尾延迟与性能回归](/deep-dives/capacity-performance/03-tail-latency-regression)
:::

**核心结论**  
性能优化的第一步不是改代码，而是先明确瓶颈、目标和基线。

**标准答案**  
我做性能优化时，第一步通常不会直接上手调参数，而是先回答三个问题：现在到底慢在哪里、希望优化到什么程度、当前基线是多少。之所以这样做，是因为很多所谓优化最后只是在优化错方向，比如用户感知慢其实是下游超时，结果团队花了很多时间去调 JVM。工程上更稳的做法是先建立基线，包括吞吐、TP95/TP99、CPU、内存、IO、下游耗时和错误率，再决定优化顺序。面试里如果能补一句“性能优化的第一步是测量，而不是猜测”，会很成熟。

**高分回答**
第一步是把模糊的“慢”变成可证伪的问题和性能预算。我先明确用户旅程、流量结构、SLO 与约束，例如“支付确认在峰值 3k TPS 下 P99 小于 800ms、错误率低于 0.1%，单请求成本不增加”。随后固定版本、数据量、缓存状态和依赖延迟建立基线，将端到端耗时分解为网关排队、应用执行、连接池等待、数据库和外部依赖，并同时记录吞吐、资源饱和与错误率。根据证据提出单一假设，例如“连接池等待贡献 P99 的 45%，根因是慢 SQL 占用连接”，先用 trace、池指标和数据库执行计划交叉验证，再做最小改动。优化必须和相同场景及对照组比较，检查 P50/P99、容量拐点、正确性和单位成本，防止通过拒绝请求、放宽一致性或增加三倍机器获得漂亮数字。若收益不显著或副作用超预算立即回退，并把压测脚本、配置和结果纳入版本管理，形成可重复的性能回归门禁。

**追问点答案**  
- 为什么不能直接调参数？  
因为没有基线就无法判断优化是否有效，甚至可能越调越差。  
- 基线里最重要的是哪类指标？  
通常是用户感知指标和资源瓶颈指标一起看。  
- 为什么要先定目标？  
因为目标决定优化边界和停止条件。没有明确负载、分位延迟、错误率、正确性与成本约束，“更快”既无法验收，也容易把问题转移给下游或基础设施。

**原理补充**  
性能优化本质上是一个受约束的工程决策问题，不是纯粹的技术试错。

**架构扩展**  
成熟架构师不会把性能优化当成“高手调优秀操作”，而是把它做成一个标准闭环：建基线、找瓶颈、做改动、再验证。真正稳定的收益来自方法，不来自灵光一现。

**工程指标**  
- TP95/TP99：用户感知核心指标。  
- QPS/TPS：吞吐指标。  
- CPU/内存/IO 水位：资源指标。  
- 错误率：稳定性指标。  
- 优化前后对比差值：收益指标。

**证据链**  
- 压测报告  
- APM 数据  
- 系统资源监控  
- 业务性能基线看板

**低风险优化路径**  
1. 先定目标和基线  
2. 再找单一主要瓶颈  
3. 最后做最小改动验证

**易错点**  
- 一上来就调参数  
- 没有明确目标  
- 优化后不做对比验证

## 2. 如何设计压测场景才真实？

::: tip L2 深度训练
识别闭环负载、协调遗漏、均匀 key 和热缓存造成的失真：[真实性压测与瓶颈证据](/deep-dives/capacity-performance/02-realistic-load-testing)
:::

**核心结论**  
真实压测场景的关键不是流量大，而是流量结构、数据分布和依赖行为都尽量像生产。

**标准答案**  
我设计压测场景时，通常会先确定是验证峰值承载、容量水位、降级效果还是回归对比，然后把生产里的访问比例、热点分布、读写结构、缓存命中率、慢请求比例和外部依赖行为尽量带进去。之所以很多压测结果不可信，是因为场景过于理想化，比如全是热缓存、全是均匀数据、没有异常流量。工程上真实压测还要覆盖冷启动、缓存未预热、消息积压、慢 SQL 和重试放大这些情况。面试里如果能补一句“压测场景决定了压测结论能不能用于生产决策”，会很完整。

**高分回答**
我先从生产最近高峰提取请求 mix、到达率、会话长度、对象大小、热点 key、地域和读写比例，再按目标设计基准、峰值、突发、耐久、故障和恢复场景。负载模型优先采用开放模型按真实到达率持续发请求；闭环模型会因服务变慢而自动降低发压量，产生协调遗漏，让 TP99 看起来更好。数据侧要复刻基数、倾斜、索引选择性、缓存命中与失效，依赖侧要模拟延迟分布、限额、超时和错误，而不是统一返回固定 20ms。先验证压测机、网络和数据生成器自身未饱和，再逐级升压找到延迟拐点和稳定容量；过程中同时校验业务结果、重复写、积压和降级，不能只看 QPS。至少覆盖冷启动、缓存击穿、单可用区故障、下游变慢、重试风暴和长时间资源泄漏。最后将生产同流量窗口与压测的延迟、命中率和资源曲线比较，记录偏差并校准；测试数据需隔离、可清理，生产压测还要有限流、熔断和停止开关。

**追问点答案**  
- 为什么热点分布很重要？  
因为热点往往比平均流量更早打出瓶颈。  
- 为什么要覆盖冷启动？  
因为很多系统上线或扩容时最脆弱。  
- 模拟生产数据是不是必须？  
不必复制敏感原始数据，但基数、对象大小、热点倾斜、索引选择性和读写关联必须接近；可用脱敏快照或合成数据，并用生产分布统计校验。

**原理补充**  
压测结果本质上取决于输入条件，输入越失真，结论越危险。

**架构扩展**  
成熟架构师不会只看“打了多少 QPS”，而会继续问：这个 QPS 是不是在真实热点、真实缓存命中、真实下游延迟条件下得到的。只有这样压测结果才有决策价值。

**工程指标**  
- 场景覆盖率：测试完整度指标。  
- 热点命中比例：真实性指标。  
- 缓存命中率：关键环境指标。  
- 下游仿真准确度：依赖真实性指标。  
- 压测结论与生产偏差率：最终可信度指标。

**证据链**  
- 场景定义文档  
- 生产流量分析  
- 压测数据样本  
- 依赖仿真配置

**低风险优化路径**  
1. 先复刻核心链路和热点  
2. 再补冷启动和异常流量  
3. 最后拿生产数据校准场景

**易错点**  
- 只压理想路径  
- 不考虑热点和冷启动  
- 压测数据和生产完全脱节

## 3. TPS 和 TP99 如何同时优化？

::: tip L2 深度训练
用联合门禁避免通过拒绝请求换取更漂亮的 TP99：[尾延迟与性能回归](/deep-dives/capacity-performance/03-tail-latency-regression)
:::

**核心结论**  
TPS 和 TP99 常常互相拉扯，要同时优化就必须同时控制平均吞吐和尾部放大因素。

**标准答案**  
我通常把 TPS 和 TP99 一起看，因为很多系统吞吐上去了，但尾延迟被排队、锁竞争、GC 或慢依赖拖得很差。之所以这两个指标容易冲突，是因为系统在接近饱和时，平均吞吐可能还能涨，但尾部请求会先恶化。工程上要同时优化，关键是减少排队、控制并发、做资源隔离、优化慢路径，并把热点请求和普通请求区分对待。面试里如果能补一句“TPS 是整体产出，TP99 是最差体验，它们必须一起守”，会很成熟。

**高分回答**
我先画负载—吞吐—P99—错误率曲线，找到排队开始非线性增长的 knee point，并把安全容量放在它之前而不是机器跑满的位置。根据 Little 定律 `L = λW`，服务时间或并发不变时盲目增加到达率只会扩大在途请求和等待；因此先用 trace 拆出执行与排队，定位锁、池等待、GC、热点 key 或慢依赖。优化优先减少每请求服务时间与长尾：批量但设上限、消除 N+1、缩短临界区、隔离慢租户与长短任务；随后用有界队列、并发上限、截止时间和背压防止过载。限流能保护 TP99，但吞吐必须统计“成功完成的有效 TPS”，并同时守错误率与业务正确性，不能靠快速拒绝伪造提升。按请求类型分别设预算，避免低价值批任务挤压在线交易；多实例还需检查负载倾斜。最终在峰值和耐久压测中验证有效 TPS 达标、P99/SLO 合格、队列稳定、无资源持续增长，并比较单位吞吐成本。

**追问点答案**  
- 为什么吞吐提高时尾延迟容易变差？  
因为系统更接近饱和，排队和竞争会先体现在尾部。  
- 只看平均耗时为什么不够？  
因为平均值会掩盖最差那部分用户体验。  
- 如何优先保护 TP99？  
先减少长尾服务时间，再用隔离、有界队列、并发控制、截止时间和背压防止排队扩散；同时监控成功 TPS 和拒绝率，避免以大量失败换取低 TP99。

**原理补充**  
TPS 和 TP99 的平衡，本质上是在系统利用率和用户体验之间找最优点。

**架构扩展**  
成熟架构师不会追求“把机器跑满”，而是会问：在多大吞吐下 TP99 开始明显变差，这个点通常就是更真实的容量边界。

**工程指标**  
- TPS：吞吐指标。  
- TP99/TP999：尾延迟指标。  
- 队列等待时长：尾部放大指标。  
- 锁等待与慢依赖占比：根因指标。  
- 资源饱和度：边界判断指标。

**证据链**  
- 压测曲线  
- 请求耗时分布  
- 队列和线程池监控  
- 下游慢调用分析

**低风险优化路径**  
1. 先找 TP99 最差来源  
2. 再控制并发和排队  
3. 最后在安全水位内提升吞吐

**易错点**  
- 只追 TPS  
- 用平均耗时掩盖尾部问题  
- 把系统压到临界点还当成功

## 4. CPU 瓶颈怎么定位？

::: tip L2 深度训练
从吞吐曲线、火焰图、GC 和锁事件建立证据链：[真实性压测与瓶颈证据](/deep-dives/capacity-performance/02-realistic-load-testing)
:::

**核心结论**  
CPU 瓶颈定位的关键不是看到 CPU 高就结束，而是进一步搞清楚是业务计算、锁竞争、GC 还是系统调用在吃 CPU。

**标准答案**  
我定位 CPU 瓶颈时，通常先看是单实例高还是整体高，再区分 user、system、iowait 和 steal 等比例，然后结合火焰图、线程栈和热点方法找出到底是计算密集、序列化、JSON 解析、正则、锁自旋还是 GC 在消耗 CPU。之所以要这么拆，是因为 CPU 高只是现象，不同根因的优化路径完全不一样。工程上还要看高 CPU 是否伴随 TP99 恶化、线程池饱和或下游调用异常。面试里如果能补一句“CPU 高不等于业务逻辑复杂，有时是系统在为错误状态买单”，会比较有经验感。

**高分回答**
我先确认 CPU 真的是限制因子：随着负载上升，每核有效吞吐不再增长、运行队列和 P99 上升，且没有主要 IO 等待。再比较实例、核、容器 request/limit 与宿主机，排除单实例倾斜、CPU throttling、steal 和指标口径错误。按 user/system/iowait/softirq 拆分后，在问题窗口采集低开销 async-profiler 火焰图、线程状态、GC CPU 和上下文切换，并与 trace 的慢路径及发布事件对齐。user 高可能来自序列化、正则、加解密或死循环；system/softirq 高要查小包、系统调用和网络；GC 高查分配速率；CPU 不高但线程阻塞则应转查锁或 IO。优化应针对热点路径做最小改动，例如缓存稳定计算、裁剪序列化、批量系统调用或修复自旋，并警惕缓存一致性、内存增加和批量放大尾延迟。使用固定流量 A/B 或同场景压测验证每核有效 TPS、P99、GC、错误率和成本；若只是线性可扩的真实计算，再扩容或水平拆分，否则扩容只会暂时掩盖缺陷。

**追问点答案**  
- iowait 高算 CPU 问题吗？  
不是 CPU 计算饱和的直接证据，它表示任务在等待 IO；还要结合磁盘时延/队列、网络和应用 trace 找到具体通道，不能因总 CPU 看似空闲就继续加并发。
- 为什么要看火焰图？  
因为它能很快显示热点调用路径。  
- GC 也会导致 CPU 高吗？  
会，尤其在对象 churn 很大时。

**原理补充**  
CPU 瓶颈本质上是系统在某类计算或状态维护上花了过多时间片。

**架构扩展**  
成熟架构师看到 CPU 高，不会马上说“加机器”，而是会先判断是热点业务计算、代码缺陷还是 JVM/序列化/锁问题。加机器只适合吞吐型问题，不适合机制型问题。

**工程指标**  
- CPU user/system/iowait：分类指标。  
- 热点方法耗时占比：根因指标。  
- GC CPU 占比：JVM 指标。  
- TP99：业务感知指标。  
- 每核吞吐：效率指标。

**证据链**  
- `top` / `pidstat`  
- 火焰图  
- `jstack` / profiler 数据  
- APM 热点方法报告

**低风险优化路径**  
1. 先区分 CPU 类型  
2. 再抓热点调用链  
3. 最后按根因选优化方案

**易错点**  
- CPU 高就直接扩容  
- 不分 user/system/iowait  
- 不看热点方法

## 5. 内存瓶颈怎么定位？

**核心结论**  
内存瓶颈定位的核心是区分“内存用得多”与“内存回不来”，也就是高占用和泄漏是两回事。

**标准答案**  
我定位内存问题时，通常先看是堆内还是堆外、是持续增长还是周期波动、GC 后是否明显回落，再结合 dump、对象分布和分配速率判断是缓存过大、对象生命周期过长、集合持有、类加载泄漏还是堆外缓冲使用失控。之所以要这么分，是因为内存高可能是正常预热，也可能是典型泄漏。工程上还要把容器内存限制、RSS、JVM 配置和应用对象模型一起看。面试里如果能补一句“真正危险的是内存回收失灵，而不是单次高峰”，会比较准确。

**高分回答**
我先按时间和内存域定位：对齐流量、发布、GC 与容器 OOM 时间线，区分 Java heap、metaspace、direct buffer、线程栈、mmap/page cache 和 native allocator；只看 JVM heap 会漏掉 RSS 超限。堆内观察 Full GC 后 live set 是否持续爬升、分配速率和晋升速率：使用量随缓存预热增长后稳定可能正常，基线跨多个 GC 周期上升才像泄漏。随后在受控时点获取两份 heap dump 做 dominator tree 与 retained size 差分，沿 GC roots 查静态集合、ThreadLocal、监听器、无界缓存或 classloader 持有；堆外用 Native Memory Tracking、direct buffer 指标和线程数补证据。先用限流、缩小批次或缓存上限止血，抓证据后再重启，避免重启抹掉现场。修复需明确对象 owner、生命周期、容量上限和释放路径，并同时核验 OOMKill、GC P99、命中率与吞吐，防止简单减缓存把压力转给数据库。通过高基数/长连接/滚动发布耐久压测验证 GC 后基线稳定，且容器 limit 留有堆外和故障余量。

**追问点答案**  
- GC 后回落很小说明什么？  
说明 live set 较大，但不等于泄漏；要看多个完整 GC 后基线是否持续增长，并结合业务缓存预热、流量和 dump 的持有链判断。
- 堆外内存也会出问题吗？  
会，比如 direct buffer 或 native 内存。  
- 缓存是不是也可能是“假泄漏”？  
是，所以要先判断它是否符合设计预期。

**原理补充**  
内存瓶颈的本质，是对象生命周期与资源回收机制之间失去平衡。

**架构扩展**  
成熟架构师看内存问题，不会只盯着堆大小，而会把本地缓存、批处理、对象模型和容器资源限制一起看。否则很容易只治标不治本。

**工程指标**  
- Heap 使用率与回落幅度：核心指标。  
- RSS：系统内存真实占用指标。  
- 对象分配速率：压力来源指标。  
- Full GC 次数：风险指标。  
- OOM 事件数：最终结果指标。

**证据链**  
- Heap dump  
- GC 日志  
- 容器内存监控  
- 对象分布分析

**低风险优化路径**  
1. 先区分堆内外和是否回落  
2. 再找最大对象和持有链  
3. 最后优化模型、缓存或资源配置

**易错点**  
- 内存高就说泄漏  
- 只看堆不看堆外  
- 不看 GC 回落情况

## 6. IO 瓶颈怎么定位？

::: tip L2 深度训练
区分服务执行、池等待、网络和下游 IO 时间：[尾延迟与性能回归](/deep-dives/capacity-performance/03-tail-latency-regression)
:::

**核心结论**  
IO 瓶颈定位的重点是判断问题发生在磁盘、网络、序列化还是同步写策略，而不是只看到系统变慢。

**标准答案**  
我定位 IO 瓶颈时，通常先区分是磁盘读写、网络收发，还是某种大对象序列化/反序列化造成的隐性 IO 成本。之所以要这么分，是因为应用表现出来都可能是“慢”，但底层原因完全不同。工程上会结合 iowait、磁盘队列长度、网卡带宽、包大小、刷盘时延和应用日志来看；如果是数据库或 MQ 侧，还要关注 fsync、日志刷盘和批量策略。面试里如果能补一句“IO 瓶颈很多时候不是算法问题，而是资源通道问题”，会比较准确。

**高分回答**
我先用 trace 把端到端慢拆成应用执行、连接/线程池等待、磁盘、网络和下游响应，确认慢的是数据搬运而不是 CPU 或锁。磁盘侧看设备利用率、平均与 P99 延迟、队列深度、吞吐和随机/顺序读写；高 iowait 只是线索，还要排除同宿主机噪声、page cache miss、日志与数据库争盘。网络侧逐跳比较 DNS、建连、TLS、首字节和传输时间，并检查 RTT、重传、丢包、带宽、连接数、包大小和跨区路径。应用侧再查同步 fsync、逐条小写、过量日志、N+1 请求、压缩/序列化和大 payload。例如吞吐未满但磁盘 P99 与队列同时上升，说明 IOPS 或刷盘成为瓶颈；带宽未满但重传高则不能靠扩带宽解决。优化从减少 IO 次数和数据量开始：批量但有上限、裁剪字段、缓存稳定读、异步写但明确持久性语义；更换存储或拓扑是高成本后手。验证时同时守数据正确性、durability、P99、积压与恢复时间，防止把同步写改异步后只得到表面提速。

**追问点答案**  
- iowait 高一定是磁盘问题吗？  
不一定。它表示 CPU 有任务在等待块 IO，还需结合设备延迟、队列和进程 IO 定位；网络等待通常不会直接表现为 iowait，虚拟化与指标口径也可能干扰判断。
- 网络也算 IO 吗？  
算，特别是跨服务调用和大对象传输时。  
- 为什么序列化也会像 IO 瓶颈？  
因为它常和网络、日志、存储一起放大耗时。

**原理补充**  
IO 瓶颈本质上是数据搬运和持久化通道成为系统节拍限制器。

**架构扩展**  
成熟架构师谈 IO 瓶颈时，不只会说“换更快磁盘”，还会继续讨论批量化、压缩、缓存、异步写和协议裁剪，因为真正成本常在数据路径设计上。

**工程指标**  
- iowait：系统层指标。  
- 磁盘队列长度：磁盘拥塞指标。  
- fsync/刷盘时延：持久化指标。  
- 网络吞吐与丢包率：网络路径指标。  
- 大对象比例：负载结构指标。

**证据链**  
- `iostat` / `sar` / `iftop`  
- 应用 trace  
- 存储和网络监控  
- 大包与日志分析

**低风险优化路径**  
1. 先区分磁盘与网络路径  
2. 再找同步写和大对象热点  
3. 最后优化批量、异步和压缩策略

**易错点**  
- 所有慢都归到代码  
- 不区分磁盘和网络  
- 不看数据负载结构

## 7. 锁竞争瓶颈怎么定位？

**核心结论**  
锁竞争瓶颈的本质不是“线程多”，而是线程在共享资源上等待时间过长。

**标准答案**  
我定位锁竞争时，通常会先看线程状态分布和热点代码路径，再结合锁等待时间、阻塞队列、数据库锁或 JVM monitor 数据判断是应用内锁、连接池锁、缓存锁还是数据库行锁在拖慢请求。之所以锁竞争容易被忽略，是因为 CPU 可能不高，但 TP99 已经非常差。工程上常见根因包括锁粒度过大、临界区做了重逻辑、热点对象竞争和数据库事务范围太宽。面试里如果能补一句“锁竞争问题本质上是共享资源建模问题”，会更有深度。

**高分回答**
我先确认等待而非计算是主因：P99 上升、吞吐停滞、CPU 未满，同时 BLOCKED/WAITING 线程、数据库 lock wait 或分布式锁等待增加。应用内用 JFR/async-profiler lock 事件和多次线程快照找到锁对象、持有者、等待者及临界区；数据库用等待图、长事务和 deadlock 日志定位表、索引、行及锁顺序；分布式锁还要看租约、续期、网络抖动与 fencing token。关键指标不是“锁次数”，而是 `等待时长 × 受影响请求数` 和对核心链路的贡献。低风险顺序是把 IO/远程调用移出临界区、缩短事务、统一加锁顺序、拆无关状态；再考虑分段锁、按 key 分区、乐观并发或队列串行化。去锁会引入竞态，扩大粒度会降低并行度，乐观重试在高冲突下会放大负载，因此必须明确一致性不变量和冲突率。用热点与非热点混合压测验证锁等待 P99、有效吞吐、重试/死锁率和正确性，并通过故障注入确认租约过期或进程暂停时不会出现双写。

**追问点答案**  
- CPU 不高也会有严重锁问题吗？  
会，因为线程可能主要在等待。  
- 数据库锁也算性能瓶颈吗？  
当然，很多业务瓶颈就是行锁或 gap lock 竞争。  
- 如何快速确认是锁问题？  
看慢请求时间线是否主要消耗在锁等待，并用线程/JFR 锁事件或数据库等待图找到相同锁对象和持有者；单次线程栈只是快照，需要多次采样或事件数据交叉确认。

**原理补充**  
锁竞争的本质，是为了保证一致性而引入的串行区间过长或过热。

**架构扩展**  
成熟架构师不会把锁问题只理解成 `synchronized`，而是会同时看数据库事务、分布式锁、缓存互斥和线程池队列，因为这些本质上都是共享访问控制。

**工程指标**  
- 锁等待时间：核心指标。  
- 阻塞线程数：并发受损指标。  
- 临界区耗时：根因指标。  
- TP99：业务感知指标。  
- 死锁/超时次数：稳定性指标。

**证据链**  
- `jstack` / profiler  
- DB 锁监控  
- trace 中的等待段  
- 代码锁粒度分析

**低风险优化路径**  
1. 先定位最热共享资源  
2. 再缩短临界区  
3. 最后优化模型或改并发策略

**易错点**  
- 只看 CPU 不看等待  
- 只查应用锁不查 DB 锁  
- 不分析临界区内容

## 8. JVM 调优一般流程是什么？

**核心结论**  
JVM 调优不是背参数，而是先识别现象、确认目标，再对内存分配、GC 和对象模型做最小必要调整。

**标准答案**  
我做 JVM 调优时，一般先看现象是吞吐差、TP99 差、Full GC 频繁、Young GC 过密还是 OOM，然后结合 GC 日志、堆分布、对象分配速率和业务流量确认根因。之所以不能直接调堆大小，是因为有时根因根本不是参数，而是对象 churn、缓存设计、批处理模型或堆外使用异常。工程上比较稳的顺序通常是：先减对象创建和大对象，再优化新生代与老年代比例，最后再考虑垃圾收集器和参数微调。面试里如果能补一句“JVM 调优的核心是先治对象模型，再治参数”，会很成熟。

**高分回答**
我先把目标定为吞吐、最大可接受停顿、内存上限和成本，例如峰值下 GC P99 小于 100ms、GC CPU 低于 10%，而不是泛泛追求“GC 少”。固定 JDK、容器 limit、流量与数据集，采集统一 GC 日志、JFR、分配速率、live set、晋升、大对象、RSS 和应用 P99。先判断是分配过快、存活集过大、泄漏、堆外超限还是收集器目标不匹配；若批处理一次创建 2GB 临时对象，调参数只是推迟问题。低风险顺序是限制批次和缓存、减少无效分配与大对象、修复持有链，再按 live set 与峰值分配量确定堆并预留 metaspace/direct buffer/线程栈空间，最后才调整收集器和少量参数。更大堆会降低 GC 频率却可能增加停顿与容器 OOM 风险，低停顿收集器也会消耗并发 CPU。每次只改一组因素，以峰值和耐久场景比较应用 P99、有效吞吐、GC 停顿分布、GC CPU、回收后占用及 RSS，保留配置和回滚入口；参数应随 JDK 版本重新验证，不能永久复制旧经验值。

**追问点答案**  
- 为什么先看 GC 日志？  
因为它能直接反映回收频率、停顿和回收效果。  
- 调大堆是不是总能缓解问题？  
不是。它可能降低频率，却拉长回收周期、压缩堆外余量并推迟泄漏暴露；应由 live set、峰值分配和容器总内存预算共同决定。
- 什么时候值得换 GC 算法？  
当当前收集器明显不匹配业务目标时。

**原理补充**  
JVM 调优的本质，是让对象生命周期模式与回收机制更匹配。

**架构扩展**  
成熟架构师谈 JVM 调优时，不会只说 G1/CMS 参数，而是会先解释业务流量、对象模型、缓存和批处理行为。这说明理解的是系统，而不是参数表。

**工程指标**  
- Young/Full GC 次数：核心指标。  
- GC 停顿时间：体验指标。  
- GC 后堆回落幅度：效果指标。  
- 对象分配速率：压力来源指标。  
- OOM 事件数：稳定性指标。

**证据链**  
- GC 日志  
- Heap dump  
- APM 对象分配分析  
- 流量与延迟对照数据

**低风险优化路径**  
1. 先定位现象与根因  
2. 再优化对象模型  
3. 最后微调 JVM 参数

**易错点**  
- 上来就调参数  
- 不看 GC 日志  
- 忽略业务对象模型

## 9. GC 停顿怎么降？

**核心结论**  
降低 GC 停顿的关键，不是单纯减少 GC 次数，而是减少会触发大停顿的对象形态和回收压力。

**标准答案**  
我通常会先看停顿主要发生在 Young 还是 Full GC，再看是不是大对象分配、老年代积压、缓存过大、批量处理一次拉太多数据或类元数据异常导致。之所以 GC 停顿难降，是因为它常常是对象模型和业务流量共同作用的结果。工程上常见手段包括分页和流式处理、限制大对象、减少临时对象、控制缓存上限、调整堆比例和选择更匹配的收集器。面试里如果能补一句“GC 停顿本质上是对象生命周期管理失败的表现”，会很有深度。

**高分回答**
我先从 GC 日志按原因拆分停顿：Young evacuation、并发周期跟不上、to-space exhausted、humongous allocation、Full GC 或元空间触发，并将停顿时间线与请求 P99、分配峰值和批任务对齐。Young GC 频繁但每次 10ms 且吞吐达标不一定是问题；真正要优化的是超过 SLO 的停顿及其业务影响。若分配速率高，先减少临时对象和反序列化；大对象多则分页、流式化并限制请求体；live set 大则治理无界缓存和长生命周期持有；并发标记启动过晚才调整阈值或预留空间。选择 G1/ZGC 等收集器要基于堆规模、JDK 版本、CPU 余量和停顿目标，并用相同工作负载验证，不能只换名字。还要检查容器 CPU throttling，因为 GC 线程拿不到 CPU 会延长周期。优化同时约束吞吐、GC CPU、RSS、错误率和数据处理完成时间，避免用更多 CPU 或更大堆换停顿。通过长时间压测和突发流量确认没有晋升失败、Full GC、分配停顿或恢复后积压。

**追问点答案**  
- Young GC 多就一定不好吗？  
不一定，要看单次停顿时间和整体吞吐。  
- 为什么 Full GC 更危险？  
它通常需要处理更大存活集并可能 Stop-The-World，容易让请求超时和重试放大；更重要的是查清触发原因，单次显式 GC 与持续回收失败的风险并不相同。
- 大对象为什么容易出问题？  
因为它们更容易直接进入老年代并增加回收成本。

**原理补充**  
GC 停顿本质上是 JVM 为恢复内存可用性而让业务线程暂停付出的代价。

**架构扩展**  
成熟架构师处理 GC 停顿，不会只把锅甩给 JVM，而是会回到代码、缓存、批任务和请求模型上找原因。真正长期有效的优化常在业务层。

**工程指标**  
- GC 停顿 P99：核心体验指标。  
- Full GC 次数：高风险指标。  
- 大对象比例：根因线索。  
- 老年代使用率：压力指标。  
- 请求超时率：业务结果指标。

**证据链**  
- GC 日志  
- 对象大小分布  
- Heap dump  
- 慢请求与 GC 时间线

**低风险优化路径**  
1. 先区分 Young / Full 问题  
2. 再找大对象和长生命周期对象  
3. 最后调参数和收集器

**易错点**  
- 只盯 GC 算法  
- 不看对象模型  
- 只追求 GC 次数少

## 10. SQL 性能优化优先级怎么排？

::: tip L2 深度训练
把 SQL 延迟放回端到端预算和真实性混合负载中验证：[尾延迟与性能回归](/deep-dives/capacity-performance/03-tail-latency-regression)
:::

**核心结论**  
SQL 优化优先级不是看谁最慢，而是看谁最频繁、影响面最大、收益最高。

**标准答案**  
我排 SQL 优化优先级时，通常会综合看三件事：调用频次、单次成本和业务影响。之所以最慢的 SQL 不一定最该优先优化，是因为有些 SQL 虽然慢，但调用很少；而有些“中等慢”的 SQL 可能每天跑几百万次。工程上比较稳的排序方法是先找高频高耗时、核心链路高影响、容易低风险优化的 SQL，再逐步处理长尾。面试里如果能补一句“SQL 优化优先级本质上是收益排序，而不是技术难度排序”，会比较成熟。

**高分回答**
我先把连接池等待、网络和数据库执行分开，避免接口慢却误判为 SQL 慢；随后用慢日志和 digest 按 `总数据库时间 ≈ 调用次数 × 平均耗时` 排出资源大户，再叠加核心链路 P99、锁影响、CPU/IO、扫描行数与业务风险。最高优先级通常是高频、高总耗时、影响核心交易且可低风险修复的查询；偶发报表即使单次最慢，也可先隔离到只读库或离线任务。对候选 SQL 收集真实参数分布、表规模、索引基数和 EXPLAIN/实际执行证据，检查 rows examined/returned、回表、排序/临时表、谓词选择性、复合索引顺序与锁等待。改造顺序是 SQL/访问方式、匹配具体谓词和排序的索引、缩短事务，再考虑缓存/读写分离/模型重构；索引会增加写放大、空间和 DDL 风险，读写分离还要处理复制延迟。上线采用影子查询或灰度，对比结果正确性、P99、扫描行、buffer 命中、锁等待、主从延迟和写入成本，并准备删索引或回退 SQL。只有明确单机瓶颈和增长阈值后，才讨论分片。

**追问点答案**  
- 为什么要看业务影响？  
因为优化资源有限，优先解决最影响用户和收入的。  
- 低风险优化是什么意思？  
是影响面可控、兼容性明确、能灰度且可快速回滚的改动。加索引也不天然低风险：要评估在线 DDL、写放大、空间、锁和执行计划漂移。
- 长尾 SQL 要不要管？  
要，但通常放在高收益项之后。

**原理补充**  
SQL 治理本质上是把数据库优化资源投入到收益最大的点上。

**架构扩展**  
成熟架构师做 SQL 治理，不会拿一个排行榜就一条条机械处理，而是会结合接口价值、调用量和可改造成本做组合排序。

**工程指标**  
- SQL 总耗时占比：收益排序核心指标。  
- 调用频次：影响面指标。  
- 单次扫描行数：成本指标。  
- 关联接口 TP99：业务影响指标。  
- 优化实施成本：决策指标。

**证据链**  
- 慢日志聚合  
- SQL 调用统计  
- 业务链路映射  
- 优化方案评估

**低风险优化路径**  
1. 先做高频高耗时排序  
2. 再叠加业务影响  
3. 最后按低风险高收益优先

**易错点**  
- 只盯单条最慢 SQL  
- 不看调用频次  
- 不考虑业务价值

## 11. 缓存命中率低怎么提升？

**核心结论**  
提升缓存命中率的关键，不是盲目扩大缓存，而是先搞清楚 miss 来自冷启动、TTL、键设计还是业务访问分散。

**标准答案**  
我通常会先区分缓存命中率低是因为数据本来就长尾分散，还是因为 TTL 太短、key 设计不合理、热点未预热、缓存容量不够或者回源逻辑有问题。之所以不能一看到命中率低就扩容，是因为很多时候命中率低不是容量问题，而是建模或策略问题。工程上常见优化包括热数据预热、分层缓存、TTL 优化、合并 key、热点保护和只缓存真正有重复读取价值的数据。面试里如果能补一句“命中率优化的本质是让缓存承接重复价值，而不是承接所有读请求”，会比较准确。

**高分回答**
我先按缓存层、业务接口、key 空间和 miss 类型拆指标，而不是看一个全局命中率：compulsory miss 来自首次访问，capacity miss 来自淘汰，expiration miss 来自 TTL，consistency miss 来自主动作废或版本切换。结合 key 访问频次、对象大小、TTL 年龄和 eviction reason 判断工作集是否能装下，以及低命中是否真的造成高回源成本。例如图片元数据命中 90% 但回源便宜，价值可能低于命中 60% 却保护昂贵聚合查询的缓存。低风险优化是修正 key 维度、只缓存有重复价值的数据、对热点预热、TTL 加抖动并用 singleflight 合并并发回源；扩容只解决 capacity miss，不能解决一次性长尾访问。延长 TTL 要明确陈旧容忍度和失效协议，负缓存要限制时长，防止不存在数据长期污染结果。用分阶段灰度观察加权命中收益、回源 QPS/数据库 P99、淘汰率、内存成本、陈旧读和热点倾斜；缓存故障演练还要验证回源限流，避免高命中率掩盖数据库无承载余量。

**追问点答案**  
- 命中率越高越好吗？  
不一定。应看避免的回源成本、用户延迟和一致性风险，可用“命中节省的后端时间/成本”做加权；缓存低价值小查询来提高比例可能毫无收益。
- 为什么热点预热有效？  
因为很多 miss 都发生在高峰前几分钟或新实例刚起来时。  
- 缓存长尾数据有意义吗？  
通常要看重复访问概率，不是所有数据都值得缓存。

**原理补充**  
缓存命中率本质上反映的是“缓存和业务访问模式的匹配程度”。

**架构扩展**  
成熟架构师不会把命中率当唯一目标，而会一起看回源成本、内存占用、用户体验和热点分布。命中率高但价值低，可能只是浪费资源。

**工程指标**  
- 缓存命中率：基础指标。  
- 回源 QPS：实际压力指标。  
- 热点 key 占比：结构指标。  
- 缓存容量利用率：成本指标。  
- 预热命中率：策略效果指标。

**证据链**  
- 命中/未命中日志  
- key 分布数据  
- 回源监控  
- 缓存配置和 TTL 策略

**低风险优化路径**  
1. 先分清 miss 类型  
2. 再优化热点、TTL 和 key 设计  
3. 最后再考虑扩容

**易错点**  
- 命中率低就只加缓存  
- 不看长尾访问模式  
- 不区分容量问题和建模问题

## 12. 网络延迟高怎么排查？

**核心结论**  
网络延迟高的排查关键，是先判断问题在客户端、本机网络、跨机房链路、服务端处理还是序列化传输成本。

**标准答案**  
我排查网络延迟时，通常会先通过 trace 判断时延到底集中在网络传输还是服务端处理，再看 RTT、丢包、重传、DNS、TLS 握手、连接复用和包大小这些因素。之所以网络问题容易误判，是因为业务上看到的是“接口慢”，但慢不一定真的发生在链路上。工程上如果跨机房调用多，还要重点关注链路稳定性和重试放大。面试里如果能补一句“网络延迟排查本质上是把总耗时拆成传输时间和处理时间”，会很准确。

**高分回答**
我先把客户端观察到的总耗时拆为排队、DNS、TCP/QUIC 建连、TLS、请求发送、服务端处理、首字节和响应传输，并在客户端、网关、服务端使用同一 request ID 与校准时间线。若 trace 只记录服务端 30ms、客户端却 500ms，再按地域、可用区、运营商、协议版本和连接复用率分组，避免平均值掩盖单路径故障。链路证据包括 RTT 分布、丢包/重传、拥塞窗口、SYN/TLS 失败、NAT/conntrack、网卡队列与跨区路由；大 payload 还要拆序列化 CPU 和线上传输。`ping` 正常不能证明应用正常，因为协议、端口、连接池和负载路径不同。优化按根因选择：短连接多就复用与预连接，跨区 chatty 调用就合并或迁移计算，包大就裁字段/分页，丢包则处理链路和 MTU；重试必须有预算、退避和截止时间，避免放大。通过同地域对照、抓包或 eBPF 样本和故障时段回放验证 P99、重传、握手率、有效吞吐及错误率。

**追问点答案**  
- 为什么先看 trace？  
它能先分出服务执行和跨进程空白，但 trace 也可能漏掉客户端排队、DNS 与采样外请求，所以还要结合客户端分段计时、网络指标和必要的抓包。
- 包太大也会导致延迟高吗？  
会，特别是在序列化和跨机房场景。  
- TLS 握手也要关注吗？  
要，短连接或频繁建连时影响很明显。

**原理补充**  
网络延迟本质上是链路质量、连接策略和数据大小共同作用的结果。

**架构扩展**  
成熟架构师谈网络慢，不会只说“查 ping”，而是会继续看连接复用、跨地域链路、协议设计和 payload 大小。这才是真正的系统视角。

**工程指标**  
- RTT：链路基础指标。  
- 丢包/重传率：稳定性指标。  
- TLS 握手时长：连接指标。  
- 平均包大小：负载结构指标。  
- 跨机房调用 TP99：业务结果指标。

**证据链**  
- trace  
- 网络监控  
- 连接池和握手数据  
- 抓包或协议日志

**低风险优化路径**  
1. 先区分传输慢还是处理慢  
2. 再看链路质量和连接策略  
3. 最后优化 payload、复用和路由

**易错点**  
- 接口慢就默认网络慢  
- 不看连接建立成本  
- 不分析包大小和跨机房因素

## 13. 线程池耗尽怎么治理？

::: tip L2 深度训练
用 Little's Law 解释流量不变、等待变长为何仍会耗尽资源：[容量模型与安全水位](/deep-dives/capacity-performance/01-capacity-model)
:::

**核心结论**  
线程池耗尽治理的重点不是把线程数加大，而是减少排队、缩短任务占用时间并控制流量进入。

**标准答案**  
我治理线程池耗尽时，通常先看是任务执行时间变长、下游依赖变慢、队列积压、线程数配置不合理还是任务模型根本不适合放在线程池里。之所以不能简单把线程开大，是因为线程更多只会增加上下文切换，还可能把下游压得更厉害。工程上更有效的做法通常是拆分线程池、缩短任务、异步化、限流、超时控制和拒绝策略优化。面试里如果能补一句“线程池耗尽本质上是系统在用排队掩盖瓶颈”，会比较有深度。

**高分回答**
我先用 `L = λW` 判断耗尽机制：到达率不变时，下游从 100ms 变 2s，在途任务会扩大 20 倍；此时加线程只是把压力传给连接池和下游。证据上同时看 active/max、队列长度与等待年龄、任务执行 P99、拒绝、线程状态和下游耗时，区分 CPU 任务、阻塞 IO、锁等待和任务泄漏。止血先缩短超时到业务截止时间、暂停低优先任务、限流并启用明确拒绝/降级，防止无界队列把快速过载变成数分钟超时。治理上按依赖和优先级做 bulkhead，在线请求与批任务分池；队列必须有界，线程数按 CPU 或阻塞系数及下游容量设定，而不是越大越好。异步化只有在全链路支持背压、取消和超时传播时才有效，否则只是把队列藏到别处。验证下游变慢、线程卡死和流量突发时，高优先链路仍满足 SLO，队列可回落且恢复后无请求风暴；持续统计等待 P99、拒绝的业务影响、下游并发和恢复时间。

**追问点答案**  
- 为什么队列太长也是问题？  
因为它会把故障延迟暴露成大面积超时。  
- 拒绝策略有价值吗？  
有，但必须按业务语义选择快速失败、降级、转异步或调用方执行，并携带可识别错误；静默丢弃会造成数据不一致，调用方执行也可能拖垮入口线程。
- 什么情况下该拆线程池？  
不同类型任务相互影响时就很值得拆。

**原理补充**  
线程池耗尽的本质，是有限执行资源被长期占满，而新的工作没有及时被拒绝或转移。

**架构扩展**  
成熟架构师遇到线程池耗尽，不会只调 `maxPoolSize`，而是会重新审视任务模型和下游依赖，因为真正的问题通常在线程池外面。

**工程指标**  
- 活跃线程数：基本指标。  
- 队列长度：核心风险指标。  
- 任务等待时长：用户体验指标。  
- 拒绝次数：保护动作指标。  
- 下游超时率：根因相关指标。

**证据链**  
- 线程池监控  
- 任务耗时分布  
- 拒绝日志  
- 下游依赖性能数据

**低风险优化路径**  
1. 先找线程被占住的原因  
2. 再限流、隔离和缩短任务  
3. 最后微调线程池参数

**易错点**  
- 线程耗尽就加线程  
- 不看队列积压  
- 不区分任务类型

## 14. 如何做性能回归防退化？

::: tip L2 深度训练
建立吞吐、TP99、错误率、资源效率和成本联合门禁：[尾延迟与性能回归](/deep-dives/capacity-performance/03-tail-latency-regression)
:::

**核心结论**  
性能防回归的关键是把关键链路基线自动化，让性能问题在变更阶段就暴露，而不是上线后才发现。

**标准答案**  
我做性能防回归时，通常会先挑出最关键的几条链路，给它们建立稳定基线，比如核心接口 RT、吞吐、关键 SQL 耗时和缓存命中率，然后把这些基线接到 CI、压测或灰度校验里。之所以性能退化难防，是因为它常常不是一次明显故障，而是版本一点点变差。工程上更稳的做法是设定阈值区间而不是绝对值，同时保留样本数据和环境条件，避免误报。面试里如果能补一句“性能回归治理的目标是让退化在变更期暴露，而不是在生产期暴露”，会比较成熟。

**高分回答**
我建立分层门禁：PR 阶段跑微基准和关键算法，主干阶段跑固定环境的服务级场景，候选版本跑真实混合负载，灰度阶段再以稳定版本作同期对照。每个门禁版本化代码、JDK、机器、配置、数据集和负载模型，并联合约束有效吞吐、P50/P99、错误率、CPU/内存/IO、数据库扫描与单位请求成本；只看平均 RT 或 QPS 会漏掉尾延迟和资源退化。阈值采用历史方差、重复试验和最小效应量，例如 P99 连续 3 次恶化超过 10% 且置信区间不重叠才阻断，减少共享环境噪声误报。门禁失败自动关联 commit、profile 和 trace，但允许带 owner、到期时间和风险说明的例外，不能永久放行。生产侧通过版本 cohort 和 SLO 检测测试未覆盖的慢性退化，并回灌场景。衡量回归拦截率、误报/漏报、从引入到发现时间、基线波动和退化上线率，定期删除不稳定或无业务价值的测试。

**追问点答案**  
- 为什么不能只靠手工压测？  
因为频率低，且很难持续守住每次变更。  
- 为什么要固定基线环境？  
环境变化会让性能对比失真。  
- 阈值为什么不能太死？  
因为测量存在噪声和自然方差；应通过重复运行、稳定对照、统计区间和最小有意义退化判断，但核心 SLO 与安全容量仍要有不可突破的硬门槛。

**原理补充**  
性能防回归本质上是把性能当成一类持续交付质量门禁。

**架构扩展**  
成熟架构师做性能工程，不会只在大促前才想到压测，而是会把关键性能指标纳入日常发布质量保障。这样才是真正的“工程能力”。

**工程指标**  
- 基线通过率：核心指标。  
- 回归触发次数：风险暴露指标。  
- 误报率：门禁质量指标。  
- 核心链路性能趋势：长期指标。  
- 退化上线率：最终失败指标。

**证据链**  
- 基线定义  
- CI/压测记录  
- 性能趋势图  
- 发布与回归关联数据

**低风险优化路径**  
1. 先挑关键链路建基线  
2. 再接入发布和测试流程  
3. 最后持续校准阈值

**易错点**  
- 只在上线前临时测  
- 没有稳定基线环境  
- 性能门禁无业务优先级

## 15. 如何做性能基线管理？

::: tip L2 深度训练
版本化负载模型、数据集、环境、结果波动和生产校准：[尾延迟与性能回归](/deep-dives/capacity-performance/03-tail-latency-regression)
:::

**核心结论**  
性能基线管理的核心不是存一组数字，而是维护“在什么条件下系统应该表现成什么样”的持续参照系。

**标准答案**  
我做性能基线管理时，通常会明确链路、负载模型、环境、版本、数据样本和关键指标，然后把这些信息一起保存，而不是只记一个 RT 或 QPS 数字。之所以基线需要完整上下文，是因为没有条件描述的性能数字几乎没有比较价值。工程上比较成熟的做法是按场景维护多套基线，比如单接口基线、全链路基线、高峰场景基线和灰度版本基线。面试里如果能补一句“性能基线本质上是系统的性能档案”，会比较完整。

**高分回答**
一条可用基线应是可复现实验包，而不是“P99=200ms”：它包含业务旅程、开放/闭环负载模型、请求 mix、热点和数据规模、缓存冷热、依赖延迟、软硬件/JDK/配置版本、预热与采样窗口，以及吞吐、分位延迟、错误、资源和成本结果。按用途维护微基准、服务级、全链路、峰值/突发/耐久与故障降级基线，但只覆盖高价值链路，避免维护失控。每次运行保存原始分布和环境指纹，显示历史中位数与波动带；环境或模型变化时先并行跑旧/新条件，解释断点并新建基线版本，不能直接覆盖历史。基线需要定期用生产流量结构、数据增长和真实 P99 校准，否则测试长期稳定也可能与现实脱节。owner 由业务与平台共同承担：业务确认场景和正确性，平台保证环境与工具可重复。通过覆盖率、可重复性、生产偏差、失效原因和基线陈旧时间判断质量，而不是追求基线数量。

**追问点答案**  
- 为什么一条基线不够？  
因为不同链路、不同负载和不同环境表现差异很大。  
- 基线多久更新一次？  
架构、运行时、硬件、数据规模或负载结构发生实质变化时版本化更新，并按固定周期用生产数据校准；更新必须保留旧基线和变化说明，不能为让门禁通过而抬高标准。
- 基线谁来维护？  
最好由平台和业务研发共同维护。

**原理补充**  
性能基线管理本质上是在变化中的系统里维持一个可持续比较的参照坐标。

**架构扩展**  
成熟架构师做基线管理，不会只让测试团队单独维护，而是会让架构、业务和平台共同确认“哪几条链路最值得长期观察”。

**工程指标**  
- 基线覆盖率：建设指标。  
- 基线失效率：说明条件是否过期。  
- 基线更新频次：维护节奏指标。  
- 回归判断准确率：管理质量指标。  
- 关键链路基线稳定度：长期变化指标。

**证据链**  
- 基线文档  
- 环境配置  
- 测试样本  
- 版本与性能历史

**低风险优化路径**  
1. 先定义少量关键基线  
2. 再补条件上下文  
3. 最后纳入版本和变更管理

**易错点**  
- 只有数字没有上下文  
- 所有链路都试图建基线  
- 基线长期不更新

## 16. 如何评估优化收益是否值得？

**核心结论**  
优化值不值得，不只看性能能提升多少，还要看实现成本、风险、稳定性收益和后续维护代价。

**标准答案**  
我评估优化收益时，通常会把问题量化成几个维度：当前损失有多大、优化后能提升多少、需要投入多少开发和测试成本、会不会引入新复杂度、能否复用到其他场景。之所以不能只看“快了 20%”就做，是因为有些优化会显著增加代码复杂度和运维成本，但业务收益很有限。工程上我更偏向优先做高频链路、低风险、高复用的优化项。面试里如果能补一句“性能优化本质上也是投资决策”，会比较有架构感。

**高分回答**
我把性能优化当投资组合：先算当前问题的年度损失，包括受影响请求、转化/超时损失、额外机器和排障工时；再在真实负载下估计收益，而不是引用微基准的 20%。决策可用 `预期净收益 = 成功概率 × 业务与成本收益 - 实施成本 - 风险期望损失 - 长期维护成本`，同时检查是否解决近期容量缺口或 SLO。候选方案按最低风险排序：例如 SQL 改写两天可让核心链路 P99 降 25%，通常优于耗时两月、只再提升 5% 的存储重构。收益要包含容量推迟多久、单位请求成本和稳定性，成本则包含开发、测试、迁移、可观测、认知负担和供应商锁定。先用原型或 5% 灰度验证关键假设，设置达到/停止门槛；若收益低于最小有效值或副作用超预算就回退。上线后核对实际收益、复发和维护工时，修正后续估算，防止所有项目都用乐观数字立项。

**追问点答案**  
- 为什么不是所有慢问题都值得优化？  
因为资源有限，应该优先解决高价值问题。  
- 成本除了开发时间还有什么？  
测试、运维、复杂度、可维护性和潜在风险。  
- 哪类优化最值得优先做？  
影响核心 SLO 或近期容量、证据充分、改动小且可灰度回滚的优化；可复用是加分项，但不能用平台化愿景掩盖当前收益不确定。

**原理补充**  
优化收益评估本质上是投入产出比和风险收益比的综合平衡。

**架构扩展**  
成熟架构师不会把优化当作纯技术秀，而是会说明：这个问题一年会影响多少请求、多少收入、多少投诉，优化之后值不值。这样才能推动资源投入。

**工程指标**  
- 优化前后 TP99 差值：直接收益指标。  
- 受影响请求占比：问题覆盖面指标。  
- 实施人天成本：投入指标。  
- 风险变更点数量：风险指标。  
- 复用收益范围：长期价值指标。

**证据链**  
- 性能数据对比  
- 业务影响评估  
- 实施成本估算  
- 变更复杂度分析

**低风险优化路径**  
1. 先量化问题规模  
2. 再估收益和实施成本  
3. 最后优先做高 ROI 项

**易错点**  
- 只看技术提升不看业务价值  
- 不评估复杂度成本  
- 优化优先级靠主观判断

## 17. 何时水平扩展优于垂直扩展？

**核心结论**  
当单机扩容收益开始递减，或者瓶颈更适合通过分散负载解决时，水平扩展通常比垂直扩展更值。

**标准答案**  
我通常在几种情况下优先考虑水平扩展：第一，单机资源已经接近规格上限；第二，瓶颈是并发承载而不是单线程性能；第三，业务具备无状态或易分片特征；第四，需要更好的容灾和弹性。之所以垂直扩展不总是最优，是因为它虽然简单，但会越来越贵，而且单机失败风险也更集中。工程上比较稳的判断方式是看单机扩容后收益是否仍接近线性，如果开始明显递减，就该考虑水平扩展。面试里如果能补一句“水平扩展解决的是规模问题，垂直扩展解决的是单点能力问题”，会比较完整。

**高分回答**
我先识别瓶颈能否被复制分摊。无状态计算在增加实例后吞吐近线性、负载均衡且下游有余量，适合水平扩展；单线程热点、全局锁、单主写入或共享数据库饱和不会因应用实例增加而消失。短期流量缺口且更大规格仍有良好性价比时，垂直扩展改动小、风险低；当规格接近上限、价格非线性、需要弹性或单机故障域不可接受时，水平扩展更合理。水平方案必须计入状态外置、分片键、再均衡、跨节点一致性、连接数放大、负载倾斜和运维复杂度。例如应用扩到 20 个实例却把数据库连接从 400 放大到 2000，整体容量可能更差。决策前分别测 scale-up 与 scale-out 曲线，比较有效吞吐、P99、单 TPS 成本、N-1 容量和扩缩时间；有状态系统先读副本、分区或业务拆分，不能直接复制写节点。通常先用垂直扩展争取时间，同时消除共享瓶颈并验证小规模水平扩展，再按增长阈值切换。

**追问点答案**  
- 垂直扩展有什么优势？  
实现简单、改动小、短期见效快。  
- 水平扩展一定更复杂吗？  
通常更复杂，因为涉及状态、路由和一致性。  
- 哪类系统最适合水平扩展？  
无状态、请求可独立处理且负载可均匀路由的服务最容易；有状态系统只有在分区键、再均衡、一致性和故障恢复都明确后才适合。

**原理补充**  
水平与垂直扩展的选择，本质上是在“单点能力提升”和“负载分散能力提升”之间做决策。

**架构扩展**  
成熟架构师不会机械说“互联网系统都该水平扩展”，而是会先判断：当前问题是不是加机器能平摊掉，还是单线程/锁/存储真相源根本不能靠多实例解决。

**工程指标**  
- 单机扩容收益曲线：决策核心指标。  
- 单实例吞吐：能力指标。  
- 扩容后成本变化：成本指标。  
- 容灾等级变化：稳定性收益指标。  
- 状态迁移复杂度：实施风险指标。

**证据链**  
- 单机性能测试  
- 成本对比  
- 状态分布和路由模型  
- 容灾需求说明

**低风险优化路径**  
1. 先看单机扩容收益是否递减  
2. 再判断业务是否适合分散  
3. 最后设计最小水平扩展方案

**易错点**  
- 还没到瓶颈就过早分布式化  
- 只看技术潮流不看问题类型  
- 无状态和有状态服务一视同仁

## 18. 如何避免过度优化？

**核心结论**  
避免过度优化的关键，是始终围绕真实瓶颈和明确收益做改动，而不是为了“更优雅”或“理论更快”去复杂化系统。

**标准答案**  
我避免过度优化时，通常会先问：当前问题是否真实影响了用户、业务或成本；优化后是否有明确量化收益；如果不做是否真的会带来明显损失。之所以过度优化常见，是因为工程师容易对热点代码过度敏感，却忽略了系统真正的瓶颈可能根本不在那里。工程上比较稳的原则是：先用最小改动解决最大的瓶颈，保留可回退路径，不为了 5% 理论收益引入 50% 复杂度。面试里如果能补一句“性能优化的边界是收益开始小于复杂度时”，会比较成熟。

**高分回答**
我在开始前写清问题、证据、目标、约束和停止条件，例如“核心链路峰值 P99 超出 800ms，目标降到 600ms，错误率与成本不恶化；达标即停止”。先用 profile 和端到端预算确认热点对用户结果的贡献，Amdahl 定律也提醒我们：只占总耗时 5% 的代码即使加速 10 倍，整体收益仍有限。方案从删除无效工作、改配置或局部算法开始，避免为假设中的未来规模提前引入分布式缓存、分片和自研协议。每个优化都记录复杂度、兼容性、故障模式、可观测与回滚成本，并用相同基线验证净收益；若收益落在测量噪声内、SLO 已有充分余量，或维护成本超过预计损失，就停止。保留简单实现作为对照和回退，定期删除未命中、未使用的“优化”路径。评审要求业务影响和数据证据，防止以技术偏好替代优先级，但也为已知增长阈值预留可逆扩展点，而非完全忽略未来。

**追问点答案**  
- 为什么理论更快不一定值得做？  
因为复杂度、维护成本和风险可能更高。  
- 什么算真实瓶颈？  
负载增长时它先达到饱和或显著贡献端到端 P99/错误，并能通过 profile、等待指标和受控改动重复验证；单一高指标或热点方法不一定就是限制因子。
- 怎么防止团队陷入过度优化？  
建立基线、做 ROI 评估、要求优化前后验证。

**原理补充**  
过度优化本质上是资源投入和复杂度增加脱离了真实问题规模。

**架构扩展**  
成熟架构师会主动阻止一些“看起来很厉害”的优化，因为系统长期可维护性往往比局部极致性能更重要。真正好的优化是收益稳、复杂度可控、后人看得懂。

**工程指标**  
- 优化 ROI：核心指标。  
- 代码复杂度变化：副作用指标。  
- 回滚需求次数：风险指标。  
- 维护工时变化：长期成本指标。  
- 业务收益兑现率：结果指标。

**证据链**  
- 基线和收益评估  
- 代码复杂度分析  
- 业务影响说明  
- 优化前后对比数据

**低风险优化路径**  
1. 先确认问题真实存在  
2. 再做收益与复杂度评估  
3. 最后用最小方案验证

**易错点**  
- 为了技术美感优化  
- 没有量化收益  
- 小问题大改造

## 19. 大促前性能保障清单有哪些？

::: tip L2 深度训练
从容量合同、阶梯压测、故障冗余到恢复流量完整验收：[真实性压测与瓶颈证据](/deep-dives/capacity-performance/02-realistic-load-testing)
:::

**核心结论**  
大促前性能保障不是只做一轮压测，而是把容量、热点、降级、扩容、发布冻结和应急手册一起准备好。

**标准答案**  
我做大促前性能保障时，通常会检查几大块：容量模型是否更新、压测是否覆盖热点场景、缓存是否预热、数据库和 MQ 是否有安全余量、限流降级开关是否可用、线程池和连接池是否完成预算、扩容策略是否演练、发布窗口是否冻结、值班和应急手册是否就绪。之所以必须用清单，是因为大促风险很多，靠临场记忆不可靠。工程上真正稳的方式是提前一到两周做多轮验证，并留足整改窗口。面试里如果能补一句“性能保障清单的本质是把隐性风险显性化”，会很有经验感。

**高分回答**
我按“预测—验证—保护—响应—恢复”收口。预测侧统一活动漏斗、峰值倍率、热点商品/租户与不确定性，按 N-1 故障仍有余量计算应用、数据库、缓存、MQ、第三方限额和网络容量。验证侧用真实 mix、热点、冷缓存、突发与耐久阶梯压测，覆盖单可用区故障、下游变慢、重试和恢复流量，并校验订单、库存、支付与消息最终一致。保护侧完成热点预热、配额、分级限流降级、超时/重试预算、有界队列和自动扩容；所有开关都有 owner、影响说明、演练记录和回退。响应侧冻结非必要变更，确认主备值班、事故指挥、依赖联系人、看板、Runbook 和业务沟通模板。恢复侧特别验证缓存回填、积压消费、补偿对账和扩容回收，防止主峰过后被恢复风暴击穿。设置 go/no-go 门槛，例如目标峰值 1.5 倍下 P99/SLO 合格、N-1 仍不超安全水位、所有关键降级演练通过；未达标就缩小活动、提前扩容或取消高风险功能，而不是带病上线。

**追问点答案**  
- 为什么要提前冻结发布？  
因为高风险窗口引入新变更收益很低、风险很高。  
- 只压测不演练可以吗？  
不够。压测验证正常容量，演练验证依赖失败、人员权限、降级开关、扩容、沟通和恢复；大促事故往往发生在异常组合与人工处置环节。
- 缓存预热为什么重要？  
因为大促前几分钟常是最脆弱阶段。

**原理补充**  
大促性能保障本质上是把高峰下的容量风险、变更风险和操作风险一起收敛。

**架构扩展**  
成熟架构师做大促保障，不会只看 QPS，而是会从流量、资源、缓存、依赖、发布、值班、回滚、开关一整套视角来准备。这是典型架构岗位视角。

**工程指标**  
- 峰值承接能力：容量指标。  
- 缓存预热完成率：准备度指标。  
- 降级开关可用率：止损能力指标。  
- 线程池/连接池安全余量：资源指标。  
- 演练通过率：应急能力指标。

**证据链**  
- 大促保障清单  
- 压测报告  
- 预热与扩容记录  
- 演练和发布冻结记录

**低风险优化路径**  
1. 先列清单盘风险  
2. 再分模块验证和整改  
3. 最后冻结变更并演练应急

**易错点**  
- 只压测不演练  
- 不冻结发布  
- 没有缓存和扩容准备

## 20. 如何向管理层汇报性能项目成果？

**核心结论**  
向管理层汇报性能成果时，重点不是技术细节炫耀，而是清楚说明投入、收益、风险下降和后续计划。

**标准答案**  
我向管理层汇报性能项目时，通常会用业务语言讲四件事：第一，原来问题造成了什么损失，比如慢、超时、投诉、峰值风险；第二，这次做了哪些关键动作；第三，结果提升了多少，比如 TP99 降了多少、吞吐提了多少、机器成本省了多少、事故风险降了多少；第四，剩余风险和下一步计划是什么。之所以不能只讲技术细节，是因为管理层更关心投入产出和业务风险变化。工程上最好用前后对比图、极限容量变化、SLO 改善和成本变化来表达。面试里如果能补一句“性能项目汇报的核心是让技术收益变成业务可理解结果”，会很有管理视角。

**高分回答**
我用一页结论回答“为什么做、投入多少、结果是否可信、下一步要什么决策”。先给业务基线：核心交易 P99、超时用户、峰值缺口、容量成本和事故风险；再说明范围与投入，不把其他项目收益归到性能项目。结果同时展示同条件前后对照和业务翻译，例如峰值有效 TPS 从 8k 提到 12k、P99 从 1.2s 降到 650ms、错误率不变，按预测峰值可多留 30% N-1 余量并推迟半年扩容，年度节省约多少。说明测试环境、样本和灰度生产证据，以及一致性、错误率和单位成本约束，防止只报最好数字。然后透明列出仍未解决的单主瓶颈、第三方限额与估算置信区间，给出继续投入、接受风险或业务限流三种选择及成本。最后报告工程化成果，如回归门禁和容量看板，确保收益可持续；后续用真实高峰复核预测并更新 ROI。

**追问点答案**  
- 为什么要讲剩余风险？  
因为成熟汇报不是只报喜，还要让管理层知道后续投入方向。  
- 成果最适合用什么形式表达？  
用同一负载下的吞吐—P99—错误率曲线、容量安全余量和成本前后对照，再附一句业务影响；单个百分比容易缺少基线和约束。
- 只讲技术细节为什么效果差？  
因为管理层通常关注业务结果和投入产出。

**原理补充**  
性能项目汇报本质上是把技术改进翻译成业务收益和风险控制成果。

**架构扩展**  
成熟架构师向上汇报时，不会停留在“换了什么算法、调了什么参数”，而是会说“高峰能多扛多少、故障概率降多少、是否减少扩容投入”。这才是架构岗位的表达方式。

**工程指标**  
- TP99 降幅：体验收益指标。  
- 峰值吞吐提升：容量收益指标。  
- 资源成本变化：成本收益指标。  
- 事故风险下降估算：稳定性收益指标。  
- 后续待优化项数量：剩余风险指标。

**证据链**  
- 前后对比看板  
- 压测报告  
- 成本账单  
- 故障与投诉趋势数据

**低风险优化路径**  
1. 先讲问题和损失  
2. 再讲动作和数据结果  
3. 最后讲剩余风险与下一步计划

**易错点**  
- 只讲技术不讲业务  
- 没有前后对比  
- 汇报里不提剩余风险
