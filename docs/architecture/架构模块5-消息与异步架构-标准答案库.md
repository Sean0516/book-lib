## 1. 什么时候必须引入 MQ？

::: tip L2 深度训练
先判断是否跨聚合传播已发生事实，而不是为了“解耦”盲目引入：[聚合、不变量与领域事件](/deep-dives/business-evolution/02-aggregate-invariants)
:::

**核心结论**  
引入 MQ 的关键不是“系统大了就上”，而是当同步调用已经无法同时满足解耦、削峰、异步化和可靠性要求时，MQ 才真正有价值。

**标准答案**  
我通常会在几类场景明确考虑 MQ。第一类是主链路不适合同步串行做完，比如订单创建后还要通知库存、积分、消息、风控，这时候异步解耦收益很高；第二类是流量有明显峰谷，需要削峰填谷，比如秒杀、日志采集、埋点上报；第三类是需要跨系统可靠传递事件，而又不想让调用方和被调用方强耦合。之所以不能把 MQ 当默认组件，是因为它会引入顺序、重复、积压、可观测性和补偿复杂度。面试里如果能补一句“MQ 解决的是同步调用做不好或做不起的问题”，会比较成熟。

**高分回答**
我会先证明同步方案违反了明确约束，再引入 MQ。若订单接口 SLO 是 200ms，而积分、通知、搜索刷新合计 P99 600ms 且这些动作不影响订单受理，就把“订单已创建”作为已发生事实异步传播；若秒杀入口峰值 5 万 TPS、库存服务安全消费 8000 TPS，队列可削峰，但必须算出积压 `峰值差 × 持续时间` 和恢复时长，确认磁盘、保留期与业务窗口可承受。跨团队消费者不断增加、生产者不应知道所有下游时，事件也能降低时间耦合。反过来，本地可丢弃埋点可用线程池，单机可靠小任务可用任务表；需要立即返回结果、跨服务强不变量或消费者无法容忍延迟时，MQ 不是答案。设计 MQ 的同时必须定义事实源、至少一次语义、业务幂等键、顺序范围、重试/死信、积压降级、schema 演进和对账，否则只是把失败从调用栈移到后台。上线前演练 broker 不可用、消费者停机、重复乱序与峰值追赶，观察端到端延迟、backlog age、重复副作用和恢复时间。引入标准是异步缓冲/解耦收益大于新增的最终一致与运维成本。

**追问点答案**  
- 只有高并发才需要 MQ 吗？  
不一定，跨系统解耦和可靠事件传递也很常见。  
- 为什么不是所有异步都要上 MQ？  
可丢失的进程内工作可用有界线程池，小规模可靠任务可用数据库任务表；MQ 只有在跨实例缓冲、重放、多个消费者或独立扩缩容收益明确时才值得。
- 上 MQ 最大的代价是什么？  
一致性和链路可观测性复杂度会上升。

**原理补充**  
MQ 的本质，是在生产者和消费者之间增加一个可缓冲、可解耦、可重放的异步中介层。

**架构扩展**  
成熟架构师不会为了“架构好看”引入 MQ，而是会先问三个问题：同步做完值不值、失败是否必须可补、峰值是否需要被缓冲。只有这三类痛点足够强，MQ 才是正收益。

**工程指标**  
- 同步链路耗时：判断异步化价值。  
- 峰值流量抖动幅度：削峰需求指标。  
- 异步任务量级：评估 MQ 必要性。  
- 生产/消费失败率：上线后稳定性指标。  
- 补偿和人工介入次数：衡量复杂度成本。

**证据链**  
- 调用链 tracing  
- 流量曲线  
- 下游调用耗时分布  
- 失败和重试日志

**低风险优化路径**  
1. 先确认异步化目标  
2. 再评估 MQ 是否比本地异步更值  
3. 最后补一致性和可观测性设计

**易错点**  
- 系统一大就默认上 MQ  
- 只看削峰，不看治理成本  
- 把所有异步问题都丢给 MQ

## 2. 如何做 MQ 选型？

**核心结论**  
MQ 选型的关键不是比较谁“更强”，而是看你的业务到底更看重吞吐、顺序、事务、延迟、生态还是运维复杂度。

**标准答案**  
做 MQ 选型时，我通常先列需求，不会一上来报中间件名字。比如要不要严格顺序、要不要事务消息、峰值吞吐多大、延迟要求多严、消息积压多久可接受、团队有没有现成运维经验。之所以不同 MQ 差异大，是因为有的偏吞吐，有的偏事务和业务语义，有的偏生态成熟和云托管能力。工程上比较稳的做法是先明确核心链路需求，再用 2 到 3 个最重要维度筛选，而不是试图选一个“全能王”。面试里如果能补一句“消息队列选型本质上是业务语义和运维能力的匹配”，会比较完整。

**高分回答**
我先把需求变成可测试矩阵：峰值/持续吞吐与平均消息大小、端到端 P99、积压保留量与追赶 RTO、顺序单位、投递语义、延时/事务需求、多租户隔离、跨区域 RPO/RTO、合规、单位消息成本和团队 on-call 能力。日志流更重高吞吐、长保留和流处理生态；订单事件可能更看重按订单有序、延时与事务/outbox 集成；但不会仅凭产品标签选型。候选方案用生产分布 POC：正常写读、消费者停机形成目标积压、单 broker/可用区故障、扩分区与 rebalance、消息大小长尾、磁盘接近水位和版本升级，测生产确认、消费延迟、恢复吞吐、数据损失与运维动作。还要验证客户端生态、schema 管理、权限审计、容量扩展、托管服务配额与退出成本。通常优先采用组织已有且能满足关键约束的平台，避免为了边缘功能引入第二套 MQ；缺少能力可由调度/事务层补，但要计算复杂度。最终决策记录未选方案、剩余风险和触发换型的阈值，并以单位消息成本、事故数和演练 RTO 持续复核，而不是拿实验室最高吞吐做唯一结论。

**追问点答案**  
- 为什么吞吐不是唯一指标？  
因为顺序、可靠性、事务能力和运维成本同样关键。  
- 团队经验为什么影响选型？  
因为再强的中间件，如果团队不会运维，也会变成风险。  
- 云上托管服务值得考虑吗？  
值得，但要实测配额、跨区能力、故障语义、可观测性、成本和数据导出/迁移路径；托管减少日常运维，不会替你解决消息幂等与业务一致性。

**原理补充**  
MQ 选型本质上是在“消息语义、性能特征和组织能力”之间找平衡点。

**架构扩展**  
成熟架构师不会把选型讨论成“Kafka vs RocketMQ 谁更好”，而是会先把场景拆清：日志平台、订单事件、支付通知、延迟消息、IoT 上报，需求根本不是一类。

**工程指标**  
- 峰值吞吐：基础性能指标。  
- 端到端延迟：实时性指标。  
- 顺序保证覆盖率：语义指标。  
- 运维事件数：平台稳定性指标。  
- 单位消息成本：长期成本指标。

**证据链**  
- 业务需求清单  
- POC 压测数据  
- 运维能力评估  
- 历史故障和使用经验

**低风险优化路径**  
1. 先列核心需求排序  
2. 再做小范围 POC  
3. 最后结合团队能力定型

**易错点**  
- 只按吞吐选型  
- 忽略事务和顺序语义  
- 不考虑运维能力

## 3. 如何保证消息不丢？

**核心结论**  
“消息不丢”不是单个环节的承诺，而是生产、存储、消费、确认和补偿整条链路共同达成的结果。

**标准答案**  
我理解消息不丢，一般会把链路拆成三段来看。生产侧要确保发送结果可确认，必要时本地落库或事务消息；Broker 侧要有持久化、副本和刷盘策略；消费侧则要做到处理成功后再提交消费位点，并对失败消息保留重试和死信路径。之所以要这样拆，是因为很多团队说“不丢消息”，其实只是某一层没问题，但其他层仍然可能丢。工程上真正稳的方案通常还会加对账和补发机制，因为理论上再强的 MQ 也不能替代业务核对。面试里如果能补一句“消息不丢是系统设计结果，不是中间件单独能力”，会很加分。

**高分回答**
我会沿“业务事实—生产—broker—消费副作用”逐段建立可恢复落点。生产者不能先提交数据库再裸发消息，否则宕机会留下事实有了、事件没有；使用同一事务写 outbox，发布器至少一次发送，收到满足副本/持久化要求的 broker ack 后才标记已发，超时状态未知时用同一消息 ID 重试。Broker 配置持久化、多副本与最小同步副本，明确异步刷盘可能丢多少数据的 RPO，并监控 ISR/磁盘水位。消费者先以业务键在本地事务中完成幂等副作用和 inbox/状态记录，再提交 offset；处理失败不提交，有限重试后进入有 owner 和 SLA 的死信。进程在处理完成但提交前崩溃会重复，所以追求的是“不丢且副作用一次”，不是幻想端到端 exactly-once。独立对账从数据库事实源计算应有事件/结果，发现永久漏项后可重放或补偿。演练生产提交点宕机、ack 丢失、broker 失副本、消费完成前后崩溃与死信积压，观察 outbox age、未同步副本、消费 lag、差异数和补发成功率；只有业务结果可核验，才有资格说消息不丢。

**追问点答案**  
- 生产端 ack 成功就一定不丢吗？  
不一定；ack 的含义取决于刷盘与同步副本配置，生产者还要处理 ack 丢失导致的结果未知，并用稳定消息 ID 幂等重发。
- 消费成功后为什么再提交位点？  
避免还没真正处理成功就把消息标记成已消费。  
- 为什么还需要对账？  
因为极端情况下业务层仍然需要最终核验。

**原理补充**  
消息不丢的本质，是在每个状态转换点都给消息一个“可确认、可恢复、可补偿”的落点。

**架构扩展**  
支付通知、订单状态变更、库存事件、积分流水这些链路最怕消息丢。成熟架构师不会只说“开持久化”，而是会继续讲消费提交时机、失败补发和业务核对。

**工程指标**  
- 发送失败率：生产侧稳定性。  
- broker 持久化延迟：存储链路健康度。  
- 消费确认成功率：消费侧质量指标。  
- 死信积压量：尾部风险指标。  
- 对账差异数：最终结果指标。

**证据链**  
- 发送确认日志  
- broker 持久化和副本状态  
- 消费提交记录  
- 对账和补发记录

**低风险优化路径**  
1. 先拆生产、存储、消费三个环节  
2. 再分别补确认和持久化  
3. 最后加对账与补发

**易错点**  
- 把“不丢”理解成 broker 配置问题  
- 提交位点时机错误  
- 没有业务层对账

## 4. 如何处理重复消费？

**核心结论**  
重复消费是常态假设，不是异常假设，所以消费端必须默认自己会收到重复消息。

**标准答案**  
处理重复消费时，我一般不会先想怎么完全杜绝，而是优先做消费幂等。最常见的方法是给消费动作绑定业务唯一键，用唯一约束、去重表、状态机或版本校验保证同一条业务效果只生效一次。之所以重复消费这么常见，是因为消息重投、消费确认失败、消费者重启和 rebalance 都可能导致同一条消息再次被看到。工程上如果消费逻辑有多步副作用，还要把“哪一步已经做过”记录清楚，避免局部重复。面试里如果能补一句“重复消费问题最终是业务副作用去重问题”，会更准确。

**高分回答**
我默认至少一次投递会重复，并定义业务幂等键而非只用 broker messageId，例如发券用 `campaignId + userId + benefitType`，即使上游重新生成消息 ID 也不会发两次。单库副作用在一个事务中插入 inbox/幂等记录并执行业务写，唯一约束处理并发竞争；若重复到达则校验消息摘要和版本，返回已完成结果。状态型事件使用 `aggregateId + version` 条件更新，只接受期望的下一版本，旧事件忽略、缺版本进入等待或重放。跨支付、短信等外部副作用要把同一业务键传给对方，超时先查询结果；本地“标记已处理”与外部调用无法原子时，用显式状态机记录处理中、结果未知和补偿，不能提前提交 offset。去重记录 TTL 覆盖消息保留、灾备回放和人工补发周期，过期后仍应有账本唯一约束。测试消费完成后提交前崩溃、rebalance、并发重复、乱序和历史重放，验收重复副作用为零，并观察幂等冲突、处理中超时和人工修复。消息去重只是入口，真正目标是每个业务副作用都能识别并安全复用既有结果。

**追问点答案**  
- 为什么 MQ 很难彻底保证不重复？  
因为可靠性语义往往更偏向至少一次。  
- 幂等表和唯一索引怎么选？  
副作用与业务表同库且有天然唯一键时优先唯一约束；需要保存摘要、处理中状态、多步结果和审计时用幂等/inbox 表，并与本地业务写同事务提交。
- 读操作也要防重吗？  
通常不用，重点是有副作用的写操作。

**原理补充**  
重复消费的核心不是消息被看两次，而是同一业务副作用是否会被执行两次。

**架构扩展**  
发券、扣库存、积分发放、账单记账都特别怕重复消费。成熟架构师设计消息消费时，默认前提是“这条消息随时可能再来一次”。

**工程指标**  
- 重复消费命中率：问题频度指标。  
- 幂等冲突数：防重执行情况。  
- 重复副作用事件数：最终质量指标。  
- 消费重试次数：间接风险指标。  
- 人工修复次数：治理成熟度指标。

**证据链**  
- 消息唯一标识  
- 幂等表或唯一约束记录  
- 消费日志  
- 业务状态流转记录

**低风险优化路径**  
1. 先定义业务唯一键  
2. 再把消费逻辑做幂等  
3. 最后补监控和审计

**易错点**  
- 试图彻底消灭重复，而不做幂等  
- 只做技术防重，不做业务防重  
- 多步副作用没有状态记录

## 5. 顺序消息怎么保证？

**核心结论**  
顺序消息的关键不是全局有序，而是先明确业务到底需要“全局顺序”还是“局部键顺序”。

**标准答案**  
我做顺序消息设计时，通常会先问顺序要求到底落在哪个维度。如果只是同一个订单、同一个用户、同一个账户内要求顺序，那一般通过同一分区、同一队列或同一 shard key 路由就够了；如果要求全局顺序，系统吞吐和可用性成本会非常高。之所以这点重要，是因为很多业务其实只需要局部顺序，却误以为自己要全局有序。工程上除了生产侧同键路由，还要保证消费侧单分区串行处理，避免并发消费把顺序打乱。面试里如果能补一句“顺序保证的前提是先收缩顺序范围”，会很成熟。

**高分回答**
我先把顺序收缩为业务聚合，例如同一订单的 `CREATED→PAID→SHIPPED` 必须有序，不同订单可并行。生产者用稳定 `orderId` 作为 partition key，并由单一事实源/outbox 按聚合版本发布，避免多个生产者各自时钟决定顺序；事件包含 `aggregateVersion`。消费者一个分区内按序拉取，但可在不同分区并行，业务更新使用版本条件，只接受当前版本的下一条。失败处理是关键：若版本 5 失败，不能把版本 6 提交后继续并假装有序；可暂停该 key/分区重试、将该 key 路由到有序重试通道，或在可证明独立时隔离，均需权衡一个坏 key 对整个分区的阻塞。扩分区会改变 key 映射，迁移期间要固定路由版本或排空旧分区，不能直接增加分区数。全局顺序意味着单分区/单写者和更低可用吞吐，除非业务不变量确实要求，否则不采用。演练生产重试、乱序到达、消费者崩溃、毒消息和扩分区，监控版本缺口/倒退、分区倾斜、阻塞年龄与吞吐；顺序保证是生产、broker 路由、消费与业务版本共同形成的端到端约束。

**追问点答案**  
- 为什么全局顺序代价高？  
因为会显著限制并行度和吞吐。  
- 顺序消息和并发消费冲突吗？  
在同一个顺序单元上通常冲突，需要串行处理。  
- 重试会影响顺序吗？  
会；前一事件失败时若后续先提交就会越序。需按 key/分区阻塞或进入保持版本的有序重试通道，并监控毒消息造成的队头阻塞。

**原理补充**  
顺序消息本质上是在并行吞吐和事件先后关系之间做约束性取舍。

**架构扩展**  
订单状态流转、账户流水、物流轨迹、聊天消息常需要局部顺序。成熟架构师不会轻易承诺“顺序消息”，而是会先和业务确认顺序边界和失败策略。

**工程指标**  
- 顺序违规事件数：核心质量指标。  
- 单分区消费延迟：顺序代价体现。  
- 分区热点度：影响吞吐和扩展。  
- 顺序消息重试阻塞时长：风险指标。  
- 吞吐下降比例：顺序保障成本。

**证据链**  
- 路由 key 设计  
- 分区映射规则  
- 消费顺序日志  
- 顺序异常复盘记录

**低风险优化路径**  
1. 先收缩顺序范围到业务局部  
2. 再设计同键同分区  
3. 最后处理失败阻塞和回退

**易错点**  
- 默认要求全局顺序  
- 生产侧和消费侧只保证一头  
- 不考虑失败重试打乱顺序

## 6. 死信队列怎么使用？

**核心结论**  
死信队列的目标不是“把坏消息丢到别处”，而是把无法正常处理的尾部问题从主消费链路里隔离出来并可追踪处理。

**标准答案**  
我通常会在消息多次重试后仍然失败、业务校验无法通过、下游长期不可用或者消息格式异常时，把它送入死信队列。之所以死信队列重要，是因为它能避免少量坏消息长期阻塞正常消费，同时保留后续人工或自动补救的机会。工程上死信消息不能只堆着不管，还要给它定义分类、告警、重放、丢弃条件和人工处理规范。面试里如果能补一句“死信队列是主链路稳定性和尾部恢复能力之间的隔离带”，会比较有架构感。

**高分回答**
我把死信定义为有 owner、SLA 和退出条件的异常工单，而不是垃圾桶。消费者先按错误分类：网络抖动等瞬时错误有界退避；schema 不兼容、缺字段等确定性错误立即隔离；下游长期故障在重试预算耗尽后进入死信，避免阻塞主分区。死信记录原消息、topic/partition/offset、业务键、schema 版本、错误码、尝试轨迹、消费者版本和 trace，敏感字段按权限脱敏；同时保留原始 payload 或可追溯对象，防止排障证据丢失。平台按业务等级告警最老年龄、增长率和金额/订单影响，自动关联 owner。重放前必须修复根因并在沙箱或影子模式验证，按业务键幂等、限速、可暂停，记录新旧消息映射，防止一次全量重放再次压垮下游；永久非法或合规到期消息经审批后才可丢弃。对于有顺序要求的 key，后续消息需等待或进入同一隔离通道，不能绕过死信破坏状态机。定期演练坏 schema、下游长故障和重放中断，指标看死信进入率、P95 滞留、自动/人工修复率、重放成功率与重复副作用。死信闭环的完成标准是业务状态已恢复并有审计证据。

**追问点答案**  
- 死信能不能自动重放？  
只有根因已修复、消息通过预检且消费者幂等时才自动限速重放；确定性坏数据或顺序敏感消息需人工/专用流程确认。
- 死信消息要保存多久？  
取决于业务价值和审计要求。  
- 为什么不能无限重试代替死信？  
因为会阻塞主链路并浪费资源。

**原理补充**  
死信队列的本质，是把异常消息从吞吐路径中剥离到治理路径。

**架构扩展**  
成熟团队的死信体系不会停留在“有个队列”，而是会继续建设死信分类、重放平台、告警面板和人工工单闭环。否则死信只是把问题藏起来。

**工程指标**  
- 死信消息量：问题规模指标。  
- 死信增长速度：故障爆发信号。  
- 死信处理完成率：治理效果指标。  
- 自动重放成功率：恢复能力。  
- 死信滞留时长：尾部风险指标。

**证据链**  
- 重试失败日志  
- 死信队列积压  
- 重放记录  
- 人工处理工单

**低风险优化路径**  
1. 先定义进入死信的条件  
2. 再给死信配告警和处理规则  
3. 最后建设重放和人工闭环

**易错点**  
- 有死信但没人处理  
- 不区分失败类型  
- 把死信当无限垃圾桶

## 7. 延时消息典型场景有哪些？

**核心结论**  
延时消息适合那些“必须在未来某个时间点触发且不适合同步阻塞等待”的业务动作。

**标准答案**  
我通常会在订单超时关闭、优惠券延迟生效、支付超时检查、定时提醒、延迟重试和状态超时兜底这类场景考虑延时消息。之所以用延时消息而不是线程睡眠或本地定时器，是因为这些任务通常需要跨实例可靠执行，且服务重启后不能丢。工程上除了发送延时消息本身，还要考虑取消、幂等执行和时间精度要求，因为很多业务并不是“到了就一定做”，而是“到了再确认是否还需要做”。面试里如果能补一句“延时消息本质上是把未来动作持久化成一条可调度事件”，会比较准确。

**高分回答**
我先明确需求是“最早何时可执行”还是“必须准点执行”。订单 30 分钟未支付后关闭通常允许秒级误差，适合持久化延时消息；金融清算的严格时间窗或海量可取消任务可能更适合专门调度系统和时间轮/任务表。事件包含业务键、期望触发时间、创建版本和动作类型，broker 持久化后即使服务重启也不会丢。到期消费者不能直接执行，而要查询权威状态并用条件更新，例如只有 `UNPAID` 且版本匹配才能关闭，用户已支付则消息自然失效；因此取消不必物理删除消息，也能通过状态机安全处理。重复投递用业务键幂等，时钟偏差、broker 堆积和消费暂停可能让消息晚到，业务需定义最大迟到、补扫与告警；早到则按事件时间重新调度，不能提前执行。大量同一时刻到期要分桶、加抖动和限流，防止定时洪峰。演练重启、时钟偏差、消息晚到、重复与支付/关闭并发，监控触发偏差 P99、积压最老年龄、状态校验拒绝率、重复副作用和漏执行对账。延时消息保证未来动作可恢复，不保证业务状态在等待期保持不变。

**追问点答案**  
- 延时消息和定时任务有什么区别？  
延时消息更适合跟具体业务事件绑定，定时任务更适合周期扫描。  
- 为什么还要做执行前校验？  
因为业务状态可能在等待期间已经变化。  
- 延时精度要求很高怎么办？  
先量化允许的提前/迟到分位数；若 MQ 的时间粒度、积压和故障恢复无法满足，应使用专门调度系统，并仍保留幂等、状态校验和漏执行对账。

**原理补充**  
延时消息的核心，是把“未来触发条件”放进可靠事件系统里，而不是依赖进程内存状态。

**架构扩展**  
成熟架构师会区分“业务延时事件”和“平台定时任务”。前者更适合延时消息，后者常适合调度平台或扫描任务，不能混用。

**工程指标**  
- 触发准点率：延时能力指标。  
- 延时消息积压量：风险指标。  
- 取消率：说明是否需要执行前校验。  
- 执行成功率：业务结果指标。  
- 超时误触发数：质量风险指标。

**证据链**  
- 延时消息发送记录  
- 触发日志  
- 状态校验记录  
- 超时处理复盘

**低风险优化路径**  
1. 先确认是单事件延时而非周期任务  
2. 再设计取消和幂等  
3. 最后验证时间精度和积压能力

**易错点**  
- 用本地定时器替代可靠延时  
- 不做触发前状态校验  
- 把周期任务和延时事件混为一谈

## 8. 如何处理消息积压？

**核心结论**  
处理消息积压的关键不是单纯提速消费，而是先分清积压原因，再决定扩容、降级、跳过还是削峰。

**标准答案**  
面对消息积压时，我通常会先判断是生产流量暴涨、消费能力不足、消费者故障、顺序阻塞、下游依赖变慢，还是某类坏消息卡住了队列。之所以不能上来就扩消费者，是因为如果问题在下游或热点分区，盲目扩容不一定有效。工程上较稳的做法是先止血，比如限流生产侧、隔离坏消息、跳过低优先级任务，再按分区扩容、消费优化、批量消费或异步降级逐步追平。面试里如果能补一句“积压问题的核心不是 backlog 条数，而是业务允许多久恢复”，会比较成熟。

**高分回答**
我先把 backlog 换算成业务时间与恢复能力：积压 3600 万条、稳定消费 1 万条/秒、生产仍为 6000 条/秒，净追赶只有 4000 条/秒，至少 2.5 小时才能清空；若业务 SLA 是 30 分钟，就必须先降生产或提高有效消费。诊断按分区比较生产率、消费率、最老消息年龄、处理 P99 和下游水位，区分流量突增、热点分区、毒消息、消费者故障、rebalance 或数据库变慢。止血优先保护下游：限制非核心生产、将不同优先级/租户隔离，暂停会产生次生消息的功能，毒消息进死信；不能盲目加消费者把数据库打垮。恢复时只有分区足够且下游有余量才水平扩容，同时优化批量拉取/提交、连接复用和非必要同步调用；追赶流量设上限，核心在线请求保留资源。若消息超过业务有效期，按规则压缩、跳过或重建状态，但必须审批和审计，不能随意丢。每阶段计算新的 ETA，清空后逐步恢复生产并检查重试风暴与数据对账。演练消费者全停、单分区热点和下游限流，观察 age、净消费率、恢复 ETA、下游 SLO、丢弃/死信与最终差异。

**追问点答案**  
- backlog 多大算危险？  
用最老消息年龄和 `积压量 ÷（消费率−生产率）` 估算恢复时间；当净消费率不为正或 ETA 超过业务一致性 SLA 时即危险。
- 为什么扩消费者不一定有效？  
可能受限于分区数、顺序约束或下游瓶颈。  
- 低优先级消息为什么要隔离？  
因为高峰时核心消息应优先恢复。

**原理补充**  
消息积压本质上是生产与消费的供需失衡在时间维度上的累积。

**架构扩展**  
成熟架构师面对积压，不会只问“堆了多少”，而会继续追问“哪些业务受影响、多久必须追平、是否要牺牲部分低价值消息保主链路”。这才是架构视角。

**工程指标**  
- backlog 条数：基础规模。  
- 消费延迟：业务感知核心指标。  
- 恢复时间估算：决策指标。  
- 分区负载均衡度：找热点瓶颈。  
- 低优先级消息占比：降级空间指标。

**证据链**  
- 生产/消费 TPS 曲线  
- backlog 监控  
- 分区消费情况  
- 下游依赖性能数据

**低风险优化路径**  
1. 先止血和分因  
2. 再做扩容或隔离  
3. 最后优化消费模型和恢复速度

**易错点**  
- 不分析原因直接扩容  
- 不看业务恢复时限  
- 高低优先级消息混在一起恢复

## 9. 消费者重平衡会带来什么问题？

**核心结论**  
重平衡本身不是问题，问题在于它会打断消费节奏、触发重复消费，并放大局部抖动。

**标准答案**  
消费者重平衡时，我通常会关注几个风险：分区短暂停止消费、位点提交不一致导致重复消费、热点分区重新分配后消费抖动，以及在高峰期频繁 rebalance 让整个消费组变得很不稳定。之所以它重要，是因为很多表面上的“消费偶发变慢”其实都和 rebalance 有关。工程上通常要通过减少无意义实例频繁上下线、优化消费时间、控制心跳和 session 配置、把慢消费者隔离等方式降低重平衡频率。面试里如果能补一句“重平衡是消费组的组织动作，但业务会感知到它的副作用”，会比较完整。

**高分回答**
重平衡改变分区所有权，风险发生在“旧消费者停止—提交已完成位点—新消费者接管”这个交接窗口。旧实例若处理完成但尚未提交，新实例会重复；若先提交再完成副作用则会丢，因此仍应先以业务幂等完成处理，再提交 offset。长批处理超过 poll/session 限制、实例频繁弹性伸缩、GC 暂停和网络抖动都会触发全组停顿；我会拆小批次、把耗时处理与 poll 心跳解耦、设置符合 P99 处理时长的超时，并采用 cooperative/sticky 分配或静态成员资格减少无关分区迁移。优雅下线先停止拉取、等待在途任务和同步提交，再离组；扩缩容设置稳定窗口，不能跟 CPU 瞬时变化。分区 revoke 回调只提交已完成的连续位点，不能跨过尚未完成消息。热点分区仍不会因增加消费者变快，需要改 key 或隔离。演练实例 kill、长 GC、慢消息和滚动发布，监控 rebalance 次数/耗时、停顿产生的 lag、重复率、分区迁移量和处理超时。目标不是消灭重平衡，而是让所有权变化可预期、局部化且不破坏业务结果。

**追问点答案**  
- 为什么重平衡会导致重复消费？  
因为分区交接时位点和处理状态可能没有完全同步。  
- 为什么慢消费者更容易触发问题？  
因为它会影响心跳和分配稳定性。  
- 如何减少重平衡影响？  
再配合 cooperative/sticky 分配、静态成员、优雅下线和扩缩容稳定窗口，让分区渐进迁移；业务端仍必须幂等承受重复。

**原理补充**  
重平衡的本质，是消费组内部重新分配分区所有权，而所有权变化天然伴随短暂不稳定。

**架构扩展**  
成熟架构师在设计消费组时，会把重平衡成本算进去，而不是只看“多加几个消费者就能更快”。稳定性往往比理论吞吐更重要。

**工程指标**  
- 重平衡次数：核心健康指标。  
- 重平衡耗时：对消费停顿的影响。  
- 重复消费率：副作用风险。  
- 分区重新分配抖动：吞吐波动指标。  
- 慢消费者数量：潜在诱因指标。

**证据链**  
- rebalance 日志  
- 位点提交记录  
- 消费耗时分布  
- 消费组实例变化记录

**低风险优化路径**  
1. 先找 rebalance 诱因  
2. 再稳定实例和消费时长  
3. 最后补幂等和位点治理

**易错点**  
- 只把它当中间件内部行为  
- 忽略业务重复消费影响  
- 实例频繁伸缩导致抖动

## 10. 消费幂等怎么落地？

**核心结论**  
消费幂等的落地点不在消息系统，而在业务副作用真正发生的地方。

**标准答案**  
做消费幂等时，我通常会先定位副作用发生在哪，比如数据库插入、状态推进、外部调用或积分发放，然后用业务唯一键、去重表、唯一索引、版本号或状态机把这一步保护住。之所以强调副作用落点，是因为很多团队只在消费框架层做判断，真正写库或调用外部系统时仍然会重复执行。工程上如果消费逻辑跨多步，就要记录阶段状态，让系统知道“已经做到哪一步”，否则失败重来会很危险。面试里如果能补一句“幂等不是框架特性，而是业务状态设计”，会很有说服力。

**高分回答**
我从副作用的不变量落地。发积分用 `sourceType + sourceId + accountId` 作为业务唯一键，在同一数据库事务中插入积分流水、更新余额并记录 inbox；唯一冲突时校验请求摘要并返回既有结果，这比单独缓存 messageId 更可靠。状态事件携带聚合版本，使用 `UPDATE ... WHERE version = expected` 单向推进，旧版本忽略、缺版本等待补齐，防止重复和乱序。多步流程持久化 `RECEIVED→PROCESSING→EXTERNAL_UNKNOWN→DONE/COMPENSATING`，每步幂等且可恢复；调用外部支付或发券时传递同一业务键，超时先查询对方结果，不能本地标记完成后盲目重复。offset 只在本地事务结果持久化后提交，提交失败导致重投由幂等吸收。去重记录保留周期覆盖消息保留、灾备重放和人工补发；高价值账本的唯一约束长期保留。用并发重复、消费者提交前崩溃、外部响应丢失、乱序和历史回放验证，指标要求重复副作用为零，并观察幂等冲突、处理中超时、版本缺口和人工补偿。幂等成功的证据是业务状态与审计结果一致，而不是框架日志写着“已去重”。

**追问点答案**  
- 为什么消息 ID 不总是够用？  
同一业务动作可能因重发生成不同消息 ID，或一个消息触发多个副作用；必须用稳定业务键和每个副作用的不变量防重。
- 多步副作用怎么幂等？  
通常要把每步状态持久化并按状态推进。  
- 外部接口也要幂等吗？  
如果会重复调用，最好要有幂等协议或补偿策略。

**原理补充**  
消费幂等的本质，是让重复触发映射成同一业务状态，而不是重复副作用。

**架构扩展**  
发券、扣库存、账务记账、风控事件入库都属于“副作用很贵”的消费场景。成熟架构师在设计消费端时，会从业务状态机出发，而不是只靠技术拦截。

**工程指标**  
- 幂等命中率：观察重复触发频度。  
- 重复副作用事件数：核心质量指标。  
- 状态回滚次数：说明流程设计是否不稳。  
- 外部接口重复调用率：跨系统风险指标。  
- 人工修复数：治理成熟度指标。

**证据链**  
- 业务唯一键定义  
- 幂等表或状态记录  
- 副作用执行日志  
- 外部调用审计

**低风险优化路径**  
1. 先找副作用落点  
2. 再围绕业务唯一键做保护  
3. 最后补多步状态推进和审计

**易错点**  
- 只在框架层判断  
- 多步副作用无状态记录  
- 不把外部调用纳入幂等范围

## 11. 事务消息怎么用？

::: tip L2 深度训练
让业务状态与待发布领域事实进入同一本地事务：[聚合、不变量与领域事件](/deep-dives/business-evolution/02-aggregate-invariants)
:::

**核心结论**  
事务消息最适合用在“本地事务成功后必须可靠地触发异步动作”的关键链路上。

**标准答案**  
使用事务消息时，我一般会先确认业务是不是满足两个条件：本地数据更新必须成功，后续异步动作又必须可靠触发；同时这条链路又不值得做跨服务强事务。典型场景像订单创建后发库存事件、支付成功后发账务和通知事件。之所以事务消息适合这里，是因为它能把“本地事务结果”和“消息是否可继续投递”绑定起来，降低 DB 成功但消息丢失的风险。工程上还要配合消息回查、消费幂等和对账机制，否则只解决了生产端局部问题。面试里如果能补一句“事务消息是同步强事务和普通异步通知之间的折中”，会比较完整。

**高分回答**
我只在“本地事实一旦成立，事件必须可靠传播，但下游允许最终一致”的链路使用事务消息。实现有两类：中间件半消息先进入 prepare，本地事务提交后 confirm，状态未知由 broker 回查权威业务表；或本地事务同时写 outbox，发布器再投递。无论哪种，事件都携带稳定业务 ID、聚合版本和 schema 版本，生产状态机可区分待确认、已提交、已回滚和未知，回查不能依赖内存。回查接口必须幂等、限流并能在历史保留期内回答，broker/发布器反复查询不会改变业务。消费者仍按至少一次设计，用业务键与版本幂等，失败进入重试/死信；订单已创建到库存已处理的收敛由端到端 SLA 和对账证明。事务消息不适合跨服务立即原子可见，也不值得用于可丢弃通知；它只闭合“本地提交—消息资格”这个缺口。演练本地提交前后宕机、confirm 丢失、回查超时、重复投递和消费者停机，监控未知/未决消息年龄、回查率、投递延迟、消费收敛和差异。最终必须能从本地事实重建事件，而不能把 broker 状态当成唯一真相。

**追问点答案**  
- 事务消息能解决最终一致所有问题吗？  
不能；它只可靠关联本地事实与发送资格，消费幂等、乱序、下游副作用、补偿和业务对账仍需单独设计。
- 为什么还要消息回查？  
因为生产端在临界阶段可能状态不确定。  
- 什么时候不值得用事务消息？  
低价值通知或简单链路，普通可靠消息就够了。

**原理补充**  
事务消息的核心，是让消息发送资格依附于本地事务提交结果，而不是让两边真正强一致提交。

**架构扩展**  
成熟架构师在用事务消息时，不会把它神化成“分布式事务替代品”，而是会清楚界定它适合最终一致场景，且只覆盖生产端的一部分复杂度。

**工程指标**  
- 事务消息提交成功率：生产侧健康度。  
- 回查次数：不确定状态频度。  
- 未决消息数：高风险尾部问题。  
- 消费收敛时长：业务感知指标。  
- 对账差异率：最终质量指标。

**证据链**  
- 本地事务日志  
- 事务消息状态记录  
- 回查结果  
- 消费和对账数据

**低风险优化路径**  
1. 先确认适用场景  
2. 再设计回查和幂等  
3. 最后补对账和人工兜底

**易错点**  
- 过度使用事务消息  
- 不做回查  
- 只看生产端不看消费收敛

## 12. 如何避免消息风暴？

**核心结论**  
消息风暴的本质是某个事件被系统层层放大，治理重点是限制放大链、控制 fan-out 和约束重试。

**标准答案**  
避免消息风暴时，我通常先识别是消息生产过多、消费者回写又触发新事件、重试放大，还是一个事件被过度 fan-out 到太多下游。之所以风暴危险，是因为它会把系统从“处理业务事件”变成“处理自己制造的事件洪水”。工程上常见治理手段包括事件分级、限制 fan-out、幂等防抖、合并同类事件、降低低优先级事件频率、重试限次和熔断下游。面试里如果能补一句“消息风暴很多时候不是流量大，而是事件传播模型设计错了”，会比较有深度。

**高分回答**
我先画事件传播图并为每条边标出生产率、fan-out、重试和是否回写上游，计算单位业务动作的消息放大系数。例如一次商品更新触发 8 个消费者，其中 3 个又各发 4 条事件，理论放大已超过 20 倍；故障时每层重试 3 次会进一步乘法增长。设计上区分领域事实和派生通知，禁止消费者为“处理完成”无意义地产生新领域事件；为事件携带 causationId、correlationId 和 hop count，检测环路与异常深度。可覆盖的状态变更按业务键合并，只保留最新版本；低价值遥测采样/批量，fan-out 通过订阅准入、配额和优先级控制。重试进入独立延迟通道，受全局 token budget 和熔断约束，不能立即回灌主 topic；消费者幂等避免重复继续产新事件。平台监控生产/业务动作比、单事件后代数、重试占比、循环因果和 topic 增长斜率，越界自动暂停非核心生产者。演练一个消费者持续失败、事件环和大规模回放，验证 broker、下游安全水位及恢复策略。风暴治理重点是控制事件拓扑的增益，而不是等 broker 满了再扩容。

**追问点答案**  
- fan-out 为什么危险？  
fan-out 本身可合理，但与多跳派生和重试叠加会形成乘法放大；需要量化单位业务动作的总后代消息数并限制环路与深度。
- 合并事件什么时候有效？  
当多个短时间内连续事件业务上可被折叠时。  
- 重试为什么会助推风暴？  
因为它会让同一批失败事件不断回灌系统。

**原理补充**  
消息风暴本质上是事件传播图在错误设计下形成了高倍数放大。

**架构扩展**  
成熟架构师面对事件驱动系统，不会只看“解耦漂亮”，还会继续追问：一个事件最多扩散到几跳、多少个消费者、失败时会不会反向再产消息。这才是真正控制复杂度。

**工程指标**  
- 单事件 fan-out 倍数：风暴风险核心指标。  
- 单位业务动作产生消息数：放大强度指标。  
- 重试消息占比：次生风险指标。  
- 下游消费者错误率：传播后果指标。  
- 事件合并率：治理效果指标。

**证据链**  
- 事件拓扑图  
- 生产/消费量对照  
- 重试记录
- fan-out 配置和日志

**低风险优化路径**  
1. 先找到放大源  
2. 再限 fan-out 和重试  
3. 最后把可折叠事件做聚合

**易错点**  
- 只看 producer 不看传播链  
- 一个事件挂太多下游  
- 不限制重试回灌

## 13. 生产端批量发送利弊是什么？

**核心结论**  
批量发送的价值在于提升吞吐和降低固定开销，但代价是错误定位、重试粒度和时延控制会变复杂。

**标准答案**  
我通常在高吞吐、低单条价值、允许小范围延迟的场景考虑批量发送，比如埋点、日志、监控指标、批量业务事件。之所以批量发送有效，是因为它能摊薄网络往返、序列化和 broker 写入的固定成本。代价是某一批里如果部分失败，重试和排查都会更复杂，且单条消息的可见延迟也会被批次聚合时间拉长。工程上更稳的做法是按消息类型分批、控制批大小和等待时间上限，而不是一味追求大 batch。面试里如果能补一句“批量发送是在吞吐、时延和故障粒度之间换取平衡”，会比较完整。

**高分回答**
我会把 batch 作为同时受条数、字节和等待时间约束的自适应窗口，而不是只设“1000 条”。批量能摊薄网络往返、压缩和 broker fsync，但第一条消息必须等待凑批，单批过大还会占内存、超过 broker 最大消息、造成长 GC，并让失败重试放大。参数从 SLO 反推：若端到端 P99 需 200ms，可只给聚合 10～20ms；同时根据平均/P99 消息大小限制批字节，并在低流量时由时间上限及时发送。不同 topic、分区和优先级不混批，有顺序要求的同 key 维持顺序。发送响应需能识别整批还是单条/分区失败，保留每条业务 ID 和 outbox 状态；结果未知时以原 ID 幂等重试，必要时二分拆批隔离坏消息，不能让一条毒消息导致整批无限重发。背压时有界缓冲并落盘或快速失败，禁止内存无限堆积。用真实消息大小压测吞吐、P99、CPU、内存和失败重试流量，逐步寻找收益拐点。批量的正确大小是满足延迟与故障预算下单位成本最低的点，不是吞吐数字最大。

**追问点答案**  
- 为什么不把 batch 开得越大越好？  
因为会增加延迟、失败重试成本和内存占用。  
- 哪些场景不适合批量？  
单条消息价值高、实时性要求高、失败必须精确定位的场景。  
- 批量失败怎么处理？  
保留批内业务 ID 与发送状态；可定位单条失败时只重试失败项，结果未知时用原 ID 幂等重试，无法定位则拆小批隔离毒消息并限制放大。

**原理补充**  
批量发送的核心，是用更粗粒度的发送单位换更低单位成本。

**架构扩展**  
成熟架构师不会把批量发送当成默认优化，而是先看业务是否真的需要吞吐优先、是否能接受更粗粒度故障处理。

**工程指标**  
- 批量吞吐提升比例：优化收益。  
- 单条平均发送时延：体验成本。  
- 批失败率：稳定性指标。  
- 批大小分布：运营监控指标。  
- 重试放大倍数：故障成本指标。

**证据链**  
- 批量配置  
- 吞吐和时延对比  
- 失败日志  
- 重试记录

**低风险优化路径**  
1. 先识别适合批量的消息类型  
2. 再控制批大小和时间窗口  
3. 最后补细粒度失败处理

**易错点**  
- 所有消息统一批量  
- 批次过大  
- 不设计批失败回放

## 14. 消费失败重试策略如何设计？

**核心结论**  
消费失败重试策略的关键，是按失败原因和业务价值分级，而不是统一“失败就重试三次”。

**标准答案**  
设计消费失败重试时，我一般先区分是瞬时依赖异常、业务校验失败、数据脏消息还是下游长期不可用。之所以不能统一重试，是因为只有一部分失败会因为时间推移而变得更容易成功。工程上比较稳的策略通常是：瞬时错误短退避重试，长期错误进入延迟重试或死信，不可恢复错误直接隔离；同时整个过程要配幂等和告警。面试里如果能补一句“重试策略本质上是在用时间换成功率，但前提是失败真的值得再试”，会比较成熟。

**高分回答**
我建立错误分类表而不是统一三次重试：网络瞬断、429 和短暂超时可重试，遵循 `Retry-After` 并指数退避加 full jitter；schema 不兼容、必填字段缺失和业务拒绝不会随时间恢复，立即隔离并通知 owner；结果未知的外部写先按业务键查询。每条消息有最大尝试次数、总时间 deadline 和业务有效期，重试进入独立延迟 topic，避免占住主分区或立即回灌；全局 retry token bucket 限制重试流量，依赖过载时熔断。消费者每次尝试都幂等，记录 attempt、错误类型和首次失败时间；超过预算进入死信及人工/自动修复，而不是永久循环。顺序消息要决定是否阻塞该 key、隔离同 key 后续版本或人工跳过，不能悄悄越序。重放前验证根因并限速，过期消息按业务规则重建当前状态或丢弃并审计。通过注入 429、超时、坏 schema、长故障和恢复抖动验证，观察各错误类别的重试成功收益、放大倍数、最老年龄、死信率和业务收敛。只有新增成功率足以覆盖资源与延迟成本的错误才值得重试。

**追问点答案**  
- 为什么业务校验失败通常不该重试？  
因为再试也大概率不会成功。  
- 延迟重试和立即重试怎么选？  
极短瞬断且下游健康可立即少量重试；限流、过载和预计恢复较慢时进入带抖动的延迟通道，并同时受业务 deadline 与全局重试预算约束。
- 死信是什么时候介入的？  
当继续重试收益已经很低时。

**原理补充**  
重试策略的本质，是把失败类型映射成不同的恢复路径，而不是统一机械处理。

**架构扩展**  
成熟架构师设计消费失败路径时，会明确哪些失败属于系统波动，哪些属于业务问题，哪些属于数据垃圾。这样主链路和异常链路才不会搅在一起。

**工程指标**  
- 重试成功率：判断策略有效性。  
- 重试次数分布：识别高成本失败。  
- 死信转入率：尾部问题规模。  
- 消费恢复时长：业务结果指标。  
- 不可恢复错误占比：分类治理指标。

**证据链**  
- 错误分类日志  
- 重试轨迹  
- 死信统计  
- 下游依赖故障记录

**低风险优化路径**  
1. 先做失败分类  
2. 再按类别配置不同重试策略  
3. 最后把死信和告警接起来

**易错点**  
- 所有失败统一重试  
- 没有失败分类  
- 不把业务错误直接隔离

## 15. 如何做消息链路追踪？

**核心结论**  
消息链路追踪的关键，不是只记录 topic 和 offset，而是把“哪条业务事件从哪里来、经过哪些处理、最终到哪”串成一条完整路径。

**标准答案**  
做消息链路追踪时，我一般会给每条业务事件携带统一 traceId、业务主键和事件类型，在生产、broker、消费、重试、死信和补偿环节都打统一日志或埋点。之所以消息链路比同步调用更难追，是因为它跨越时间、线程、服务甚至多套系统，天然不连续。工程上比较有价值的做法是把消息和原始业务动作关联起来，比如订单号、支付号、用户号，而不是只看 MQ 内部元数据。面试里如果能补一句“异步链路追踪的目标是恢复因果关系”，会比较有深度。

**高分回答**
我为事件定义统一 envelope：eventId、业务聚合 ID/version、eventType/schemaVersion、trace context、correlationId、causationId、producer/version、发生时间和投递尝试。同步请求产生事件时保留 trace context；生产 span 与消费 span 通过 span link 表达跨时间因果，不能把数小时后的消费伪装成一个一直未结束的同步父子 span。每次重试生成新 attempt span 但保留原 eventId，死信、重放和补偿记录新旧事件映射，才能回答“这笔订单为何到这里、重放了几次、最终谁修复”。Broker topic/partition/offset 用于定位传输位置，业务键用于核对结果，二者缺一不可。日志与 trace 对敏感业务键脱敏，高基数字段不进入指标标签；采样采用正常低比例、错误/慢/死信尾采样，遥测后端故障时有界丢弃而不阻塞消费。平台基于这些元数据计算端到端延迟、各阶段等待、断链和 schema 错误。验收跨线程、批量、重试、死信与补偿的完整链路，并现场从业务 ID 定位根因；追踪成功标准是能恢复业务因果和责任，而不是只查到一个 offset。

**追问点答案**  
- 为什么 offset 不够用？  
offset 只标识某 topic/partition 的传输位置，重放或跨 topic 后会变化；还需稳定 eventId、业务聚合 ID、版本和因果 ID 才能串起结果。
- 重试和死信也要追踪吗？  
必须要，否则尾部问题无法定位。  
- traceId 要不要跨同步链路传递？  
最好统一，这样异步前后的因果链才完整。

**原理补充**  
消息追踪的本质，是让分散在时间和空间中的异步片段恢复成一条可理解的业务事件链。

**架构扩展**  
成熟架构师不会把消息追踪当成“可选增强”，因为异步系统一旦没追踪，排障和对账成本会飙升。很多团队痛苦的根因，不是 MQ 难，而是消息看不见。

**工程指标**  
- 追踪覆盖率：核心治理指标。  
- trace 断链率：识别追踪缺口。  
- 平均定位时长：追踪系统价值指标。  
- 死信链路可追溯率：尾部治理能力。  
- 业务主键关联成功率：完整性指标。

**证据链**  
- trace 日志  
- 业务主键映射  
- 重试与死信记录  
- 链路恢复样本

**低风险优化路径**  
1. 先统一消息元数据  
2. 再打通生产、消费、重试各环节追踪  
3. 最后把业务主键挂进去

**易错点**  
- 只追踪同步调用  
- 只有 MQ 元数据没有业务主键  
- 死信和补偿链路不可见

## 16. Topic / 分区如何规划？

::: tip L2 深度训练
以业务事实、契约所有权和聚合顺序需求规划，而非按团队随意建 Topic：[领域发现与限界上下文](/deep-dives/business-evolution/01-domain-discovery)
:::

**核心结论**  
Topic 和分区规划的核心不是越多越好，而是让消息模型、并发度、顺序需求和运维成本保持平衡。

**标准答案**  
我规划 topic 时，通常会先按事件域或业务语义拆分，而不是按团队或接口名字随意起；分区数则会结合预期吞吐、消费者并行度、顺序要求和未来扩容空间来定。之所以不能把 topic 和分区开得特别随意，是因为 topic 太细会带来管理复杂度，分区太多会增加 broker 元数据和运维成本，分区太少又限制并发。工程上比较稳的思路是先按“是否同类事件、是否需要共享消费语义”划 topic，再按峰值吞吐和顺序边界规划分区。面试里如果能补一句“topic 规划反映事件建模，分区规划反映吞吐和顺序取舍”，会比较完整。

**高分回答**
我把 topic 当作有 owner、契约和生命周期的事件语义边界，而不是按某个 API 或消费者随意建。只有事件含义、权限/数据等级、保留期、顺序键和重试策略兼容时才共用；支付事件与低价值通知即使生产者相同，也应隔离故障域。分区数从目标吞吐和单分区实测能力反推，例如峰值 12 万条/秒、单分区在 P99 与副本要求下安全承载 8000 条/秒，至少需要 15 个，再为 N-1、增长和消费者并行度留余量。partition key 选业务聚合 ID 保证局部顺序，并用真实分布验证最大/平均流量比；头部租户需要加桶或独立 topic。分区不是越多越好，过多会增加文件句柄、复制、选主、rebalance 和客户端内存；后续扩分区还会改变 key 映射并破坏顺序，因此迁移要有路由版本、排空或新 topic。平台对命名、schema、owner、TTL、ACL、容量和退场做准入，监控单分区吞吐/lag、倾斜、元数据水位和消费者利用率。压测 broker 故障、积压追赶和扩容，证明语义隔离、吞吐和顺序都满足，而不是拍脑袋预建几百个分区。

**追问点答案**  
- 分区越多吞吐一定越高吗？  
不是；吞吐还受 broker 磁盘/网络、消费者和下游限制，过多分区会增加复制、选主、rebalance、句柄与客户端开销。
- 为什么不同业务不该随便共用 topic？  
因为重试策略、保留周期和消费者语义可能完全不同。  
- 顺序消息分区怎么规划？  
要让同一顺序键稳定落到同一分区。

**原理补充**  
topic 是语义边界，分区是并发和顺序边界，两者解决的问题不同。

**架构扩展**  
成熟架构师做 topic 规划时，通常会考虑：这是不是事件资产、生命周期多久、是否允许多类消费、是否会长期演化。好的 topic 设计会让后面的治理轻很多。

**工程指标**  
- topic 数量：资产规模指标。  
- 分区负载均衡度：并发利用率指标。  
- 单分区吞吐：热点识别指标。  
- 元数据管理成本：平台压力指标。  
- topic 复用冲突事件数：治理质量指标。

**证据链**  
- 事件分类清单  
- topic / 分区配置  
- 吞吐和负载监控  
- 消费模型说明

**低风险优化路径**  
1. 先按事件语义划 topic  
2. 再按吞吐和顺序需求定分区  
3. 最后保留平滑扩容空间

**易错点**  
- topic 乱分或乱复用  
- 分区数拍脑袋  
- 不把顺序需求纳入规划

## 17. 如何做削峰填谷评估？

**核心结论**  
削峰填谷的评估核心，不是“能不能堆住”，而是“能堆多久、多久能追平、业务能否接受这个延迟”。

**标准答案**  
评估削峰填谷时，我通常会先算峰值流量、稳态消费能力和高峰持续时间，然后换算 backlog 会堆到多大、业务延迟会被拉长多久，以及高峰过去后需要多久追平。之所以这样评估，是因为削峰不是免费午餐，它只是把吞吐压力从实时搬到了时间维度。工程上还要按业务价值分层，有些消息可以晚几分钟，有些链路晚几十秒就不行。面试里如果能补一句“削峰评估的核心不是能不能堆，而是堆完还能不能及时恢复”，会比较成熟。

**高分回答**
我会画分时生产曲线而不是只用一个峰值。假设高峰 20 分钟生产 5 万条/秒，消费者为保护数据库只能稳定处理 2 万条/秒，则积压峰值约 `3万 × 1200 = 3600万` 条；高峰后生产回落到 5000 条/秒，净追赶 1.5 万条/秒，至少还要 40 分钟清空。然后检查 MQ 保留字节、磁盘/副本、最老消息年龄和业务 SLA：若库存释放要求 10 分钟内完成，60 分钟结果显然不可接受，必须增加安全消费能力、拆高优先级通道或降低入口。容量按 P99 消息大小和副本系数计算，不能只算条数；消费压测要包含真实下游写入与故障一份容量。不同业务分 topic/优先级，高价值事件保留资源，过期可重建的低价值事件可压缩或丢弃并审计。还需模拟峰值期间消费者故障和峰后追赶，确认重试、缓存冷启动及数据库不会被追赶流量二次压垮。指标看 backlog 面积、age P99、恢复 ETA、业务完成延迟、存储水位和单位成本。削峰成立的条件是缓冲容量、恢复速度与业务时效三者同时满足。

**追问点答案**  
- backlog 大一点是不是无所谓？  
还取决于消息字节、保留容量和净追赶速度；若最老年龄或预计恢复时间超过业务 SLA，即使 broker 存得下也不可接受。
- 削峰是不是一定依赖 MQ？  
不一定，本地队列、任务表也可以，但规模和可靠性不同。  
- 为什么要算追平时间？  
因为用户和业务感知的是恢复时间，而不是堆积条数。

**原理补充**  
削峰填谷本质上是用时间缓冲替代瞬时资源峰值。

**架构扩展**  
秒杀、埋点、批量通知、异步积分都常做削峰。成熟架构师会特别强调“业务时效窗口”，因为这决定削峰是不是合理。

**工程指标**  
- 高峰生产 TPS：输入压力。  
- 稳态消费 TPS：输出能力。  
- backlog 峰值：缓冲容量。  
- 追平时长：恢复能力。  
- 业务延迟 SLA：可接受边界。

**证据链**  
- 流量预测  
- 消费能力压测  
- backlog 曲线  
- 业务 SLA 说明

**低风险优化路径**  
1. 先算峰值与稳态差额  
2. 再评估可接受延迟窗口  
3. 最后按业务等级设计削峰策略

**易错点**  
- 只看能不能堆住  
- 不看业务时效  
- 不评估追平时间

## 18. MQ 故障时系统怎么降级？

**核心结论**  
MQ 故障降级的目标不是强行保持原链路不变，而是优先保核心业务，再决定哪些异步动作可以延后、丢弃或转人工。

**标准答案**  
当 MQ 故障时，我通常先按业务重要性分层。核心链路如果必须继续运行，就要考虑临时落本地表、写任务表、走同步补救或进入受控降级模式；低价值异步消息则可以限流、延后、丢弃或暂停。之所以不能统一处理，是因为不同消息的业务价值和时效完全不同。工程上成熟方案通常会提前设计本地兜底、消息开关、低优先级丢弃策略和故障期间的补发机制。面试里如果能补一句“MQ 故障治理的本质是把业务从消息系统依赖中暂时解耦”，会比较完整。

**高分回答**
我先按事件是否影响核心不变量分级。订单落库后必须最终通知履约的事件，正常路径就用本地事务 outbox，MQ 不可用时只积在数据库任务表，订单接口仍成功；发布器退避重试且 outbox 有容量/年龄告警。可重建的搜索刷新、埋点和通知可暂停、采样或丢弃；依赖 MQ 才能完成库存预留且业务不能接受延迟时，应限流或停止接单，不能假装成功。兜底表必须分区、限额并与在线表隔离，防 broker 长故障把数据库填满；同步直调只在下游容量和幂等已验证时作为小流量应急，不能把异步洪峰直接转成同步雪崩。故障期间冻结大规模发布、关闭非核心生产，向用户暴露处理中状态。恢复时按业务优先级、事件版本和创建时间限速补发，先确认消费者与下游余量，避免所有 producer 同时重连；对账事实源捕获任何漏项，补发后再清理 outbox。演练 broker 全挂、半失败、ack 未知和恢复惊群，监控核心成功率、outbox 最老年龄/容量、丢弃量、补发 ETA、重复副作用和差异。MQ 降级不是把消息换个地方无限堆，而是有容量上限的业务连续性方案。

**追问点答案**  
- 所有消息都要本地落表兜底吗？  
不需要；只有必须由本地事实可靠派生的高价值事件使用 outbox。可重建或可丢弃事件应暂停/采样，否则兜底表会成为新的单点和容量风险。
- 为什么有些消息可以直接丢？  
因为业务价值低或可以由后续重建。  
- 故障恢复后最重要的动作是什么？  
补发、对账和追平状态。

**原理补充**  
MQ 故障降级本质上是在“可靠性损失”和“业务连续性”之间做短期折中。

**架构扩展**  
成熟架构师面对 MQ 故障，不会只盯中间件恢复，而是会先看“订单还能不能下、支付还能不能确认、通知可不可以晚点发”。这是业务优先级思维。

**工程指标**  
- MQ 不可用时长：故障规模指标。  
- 本地兜底写入成功率：核心业务保护指标。  
- 丢弃消息占比：降级代价指标。  
- 补发成功率：恢复质量指标。  
- 对账差异数：尾部风险指标。

**证据链**  
- 故障期间日志  
- 本地兜底记录  
- 降级开关状态  
- 补发与对账数据

**低风险优化路径**  
1. 先给消息按业务价值分级  
2. 再为高价值链路设计本地兜底  
3. 最后补恢复后的补发和对账

**易错点**  
- 所有消息一刀切处理  
- 平时不准备故障兜底  
- 故障恢复后不补发不核对

## 19. 如何做消息平台 SLA？

**核心结论**  
消息平台 SLA 不能只写“可用”，而要把可用性、时延、可靠性和恢复能力都量化出来。

**标准答案**  
我定义消息平台 SLA 时，通常会分成四类指标：第一是可用性，比如生产和消费接口成功率；第二是时延，比如消息从生产到可消费、到实际消费完成的 P95/P99；第三是可靠性，比如消息丢失率、重复率、死信处理时限；第四是恢复能力，比如故障恢复时间、backlog 追平时间。之所以要这么分，是因为消息平台不是简单的“能收发”就算合格，它承载的是业务事件通路。工程上 SLA 最后要能映射成具体告警、预案和容量预算，而不是只写在文档里。面试里如果能补一句“消息平台 SLA 的核心是业务事件承诺”，会很加分。

**高分回答**
我会把平台 SLI 和业务端到端 SLI 分开。平台侧定义生产 ack 成功率、broker 持久化/复制达标率、可消费延迟和控制面可用性；业务侧从事实提交时间到消费者副作用完成，按事件等级承诺 P99/P99.9，例如核心订单 99.9% 在 5 秒、99.99% 在 1 分钟内收敛。可靠性不能只写“零丢失”，要明确 ack/RPO 语义、重复由业务幂等吸收、每百万事件未解释差异和死信最老处理时间。恢复项包括 broker RTO、单可用区故障容量、目标积压追平时间及最大可接受 age。每项有错误预算与 burn-rate 告警，快速消耗时冻结变更、扩容或降级；告警直接关联 owner 和运行手册。测量用端到端探针事件、broker 指标、消费状态和独立对账交叉验证，避免平台绿但业务消费者已停。按 topic/租户隔离指标，防平均值掩盖关键链路。季度故障演练验证生产、消费、跨区和追赶，并公开未达标与容量成本。SLA 是生产者和消费者共同依赖的事件承诺，必须说明边界、分位数、错误预算与恢复证据。

**追问点答案**  
- 为什么时延要分生产到可消费和消费完成？  
前者反映 broker 传输与调度，后者还包含消费者 lag 和副作用；二者分开才能定位责任，同时端到端结果才代表业务承诺。
- 重复率也算 SLA 吗？  
对很多高价值业务来说，绝对算。  
- SLA 和内部监控有什么关系？  
SLA 是对外承诺，监控是验证承诺的手段。

**原理补充**  
消息平台 SLA 的本质，是把平台能力翻译成业务可依赖的事件服务承诺。

**架构扩展**  
成熟架构师做消息平台时，不会只讲吞吐，还会讲“业务多久能收到、出问题多久恢复、重复和丢失风险怎么算”。这才是平台思维。

**工程指标**  
- 生产接口成功率：可用性指标。  
- 端到端延迟 P99：实时性指标。  
- 死信处理时长：尾部治理指标。  
- 丢失/重复率：可靠性指标。  
- backlog 追平时间：恢复能力指标。

**证据链**  
- SLA 定义文档  
- 监控面板  
- 故障和恢复记录  
- 对账和审计结果

**低风险优化路径**  
1. 先定义平台承诺边界  
2. 再把承诺映射到监控和告警  
3. 最后定期用演练验证 SLA

**易错点**  
- 只写可用性不写时延和可靠性  
- SLA 不可监控  
- 不把恢复时间纳入承诺

## 20. 如何验证异步架构收益？

::: tip L2 深度训练
同时衡量跨域等待、业务完成延迟、一致性差异和运维复杂度：[单体渐进迁移](/deep-dives/business-evolution/03-strangler-migration)
:::

**核心结论**  
验证异步架构收益不能只看“接口快了”，还要看系统吞吐、稳定性、可恢复性和业务体验有没有一起变好。

**标准答案**  
我验证异步架构收益时，通常会从四个维度看。第一是主链路性能，比如接口时延、线程占用、下游调用数量有没有下降；第二是吞吐和峰值承接能力，比如高峰时是否更稳；第三是恢复和治理能力，比如积压是否可控、失败后能否补偿；第四是业务结果，比如成功率、投诉率、延迟是否在可接受范围内。之所以不能只看“接口从 300ms 变 80ms”，是因为异步化可能只是把问题延后了。工程上最有价值的是对照实验和灰度验证，而不是只凭主观感受。面试里如果能补一句“异步架构的收益必须覆盖系统和业务两个层面”，会比较完整。

**高分回答**
我在改造前先写可证伪假设，例如“订单接口 TP99 从 600ms 降至 200ms，峰值承接从 5000 提升到 1 万 TPS，同时 99.9% 履约事件 30 秒内完成，人工补偿不增加”。灰度按稳定用户/租户分组，对照同步旧链路与异步新链路，保持流量结构和下游容量可比；影子事件验证结果但不产生真实副作用。系统收益看主链路 TP99、线程/连接占用、同步 fan-out、峰值错误率和单位成本；异步成本看端到端完成时间、backlog age/恢复 ETA、重复/乱序、死信、对账差异、平台运维与开发认知成本；业务看下单/履约成功率、状态处理中时长、投诉和客服工单。压测不仅打正常峰值，还停消费者、注入 broker 故障与重试，证明缓冲和恢复而不是把问题推迟。若接口快 400ms 但 2% 订单分钟级不收敛，改造不达标。观察完整业务周期后再扩大灰度，并保留回切和补偿。最终以收益减去一致性、延迟、成本和事故风险的净结果决策，而不是只展示入口延迟下降。

**追问点答案**  
- 为什么接口变快不代表一定收益大？  
异步只是把等待移到后台；必须同时证明端到端完成 SLA、积压恢复、差异/补偿、用户体验和总成本没有恶化。
- 最好怎么验证？  
通过灰度、压测和上线前后指标对照。  
- 业务体验怎么评估？  
看成功率、延迟可接受度、投诉和人工介入。

**原理补充**  
异步化本质上是在重新分配时延和可靠性成本，所以收益验证必须覆盖完整生命周期。

**架构扩展**  
成熟架构师不会只说“我们异步化后快了很多”，而是会继续给出：高峰扛住了多少、MQ 是否稳、补偿率多少、用户有没有明显抱怨。这才是真正能站住的收益证明。

**工程指标**  
- 主链路 TP99：性能收益指标。  
- 峰值吞吐：容量收益指标。  
- backlog 与追平时长：稳定性成本指标。  
- 补偿和死信率：治理复杂度指标。  
- 用户投诉率：业务结果指标。

**证据链**  
- 灰度对照数据  
- 压测报告  
- 主链路 tracing  
- 异步链路监控和用户反馈

**低风险优化路径**  
1. 先定义收益目标  
2. 再做灰度对照和压测  
3. 最后同时评估系统指标和业务结果

**易错点**  
- 只看接口变快  
- 不看异步尾部问题  
- 没有灰度和对照验证
