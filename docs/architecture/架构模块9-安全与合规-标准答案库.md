# 架构模块9 - 安全与合规 - 标准答案库

## 1. 认证与鉴权如何分层设计？

::: tip L2 深度训练
区分认证、策略决策、对象级执行与业务前置条件：[身份、授权与租户隔离](/deep-dives/security-compliance/01-identity-authorization)
:::

**核心结论**  
认证解决“你是谁”，鉴权解决“你能做什么”，分层设计的关键是把身份确认、权限判断和业务敏感操作校验拆开。

**标准答案**  
我做认证与鉴权设计时，通常会分成入口认证、服务鉴权和业务敏感操作校验三层。入口层负责确认用户或系统身份，比如登录态、Token、签名；服务层负责判断角色、权限、租户范围；业务层再对高风险操作做更细粒度校验，比如是否本人、是否满足审批链、是否命中风控规则。之所以要分层，是因为很多系统把所有安全判断都压在网关或接口注解上，最后业务细节保护不住。工程上比较稳的设计是统一身份体系、统一权限模型，再把敏感业务规则留在业务层兜底。面试里如果能补一句“认证和鉴权不是一个组件的事，而是多层防线”，会比较成熟。

**高分回答**
我先区分主体、资源、动作和上下文。身份提供方完成用户认证与 MFA，入口验证凭证的签名、签发方、受众、时效和会话状态；服务间使用工作负载身份和短期凭证，不能把“来自内网”当身份。策略决策层基于角色、租户、资源归属和环境输出 allow/deny，业务执行点仍要校验对象级权限与业务前置条件，例如客服有“查订单”权限不代表能查看任意租户订单。默认拒绝、最小权限，资源 ID 必须在授权范围内解析，防止只鉴权接口却发生 IDOR。高风险动作加入 step-up 认证、双人审批或交易签名，并对授权决策记录主体、资源、策略版本和结果，但不泄露敏感凭证。缓存权限时要处理撤权时效，身份系统故障时敏感写操作 fail closed，低风险读是否降级需预先定义。通过跨租户、撤权、伪造服务身份和策略服务故障测试验证，并监控越权事件、撤权生效时间、策略误拒与无身份内部调用。

**追问点答案**  
- 为什么网关做了认证，业务层还要校验？  
因为高风险业务规则通常只有业务层最清楚。  
- 认证失败和鉴权失败为什么要区分？  
因为一个是身份无效，一个是身份有效但权限不足。  
- 服务间调用也要做认证吗？  
要。使用可轮换的工作负载身份或 mTLS/短期令牌，并校验目标受众和服务权限；共享永久密钥与来源 IP 都不足以证明具体调用者。

**原理补充**  
认证与鉴权分层的本质，是把“身份可信”和“动作被允许”这两个问题分别治理。

**架构扩展**  
成熟架构师不会只说“用 JWT 做认证”，而是会继续讲：角色权限在哪里判、租户边界在哪里控、敏感操作谁来二次确认。这说明理解的是整条安全链路，而不是一个 token 技术点。

**工程指标**  
- 认证失败率：身份系统健康度指标。  
- 鉴权拒绝率：权限控制效果指标。  
- 越权操作事件数：核心风险指标。  
- 敏感操作二次校验命中率：高风险保护指标。  
- 内部服务未认证调用数：内部安全治理指标。

**证据链**  
- 认证流程图  
- 权限模型  
- 网关和服务层鉴权日志  
- 敏感操作审计记录

**低风险优化路径**  
1. 先统一身份体系  
2. 再统一权限判断入口  
3. 最后给高风险业务补二次校验

**易错点**  
- 认证鉴权混在一起讲  
- 只靠入口层保护  
- 忽略内部调用身份可信问题

## 2. RBAC 和 ABAC 怎么选？

::: tip L2 深度训练
组合岗位、资源关系、租户和环境条件，避免角色爆炸：[身份、授权与租户隔离](/deep-dives/security-compliance/01-identity-authorization)
:::

**核心结论**  
RBAC 更适合角色稳定、规则清晰的系统，ABAC 更适合属性复杂、上下文敏感的权限场景，选型关键看变化维度和治理成本。

**标准答案**  
我选 RBAC 或 ABAC 时，通常先看权限规则主要依赖什么。如果权限主要由岗位或角色决定，比如管理员、审核员、客服，这类场景 RBAC 更简单、稳定、易管理；如果权限和租户、地域、数据归属、时间窗口、资源标签等属性强相关，ABAC 会更灵活。之所以不能只追求灵活，是因为 ABAC 的规则解释和治理成本通常比 RBAC 高很多。工程上很多成熟系统其实会混用：基础权限用 RBAC，敏感数据范围和细粒度限制用 ABAC。面试里如果能补一句“权限模型不是追求最强，而是追求可治理”，会比较完整。

**高分回答**
我从规则变化维度和可解释性选型。岗位稳定、权限集合有限时用 RBAC，便于审批、审计和定期复核；若规则依赖资源 owner、租户、数据级别、地域、时间或设备风险，纯 RBAC 会产生“华东夜班高级客服”等角色爆炸，此时用 ABAC 表达动态条件。实际常采用组合模型：RBAC 决定能否发起某类动作，ABAC/关系判断限制能操作哪些对象，业务规则再校验状态机前置条件。属性必须有可信来源、owner、更新时效和缺失语义；关键属性缺失默认拒绝，不能让调用方自行声明。策略集中版本化与静态检查，执行点靠近资源并携带完整上下文；上线先影子评估新策略，比较旧/新决策，随后灰度和可回滚。治理上提供“为什么允许/拒绝”的解释、冲突优先级、权限申请到期和周期性 recertification。选型指标不是规则数量越少越好，而是越权/误拒、角色增长、决策 P99、撤权生效时间和审计解释耗时。

**追问点答案**  
- 为什么很多系统先用 RBAC？  
因为实现和理解成本低，更容易快速落地。  
- ABAC 为什么更复杂？  
因为决策要依赖更多上下文属性，规则也更难解释。  
- 两者能混用吗？  
可以。常见做法是 RBAC 控制动作集合，ABAC 或资源关系控制数据范围；必须明确冲突顺序、属性可信来源和统一审计，避免两套规则各自放行。

**原理补充**  
权限模型选型的本质，是在“规则表达力”和“治理可控性”之间做平衡。

**架构扩展**  
成熟架构师不会把 RBAC 和 ABAC 当成书本概念对立，而是会先问：这个系统的权限变化频率高不高、边界多不多、谁来维护规则、怎么审计。

**工程指标**  
- 权限规则数量：治理复杂度指标。  
- 权限变更频率：模型适配度指标。  
- 越权/误拒事件数：最终质量指标。  
- 权限规则解释耗时：运维和审计成本指标。  
- 规则复用率：模型稳定性指标。

**证据链**  
- 权限设计文档  
- 角色与属性清单  
- 审批与审计记录  
- 权限异常案例

**低风险优化路径**  
1. 先用简单模型承接主流程  
2. 再针对复杂边界补属性规则  
3. 最后给规则管理和审计留平台能力

**易错点**  
- 一上来就做非常复杂的 ABAC  
- 只讲模型能力不讲治理成本  
- 不了解真实业务变化维度

## 3. JWT 常见风险有哪些？

::: tip L2 深度训练
验签之后继续验证受众、时效、撤权和对象归属：[身份、授权与租户隔离](/deep-dives/security-compliance/01-identity-authorization)
:::

**核心结论**  
JWT 的风险不在于它是不是流行，而在于使用不当时会放大身份伪造、泄露、过期控制和撤销困难问题。

**标准答案**  
我看 JWT 风险时，通常先关注几个点：签名算法是否安全、密钥管理是否规范、Token 生命周期是否过长、敏感信息是否被放进载荷、丢失后是否有撤销机制。之所以 JWT 经常被误用，是因为它“看起来很方便”，但很多团队把它当成万能会话方案，结果忽略了无状态带来的撤销困难。工程上比较稳的做法是短生命周期、只放必要声明、服务端保留刷新和黑名单机制，并对高敏感操作做二次校验。面试里如果能补一句“JWT 的最大风险不是技术本身，而是无状态便利背后的治理代价”，会比较成熟。

**高分回答**
JWT 首先是签名声明而不是加密容器，payload 可读，不能放密码、密钥或不必要隐私。验证端必须固定允许的算法并验证 `iss`、`aud`、`exp`、`nbf`，不能信任令牌自带算法选择；`kid` 只能在受控密钥集合中查找，避免算法混淆和任意密钥加载。访问令牌短期、最小 scope，刷新令牌采用轮换与复用检测并安全存储；浏览器场景还要考虑 XSS、Cookie 的 HttpOnly/Secure/SameSite 和 CSRF 边界。密钥经 KMS 托管、版本化轮换，验证端支持重叠窗口与缓存刷新，避免轮换造成全站登出。无状态令牌不能天然实时撤权，因此高风险系统要结合会话版本、撤销列表或令牌内省，并明确账号禁用和权限变更的最大生效时间。日志只记录散列 token ID 等审计字段，不记录完整令牌。测试覆盖过期、错误受众、旧密钥、时钟偏差、重放、登出和刷新令牌被盗；高价值交易即使令牌有效也需二次验证与业务鉴权。

**追问点答案**  
- 为什么不要把敏感信息放进 JWT？  
因为载荷通常可解码，不适合承载隐私和机密数据。  
- 为什么 JWT 撤销困难？  
因为它天然偏无状态，服务端默认不追踪每个 token。  
- 短 token + refresh token 为什么常见？  
短访问令牌限制失窃暴露窗口，刷新令牌维持会话；后者必须轮换、可撤销并检测重复使用，否则只是把长期风险转移到另一枚令牌。

**原理补充**  
JWT 本质上是自包含身份声明载体，它的便利来自去状态化，它的风险也来自去状态化。

**架构扩展**  
成熟架构师在谈 JWT 时，不会只说“用它做登录”，而是会继续讲密钥轮换、刷新策略、失窃处置和高风险动作保护。这才是真正上线视角。

**工程指标**  
- Token 过期时间分布：风险暴露指标。  
- 黑名单命中率：撤销机制指标。  
- 非法 token 拒绝率：攻击防护指标。  
- 敏感操作二次验证率：高风险保护指标。  
- 密钥轮换成功率：密钥治理指标。

**证据链**  
- JWT 配置  
- 密钥管理策略  
- 登录与刷新日志  
- 非法访问审计

**低风险优化路径**  
1. 先缩短 token 生命周期  
2. 再建立刷新和撤销机制  
3. 最后补密钥轮换与高风险操作二次校验

**易错点**  
- JWT 放太多敏感信息  
- token 生命周期太长  
- 没有撤销和轮换策略

## 4. 接口防刷如何落地？

**核心结论**  
接口防刷的重点不是简单限流，而是识别请求来源、行为模式和业务风险等级，再做分层防护。

**标准答案**  
我做接口防刷时，通常会先按接口价值和风险分级。高风险接口比如登录、验证码、下单、支付、抢券，会在网关和业务层同时加保护，包括频率限制、设备指纹、IP 信誉、验证码、行为验证、黑白名单和风控策略。之所以不能只靠限流，是因为恶意流量常常会模拟正常速率或分布式来源。工程上更稳的策略是把防刷做成组合拳：入口快速拦截，业务层做强语义校验，风控层做动态策略更新。面试里如果能补一句“防刷的本质是识别异常行为，而不只是限制流量大小”，会比较完整。

**高分回答**
我先给登录、短信、搜索、抢券、下单和支付定义不同保护目标与攻击成本，不能共用一套 IP 阈值。入口按 IP/网段、账号、设备、会话和 API key 做多维令牌桶，业务层再限制“每账号每活动可领取一次”等语义不变量，并用幂等键阻止重复副作用；风控根据失败率、速度、设备关联、行为序列和信誉做风险评分。低风险请求无感通过，中风险增加 proof-of-work/验证码或二次验证，高风险拒绝并短期冷却，避免所有用户都承担摩擦。规则需考虑 NAT 和无障碍用户，不能仅凭 IP 永久封禁；响应也避免暴露账号是否存在。防护系统与核心服务隔离容量，策略服务不可用时对高风险写保守关闭、普通读采用本地兜底。通过历史攻击回放、红队和灰度校准，联合看攻击成功率、误伤/申诉率、验证完成率、单次攻击成本、下游节省流量和规则漂移；每条封禁都有原因、期限与人工申诉路径。

**追问点答案**  
- 为什么登录接口特别需要防刷？  
因为它容易被撞库、爆破和账号探测。  
- 验证码能解决所有问题吗？  
不能。验证码可被代答、自动识别或中继，还会伤害体验；应只作为风险分层中的加摩擦手段，并配合多维限额、业务不变量和异常行为检测。
- 业务层为什么也要参与？  
因为只有业务层最了解哪些行为模式异常。

**原理补充**  
接口防刷本质上是在成本可控前提下，把异常流量尽量挡在业务副作用发生之前。

**架构扩展**  
成熟架构师做防刷时，不会只说“加限流”，而是会继续讨论风险接口分级、攻击成本提升、用户体验平衡和误伤控制。这才是完整方案。

**工程指标**  
- 防刷命中率：防护效果指标。  
- 误伤率：体验成本指标。  
- 异常来源覆盖率：识别能力指标。  
- 风险接口攻击成功率：最终安全结果指标。  
- 验证码/风控触发率：策略执行指标。

**证据链**  
- 风险接口清单  
- 攻击样本日志  
- 限流与风控规则  
- 用户申诉和误伤记录

**低风险优化路径**  
1. 先给接口按风险分级  
2. 再做入口 + 业务 + 风控三层防护  
3. 最后持续优化误伤与命中平衡

**易错点**  
- 所有接口一套防刷规则  
- 只做单点限流  
- 不评估误伤用户体验

## 5. 敏感数据脱敏怎么做？

::: tip L2 深度训练
区分展示脱敏、哈希、令牌化和加密的适用边界：[数据保护与生命周期](/deep-dives/security-compliance/02-data-lifecycle)
:::

**核心结论**  
敏感数据脱敏的核心不是把字段打星号，而是按场景控制“谁、在什么条件下、能看到多完整的数据”。

**标准答案**  
我做敏感数据脱敏时，通常先按数据类型分级，比如手机号、身份证、银行卡、地址、邮箱、业务隐私信息，然后再按使用场景设计展示脱敏、日志脱敏、导出脱敏和测试数据脱敏。之所以不能只做统一打码，是因为不同角色和场景对数据可见范围要求不同。工程上更稳的做法是把脱敏策略收敛到统一组件或平台层，而不是每个服务自己手写。面试里如果能补一句“脱敏不是表现层技巧，而是数据最小可见原则的落地”，会比较成熟。

**高分回答**
我先做数据分类、用途和流转图，区分展示掩码、不可逆哈希、可逆令牌化和加密：手机号给客服展示部分字符是掩码，不降低数据库泄露风险；密码用带盐慢哈希；需要关联但不见原值可令牌化；必须恢复的高敏字段用受控加密。策略按主体、用途、字段和环境决定最小可见范围，后端返回前执行，不能只靠前端 CSS。日志、trace、消息、搜索索引、导出、备份和测试副本都要纳入，默认不采集敏感值；排障临时解敏采用审批、短时授权、水印、审计和到期回收。保持格式的脱敏数据仍可能被关联重识别，因此测试数据应合成或保持统计分布的不可逆处理，并评估小样本组合风险。统一策略服务要版本化、失败时默认遮蔽，高风险导出增加行数限制与异常检测。用字段覆盖率、明文扫描、解敏次数、未授权访问、到期删除和抽样人工核验验证，而不是只检查页面截图。

**追问点答案**  
- 为什么日志脱敏很重要？  
因为很多泄露不是来自业务界面，而是来自日志和排障导出。  
- 测试环境为什么也要脱敏？  
因为测试环境通常权限和安全边界更弱。  
- 完全不可见和部分脱敏怎么选？  
按最小必要用途决定：不需要识别个体就完全不可见或令牌化，确需核对时只展示足够区分的部分；查看原文应是高风险例外，需短时授权和完整审计。

**原理补充**  
脱敏本质上是在数据流转路径上持续执行“最小必要暴露”原则。

**架构扩展**  
成熟架构师不会把脱敏理解成前端格式化，而是会从存储、查询、日志、导出、排障、测试全链路看数据暴露面。

**工程指标**  
- 敏感字段脱敏覆盖率：建设指标。  
- 日志泄露事件数：风险结果指标。  
- 测试数据脱敏完成率：环境治理指标。  
- 导出审计命中率：高风险动作指标。  
- 违规访问告警数：防护效果指标。

**证据链**  
- 数据分类清单  
- 脱敏规则配置  
- 日志与导出审计  
- 测试环境数据样本

**低风险优化路径**  
1. 先做数据分级  
2. 再统一展示/日志/导出脱敏策略  
3. 最后补审计和测试环境治理

**易错点**  
- 只做前端脱敏  
- 忽略日志和测试环境  
- 各系统各自实现脱敏逻辑

## 6. 传输加密和存储加密边界是什么？

**核心结论**  
传输加密保护数据在路上不被窃听和篡改，存储加密保护数据落盘后不被直接读取，两者边界不同但通常都需要。

**标准答案**  
我理解传输加密时，重点是客户端到服务端、服务间调用、跨机房同步、外部依赖访问这些链路的机密性和完整性；存储加密则关注数据库、对象存储、备份文件和磁盘层面的数据保护。之所以两者不能互相替代，是因为数据在路上和在盘上面对的是不同威胁模型。工程上如果业务敏感度高，除了 TLS 和磁盘加密，还可能要做字段级加密或应用层加密。面试里如果能补一句“加密边界取决于威胁模型，不是统一打勾项”，会比较成熟。

**高分回答**
我先画数据从终端、边缘、服务、消息、数据库、分析平台到备份的流转图，并标注每一段的攻击者与信任边界。传输中用现代 TLS 保护机密性、完整性和端点身份，服务间也应认证，终止 TLS 的网关之后若跨越共享网络仍需重新加密；同时治理证书签发、轮换、吊销和主机名校验。静态加密可防磁盘、快照和备份介质丢失，但拥有数据库逻辑权限的人仍可能读明文；高敏字段因此使用信封加密或令牌化，将数据密钥与密文分离，并以 KMS 控制用途和审计。应用层加密会牺牲查询、索引和密钥轮换便利，要按字段威胁模型选用，不能全库盲目加密。还要覆盖临时文件、日志、搜索索引和导出，内存中的明文则靠最小权限、进程隔离和缩短驻留时间降低风险。验收不仅看“已开启”，还测试旧协议拒绝、证书过期、密钥失效、备份恢复与轮换不中断，并监控明文路径、解密调用和未加密资产。

**追问点答案**  
- TLS 做了是不是就够了？  
不够，落盘后的数据仍可能被非法读取。  
- 磁盘加密是不是能代替字段加密？  
不能。磁盘加密主要防介质、快照或底层存储泄露，无法约束合法数据库接口、DBA 或被攻陷应用读取；字段级保护用于更窄的高敏威胁边界，但会增加查询和轮换成本。
- 为什么跨服务调用也要加密？  
因为内网不是绝对可信环境。

**原理补充**  
加密的本质，是按数据所处位置分别降低不同攻击面的可见性和可篡改性。

**架构扩展**  
成熟架构师不会只说“上 HTTPS”，而是会继续解释：什么链路靠传输加密、什么数据要落盘加密、什么高敏字段要应用层保护。这体现的是威胁建模能力。

**工程指标**  
- TLS 覆盖率：传输安全指标。  
- 弱加密协议命中率：风险指标。  
- 存储加密覆盖率：落盘保护指标。  
- 高敏字段加密覆盖率：数据核心保护指标。  
- 明文传输/存储告警数：治理结果指标。

**证据链**  
- TLS 配置  
- 存储与备份加密策略  
- 字段级加密清单  
- 安全扫描结果

**低风险优化路径**  
1. 先画清数据流转路径  
2. 再按路上与盘上分别补保护  
3. 最后针对高敏字段做更强加密

**易错点**  
- 只做传输加密  
- 把内网默认当可信  
- 不区分磁盘加密和字段加密

## 7. 密钥管理怎么做？

::: tip L2 深度训练
设计信封加密、版本轮换、短期身份、吊销与恢复：[数据保护与生命周期](/deep-dives/security-compliance/02-data-lifecycle)
:::

**核心结论**  
密钥管理的核心不是把密钥藏起来，而是让密钥生成、分发、使用、轮换、撤销全流程可控可审计。

**标准答案**  
我做密钥管理时，通常先确保密钥不出现在代码仓库、配置文件明文和日志里，再把密钥统一放到专门的 KMS、密钥服务或受控的机密管理系统中。之所以这件事重要，是因为很多安全事故不是算法弱，而是密钥泄露。工程上成熟的密钥管理还要覆盖最小权限、短期凭证、自动轮换、密钥版本、吊销和审计。面试里如果能补一句“密钥管理的风险点不在加密算法，而在密钥生命周期治理”，会比较成熟。

**高分回答**
我按生成、托管、授权、使用、轮换、吊销、备份和销毁管理完整生命周期。主密钥在 KMS/HSM 内生成并尽量不可导出，业务使用信封加密：KMS 保护数据密钥，应用只在短时间内获得明文数据密钥并清理内存。人和工作负载通过独立身份获取最小用途权限，区分 encrypt/decrypt/admin，生产解密与密钥管理不能由同一角色长期兼有；CI/CD 使用短期凭证，不共享永久 secret。密文携带 key version，轮换先发布新版本写入、保留旧版本解密，再后台重加密并验证，避免一次切换导致历史数据不可读。泄露响应包括立即限制用途、吊销凭证、评估被加密数据、重签/重加密和通知，而不是只“换配置”。密钥备份和灾备恢复要定期演练，否则 KMS 故障可能比数据泄露更快造成停机；审计记录谁因何解密何类资源并对异常批量调用告警。衡量未托管密钥、轮换逾期、长期凭证、吊销时间、解密异常和恢复演练通过率。

**追问点答案**  
- 为什么密钥不能写配置文件？  
因为配置传播面广，泄露风险高。  
- 自动轮换为什么重要？  
因为长期不换的密钥一旦泄露，影响面很大。  
- 应用真的需要看到明文密钥吗？  
长期主密钥通常不应离开 KMS/HSM；应用可能短暂使用数据密钥，但应限作用域和缓存时间、避免日志/转储泄露，并优先让受控加解密服务完成操作。

**原理补充**  
密钥管理本质上是在控制“谁能在什么时间、以什么方式接触到密钥”。

**架构扩展**  
成熟架构师谈密钥管理时，不会只提 KMS，而是会继续讲接入权限、轮换策略、泄露处置和审计链。这说明理解的是治理，不只是工具。

**工程指标**  
- 密钥轮换周期：治理成熟度指标。  
- 明文暴露事件数：核心风险指标。  
- 未受管密钥数量：资产治理指标。  
- 最小权限达标率：访问控制指标。  
- 密钥吊销执行时长：应急能力指标。

**证据链**  
- 密钥资产清单  
- KMS 配置  
- 轮换记录  
- 访问审计日志

**低风险优化路径**  
1. 先盘点密钥资产  
2. 再统一托管和审计  
3. 最后推进自动轮换和最小权限

**易错点**  
- 密钥散落在代码和配置中  
- 无轮换策略  
- 泄露后没有快速吊销机制

## 8. SQL 注入 / XSS / CSRF 如何防护？

**核心结论**  
这三类攻击问题不同，防护也不能混为一谈：SQL 注入靠参数化和边界校验，XSS 靠输出编码和内容约束，CSRF 靠请求来源校验和会话保护。

**标准答案**  
我处理 SQL 注入时，第一原则是参数化查询和 ORM 安全接口，绝不拼接不可信输入；处理 XSS 时，核心是输出编码、富文本白名单和前端展示约束；处理 CSRF 时，则要结合 SameSite、CSRF token、来源校验和敏感操作二次确认。之所以这三类常被一起问，是因为它们都来自输入边界失守，但真正攻击面不一样。工程上更稳的做法是框架层默认防护 + 代码评审 + 安全扫描一起做，而不是依赖开发手工记忆。面试里如果能补一句“输入不可信、输出要约束、请求要验来源”，会比较清晰。

**高分回答**
我会按解释器边界分别治理。SQL 注入用参数化值绑定，动态表名、列名和排序方向只能从服务端白名单映射，数据库账号再以最小权限限制失陷影响；ORM 的 raw query 仍需专项审查。XSS 要按 HTML 文本、属性、URL、JavaScript 等输出上下文编码，富文本采用成熟白名单 sanitizer，并用 CSP/Trusted Types 作为纵深防御；输入过滤不能替代输出编码，已存储恶意内容也必须安全呈现。CSRF 针对浏览器自动携带凭证：状态变更使用不可预测且绑定会话的 token，校验 Origin/Referer，Cookie 设置 SameSite/Secure/HttpOnly，GET 不产生副作用；Bearer token 若不由浏览器自动附带，威胁边界不同，但仍需防 XSS 窃取。高风险动作再验证密码/MFA 与业务幂等。将安全 API 封装为框架默认项，SAST/DAST、单元测试和渗透验证覆盖动态 SQL、富文本、旧浏览器与跨域配置；指标关注可利用漏洞和修复时长，不以扫描告警数量当成果。

**追问点答案**  
- ORM 用了是不是就不会 SQL 注入？  
不一定。参数绑定通常安全，但 raw query、字符串拼接以及动态表名/排序字段仍可能引入风险；后两者应使用服务端枚举白名单而不是把标识符当普通参数。
- XSS 为什么不是前端单独问题？  
因为服务端富文本和内容存储同样会引入风险。  
- 为什么 CSRF 在 Cookie 会话场景更常见？  
因为浏览器会自动带上 Cookie。

**原理补充**  
这三类攻击本质上都与“信任了不该信任的输入或请求上下文”有关。

**架构扩展**  
成熟架构师回答这题时，不会只背 OWASP 名词，而是会讲默认防护策略在哪里、哪些框架能力能复用、哪些高风险点要额外审查。

**工程指标**  
- 注入/XSS/CSRF 漏洞数：最终风险指标。  
- 参数化覆盖率：SQL 防护指标。  
- 输出编码覆盖率：XSS 防护指标。  
- CSRF Token 覆盖率：请求来源防护指标。  
- 安全扫描命中率：治理过程指标。

**证据链**  
- 安全扫描结果  
- 代码审计报告  
- 框架配置  
- 漏洞修复记录

**低风险优化路径**  
1. 先让框架默认安全  
2. 再补高风险代码审查  
3. 最后通过扫描和渗透验证

**易错点**  
- 把三类问题一锅端  
- 只靠开发自觉  
- 框架有能力但没开启

## 9. 文件上传安全如何做？

**核心结论**  
文件上传安全的核心不是“能上传成功”，而是限制类型、大小、内容和后续处理路径，防止文件成为执行入口或传播载体。

**标准答案**  
我做文件上传安全时，通常会先做类型白名单、大小限制、文件名规范化和存储路径隔离，再结合病毒扫描、内容识别、权限控制和访问签名。之所以文件上传危险，是因为它既可能带来恶意代码，也可能带来存储滥用、越权访问和内容分发风险。工程上更稳的做法是上传后不直接落到可执行目录，下载时走受控链接或对象存储签名访问。面试里如果能补一句“文件上传问题本质上是让外部内容进入内部系统的入口治理”，会比较成熟。

**高分回答**
我把上传拆为接收、隔离、验证、处理和发布。入口限制认证主体、配额、声明大小与实际流式字节数，服务端生成随机对象 ID，丢弃原始路径并规范化展示名；扩展名、Content-Type 和 magic bytes 联合白名单仍只是初筛。新文件先进入不可执行、私有的 quarantine bucket，病毒/恶意内容扫描及必要的安全重编码通过后才改变状态并发布；扫描器在隔离沙箱内运行，限制 CPU、内存、解压层数和总展开大小，防止压缩炸弹与解析器漏洞。图片/文档处理结果写入新对象，不信任原文件；下载时重新鉴权资源归属，使用短时签名、正确 Content-Disposition 和 `nosniff`，避免把上传域与主站同源执行。异步扫描期间用户只能看到“处理中”，扫描失败默认不发布，并有超时、重试和人工处置。监控恶意命中、扫描滞留、越权下载、配额滥用与处理器异常，通过伪造类型、双扩展、超大压缩包和租户越权测试验证全链路。

**追问点答案**  
- 为什么只看扩展名不够？  
因为扩展名可以伪造。  
- 上传后为什么不能直接公网直链？  
因为权限和审计会失控。  
- 病毒扫描是不是一定要同步做？  
不必同步，但扫描完成前文件必须保持隔离且不可公开访问；异步流程要有明确状态、超时、失败关闭、重试上限和人工处置。

**原理补充**  
文件上传安全本质上是在控制“外部字节流”进入系统后的风险扩散路径。

**架构扩展**  
成熟架构师不会把文件上传理解成前端组件问题，而是会从接收、存储、扫描、访问、分发全链路来看，这样才能真正收住风险。

**工程指标**  
- 非白名单文件拦截率：入口防护指标。  
- 病毒扫描命中率：内容安全指标。  
- 越权下载事件数：访问控制指标。  
- 上传失败率：体验平衡指标。  
- 可疑文件滞留时长：治理效率指标。

**证据链**  
- 上传策略配置  
- 扫描日志  
- 对象存储权限记录  
- 下载访问审计

**低风险优化路径**  
1. 先限制类型和大小  
2. 再隔离存储和访问路径  
3. 最后加扫描和审计

**易错点**  
- 只校验扩展名  
- 上传目录与可执行目录混用  
- 下载链路无权限控制

## 10. 审计日志要记录哪些字段？

::: tip L2 深度训练
记录主体、资源、动作、策略版本、结果与原因，同时保护日志本身：[身份、授权与租户隔离](/deep-dives/security-compliance/01-identity-authorization)
:::

**核心结论**  
审计日志的关键不是记录得多，而是能在事后准确回答“谁、在什么时候、对什么对象、做了什么、结果如何”。

**标准答案**  
我设计审计日志时，通常会固定几个核心字段：操作者身份、租户或组织、时间、来源 IP/设备、目标资源、操作类型、前后状态摘要、结果、失败原因、traceId 和审批关联信息。之所以审计日志重要，是因为很多安全和合规问题都要靠它还原责任链和事件链。工程上对于高风险操作，还要保证日志不可篡改、可长期保留、支持检索和导出。面试里如果能补一句“审计日志的本质是为责任追溯和合规取证服务”，会比较成熟。

**高分回答**
我从取证问题反推 Schema：`who` 包含不可复用的主体 ID、主体类型、租户与代理/代操作关系；`what` 包含资源类型和 ID、动作、策略/权限版本；`when/where` 包含可信服务端时间、请求/trace ID、来源网络与设备上下文；`result` 包含允许/拒绝、原因码、审批/工单和变更 ID。对状态变更记录字段级差异摘要或旧/新值的安全哈希，敏感原值、密码和完整 token 不进入日志。事件在执行点产生并带业务结果，避免网关记录“请求通过”却不知道落库是否成功。传输使用认证加密，写入独立受限存储，采用追加写、WORM 或签名链提升篡改可见性；生产管理员不能同时修改业务数据和删除审计证据。时间同步、去重、丢失检测、跨区域保留和 legal hold 都需设计。查询与导出审计日志本身也要审计，保留期按法规与数据最小化分层。通过高风险动作覆盖率、字段完整率、端到端到达延迟、丢失率、检索耗时和篡改演练验证可用性。

**追问点答案**  
- 为什么前后状态摘要重要？  
因为很多操作要知道改了什么，而不仅是做了操作。  
- 审计日志和普通业务日志区别是什么？  
审计日志更强调完整、可靠和责任追踪。  
- 审计日志能删吗？  
按法规、合同、调查保全和数据最小化策略到期处置，任何提前删除都需授权且留下独立证据；“不可篡改”不等于无限期保留。

**原理补充**  
审计日志本质上是为关键动作建立一条不可轻易否认的事实链。

**架构扩展**  
成熟架构师设计审计体系时，不会只关注字段完整性，还会继续讲存储隔离、权限控制、保留周期和检索能力，这才是完整的审计观。

**工程指标**  
- 审计覆盖率：建设指标。  
- 缺字段率：质量指标。  
- 高风险操作审计完整率：关键合规指标。  
- 审计检索时长：取证效率指标。  
- 审计日志篡改事件数：底线风险指标。

**证据链**  
- 审计字段规范  
- 日志样本  
- 存储与权限配置  
- 合规审查记录

**低风险优化路径**  
1. 先定义关键动作和必填字段  
2. 再统一接入和存储  
3. 最后做不可篡改与检索能力建设

**易错点**  
- 只记录“操作成功/失败”  
- 缺少资源对象和前后状态  
- 审计日志和普通日志混存混权

## 11. 如何做权限最小化治理？

::: tip L2 深度训练
补齐工作负载身份、短期提权、休眠权限与撤权传播：[身份、授权与租户隔离](/deep-dives/security-compliance/01-identity-authorization)
:::

**核心结论**  
权限最小化治理的核心不是一开始配得多严格，而是持续让账号和服务只保留当前确实需要的最小权限集。

**标准答案**  
我做最小权限治理时，通常会先盘点人和服务账号的实际访问需求，再基于角色、租户、环境和资源类型做授权模板，然后定期回收不用的权限。之所以最小权限难，是因为系统一旦跑起来，权限往往只加不减，最后人人都有管理员。工程上比较稳的做法是权限申请、审批、过期、定期复查和高权限告警一起上。面试里如果能补一句“最小权限不是一次性配置，而是持续收缩过程”，会比较成熟。

**高分回答**
我先建立人、服务、CI/CD、机器人和第三方身份资产，关联 owner、用途、环境、最近使用与实际访问动作；共享账号优先拆分，长期密钥改为工作负载身份和短期凭证。基础权限通过岗位模板授予，生产高权采用 JIT：基于工单审批、限定资源与命令、自动到期并全程审计，break-glass 账号独立告警且事后复核。撤权要覆盖离职、转岗、服务下线和依赖变更，并验证从身份源到网关、策略缓存和目标系统的传播时延。对 90 天未使用权限先影子告警、通知 owner，再自动回收，降低一次性“大扫除”造成停机。关键职责做 SoD，例如开发者不能独自修改生产数据并删除审计。季度 recertification 由资源 owner 确认而非安全团队代签。指标包括高权与无 owner 身份、休眠权限、永久凭证、临时权限自动回收、撤权 SLA 和因权限不足导致的生产事件，在降低爆炸半径和保持可运维之间校准。

**追问点答案**  
- 为什么权限总是只加不减？  
因为回收权限需要确认影响，组织上常常容易拖延。  
- 服务账号也要最小权限吗？  
更要，因为它们一旦泄露往往影响更大。  
- 临时权限怎么处理？  
采用 JIT 授权，绑定工单、资源范围、用途和最短有效期，自动回收并记录使用行为；续期重新审批，不能把“临时”权限转成无期限常驻。

**原理补充**  
最小权限本质上是在降低“账号被滥用或泄露后能造成的最大损害”。

**架构扩展**  
成熟架构师做权限治理时，不会只看用户角色，还会同时看服务账号、CI/CD 账号、运维工具账号，因为真正高风险的往往是这些非人类身份。

**工程指标**  
- 高权限账号数量：风险指标。  
- 过期未回收权限占比：治理漏洞指标。  
- 临时权限自动回收率：成熟度指标。  
- 越权访问事件数：结果指标。  
- 权限复查完成率：持续治理指标。

**证据链**  
- 权限清单  
- 审批与回收记录  
- 高权限账号审计  
- 越权告警

**低风险优化路径**  
1. 先盘点高风险账号  
2. 再引入到期和回收机制  
3. 最后做定期复查和告警

**易错点**  
- 只治理人账号不治理服务账号  
- 权限申请有流程，回收没流程  
- 高权限长期无人审查

## 12. 多租户数据隔离怎么设计？

::: tip L2 深度训练
不要信任请求 tenantId，在服务与查询层强制对象范围：[身份、授权与租户隔离](/deep-dives/security-compliance/01-identity-authorization)
:::

**核心结论**  
多租户隔离的关键不是用了哪种模式，而是确保任何一层出现缺陷时都不容易跨租户读写数据。

**标准答案**  
我设计多租户隔离时，通常先看租户规模、合规要求和风险等级，再选择共享库共享表、共享库分表、独立 schema 或独立库等模式。之所以要慎重，是因为多租户问题一旦出错，往往直接升级成数据泄露事故。工程上除了存储层隔离，还要补应用层租户上下文、查询条件约束、缓存 key 隔离、对象存储路径隔离和日志脱敏。面试里如果能补一句“多租户隔离不是数据库设计问题，而是全链路边界问题”，会比较成熟。

**高分回答**
我先按租户敏感度、规模、监管、定制和 noisy-neighbor 风险选择共享表、独立 schema、独立库或独立部署，支持高风险租户升级隔离等级。租户上下文只能从已认证身份和服务间可信声明派生，不能信任请求体里的 `tenantId`；数据访问层自动追加租户谓词或使用数据库 RLS，主键/唯一索引也包含租户维度，后台任务显式携带并校验上下文。缓存 key、消息 topic/header、搜索索引、对象路径、向量库、导出与备份恢复都需隔离；跨租户管理操作走独立权限、审批和审计。共享资源还要按租户配额与并发隔离，避免一个租户耗尽连接池或消息容量。测试采用随机租户 ID、属性测试和负向用例，尝试读取、更新、批量导出及异步消费其他租户数据；生产对响应 tenant 与身份 tenant 不一致、无上下文查询和跨租户扫描告警。恢复演练必须证明只恢复目标租户且不会覆盖邻居数据。隔离越强成本越高，选择依据是风险与合规，不把独立库当万能答案。

**追问点答案**  
- 为什么只靠 where tenant_id 不够？  
因为开发者可能漏写条件，异步任务还可能丢失上下文；应在数据访问或数据库层强制，并把缓存、消息、搜索、对象存储、导出和恢复纳入同一边界。
- 独立库是不是一定最安全？  
隔离性更强，但成本和运维复杂度也更高。  
- 怎么快速发现跨租户风险？  
通过审计、自动化测试和高风险查询扫描。

**原理补充**  
多租户隔离本质上是在共享基础设施上人为建立清晰的数据边界。

**架构扩展**  
成熟架构师做多租户设计时，不会停留在“tenant_id 字段”层面，而是会继续讲缓存、对象存储、权限、审计、导出和测试数据隔离。

**工程指标**  
- 跨租户访问事件数：核心风险指标。  
- 多租户隔离测试通过率：质量指标。  
- 高风险查询未带租户条件占比：代码治理指标。  
- 租户级缓存串扰事件数：运行风险指标。  
- 隔离架构成本：设计权衡指标。

**证据链**  
- 数据模型  
- 访问控制规则  
- 自动化测试报告  
- 导出和缓存日志

**低风险优化路径**  
1. 先选合适隔离模式  
2. 再在缓存、存储、导出等链路补边界  
3. 最后用自动化测试持续验证

**易错点**  
- 只看数据库隔离  
- 忽略缓存和对象存储  
- 不做串租户专项测试

## 13. 安全漏洞响应流程怎么建？

::: tip L2 深度训练
串联证据保护、隔离、根除、可信恢复和同路径复测：[安全事件响应与 AI 边界](/deep-dives/security-compliance/04-security-incident-ai)
:::

**核心结论**  
漏洞响应流程的关键不是收到漏洞后大家都很紧张，而是有明确的分级、止血、修复、验证和通报闭环。

**标准答案**  
我建漏洞响应流程时，通常会先定义漏洞分级标准和响应时限，再明确谁负责初判、谁有权止血、谁推进修复、谁负责验证和通报。之所以流程很重要，是因为真正出问题时最怕信息混乱、决策分散。工程上漏洞响应一般会包括：接收与确认、影响评估、临时缓解、正式修复、回归验证、外部沟通和事后复盘。面试里如果能补一句“漏洞响应的核心是把不确定性快速收敛为可执行动作”，会比较有管理感。

**高分回答**
我先区分“发现漏洞但未利用”和“已有入侵证据”，分级同时看可利用性、外网暴露、数据/权限影响、在野利用与业务关键性。接报后指定事件指挥、技术 owner、取证和沟通角色，先保护日志、镜像、时间线与受影响资产，避免重启或清理销毁证据。止血可以下线入口、封禁凭证、关闭功能或加虚拟补丁，但要评估攻击者是否已持久化；随后确定受影响版本、数据、租户和时间窗，修复根因、轮换相关 secret、清除持久化并从可信制品恢复。验证不仅扫描补丁，还用原利用路径和横向移动假设复测，完成数据完整性、监控与业务恢复确认。法务、隐私、客户和监管通知依据事实与时限统一出口，避免研发自行对外承诺。复盘追踪根因控制、同类资产横向排查和行动项验收。指标使用确认/止血/修复时间、暴露窗口、资产覆盖、复测通过、复发率和证据完整率，不以“快速关闭工单”代替可信恢复。

**追问点答案**  
- 为什么漏洞也要分级？  
因为不同风险等级需要不同响应速度和资源投入。  
- 临时缓解和正式修复有什么区别？  
缓解是缩小可利用面或影响，如关功能、限访问；正式修复消除根因并处理已受影响资产。缓解必须有 owner、到期时间和残余风险，不能永久替代修复。
- 为什么通报也重要？  
因为上下游和业务方需要知道风险和处置状态。

**原理补充**  
漏洞响应流程本质上是在安全事件中建立一条快速、清晰、可协调的处置链。

**架构扩展**  
成熟架构师不会把漏洞响应完全丢给安全团队，而是会让研发、运维、业务和法务都知道自己在不同等级漏洞中的角色。

**工程指标**  
- 平均响应时长：流程效率指标。  
- 高危漏洞修复时长：核心结果指标。  
- 漏洞复发率：治理效果指标。  
- 临时缓解覆盖率：止损能力指标。  
- 通报完整率：协作质量指标。

**证据链**  
- 漏洞工单  
- 分级规则  
- 止血与修复记录  
- 通报与复盘文档

**低风险优化路径**  
1. 先定义分级和时限  
2. 再明确角色与审批链  
3. 最后通过演练和复盘持续打磨

**易错点**  
- 没有统一分级  
- 只修代码不做通报和复盘  
- 角色责任不清

## 14. 如何做依赖组件漏洞治理？

::: tip L2 深度训练
结合可达性、暴露面、业务影响、SBOM 和来源证明排序：[供应链安全与威胁建模](/deep-dives/security-compliance/03-supply-chain-threat-modeling)
:::

**核心结论**  
依赖漏洞治理的核心不是看到 CVE 就全量升级，而是按可利用性、影响面和修复风险做优先级排序。

**标准答案**  
我做依赖组件漏洞治理时，通常先确认三件事：这个漏洞是否真的在当前使用路径上可达、影响范围有多大、修复升级会不会引入兼容性风险。之所以不能机械地“见洞就升”，是因为有些漏洞虽然评级高，但在当前系统里不可触发；也有些漏洞评级一般，但路径极常用且暴露在外网。工程上比较稳的方式是持续做依赖清单、漏洞扫描、版本管理和灰度升级，同时对高危组件准备快速替换和回滚方案。面试里如果能补一句“依赖漏洞治理本质上是风险优先级管理”，会比较成熟。

**高分回答**
我先让每个构建生成可追溯 SBOM，包含直接、传递依赖、运行镜像、来源、版本与 owner，并将制品和源码 commit 关联。风险排序不只看 CVSS，还看生产是否部署、漏洞代码是否可达、是否公网暴露、所需权限、在野利用/EPSS、数据影响和补偿控制；不可达结论也需证据与复核日期。高危可利用项立即隔离入口、禁用功能或加 WAF 规则争取时间，同时在兼容矩阵中升级最小安全版本，跑功能、性能和安全回归，灰度并保留镜像回退。若上游无补丁，评估替换、隔离或维护受控 fork，并设置风险接受的审批与到期时间。供应链还需锁定版本和哈希、验证签名/来源、最小化构建权限与可复现构建，防止“升级到恶意最新版”。度量资产/SBOM 覆盖、可利用高危的 MTTR、例外逾期、来源验证、升级失败与相同组件重复引入率；避免用扫描总数驱动无意义升级。

**追问点答案**  
- 为什么 CVSS 高不一定优先级最高？  
因为还要看可利用性和实际暴露面。  
- 为什么要做依赖清单？  
没有资产清单就无法知道问题范围。  
- 升级失败怎么办？  
先回退到已签名的上一安全制品并保留临时缓解；若旧版同样有高风险漏洞，应限制暴露、替换组件或接受经审批的短期残余风险，而不是无条件回滚后关闭事件。

**原理补充**  
依赖漏洞治理本质上是在“安全风险”和“变更风险”之间做动态平衡。

**架构扩展**  
成熟架构师面对依赖漏洞，不会只看扫描平台红色数量，而是会继续问：哪些暴露在公网、哪些跑在核心链路、哪些有可用 exploit、哪些升级风险高。这才是有判断力的治理。

**工程指标**  
- 高危依赖漏洞存量：风险规模指标。  
- 漏洞修复时长：治理效率指标。  
- 依赖清单覆盖率：资产可见性指标。  
- 升级回滚率：变更风险指标。  
- 漏洞复发率：长期治理指标。

**证据链**  
- SBOM / 依赖清单  
- 漏洞扫描报告  
- 升级验证记录  
- 回滚和兼容问题复盘

**低风险优化路径**  
1. 先做依赖资产清单  
2. 再按风险排序治理  
3. 最后把升级和回滚纳入标准流程

**易错点**  
- 扫描一堆洞但没有排序  
- 只看漏洞等级不看暴露面  
- 升级没有验证和回滚准备

## 15. 数据删除与合规要求如何满足？

::: tip L2 深度训练
覆盖主库、缓存、索引、日志、向量库、第三方与备份：[数据保护与生命周期](/deep-dives/security-compliance/02-data-lifecycle)
:::

**核心结论**  
数据删除合规的难点不是删主库一条记录，而是确认副本、缓存、索引、备份和日志链路都符合保留与删除要求。

**标准答案**  
我满足数据删除合规要求时，通常会先明确法规和业务规则要求的是“逻辑删除、物理删除、匿名化”中的哪一种，再把主库、缓存、ES 索引、对象存储、备份和日志一起纳入删除链路。之所以这件事难，是因为数据一旦在多个系统里扩散，删掉主表并不等于真的删除完成。工程上还要保留删除审计、失败重试和结果确认机制，避免“看起来删了，实际还能查到”。面试里如果能补一句“合规删除本质上是跨存储副本的数据生命周期治理”，会比较成熟。

**高分回答**
我从数据地图和处理目的开始，为每类数据定义法定依据、最短/最长保留、删除或匿名化方式、legal hold 与 owner。收到删除请求先验证主体和范围，生成不可复用的 deletion ID，通过编排任务传播到主库、缓存、搜索、对象存储、分析仓库、向量库、消息重放区和第三方处理者；每个处理器幂等、可重试并返回证明，避免部分成功。备份通常不能逐条修改，因此限制访问、缩短保留并确保恢复流程在数据回放后重新应用删除 tombstone；若法律要求不同，应由法务确认而非技术自行承诺。审计只保留证明所需的最小元数据和安全哈希，不能为了证明删除继续保存被删原文。匿名化必须评估与其他字段组合后的重识别风险，逻辑删除也不能继续参与推荐或导出。以删除传播 P95、逾期和失败队列、第三方确认、恢复后复删演练、删除后查询抽样及 DSAR SLA 验收，并对 legal hold 的建立和解除独立审计。

**追问点答案**  
- 为什么逻辑删除不总是够？  
因为有些法规要求数据真正不可恢复。  
- 备份中的数据怎么处理？  
要有独立策略，通常结合保留周期和恢复约束。  
- 删除和审计会冲突吗？  
可能冲突。应仅保留证明动作所必需的主体替代标识、时间和结果，避免保留原始敏感内容；法定审计保留与删除权的优先关系需由适用法规和 legal hold 明确。

**原理补充**  
数据删除本质上是在多个数据副本和法律要求之间收敛出一套可执行的一致策略。

**架构扩展**  
成熟架构师不会把数据删除只当成“加个 delete 接口”，而是会继续追踪缓存、搜索、导出、备份、对象存储和日志。真正难的是全链路一致。

**工程指标**  
- 删除完成率：执行指标。  
- 删除传播时长：全链路收敛指标。  
- 删除失败重试率：稳定性指标。  
- 合规审计通过率：结果指标。  
- 删除后仍可查事件数：核心风险指标。

**证据链**  
- 数据生命周期策略  
- 删除任务日志  
- 缓存/索引同步记录  
- 合规审计报告

**低风险优化路径**  
1. 先明确删除类型和法规要求  
2. 再梳理所有数据副本  
3. 最后做删除审计和验证

**易错点**  
- 只删主库  
- 不管搜索和缓存副本  
- 无审计无法证明已删除

## 16. 安全与性能冲突怎么平衡？

**核心结论**  
安全与性能冲突时，不能简单二选一，关键是按风险等级把最重的安全措施用在最该保护的链路上。

**标准答案**  
我平衡安全和性能时，通常会先按业务风险和数据敏感度分级，再决定哪些链路必须启用强校验、强加密、强审计，哪些链路可以用更轻量的方式。之所以不能为了性能把安全全关掉，也不能为了安全把所有链路都套最重机制，是因为两者都要服务业务目标。工程上常见做法包括高敏操作二次校验、普通读取走轻量验证、冷热数据不同加密策略、热点接口局部缓存授权结果等。面试里如果能补一句“安全和性能冲突本质上是风险与体验的分级治理问题”，会比较成熟。

**高分回答**
我先把“冲突”量化：在目标峰值下，TLS、授权、审计或风控分别增加多少 P99/CPU，同时它们降低哪种威胁和合规风险。身份校验、对象级授权、数据完整性等底线控制不能因性能直接关闭，应优化实现：连接复用和硬件加速降低加密成本，策略本地只读副本与短 TTL 缓存降低远程授权延迟，审计采用有界异步管道但高风险写入失败时按策略 fail closed。缓存授权结果必须把主体、资源、动作、租户和策略版本纳入 key，并明确撤权最大窗口；不能只按用户角色粗粒度缓存。风控采用分层挑战，把强校验集中到高风险请求，普通请求走低成本特征。任何异步或采样都要明确不能丢的数据，不能用性能名义采样掉安全事件。以 P99、有效吞吐、误拒/误放、撤权时延、审计丢失和攻击模拟结果联合验收；若仍不达标，增加专用容量或调整用户流程，而不是悄悄削弱控制。

**追问点答案**  
- 为什么高敏链路值得更高开销？  
因为一旦出问题损失更大。  
- 授权结果缓存为什么有风险？  
它会延长撤权和策略更新生效时间，还可能因 key 缺少资源、租户或策略版本发生串权；应短 TTL、事件失效并对高风险动作重新实时决策。
- 怎么说服业务接受性能损耗？  
要换算成风险降低和合规收益。

**原理补充**  
安全与性能平衡本质上是在不同风险等级场景下分配不同强度的控制成本。

**架构扩展**  
成熟架构师不会一提安全就说“肯定影响性能”，而是会继续问：影响在哪、值不值、能不能按链路分层、能不能通过缓存或异步降低代价。

**工程指标**  
- 安全链路 TP99：体验成本指标。  
- 高风险操作防护覆盖率：安全收益指标。  
- 误拒/误放率：策略质量指标。  
- 授权缓存命中率：优化效果指标。  
- 风险事件下降幅度：最终结果指标。

**证据链**  
- 风险分级文档  
- 性能对比数据  
- 安全事件统计  
- 授权与校验策略配置

**低风险优化路径**  
1. 先按风险分级  
2. 再给高风险链路上重防护  
3. 最后对中低风险链路做性能优化

**易错点**  
- 所有链路同一套重防护  
- 为了性能牺牲底线安全  
- 不做风险分级

## 17. 如何做风控规则架构？

**核心结论**  
风控规则架构的关键不是规则越多越好，而是让规则能快速迭代、可解释、可灰度、可审计。

**标准答案**  
我做风控规则架构时，通常会把规则执行引擎、规则配置、特征数据、决策结果、实验灰度和人工审核接口分开设计。之所以要这么拆，是因为风控规则变化快，如果代码和规则强绑定，迭代会非常慢。工程上成熟的风控架构通常支持规则版本、命中解释、灰度试验、租户差异化和误伤回溯。面试里如果能补一句“风控架构不是写 if-else，而是决策系统工程”，会比较成熟。

**高分回答**
我把在线决策拆为事件接入、实时/离线特征、规则与模型编排、决策、动作和反馈。特征必须有定义、owner、时间戳、缺失值和数据新鲜度，线上线下使用同一口径；规则版本不可变，发布前做历史回放，先 shadow 只记录决策，再按 cohort 灰度，异常可一键回退。决策输出不仅是 allow/deny，还包括原因码、证据特征版本、置信度和建议动作，支持申诉、人工审核与审计；敏感特征最小化收集并限制用途。延迟或特征服务故障时按风险定义 fallback：支付等高风险可能转人工或保守拒绝，低风险浏览可降级通过，不能全局默认同一结果。规则冲突有明确优先级和组合方式，设置全局 kill switch、防循环调用与单用户决策幂等。反馈闭环将确认欺诈、拒付、申诉和人工结论回灌，监控 precision/recall、损失金额、误伤、人工队列、决策 P99、特征漂移和规则边际贡献，及时淘汰重复无效规则。

**追问点答案**  
- 为什么规则要可解释？  
因为业务申诉、人工审核和合规都需要知道为什么被拦。  
- 为什么风控要支持灰度？  
因为真实分布和离线样本可能不同；应先 shadow 对比，再小流量灰度，联合观察漏拦、误伤、申诉和业务转化，并保留按规则版本快速回退。
- 特征数据为什么要单独治理？  
因为风控效果很大程度取决于特征质量。

**原理补充**  
风控规则架构本质上是在实时风险判断和持续策略迭代之间建立一套可控闭环。

**架构扩展**  
成熟架构师做风控时，不只谈“拦不拦”，还会谈命中率、误伤率、规则回溯、人工介入和在线实验。这说明理解的是系统而不是单条规则。

**工程指标**  
- 规则命中率：效果指标。  
- 误伤率：体验成本指标。  
- 规则灰度成功率：迭代能力指标。  
- 人工复核率：策略成熟度指标。  
- 风险事件漏拦率：核心安全结果指标。

**证据链**  
- 规则引擎配置  
- 命中日志  
- 灰度实验记录  
- 申诉与人工审核数据

**低风险优化路径**  
1. 先拆引擎、规则和特征数据  
2. 再补灰度与解释能力  
3. 最后形成命中与误伤闭环

**易错点**  
- 规则写死在代码里  
- 无灰度直接全量启用  
- 不追踪误伤和申诉

## 18. 如何做账号体系防撞库？

**核心结论**  
防撞库的关键不是只加验证码，而是识别异常登录模式并分层提高攻击成本。

**标准答案**  
我做账号体系防撞库时，通常会从账号、设备、IP、行为模式和结果反馈几层一起做。工程上常见手段包括登录频率限制、异常设备识别、IP/ASN 信誉、验证码、二次验证、登录失败熔断、异地登录提醒和密码泄露库比对。之所以撞库难防，是因为攻击者往往不追求高频爆破，而是模拟正常节奏批量试。面试里如果能补一句“撞库问题本质上是批量弱凭证验证被异常自动化利用”，会比较准确。

**高分回答**
我先让认证本身抗泄露：密码使用自适应慢哈希与独立盐，注册/改密时检查已泄露密码，支持 MFA/passkey，并避免响应暴露账号是否存在。检测同时看 IP/ASN、设备、账号、失败用户名基数、成功后行为和跨地域速度；撞库常是“单 IP 低速、整体高基数”，只按 IP 限流会漏掉。风险分层后，低风险正常登录无感，高风险要求 MFA/验证码、限速或暂缓；不要因少量失败永久锁死账号，否则攻击者可制造拒绝服务。命中可疑成功时撤销会话、保护高危操作、通知用户并提供安全恢复，恢复流程本身也需防冒用。使用泄露凭证比对时采用保护隐私的查询方式，不上传完整密码。规则故障时维持基础限额和 MFA，风控不能成为认证单点。通过历史攻击回放和受控演练衡量账号接管率、可疑成功确认率、误伤、挑战完成率、恢复时长和正常登录 P99，而不是只报拦截次数。

**追问点答案**  
- 为什么验证码不能解决所有撞库？  
因为攻击者可以分布式尝试，也可能通过打码平台绕过。  
- 为什么要看设备和 IP 行为？  
因为撞库往往在行为模式上和正常用户明显不同。  
- 密码泄露库比对有什么价值？  
可在注册、改密和风险登录时阻止已知泄露密码，但查询方式要保护用户输入，且不能替代慢哈希、MFA、行为检测和会话处置。

**原理补充**  
防撞库本质上是在认证链路中不断提高自动化攻击成本，并尽量减少对正常用户的影响。

**架构扩展**  
成熟架构师谈防撞库，不会只说验证码，而是会继续讲行为检测、分层挑战、风控和通知联动。这才是现实世界里的打法。

**工程指标**  
- 异常登录拦截率：核心防护指标。  
- 误伤率：用户体验指标。  
- 撞库成功率：最终风险指标。  
- 登录挑战触发率：策略执行指标。  
- 账号异常申诉率：副作用指标。

**证据链**  
- 登录行为日志  
- 风控规则  
- 验证码和挑战记录  
- 安全事件复盘

**低风险优化路径**  
1. 先识别异常登录特征  
2. 再按风险级别提升挑战  
3. 最后联动通知和密码治理

**易错点**  
- 全靠验证码  
- 不区分正常与异常行为  
- 不做异常登录提醒

## 19. 如何做高危操作二次确认？

**核心结论**  
高危操作二次确认的重点不是多弹一个框，而是确保真正的操作者、真正理解风险并留下可审计记录。

**标准答案**  
我做高危操作二次确认时，通常会先界定哪些操作算高危，比如删库、提权、批量导出、支付确认、重要配置变更，然后再按场景设计确认方式，比如再次输入密码、短信/OTP、审批流、双人确认或延时生效。之所以不能只做一个前端确认弹窗，是因为真正要防的是误操作、账号被盗用和越权操作。工程上更稳的做法是把二次确认和审计日志、权限校验、风控规则一起联动。面试里如果能补一句“高危操作确认的本质是把高损失动作的执行门槛提高”，会比较成熟。

**高分回答**
我先按不可逆性、影响范围、数据敏感度和欺诈损失维护高危动作目录，例如批量删除、提权、密钥导出、付款与跨租户导出。执行前重新做对象级授权，并使用与当前操作绑定的 step-up 挑战：确认页面展示具体资源、数量、金额和不可逆后果，后端签发短时、单次、绑定主体/动作/参数的确认凭证，防止攻击者把“确认 A”重放为“执行 B”。仅弹窗不能证明身份；高风险场景用抗钓鱼 MFA、双人审批或职责分离，审批人不能与申请人同一主体。延时生效与撤销窗口适合批量删除等操作，支付等流程则依赖幂等和交易签名。执行后生成不可篡改审计、通知相关 owner，并验证实际结果；审批服务不可用时默认不执行，break-glass 有独立强审计。测试参数替换、重放、审批人撤权、并发执行和回滚失败，指标关注未确认执行、误操作拦截、审批时延、绕过尝试和确认后参数变更。

**追问点答案**  
- 为什么简单弹窗不够？  
因为它既不能确认身份，也很难形成责任链。  
- 双人确认适合什么场景？  
特别高风险、不可逆、影响面大的操作。  
- 为什么延时生效也有价值？  
它为误操作、账号失陷告警和人工复核提供撤销窗口，适合可排队的不可逆动作；紧急止血不应被固定延时阻断，需要受控 break-glass。

**原理补充**  
高危操作二次确认本质上是在高损失动作前增加额外的身份和意图验证。

**架构扩展**  
成熟架构师设计高危操作防护时，不会只看 UI，而会继续讲：权限、身份、审计、审批、回滚和延时撤回是不是都打通了。

**工程指标**  
- 高危操作二次确认覆盖率：建设指标。  
- 未经确认执行事件数：底线风险指标。  
- 审批通过时长：效率指标。  
- 误操作拦截率：价值指标。  
- 高危操作审计完整率：合规指标。

**证据链**  
- 高危操作清单  
- 二次确认流程  
- 审计日志  
- 审批与回滚记录

**低风险优化路径**  
1. 先定义高危动作清单  
2. 再按风险设计确认方式  
3. 最后打通审计和回滚机制

**易错点**  
- 只做简单弹窗确认  
- 无审计无审批  
- 不区分动作风险等级

## 20. 如何做安全基线检查自动化？

::: tip L2 深度训练
从源码、依赖、构建、制品、部署到运行时建立分层门禁：[供应链安全与威胁建模](/deep-dives/security-compliance/03-supply-chain-threat-modeling)
:::

**核心结论**  
安全基线自动化的关键不是扫描工具多，而是把关键检查嵌进开发、发布和运行时流程里，形成持续守门能力。

**标准答案**  
我做安全基线检查自动化时，通常会先挑最稳定、最通用、最值得自动化的规则，比如弱口令、明文密钥、TLS 配置、依赖漏洞、容器镜像基线、权限过宽和高危端口暴露，然后分别接入代码扫描、CI、镜像构建、发布前检查和运行时巡检。之所以自动化重要，是因为人工检查既不稳定也不可持续。工程上成熟的做法是把基线检查分成阻断项和观察项，避免一上来就把流水线全部卡死。面试里如果能补一句“安全基线自动化的目标是把低级高频错误前置消灭”，会比较成熟。

**高分回答**
我按源码、依赖、构建、制品、部署和运行时设计分层控制。提交阶段检测 secret、危险 API 与 IaC 错配；构建阶段生成 SBOM、锁定依赖并签名制品；发布阶段验证签名、镜像用户、权限、网络暴露和策略；运行时持续检查配置漂移、过宽身份与未授权资产。规则即代码，带 owner、风险说明、测试样例、适用范围和版本，先在存量系统观察建立误报基线，新引入的严重问题立即阻断，存量按 SLA 治理，避免一次上线让全公司无法发布。阻断依据应是可利用性和业务风险，例外必须审批、补偿控制、到期和自动提醒，不能形成永久白名单。平台结果关联到具体资产和修复建议，重复问题推动安全默认模板而不是逐个工单。还要定期模拟扫描器失效、签名验证绕过和运行时漂移，保证控制真的在路径上。衡量覆盖的真实资产、门禁绕过、严重问题逃逸、误报处理时间、例外逾期、修复 SLA 和复发率，而非扫描次数。

**追问点答案**  
- 为什么要分阻断项和观察项？  
因为并不是所有问题都值得立即阻断发布。  
- 运行时巡检为什么也重要？  
因为很多风险是上线后通过配置漂移或临时变更引入的。  
- 自动化会不会误报很多？  
会。应以测试样例、适用上下文、灰度观察和开发反馈校准；高误报规则先降为观察，不能让团队习惯无脑绕过所有安全门禁。

**原理补充**  
安全基线自动化本质上是在系统生命周期各阶段建立低成本、可重复的风险守门点。

**架构扩展**  
成熟架构师推动基线自动化时，不会幻想一步到位，而是会先从高收益低争议规则开始，逐步扩大覆盖面。这种推进方式更容易真正落地。

**工程指标**  
- 基线检查覆盖率：建设进度指标。  
- 阻断问题数量：前置收益指标。  
- 误报率：规则质量指标。  
- 运行时漂移检出率：持续治理指标。  
- 基线违规复发率：长期成效指标。

**证据链**  
- 基线规则清单  
- CI 和巡检结果  
- 阻断记录  
- 误报复盘与规则调整记录

**低风险优化路径**  
1. 先挑高收益规则自动化  
2. 再分阻断与观察  
3. 最后把运行时巡检接进来

**易错点**  
- 想一次性自动化所有规则  
- 不分阻断项和观察项  
- 上线后不持续巡检
