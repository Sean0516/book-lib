# RAG 质量闭环

## 90 秒速答

我先定义 RAG 要解决的业务任务、知识时效和无答案策略，再把链路拆成摄取、切块、召回、重排、
上下文组装、生成与引用。离线分别评估检索 Recall@K、MRR/nDCG，以及答案正确性、忠实度、
引用覆盖和拒答准确率，不能只看最终回答。线上观察任务成功率、无答案率、引用点击、延迟和单次
成功成本。失败样本按“知识缺失、权限过滤、查询理解、召回、重排、生成”归因，修复后进入
版本化回归集；低置信度时拒答、澄清或转人工，而不是强行生成。

## 先定义事实边界

RAG 适合知识频繁变化、需要私有数据或必须引用溯源的任务。若任务主要靠复杂推理、知识稳定且
上下文很短，直接提示或结构化查询可能更简单。文档入库前要确定事实源、版本、权限、有效期和
删除传播，避免向量库成为无法治理的数据副本。

## 分层指标树

| 层 | 指标 | 回答的问题 |
| --- | --- | --- |
| 摄取 | 新鲜度、解析成功率、权限同步延迟 | 知识是否完整可用 |
| 召回 | Recall@K | 证据有没有进入候选集 |
| 排序 | MRR、nDCG | 正确证据是否排在前面 |
| 上下文 | 有效 token 比、重复率 | 给模型的证据是否干净 |
| 生成 | 正确性、忠实度、引用覆盖 | 回答是否由证据支持 |
| 产品 | 任务成功率、转人工率 | 用户问题是否解决 |
| 运营 | TP99、单次成功成本 | 系统能否持续运行 |

Recall@K 高但答案差，可能是重排、上下文噪声或生成问题；答案看似正确但引用不支持，则是忠实度
问题。只有最终总分无法指导修复。

## 检索链路选择

```mermaid
flowchart LR
  Q["用户问题"] --> A["权限与租户过滤"]
  A --> R["查询改写"]
  R --> H["BM25 + 向量召回"]
  H --> X["融合去重"]
  X --> E["重排"]
  E --> C["上下文组装"]
  C --> G["生成、引用、置信降级"]
```

- 专有名词、编号和精确短语较多时，BM25 很重要。
- 语义改写和同义表达较多时，向量召回更有价值。
- 混合检索提高覆盖后再重排，代价是延迟与成本。
- 权限过滤应进入检索条件，不能先召回敏感内容再靠 Prompt 隐藏。

## 切块不是固定字符数

按标题、段落、表格和代码等语义边界切块，保留父文档和章节关系。块太小会丢上下文，太大会
降低召回精度并浪费 token。用问题—证据集评测不同 chunk size、overlap 和父子检索策略，
以召回、答案质量、延迟和成本共同选择。

## 无答案和引用策略

无答案不是异常，而是必须设计的产品路径。可组合检索分数、重排分数、证据覆盖和生成校验设置
风险分级：低风险提供“基于现有资料”的有限回答；高风险领域证据不足时明确拒答或转人工。
引用必须定位到真实文档版本与片段，并在生成后验证关键主张是否被引用支持。

## 场景推演：知识库更新后正确率下降

观察：Recall@10 从 91% 降到 74%，生成模型未变；失败集中在新版产品文档，旧版引用比例升高。
先检查摄取位点、文档版本过滤和向量索引完成度，而不是直接更换大模型。止血可回滚索引别名、
提高新版精确检索权重；长期建立双索引切换、摄取完整性校验和版本新鲜度 SLI。

## 面试官三级追问

### L1：向量相似度高是否代表答案可信？

不代表。相似度只说明表示空间接近，还需验证权限、时效、事实支持、上下文冲突和生成忠实度。

### L2：如何评估没有唯一标准答案的问题？

拆成关键事实点、可接受表达和禁止错误，结合规则、模型评审与人工抽样；对评审器做一致性校准，
保留专家仲裁集，避免用一个模型分数代替真值。

### L3：如何防止删除文档后仍被检索？

删除事件携带文档版本进入摄取流水线，同步清理向量、关键词索引和缓存；记录位点并有删除传播
SLO。高敏感场景在检索时再次校验事实源权限与有效状态。

## 25 分自测

| 维度 | 5 分要求 |
| --- | --- |
| 正确性 | 能区分召回、排序、生成与产品指标 |
| 深度 | 覆盖版本、权限、无答案和引用验证 |
| 取舍 | 质量、延迟、成本和新鲜度权衡明确 |
| 表达 | 目标 → 分层指标 → 失败归因 → 修复 |
| 可运维性 | 有索引切换、回滚、漂移和删除传播 |
