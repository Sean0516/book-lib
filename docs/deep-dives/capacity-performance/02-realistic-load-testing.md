# 真实性压测与瓶颈证据

## 90 秒速答

压测先定义要验证的容量假设和通过标准，再复刻生产的接口比例、到达率、热点分布、数据规模、
缓存状态和依赖时延。优先用开放模型按目标速率发压，避免系统变慢后施压端也变慢而漏测排队。
按阶梯升压寻找吞吐不再线性增长、TP99 和队列突然恶化的拐点；同时采集业务、应用、运行时、
中间件和基础设施证据。每次只验证一个主要假设，优化后用同一基线回归，并做故障和恢复测试。

## 压测前先写验收合同

```text
场景：订单查询 70%、创建 20%、取消 10%，热点用户占流量 35%
目标：峰值 12,000 QPS 持续 30 分钟
通过：成功率 >= 99.95%，TP99 < 200ms，DB CPU < 60%，连接等待 P99 < 10ms
终止：数据差异 > 0、错误率 > 2% 持续 2 分钟、任一事实库水位 > 80%
```

没有明确假设和停止条件的压测，很容易变成“把 CPU 打高”的展示活动。

## 开放模型与闭环模型

- **开放模型**按外部速率持续到达，更接近用户不会因为服务变慢就自动减少点击的场景。
- **闭环模型**每个虚拟用户等上次完成再发下一次；系统越慢，发送越慢，可能掩盖过载。

还要防止协调遗漏（coordinated omission）：若只记录已完成请求的服务时间，排队和无法按时发出
的请求会从统计中消失，使 TP99 看起来过于漂亮。

## 五类最常见的失真

| 失真 | 错误结论 | 修正方法 |
| --- | --- | --- |
| 均匀 key | 没有热点问题 | 复刻 Zipf/头部流量分布 |
| 小数据集 | 索引和缓存都命中内存 | 使用接近生产的数据量与基数 |
| 永远热缓存 | 忽略启动和切换回源 | 测冷启动、预热和缓存失效 |
| Mock 下游 | 忽略真实连接与尾延迟 | 回放延迟分布并做故障注入 |
| 只压单接口 | 忽略资源争用 | 保留读写、任务和消息混合比例 |

## 用阶梯升压找拐点

每阶保持足够时间让队列、GC、缓存和数据库达到稳态，例如 30% → 50% → 70% → 85% → 100%
目标流量。记录吞吐、TP50/95/99、错误率、利用率和队列等待。

```mermaid
flowchart LR
  A["吞吐线性增长"] --> B["资源逼近水位"]
  B --> C["排队开始增长"]
  C --> D["TP99 陡升"]
  D --> E["超时、重试、吞吐下降"]
```

安全容量通常取拐点之前，并预留单 AZ 故障和预测误差，而不是取最后一个“还能返回”的点。

## 从症状到证据链

| 观察 | 候选假设 | 继续取证 |
| --- | --- | --- |
| CPU 高、吞吐不涨 | 热代码、GC、锁自旋 | 火焰图、GC 日志、锁事件 |
| CPU 低、线程满 | IO、连接池或下游等待 | 线程栈、池等待、依赖 trace |
| DB CPU 低但接口慢 | 锁、连接排队、网络 | 锁等待图、活跃会话、RTT |
| 只有 TP99 恶化 | 热点、暂停、排队、慢分片 | 分桶指标、trace、分片维度 |
| 吞吐周期性下降 | GC、批任务、限速 | 时间线对齐变更和任务 |

结论必须能被反证。例如“数据库慢”不是根因；需要说明是哪类等待、由哪条 SQL 或哪种负载触发，
并通过隔离变量或改动前后实验闭环。

## 性能优化实验怎么做

1. 固定代码、数据、机器、负载模型和预热条件。
2. 每次只改变一个主要变量，保留原始数据和版本标识。
3. 多轮运行报告中位结果与波动范围，不只挑最好的一次。
4. 同时比较吞吐、尾延迟、错误、资源和成本，防止指标转移。
5. 在故障、冷启动和恢复场景再次验证。

## 面试官三级追问

### L1：为什么压测 QPS 达标仍不能上线？

可能是数据量过小、热点缺失、缓存全热、下游被 Mock 或错误率和 TP99 已越线。必须验证负载
真实性、业务正确性、安全水位和故障冗余。

### L2：生产环境能否做全链路压测？

可以，但要有压测流量标识、数据隔离、影子写或清理方案、下游容量保护、逐级放量和停止开关。
核心不是“能不能”，而是爆炸半径和副作用是否受控。

### L3：两次压测结果相差 15% 怎么判断优化有效？

先检查环境、数据、预热、邻居噪声和负载生成器是否一致；运行多轮得到波动区间，再判断差异是否
超过自然噪声。没有重复性和置信区间的单次结果不能证明优化。

## 25 分自测

| 维度 | 5 分要求 |
| --- | --- |
| 正确性 | 流量模型、数据规模和验收目标完整 |
| 深度 | 理解开放模型、协调遗漏和容量拐点 |
| 取舍 | 真实性、风险、成本和测试效率有权衡 |
| 表达 | 假设 → 实验 → 证据 → 结论清晰 |
| 可运维性 | 有终止条件、数据隔离和结果留档 |
