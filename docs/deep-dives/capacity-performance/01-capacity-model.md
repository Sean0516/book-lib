# 容量模型与安全水位

## 90 秒速答

容量规划先从业务事件推导峰值到达率，再按接口扇出换算各下游流量；用 Little's Law 估算
在途并发，以压测得到单实例在目标 TP99、错误率和资源水位下的安全吞吐。实例数不能只做
峰值除法，还要考虑单 AZ 故障、发布、扩容时延和预测误差。线程池、连接池与队列分别受 CPU、
下游容量和允许排队时间约束，不能互相照抄。最后用分层告警、阶梯压测和生产数据持续校准。

## 从 DAU 推导峰值 QPS

假设 DAU 1,200 万，每人每天平均下单查询 8 次，35% 请求集中在晚高峰 2 小时，峰值分钟又是
高峰平均的 2.2 倍：

```text
日请求量 = 12,000,000 × 8 = 96,000,000
高峰平均 QPS = 96,000,000 × 35% / 7,200 ≈ 4,667
峰值 QPS = 4,667 × 2.2 ≈ 10,267
```

这只是入口流量。若每次请求平均查缓存 2 次、10% 回源数据库，则缓存约 20,534 QPS，数据库
约 2,053 QPS。还要叠加重试、定时任务、消息消费和运营活动。

## Little's Law 只给第一版并发

稳定系统近似满足：

```text
L = λ × W
```

入口 10,000 QPS、平均响应 40 ms，平均在途约 400；若下游劣化到 400 ms，在流量不变时在途
会升到 4,000。它说明为什么“流量没涨”也会耗尽线程和连接，但不能代替分布、尾延迟和压测。

## 从单实例安全容量推导实例数

压测显示单实例在 600 QPS 时满足 TP99 `<80 ms`、错误率 `<0.1%`、CPU `<60%`。目标峰值
10,267 QPS：

```text
业务所需实例 = ceil(10,267 / 600) = 18
若 3 AZ 且任一 AZ 故障后仍需低于安全水位，至少 ceil(18 × 3 / 2) = 27
再考虑发布批次和预测误差，可将常驻量设为 30，并用压测复核
```

不能直接用 CPU 100% 时的极限 QPS。安全容量要同时满足延迟、错误率、下游水位和故障冗余。

## 线程、连接和队列不是同一个参数

| 资源 | 主要约束 | 设计方法 |
| --- | --- | --- |
| CPU 线程 | 核数、计算时间、上下文切换 | CPU 密集通常接近核数，再用压测校准 |
| 阻塞并发 | 到达率、等待时间 | Little's Law 初估，但受下游安全容量封顶 |
| DB 连接 | DB 可承受活跃会话 | 全部实例连接总数必须反向校验数据库 |
| 队列 | 允许排队时延 | `队列量 ≈ 到达率 × 可接受排队时间` |

例如 10,000 QPS 最多允许排队 20 ms，平均排队量约 200。把队列设成 10 万只会让过期请求堆积，
并不能创造吞吐。

## 自动扩容为什么救不了短突发

如果监控采样、决策、拉起实例和预热合计需要 3 分钟，持续 20 秒的三倍突发不能依赖 HPA。
短突发应由预留容量、准入控制、缓存和有限队列处理；HPA 负责分钟级持续增长。扩容阈值除了
CPU，还应观察并发、队列等待、TP99、连接水位和业务流量预测。

## 安全水位要包含失败场景

| 水位 | 示例 | 动作 |
| --- | ---: | --- |
| 关注 | CPU 50% 或连接 55% | 校验趋势和预测 |
| 预警 | CPU 60% 或 TP99 逼近 SLO | 扩容、冻结非必要任务 |
| 紧急 | CPU 75% 或队列持续增长 | 限流、降级、升级值班 |

阈值必须按资源特性和历史拐点校准；数据库锁等待在 CPU 低时也可能已经不可用。

## 面试官三级追问

### L1：平均响应时间能否用来计算线程数？

只能得到稳态平均在途量。线程和并发上限还要考虑 TP99、突发、下游容量和排队目标，并通过
与生产分布一致的压测校准。

### L2：为什么连接池不能按每个实例最优值配置？

单实例配置乘以实例数可能超过数据库可承受会话。应先从数据库安全并发预算反向分配给服务、
租户和任务，再预留运维与故障恢复连接。

### L3：流量预测偏差 50% 怎么办？

显式记录预测区间，保留故障容量，设计限流降级和异步削峰；用生产峰值、请求构成和扩容事件
每周校准。不能用无限冗余掩盖模型不确定性，也不能只给单点数字。

## 25 分自测

| 维度 | 5 分要求 |
| --- | --- |
| 正确性 | 峰值、扇出、在途量和实例计算自洽 |
| 深度 | 区分极限容量、安全容量和故障容量 |
| 取舍 | 性能、成本、冗余和拒绝策略权衡清楚 |
| 表达 | 假设 → 计算 → 风险 → 验证结构完整 |
