# SLO、演练与事故闭环

## 90 秒速答

稳定性建设先把用户感知转成少量 SLI，为不同业务等级定义 SLO，再用错误预算决定发布速度和
治理优先级。事故前通过容量压测、故障演练和 Runbook 验证保护措施；事故中按影响面、时间线
和变更线索先止血后定因；恢复时校验业务事实而不只是进程存活；事故后做无责复盘，把根因、
监控缺口、决策缺口转成有 owner、期限和验收方式的行动项。

## 从用户旅程定义 SLI

| 用户旅程 | 好事件定义 | SLI 示例 |
| --- | --- | --- |
| 提交订单 | 在 3 秒内返回明确受理结果 | 好请求数 / 有效请求数 |
| 完成支付 | 资金成功且订单 30 秒内收敛 | 按时收敛订单 / 成功支付订单 |
| 消费消息 | 在业务时限内完成且副作用正确 | 按时正确处理数 / 应处理数 |

只看 HTTP 200 会掩盖“返回成功但业务没完成”。SLI 的分母也要排除无效参数、恶意请求等不在
服务承诺内的流量，并把规则写清楚。

## 错误预算如何驱动决策

月度 99.95% 可用性意味着约 21.6 分钟不可用预算。更重要的是消耗速度：若 1 小时内消耗了
月预算的 20%，即使当前已恢复，也应升级响应并冻结高风险发布。

```text
错误预算 = 1 - SLO
燃烧率 = 当前窗口错误率 / 允许错误率
```

错误预算充足时可以加快实验；持续超支时优先做稳定性整改、收紧发布并减少风险变更。它不是
“允许故障”的免责额度，而是产品速度与可靠性之间的共同决策语言。

## RTO、RPO 要落到业务恢复顺序

- **RTO**：从中断到业务恢复的最长可接受时间。
- **RPO**：恢复后最多可接受丢失多长时间范围的数据。

例如订单核心写入 RTO 15 分钟、RPO 0，商品搜索 RTO 2 小时、RPO 30 分钟。两者会决定同步
复制、异步复制、备份频率、切换自动化和成本。不能只写在文档里，必须通过切换演练测得。

## 一次有效的故障演练

1. 定义假设：单 AZ 网络丢包 20% 时，核心下单仍保持 99.9%。
2. 设定爆炸半径：测试租户 → 1% 实例 → 单 AZ，任何越线立即停止。
3. 准备观测和终止条件：业务成功率、P99、错误预算燃烧率、数据差异。
4. 注入故障并记录决策时间线，不提前告诉操作者具体故障点。
5. 验证自动保护、人工 Runbook、恢复顺序和数据校验。
6. 把发现转成行动项，再演练验证关闭。

## 事故响应：先恢复服务，再证明根因

```mermaid
flowchart LR
  A["确认用户影响"] --> B["控制变更与爆炸半径"]
  B --> C["可逆止血"]
  C --> D["业务恢复验证"]
  D --> E["证据链定因"]
  E --> F["复盘与整改验收"]
```

事故指挥者负责决策和节奏，技术人员分线取证，记录员维护时间线，对外沟通者统一更新影响。
不要让所有人同时 SSH，也不要为了找“完美根因”延迟可逆的止血动作。

## 恢复验证不能只看机器绿了

至少验证四层：实例和依赖健康、核心接口成功率与时延、订单/支付等业务结果、数据对账与积压。
恢复顺序通常是事实源 → 核心服务 → 异步传播 → 查询投影 → 非核心流量，且要限速放量。

## 复盘怎样避免流于形式

复盘区分触发因素、直接技术原因和系统性原因。每个行动项必须有 owner、截止时间、优先级和
验收证据；“加强监控”“提高意识”不是合格行动项。更好的写法是：“7 月 30 日前由平台组
增加支付状态收敛 SLI，并通过延迟注入演练验证 5 分钟告警”。

## 核心运营指标

| 指标 | 说明 | 关注点 |
| --- | --- | --- |
| MTTD | 从影响开始到发现 | 告警是否贴近用户感知 |
| MTTA | 从告警到有人响应 | 分级与升级路径 |
| MTTR | 从影响到业务恢复 | 拆分止血和恢复阶段 |
| 变更失败率 | 变更造成回滚或事故的比例 | 发布质量 |
| 整改按期完成率 | 行动项按期验收比例 | 复盘是否落地 |
| 同类事故复发率 | 同模式再次发生 | 知识是否转成系统能力 |

## 面试官三级追问

### L1：SLO 是否越高越好？

不是。目标越高，冗余、研发流程和运维成本越高。应结合用户损失、合同承诺、依赖能力和投入
成本确定，并按核心链路分级。

### L2：自动切换为什么仍需要演练？

配置漂移、容量不足、数据落后和错误健康检查都会让“自动”在真实故障时失效。演练验证的是
整条恢复链，而不是切换脚本能否执行。

### L3：事故恢复后为什么不能立刻全量放流？

恢复期可能存在积压重放、缓存冷启动、连接重建和客户端重试洪峰。应阶梯放量，持续观察业务
SLI、资源水位和数据差异；越线立即回退。
