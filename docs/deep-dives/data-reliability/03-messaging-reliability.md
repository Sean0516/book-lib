# 消息不丢、不重、不乱和不积压如何系统回答？

> [!IMPORTANT]
> 消息系统通常只能提供某种投递语义。业务正确性必须端到端设计，不能只依赖 Broker 的一个开关。

## 60–90 秒速答 {#quick-answer}

我会按生产者、Broker、消费者和业务结果四段设计可靠性。生产端用本地事务 Outbox 或事务消息
避免“数据库成功、发消息失败”；Broker 开启满足业务要求的副本与确认；消费端处理成功后才
提交位点，并用稳定业务键、唯一约束或状态机实现幂等。这样接受至少一次投递带来的重复，而
不是承诺无法端到端证明的“绝对只消费一次”。

顺序只在业务确实需要的实体范围内保证，例如同一 `orderId` 路由到同一分区，再用状态版本
拒绝旧事件。失败重试要指数退避、设置上限并进入死信或人工处理，不能让毒消息无限阻塞分区。

积压时先比较生产速率与有效消费速率，估算清空时间，再区分消费者变慢、分区不足、下游限流
还是重平衡。止血时暂停非核心生产、隔离毒消息、扩消费者到分区上限，并保护数据库和下游；
Broker 故障时核心写入保留本地事实和待发记录，不能绕过可靠链路假装成功。

## 端到端可靠性 {#loss}

```mermaid
flowchart LR
  B["业务事务"] --> O["Outbox / 事务消息"]
  O --> P["生产确认"]
  P --> M["Broker 副本与持久化"]
  M --> C["消费者处理"]
  C --> I["幂等业务写入"]
  I --> A["提交位点"]
  O -.重扫.-> P
  I -.对账.-> B
```

每一段都要回答失败窗口：

| 失败点 | 风险 | 典型保护 |
| --- | --- | --- |
| DB 已提交，消息未发 | 业务状态无法传播 | Outbox/CDC、事务消息、补偿扫描 |
| Broker 收到但生产者超时 | 生产者重试导致重复 | 稳定消息 ID，消费端幂等 |
| 消费成功但位点未提交 | 消息再次投递 | 业务唯一键或状态机幂等 |
| 位点先提交、业务后失败 | 消息逻辑丢失 | 业务成功后再提交位点 |
| 下游部分成功 | 数据状态分裂 | 补偿、对账、人工异常队列 |

“消息不丢”必须一直讲到业务结果可恢复，仅说生产者 ACK 和 Broker 持久化还不够。

## 重复消费与幂等 {#duplicate-idempotency}

教学场景：支付回调消息因消费者处理成功后网络断开而重复投递。若消费者每次都执行“余额加
100”，就会重复入账；如果使用 `paymentId` 唯一约束创建入账流水，再由流水驱动余额变化，
重复消息会命中同一业务结果。

幂等键必须来自稳定业务身份，不能使用每次重试都变化的 traceId。去重记录的保留时间要覆盖
消息最大重试、回放和人工补发窗口。只用短 TTL Redis 去重，缓存丢失后仍可能重复执行；强业务
结果优先依靠数据库唯一约束、条件更新或显式状态机。

## 顺序与乱序 {#ordering}

全局顺序吞吐低、扩展困难，大多数业务只需要同一实体的局部顺序：

1. 使用 `orderId` 等聚合键路由到固定分区。
2. 单分区由一个消费权属顺序读取。
3. 事件携带状态版本或业务序号。
4. 数据库条件更新拒绝旧版本和非法状态跃迁。
5. 被拒事件进入审计与对账，不应静默丢弃。

即使 Broker 保证分区顺序，重试转移、并发异步处理和跨系统回放仍可能让业务完成顺序变化，
所以业务状态机仍然必要。

## 重试与死信 {#retry-dlq}

| 错误类型 | 动作 |
| --- | --- |
| 瞬时网络错误 | 有上限的指数退避 + jitter |
| 下游限流 | 尊重 Retry-After 或降低消费速率 |
| 参数/Schema 错误 | 不做高频重试，进入隔离队列 |
| 永久业务冲突 | 记录原因并人工/规则处理 |
| 未知错误 | 有限重试后死信，保留原始上下文 |

死信队列不是垃圾桶。必须有责任人、积压告警、重放工具、幂等保证和处理时限。重放前先修复
根因并控制速率，否则会把故障重新注入主链路。

## 积压如何计算 {#backlog}

假设生产速率 20,000 条/s，故障期间有效消费降到 8,000 条/s，持续 10 分钟：

```text
新增积压 = (20,000 - 8,000) × 600 = 7,200,000 条
```

恢复后消费能力提升到 32,000 条/s，而生产仍为 20,000 条/s：

```text
净清理速度 = 32,000 - 20,000 = 12,000 条/s
理论清空时间 = 7,200,000 ÷ 12,000 = 600 秒
```

这是理想下限，还未考虑重试、毒消息、缓存未热和下游限流。扩容消费者也受分区数限制，并且
可能把压力转移到数据库。正确顺序是先定位有效消费率下降原因，再决定扩容、增加分区、暂停
非核心生产或对下游限速。

## Broker 故障与降级 {#broker-failure}

核心交易不应在 MQ 不可用时绕过事件链路直接标记“全部完成”。可选策略：

- 本地事务完成并写 Outbox，接口返回业务已受理，等待 Broker 恢复后补发。
- 如果业务不能接受异步传播延迟，则快速失败并让上游安全重试。
- 低价值通知可以降级或暂存，但必须明确是否允许丢失。
- 本地内存缓冲只能吸收很短抖动，进程重启会丢，不能替代持久待发记录。

恢复后按租户和业务优先级限速补发，先确认消费者与下游水位，再逐步释放积压。

## 指标与动作 {#metrics}

| 指标 | 教学信号 | 动作 |
| --- | --- | --- |
| 生产/消费速率差 | 连续 5 分钟为正 | 估算清空时间并查瓶颈 |
| 最老消息年龄 | 越过业务时效承诺 | 暂停非核心生产、提升优先级 |
| 重试率 | 突增且集中同类错误 | 隔离毒消息，停止无效重试 |
| 幂等冲突率 | 高于历史基线 | 检查生产超时、位点与重放 |
| 下游 TP99 / 错误率 | 扩容消费后恶化 | 降低消费并发，保护下游 |
| Outbox 未发送年龄 | 越过传播 SLO | 修复发布链路并告警业务风险 |

## 面试官追问 {#follow-ups}

1. Broker 返回超时，生产者如何判断消息到底有没有写成功？
2. 消费成功但提交位点失败，为什么必须允许重复？
3. 同一订单事件进入同一分区，为什么业务状态仍可能乱序？
4. 消息积压时把消费者扩十倍，可能引发什么二次故障？
5. MQ 故障两小时，恢复后如何补发而不再次打垮系统？

## 25 分自评

| 维度 | 5 分要求 |
| --- | --- |
| 正确性 | 覆盖生产、Broker、消费和业务结果四段 |
| 深度 | 能解释重复、乱序、毒消息和积压传播 |
| 取舍 | 明确至少一次、顺序范围和降级语义 |
| 表达 | 先端到端链路，再分别回答异常 |
| 可运维性 | 能计算积压，有重放、对账、限速和恢复验证 |
