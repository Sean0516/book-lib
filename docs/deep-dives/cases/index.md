# 完整案例库

这里不是“看答案”的案例摘要，而是可以暂停推演的完整面试训练。每篇先给现象和证据，
再展开假设、取证、根因、止损、长期修复与验收。

> [!IMPORTANT]
> 所有案例均基于常见生产模式构造，不对应任何实际企业或未公开事故。数字仅服务于同一
> 教学场景，重点是建立可验证的证据链。

## 案例地图

| 模块 | 故障案例 | 架构案例 | 建议顺序 |
| --- | --- | --- | --- |
| [JVM](./jvm/) | CPU 100%、线程池耗尽；另含既有 Full GC 案例 | 低延迟 Java 服务 | Full GC → CPU → 线程池 → 低延迟设计 |
| [数据库](./database/) | 慢 SQL、死锁与锁等待 | 高并发订单存储 | 慢 SQL → 锁竞争 → 存储设计 |
| [Redis](./redis/) | 热 Key、缓存击穿与不一致 | 高可用缓存体系 | 热点 → 一致性 → 高可用设计 |
| [消息队列](./messaging/) | 消息积压、重复/丢失/乱序 | 可靠事件驱动架构 | 积压 → 可靠性 → 事件架构 |
| [系统设计](./system-design/) | 秒杀击穿、级联雪崩 | 高并发订单系统 | 流量事故 → 依赖事故 → 综合设计 |

## 两种练习方式

### 故障排查

先只读“场景数据、时间线、观察证据”，口述你的前三个假设和取证顺序，再对照根因。
优秀回答必须区分事实、推断和待验证结论。

### 架构设计

按“目标与约束 → 容量 → 核心链路 → 取舍 → 故障模型 → 指标”回答。不要从技术名词开始，
先说明为何这个方案适合题目给定的负载和一致性要求。

## 评分

每篇按正确性、证据深度、取舍、可运维性和表达各 5 分，总分 25 分。`22–25` 分代表可以
进入高级岗位追问，`17–21` 分说明主干正确但证据或边界不足，低于 17 分建议按案例证据链
重答。
