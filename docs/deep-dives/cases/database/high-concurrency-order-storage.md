# 案例：高并发订单存储设计

> [!IMPORTANT]
> 本文是架构面试教学场景，所有容量均为显式假设。

## 业务现场与现状约束

现有订单单库 8 TB，同时承担用户列表、客服查单和商户对账，维护窗口已超过 6 小时。未来
大促目标是 50,000 写/s，但主要访问仍是“用户最近订单”和“订单号精确查询”。财务对账
允许分钟级延迟，用户建单不允许重复；迁移必须不停机并可在 30 分钟内回退。

> [!NOTE]
> 先按访问模式选分片键，再讨论中间件。商户对账为什么不一定要在在线订单库完成？
## 需求与约束

| 目标 | 数值 |
| --- | ---: |
| 峰值创建 | 50,000 写/s |
| 年增长 | 20 亿订单 |
| 保留 | 热数据 180 天，总计 7 年 |
| 查询 | 用户订单、商户对账、按订单号查询 |
| 一致性 | 不重复建单；支付状态最终收敛 |
| 可用性 | 99.99% |

## 面试版设计回答

先以订单号做全局幂等键，入口把创建命令写入分区日志，再由订单服务按用户哈希路由到 64
个逻辑分片。主表只保留查询必需字段，明细分表；用户列表索引以
`(user_id,created_at,id)` 支持游标分页，商户对账走独立 CDC 明细库，避免跨分片聚合压
在线库。180 天后归档对象存储并保留订单号路由索引。扩容采用逻辑分片到物理集群的映射，
先影子回放、双读校验再切换，不直接做不可回滚的全量双写。

## 容量估算

20 亿/年约 63 单/s 平均，50,000/s 峰值说明峰均比极高。按每订单主表 1 KiB、明细与索引
4 KiB 估算，年原始量约 10 TB，三副本和临时空间需至少 40 TB。64 逻辑分片在峰值平均
781 写/s；物理集群需按热点用户、复制和故障后 70% 水位压测。

  R --> D2["MySQL 集群 B..."]

## 数据模型与接口

```sql
CREATE TABLE orders (
  id BIGINT PRIMARY KEY,
  request_id VARBINARY(32) NOT NULL,
  user_id BIGINT NOT NULL,
  merchant_id BIGINT NOT NULL,
  status TINYINT NOT NULL,
  amount BIGINT NOT NULL,
  created_at DATETIME(3) NOT NULL,
  UNIQUE KEY uk_request(request_id),
  KEY idx_user_time(user_id, created_at, id)
);
```

`POST /orders` 必须携带 `request_id`；重复请求返回原订单，而不是再次扣库存。

## 关键链路

创建日志确认后进入订单状态机；本地事务同时写订单和 outbox，CDC 发布订单事件。按订单号
查询先从 ID 中解析逻辑分片；用户列表天然单分片；商户跨分片对账使用 CDC 库并与财务
账本核对。

## 方案取舍

| 方案 | 优点 | 风险 | 决策 |
| --- | --- | --- | --- |
| 单库分区 | 简单 | 写入和维护上限低 | 早期可用 |
| 直接按用户分片 | 用户查询局部 | 商户查询跨片 | 主方案+对账库 |
| 日志先行 | 削峰、可重放 | 可见性稍延迟 | 创建链路采用 |
| 全局二级索引 | 查询灵活 | 一致性与写放大 | 仅订单号路由索引 |

## 一致性与故障处理

- `request_id` 唯一约束保证建单幂等。
- 状态机只允许合法跃迁，事件携带版本拒绝乱序。
- 日志已确认但数据库未写入时可重放；写入成功但事件未发由 outbox/CDC 补发。
- 对账比较订单、支付、库存和消息位点，差异进入补偿队列与人工审批。
- 单集群故障时暂停对应逻辑分片写入或切主，禁止跨分片随意重建订单。

## 扩容与演进

逻辑分片数预留，物理映射可变。迁移一个逻辑分片时：快照回填 → CDC 追平 → 影子读校验
→ 短暂写栅栏 → 路由切换；保留旧库只读 24 小时用于回退。超过单集群 60% 写水位或维护
窗口无法满足时触发拆分。

## 指标与验收

| 指标 | 目标 |
| --- | ---: |
| 创建 P99 | `< 200 ms` |
| 重复订单 | `0` |
| 分片最高写水位 | `< 60%` |
| CDC 延迟 P99 | `< 5 s` |
| 对账差异未闭环 | `< 10 min` |
| 单集群恢复 RTO/RPO | `< 15 min / < 5 s` |

## 面试官追问与评分

### 追问一：为什么按 userId 分片，而不是 merchantId？

**参考回答：**核心在线访问是用户最近订单，按 userId 可将列表查询限制在单分片；商户
对账允许分钟级延迟，可通过 CDC 汇聚到独立对账库。代价是商户在线查询不能直接走单分片。
若访问模式以商户后台为主，分片键可能需要重新选择。

### 追问二：订单号如何设计，为什么不能只用数据库自增 ID？

**参考回答：**订单号需要全局唯一、趋势递增并能定位逻辑分片，可使用时间戳、节点/序列和
路由位组合。直接暴露自增 ID 在多分片下不唯一，也可能泄露业务量。生成器必须处理时钟
回拨和节点冲突，并保留 requestId 唯一约束防重复建单。

### 追问三：日志确认后什么时候可以向用户返回成功？

**参考回答：**取决于产品语义。若“成功”代表订单已可查询，应在订单落库后返回；命令只
写入日志时应返回“已受理/处理中”并提供查询接口。不能把消息持久化等同于业务完成，否则
后续建单失败会造成用户认知和财务状态冲突。

### 追问四：商户对账库延迟或不可用会影响建单吗？

**参考回答：**不应影响核心建单。在线库通过 outbox/CDC 异步提供变更，对账库可从位点
恢复；延迟超过 SLO 时暂停报表或标记数据新鲜度。财务结算前需要等待位点追平并执行订单、
支付和退款对账，不能直接用延迟副本做最终结算。

### 追问五：如何在线扩容逻辑分片？

**参考回答：**逻辑分片与物理库解耦。先快照回填目标分片，再用 CDC 追平增量，通过影子读
比较数量、金额和状态；切换时使用写栅栏或可证明的双写协议，更新路由后保留旧库只读。
校验或延迟越线时将路由切回，不能在验证完成前删除旧数据。

失分信号：用平均 63 单/s 代替峰值规划；只说分库分表；没有全局 ID 路由、迁移和对账。

| 证据 | 容量、热点和水位量化 |
| 取舍 | 在线查询与对账分离合理 |
| 可运维性 | 迁移、对账、灾备完整 |
