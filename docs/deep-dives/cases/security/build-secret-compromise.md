# 案例：构建密钥泄露与恶意制品

## 业务现场

CI 使用长期云密钥并允许依赖安装脚本访问网络。某依赖更新后读取环境变量，把密钥外发，并上传了
同名镜像标签。集群只按标签部署，6 个服务在 38 分钟内运行了来源不明的制品。

## 面试版事故回答

先吊销密钥、冻结部署和制品仓库写入，隔离相关工作负载并保全构建日志、镜像 digest 与云审计。
不能只回滚标签，因为标签已被覆盖。长期使用隔离临时构建和工作负载短期身份，限制安装阶段网络；
锁定依赖、生成 SBOM；制品使用不可变 digest、签名和来源证明，部署策略只接受受信构建身份。

## 恢复与验收

从已审计源码和干净构建环境重建制品，轮换派生凭证，检查云资源横向移动后阶梯恢复。演练恶意依赖
和制品替换，要求长期密钥为 0、未签名制品部署为 0、吊销传播 `<5min`、所有运行制品可追溯。

## 追问与评分

1. 扫描没有 CVE 为什么仍中招？——攻击来自构建脚本与凭证，不是已知组件漏洞。
2. 删除恶意依赖是否足够？——还需吊销密钥、重建制品和调查已发生行为。
3. 签名私钥如何保护？——优先使用受控签名服务和短期身份，限制使用并审计。

| 维度 | 5 分要求 |
| --- | --- |
| 正确性 | 覆盖凭证、构建、制品、部署整条链 |
| 证据 | digest、审计与来源证明可追踪 |
| 取舍 | 快速恢复与可信重建平衡 |
| 可运维性 | 吊销、签名、策略和演练完整 |
| 表达 | 不把漏洞扫描当全部安全 |
