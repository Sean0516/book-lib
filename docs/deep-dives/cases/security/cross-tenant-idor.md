# 案例：合法 JWT 导致跨租户订单泄露

## 业务现场

导出接口验证 JWT 和 `report:read` 角色，却直接使用请求中的 `tenantId` 查询。普通管理员修改参数后
下载了其他 17 个租户的 8.2 万条订单；网关和应用日志均显示 200，未记录授权决策资源范围。

## 面试版事故回答

立即关闭导出、吊销相关会话并限制对象存储下载，保全网关、应用、数据库与审计日志，通知安全与
合规负责人评估影响。根因是把认证和粗粒度角色当作对象级授权，并信任客户端租户参数。长期从可信
身份推导租户，在服务和查询层强制范围；高风险导出审批、限额、水印和短期下载；记录策略版本、
主体、资源、结果和原因，并增加跨租户负向测试。

## 证据链与验收

`JWT 合法 → 角色允许 → tenantId 未绑定身份 → 跨租户查询 → 下载链接继续有效`。验收要求跨租户
测试全部拒绝、撤权 5 分钟内传播、导出全量可追踪、旧下载链接失效且历史访问范围完成对账。

## 追问与评分

1. 网关统一鉴权为何不够？——网关不知道具体订单归属和业务状态。
2. 数据库 RLS 能否替代应用授权？——可做纵深防御，但仍需应用语义、审批和审计。
3. 是否直接删除日志？——不能，需保全证据并限制访问。

| 维度 | 5 分要求 |
| --- | --- |
| 正确性 | 区分认证、角色与对象授权 |
| 证据 | 身份、查询、导出和下载链闭环 |
| 取舍 | 止损、取证和业务连续性平衡 |
| 可运维性 | 撤权、审计、负向测试完整 |
| 表达 | 先影响和止损，再长期修复 |

用 90 秒解释“JWT 完全合法为何仍是严重越权”。参考
