# 案例：Java 服务 CPU 持续 100%

> [!IMPORTANT]
> 本文是基于常见生产模式构造的教学案例，不对应任何实际企业。请先读证据再看根因。

## 业务现场与变更线索

规则决策服务负责优惠资格和风控放行，工作日下午开始出现“提交后转圈”，但入口 QPS 没有
明显增长。当天新版本将规则缓存封装为通用组件，并修改了 Key 构造；运维同时看到 CPU
告警、Young GC 增多和 TP99 上升。需要判断是业务死循环、锁自旋、GC、序列化还是流量倾斜。

> [!NOTE]
> 先写出取证顺序：你会如何从进程 CPU 走到线程、方法、对象分配和代码变更？

## 场景数据

| 项目 | 正常 | 故障 |
| --- | ---: | ---: |
| 流量 | 1,200 QPS | 1,180–1,220 QPS |
| 规格 | 8 vCPU / 12 GiB / JDK 21 | 未变化 |
| CPU | 38% | 97% |
| TP99 | 210 ms | 3.2 s |
| 错误率 | 0.15% | 8.4% |
| 规则载荷 | 1.8 MiB JSON | 每请求重复解析 |
| 分配速率 | 420 MiB/s | 2.9 GiB/s |

## 面试版事故回答

流量和机器规格没有变化，因此先排除单纯容量增长。我用 `top -H` 找到持续占用 CPU 的
工作线程，把十进制线程号转成十六进制后在 `jstack` 中定位到 Jackson 解析；连续两次
线程栈相同只能证明热点，再用 async-profiler 确认 `RuleCompiler.parse` 占 61% CPU，
同时分配速率从 420 MiB/s 增至 2.9 GiB/s。发布差异显示缓存 key 从规则版本改成了请求
ID，1.8 MiB 规则因此每次都重新解析。先回滚并限流，长期恢复稳定 key、有界缓存和单飞
编译。以 1,200 QPS 回放 30 分钟，CPU 低于 60%、TP99 低于 300 ms 才恢复全量。

## 架构与故障传播

```mermaid
flowchart LR
  R["请求 1,200 QPS"] --> K["requestId 作为缓存 key"]
  K --> M["缓存命中率 96% → 0.3%"]
  M --> P["重复解析 1.8 MiB JSON"]
  P --> C["CPU 97% / 分配 2.9 GiB/s"]
  C --> Q["请求排队"]
  Q --> T["TP99 3.2 s / 错误率 8.4%"]
```

## 时间线

| 时间 | 证据 | 动作 |
| --- | --- | --- |
| 14:00 | 新版本完成 20% 灰度 | 继续观察 |
| 14:06 | 灰度实例 CPU 82% | 停止放量 |
| 14:10 | TP99 1.4 s | 摘一台灰度实例取证 |
| 14:14 | 热线程停在 JSON 解析 | 启动 60 秒 profiler |
| 14:19 | 解析占 61% CPU | 对比发布 diff 与缓存指标 |
| 14:24 | 命中率降至 0.3% | 回滚、保护性限流 |
| 14:41 | CPU 回到 41% | 同流量验证 |

## 从观察到结论

| 观察 | 支持的判断 | 不能直接断言 |
| --- | --- | --- |
| 流量稳定、CPU 上升 | 单请求成本增加 | 一定是代码问题 |
| 热线程都在解析 | 解析是热点 | 解析为何重复 |
| 火焰图占比 61% | 优化该路径收益高 | 缓存就是根因 |
| 命中率与发布同步下降 | 缓存身份变化可疑 | 修复后不会击穿 |
| key 包含 requestId | 相同规则无法复用 | 缓存边界已经合理 |

三轮证据应依次收敛：`top -H+jstack` 证明热线程在解析；火焰图与分配 profile 证明解析占
61% CPU 且制造大量短命对象；命中率和发布 diff 最终解释“为何重复解析”。缺任何一层都
不应直接宣布根因。

## 取证过程

```bash
PID=2481
top -H -p "$PID"
printf '%x\n' 2764                 # 线程号转 nid
jstack "$PID" | sed -n '/nid=0xacc/,/^$/p'
asprof -d 60 -e cpu -f /tmp/cpu.html "$PID"
jcmd "$PID" GC.class_histogram
```

取证要连续采样并与发布、流量和 GC 对齐。单张线程栈可能恰好落在正常解析路径。

## 止血决策

1. 停止灰度并回滚缓存 key 变更。
2. 按租户限制规则编译并发，避免剩余实例同时过载。
3. 降级到上一版已编译规则；规则更新暂时延迟。
4. 摘流单实例保留 profiler、线程栈和命中率数据。

加机器只能暂时稀释 CPU，错误 key 仍会让成本随请求线性增长。

## 永久修复

```java
record RuleKey(String tenantId, String ruleVersion) {}

CompiledRule getRule(Request r) {
    var key = new RuleKey(r.tenantId(), r.ruleVersion());
    return cache.get(key, ignored -> compiler.parse(r.ruleJson()));
}
```

缓存设置最大权重、30 分钟 TTL 和每 key 单飞编译；指标增加命中率、key 基数、编译耗时及
分配速率。测试必须证明不同 requestId 的同版本规则产生同一 key。

## 方案取舍

| 方案 | 收益 | 代价/风险 | 选择 |
| --- | --- | --- | --- |
| 扩容实例 | 快速争取时间 | 成本高且根因保留 | 仅止血 |
| 缓存原始 JSON | 减少网络读取 | 仍需每次解析 | 不选 |
| 缓存编译结果 | 消除重复 CPU | 需限制重量和版本 | 主方案 |
| 预编译所有规则 | 请求路径最轻 | 长尾规则浪费内存 | 仅高频规则 |

## 验证与回滚

| 指标 | 故障值 | 通过标准 |
| --- | ---: | ---: |
| CPU | 97% | `< 60%` 持续 30 分钟 |
| TP99 | 3.2 s | `< 300 ms` |
| 缓存命中率 | 0.3% | `> 95%` |
| 分配速率 | 2.9 GiB/s | `< 600 MiB/s` |
| 规则错误率 | 8.4% | `< 0.3%` |

灰度出现 CPU 超过 70%、命中率低于 90% 或规则版本错误即回滚。

## 复盘与防复发

- 缓存 key 评审必须说明每个字段的业务身份含义。
- CPU 告警同时展示 QPS、命中率、分配速率和发布标记。
- 大载荷解析路径加入基准测试和稳态压测。
- 高成本计算统一使用并发上限与 single-flight。

## 面试官追问与评分

### 追问一：CPU 100% 时为什么不能只看一次 `jstack`？

**参考回答：**线程栈只是一个瞬时采样，可能恰好落在正常高频方法上。应先用 `top -H`
确认持续占 CPU 的线程，再连续采样栈；随后用 CPU 火焰图量化方法占比，并与请求量、GC、
分配率和发布变更对齐。多种独立证据指向同一路径，才能提高根因可信度。

### 追问二：如何区分业务死循环和高频正常计算？

**参考回答：**死循环通常表现为同一线程长期停留在相同栈帧、吞吐不随 CPU 上升且可能无
正常返回；高频计算会在相似调用路径间变化，并与请求量或缓存命中率相关。可结合方法计数、
请求完成率、火焰图和最小复现判断，不能仅凭“栈看起来一样”下结论。

### 追问三：为什么分配速率高会进一步放大 CPU 问题？

**参考回答：**重复解析既消耗 CPU，也产生大量短命对象，增加 TLAB 分配、Young GC 和对象
复制成本。GC 可能是放大器而非最初根因。应同时观察 allocation profile、GC CPU 占比和
暂停分布；直接调大堆只会改变回收节奏，不会消除重复工作。

### 追问四：缓存 Key 稳定后还存在哪些风险？

**参考回答：**稳定 Key 仍可能因租户、版本组合过多形成高基数，也可能缓存错误规则或在
统一过期时击穿。需要最大权重、TTL 抖动、按租户配额、命中率和 Key 基数监控，并用版本
控制更新。缓存不是“加上就结束”，必须定义身份、容量和失效边界。

### 追问五：无法回滚时如何止损？

**参考回答：**通过动态配置恢复旧 Key 或关闭新缓存路径，限制解析并发，对高风险租户限流，
并使用上一版编译结果兜底。先在少量实例验证 CPU、命中率和规则正确性，再逐步放量。若
无法保证规则正确，宁可降级功能，也不能只追求 CPU 恢复。

失分信号：看到 CPU 高就扩容；只贴一张线程栈；把 GC 相关性当因果；修 Key 却不限制缓存容量。

| 维度 | 5 分要求 |
| --- | --- |
| 正确性 | 区分流量增长与单请求成本 |
| 证据 | 线程 CPU、栈、火焰图、命中率闭环 |
| 取舍 | 说明扩容只止血 |
| 可运维性 | 有阈值、灰度和回滚 |
| 表达 | 事实、推断、结论分层 |
