# 案例：平台采用率很高但交付更慢

## 业务现场

公司强制 42 个团队迁移内部开发者平台，采用率达到 96%。半年后部署前置时间从 1.8 天升到 3.1
天，平台工单每月 1,400 个，27% 团队维护绕行脚本；平台团队以“接入服务数”汇报成功。

## 从指标到结论

| 指标 | 观察 | 结论 |
| --- | --- | --- |
| 强制采用率 | 96% | 不能证明自愿价值 |
| 自助成功率 | 61% | 黄金路径不完整 |
| 工单等待 | 46% 前置时间 | 平台成为人工瓶颈 |
| 绕行率 | 27% | 用户需求未被覆盖 |

## 面试版回答

先停止用接入数作为唯一目标，按“创建服务—配置环境—发布—观测”拆用户旅程，定位工单等待和高频
绕行。选择发布权限与数据库接入做薄切片自助化，与三个团队共创；保留有期限例外。长期以交付
前置时间、变更失败率、自助成功率、绕行率和满意度运营平台，并自动升级模板和策略版本。

## 治理改造

平台原则写入 ADR；高频规则进入流水线守护；例外记录风险、owner 和到期日；低价值功能设退场
标准。三个月目标不是“新增 20 个功能”，而是发布等待降低 50%、自助成功率超过 90%、变更失败率
不升高。

## 面试官追问与评分

1. 是否应取消强制标准？——保留风险底线，但通过更好默认路径获得采用，而非全部放任。
2. 团队特殊需求怎么办？——提供透明例外，并用例外聚类反哺平台路线图。
3. 如何证明 ROI？——节省等待和重复工程时间，同时验证故障与成本没有转移。

| 维度 | 5 分要求 |
| --- | --- |
| 正确性 | 区分采用率与用户价值 |
| 证据 | 流程、工单、绕行和稳定性指标完整 |
| 取舍 | 标准化与自主性平衡 |
| 可运维性 | 产品路线、例外、版本升级可持续 |
| 表达 | 从用户问题而非平台功能出发 |
