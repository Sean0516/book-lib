# 案例：错误探针引发重启风暴

## 业务现场

订单库发生 40 秒延迟抖动，应用 liveness 直接检查数据库查询。3 个 AZ 的 120 个 Pod 在两分钟内
重启 87%，本地缓存清空后回源继续压高数据库，订单成功率从 99.96% 降到 58%。

## 证据链

| 证据 | 结论 |
| --- | --- |
| 容器 CPU 仅 31% | 应用不是计算饱和 |
| 探针超时与 DB P99 对齐 | liveness 错误依赖下游健康 |
| 重启后回源升高 6.4× | 自愈动作放大故障 |
| 回滚探针后 3 分钟恢复 | 传播路径被证实 |

## 面试版事故回答

先暂停发布、回滚 liveness 并限制非核心流量，避免继续重启；数据库抖动通过 readiness、熔断和降级
表达，不能把所有实例判死。长期将 liveness 限于进程无法自愈状态，startup 保护冷启动，readiness
只判断能否接核心流量；加探针抖动、PDB、缓存预热和数据库慢调用演练。

```mermaid
flowchart LR
  D["数据库慢"] --> L["liveness 失败"]
  L --> R["Pod 批量重启"]
  R --> C["缓存冷启动"]
  C --> D
```

## 验收与回滚

注入数据库 P99 2 秒但不完全宕机，要求 Pod 非预期重启为 0、核心成功率 `>99.9%`、回源放大
`<1.2`。任何跨 AZ 同时摘流超过 20% 即停止演练。

## 面试官追问与评分

1. readiness 也检查数据库会怎样？——全摘流可能让仍可降级的服务彻底不可用，应按核心语义设计。
2. PDB 为什么没阻止？——突发探针重启不是自愿驱逐，PDB 不是该故障的保护机制。
3. 是否应删除所有探针？——不应，应修正语义并独立验证三类探针。

| 维度 | 5 分要求 |
| --- | --- |
| 正确性 | 区分三类探针语义 |
| 证据 | 重启、数据库和缓存形成闭环 |
| 取舍 | 摘流、降级、重启边界明确 |
| 可运维性 | 有演练、阈值和回滚 |
| 表达 | 解释“自愈为何放大故障” |
