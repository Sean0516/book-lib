# 案例：数据库不兼容导致发布回滚失败

## 业务现场

订单新版把 `status` 字段拆成两个字段，并在发布脚本中直接删除旧列。金丝雀 5% 时新版本正常，
扩到 50% 后旧版本大量报错。团队回滚镜像，却因旧列已删除无法恢复，故障持续 47 分钟。

## 根因链

金丝雀只验证新版本，没有验证新旧共存；数据库迁移不可逆；流水线把“镜像可回滚”误当成“系统
可回滚”；发布门禁只看 HTTP 错误，没有数据兼容测试。

## 面试版事故回答

立即停止放量并关闭新功能写入，优先前滚恢复兼容列或切换到只读降级，不能反复回滚镜像。长期按
Expand–Migrate–Contract：先新增字段并双写校验，后台可重放迁移，切读后保留旧字段观察，最后
独立发布删除；流水线增加契约、混部、影子读和数据差异门禁。

## 发布阶段

```mermaid
flowchart LR
  A["新增兼容字段"] --> B["新旧版本混部"]
  B --> C["双写/回填/对账"]
  C --> D["切换读路径"]
  D --> E["观察期"]
  E --> F["删除旧字段"]
```

## 验收

新旧版本双向兼容，迁移可暂停和重放；数据差异为 0；回滚演练能在 5 分钟恢复业务；旧字段删除
作为独立变更，并要求确认无旧版本和旧消费者。

## 面试官追问与评分

1. 蓝绿发布能解决吗？——不能自动解决共享数据库的协议不兼容。
2. 双写失败怎么办？——记录可重放意图、对账差异并明确事实源，不能静默忽略。
3. 为什么 5% 未发现？——新旧交互和特定数据路径样本不足，门禁设计不完整。

| 维度 | 5 分要求 |
| --- | --- |
| 正确性 | 识别数据变更不可由镜像回滚 |
| 证据 | 兼容、混部和数据差异链完整 |
| 取舍 | 前滚、降级和回滚选择合理 |
| 可运维性 | 分阶段门禁和演练可执行 |
| 表达 | 区分应用发布与数据迁移 |
