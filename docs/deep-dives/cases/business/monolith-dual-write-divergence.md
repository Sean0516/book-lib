# 案例：单体拆分双写造成数据分叉

## 业务现场

会员域从单体拆出后，网关同时调用单体和新服务写入。新服务超时会重试，单体失败则记录日志人工
处理。三周后 4.8% 会员等级不同，两个团队都认为自己的数据库是事实源，回滚也无法决定保留哪边。

## 面试版事故回答

立即停止无边界双写，按业务键冻结冲突会员的等级变更，用原始交易流水重建可信状态。根因是迁移
没有唯一事实源、持久化复制意图、版本和对账。改为单体暂时唯一写入，通过 Outbox/CDC 构建新服务
投影并影子比较；验证稳定后按租户设置写栅栏切换所有权，新服务成为事实源，旧系统只读回退。

```mermaid
flowchart LR
  C["会员命令"] --> M["单体事实源"]
  M --> O["Outbox/CDC"]
  O --> N["新服务投影"]
  N --> V["影子对账"]
  V --> G["按租户切写所有权"]
```

## 迁移门禁

影子差异 `<0.01%` 且都有可解释原因；CDC 延迟 `<5s`；切换租户可在 30 分钟内路由回旧读路径；
切写后旧库写入为 0；迁移结束时删除双写代码和人工补偿脚本。

## 面试官追问与评分

1. 为什么日志补偿不够？——没有结构化意图、幂等和完成状态，无法证明全部恢复。
2. CDC 延迟时用户读哪边？——切换前读事实源；切换后按新所有权读，必要时展示处理中，不能随机读。
3. 何时算拆分完成？——旧写入、调用、作业和值班归零，且回退转为备份恢复而非双轨运行。

| 维度 | 5 分要求 |
| --- | --- |
| 正确性 | 始终只有一个写事实源 |
| 证据 | 交易流水、位点、版本和对账闭环 |
| 取舍 | 影子、切写、回退顺序合理 |
| 可运维性 | 有门禁、写栅栏和旧路径退场 |
| 表达 | 不把“双写”当天然迁移方案 |
