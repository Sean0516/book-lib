# 案例：超大订单聚合造成锁竞争

## 业务现场

批发订单最多包含 8 万条明细。系统每次更新单条备注都加载完整订单并更新聚合版本；大客户活动时
同一订单每秒 120 次操作，乐观锁冲突率升到 67%，保存 TP99 达到 9.4 秒。

## 面试版事故回答

先限制高频非核心修改并把批量任务排队，保护确认和支付命令。根因是把“有关联”误当成“必须同一
强一致边界”。重新列不变量：订单总额在确认前可异步重算，确认时冻结版本并校验；备注、附件和
统计不属于订单核心聚合，可独立存储。大批量明细采用分段命令和状态机，最终确认用条件更新保证
版本与金额摘要一致。

## 证据链

| 证据 | 结论 |
| --- | --- |
| 冲突集中于大订单 | 非全局 DB 容量问题 |
| 修改备注也更新订单版本 | 无关操作共享聚合锁 |
| 加实例冲突率不降 | 瓶颈是业务串行点 |

## 验收

备注更新不竞争订单版本；确认命令保持金额不变量；热点订单冲突 `<5%`、确认 TP99 `<500ms`；
注入重复、乱序和确认并发，不能出现确认后金额变化。

## 面试官追问与评分

1. 为什么不删除所有版本锁？——确认不变量仍需串行化或条件更新。
2. 分小聚合会失去一致性吗？——跨聚合允许明确窗口，关键确认点重新校验核心摘要。
3. CQRS 是否必须？——不是；只有读写复杂度和扩展收益足够时才引入独立投影。

| 维度 | 5 分要求 |
| --- | --- |
| 正确性 | 从不变量而非对象关系定义聚合 |
| 证据 | 热点、冲突和命令类型能闭环 |
| 取舍 | 一致性与并发权衡明确 |
| 可运维性 | 有冲突、延迟和不变量测试 |
| 表达 | 说明哪些拆、哪些仍强一致 |
