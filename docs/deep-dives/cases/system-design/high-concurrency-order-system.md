# 案例：高并发订单系统设计
> [!IMPORTANT]
> 架构教学场景，容量是假设。

## 业务现场

平台准备将订单中心从单体拆出，服务普通电商、大促秒杀和直播带货三类流量。峰值请求中
约 80% 会因库存、风控或限购失败，成功订单为 20,000/s。用户可能重复点击，支付回调可能
延迟或重复；商家要求实时看到新订单，财务要求任何金额差异都能追溯。

## 现有系统与演进约束

- 现有 MySQL 单库 8 TB，订单、明细、库存写在同一事务，峰值只能稳定 6,000 单/s。
- 订单号被客服、支付和物流使用，迁移期间不能改变查询接口。
- 团队可接受秒杀订单 3 秒内可见，普通订单仍希望同步返回订单号。
- 不允许停机迁移；必须支持灰度、双读校验和 30 分钟内回退。

> [!NOTE]
> 设计前先明确：100,000/s 是尝试还是成功量？哪些步骤必须同步？系统的事实源分别是谁？
## 需求与约束
尝试 100,000/s，成功 20,000/s；不重复建单；99.99% 可用；支付窗口 15 分钟。
## 面试版设计回答
入口用活动令牌和幂等键准入，库存原子预占后异步建单；订单状态机与 outbox 同事务，支付
和库存通过事件协作并对账。按用户逻辑分片，订单号可路由；失败释放预占，超时订单由定时
扫描关闭。所有层按故障后 70% 水位部署。
## 容量估算
成功 20k/s、每单 5KiB 约 100MiB/s 写入；按三副本、峰值两倍和热点预留规划。
## 核心架构
```mermaid
flowchart LR
 G["准入/幂等"]-->I["库存预占"]-->Q["订单队列"]-->O["分片订单库"]
 O-->E["Outbox"]-->P["支付"]-->R["对账补偿"]
```
## 数据模型与接口
```json
{"requestId":"唯一键","userId":7,"skuId":8848,"quantity":1}
```
## 关键链路
同步返回受理结果；异步建单；支付回调按 requestId 幂等；15 分钟关闭未支付并释放库存。
## 方案取舍
| 方案 | 优点 | 风险 |
| --- | --- | --- |
| 同步建单 | 语义直接 | 峰值脆弱 |
| 队列削峰 | 吸收突发 | 最终可见 |
| Redis 预占 | 高吞吐 | 需账实对账 |

## 推演岔路

**如果要求同步返回订单号：** 可先生成全局 ID 并写入持久命令日志，返回“处理中”，而不是
谎称订单已完成。**如果消息队列不可用：** 准入层应快速失败或使用受限本地缓冲，不能绕过
库存直接写库。**如果支付先于订单事件到达：** 以业务键暂存并重试，状态机不得跳过合法
前置状态。**如果一个用户成为热点：** 用户分片需要识别热点租户并设置独立水位。
## 一致性与故障处理
唯一键防重复；状态版本防乱序；Outbox 防消息丢失；订单、库存、支付四方对账；单写地域以
fencing 防脑裂。
## 扩容与演进
逻辑分片映射物理集群；快照回填、CDC 追平、影子读、写栅栏后切换，保留旧库回退。
## 指标与验收
受理 P99 `<100ms`、建单 P99 `<3s`、重复/超卖 `0`、对账差异 `<10min`、可用性 99.99%。
## 面试官追问与评分

### 追问一：用户重复点击，如何保证返回同一个订单？

**参考回答：**客户端生成稳定 requestId，服务端用唯一约束建立 requestId 到 orderId 的映射。
首次请求处理中时，后续请求返回同一受理状态；完成后返回同一订单。幂等记录必须覆盖最长
重试窗口，不能使用时间戳或每次变化的 traceId。

### 追问二：库存预占成功，但建单失败如何恢复？

**参考回答：**库存预占包含业务键、状态和过期时间；建单命令写入持久日志后可重放。短暂
失败继续幂等重试，超过支付窗口才通过补偿状态机释放。后台对账比较预占、订单和支付，
无法自动判断的高价值订单进入人工审核，不能直接删除。

### 追问三：为什么按 userId 分片，而不是 merchantId？

**参考回答：**在线高频访问是用户最近订单，按 userId 可以把列表查询限制在单分片；商户
对账允许分钟级延迟，可通过 CDC 汇聚到对账库。代价是商户在线查询不能直接单分片完成。
如果业务主要是商户后台，访问模式变化后分片键也可能不同。

### 追问四：消息晚到或乱序导致订单状态倒退怎么办？

**参考回答：**事件按 orderId 保证局部顺序，同时携带状态版本；数据库以条件更新拒绝旧
版本和非法跃迁。不能只依赖消息顺序，因为重放、跨地域复制和补偿都会产生乱序。被拒绝
事件进入审计记录，周期对账判断是否需要补偿。

### 追问五：64 个逻辑分片如何在线扩到 128？

**参考回答：**先保持逻辑分片与物理库解耦。新分片做快照回填，再用 CDC 追平增量；通过
影子读比较数量、金额和状态。切换时使用短暂写栅栏或可证明的双写协议，更新路由后保留旧
库只读一段时间。任何校验越线都能把路由切回，不能一次性搬迁后删除旧数据。

失分信号：直接说分库分表而无访问模式；承诺分布式“强一致”却无协议；忽略失败订单；
没有迁移、对账、容量水位和灾备。

| 维度 | 5 分要求 |
| --- | --- |
| 正确性 | 幂等、状态机和事实源明确 |
| 证据 | 容量和热点估算自洽 |
| 取舍 | 同步/异步、分片键有理由 |
| 可运维性 | 迁移、回退、对账、灾备 |
| 表达 | 从业务约束到演进闭环 |
