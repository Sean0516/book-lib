# 渐进式交付与多集群

## 90 秒速答

发布策略先由变更风险、回滚速度、容量和数据兼容决定。无状态小改可滚动发布；高风险版本采用
金丝雀或蓝绿，但数据库和消息 schema 必须遵循先扩展、后迁移、再收缩，确保新旧版本可并存。
每一阶段按用户、租户或地域稳定分流，比较业务 SLI、TP99、错误预算和资源，不只看 Pod 健康；
硬门槛越线自动停止并回滚。多集群用于故障域、合规或规模边界，但要提前解决流量调度、数据事实源、
配置漂移和剩余容量，恢复时阶梯放量，防止缓存冷启动和连接重建形成二次冲击。

## 发布策略选择

| 策略 | 适用 | 代价 |
| --- | --- | --- |
| 滚动 | 低风险、前后兼容 | 新旧版本共存，回滚受数据变化限制 |
| 金丝雀 | 可量化业务指标、高风险变更 | 需要稳定分流和自动分析 |
| 蓝绿 | 需要快速流量切换、双环境可承受 | 成本高，数据层仍非一键回滚 |
| 功能开关 | 代码部署与能力启用分离 | 开关债务和组合复杂度 |

## Expand–Migrate–Contract

```mermaid
flowchart LR
  A["新增兼容字段/接口"] --> B["双读或双写校验"]
  B --> C["后台迁移历史数据"]
  C --> D["切换读路径"]
  D --> E["观察窗口"]
  E --> F["删除旧字段/旧协议"]
```

“回滚应用镜像”无法撤销已经写入的新数据格式。发布计划必须写清应用、数据库、消息和配置各自的
前向兼容、回滚点与不可逆边界。

## 金丝雀门禁

每阶段 1% → 5% → 20% → 50% → 100%，观察至少覆盖关键业务周期。比较实验组与基线组的业务
成功率、TP99、错误类型、资源效率和数据差异。支付错误、越权、数据不一致等硬指标零容忍；低流量
阶段样本不足时，配合影子请求和合成交易，不能因为“没报警”就自动晋级。

## 多集群的真实代价

- 全局流量调度可能受 DNS TTL、连接保持和客户端缓存影响。
- 每个集群都要有独立故障容量，不能把总副本均匀分完后宣称容灾。
- 有状态数据要定义事实源、写入拓扑、RPO/RTO 和脑裂处理。
- 配置、Secret、策略与镜像必须版本化同步并检测漂移。
- 切换和回切都要演练，回切往往比切走更危险。

## 面试官三级追问

### L1：金丝雀 1% 没问题为什么 100% 出问题？

小流量可能未触发热点、锁竞争、配额或特定租户路径。需要分层流量、容量阶梯和关键场景合成测试。

### L2：数据库变更如何回滚？

优先做兼容性前滚。先新增不删除，迁移可暂停可重放，切读后保留旧结构观察；若已做不可逆转换，
镜像回滚并不能恢复数据，必须有备份、反向迁移或业务补偿。

### L3：自动回滚什么时候不安全？

数据协议已经变化、外部副作用已发生或回滚版本不兼容当前状态时。此时应停止放量、关闭功能或
前滚修复，并由事故指挥明确选择。
