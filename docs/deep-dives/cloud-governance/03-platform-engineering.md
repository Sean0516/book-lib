# 平台工程与黄金路径

## 90 秒速答

平台工程的目标不是集中接单，而是降低开发者认知负担并把可靠、安全、合规的默认路径做成自助
产品。我先调研用户旅程和最高频摩擦，选择少量黄金路径，例如创建服务、发布、观测、数据库接入
和事故响应；通过模板、API、门户与流水线提供 paved road，同时允许有成本、有期限、有 owner 的
例外。平台用产品方式运营：有用户研究、路线图、SLO 和支持模型。价值通过交付前置时间、部署频率、
变更失败率、恢复时间、平台采用率和开发者满意度共同衡量，不能只统计平台功能数。

## 平台、门户和 DevOps 的区别

- 门户是入口，不等于平台；背后必须有可组合的自助能力。
- 平台团队提供能力和契约，不应成为所有发布的人工审批瓶颈。
- DevOps 是协作与责任方式，平台工程是降低其落地成本的一种产品化手段。

## 黄金路径结构

```mermaid
flowchart LR
  A["服务模板"] --> B["代码仓库与所有权"]
  B --> C["CI、安全与架构检查"]
  C --> D["环境与基础设施自助"]
  D --> E["渐进式发布"]
  E --> F["SLO、告警与 Runbook"]
```

默认生成健康检查、资源基线、仪表盘、告警、服务目录、依赖元数据和回滚配置，让正确做法比绕开
平台更省力。模板生成后仍要能持续升级，避免复制一次便永久漂移。

## 平台产品化路线

1. 访谈并量化等待时间、失败率、重复工作和认知负担。
2. 选择覆盖面大、标准化收益高的一个旅程做最小黄金路径。
3. 与早期团队共创，保留逃生通道和反馈。
4. 用采用率、任务完成时间和可靠性验证，再扩展能力。
5. 淘汰低价值功能和重复工具，控制平台自身复杂度。

## 例外不是失败

核心域、高性能或合规场景可能不适合默认路径。例外申请应记录原因、风险、补偿控制、owner、
到期时间和回归计划。若同类例外持续增加，说明黄金路径需要产品改进，而不是团队都“不配合”。

## 平台指标

| 维度 | 指标 |
| --- | --- |
| 流动效率 | 从提交到生产的前置时间、等待时间占比 |
| 稳定性 | 变更失败率、MTTR、默认告警覆盖 |
| 采用 | 活跃团队、黄金路径覆盖、绕行率、例外量 |
| 体验 | 自助任务成功率、满意度、支持工单量 |
| 经济性 | 每次部署平台成本、重复工具削减、工程时间节省 |

## 面试官三级追问

### L1：为什么不能强制所有团队立即迁移？

平台价值尚未验证，强制迁移可能转移成本并阻塞业务。先证明黄金路径更快更稳，再分风险迁移。

### L2：模板如何避免过时？

把关键策略放在可升级组件和流水线守护中；服务目录记录版本，自动生成升级 PR，度量版本分布。

### L3：平台采用率高是否等于成功？

不等于。强制使用也能产生高采用率，必须同时看交付速度、失败率、开发者体验和业务结果。
