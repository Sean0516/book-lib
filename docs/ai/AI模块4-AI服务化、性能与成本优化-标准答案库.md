# AI模块4 - AI服务化、性能与成本优化 - 标准答案库

## 1. AI 服务 SLA 如何定义？

::: tip L2 深度训练
同时定义任务成功、TTFT、TP99、可用性和单位成功成本：[模型路由、SLA 与成本](/deep-dives/ai-architecture/03-model-routing-cost)
:::

**核心结论**  
AI 服务 SLA 不能只写可用性，要同时定义质量、延迟、成本和降级边界，因为 AI 服务的“可用”不等于“可用且有价值”。

**标准答案**  
我定义 AI 服务 SLA 时，通常会拆成四类指标：可用性、时延、质量和成本。因为 AI 服务和普通 API 不一样，就算 99.9% 可用，如果结果质量飘、首字延迟过高或成本失控，业务依然会觉得不可用。工程上比较稳的做法，是按场景定义不同 SLA，比如聊天助手更看 TTFT 和满意度，抽取任务更看结构化成功率，审核任务更看高风险误判率。成熟团队会把“最低可接受质量”和“超预算后的降级策略”写进 SLA，而不是只写 uptime。

**追问点答案**  
- 为什么 AI SLA 要包含质量？  
因为 AI 服务返回结果不代表业务任务已完成。  
- 成本为什么也是 SLA 维度？  
因为成本失控会直接影响服务能否长期运营。  
- 不同场景的 SLA 能一样吗？  
通常不能，任务目标差异很大。

**原理补充**  
AI SLA 的本质，是把“模型能力不稳定”转成业务可接受边界的组合约束。

**架构扩展**  
成熟平台通常会按实时交互、后台异步、高风险审批三类服务分别定义 SLA，而不是把所有 AI 能力放在一套统一指标下。

**工程指标**  
- 可用性：看基础稳定性。  
- TTFT/TP99：看交互体验。  
- 任务成功率：看业务质量。  
- 单请求成本：看经济性。  
- 降级触发率：看韧性。

**证据链**  
- SLA 文档  
- 分场景看板  
- 用户反馈  
- 降级记录

**低风险优化路径**  
1. 先按场景拆 SLA  
2. 再给每类场景定义质量和成本边界  
3. 最后把降级策略纳入运行手册

**易错点**  
- SLA 只写可用性  
- 不区分场景差异  
- 没有超边界后的止损方案

## 2. 推理网关应具备哪些能力？

**核心结论**  
推理网关的核心价值，是把鉴权、路由、限流、审计、缓存和观测统一收口，避免业务服务直接耦合各类模型供应商。

**标准答案**  
我设计推理网关时，通常会把它定位成 AI 调用平面的统一入口，而不是简单的请求转发层。因为业务服务一旦直接对接不同模型和供应商，后面做配额、路由、审计、成本治理会非常碎。工程上比较稳的做法，是让网关统一处理鉴权、模型选择、流量路由、重试、超时、缓存、日志和指标，并向上暴露统一 API。成熟平台通常还会把 Prompt 版本、模型版本和租户配额也收进网关治理，这样 AI 服务才能真正平台化。

**追问点答案**  
- 为什么不能让业务直接调模型 API？  
因为后续治理、切换和成本控制会很难统一。  
- 网关和模型路由器是一回事吗？  
相关但不完全一样，网关更强调统一治理平面。  
- 推理网关会不会成为单点？  
会，所以本身也要高可用和可观测。

**原理补充**  
推理网关的本质，是把分散的模型调用变成统一治理的服务流量入口。

**架构扩展**  
成熟企业在 AI 平台化过程中，推理网关往往是最先成型的核心组件，因为没有它，模型调用会像早期微服务没网关一样迅速失控。

**工程指标**  
- 网关可用性：看平台稳定性。  
- 路由正确率：看治理质量。  
- 限流命中率：看保护效果。  
- 成本归因覆盖率：看经营能力。  
- 模型切换成功率：看弹性。

**证据链**  
- 网关架构图  
- 路由配置  
- 调用日志  
- 成本看板

**低风险优化路径**  
1. 先统一 API 入口  
2. 再补路由、限流和审计  
3. 最后做成本与版本治理

**易错点**  
- 业务直接耦合多家模型接口  
- 网关只做转发不做治理  
- 缺少统一日志和配额能力

## 3. 流式输出架构如何设计？

**核心结论**  
流式输出架构的重点，是缩短感知等待时间，同时确保中途中断、超时、审计和下游消费都能被可靠处理。

**标准答案**  
我设计流式输出时，通常先区分对用户展示的流式体验和系统内部的完整结果管理。因为流式能显著改善感知延迟，但也会引入中途中断、分片拼接、客户端断连和审计存档问题。工程上比较稳的做法，是前端按 token 或 chunk 流式接收，后端保留完整结果聚合和状态记录，并对中断场景做好重试或终止处理。成熟系统会把“流式体验”和“最终可审计结果”同时设计进去，而不是只顾前端看起来更快。

**追问点答案**  
- 为什么流式不等于更快完成？  
它主要改善的是感知延迟，不一定减少总耗时。  
- 客户端断开时怎么处理？  
要看任务是否继续执行、是否落盘、是否终止。  
- 审计为什么更复杂？  
因为输出是分段产生的，不是一次性结果。

**原理补充**  
流式输出的本质，是把完整推理结果拆成增量可消费片段，提前反馈给用户。

**架构扩展**  
成熟平台通常会把流式传输、完整结果持久化和异常中断处理分成不同层次治理，避免前端体验优化反过来破坏后端可审计性。

**工程指标**  
- TTFT：看感知起始速度。  
- 流式完成率：看链路稳定性。  
- 中断率：看连接质量。  
- 最终结果落盘成功率：看审计能力。  
- 用户中途退出率：看体验质量。

**证据链**  
- SSE/WebSocket 连接日志  
- 输出片段记录  
- 完整结果存储  
- 中断案例

**低风险优化路径**  
1. 先实现基础流式传输  
2. 再补完整结果落盘和中断处理  
3. 最后优化前端消费体验

**易错点**  
- 只有流式展示没有完整结果记录  
- 断连后状态不清  
- 只追求感知快忽略治理

## 4. 缓存策略如何降低 token 成本？

**核心结论**  
降低 token 成本的缓存策略要分层设计，优先复用高频请求、检索结果和结构化中间产物，而不是只缓存最终大段答案。

**标准答案**  
我做 AI 缓存时，通常会先看哪些内容最适合复用，比如标准化后的相似问题、RAG 检索结果、工具中间结果和固定结构化输出。因为如果只缓存最终答案，命中收益虽然大，但时效性和权限风险也最大。工程上比较稳的做法，是先缓存高复用、低风险层，再按业务价值谨慎扩到最终结果层。成熟团队会把缓存命中率、错误复用率和版本失效机制一起设计，不会为了省 token 把系统做成脏缓存中心。

**追问点答案**  
- 为什么最终答案缓存风险更大？  
因为更容易受知识更新和权限变化影响。  
- 哪类缓存通常最稳？  
检索结果、相似问题和结构化中间层。  
- 缓存命中是不是越高越好？  
不是，错误复用和陈旧复用也要控制。

**原理补充**  
AI 缓存的本质，是用“重复任务复用”抵消“高成本推理”。

**架构扩展**  
成熟平台一般会把缓存和 Prompt/模型/知识版本绑在一起，这样版本变化时可以精细失效，而不是全量清空或完全不清。

**工程指标**  
- Token 节省比例：看收益。  
- 缓存命中率：看复用程度。  
- 错误复用率：看风险。  
- 版本失效正确率：看一致性。  
- 单请求平均成本下降幅度：看经济性。

**证据链**  
- 缓存日志  
- 版本绑定策略  
- 错误复用样本  
- 成本看板

**低风险优化路径**  
1. 先缓存低风险中间层  
2. 再补版本化失效  
3. 最后按业务价值扩展到答案层

**易错点**  
- 直接缓存所有最终答案  
- 缓存键不带版本和权限  
- 只看命中率不看错复用率

## 5. 并发请求如何做优先级调度？

**核心结论**  
并发请求优先级调度的目标，是让高价值、低延迟容忍度的请求优先获得模型和资源，而不是所有请求先到先服务。

**标准答案**  
我做 AI 请求优先级调度时，通常先按业务价值和时延要求分层。比如在线客服、实时 Copilot、审批辅助这类交互型请求通常优先级更高，而批量摘要、离线标签、低价值分析任务可以让位。因为 AI 推理资源往往昂贵且不无限，先到先服务在高峰期很容易把关键请求拖死。工程上比较稳的做法，是按优先级建队列、设超时和配额，并允许低优先级任务降级、排队或延后执行。成熟平台会把业务优先级显式纳入调度，而不是让所有调用共享一个平面。

**追问点答案**  
- 为什么先到先服务不够？  
因为它无法体现业务价值差异。  
- 低优先级任务如何处理？  
排队、降级、批量化或转离线。  
- 高优先级是否永远优先？  
也要防止长期饥饿，通常要配额和平衡策略。

**原理补充**  
优先级调度的本质，是让资源分配贴近业务价值，而不是时间顺序。

**架构扩展**  
成熟平台通常会把优先级和租户、场景、费用策略一起设计，比如 VIP 客户、核心链路和付费场景可以拿到更稳定保障。

**工程指标**  
- 高优先级请求 TP99：看保护效果。  
- 低优先级积压时长：看代价。  
- 优先级队列饥饿率：看调度平衡。  
- 高峰期成功率：看实际收益。  
- 配额触发率：看资源紧张程度。

**证据链**  
- 队列数据  
- 业务优先级规则  
- 高峰监控  
- 配额日志

**低风险优化路径**  
1. 先给场景做优先级分层  
2. 再做队列和配额控制  
3. 最后优化饥饿保护和离线转移

**易错点**  
- 所有请求同优先级  
- 只保高优不管低优饥饿  
- 没有高峰期降级策略

## 6. 批量推理与实时推理如何并存？

**核心结论**  
批量推理和实时推理并存的关键，是从资源池、队列、SLA 和成本目标上分层治理，避免互相抢占。

**标准答案**  
我让批量和实时推理并存时，通常不会让它们共用完全相同的资源池。因为实时请求最怕排队和长尾，而批量任务更看吞吐和单位成本。工程上比较稳的做法，是把实时任务放进低延迟通道，把批量任务放进可中断、可延后的吞吐通道，并在高峰期允许实时任务优先抢占资源。成熟系统会把两者当成两类不同业务，而不是一个模型服务的两种用法。

**追问点答案**  
- 为什么批量任务适合低峰执行？  
因为对延迟不敏感，更适合让位给实时流量。  
- 批量和实时能共用模型吗？  
可以，但资源和调度最好分层。  
- 最容易冲突的点是什么？  
GPU/并发资源争抢和队列积压。

**原理补充**  
并存设计的本质，是用不同调度策略承接不同 SLA 的推理需求。

**架构扩展**  
成熟 AI 平台常把推理服务分成 online / async / batch 三层，并对不同层配置不同的模型档位、缓存策略和资源水位。

**工程指标**  
- 实时 TP99：看体验。  
- 批量吞吐：看资源效率。  
- 资源抢占冲突数：看隔离效果。  
- 低峰资源利用率：看经济性。  
- 批处理积压量：看运行健康度。

**证据链**  
- 调度配置  
- 队列统计  
- 资源利用率看板  
- 高峰/低峰对比数据

**低风险优化路径**  
1. 先分离队列和 SLA  
2. 再分层资源池  
3. 最后加抢占和离线窗口策略

**易错点**  
- 批量和实时完全混跑  
- 只看平均资源利用率  
- 高峰期无资源保护

## 7. 高峰期限流策略如何设计？

**核心结论**  
高峰期限流不是单纯挡流量，而是按业务价值、用户级别和请求成本分层保护核心能力。

**标准答案**  
我设计 AI 高峰期限流时，通常先看哪些流量是必须保住的，比如核心用户、核心功能、实时会话，然后再决定哪些请求可以排队、降级或直接拒绝。因为 AI 请求成本差异很大，同样一个 QPS，长上下文复杂任务和短问短答的资源消耗完全不同。工程上比较稳的做法，是结合用户级配额、场景级优先级、单请求预算和令牌桶/漏桶策略一起做，而不是只按 QPS 一刀切。成熟系统会把限流和降级联动，这样在保核心的同时还能给用户一个可预期反馈。

**追问点答案**  
- 为什么 AI 限流不能只看 QPS？  
因为不同请求的 token 和计算成本差别很大。  
- 限流后最好的用户反馈是什么？  
明确告知排队、降级或稍后重试，而不是无响应。  
- 限流和配额有什么关系？  
配额是长期约束，限流是瞬时保护。

**原理补充**  
限流的本质，是在资源不足时把损失控制在最小可接受范围内。

**架构扩展**  
成熟平台一般会同时做“入口限流 + 任务级预算 + 模型级配额”，这样既能挡住洪峰，也不会把资源消耗全压到后端模型层才发现出问题。

**工程指标**  
- 高峰限流命中率：看保护触发。  
- 核心请求成功率：看保核心效果。  
- 被限流请求平均等待时长：看体验。  
- 降级后完成率：看韧性。  
- 资源饱和告警数：看压力水平。

**证据链**  
- 限流规则  
- 高峰时段流量数据  
- 用户反馈  
- 降级记录

**低风险优化路径**  
1. 先按业务价值分层流量  
2. 再加限流和配额联动  
3. 最后补排队和降级反馈

**易错点**  
- 全部请求统一限流  
- 不区分请求成本  
- 限流后无用户反馈

## 8. 多模型服务如何做熔断降级？

::: tip L2 深度训练
不要透明替换模型，先验证任务能力和工具 schema 兼容：[模型路由、SLA 与成本](/deep-dives/ai-architecture/03-model-routing-cost)
:::

**核心结论**  
多模型服务的熔断降级要先明确哪些任务可以降质量、哪些只能降速度、哪些必须直接拒绝，避免故障时全线失控。

**标准答案**  
我做多模型熔断降级时，通常会先给不同任务定义可接受的退化方式。比如高价值实时问答可以从大模型降到中模型，低风险任务可以缩短上下文或减少重排，高风险审批类任务则宁可拒绝也不能错误放行。工程上比较稳的做法，是结合模型可用性、时延、成本和质量阈值做分层开关，触发时按预设顺序切换，而不是现场人工拍脑袋。成熟系统会把“降级路径”视作上线前就要演练的能力，而不是出了事故再想怎么办。

**追问点答案**  
- 为什么不是所有任务都能降级到小模型？  
因为有些任务的质量底线不能被打破。  
- 熔断和降级有什么区别？  
熔断是停止不健康路径，降级是切到更低成本或更弱能力路径。  
- 为什么要演练？  
因为故障发生时没有时间临场试错。

**原理补充**  
熔断降级的本质，是把故障时的资源和能力收敛到业务可接受范围内。

**架构扩展**  
成熟平台通常会给每条能力链路维护降级树，比如“重排关掉 -> 小模型替代 -> 缩上下文 -> 仅返回检索结果 -> 拒绝服务”。这样故障时才有秩序。

**工程指标**  
- 熔断命中率：看保护触发。  
- 降级后任务完成率：看业务保留度。  
- 错误扩散范围：看止损效果。  
- 故障恢复时长：看韧性。  
- 降级演练成功率：看准备程度。

**证据链**  
- 降级策略文档  
- 开关配置  
- 演练记录  
- 事故复盘

**低风险优化路径**  
1. 先定义任务级可接受降级路径  
2. 再接入自动触发机制  
3. 最后持续演练和复盘

**易错点**  
- 所有任务统一降级方式  
- 没有预定义切换顺序  
- 故障前不演练

## 9. GPU/CPU 混合部署如何取舍？

**核心结论**  
GPU/CPU 混合部署的关键，是根据模型规模、请求时延、吞吐需求和成本结构，把不同任务放到最合适的算力层，而不是简单追求“都上 GPU”。

**标准答案**  
我做 GPU/CPU 混合部署时，通常会先看任务是否需要高吞吐、高并发和复杂推理。大模型在线推理、高并发生成通常更适合 GPU；而轻量模型、预处理、规则判定、部分 embedding 或离线任务，则可能放在 CPU 上更划算。之所以不能所有任务都压 GPU，是因为 GPU 成本高，且很多请求并不值得用最贵资源。工程上比较稳的做法，是按任务类型、模型大小和 SLA 分层，把 GPU 用在最值钱的路径上。成熟团队会把混合部署看成资源经营问题，而不是单纯的技术偏好。

**追问点答案**  
- CPU 跑推理一定不划算吗？  
不一定，小模型或低并发场景可能很合适。  
- GPU 最大价值是什么？  
在复杂模型和高并发下提供更好吞吐与时延。  
- 混合部署为什么更难？  
因为调度、路由和运维复杂度都会上升。

**原理补充**  
混合部署的本质，是让不同算力资源承接不同价值密度的任务。

**架构扩展**  
成熟平台常见模式是“GPU 做主生成，CPU 做前后处理和轻任务”，这样整体成本结构通常比全 GPU 更健康。

**工程指标**  
- GPU 利用率：看昂贵资源效率。  
- CPU 任务成功率：看替代能力。  
- 单位请求算力成本：看经济性。  
- 路由命中准确率：看调度质量。  
- 算力空转比例：看资源浪费。

**证据链**  
- 模型与任务分类  
- 资源监控  
- 成本报表  
- 路由策略

**低风险优化路径**  
1. 先按任务和模型做分层  
2. 再局部试点 CPU 替代或前后处理下沉  
3. 最后优化调度和资源配比

**易错点**  
- 所有任务都上 GPU  
- 不看资源利用率  
- 路由策略和算力层脱节

## 10. 长文本请求如何切分执行？

**核心结论**  
长文本切分执行的关键，是按语义和任务目标拆段，并设计中间聚合逻辑，避免简单硬切导致上下文断裂和结果失真。

**标准答案**  
我处理长文本请求时，通常不会直接把文本机械分段后独立生成，因为很多任务依赖上下文连贯性。比较稳的做法，是先按章节、条款、函数、段落等自然边界切分，再按任务类型选择摘要、局部问答、分段抽取或 map-reduce 聚合。工程上还要定义段间共享信息和最终汇总策略，避免每段都各说各话。成熟团队不会把长文本问题只理解成“长度问题”，而会把它看成任务拆解和结果聚合问题。

**追问点答案**  
- 为什么不能只按 token 数硬切？  
因为会破坏语义结构。  
- map-reduce 适合什么场景？  
长文摘要、批量抽取和分段分析。  
- 聚合阶段为什么关键？  
因为最终答案质量常取决于聚合而不只是分段输出。

**原理补充**  
长文本切分的本质，是在上下文限制下保住任务语义和最终结果一致性。

**架构扩展**  
成熟系统通常会把长文本处理拆成“分段理解 + 中间结构化结果 + 汇总生成”，这样比直接全文硬塞模型更稳，也更便于回放和审计。

**工程指标**  
- 分段任务成功率：看局部质量。  
- 聚合一致率：看最终稳定性。  
- 平均 token 长度：看成本控制。  
- 漏关键信息率：看语义损失。  
- 总体 TP99：看执行效率。

**证据链**  
- 切分规则  
- 中间结果样本  
- 聚合失败案例  
- 成本和延迟数据

**低风险优化路径**  
1. 先按自然边界切分  
2. 再设计中间结构化结果  
3. 最后优化汇总逻辑

**易错点**  
- 只按长度硬切  
- 没有中间结果结构  
- 聚合逻辑过于随意

## 11. 结果复用策略如何设计？

**核心结论**  
结果复用策略要区分“可以直接复用的稳定结果”和“只能局部复用的中间结果”，避免为了省成本复用错误语境下的旧答案。

**标准答案**  
我设计结果复用时，通常先判断任务输出是不是强依赖上下文、时效和权限。像固定结构化抽取、中间检索结果、稳定规则判断，这些更适合复用；而高个性化、强时效、强上下文依赖的答案则需要更谨慎。工程上比较稳的做法，是把复用策略做成分层：先复用中间步骤，再视业务风险决定是否复用最终答案。成熟平台会给复用绑定版本、租户和权限信息，不会让“看起来相似”的请求直接共享结果。

**追问点答案**  
- 为什么中间结果常比最终答案更适合复用？  
因为语义更稳定、风险更低。  
- 最终答案复用最大风险是什么？  
上下文、权限或知识已变化。  
- 结果复用和缓存是一回事吗？  
相关，但复用更强调业务语义上的可重用性判断。

**原理补充**  
结果复用的本质，是把高重复、低变化的工作从在线推理里剥离出来。

**架构扩展**  
成熟系统通常会优先做“检索复用、摘要复用、工具结果复用”，最后才谨慎做最终答案复用，因为后者的语义风险最高。

**工程指标**  
- 复用命中率：看收益。  
- 错复用率：看风险。  
- 中间结果复用占比：看分层成熟度。  
- 成本下降幅度：看经济收益。  
- 版本失效正确率：看治理质量。

**证据链**  
- 复用策略文档  
- 命中日志  
- 错复用案例  
- 成本看板

**低风险优化路径**  
1. 先复用稳定中间结果  
2. 再补版本和权限约束  
3. 最后按低风险场景扩到最终结果

**易错点**  
- 不区分中间结果和最终答案  
- 忽略权限和时效  
- 只看复用收益不看错复用损失

## 12. token 消耗异常如何告警？

**核心结论**  
token 异常告警要同时看请求粒度和聚合粒度，识别单次异常、租户异常和链路退化，而不是只看总账单上涨。

**标准答案**  
我做 token 异常告警时，通常会从三个层次看：单请求是否异常放大、某租户或某功能是否持续高消耗、整体链路是否在某次升级后出现系统性上涨。因为 AI 成本异常很多时候不是业务真的增长，而是 Prompt 漏控、循环调用、上下文拼装膨胀或者缓存失效。工程上比较稳的做法，是按模型、功能、租户、版本维度做看板和阈值，触发后能快速定位到具体链路。成熟团队不会等月底对账才发现成本炸了。

**追问点答案**  
- 为什么总账单告警不够？  
因为发现得太晚，也无法快速定位责任点。  
- 哪些问题最常导致 token 异常？  
上下文膨胀、循环调用、重试失控、缓存失效。  
- 为什么版本维度也重要？  
因为升级经常会改变 token 消耗曲线。

**原理补充**  
异常告警的本质，是把成本问题从事后统计变成实时运行信号。

**架构扩展**  
成熟 AI 平台通常会把 token 异常和 Prompt 版本、模型版本、缓存命中一起联看，否则很难真正找到根因。

**工程指标**  
- 单请求 token 异常率：看局部风险。  
- 租户级成本突增次数：看经营风险。  
- 版本升级后 token 漂移幅度：看回归问题。  
- 告警到定位时长：看可运维性。  
- 缓存失效相关异常占比：看优化空间。

**证据链**  
- token 看板  
- 版本变更记录  
- 缓存监控  
- 异常样本

**低风险优化路径**  
1. 先建立分维度 token 监控  
2. 再给异常模式设阈值  
3. 最后联动止损和回归

**易错点**  
- 只看总成本  
- 无法按功能/租户定位  
- 发现异常后没有自动止损

## 13. 供应商故障切换怎么做？

::: tip L2 深度训练
设计切换、排队、确定性兜底、半开探测和阶梯回切：[模型路由、SLA 与成本](/deep-dives/ai-architecture/03-model-routing-cost)
:::

**核心结论**  
供应商故障切换要提前准备统一接口、可替代模型、回归基线和流量开关，不能等故障发生时临时接别家。

**标准答案**  
我做供应商切换时，通常会先在架构上保留统一调用层和至少一个可接受的备用方案，因为故障时最怕的是“理论上能切，实际上从没切过”。工程上比较稳的做法，是把调用抽象、路由策略、Prompt 兼容性、回退开关和可用性监控提前做好，并定期演练故障转移。成熟团队不会等主供应商挂了才开始适配备用模型，而是把切换当成常规韧性能力建设。

**追问点答案**  
- 为什么统一调用层很重要？  
因为它能减少业务侧切换成本。  
- 备用模型一定要完全等价吗？  
不一定，但至少要满足最低业务底线。  
- 为什么要演练？  
因为未演练的切换大概率会在最差的时候失败。

**原理补充**  
供应商切换的本质，是把外部依赖故障对业务的影响控制在最小范围内。

**架构扩展**  
成熟平台通常会把“热切换”和“降级切换”区分开：前者追求无感替换，后者至少保证核心链路不停。

**工程指标**  
- 故障切换成功率：看韧性。  
- 切换耗时：看止损速度。  
- 切换后质量退化幅度：看备用方案质量。  
- 演练覆盖率：看准备程度。  
- 业务无感比例：看用户影响。

**证据链**  
- 路由配置  
- 故障演练记录  
- 切换后指标对比  
- 供应商健康监控

**低风险优化路径**  
1. 先统一调用抽象  
2. 再准备最低可用备用模型  
3. 最后定期演练切换

**易错点**  
- 没有备用路径  
- 备用模型从未验证  
- 供应商切换全靠手工临场处理

## 14. TTFT 和 TP99 如何同时优化？

**核心结论**  
同时优化 TTFT 和 TP99 的关键，是把“首字快”和“整体稳”拆开治理，分别找前置耗时和长尾瓶颈。

**标准答案**  
我优化 TTFT 和 TP99 时，通常不会把它们混成一个问题。TTFT 更受前置处理、排队、流式返回和上下文长度影响，而 TP99 往往还叠加模型生成时长、重排、工具调用和长尾请求。工程上比较稳的做法，是先拆链路看每一段耗时，再针对 TTFT 做前置缓存和并行，对 TP99 做长尾请求分层、限时、降级和资源保护。成熟团队会接受“不是所有优化都同时拉满”，而是按业务优先级做平衡。

**追问点答案**  
- 为什么 TTFT 和 TP99 会矛盾？  
因为让首字快不代表整体生成更稳，反之亦然。  
- 长尾请求为什么要单独处理？  
因为它们会显著拉高 TP99。  
- 流式输出能解决 TP99 吗？  
主要改善 TTFT 和感知体验，不一定改善总时长。

**原理补充**  
这两个指标本质上反映的是不同维度的用户体验：开始等待和整体尾延迟。

**架构扩展**  
成熟 AI 平台通常会对长上下文、大输出、重链路请求单独建档，这样优化时才不会让少数极端请求拖垮整体看板。

**工程指标**  
- TTFT：看首响体验。  
- TP99：看长尾稳定性。  
- 前置链路耗时：看 TTFT 瓶颈。  
- 长尾请求占比：看 TP99 根因。  
- 分场景指标差异：看是否需要分层治理。

**证据链**  
- tracing  
- 分场景延迟看板  
- 请求长度分布  
- 长尾样本

**低风险优化路径**  
1. 先拆分 TTFT 和 TP99 根因  
2. 再分别做前置优化和长尾治理  
3. 最后按场景设差异化目标

**易错点**  
- 只看平均延迟  
- 不拆分前置和生成阶段  
- 用一套优化手段解决所有延迟问题

## 15. 成本中心如何分摊到业务线？

**核心结论**  
AI 成本分摊要能映射到业务线、产品功能、租户和模型版本，否则成本就只是平台账单，不是经营指标。

**标准答案**  
我做成本中心分摊时，通常会先统一标签和调用维度，比如业务线、场景、租户、模型、Prompt 版本、功能入口。因为没有这些维度，月底只能看到总账单，没法知道谁在花、值不值。工程上比较稳的做法，是在推理网关层统一打点和归因，让每次调用都能落到一个明确成本中心。成熟团队会把成本分摊和配额、预算、ROI 一起看，而不是只做财务归集。

**追问点答案**  
- 为什么推理网关最适合做归因？  
因为它最接近统一调用入口。  
- 只按业务线分摊够吗？  
通常不够，功能和租户维度也很关键。  
- 成本分摊为什么和 ROI 有关？  
因为只有知道谁花了多少，才能判断值不值。

**原理补充**  
分摊的本质，是把共享 AI 平台成本转换成业务可经营的责任单元。

**架构扩展**  
成熟平台通常会把成本看板开放给业务负责人，而不是只留在平台团队内部，这样成本治理才真正能形成闭环。

**工程指标**  
- 成本归因完整率：看治理基础。  
- 业务线单位请求成本：看经济性。  
- 功能级预算偏差：看控制质量。  
- 未归因调用比例：看数据缺口。  
- ROI 可计算覆盖率：看经营成熟度。

**证据链**  
- 标签规范  
- 网关打点日志  
- 成本看板  
- 预算配置

**低风险优化路径**  
1. 先统一标签与归因维度  
2. 再打通成本看板  
3. 最后接预算和 ROI 复盘

**易错点**  
- 只有平台总账单  
- 归因维度缺失  
- 成本分摊与预算脱节

## 16. 计费与配额系统如何设计？

**核心结论**  
计费与配额系统要同时支持实时控制和事后结算，既能在请求发生时限额，也能在周期结束时准确归账。

**标准答案**  
我设计计费与配额系统时，通常会把“实时配额控制”和“离线账单核算”分开。因为实时链路更关心能不能放行、还剩多少额度，账单链路则关心统计准确、可追溯和可对账。工程上比较稳的做法，是在网关层记录标准化使用量，在配额服务里做实时判断，在结算链路里做按日或按周期汇总。成熟团队会把 token、模型档位、重试、缓存节省和特殊优惠都纳入统一计费模型，而不是各算各的。

**追问点答案**  
- 为什么实时控制和账单核算要分开？  
因为它们对时延和准确性的要求不同。  
- 配额只按 token 控制够吗？  
不一定，还可能要按请求数、模型档位和租户等级控制。  
- 对账为什么重要？  
因为计费一旦有争议，就需要证据闭环。

**原理补充**  
计费与配额系统的本质，是把 AI 消耗从资源使用转成业务规则可理解的收费和限制单位。

**架构扩展**  
成熟平台通常会把计费、配额、成本归因和降级打通：额度耗尽时不只是拒绝，还可能切换到低成本模型或只提供基础能力。

**工程指标**  
- 实时配额判断时延：看链路可用性。  
- 账单对账准确率：看财务可信度。  
- 超额调用拦截率：看控制效果。  
- 计费争议率：看模型质量。  
- 未结算调用比例：看闭环完整度。

**证据链**  
- 计费规则  
- 配额日志  
- 对账报表  
- 争议工单

**低风险优化路径**  
1. 先统一计量口径  
2. 再分离实时控制和账单核算  
3. 最后补对账和争议处理

**易错点**  
- 把配额和账单混在一个慢系统里  
- 计量口径不统一  
- 没有对账能力

## 17. 服务发布如何防回归？

**核心结论**  
AI 服务发布防回归的关键，是把模型、Prompt、工具、知识和缓存策略一起纳入回归，而不是只测接口可用。

**标准答案**  
我做 AI 服务发布防回归时，通常会把回归对象拆成几个层次：接口层、模型输出层、工具调用层、RAG 结果层和成本层。因为 AI 服务的退化往往不是 HTTP 500，而是结构化失败、检索偏移、成本上涨或高风险输出变多。工程上比较稳的做法，是维护多层评测集和线上小流量双跑，确保质量、时延和成本一起过门禁。成熟团队会把发布看作行为变化管理，而不是普通 API 发布。

**追问点答案**  
- 为什么接口正常不代表没回归？  
因为 AI 行为问题常藏在结果质量和稳定性里。  
- 哪些东西都需要版本化？  
模型、Prompt、工具 schema、知识版本、缓存策略。  
- 双跑的价值是什么？  
在不影响用户的情况下验证新版本行为。

**原理补充**  
防回归的本质，是让“行为变化”在上线前可见、可比较、可止损。

**架构扩展**  
成熟平台通常会把 AI 发布和传统代码发布分开治理，因为 AI 发布的风险对象更多，也更难通过单元测试发现。

**工程指标**  
- 回归通过率：看发布质量。  
- 双跑差异率：看行为变化。  
- 发布后回滚率：看门禁有效性。  
- 高风险错误增量：看上线风险。  
- 成本漂移幅度：看隐性退化。

**证据链**  
- 回归报告  
- 双跑样本  
- 版本记录  
- 回滚记录

**低风险优化路径**  
1. 先版本化所有关键组件  
2. 再建设多层回归集  
3. 最后接入双跑和灰度

**易错点**  
- 只测接口不测行为  
- 关键组件不版本化  
- 上线前无灰度和双跑

## 18. AI API 的幂等与重试怎么做？

**核心结论**  
AI API 的幂等与重试要区分“纯推理请求”和“带副作用请求”，前者重点控制重复成本，后者重点避免重复执行。

**标准答案**  
我做 AI API 幂等与重试时，通常先看请求有没有副作用。像普通问答、摘要、分类这类纯推理请求，重复执行主要影响成本和一致性；但如果请求会落工单、发消息、写数据库，那就必须有幂等键、状态查询和防重复执行机制。工程上比较稳的做法，是让 API 层统一接受 request_id，并对可缓存结果做复用，对副作用类请求做状态落盘和去重。成熟团队会把 AI API 当成分布式服务接口来治理，而不是因为它是“AI 请求”就忽略可靠性问题。

**追问点答案**  
- 为什么纯推理请求也要考虑幂等？  
因为重试会带来重复成本和结果波动。  
- 有副作用请求最怕什么？  
重试导致重复创建、重复发送或重复计费。  
- request_id 为什么重要？  
因为它是幂等识别和追踪的统一锚点。

**原理补充**  
幂等治理的本质，是让重复请求不会产生不可接受的额外后果。

**架构扩展**  
成熟平台通常会把 AI API 的幂等和计费、审计、缓存打通，这样重复请求既能避免副作用，也能避免重复花钱。

**工程指标**  
- 重试成功率：看韧性。  
- 重复执行事件数：看副作用风险。  
- request_id 覆盖率：看治理基础。  
- 重试引发成本增量：看经济影响。  
- 状态查询命中率：看可靠性设计。

**证据链**  
- API 协议  
- 幂等日志  
- 去重记录  
- 成本数据

**低风险优化路径**  
1. 先统一 request_id 机制  
2. 再分纯推理和副作用请求治理  
3. 最后打通缓存、计费和审计

**易错点**  
- 认为 AI API 不需要幂等  
- 副作用请求无状态记录  
- 重试策略无差别

## 19. 全链路压测如何覆盖 LLM 依赖？

**核心结论**  
覆盖 LLM 依赖的全链路压测不能只测业务接口，要同时模拟模型响应特征、外部供应商波动、缓存命中变化和降级策略触发。

**标准答案**  
我做 AI 链路压测时，通常不会只压入口 API，因为真正的瓶颈可能在模型调用、检索、重排、缓存失效或外部供应商限流。比较稳的做法，是分层做压测：先压内部组件，再做端到端高峰模拟，同时覆盖供应商变慢、部分失败和高成本长请求比例上升这类异常情况。工程上还会特别看限流、降级和回滚逻辑在高峰下是否真的生效。成熟团队把 AI 压测看成“业务链路 + 外部依赖 + 成本韧性”的综合演练。

**追问点答案**  
- 为什么不能只压业务入口？  
因为外部模型依赖和内部检索层都可能是主瓶颈。  
- 供应商异常为什么也要模拟？  
因为真实高峰时经常和外部波动叠加。  
- 成本为什么也要压测？  
因为高峰时请求结构变化可能让成本飙升。

**原理补充**  
全链路压测的本质，是在接近真实生产条件下验证容量、韧性和止损能力。

**架构扩展**  
成熟平台通常会准备“轻压测、重压测、故障注入演练”三层，不会只做正常路径高并发测试。

**工程指标**  
- 端到端 TP99：看体验。  
- 供应商异常下成功率：看韧性。  
- 降级触发成功率：看保护能力。  
- 高峰期单请求成本变化：看经济风险。  
- 核心链路可用性：看总体承载能力。

**证据链**  
- 压测脚本  
- tracing  
- 外部依赖模拟结果  
- 降级和限流日志

**低风险优化路径**  
1. 先分层压内部链路  
2. 再压端到端和异常场景  
3. 最后把止损逻辑纳入演练

**易错点**  
- 只测正常路径  
- 不模拟供应商波动  
- 不观察成本变化

## 20. 如何做 AI 平台 FinOps？

::: tip L2 深度训练
用业务结果成本、路由升级率、重试和预算燃烧率运营：[模型路由、SLA 与成本](/deep-dives/ai-architecture/03-model-routing-cost)
:::

**核心结论**  
AI 平台 FinOps 的核心，是把模型调用、算力、缓存、重试和知识链路成本变成可归因、可预算、可优化的经营指标体系。

**标准答案**  
我做 AI 平台 FinOps 时，通常不会只看云账单或模型账单，而是先把成本拆成可经营单元：模型调用、GPU/CPU 资源、embedding 重建、检索与重排、缓存、重试和失败浪费。因为 AI 成本失控往往不是单点问题，而是多个细碎环节叠加。工程上比较稳的做法，是建立业务线/租户/功能级成本归因，把成本与质量和收入、人效一起看，再做预算、配额和优化。成熟团队做 FinOps 的目标，不是单纯压低成本，而是提升单位价值产出。

**追问点答案**  
- 为什么 AI FinOps 比普通云成本更复杂？  
因为它还叠加了模型行为、token 消耗和任务成功率。  
- 为什么不能只盯低价模型？  
因为质量下降可能带来更高业务损失。  
- FinOps 和架构设计有什么关系？  
架构分层、缓存、路由和降级都会直接影响成本曲线。

**原理补充**  
AI FinOps 的本质，是把 AI 能力从“昂贵实验”变成“可经营服务”。

**架构扩展**  
成熟 AI 平台通常会把 FinOps 嵌进日常研发和平台治理，而不是等财务提醒超预算才开始处理。模型路由、缓存和预算本身就是 FinOps 设计的一部分。

**工程指标**  
- 单位业务价值成本：看投入产出。  
- 模型/租户/功能成本归因率：看治理基础。  
- 缓存节省比例：看优化收益。  
- 失败浪费成本占比：看无效消耗。  
- 预算达成率：看经营能力。

**证据链**  
- FinOps 看板  
- 成本拆解模型  
- 预算与配额记录  
- 优化前后对比

**低风险优化路径**  
1. 先做成本归因和看板  
2. 再接预算、配额和优化手段  
3. 最后按业务价值持续复盘

**易错点**  
- 把 FinOps 理解成“压价”  
- 不做成本归因  
- 成本治理与质量指标脱节
