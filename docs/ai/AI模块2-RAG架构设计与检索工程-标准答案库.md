# AI模块2 - RAG架构设计与检索工程 - 标准答案库

## 1. 什么时候必须做 RAG？

**核心结论**  
当业务需要基于私有知识、最新资料、可审计引用或高事实准确率回答问题时，通常就必须做 RAG，而不是只靠基础模型裸答。

**标准答案**  
我判断一个场景是否必须做 RAG 时，通常先看回答是不是依赖企业私有数据、频繁更新的知识、带权限边界的文档，或者必须给出引用依据。如果这些要求成立，只靠基础模型通常很难稳定满足，因为模型参数里的知识不一定最新，也无法天然满足企业内数据和审计要求。工程上比较稳的做法，是让模型只负责理解和生成，把“提供事实依据”交给检索层完成。成熟架构师谈 RAG，不会把它说成“让模型更聪明”，而会说成“让模型在证据边界内回答问题”。

**追问点答案**  
- 为什么有最新知识就更适合 RAG？  
因为基础模型参数更新无法跟上业务知识变化。  
- RAG 能完全消除幻觉吗？  
不能，但能显著降低无依据输出。  
- 什么场景不一定需要 RAG？  
纯创意生成或完全不依赖外部知识的任务。

**原理补充**  
RAG 的本质，是把知识访问从模型内部参数转移到外部可更新、可治理的检索系统。

**架构扩展**  
真实业务里，客服、内部知识库、合同问答、代码库问答、研究助手这类场景，RAG 往往不是锦上添花，而是能否上线的基础条件。

**工程指标**  
- 引用覆盖率：看证据链完整性。  
- 无依据回答率：看幻觉控制效果。  
- 知识更新时间到生效时长：看新鲜度。  
- 权限命中准确率：看安全边界。  
- 任务成功率：看业务效果。

**证据链**  
- 评测集  
- 引用日志  
- 文档更新记录  
- 用户反馈

**低风险优化路径**  
1. 先锁定必须依赖私有知识的场景  
2. 再做最小 RAG 链路试点  
3. 最后补引用、权限和评测

**易错点**  
- 需要私有知识却仍然裸用模型  
- 把 RAG 当“搜索 + 拼接”不做治理  
- 不定义哪些场景必须带引用

## 2. 文档切块策略怎么设计？

**核心结论**  
切块策略要服务检索目标，核心是让每块既保留足够语义完整性，又不过度冗长，能被高效召回和准确引用。

**标准答案**  
我设计切块策略时，通常不会先固定字符数，而是先看文档类型和问答粒度。比如技术文档、合同、FAQ、代码说明，它们天然结构就不一样，切块策略也不该一样。工程上比较稳的做法，是优先按自然边界切，比如标题、段落、列表或章节，再在长度超标时做二次切分，并保留适度 overlap。这样做的原因，是如果切得太碎，语义会丢；切得太大，召回噪音和成本都会上升。成熟团队通常会把切块和评测一起调，而不是把它当固定参数。

**追问点答案**  
- 为什么不能只按固定字数切？  
因为语义边界可能被硬切断。  
- overlap 为什么重要？  
因为跨段信息经常需要一点上下文承接。  
- 代码和合同文档切块能一样吗？  
通常不能，结构和问答粒度差异很大。

**原理补充**  
切块的本质，是在“语义完整”和“检索精度”之间找最合适的平衡。

**架构扩展**  
成熟 RAG 系统往往会按文档类型维护不同切块器，比如 FAQ 按问答对切，合同按条款切，技术文档按标题层级切，这样召回和引用质量通常明显更好。

**工程指标**  
- 召回命中率：看切块是否适配检索。  
- 引用精确率：看语义边界是否合理。  
- 平均 chunk 长度：看成本和噪音。  
- 用户追问率：看一次召回信息是否够用。  
- Chunk 冗余度：看索引效率。

**证据链**  
- 切块样本  
- 检索命中案例  
- 引用偏差样本  
- 评测结果

**低风险优化路径**  
1. 先按文档类型设计自然边界切块  
2. 再调长度和 overlap  
3. 最后用评测集迭代优化

**易错点**  
- 全文档统一一个切块参数  
- 只看长度不看语义边界  
- 不做切块效果评测

## 3. 向量库选型看哪些指标？

**核心结论**  
向量库选型要同时看检索效果、写入更新能力、过滤能力、延迟、成本和运维复杂度，而不是只看 ANN 指标。

**标准答案**  
我选向量库时，通常先看业务对召回质量、并发规模、更新频率和过滤条件的要求。因为有的场景是高并发在线检索，有的则更偏离线知识库更新，瓶颈完全不同。工程上比较稳的做法，是重点比较查询延迟、过滤能力、多租户支持、增量更新代价、索引重建成本和运维方式，而不是只看理论向量搜索性能。成熟团队会把向量库当成 RAG 数据层的一部分来评估，明确它到底要解决的是“更快”“更准”还是“更好管”。

**追问点答案**  
- 为什么过滤能力很重要？  
因为权限、租户和文档类型筛选往往是业务刚需。  
- 增量更新为什么要单独评估？  
因为知识库不是只建一次，后续持续更新是常态。  
- 自建和托管怎么选？  
看团队运维能力和可控性需求。

**原理补充**  
向量库不是孤立组件，它的价值取决于是否贴合检索链路和知识治理要求。

**架构扩展**  
成熟团队在做向量库选型时，往往会同时测混合检索和 metadata filter，因为生产里真正痛的常常不是 ANN 本身，而是业务过滤和更新治理。

**工程指标**  
- 查询 TP99：看在线体验。  
- 召回率：看检索效果。  
- 增量写入延迟：看知识更新能力。  
- 过滤正确率：看业务可用性。  
- 运维成本：看长期可持续性。

**证据链**  
- 压测结果  
- 业务过滤需求清单  
- 更新频率统计  
- 成本评估

**低风险优化路径**  
1. 先明确检索和更新需求  
2. 再做小规模压测和效果对比  
3. 最后结合团队运维能力定方案

**易错点**  
- 只看搜索性能榜单  
- 忽略过滤和更新能力  
- 不评估实际运维成本

## 4. embedding 模型如何选择？

**核心结论**  
Embedding 模型选择要看语种、领域语义、向量质量、成本和延迟，核心是它是否能把“你的文档和你的查询”映射到有效相似空间。

**标准答案**  
我选 embedding 模型时，通常会先看知识库的语料类型和查询方式，是通用文本、代码、法律文档、产品知识，还是多语言混合内容。因为 embedding 的好坏，最终决定的是检索是否能把真正相关的 chunk 拉出来。工程上比较稳的做法，是直接用自己的文档和真实问题做召回评测，而不是只看公开 benchmark。成熟团队会把 embedding 模型的质量、延迟和单次构建成本一起评估，因为知识库规模一大，向量重建代价就很明显。

**追问点答案**  
- 为什么 embedding 选型不能只看公开分数？  
因为你的领域语料和 benchmark 语料可能差很多。  
- 多语言场景为什么更要谨慎？  
因为不同语种 embedding 质量差异可能很大。  
- embedding 模型更新为什么有成本？  
因为通常意味着全量或部分重建向量索引。

**原理补充**  
Embedding 模型的本质，是决定“语义相似”在向量空间里如何被表示。

**架构扩展**  
成熟团队通常会保留一套 embedding 评测集和重建流程，避免后续换模型时完全失控。因为一旦知识库规模起来，embedding 升级本身就是工程项目。

**工程指标**  
- TopK 召回率：看语义表示质量。  
- 多语种召回差异：看语言适配性。  
- 向量构建成本：看重建代价。  
- 查询时延：看线上可用性。  
- 文档重建周期：看运维压力。

**证据链**  
- 召回评测集  
- 多语种样本对比  
- 向量重建记录  
- 成本测算

**低风险优化路径**  
1. 先用真实问答样本做 embedding 对比  
2. 再评估重建成本  
3. 最后定版本策略和升级路径

**易错点**  
- embedding 选型脱离真实文档  
- 忽略重建成本  
- 多语言场景不单独评测

## 5. 混合检索（BM25+向量）何时必要？

**核心结论**  
当查询同时依赖关键词精确匹配和语义相似理解，尤其涉及术语、编号、缩写、实体名时，混合检索通常比纯向量更稳。

**标准答案**  
我判断要不要做混合检索时，通常先看业务问题里是否经常出现精确关键词，比如产品型号、报错码、条款编号、表名字段名这类内容。如果是，这类查询仅靠向量相似可能会漏掉最关键的命中；但如果只用关键词，又会丢掉语义改写和自然表达的能力。工程上比较稳的做法，是让 BM25 承担关键词召回，让向量承担语义召回，再通过融合或重排选出最终结果。成熟团队不会机械地说“向量比关键词高级”，而是会根据查询分布决定是否混合。

**追问点答案**  
- 什么场景最适合混合检索？  
实体名、代码、编号、专业术语密集的场景。  
- 混合检索一定比纯向量好吗？  
不一定，要看查询分布和治理成本。  
- 多一步融合会不会更慢？  
会增加一些开销，但很多场景收益明显。

**原理补充**  
混合检索的核心，是把“词面相关性”和“语义相关性”两种信号结合起来。

**架构扩展**  
在企业知识库、代码问答、工单系统和法规检索里，混合检索往往是事实上的默认方案，因为这些场景既有术语精确性，又有自然语言表达多样性。

**工程指标**  
- TopK 召回率提升幅度：看混合收益。  
- 术语类问题命中率：看关键词价值。  
- 最终重排前噪音率：看召回质量。  
- 查询延迟增量：看成本。  
- 用户追问率：看答案相关性。

**证据链**  
- 按问题类型拆分的评测结果  
- 关键词遗漏案例  
- 语义召回失败案例  
- 延迟对比数据

**低风险优化路径**  
1. 先识别术语/编号型问题占比  
2. 再对这类查询试点混合检索  
3. 最后视收益决定是否全面推广

**易错点**  
- 默认纯向量解决一切  
- 引入混合检索后不评估延迟和收益  
- 不按查询类型分层

## 6. 重排模型怎么接入主链路？

**核心结论**  
重排模型适合在初步召回后缩小候选集，用更细粒度语义判断提升最终相关性，但必须控制好延迟和调用成本。

**标准答案**  
我接入重排模型时，通常不会让它直接面对全量知识库，而是先由向量或混合检索召回一批候选，再用重排模型对前几十条结果重新排序。这样做的原因，是重排模型的精度通常更高，但成本和时延也更高，不适合直接做第一层检索。工程上比较稳的做法，是把重排放在高价值链路或高模糊度问题上，并监控它对召回质量和 TP99 的影响。成熟团队会把重排当成“精排层”，而不是默认每个请求都必须经过的固定步骤。

**追问点答案**  
- 为什么重排不直接全库使用？  
因为成本和延迟通常扛不住。  
- 什么请求最适合加重排？  
候选较多、语义区分难、答案要求高准确的请求。  
- 重排也可以按需触发吗？  
可以，这样更节省成本。

**原理补充**  
重排的本质，是在较小候选集上做更贵但更准确的相关性判断。

**架构扩展**  
成熟 RAG 系统常见做法是“两段式”检索：第一段追求不漏，第二段追求更准。重排是否常驻主链路，取决于收益能否覆盖它的时延和成本。

**工程指标**  
- 最终命中率提升：看效果。  
- TP99 增量：看性能代价。  
- 重排触发率：看使用范围。  
- 重排后引用正确率：看业务价值。  
- 单请求成本增加：看经济性。

**证据链**  
- 无重排/有重排对比实验  
- 候选集质量数据  
- 延迟曲线  
- 成本分析

**低风险优化路径**  
1. 先在高价值场景试点重排  
2. 再优化候选数量和触发条件  
3. 最后决定是否扩展到主链路

**易错点**  
- 让所有请求都强制走重排  
- 不控制候选集规模  
- 只看效果不看 TP99

## 7. 查询改写在 RAG 中有什么价值？

**核心结论**  
查询改写的价值，在于把用户自然表达转成更适合检索系统理解的查询形式，从而提升召回质量和检索稳定性。

**标准答案**  
我用查询改写时，通常会针对用户输入过短、过口语化、上下文依赖强或者存在指代省略的情况。因为用户真实问题往往并不是最适合检索的表达，比如“这个怎么配”“它会报什么错”这种问题，如果不结合上下文补全，检索效果通常很差。工程上比较稳的做法，是把改写限定在提高检索表达质量，而不是让模型自由发挥生成新问题。成熟团队会持续监控改写前后召回质量，避免改写带来语义偏移。

**追问点答案**  
- 为什么查询改写容易带来风险？  
因为改写可能偏离用户原意。  
- 什么情况下最值得改写？  
问题过短、指代强、上下文依赖高时。  
- 改写一定要用 LLM 吗？  
不一定，规则和模板也能处理一部分。

**原理补充**  
查询改写的核心，是减少用户表达和知识表达之间的语义落差。

**架构扩展**  
成熟 RAG 产品里，查询改写经常和会话记忆、术语扩展、模板化增强一起用，但都会保持“尽量不改变真实意图”这条底线。

**工程指标**  
- 改写后召回提升率：看收益。  
- 改写偏移率：看风险。  
- 短问题命中率：看特定价值。  
- 改写耗时：看性能代价。  
- 用户追问率：看实际效果。

**证据链**  
- 改写前后对比集  
- 偏移样本  
- 会话上下文日志  
- 召回结果对比

**低风险优化路径**  
1. 先只处理最明显的短问题和指代问题  
2. 再评估偏移率  
3. 最后逐步扩到更多场景

**易错点**  
- 改写过度自由发挥  
- 不监控语义偏移  
- 改写链路无回退

## 8. 多跳检索如何落地？

**核心结论**  
多跳检索适合答案需要跨多个文档或多个事实组合推理的场景，关键是把检索拆成可解释步骤，而不是一次性盲搜。

**标准答案**  
我做多跳检索时，通常会先判断问题是不是单次召回就能回答，如果需要先找到 A 再通过 A 找 B，这时才值得做多跳。因为多跳链路一旦上来，复杂度、成本和失败点都会增加。工程上比较稳的方式，是先让模型或规则识别关键中间实体，再分步检索和聚合，同时保留每一步的证据。成熟系统不会把多跳当默认模式，而会针对复杂问题或高价值场景专门触发。

**追问点答案**  
- 多跳为什么复杂？  
因为每一跳都可能带来误差累积。  
- 什么时候不该上多跳？  
单跳问题占绝大多数且收益不明显时。  
- 为什么要保留每一步证据？  
便于解释、调试和审计。

**原理补充**  
多跳检索的本质，是把复杂问题拆成多个可检索的中间子问题。

**架构扩展**  
在研究助手、代码依赖分析、复杂合规问答里，多跳检索会带来明显收益，但成熟团队通常会严格限制触发条件，避免把所有请求都拖进重链路。

**工程指标**  
- 多跳问题任务成功率：看收益。  
- 每跳命中率：看链路质量。  
- 多跳整体延迟：看性能代价。  
- 误差累积率：看风险。  
- 证据完整率：看可解释性。

**证据链**  
- 多跳评测集  
- 每跳日志  
- 中间实体识别结果  
- 失败样本库

**低风险优化路径**  
1. 先识别真正需要多跳的问题类型  
2. 再用可解释步骤试点  
3. 最后控制触发范围和链路长度

**易错点**  
- 所有复杂问题都盲目上多跳  
- 不记录每一步证据  
- 多跳收益没有单独评估

## 9. 引用溯源如何设计成可审计？

**核心结论**  
引用溯源要做到“答案中的关键结论都能追到具体文档片段、具体版本和具体权限上下文”，这样才真正可审计。

**标准答案**  
我设计引用溯源时，通常会把文档 ID、chunk ID、文档版本、更新时间和检索时间一起记录下来，而不是只在前端简单显示一个来源名称。因为真正出问题时，团队需要知道模型到底基于哪一版内容回答的，用户当时有没有权限看到这段内容。工程上比较稳的做法，是让生成阶段保留答案和引用片段的对应关系，并在日志里记录完整链路，便于回放和追责。成熟团队谈引用，不会只讲“给用户看来源”，还会讲后端审计和版本追踪。

**追问点答案**  
- 为什么来源名称不够？  
因为无法精确定位到片段和版本。  
- 权限上下文为什么也要记录？  
因为可见不等于对所有用户都可见。  
- 引用展示和审计是同一层吗？  
不是，展示是产品能力，审计是治理能力。

**原理补充**  
可审计引用的核心，是让答案与证据之间存在可回放、可验证的强关联。

**架构扩展**  
高风险场景里，引用溯源是决定系统能否真正上线的重要条件之一。没有它，出现错误时就无法解释是知识库问题、检索问题还是生成问题。

**工程指标**  
- 引用完整率：看证据覆盖度。  
- 引用可回放成功率：看审计质量。  
- 错误引用率：看风险。  
- 版本追溯成功率：看治理成熟度。  
- 高风险答案带引用比例：看保护力度。

**证据链**  
- 检索日志  
- chunk 与答案映射  
- 文档版本记录  
- 审计回放结果

**低风险优化路径**  
1. 先记录文档/片段/版本信息  
2. 再建立答案到引用的映射  
3. 最后做回放和审计工具

**易错点**  
- 只有前端来源展示没有后端审计  
- 不记录版本  
- 引用和答案没有精确映射

## 10. 知识库增量更新如何保证一致性？

**核心结论**  
知识库增量更新的一致性重点，是让文档原文、向量索引、元数据和权限信息保持同一版本语义，避免“文档已经变了，检索还是旧的”。

**标准答案**  
我做知识库增量更新时，通常会把更新链路拆成文档入库、清洗切块、向量生成、索引替换和权限元数据刷新几步，并给每次更新打版本标记。因为 RAG 里最常见的问题之一就是数据已经改了，但检索层和权限层没同步好。工程上比较稳的做法，是采用可追踪的增量任务和幂等更新逻辑，必要时用双索引或版本切换避免在线查询读到半更新状态。成熟团队不会只关心“更新快不快”，还会关注“更新过程是否可审计、可回退”。

**追问点答案**  
- 为什么知识库更新也要版本化？  
因为要避免线上读到混合状态。  
- 双索引切换适合什么场景？  
大批量更新或高风险知识库。  
- 权限信息为什么要一起更新？  
因为内容可见性和文档内容一样重要。

**原理补充**  
增量更新一致性的本质，是保证检索视图和知识源在同一语义时间点上对齐。

**架构扩展**  
成熟系统通常会把“知识更新时间”和“知识生效时间”区别开，尤其在法规、价格、产品政策类场景里，这个差异很关键。

**工程指标**  
- 更新成功率：看链路稳定性。  
- 文档到索引生效时长：看时效性。  
- 半更新状态错误数：看一致性风险。  
- 回滚成功率：看治理能力。  
- 权限同步延迟：看安全边界。

**证据链**  
- 更新任务日志  
- 版本记录  
- 索引切换记录  
- 权限同步状态

**低风险优化路径**  
1. 先建立版本化更新链路  
2. 再做幂等与审计  
3. 最后针对大批量更新引入双索引切换

**易错点**  
- 只更新文档不更新索引  
- 没有版本标记  
- 权限元数据滞后

## 11. 多租户知识库怎么隔离？

**核心结论**  
多租户知识库隔离要同时控制数据存储、检索过滤、缓存和审计，不能只靠前端带一个 tenant_id。

**标准答案**  
我做多租户隔离时，通常会先明确隔离级别，是逻辑隔离、索引级隔离还是物理隔离。因为不同业务和合规要求下，代价和风险完全不同。工程上比较稳的做法，是让租户信息从文档入库到检索过滤、缓存键、日志审计一路贯穿，并对高敏感租户做更强隔离。成熟团队不会把多租户隔离看成单点过滤条件，而会把它当作整条 RAG 链路的安全边界。

**追问点答案**  
- 为什么只靠查询过滤不够？  
因为缓存、日志和索引层都可能泄露租户边界。  
- 什么时候需要更强的物理隔离？  
高合规、高敏感行业或大客户专属场景。  
- 审计为什么重要？  
因为要能证明没有越权访问。

**原理补充**  
多租户隔离的核心，是确保任何路径下都不会跨租户召回或暴露不该看的知识。

**架构扩展**  
成熟平台通常会把多租户隔离和权限裁剪放在同一层治理，而不是分开做。因为真实风险很多发生在“租户对了，但角色不对”的细粒度边界上。

**工程指标**  
- 跨租户误召回率：看隔离质量。  
- 权限过滤命中准确率：看安全边界。  
- 高敏感租户独立索引比例：看隔离强度。  
- 审计日志完整率：看可证明性。  
- 相关安全告警数：看风险水平。

**证据链**  
- 租户隔离设计  
- 检索过滤日志  
- 缓存键策略  
- 审计记录

**低风险优化路径**  
1. 先把租户标识贯穿全链路  
2. 再补缓存和日志隔离  
3. 最后按敏感等级决定是否做更强隔离

**易错点**  
- 只在检索层加 tenant filter  
- 缓存未按租户分隔  
- 审计缺失

## 12. RAG 如何做权限裁剪？

**核心结论**  
权限裁剪要在检索前和检索后都考虑，目标是让模型只看到用户有权看到的证据，而不是先查出来再临时遮挡。

**标准答案**  
我做 RAG 权限裁剪时，通常优先在召回阶段就把无权限内容挡掉，因为如果先召回再交给模型，即使最后不展示，也存在泄露和污染回答的风险。工程上比较稳的方式，是让文档和 chunk 都带权限元数据，查询时结合用户身份、角色、租户和资源边界做过滤，再在生成前做一次结果级复核。成熟团队会把权限裁剪和引用审计打通，这样高风险场景下能证明模型看到的内容是合规的。

**追问点答案**  
- 为什么不能先让模型看到再隐藏？  
因为模型可能已经利用了不该看的内容。  
- 为什么还要二次复核？  
因为检索层和生成层都可能有边界风险。  
- 权限裁剪最难的是什么？  
细粒度权限和动态权限变化。

**原理补充**  
权限裁剪的本质，是把“谁能看到什么”变成检索和生成链路的硬约束。

**架构扩展**  
成熟企业 RAG 系统往往会优先做文档级权限裁剪，再逐步演进到 chunk 级和段落级裁剪，因为越细粒度收益越大，但治理成本也越高。

**工程指标**  
- 越权召回率：看风险核心指标。  
- 权限过滤正确率：看实现质量。  
- 权限变更生效时长：看动态性。  
- 高风险答案审计通过率：看闭环能力。  
- 误拒召回率：看可用性代价。

**证据链**  
- 权限模型  
- 检索过滤日志  
- 审计回放结果  
- 变更记录

**低风险优化路径**  
1. 先做文档级过滤  
2. 再补生成前复核  
3. 最后按风险决定是否升级到更细粒度

**易错点**  
- 先召回后遮挡  
- 权限元数据不完整  
- 权限变更不同步

## 13. RAG 的缓存层如何设计？

**核心结论**  
RAG 缓存不只是缓存最终答案，还应按查询改写结果、检索结果和生成结果分层设计，兼顾命中率、时效性和权限安全。

**标准答案**  
我设计 RAG 缓存时，通常会分三层考虑：查询标准化后的结果缓存、检索候选缓存和最终回答缓存。因为不同层的收益和风险不一样，最终答案缓存命中收益最大，但最容易受到知识更新和权限变化影响；检索结果缓存更稳，但不能完全省掉生成成本。工程上比较稳的做法，是按业务时效性设置 TTL，并让缓存键包含租户、权限和知识版本。成熟系统不会为了省 token 把缓存做成安全隐患或一致性黑洞。

**追问点答案**  
- 为什么缓存键要带知识版本？  
因为知识更新后旧答案可能失效。  
- 最终答案缓存为什么风险更高？  
因为它最容易跨权限、跨时效复用。  
- 检索缓存有什么好处？  
能降低重复检索开销，同时比最终答案更稳。

**原理补充**  
RAG 缓存的本质，是在时效性、权限边界和成本收益之间做分层治理。

**架构扩展**  
成熟团队通常会从检索缓存开始，再谨慎引入最终答案缓存，尤其在多租户和高时效场景里，不会一上来就缓存大段生成结果。

**工程指标**  
- 分层缓存命中率：看优化收益。  
- 因缓存过期问题导致的错误率：看一致性风险。  
- 跨权限缓存误用次数：看安全风险。  
- Token 节省比例：看成本收益。  
- 缓存失效率：看时效管理质量。

**证据链**  
- 缓存命中日志  
- 版本与权限键设计  
- 错误案例  
- 成本对比数据

**低风险优化路径**  
1. 先做检索层缓存  
2. 再按低风险场景引入答案缓存  
3. 最后完善版本和权限失效机制

**易错点**  
- 直接缓存最终答案而不考虑权限  
- 缓存键不含知识版本  
- 不分层一刀切缓存

## 14. 召回率和准确率冲突如何权衡？

**核心结论**  
召回率和准确率冲突时，通常先保证“不漏关键证据”，再用重排和生成约束控制噪音，因为漏掉关键资料比多带几条噪音往往更危险。

**标准答案**  
我在召回率和准确率之间取舍时，通常会先看业务容错。像合同问答、合规检索、故障排查这类场景，漏掉关键证据通常比多带几条无关内容更危险，所以会优先保证召回，再通过重排、引用筛选和回答约束提升最终精度。之所以不能一味追求 Top1 很准，是因为很多复杂问题本身就需要多个证据。工程上比较稳的做法，是把召回层和精排层职责分开，分别优化。成熟团队会按问题类型设策略，而不是全场景统一取舍。

**追问点答案**  
- 为什么高风险场景更怕漏召回？  
因为缺关键证据会直接导致错误结论。  
- 准确率低会有什么问题？  
噪音多会拉高生成阶段出错概率。  
- 可以按问题类型动态调吗？  
可以，而且通常更合理。

**原理补充**  
这类权衡本质上是在“信息覆盖度”和“候选纯度”之间做平衡。

**架构扩展**  
成熟 RAG 系统经常会对“事实核查型问题”和“导航类问题”使用不同召回策略，因为两类问题对漏和噪音的容忍完全不同。

**工程指标**  
- TopK 召回率：看覆盖度。  
- 最终引用正确率：看精度。  
- 噪音 chunk 比例：看纯度。  
- 任务成功率：看端到端收益。  
- 高风险场景误答率：看业务代价。

**证据链**  
- 召回/精排对比实验  
- 问题类型分层数据  
- 引用错误案例  
- 用户反馈

**低风险优化路径**  
1. 先按场景定义漏召回代价  
2. 再分层优化召回和精排  
3. 最后按问题类型做动态策略

**易错点**  
- 只盯单一指标  
- 不区分业务场景  
- 召回和重排职责混乱

## 15. 无答案场景如何优雅降级？

**核心结论**  
无答案场景的关键，不是硬答，而是能识别证据不足、明确表达边界，并给出下一步可执行建议。

**标准答案**  
我处理无答案场景时，通常会先定义什么叫“证据不足”，比如召回分数太低、引用不足、多个来源冲突或权限不完整。因为用户体验最差的并不是“我不知道”，而是模型很自信地给错答案。工程上比较稳的做法，是在低置信度时触发拒答、引导补充问题、推荐人工入口或返回相关文档列表。成熟团队会把无答案设计成产品能力，而不是把它当模型失败。

**追问点答案**  
- 为什么拒答也是能力？  
因为高风险场景下，拒答比错答更安全。  
- 低置信度怎么判断？  
结合召回分数、引用数量和规则阈值。  
- 如何避免用户觉得系统“太笨”？  
拒答时给明确原因和下一步建议。

**原理补充**  
优雅降级的本质，是把“不知道”变成受控的产品行为，而不是系统缺陷。

**架构扩展**  
成熟系统通常会给无答案场景留产品出口，比如转人工、展示相关文档、让用户缩小问题范围，这比单纯返回“未找到”更可用。

**工程指标**  
- 低置信度拒答准确率：看安全性。  
- 错答率：看降级效果。  
- 无答案后用户继续完成率：看体验。  
- 人工转接率：看兜底路径使用情况。  
- 用户二次追问率：看表达质量。

**证据链**  
- 拒答规则  
- 低置信度日志  
- 用户后续行为数据  
- 错答样本

**低风险优化路径**  
1. 先定义低置信度规则  
2. 再设计拒答和兜底出口  
3. 最后持续校准误拒和错答

**易错点**  
- 没证据也硬答  
- 只返回“查不到”没有引导  
- 不监控拒答质量

## 16. RAG 评测集如何构建？

**核心结论**  
RAG 评测集要覆盖真实问题、真实文档、真实答案和真实引用，既测“答得对不对”，也测“找得准不准”。

**标准答案**  
我构建 RAG 评测集时，通常会先从真实用户问题、历史工单、FAQ、搜索日志里抽样，而不是只靠人工想题。因为人工编的问题往往太干净，不能代表生产环境。工程上比较稳的做法，是给每个问题标注标准答案、关键证据片段、可接受答案范围和高风险点，再按问题类型分层，比如事实问答、流程问答、多跳问答、无答案场景。成熟团队会把评测集当长期资产持续扩充，尤其会重视失败样本回流。

**追问点答案**  
- 为什么不能只用人工想题？  
因为和真实用户问题分布差异很大。  
- 评测集为什么要带引用标注？  
因为 RAG 不只测生成，还要测检索。  
- 失败样本为什么最有价值？  
因为它最能暴露系统短板。

**原理补充**  
评测集的本质，是把生产问题抽象成可重复验证的标准样本。

**架构扩展**  
成熟团队通常会把评测集分成开发集、回归集和上线门禁集，不同集合承担不同治理目的，避免所有评测都混在一起。

**工程指标**  
- 评测集覆盖率：看代表性。  
- 失败样本占比：看难度质量。  
- 问题类型分布：看是否贴近生产。  
- 引用标注完整率：看检索可测性。  
- 评测集更新频率：看治理活性。

**证据链**  
- 问题样本来源  
- 标注规则  
- 类型分布统计  
- 失败样本库

**低风险优化路径**  
1. 先从真实日志抽样  
2. 再给高价值场景做精标  
3. 最后把失败样本持续回流

**易错点**  
- 评测题脱离真实场景  
- 只标最终答案不标证据  
- 评测集建完不更新

## 17. 幻觉率如何在 RAG 中降低？

**核心结论**  
RAG 降低幻觉的关键，不是“有检索就一定对”，而是让检索、引用、Prompt 约束和拒答机制一起工作。

**标准答案**  
我降低 RAG 幻觉时，通常会从四层入手：先提升召回质量，确保关键证据真的被找到；再控制上下文噪音，避免模型被无关内容带偏；然后要求回答尽量基于引用生成；最后在证据不足时触发拒答或降级。因为很多 RAG 系统的问题不是完全没检索，而是检索了但证据不准、噪音太多、回答不受约束。成熟团队会把“是否基于证据回答”作为回归重点，而不是只看语言是否流畅。

**追问点答案**  
- 为什么 RAG 仍然会幻觉？  
因为检索质量、上下文噪音和生成策略都可能出问题。  
- 拒答为什么重要？  
因为没有充分证据时，强答最危险。  
- 只加引用就够了吗？  
不够，引用本身也可能错。

**原理补充**  
RAG 降幻觉的本质，是把回答限制在外部证据边界内，并在边界不足时停止扩展。

**架构扩展**  
成熟产品通常会在高风险场景中增加“引用必须命中关键片段”“多证据一致才可回答”等规则，而不是只依赖模型自己克制。

**工程指标**  
- 无依据回答率：看核心风险。  
- 错误引用率：看证据可信度。  
- 高风险场景拒答率：看保护能力。  
- 噪音 chunk 比例：看上下文质量。  
- 用户纠错率：看真实效果。

**证据链**  
- 幻觉样本库  
- 引用与答案对照  
- 召回日志  
- 用户反馈

**低风险优化路径**  
1. 先提升检索和引用质量  
2. 再加回答约束和拒答规则  
3. 最后持续回流幻觉样本

**易错点**  
- 以为有 RAG 就天然低幻觉  
- 不检查引用质量  
- 没有拒答机制

## 18. 高并发下检索链路如何扩展？

**核心结论**  
高并发下检索扩展要同时考虑索引查询、过滤、缓存和下游模型调用协同，不是单纯把向量库实例数堆上去。

**标准答案**  
我做高并发检索扩展时，通常先拆链路瓶颈：是查询本身慢、过滤慢、重排慢，还是模型调用把整体拖住。因为很多时候检索层和生成层是连在一起的，只优化一边收益有限。工程上比较稳的做法，是先做查询标准化和缓存，再按查询热点、租户分片、索引副本和异步重排去扩展，同时给高峰期准备限流和降级策略。成熟团队会特别看热点查询和热点知识库，因为并发问题往往集中在少量高频模式上。

**追问点答案**  
- 为什么缓存是第一优先级？  
因为高频重复查询往往占很大比例。  
- 索引副本和分片分别解决什么？  
副本偏读扩展，分片偏容量和并行。  
- 为什么还要配降级？  
因为高峰时全量链路不一定都扛得住。

**原理补充**  
扩展的本质，是把热点流量、索引压力和下游生成成本拆开治理。

**架构扩展**  
成熟系统通常不会让每个请求都走最重链路，而是会用缓存、轻量召回、按需重排和模型分层共同吸收高峰。

**工程指标**  
- 检索 TP99：看核心性能。  
- 缓存命中率：看扩展收益。  
- 热点查询占比：看热点集中度。  
- 降级触发率：看高峰保护。  
- 端到端成功率：看整体效果。

**证据链**  
- 查询分布统计  
- 热点看板  
- 检索与重排监控  
- 限流降级日志

**低风险优化路径**  
1. 先定位热点和缓存收益  
2. 再做副本、分片和链路分层  
3. 最后补高峰降级和配额

**易错点**  
- 一味扩向量库实例  
- 不识别热点模式  
- 没有高峰期降级策略

## 19. 成本可控的 RAG 方案怎么做？

**核心结论**  
成本可控的 RAG 方案要从检索、重排、生成、缓存和知识更新全链路优化，而不是只盯模型 token 单价。

**标准答案**  
我做成本可控 RAG 时，通常会先把成本拆开：embedding 重建成本、检索查询成本、重排成本、生成 token 成本和缓存节省收益。因为很多团队以为 RAG 一定省钱，实际上如果切块太细、重排太重、上下文太长，成本照样会很高。工程上比较稳的做法，是优先控制上下文长度、缓存高频问题、只在高价值场景触发重排，并按业务价值分层模型。成熟团队会把每一步成本和质量收益一起看，避免为了省一点 token 反而把效果做垮。

**追问点答案**  
- 为什么上下文长度是大头？  
因为它直接决定生成阶段 token 消耗。  
- 成本优化为什么不能只降模型档位？  
因为检索和重排也可能是重要成本项。  
- 什么优化最常见？  
缓存、上下文压缩、按需重排、任务分层。

**原理补充**  
成本治理的核心，是让每一步开销都和业务价值对齐。

**架构扩展**  
成熟 RAG 平台通常会给不同业务线不同默认策略，比如高价值问答允许重链路，低价值场景默认轻链路，以此控制整体成本曲线。

**工程指标**  
- 单请求总成本：看整体经济性。  
- 上下文平均 token 长度：看主要消耗。  
- 重排触发率：看重链路使用情况。  
- 缓存节省比例：看优化收益。  
- 成本与任务成功率比值：看投入产出。

**证据链**  
- 成本拆解报表  
- 上下文长度分布  
- 缓存命中日志  
- 业务价值分层策略

**低风险优化路径**  
1. 先做成本拆解  
2. 再优化上下文和缓存  
3. 最后按业务分层重链路使用范围

**易错点**  
- 只盯模型单价  
- 不看上下文长度  
- 为省成本破坏高价值场景效果

## 20. 如何做 RAG 线上 A/B 实验？

**核心结论**  
RAG 线上 A/B 实验要比较的不只是点击或满意度，还要同时看引用质量、任务成功率、延迟、成本和错误边界。

**标准答案**  
我做 RAG A/B 时，通常会先明确实验目标到底是想优化召回、降低幻觉、减少成本，还是提高用户完成率。因为如果目标不清，后面很容易出现不同指标互相打架。工程上比较稳的做法，是把实验流量切到可比较人群，记录检索链路版本、Prompt 版本和知识版本，同时看用户行为指标和系统质量指标。成熟团队不会只看“满意度略升”，而会继续看错误率、拒答率、延迟和成本是不是一起健康。

**追问点答案**  
- 为什么 RAG 实验不能只看点击率？  
因为点击不一定代表答案真的更对。  
- 知识版本为什么要记录？  
因为不同时点知识变化会干扰实验结果。  
- 系统指标为什么也要纳入 A/B？  
因为效果提升如果代价过高，未必值得上线。

**原理补充**  
RAG 实验的本质，是在真实流量下验证“某条检索-生成链路”是否真的更优。

**架构扩展**  
成熟团队做 RAG A/B 时，通常会先从低风险流量或内部流量开始，等质量和成本都稳定后再逐步放大，避免把线上实验变成生产事故。

**工程指标**  
- 任务完成率：看业务效果。  
- 引用正确率：看事实质量。  
- 幻觉率：看风险变化。  
- TTFT/TP99：看体验。  
- 单请求成本：看经济性。

**证据链**  
- 实验设计文档  
- 版本记录  
- 行为和质量指标对比  
- 样本回放结果

**低风险优化路径**  
1. 先定义单一清晰实验目标  
2. 再同时采集用户指标和系统指标  
3. 最后在小流量验证后逐步放量

**易错点**  
- 目标不清就做实验  
- 不记录知识和检索版本  
- 只看用户行为不看质量和成本
