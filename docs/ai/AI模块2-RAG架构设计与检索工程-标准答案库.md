# AI模块2 - RAG架构设计与检索工程 - 标准答案库

## 1. 什么时候必须做 RAG？

**核心结论**  
当业务需要基于私有知识、最新资料、可审计引用或高事实准确率回答问题时，通常就必须做 RAG，而不是只靠基础模型裸答。

**标准答案**  
我判断一个场景是否必须做 RAG 时，通常先看回答是不是依赖企业私有数据、频繁更新的知识、带权限边界的文档，或者必须给出引用依据。如果这些要求成立，只靠基础模型通常很难稳定满足，因为模型参数里的知识不一定最新，也无法天然满足企业内数据和审计要求。工程上比较稳的做法，是让模型只负责理解和生成，把“提供事实依据”交给检索层完成。成熟架构师谈 RAG，不会把它说成“让模型更聪明”，而会说成“让模型在证据边界内回答问题”。

**高分回答**
我不会把“知识问答”直接等同于 RAG，而会先判断事实来自哪里、变化多快、是否需要权限控制和证据审计。稳定且公开的常识可以直接使用模型；结构化、实时且要求确定性的余额、库存、订单状态，更适合调用 API 或查询数据库；微调主要改变模型的行为和表达习惯，不适合承担频繁变化的事实；只有当答案依赖非结构化私有资料、知识持续更新，且必须引用来源或按用户权限裁剪时，RAG 才通常是主方案。

上线时我会明确一条证据契约：先在授权范围内检索，再让模型基于证据回答；证据不足就澄清或拒答，答案携带文档、版本和位置引用。评测也分层进行：用 Recall@K 看相关证据是否被找回，用证据充分率和引用准确率看上下文是否可用，用 groundedness、无依据回答率和任务成功率看最终答案，同时单测越权检索与知识更新时延。这样才能区分“没找到”“找到了但没用好”和“生成阶段编造”三类问题。

我会先选择一个高价值、边界清晰的知识域，用真实问题和反例建立黄金集，再逐步补混合检索、重排、权限和监控。若资料本身过期、互相冲突或缺少负责人，先治理知识源；否则 RAG 只是把低质量资料更快地送给模型。

**追问点答案**  
- 为什么有最新知识就更适合 RAG？  
因为基础模型参数更新无法跟上业务知识变化。  
- RAG 能完全消除幻觉吗？  
不能，但能显著降低无依据输出。  
- 什么场景不一定需要 RAG？  
纯创意生成或完全不依赖外部知识的任务；结构化实时事实通常优先走 API 或数据库查询，小规模稳定资料也可以先评估长上下文方案。

**原理补充**  
RAG 的本质，是把知识访问从模型内部参数转移到外部可更新、可治理的检索系统。

**架构扩展**  
真实业务里，客服、内部知识库、合同问答、代码库问答、研究助手这类场景，RAG 往往不是锦上添花，而是能否上线的基础条件。

**工程指标**  
- 引用覆盖率：看证据链完整性。  
- 无依据回答率：看幻觉控制效果。  
- 知识更新时间到生效时长：看新鲜度。  
- 权限命中准确率：看安全边界。  
- 任务成功率：看业务效果。

**证据链**  
- 评测集  
- 引用日志  
- 文档更新记录  
- 用户反馈

**低风险优化路径**  
1. 先锁定必须依赖私有知识的场景  
2. 再做最小 RAG 链路试点  
3. 最后补引用、权限和评测

**易错点**  
- 需要私有知识却仍然裸用模型  
- 把 RAG 当“搜索 + 拼接”不做治理  
- 不定义哪些场景必须带引用

## 2. 文档切块策略怎么设计？

**核心结论**  
切块策略要服务检索目标，核心是让每块既保留足够语义完整性，又不过度冗长，能被高效召回和准确引用。

**标准答案**  
我设计切块策略时，通常不会先固定字符数，而是先看文档类型和问答粒度。比如技术文档、合同、FAQ、代码说明，它们天然结构就不一样，切块策略也不该一样。工程上比较稳的做法，是优先按自然边界切，比如标题、段落、列表或章节，再在长度超标时做二次切分，并保留适度 overlap。这样做的原因，是如果切得太碎，语义会丢；切得太大，召回噪音和成本都会上升。成熟团队通常会把切块和评测一起调，而不是把它当固定参数。

**高分回答**
切块不是预处理参数，而是检索数据建模。我会先从用户问题的粒度反推可独立作答的最小语义单元：FAQ 保留问题与答案，合同按条款和子条款，技术文档保留标题层级、列表和表格表头，代码按类、函数或符号边界，并补充必要的签名与依赖上下文。超长单元才二次切分，跨边界信息则用父子块检索或窗口扩展补齐，而不是盲目增大 overlap。

每个 chunk 都要带稳定 ID、来源、版本、标题路径、时间、租户和权限元数据。稳定 ID 使增量更新、删除、引用回溯和故障审计成为可能；父子关系则允许“小块负责精准召回、大块负责完整作答”。Overlap 会增加重复召回、索引体积和上下文 token，只有边界遗漏数据证明有收益时才增加。

我会按问题类型构建评测切片，对多组切块策略比较 Recall@K、边界遗漏率、引用精度、重复上下文率、平均输入 token 和最终回答正确率。若召回命中了相邻块却缺关键条件，应调整边界或父子扩展；若总召回同一段重复内容，则先去重和降低重叠。最终选择应来自真实查询上的端到端结果，而不是固定的字符数经验。

**追问点答案**  
- 为什么不能只按固定字数切？  
因为语义边界可能被硬切断。  
- overlap 为什么重要？  
它能缓解跨边界信息丢失，但会制造重复召回并增加索引和上下文成本，应以边界遗漏评测决定，而不是默认越大越好。
- 代码和合同文档切块能一样吗？  
通常不能，结构和问答粒度差异很大。

**原理补充**  
切块的本质，是在“语义完整”和“检索精度”之间找最合适的平衡。

**架构扩展**  
成熟 RAG 系统往往会按文档类型维护不同切块器，比如 FAQ 按问答对切，合同按条款切，技术文档按标题层级切，这样召回和引用质量通常明显更好。

**工程指标**  
- 召回命中率：看切块是否适配检索。  
- 引用精确率：看语义边界是否合理。  
- 平均 chunk 长度：看成本和噪音。  
- 用户追问率：看一次召回信息是否够用。  
- Chunk 冗余度：看索引效率。

**证据链**  
- 切块样本  
- 检索命中案例  
- 引用偏差样本  
- 评测结果

**低风险优化路径**  
1. 先按文档类型设计自然边界切块  
2. 再调长度和 overlap  
3. 最后用评测集迭代优化

**易错点**  
- 全文档统一一个切块参数  
- 只看长度不看语义边界  
- 不做切块效果评测

## 3. 向量库选型看哪些指标？

**核心结论**  
向量库选型要同时看检索效果、写入更新能力、过滤能力、延迟、成本和运维复杂度，而不是只看 ANN 指标。

**标准答案**  
我选向量库时，通常先看业务对召回质量、并发规模、更新频率和过滤条件的要求。因为有的场景是高并发在线检索，有的则更偏离线知识库更新，瓶颈完全不同。工程上比较稳的做法，是重点比较查询延迟、过滤能力、多租户支持、增量更新代价、索引重建成本和运维方式，而不是只看理论向量搜索性能。成熟团队会把向量库当成 RAG 数据层的一部分来评估，明确它到底要解决的是“更快”“更准”还是“更好管”。

**高分回答**
我会先用需求约束选型，而不是从产品榜单开始：当前与三年后的向量规模、维度、查询 QPS 和 P99 目标，写入与删除生效时限，metadata 过滤选择性，多租户隔离方式，以及是否需要关键词与向量混合检索。数据量不大、现有搜索系统已满足过滤和延迟时，不一定需要额外引入专用向量库。

PoC 必须使用自己的 embedding、文档、过滤条件和查询分布，画出不同索引参数下的召回—延迟—内存曲线。尤其要验证过滤发生在候选生成之前还是之后：低选择性权限过滤若只做后过滤，可能拿不到足够结果；更严重的是实现错误会造成越权。除了平均延迟，还要压测热点租户、批量更新、节点故障和索引构建期间的尾延迟。

生产维度还包括分片与副本、备份恢复、扩缩容、删除语义、索引重建、模型版本迁移、监控和单位查询成本。我会要求供应商或自建方案完成恢复演练，并验证旧索引可回滚。最后以 Recall@K、过滤正确率、更新可见时延、P99、恢复时间和总拥有成本做决策，避免选到“单机 benchmark 很快、生产治理很弱”的组件。

**追问点答案**  
- 为什么过滤能力很重要？  
因为权限、租户和文档类型筛选是业务刚需；还要验证预过滤与后过滤语义，防止结果不足或越权泄漏。
- 增量更新为什么要单独评估？  
因为知识库不是只建一次，后续持续更新是常态。  
- 自建和托管怎么选？  
看团队运维能力和可控性需求。

**原理补充**  
向量库不是孤立组件，它的价值取决于是否贴合检索链路和知识治理要求。

**架构扩展**  
成熟团队在做向量库选型时，往往会同时测混合检索和 metadata filter，因为生产里真正痛的常常不是 ANN 本身，而是业务过滤和更新治理。

**工程指标**  
- 查询 TP99：看在线体验。  
- 召回率：看检索效果。  
- 增量写入延迟：看知识更新能力。  
- 过滤正确率：看业务可用性。  
- 运维成本：看长期可持续性。

**证据链**  
- 压测结果  
- 业务过滤需求清单  
- 更新频率统计  
- 成本评估

**低风险优化路径**  
1. 先明确检索和更新需求  
2. 再做小规模压测和效果对比  
3. 最后结合团队运维能力定方案

**易错点**  
- 只看搜索性能榜单  
- 忽略过滤和更新能力  
- 不评估实际运维成本

## 4. embedding 模型如何选择？

**核心结论**  
Embedding 模型选择要看语种、领域语义、向量质量、成本和延迟，核心是它是否能把“你的文档和你的查询”映射到有效相似空间。

**标准答案**  
我选 embedding 模型时，通常会先看知识库的语料类型和查询方式，是通用文本、代码、法律文档、产品知识，还是多语言混合内容。因为 embedding 的好坏，最终决定的是检索是否能把真正相关的 chunk 拉出来。工程上比较稳的做法，是直接用自己的文档和真实问题做召回评测，而不是只看公开 benchmark。成熟团队会把 embedding 模型的质量、延迟和单次构建成本一起评估，因为知识库规模一大，向量重建代价就很明显。

**高分回答**
Embedding 选型的核心不是公开榜单分数，而是能否在本业务的“查询—相关文档”关系上稳定区分正例和难负例。我会从搜索日志、专家标注和失败案例构建评测集，覆盖中文与英文、缩写与全称、产品型号、领域术语、长短查询、代码或表格等切片，并加入词面相似但语义错误的 hard negative，避免模型只靠关键词取巧。

离线比较 Recall@K、MRR 或 nDCG，并按场景切片看最差表现；端到端还要比较重排后的证据充分率和回答正确率。工程上同时测向量维度与存储量、索引构建吞吐、查询 P99、外部调用合规性和全量重建成本。文本清洗、查询前缀、截断与归一化必须遵循模型约定，切块策略也应一起复测，否则可能把预处理问题误判成模型问题。

Embedding 版本要进入 chunk 元数据，两个不兼容的向量空间不能混在同一索引直接比较。升级时我会构建新索引，做离线回放和线上影子流量，再灰度切换并保留旧索引回滚；只有当持续存在明确的领域语义缺口且通用模型无法通过数据与检索策略解决时，才考虑领域微调。

**追问点答案**  
- 为什么 embedding 选型不能只看公开分数？  
因为你的领域语料和 benchmark 语料可能差很多。  
- 多语言场景为什么更要谨慎？  
因为不同语种 embedding 质量差异可能很大。  
- embedding 模型更新为什么有成本？  
因为新旧模型通常不共享可比较的向量空间，需要重算向量、重建索引、双版本验证并预留回滚能力。

**原理补充**  
Embedding 模型的本质，是决定“语义相似”在向量空间里如何被表示。

**架构扩展**  
成熟团队通常会保留一套 embedding 评测集和重建流程，避免后续换模型时完全失控。因为一旦知识库规模起来，embedding 升级本身就是工程项目。

**工程指标**  
- TopK 召回率：看语义表示质量。  
- 多语种召回差异：看语言适配性。  
- 向量构建成本：看重建代价。  
- 查询时延：看线上可用性。  
- 文档重建周期：看运维压力。

**证据链**  
- 召回评测集  
- 多语种样本对比  
- 向量重建记录  
- 成本测算

**低风险优化路径**  
1. 先用真实问答样本做 embedding 对比  
2. 再评估重建成本  
3. 最后定版本策略和升级路径

**易错点**  
- embedding 选型脱离真实文档  
- 忽略重建成本  
- 多语言场景不单独评测

## 5. 混合检索（BM25+向量）何时必要？

::: tip L2 深度训练
用召回、排序、延迟和成本共同决定混合检索与重排：[RAG 质量闭环](/deep-dives/ai-architecture/01-rag-quality-loop)
:::

**核心结论**  
当查询同时依赖关键词精确匹配和语义相似理解，尤其涉及术语、编号、缩写、实体名时，混合检索通常比纯向量更稳。

**标准答案**  
我判断要不要做混合检索时，通常先看业务问题里是否经常出现精确关键词，比如产品型号、报错码、条款编号、表名字段名这类内容。如果是，这类查询仅靠向量相似可能会漏掉最关键的命中；但如果只用关键词，又会丢掉语义改写和自然表达的能力。工程上比较稳的做法，是让 BM25 承担关键词召回，让向量承担语义召回，再通过融合或重排选出最终结果。成熟团队不会机械地说“向量比关键词高级”，而是会根据查询分布决定是否混合。

**高分回答**
当查询既包含必须精确命中的标识符，又存在同义改写和自然语言描述时，我会考虑混合检索。例如报错码、合同条款号、API 名和产品型号依赖 BM25 的词面匹配，而“登录后一直回到原页面”与“认证重定向循环”需要向量语义召回。若查询主要是稳定的精确检索或单一语义检索，则先保留简单基线，避免无收益地增加链路复杂度。

实现上让 BM25 与向量检索在同一权限和时间过滤条件下并行召回，再使用 RRF 等只依赖名次的方式融合，因为两路原始分数通常不能直接相加；对融合后的有限候选去重并按需重排。必须保护编号、大小写敏感符号和领域实体，查询清洗若把这些信息抹掉，混合检索也无法补救。两路过滤语义不一致还可能造成结果缺失甚至权限风险。

评测时我会分别统计编号/实体型、自然语言改写型和混合型查询的 Recall@K、nDCG、最终引用准确率、P99 与单请求成本，并和纯 BM25、纯向量两个基线比较。如果总体提升只来自少数查询，可用查询分类按需触发；如果大部分企业查询都兼具两类特征，再将混合检索设为默认。这样能说清楚收益来自哪里，也能控制重排和双路召回的代价。

**追问点答案**  
- 什么场景最适合混合检索？  
实体名、代码、编号、专业术语密集的场景。  
- 混合检索一定比纯向量好吗？  
不一定；必须按查询类型与纯 BM25、纯向量基线比较质量、延迟和成本，必要时只对特定查询触发。
- 多一步融合会不会更慢？  
会增加一些开销，但很多场景收益明显。

**原理补充**  
混合检索的核心，是把“词面相关性”和“语义相关性”两种信号结合起来。

**架构扩展**  
在企业知识库、代码问答、工单系统和法规检索里，混合检索往往是事实上的默认方案，因为这些场景既有术语精确性，又有自然语言表达多样性。

**工程指标**  
- TopK 召回率提升幅度：看混合收益。  
- 术语类问题命中率：看关键词价值。  
- 最终重排前噪音率：看召回质量。  
- 查询延迟增量：看成本。  
- 用户追问率：看答案相关性。

**证据链**  
- 按问题类型拆分的评测结果  
- 关键词遗漏案例  
- 语义召回失败案例  
- 延迟对比数据

**低风险优化路径**  
1. 先识别术语/编号型问题占比  
2. 再对这类查询试点混合检索  
3. 最后视收益决定是否全面推广

**易错点**  
- 默认纯向量解决一切  
- 引入混合检索后不评估延迟和收益  
- 不按查询类型分层

## 6. 重排模型怎么接入主链路？

**核心结论**  
重排模型适合在初步召回后缩小候选集，用更细粒度语义判断提升最终相关性，但必须控制好延迟和调用成本。

**标准答案**  
我接入重排模型时，通常不会让它直接面对全量知识库，而是先由向量或混合检索召回一批候选，再用重排模型对前几十条结果重新排序。这样做的原因，是重排模型的精度通常更高，但成本和时延也更高，不适合直接做第一层检索。工程上比较稳的做法，是把重排放在高价值链路或高模糊度问题上，并监控它对召回质量和 TP99 的影响。成熟团队会把重排当成“精排层”，而不是默认每个请求都必须经过的固定步骤。

**高分回答**
我会把召回和重排定义成两个目标不同的阶段：第一阶段用 BM25、向量或混合检索扩大候选，优先保证相关证据不漏；第二阶段让 cross-encoder 或其他精排模型同时读取查询与候选文本，提升前几名的相关性。候选必须先完成租户、权限、有效期过滤，重排只能调整授权结果的顺序，不能成为安全过滤器。

候选数量不是越多越好。我会用离线数据画出候选数从小到大时 Recall@K、nDCG、重排 P99 和成本曲线，再确定召回窗口与最终送入模型的数量。对于精确编号查询、首名置信度很高或低价值请求，可以绕过重排；对于语义歧义大、候选相似或高风险问题才触发。重排超时或服务不可用时，应降级到融合排序结果，并在答案置信度和监控中标记降级状态。

效果评估不能只看“排序分更高”，而要比较无重排基线下的 MRR/nDCG、关键证据进入 TopN 的比例、最终引用正确率和答案成功率，同时观察 P95/P99、超时率与单请求成本。若重排提升离线排序但不改善最终答案，应继续排查切块、上下文组织或生成约束，而不是继续扩大候选集。

**追问点答案**  
- 为什么重排不直接全库使用？  
因为成本和延迟通常扛不住。  
- 什么请求最适合加重排？  
候选较多、语义区分难、答案要求高准确的请求。  
- 重排也可以按需触发吗？  
可以，可依据查询类型、首名置信度、候选分差和业务风险触发，并为超时准备回退到原始融合排序的降级路径。

**原理补充**  
重排的本质，是在较小候选集上做更贵但更准确的相关性判断。

**架构扩展**  
成熟 RAG 系统常见做法是“两段式”检索：第一段追求不漏，第二段追求更准。重排是否常驻主链路，取决于收益能否覆盖它的时延和成本。

**工程指标**  
- 最终命中率提升：看效果。  
- TP99 增量：看性能代价。  
- 重排触发率：看使用范围。  
- 重排后引用正确率：看业务价值。  
- 单请求成本增加：看经济性。

**证据链**  
- 无重排/有重排对比实验  
- 候选集质量数据  
- 延迟曲线  
- 成本分析

**低风险优化路径**  
1. 先在高价值场景试点重排  
2. 再优化候选数量和触发条件  
3. 最后决定是否扩展到主链路

**易错点**  
- 让所有请求都强制走重排  
- 不控制候选集规模  
- 只看效果不看 TP99

## 7. 查询改写在 RAG 中有什么价值？

**核心结论**  
查询改写的价值，在于把用户自然表达转成更适合检索系统理解的查询形式，从而提升召回质量和检索稳定性。

**标准答案**  
我用查询改写时，通常会针对用户输入过短、过口语化、上下文依赖强或者存在指代省略的情况。因为用户真实问题往往并不是最适合检索的表达，比如“这个怎么配”“它会报什么错”这种问题，如果不结合上下文补全，检索效果通常很差。工程上比较稳的做法，是把改写限定在提高检索表达质量，而不是让模型自由发挥生成新问题。成熟团队会持续监控改写前后召回质量，避免改写带来语义偏移。

**高分回答**
查询改写主要解决三类检索错位：会话中的指代和省略、用户词汇与知识库术语不一致，以及复杂问题包含多个检索意图。例如“它支持吗”要结合有限会话上下文还原对象；业务别名可以扩展为标准术语；多意图问题则拆成子查询。改写结果只服务检索，原始问题仍应保留给生成阶段，避免模型最终回答了“被改写的问题”而不是用户问题。

我会优先使用确定性规则处理拼写、别名和标识符保护，只在语义补全确有必要时调用模型，并要求输出结构化的独立查询、保留实体和时间条件。系统同时保留原始查询和改写查询进行多路召回，融合后去重；当改写置信度低、引入原问题没有的实体或改变否定、时间、范围条件时，回退到原查询或向用户澄清。任何来自历史对话的内容也要按当前用户权限重新检索，不能把旧答案直接当证据。

评测应包含正常样本和“语义陷阱”，比较改写前后的 Recall@K、关键实体保留率、意图偏移率、最终任务成功率和额外延迟。还要记录原始查询、改写结果、所用上下文与召回差异，失败时才能判断是改写偏了还是知识库确实没有答案。

**追问点答案**  
- 为什么查询改写容易带来风险？  
因为它可能新增实体、改变否定或时间范围，导致检索目标偏离；因此要保留原查询、多路召回并监控关键约束保留率。
- 什么情况下最值得改写？  
问题过短、指代强、上下文依赖高时。  
- 改写一定要用 LLM 吗？  
不一定，规则和模板也能处理一部分。

**原理补充**  
查询改写的核心，是减少用户表达和知识表达之间的语义落差。

**架构扩展**  
成熟 RAG 产品里，查询改写经常和会话记忆、术语扩展、模板化增强一起用，但都会保持“尽量不改变真实意图”这条底线。

**工程指标**  
- 改写后召回提升率：看收益。  
- 改写偏移率：看风险。  
- 短问题命中率：看特定价值。  
- 改写耗时：看性能代价。  
- 用户追问率：看实际效果。

**证据链**  
- 改写前后对比集  
- 偏移样本  
- 会话上下文日志  
- 召回结果对比

**低风险优化路径**  
1. 先只处理最明显的短问题和指代问题  
2. 再评估偏移率  
3. 最后逐步扩到更多场景

**易错点**  
- 改写过度自由发挥  
- 不监控语义偏移  
- 改写链路无回退

## 8. 多跳检索如何落地？

**核心结论**  
多跳检索适合答案需要跨多个文档或多个事实组合推理的场景，关键是把检索拆成可解释步骤，而不是一次性盲搜。

**标准答案**  
我做多跳检索时，通常会先判断问题是不是单次召回就能回答，如果需要先找到 A 再通过 A 找 B，这时才值得做多跳。因为多跳链路一旦上来，复杂度、成本和失败点都会增加。工程上比较稳的方式，是先让模型或规则识别关键中间实体，再分步检索和聚合，同时保留每一步的证据。成熟系统不会把多跳当默认模式，而会针对复杂问题或高价值场景专门触发。

**高分回答**
只有当问题存在依赖关系——后一跳的检索条件必须由前一跳结果产生——我才会启用多跳。例如先从项目文档确定其依赖组件，再查询该组件某版本的漏洞公告。若只是需要多个彼此独立的事实，拆成并行子查询更快也更容易控制；如果单跳已经能找到包含完整答案的文档，多跳只会增加误差和成本。

落地时先生成受约束的检索计划，明确每一跳的子问题、所需实体、停止条件和最大跳数。每一跳输出结构化事实与证据 ID，下一跳只能使用已验证实体，不能把模型猜测当检索条件；最终答案必须展示能覆盖结论的完整证据链。任一关键跳证据不足、出现矛盾或达到预算，就停止并澄清或返回部分结果，而不是继续推演。

我会为每一跳记录查询、候选、选中证据和实体来源，用多跳专用评测集观察每跳 Recall@K、实体提取正确率、路径完成率、证据链完整率、整体任务成功率、P99 和 token 成本。由于成功率近似受各跳成功率连乘影响，应优先减少不必要的跳数，并通过规则、知识图谱或确定性 API 固化可确定的关系。

**追问点答案**  
- 多跳为什么复杂？  
因为后一跳依赖前一跳的实体和证据，单跳误差会沿路径放大；必须设置最大跳数、停止条件和证据不足时的退出策略。
- 什么时候不该上多跳？  
单跳问题占绝大多数且收益不明显时。  
- 为什么要保留每一步证据？  
便于解释、调试和审计。

**原理补充**  
多跳检索的本质，是把复杂问题拆成多个可检索的中间子问题。

**架构扩展**  
在研究助手、代码依赖分析、复杂合规问答里，多跳检索会带来明显收益，但成熟团队通常会严格限制触发条件，避免把所有请求都拖进重链路。

**工程指标**  
- 多跳问题任务成功率：看收益。  
- 每跳命中率：看链路质量。  
- 多跳整体延迟：看性能代价。  
- 误差累积率：看风险。  
- 证据完整率：看可解释性。

**证据链**  
- 多跳评测集  
- 每跳日志  
- 中间实体识别结果  
- 失败样本库

**低风险优化路径**  
1. 先识别真正需要多跳的问题类型  
2. 再用可解释步骤试点  
3. 最后控制触发范围和链路长度

**易错点**  
- 所有复杂问题都盲目上多跳  
- 不记录每一步证据  
- 多跳收益没有单独评估

## 9. 引用溯源如何设计成可审计？

::: tip L2 深度训练
补齐文档版本、关键主张验证、权限和删除传播：[RAG 质量闭环](/deep-dives/ai-architecture/01-rag-quality-loop)
:::

**核心结论**  
引用溯源要做到“答案中的关键结论都能追到具体文档片段、具体版本和具体权限上下文”，这样才真正可审计。

**标准答案**  
我设计引用溯源时，通常会把文档 ID、chunk ID、文档版本、更新时间和检索时间一起记录下来，而不是只在前端简单显示一个来源名称。因为真正出问题时，团队需要知道模型到底基于哪一版内容回答的，用户当时有没有权限看到这段内容。工程上比较稳的做法，是让生成阶段保留答案和引用片段的对应关系，并在日志里记录完整链路，便于回放和追责。成熟团队谈引用，不会只讲“给用户看来源”，还会讲后端审计和版本追踪。

**高分回答**
可审计引用不是在答案末尾附几个链接，而是建立“主张—证据”的可验证映射。我会让生成阶段输出结构化主张及其 evidence_id，再由服务端根据检索结果解析为文档 ID、chunk ID、版本、原文位置和可访问链接；模型不能自行生成来源地址。前端展示便于用户核验的短引用，后端保留完整检索快照、排序分、提示版本、模型版本、请求身份与授权决策。

审计记录必须能回答三个问题：当时模型看到了什么、为什么该用户能看到、该证据是否真的支持对应主张。为避免文档更新后链接内容变化，关键场景应保存不可变版本或内容哈希；删除或保留策略则按合规要求传播到索引、缓存和审计存储。日志本身可能包含敏感原文，应加密、最小化访问并设定保留期，不能为了审计制造新的泄漏面。

质量门禁要逐主张验证引用覆盖率、引用蕴含准确率、版本可回放率和失效链接率。模型引用了真实文档但文档不支持该结论，仍然属于错误引用；多个来源冲突时，应展示冲突和版本，而不是挑一个迎合答案。线上抽样回放与高风险答案强制引用，才能把知识源、检索、重排和生成故障准确归因。

**追问点答案**  
- 为什么来源名称不够？  
因为无法精确定位到片段和版本。  
- 权限上下文为什么也要记录？  
因为可见不等于对所有用户都可见。  
- 引用展示和审计是同一层吗？  
不是；展示服务用户核验，审计还需保存检索快照、内容版本、授权决策和运行配置，并对审计数据本身实施访问控制。

**原理补充**  
可审计引用的核心，是让答案与证据之间存在可回放、可验证的强关联。

**架构扩展**  
高风险场景里，引用溯源是决定系统能否真正上线的重要条件之一。没有它，出现错误时就无法解释是知识库问题、检索问题还是生成问题。

**工程指标**  
- 引用完整率：看证据覆盖度。  
- 引用可回放成功率：看审计质量。  
- 错误引用率：看风险。  
- 版本追溯成功率：看治理成熟度。  
- 高风险答案带引用比例：看保护力度。

**证据链**  
- 检索日志  
- chunk 与答案映射  
- 文档版本记录  
- 审计回放结果

**低风险优化路径**  
1. 先记录文档/片段/版本信息  
2. 再建立答案到引用的映射  
3. 最后做回放和审计工具

**易错点**  
- 只有前端来源展示没有后端审计  
- 不记录版本  
- 引用和答案没有精确映射

## 10. 知识库增量更新如何保证一致性？

**核心结论**  
知识库增量更新的一致性重点，是让文档原文、向量索引、元数据和权限信息保持同一版本语义，避免“文档已经变了，检索还是旧的”。

**标准答案**  
我做知识库增量更新时，通常会把更新链路拆成文档入库、清洗切块、向量生成、索引替换和权限元数据刷新几步，并给每次更新打版本标记。因为 RAG 里最常见的问题之一就是数据已经改了，但检索层和权限层没同步好。工程上比较稳的做法，是采用可追踪的增量任务和幂等更新逻辑，必要时用双索引或版本切换避免在线查询读到半更新状态。成熟团队不会只关心“更新快不快”，还会关注“更新过程是否可审计、可回退”。

**高分回答**
我会把原始文档存储视为事实源，把清洗、切块、embedding 和检索索引视为可重建投影。每次变更携带 document_id、单调版本、内容哈希、权限版本和事件 ID；处理器按事件 ID 幂等，只有原文、chunk、向量和权限元数据全部成功后，才原子发布新版本。查询只读取已发布版本，避免同一文档的新旧块同时出现。

单文档更新可以先写新 chunk，再通过版本指针切换并异步清理旧块；全量重切块或 embedding 升级则构建影子索引，完成数量校验、抽样召回和权限测试后切换别名。删除使用 tombstone 立即阻断查询，再异步清除向量、关键词索引、缓存和派生数据。乱序事件用版本比较拒绝旧写覆盖新写，失败任务进入可观测的重试与死信流程，并支持按版本回放。

我会对账事实源与索引中的文档数、chunk 数、内容哈希和权限版本，监控变更到可检索的 P95/P99 时延、失败积压、孤儿 chunk、重复版本和删除传播时长。发布前做 canary 查询，切换异常就回滚索引别名。这样实现的是可解释的最终一致性与版本级原子可见，而不是声称跨异构系统存在难以兑现的强事务。

**追问点答案**  
- 为什么知识库更新也要版本化？  
因为它让查询只读完整发布版本，并能处理乱序更新、审计历史答案和在索引升级失败时回滚。
- 双索引切换适合什么场景？  
大批量更新或高风险知识库。  
- 权限信息为什么要一起更新？  
因为内容可见性和文档内容一样重要。

**原理补充**  
增量更新一致性的本质，是保证检索视图和知识源在同一语义时间点上对齐。

**架构扩展**  
成熟系统通常会把“知识更新时间”和“知识生效时间”区别开，尤其在法规、价格、产品政策类场景里，这个差异很关键。

**工程指标**  
- 更新成功率：看链路稳定性。  
- 文档到索引生效时长：看时效性。  
- 半更新状态错误数：看一致性风险。  
- 回滚成功率：看治理能力。  
- 权限同步延迟：看安全边界。

**证据链**  
- 更新任务日志  
- 版本记录  
- 索引切换记录  
- 权限同步状态

**低风险优化路径**  
1. 先建立版本化更新链路  
2. 再做幂等与审计  
3. 最后针对大批量更新引入双索引切换

**易错点**  
- 只更新文档不更新索引  
- 没有版本标记  
- 权限元数据滞后

## 11. 多租户知识库怎么隔离？

**核心结论**  
多租户知识库隔离要同时控制数据存储、检索过滤、缓存和审计，不能只靠前端带一个 tenant_id。

**标准答案**  
我做多租户隔离时，通常会先明确隔离级别，是逻辑隔离、索引级隔离还是物理隔离。因为不同业务和合规要求下，代价和风险完全不同。工程上比较稳的做法，是让租户信息从文档入库到检索过滤、缓存键、日志审计一路贯穿，并对高敏感租户做更强隔离。成熟团队不会把多租户隔离看成单点过滤条件，而会把它当作整条 RAG 链路的安全边界。

**高分回答**
我会按数据敏感度、合规要求、租户规模和“吵闹邻居”风险选择隔离级别：共享集群加逻辑过滤成本最低；独立 namespace 或索引便于配额和迁移；独立集群、密钥和网络边界适合高敏客户。隔离级别可以分层，不必让所有租户承担最高成本，但必须定义升级、迁移和删除租户数据的流程。

tenant_id 不能由客户端自由传入，而应由认证上下文在服务端注入，并贯穿文档对象、chunk、关键词与向量索引、任务队列、缓存键、限流配额和审计日志。检索必须使用不可绕过的服务端预过滤；生成前再校验候选租户归属。缓存和批处理尤其容易泄漏：语义相似的查询也不能跨租户复用私有答案，离线 embedding 任务也要防止输出写入错误 namespace。

验收时除正常查询，还要做恶意 tenant_id、缺失过滤条件、缓存命中、重试乱序、备份恢复和运维账号访问测试，并将“跨租户召回”为零设为硬门禁。运行中监控每租户 QPS、P99、容量与成本，防止单个租户拖垮共享资源；审计记录授权主体、租户、索引和证据 ID，才能证明隔离实际生效。

**追问点答案**  
- 为什么只靠查询过滤不够？  
因为身份注入、索引、缓存、异步任务、日志和备份都可能跨边界；过滤还必须由服务端强制，不能信任客户端 tenant_id。
- 什么时候需要更强的物理隔离？  
高合规、高敏感行业或大客户专属场景。  
- 审计为什么重要？  
因为要能证明没有越权访问。

**原理补充**  
多租户隔离的核心，是确保任何路径下都不会跨租户召回或暴露不该看的知识。

**架构扩展**  
成熟平台通常会把多租户隔离和权限裁剪放在同一层治理，而不是分开做。因为真实风险很多发生在“租户对了，但角色不对”的细粒度边界上。

**工程指标**  
- 跨租户误召回率：看隔离质量。  
- 权限过滤命中准确率：看安全边界。  
- 高敏感租户独立索引比例：看隔离强度。  
- 审计日志完整率：看可证明性。  
- 相关安全告警数：看风险水平。

**证据链**  
- 租户隔离设计  
- 检索过滤日志  
- 缓存键策略  
- 审计记录

**低风险优化路径**  
1. 先把租户标识贯穿全链路  
2. 再补缓存和日志隔离  
3. 最后按敏感等级决定是否做更强隔离

**易错点**  
- 只在检索层加 tenant filter  
- 缓存未按租户分隔  
- 审计缺失

## 12. RAG 如何做权限裁剪？

**核心结论**  
权限裁剪要在检索前和检索后都考虑，目标是让模型只看到用户有权看到的证据，而不是先查出来再临时遮挡。

**标准答案**  
我做 RAG 权限裁剪时，通常优先在召回阶段就把无权限内容挡掉，因为如果先召回再交给模型，即使最后不展示，也存在泄露和污染回答的风险。工程上比较稳的方式，是让文档和 chunk 都带权限元数据，查询时结合用户身份、角色、租户和资源边界做过滤，再在生成前做一次结果级复核。成熟团队会把权限裁剪和引用审计打通，这样高风险场景下能证明模型看到的内容是合规的。

**高分回答**
核心原则是“未授权内容不能进入模型上下文”。用户身份先由可信认证系统解析成主体、租户、角色、组和必要的属性，再由策略决策点计算允许访问的资源范围；检索服务把这一范围转成不可绕过的预过滤条件。文档继承、段落例外、时间限制等权限要进入索引元数据，无法安全表达的细粒度策略则宁可回源校验，也不能先宽召回后遮挡。

为防止索引权限滞后，我会让内容版本和 ACL 版本一起发布，权限收紧优先于内容更新：变更事件到达时可先将资源标记不可检索，再重建元数据。生成前对 TopN 再做一次授权校验，引用下载也重新鉴权；查询改写、重排、缓存、会话历史和日志都遵循相同权限边界。模型即便不展示原文，也可能从越权上下文推断秘密，因此输出脱敏不能替代检索前授权。

测试包含允许、拒绝、权限继承、撤权、组变更和缓存复用，并将越权召回率设为零，同时衡量误拒率和权限变更 P99 生效时长。审计保存策略版本、决策结果和证据 ID，但对日志本身最小授权。若过滤后候选不足，系统应返回证据不足，而不是放宽权限补齐结果。

**追问点答案**  
- 为什么不能先让模型看到再隐藏？  
因为模型可能已经利用了不该看的内容。  
- 为什么还要二次复核？  
因为检索层和生成层都可能有边界风险。  
- 权限裁剪最难的是什么？  
细粒度继承和动态撤权最难，需要 ACL 与内容版本协同发布、缓存失效和生成前二次鉴权。

**原理补充**  
权限裁剪的本质，是把“谁能看到什么”变成检索和生成链路的硬约束。

**架构扩展**  
成熟企业 RAG 系统往往会优先做文档级权限裁剪，再逐步演进到 chunk 级和段落级裁剪，因为越细粒度收益越大，但治理成本也越高。

**工程指标**  
- 越权召回率：看风险核心指标。  
- 权限过滤正确率：看实现质量。  
- 权限变更生效时长：看动态性。  
- 高风险答案审计通过率：看闭环能力。  
- 误拒召回率：看可用性代价。

**证据链**  
- 权限模型  
- 检索过滤日志  
- 审计回放结果  
- 变更记录

**低风险优化路径**  
1. 先做文档级过滤  
2. 再补生成前复核  
3. 最后按风险决定是否升级到更细粒度

**易错点**  
- 先召回后遮挡  
- 权限元数据不完整  
- 权限变更不同步

## 13. RAG 的缓存层如何设计？

**核心结论**  
RAG 缓存不只是缓存最终答案，还应按查询改写结果、检索结果和生成结果分层设计，兼顾命中率、时效性和权限安全。

**标准答案**  
我设计 RAG 缓存时，通常会分三层考虑：查询标准化后的结果缓存、检索候选缓存和最终回答缓存。因为不同层的收益和风险不一样，最终答案缓存命中收益最大，但最容易受到知识更新和权限变化影响；检索结果缓存更稳，但不能完全省掉生成成本。工程上比较稳的做法，是按业务时效性设置 TTL，并让缓存键包含租户、权限和知识版本。成熟系统不会为了省 token 把缓存做成安全隐患或一致性黑洞。

**高分回答**
我会先按可复用边界拆缓存：规范化或改写结果可按无敏感信息的查询缓存；检索候选缓存必须绑定租户、授权范围摘要、过滤条件、索引版本和 embedding 版本；重排缓存还要带模型版本；最终答案缓存则额外绑定提示、生成模型、证据版本和输出策略。语义缓存不是只比较向量相似度，还要验证意图、实体、时间范围和权限完全兼容，否则相似问题可能得到错误旧答案。

失效优先采用版本化键和变更事件：文档、ACL 或模型升级后，新请求自然进入新命名空间，旧值按 TTL 回收；撤权与删除等安全事件还要主动驱逐。高时效、高风险或用户个性化答案默认不缓存最终结果，可只缓存授权后的检索候选。为防缓存击穿和雪崩，可加请求合并、随机 TTL、热点预热与容量隔离，但降级时也不能省略权限检查。

我会按层统计命中率、节省的检索/重排/token 成本、额外延迟、陈旧命中率和跨权限误用次数，并设置后两者为质量门禁。缓存值保留来源版本和生成时间，命中后仍检查授权与有效期。若失效治理成本高于节省收益，就删除该层缓存，而不是让复杂度持续累积。

**追问点答案**  
- 为什么缓存键要带知识版本？  
因为知识更新后旧答案可能失效。  
- 最终答案缓存为什么风险更高？  
因为它固化了某一时刻的证据、权限、提示和模型行为，最容易被跨权限、跨版本或跨时间错误复用。
- 检索缓存有什么好处？  
能降低重复检索开销，同时比最终答案更稳。

**原理补充**  
RAG 缓存的本质，是在时效性、权限边界和成本收益之间做分层治理。

**架构扩展**  
成熟团队通常会从检索缓存开始，再谨慎引入最终答案缓存，尤其在多租户和高时效场景里，不会一上来就缓存大段生成结果。

**工程指标**  
- 分层缓存命中率：看优化收益。  
- 因缓存过期问题导致的错误率：看一致性风险。  
- 跨权限缓存误用次数：看安全风险。  
- Token 节省比例：看成本收益。  
- 缓存失效率：看时效管理质量。

**证据链**  
- 缓存命中日志  
- 版本与权限键设计  
- 错误案例  
- 成本对比数据

**低风险优化路径**  
1. 先做检索层缓存  
2. 再按低风险场景引入答案缓存  
3. 最后完善版本和权限失效机制

**易错点**  
- 直接缓存最终答案而不考虑权限  
- 缓存键不含知识版本  
- 不分层一刀切缓存

## 14. 召回率和准确率冲突如何权衡？

**核心结论**  
召回率和准确率冲突时，通常先保证“不漏关键证据”，再用重排和生成约束控制噪音，因为漏掉关键资料比多带几条噪音往往更危险。

**标准答案**  
我在召回率和准确率之间取舍时，通常会先看业务容错。像合同问答、合规检索、故障排查这类场景，漏掉关键证据通常比多带几条无关内容更危险，所以会优先保证召回，再通过重排、引用筛选和回答约束提升最终精度。之所以不能一味追求 Top1 很准，是因为很多复杂问题本身就需要多个证据。工程上比较稳的做法，是把召回层和精排层职责分开，分别优化。成熟团队会按问题类型设策略，而不是全场景统一取舍。

**高分回答**
严格说要先明确“准确率”指候选精度、Top1 相关性还是最终回答正确率，因为三者不能混为一谈。我会按业务错误成本确定目标：合规核查漏掉反例代价高，召回层应扩大覆盖；导航类问题更看重首条精度；摘要类问题可能需要多份互补证据。随后让粗召回负责覆盖，融合和重排负责候选纯度，生成阶段只消费达到阈值且能支撑主张的证据。

调参不是简单增减 TopK。先看黄金证据是否进入候选：没有则改查询、切块、embedding 或召回通道；已经进入却排位低则优化融合与重排；排位正确但答案错误则处理上下文组织和生成约束。对相互重复的 chunk 做去重和多样性选择，可以在不牺牲覆盖的前提下降低噪音。复杂问题还应按子主题设置覆盖，而不是让相似候选占满窗口。

我会按查询类型画 precision/recall 曲线，并用业务加权损失选择工作点，同时监控 Recall@K、nDCG、证据充分率、上下文利用率、最终任务成功率、延迟与 token。若高召回导致模型被冲突内容误导，应呈现冲突或拒答，而不是静默挑选。最终阈值应在独立回归集和线上小流量实验中验证。

**追问点答案**  
- 为什么高风险场景更怕漏召回？  
因为缺关键证据会直接导致错误结论。  
- 准确率低会有什么问题？  
噪音多会拉高生成阶段出错概率。  
- 可以按问题类型动态调吗？  
可以；应按漏证据和噪音的业务代价设置候选数、重排阈值和多样性策略，并保留统一安全下限。

**原理补充**  
这类权衡本质上是在“信息覆盖度”和“候选纯度”之间做平衡。

**架构扩展**  
成熟 RAG 系统经常会对“事实核查型问题”和“导航类问题”使用不同召回策略，因为两类问题对漏和噪音的容忍完全不同。

**工程指标**  
- TopK 召回率：看覆盖度。  
- 最终引用正确率：看精度。  
- 噪音 chunk 比例：看纯度。  
- 任务成功率：看端到端收益。  
- 高风险场景误答率：看业务代价。

**证据链**  
- 召回/精排对比实验  
- 问题类型分层数据  
- 引用错误案例  
- 用户反馈

**低风险优化路径**  
1. 先按场景定义漏召回代价  
2. 再分层优化召回和精排  
3. 最后按问题类型做动态策略

**易错点**  
- 只盯单一指标  
- 不区分业务场景  
- 召回和重排职责混乱

## 15. 无答案场景如何优雅降级？

**核心结论**  
无答案场景的关键，不是硬答，而是能识别证据不足、明确表达边界，并给出下一步可执行建议。

**标准答案**  
我处理无答案场景时，通常会先定义什么叫“证据不足”，比如召回分数太低、引用不足、多个来源冲突或权限不完整。因为用户体验最差的并不是“我不知道”，而是模型很自信地给错答案。工程上比较稳的做法，是在低置信度时触发拒答、引导补充问题、推荐人工入口或返回相关文档列表。成熟团队会把无答案设计成产品能力，而不是把它当模型失败。

**高分回答**
无答案至少分四类：知识库确实没有、问题含糊无法确定意图、用户无权访问可能存在的资料，以及来源互相冲突或已过期。它们不能共用一句“未找到”。系统应根据类型分别给出补充信息提示、可公开的相关资料、权限申请入口、冲突说明或人工转接，但绝不能泄露“存在某份你无权访问的秘密文档”。

拒答判断不能只依赖向量最高分，因为不同查询和索引的分数不可直接比较。我会结合黄金证据上的校准结果、关键词与向量一致性、重排分差、证据覆盖、来源时效、主张支持度和规则风险判断；高风险场景使用更保守阈值。低置信度时让模型从受控响应类型中选择，并由服务端校验引用，禁止它用参数知识补全私有事实。

评测集要显式加入不可回答、近似但错误、权限受限、冲突和过期样本，观察无答案识别率、错误作答率、误拒率、澄清后完成率和人工转接成功率。拒答日志回流后持续校准阈值，并区分“系统不知道”和“用户没说清”，这样既降低自信误答，也避免系统因过度拒答失去可用性。

**追问点答案**  
- 为什么拒答也是能力？  
因为高风险场景下，拒答比错答更安全。  
- 低置信度怎么判断？  
要用标注集校准，综合重排分差、证据覆盖与时效、来源一致性和主张支持度，不能照搬单一向量分数阈值。
- 如何避免用户觉得系统“太笨”？  
拒答时给明确原因和下一步建议。

**原理补充**  
优雅降级的本质，是把“不知道”变成受控的产品行为，而不是系统缺陷。

**架构扩展**  
成熟系统通常会给无答案场景留产品出口，比如转人工、展示相关文档、让用户缩小问题范围，这比单纯返回“未找到”更可用。

**工程指标**  
- 低置信度拒答准确率：看安全性。  
- 错答率：看降级效果。  
- 无答案后用户继续完成率：看体验。  
- 人工转接率：看兜底路径使用情况。  
- 用户二次追问率：看表达质量。

**证据链**  
- 拒答规则  
- 低置信度日志  
- 用户后续行为数据  
- 错答样本

**低风险优化路径**  
1. 先定义低置信度规则  
2. 再设计拒答和兜底出口  
3. 最后持续校准误拒和错答

**易错点**  
- 没证据也硬答  
- 只返回“查不到”没有引导  
- 不监控拒答质量

## 16. RAG 评测集如何构建？

::: tip L2 深度训练
分层评估 Recall@K、排序、忠实度、引用和拒答：[RAG 质量闭环](/deep-dives/ai-architecture/01-rag-quality-loop)
:::

**核心结论**  
RAG 评测集要覆盖真实问题、真实文档、真实答案和真实引用，既测“答得对不对”，也测“找得准不准”。

**标准答案**  
我构建 RAG 评测集时，通常会先从真实用户问题、历史工单、FAQ、搜索日志里抽样，而不是只靠人工想题。因为人工编的问题往往太干净，不能代表生产环境。工程上比较稳的做法，是给每个问题标注标准答案、关键证据片段、可接受答案范围和高风险点，再按问题类型分层，比如事实问答、流程问答、多跳问答、无答案场景。成熟团队会把评测集当长期资产持续扩充，尤其会重视失败样本回流。

**高分回答**
我会先冻结一份与线上对应的知识快照，再从搜索日志、客服工单和专家场景中按真实流量分层抽样，覆盖事实、流程、聚合、多跳、时效、冲突、权限和无答案问题。每条样本记录原始问法、必要上下文、可接受答案要点、不可接受结论、黄金证据及版本、用户权限和风险等级；同一意图还要加入口语、省略、错别字和领域别名变体。

标注不能只给最终答案。检索层需要相关与不相关 chunk、必要证据集合和 hard negative；生成层需要主张级支持关系、引用要求和拒答预期。双人标注加仲裁，并跟踪一致率；对存在多种合理答案的题目使用要点与证据规则，避免把措辞差异误判为错误。合成数据可补稀有边界，但必须标识来源，不能替代真实分布。

数据集分为可迭代的开发集、固定回归集和严格隔离的发布门禁集，按用户、文档或时间去重，避免同源泄漏。评测分解 Recall@K、MRR/nDCG、证据充分率、引用蕴含、groundedness、拒答准确率与端到端任务成功率，并按切片报告最差表现。线上失败样本经过脱敏和专家复核后持续回流，同时保留历史快照，才能判断系统是真进步还是只适配了当前题集。

**追问点答案**  
- 为什么不能只用人工想题？  
因为和真实用户问题分布差异很大。  
- 评测集为什么要带引用标注？  
因为要分别判断黄金证据是否被召回、答案主张是否被证据支持，从而定位检索、排序还是生成阶段的问题。
- 失败样本为什么最有价值？  
因为它最能暴露系统短板。

**原理补充**  
评测集的本质，是把生产问题抽象成可重复验证的标准样本。

**架构扩展**  
成熟团队通常会把评测集分成开发集、回归集和上线门禁集，不同集合承担不同治理目的，避免所有评测都混在一起。

**工程指标**  
- 评测集覆盖率：看代表性。  
- 失败样本占比：看难度质量。  
- 问题类型分布：看是否贴近生产。  
- 引用标注完整率：看检索可测性。  
- 评测集更新频率：看治理活性。

**证据链**  
- 问题样本来源  
- 标注规则  
- 类型分布统计  
- 失败样本库

**低风险优化路径**  
1. 先从真实日志抽样  
2. 再给高价值场景做精标  
3. 最后把失败样本持续回流

**易错点**  
- 评测题脱离真实场景  
- 只标最终答案不标证据  
- 评测集建完不更新

## 17. 幻觉率如何在 RAG 中降低？

**核心结论**  
RAG 降低幻觉的关键，不是“有检索就一定对”，而是让检索、引用、Prompt 约束和拒答机制一起工作。

**标准答案**  
我降低 RAG 幻觉时，通常会从四层入手：先提升召回质量，确保关键证据真的被找到；再控制上下文噪音，避免模型被无关内容带偏；然后要求回答尽量基于引用生成；最后在证据不足时触发拒答或降级。因为很多 RAG 系统的问题不是完全没检索，而是检索了但证据不准、噪音太多、回答不受约束。成熟团队会把“是否基于证据回答”作为回归重点，而不是只看语言是否流畅。

**高分回答**
我会先把“幻觉”拆成可定位类型：黄金证据没召回、召回了错误或过期证据、上下文中有证据但模型忽略、答案主张超出证据，以及引用真实却不支持主张。不同类型对应切块与查询、排序与版本治理、上下文组织、生成约束和引用校验，不能只靠一句“请勿编造”的 Prompt 解决。

链路上先做权限与时效过滤、去重和重排，只把少量充分证据送给模型；将文档内容视为不可信数据，隔离其中的提示注入指令。生成输出按原子主张绑定 evidence_id，服务端校验引用存在且可访问；证据不足、冲突或主张无法获得支持时拒答或显式说明不确定性。对于数值、日期和高风险结论，可增加规则、确定性工具或第二阶段验证，但不能把另一个模型的“感觉”当作事实证明。

评测同时观察检索 Recall@K、证据充分率、主张支持率、引用准确率、错误作答率和误拒率，并按风险切片。线上抽样主张级审查，将失败回放到知识快照和链路版本；这样才能确认下降的是无依据回答，而不是因为系统大量拒答造成的表面改善。

**追问点答案**  
- 为什么 RAG 仍然会幻觉？  
因为检索质量、上下文噪音和生成策略都可能出问题。  
- 拒答为什么重要？  
因为没有充分证据时，强答最危险。  
- 只加引用就够了吗？  
不够；还要验证每个关键主张是否被对应片段蕴含，真实来源但不支持结论仍是错误引用。

**原理补充**  
RAG 降幻觉的本质，是把回答限制在外部证据边界内，并在边界不足时停止扩展。

**架构扩展**  
成熟产品通常会在高风险场景中增加“引用必须命中关键片段”“多证据一致才可回答”等规则，而不是只依赖模型自己克制。

**工程指标**  
- 无依据回答率：看核心风险。  
- 错误引用率：看证据可信度。  
- 高风险场景拒答率：看保护能力。  
- 噪音 chunk 比例：看上下文质量。  
- 用户纠错率：看真实效果。

**证据链**  
- 幻觉样本库  
- 引用与答案对照  
- 召回日志  
- 用户反馈

**低风险优化路径**  
1. 先提升检索和引用质量  
2. 再加回答约束和拒答规则  
3. 最后持续回流幻觉样本

**易错点**  
- 以为有 RAG 就天然低幻觉  
- 不检查引用质量  
- 没有拒答机制

## 18. 高并发下检索链路如何扩展？

**核心结论**  
高并发下检索扩展要同时考虑索引查询、过滤、缓存和下游模型调用协同，不是单纯把向量库实例数堆上去。

**标准答案**  
我做高并发检索扩展时，通常先拆链路瓶颈：是查询本身慢、过滤慢、重排慢，还是模型调用把整体拖住。因为很多时候检索层和生成层是连在一起的，只优化一边收益有限。工程上比较稳的做法，是先做查询标准化和缓存，再按查询热点、租户分片、索引副本和异步重排去扩展，同时给高峰期准备限流和降级策略。成熟团队会特别看热点查询和热点知识库，因为并发问题往往集中在少量高频模式上。

**高分回答**
我会先按查询改写、关键词/向量召回、过滤、融合、重排和生成拆分端到端预算，通过 tracing、负载回放和队列时间定位瓶颈。平均延迟没有意义，必须看不同租户、过滤选择性和索引更新期间的 P95/P99；并用开放环压测观察接近容量上限时是否排队雪崩。容量模型至少包含峰值 QPS、候选数、每阶段服务时间、并发上限和目标余量。

召回层尽量无状态，读副本扩展查询，按容量或租户分片，并为热点分片提供额外副本；连接池、批量 embedding、请求合并和授权安全的缓存吸收重复工作。重排与生成使用独立并发池和有界队列，配超时、熔断、背压与每租户配额，防止慢下游拖死检索。降级顺序应预先验证，例如跳过重排、缩小候选或返回带引用的搜索结果，但权限过滤、审计和拒答不能降级。

上线前做阶梯、突发、热点、节点故障与索引切换压测，验收吞吐、端到端 P99、错误率、队列深度、资源利用率和质量损失；运行中用容量水位自动扩容并保留手动保护开关。副本解决读吞吐与可用性，分片解决容量，但分片过多会增加扇出和尾延迟，因此必须用真实查询验证。

**追问点答案**  
- 为什么缓存是第一优先级？  
它对重复查询收益高，但不是无条件第一优先；应先用追踪确认热点，并确保缓存键绑定租户、权限和知识版本。
- 索引副本和分片分别解决什么？  
副本偏读扩展，分片偏容量和并行。  
- 为什么还要配降级？  
因为高峰时全量链路不一定都扛得住。

**原理补充**  
扩展的本质，是把热点流量、索引压力和下游生成成本拆开治理。

**架构扩展**  
成熟系统通常不会让每个请求都走最重链路，而是会用缓存、轻量召回、按需重排和模型分层共同吸收高峰。

**工程指标**  
- 检索 TP99：看核心性能。  
- 缓存命中率：看扩展收益。  
- 热点查询占比：看热点集中度。  
- 降级触发率：看高峰保护。  
- 端到端成功率：看整体效果。

**证据链**  
- 查询分布统计  
- 热点看板  
- 检索与重排监控  
- 限流降级日志

**低风险优化路径**  
1. 先定位热点和缓存收益  
2. 再做副本、分片和链路分层  
3. 最后补高峰降级和配额

**易错点**  
- 一味扩向量库实例  
- 不识别热点模式  
- 没有高峰期降级策略

## 19. 成本可控的 RAG 方案怎么做？

**核心结论**  
成本可控的 RAG 方案要从检索、重排、生成、缓存和知识更新全链路优化，而不是只盯模型 token 单价。

**标准答案**  
我做成本可控 RAG 时，通常会先把成本拆开：embedding 重建成本、检索查询成本、重排成本、生成 token 成本和缓存节省收益。因为很多团队以为 RAG 一定省钱，实际上如果切块太细、重排太重、上下文太长，成本照样会很高。工程上比较稳的做法，是优先控制上下文长度、缓存高频问题、只在高价值场景触发重排，并按业务价值分层模型。成熟团队会把每一步成本和质量收益一起看，避免为了省一点 token 反而把效果做垮。

**高分回答**
成本治理先建立按租户、场景和请求的成本账本：文档解析与 embedding、索引存储、查询 embedding、关键词/向量检索、重排、输入输出 token、缓存和人工兜底都要归因。只看模型单价会漏掉全量重建、过细切块造成的索引膨胀，以及低质量答案带来的重复提问和人工处理成本。

优化顺序是先删无价值工作，再降低单价。通过语义边界切块、去重和增量更新减少索引与重建；缩小重复候选、按问题复杂度设置 TopK 和重排触发；压缩上下文但保留支撑主张的证据；对权限和版本安全的热点使用分层缓存；简单分类、改写或摘要任务可路由轻量能力，高风险回答保留更强链路。每项优化都和固定质量集比较，低于质量下限就回退。

我会看每个成功任务成本而不只是单请求成本，并设置租户预算、配额、异常告警和月度预测。评估维度包括成本分布 P50/P95、缓存后的有效成本、单位知识更新成本、任务成功率和人工兜底率。若省 20% token 却显著增加误答或追问，总拥有成本反而更高；成本曲线必须与业务价值和风险等级一起解释。

**追问点答案**  
- 为什么上下文长度是大头？  
因为它直接决定生成阶段 token 消耗。  
- 成本优化为什么不能只降模型档位？  
因为索引、检索、重排、上下文冗余、重建和错误答案的人工成本都可能更大，且降档会影响任务成功率。
- 什么优化最常见？  
缓存、上下文压缩、按需重排、任务分层。

**原理补充**  
成本治理的核心，是让每一步开销都和业务价值对齐。

**架构扩展**  
成熟 RAG 平台通常会给不同业务线不同默认策略，比如高价值问答允许重链路，低价值场景默认轻链路，以此控制整体成本曲线。

**工程指标**  
- 单请求总成本：看整体经济性。  
- 上下文平均 token 长度：看主要消耗。  
- 重排触发率：看重链路使用情况。  
- 缓存节省比例：看优化收益。  
- 成本与任务成功率比值：看投入产出。

**证据链**  
- 成本拆解报表  
- 上下文长度分布  
- 缓存命中日志  
- 业务价值分层策略

**低风险优化路径**  
1. 先做成本拆解  
2. 再优化上下文和缓存  
3. 最后按业务分层重链路使用范围

**易错点**  
- 只盯模型单价  
- 不看上下文长度  
- 为省成本破坏高价值场景效果

## 20. 如何做 RAG 线上 A/B 实验？

::: tip L2 深度训练
连接离线回归、影子流量、A/B、失败样本与漂移监测：[AI 评测与发布门禁](/deep-dives/ai-architecture/04-evaluation-release-gates)
:::

**核心结论**  
RAG 线上 A/B 实验要比较的不只是点击或满意度，还要同时看引用质量、任务成功率、延迟、成本和错误边界。

**标准答案**  
我做 RAG A/B 时，通常会先明确实验目标到底是想优化召回、降低幻觉、减少成本，还是提高用户完成率。因为如果目标不清，后面很容易出现不同指标互相打架。工程上比较稳的做法，是把实验流量切到可比较人群，记录检索链路版本、Prompt 版本和知识版本，同时看用户行为指标和系统质量指标。成熟团队不会只看“满意度略升”，而会继续看错误率、拒答率、延迟和成本是不是一起健康。

**高分回答**
线上实验前，新方案必须先通过固定知识快照的离线回归和影子流量，确认没有权限、引用和延迟回归。A/B 只改变一个主要因素，例如切块、融合权重或重排策略；若整条链路一起变化，即使指标提升也无法归因。用户或会话要稳定分桶，避免同一人跨组污染，并分层检查租户、问题类型、设备和风险等级是否均衡。

实验预先写明主指标、护栏、最小可检测效应、样本量和停止规则。主指标可用任务完成率，质量护栏包括主张支持率、引用准确率、错误作答与越权事件，系统护栏包括 P99、错误率和单成功任务成本。满意度、点击或停留只能作为代理指标；高风险错误即使总体均值改善也可直接触发停止。知识版本、文档变更、模型、embedding、提示和检索配置都要记录，必要时冻结知识域或在分析中控制时间因素。

先从内部和低风险小流量开始，监控样本比率异常与分桶污染，再逐级放量。实验结束既看统计显著性，也看实际效应和分场景退化，保留失败样本回放；通过后仍以 canary 发布并保留快速回滚。对很少出现但后果严重的问题，A/B 样本通常不足，应依靠专项红队和硬门禁而不是等待线上显著性。

**追问点答案**  
- 为什么 RAG 实验不能只看点击率？  
因为点击只是代理信号，可能被表达形式影响；必须同时验证主张支持、引用正确、越权、延迟和每个成功任务成本。
- 知识版本为什么要记录？  
因为不同时点知识变化会干扰实验结果。  
- 系统指标为什么也要纳入 A/B？  
因为效果提升如果代价过高，未必值得上线。

**原理补充**  
RAG 实验的本质，是在真实流量下验证“某条检索-生成链路”是否真的更优。

**架构扩展**  
成熟团队做 RAG A/B 时，通常会先从低风险流量或内部流量开始，等质量和成本都稳定后再逐步放大，避免把线上实验变成生产事故。

**工程指标**  
- 任务完成率：看业务效果。  
- 引用正确率：看事实质量。  
- 幻觉率：看风险变化。  
- TTFT/TP99：看体验。  
- 单请求成本：看经济性。

**证据链**  
- 实验设计文档  
- 版本记录  
- 行为和质量指标对比  
- 样本回放结果

**低风险优化路径**  
1. 先定义单一清晰实验目标  
2. 再同时采集用户指标和系统指标  
3. 最后在小流量验证后逐步放量

**易错点**  
- 目标不清就做实验  
- 不记录知识和检索版本  
- 只看用户行为不看质量和成本
