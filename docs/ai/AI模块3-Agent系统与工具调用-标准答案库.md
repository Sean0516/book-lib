# AI模块3 - Agent系统与工具调用 - 标准答案库

## 1. Agent 和 Workflow 的边界是什么？

::: tip L2 深度训练
用任务路径、风险和恢复要求决定自主性边界：[Agent 生产化与工具安全](/deep-dives/ai-architecture/02-agent-production-safety)
:::

**核心结论**  
Workflow 更适合固定步骤、规则清晰的流程，Agent 更适合目标明确但路径不确定、需要动态决策和工具选择的任务。

**标准答案**  
我区分 Agent 和 Workflow 时，通常先看任务是不是“步骤基本已知”。如果一个任务的输入、判断分支和输出路径都比较稳定，比如审批流、固定 ETL、模板化报表，这类更适合 Workflow；如果任务目标明确，但中间要不要查知识、调哪个工具、是否需要分解子任务都依赖上下文，那更适合 Agent。之所以这个边界重要，是因为很多团队会把所有自动化都叫 Agent，结果把简单问题做复杂了。工程上比较稳的做法，是优先用 Workflow 承接确定性流程，只在确实需要动态规划和推理时引入 Agent。成熟团队不会为了“更智能”牺牲系统可控性。

**高分回答**
我会从路径不确定性、动作风险和可验证性三个维度划边界。输入输出明确、分支可枚举、每一步都有确定校验的审批、同步和报表任务，用 Workflow 更合适；目标明确但信息不完整，需要根据观察动态拆解任务、选择工具或调整路径时，才引入 Agent。即便需要 Agent，高风险写操作、合规检查和最终提交仍应回到确定性节点控制。

生产上通常采用“固定骨架 + 局部自主”：Workflow 管理身份校验、预算、状态持久化、审批、提交和补偿，Agent 只在受限节点生成计划或选择白名单工具。每次 Agent 决策都输出结构化意图和依据，由执行端重新鉴权和校验参数；模型没有直接持有凭证，也不能绕过状态机。失败时可以从最近检查点恢复，或降级到预定义路径和人工队列。

选型要通过同一任务集比较端到端成功率、路径方差、人工介入率、P95 完成时间、单成功任务成本和高风险错误率。如果 Agent 仅带来表达变化，却没有提高复杂任务完成率，就保留 Workflow；只有规则分支持续爆炸、人工需要频繁临场判断，且自主方案在安全门禁下有明确收益时，再扩大 Agent 边界。

**追问点答案**  
- 为什么 Workflow 更容易上线？  
因为步骤固定、测试边界清楚、回归成本低。  
- Agent 最大价值是什么？  
在路径不确定的任务里动态选择最合适的执行方式。  
- 两者能混用吗？  
可以，常见做法是 Workflow 控制状态、审批和副作用，Agent 只负责局部规划与工具选择，并在检查点恢复。

**原理补充**  
两者本质区别在于控制权归属：Workflow 由系统预定义流程控制，Agent 由模型在约束内做运行时决策。

**架构扩展**  
成熟 AI 系统常见做法不是“全 Agent 化”，而是固定骨架 + 局部 Agent。这样既能保留关键链路可控性，又能在复杂节点引入智能决策。

**工程指标**  
- 自动化任务成功率：看方案适配度。  
- 平均执行路径长度：看任务复杂度。  
- 人工兜底率：看可控性。  
- 回归问题数：看治理成本。  
- 单任务平均成本：看工程收益。

**证据链**  
- 任务流程图  
- 动态决策样本  
- 失败案例回放  
- 人工介入记录

**低风险优化路径**  
1. 先把固定流程留给 Workflow  
2. 再把高不确定节点局部 Agent 化  
3. 最后补审计和回退能力

**易错点**  
- 把所有流程都做成 Agent  
- 忽略可测试性和回归成本  
- 没有明确 Agent 决策边界

## 2. 何时用 ReAct，何时用固定工作流？

**核心结论**  
ReAct 适合需要边想边做、边观察边纠偏的任务，固定工作流适合步骤明确、可预测、要求高稳定性的任务。

**标准答案**  
我选择 ReAct 还是固定工作流时，通常先看任务是否需要运行时探索。像开放式信息搜集、复杂排障、跨工具分析这类任务，中间往往要根据新观察到的信息调整后续动作，这时 ReAct 更有价值；但像工单分发、表单校验、固定报表生成这种任务，步骤本来就稳定，强行用 ReAct 只会增加波动。工程上比较稳的做法，是让 ReAct 承担少量真正需要推理闭环的任务，把大多数高频稳定流程交给固定工作流。成熟团队不会把 ReAct 当成默认架构，而是当成一种高弹性但高治理成本的能力。

**高分回答**
ReAct 适用于“下一步取决于刚得到的观察”的任务，例如故障排查先查指标，再根据异常选择日志或配置工具；固定工作流适用于路径能在设计期枚举、结果可用规则验证的任务。两者不是按任务听起来是否复杂来选，而是看运行时是否真的需要探索。能够提前写成状态机的流程，通常不应让模型重复临场规划。

使用 ReAct 时，我会把循环实现为受控运行时：模型只能产生结构化 action，系统校验工具、参数、权限和预算后执行，再把经过裁剪的不可信 observation 返回。运行时维护已执行动作、关键事实、最大步数、时间与费用预算，以及成功、无进展、证据不足和需审批等终止条件。副作用工具不得在思考循环中自动反复执行，应通过幂等键和审批节点隔离。

落地前用探索型与确定型任务分别比较固定流程、ReAct 和混合方案，关注成功率、无效步骤率、路径方差、循环终止率、P99 和每个成功任务成本。常见生产方案是固定工作流负责主干，只有诊断或信息搜集节点进入 ReAct，超时、重复动作或置信度不足时回退到规则路径或人工处理。

**追问点答案**  
- ReAct 最大风险是什么？  
路径和成本不稳定只是表象，更直接的风险是错误观察被迭代放大、循环无收敛，以及副作用动作被重复执行。
- 固定工作流最大短板是什么？  
面对新情况时灵活性不足。  
- 是否可以先工作流再在节点里用 ReAct？  
可以，而且这是很常见的生产方案。

**原理补充**  
ReAct 的核心是让模型在“推理-行动-观察”循环中迭代决策，而固定工作流则把循环提前固化成系统逻辑。

**架构扩展**  
成熟平台通常会把 ReAct 限制在可审计、低爆炸半径的任务里，并用预算、步数、工具白名单来约束它，否则很容易出现循环调用和不可解释行为。

**工程指标**  
- 平均工具调用步数：看执行复杂度。  
- 执行路径稳定性：看可控性。  
- 单任务成本波动：看经济风险。  
- 中途纠偏成功率：看 ReAct 价值。  
- 回归失败率：看治理压力。

**证据链**  
- 路径回放日志  
- 成本曲线  
- 任务分类清单  
- 失败案例

**低风险优化路径**  
1. 先识别必须运行时探索的任务  
2. 再小范围试点 ReAct  
3. 最后保留固定工作流兜底

**易错点**  
- 为了“高级”优先用 ReAct  
- 缺少步数和预算限制  
- 固定流程也交给模型临时决定

## 3. 工具调用协议如何设计？

**核心结论**  
工具调用协议要把输入、输出、权限、错误和幂等边界定义清楚，让模型调用是“受约束的能力消费”，而不是随意执行。

**标准答案**  
我设计工具调用协议时，通常会先把工具当成正式接口，而不是给模型开放一个模糊能力。比较稳的做法，是给每个工具定义清晰 schema、字段含义、默认值、枚举范围、错误码和权限要求，同时限制模型只能在这些约束里调用。之所以协议设计重要，是因为模型调用不稳定时，很多问题并不是模型不会用，而是工具定义本身太模糊。工程上我还会考虑幂等字段、审计标识和超时语义，确保工具可以被可靠追踪和重放。成熟系统不会把工具调用停留在“模型会吐 JSON”这一级。

**高分回答**
工具协议应像面向不可信调用者的正式 API。定义唯一名称和版本、单一明确职责、严格输入输出 schema、单位与时区、枚举和互斥字段，并用描述和正反例说明前置条件与副作用。不要让模型拼接 SQL、Shell 或 URL；应让它提交业务意图，由适配层参数化执行。敏感字段、租户和调用身份从可信上下文注入，不能由模型自由填写。

执行契约还要明确超时、可重试错误、永久错误、限流、部分成功、幂等键和取消语义。只读与写入工具分开；写操作先返回可预览的变更计划或 dry-run，高风险动作进入审批，再以冻结参数提交，防止审批后被重新规划。工具返回结构化数据、稳定错误码和可安全呈现的摘要，原始输出经过大小限制、脱敏和提示注入隔离后才能进入上下文。

协议采用向后兼容的版本策略，并用契约测试、错误注入和历史轨迹回放验证升级。关键指标包括参数校验失败率、工具选择准确率、重复副作用率、超时率、可恢复失败率和审计覆盖率。协议设计的目标不是让模型“尽量会用”，而是让错误调用在执行前被拒绝、执行后可定位且副作用可控制。

**追问点答案**  
- 为什么 schema 要尽量明确？  
因为模糊字段会让模型误解参数含义。  
- 为什么错误码也重要？  
因为 Agent 需要根据失败类型决定重试、降级还是人工介入。  
- 幂等字段什么时候需要？  
所有可能重试的副作用操作都需要，并由服务端保存幂等结果；仅让模型生成一个随机键并不能阻止同一意图重复执行。

**原理补充**  
工具协议的本质，是把模型的不确定决策收束到系统可验证、可执行的接口边界里。

**架构扩展**  
成熟平台通常会把工具协议统一做成平台资产，而不是每个团队各写一套 JSON 描述。这样后续权限治理、审计和回归测试会顺很多。

**工程指标**  
- 参数合法率：看协议清晰度。  
- 工具误调用率：看设计质量。  
- 幂等冲突数：看副作用控制。  
- 协议变更回归问题数：看可维护性。  
- 工具调用审计完整率：看治理能力。

**证据链**  
- Tool schema 文档  
- 错误码规范  
- 调用审计日志  
- 回归报告

**低风险优化路径**  
1. 先统一 schema 风格  
2. 再补错误语义和权限字段  
3. 最后把幂等与审计接入主链路

**易错点**  
- schema 太松散  
- 工具副作用边界不清  
- 协议变更无回归

## 4. 多工具编排如何避免循环调用？

**核心结论**  
避免循环调用的关键，是给 Agent 明确目标终止条件、步数预算、工具状态记忆和异常熔断，而不是只相信模型会自己停下来。

**标准答案**  
我做多工具编排时，通常会把“为什么结束”设计得和“怎么开始”一样清楚。因为 Agent 最容易出问题的场景之一，就是反复搜索、反复调用同一个工具、或者在几个工具之间来回震荡。工程上比较稳的做法，是限制最大步数、记录已执行动作、对重复调用设检测规则，并在目标已满足、证据不足或成本超预算时强制收束。成熟团队不会把终止逻辑完全交给模型，而会在系统层做护栏。

**高分回答**
循环通常来自目标不可验证、观察没有新增信息，或错误恢复策略把 Agent 送回同一状态。我会先把任务定义成状态机或有向执行图：状态包含目标、已确认事实、未满足条件、工具结果版本和预算；每个动作声明前置条件、预期状态变化与终止判定。运行时而不是 Prompt 负责最大步数、墙钟时间、token/费用和每类工具调用次数上限。

重复检测不能只比较工具名，要对工具、规范化参数和相关状态生成动作指纹。如果连续调用没有新增事实、状态哈希重复，或在 A→B→A 间震荡，就触发无进展熔断。瞬时错误按错误类型有限重试并退避；参数错误交给模型修正一次；权限、业务拒绝和确定性空结果不得盲重试。写操作还要使用业务幂等键，避免看似不同的轨迹造成重复副作用。

收束结果分为成功、部分成功、证据不足、预算耗尽和需人工介入，不能强迫每条轨迹生成成功答案。通过历史轨迹回放和故障注入统计重复动作率、每步信息增益、预算终止率、循环检测准确率、任务成功率与成本长尾；误杀正常探索时，再按任务类型调整窗口，而不是取消硬上限。

**追问点答案**  
- 为什么步数限制是必需的？  
因为它是防止成本和行为失控的最后保险。  
- 仅靠 Prompt 约束能避免循环吗？  
通常不够，系统级检测更可靠。  
- 重复调用检测怎么做？  
对工具、规范化参数和任务状态生成指纹，同时检测状态无进展和 A→B→A 震荡；只比较工具名会误伤合法分页调用。

**原理补充**  
循环调用问题本质上是“目标未收敛 + 缺少终止约束”的组合故障。

**架构扩展**  
成熟平台通常会对 Agent 运行时维护状态机和预算器，而不是只有一段 Prompt。这样才能把模型探索行为控制在可预测范围内。

**工程指标**  
- 平均步数：看链路复杂度。  
- 重复调用率：看循环风险。  
- 超预算终止率：看保护触发情况。  
- 人工介入率：看治理效果。  
- 单任务成本长尾：看风险暴露。

**证据链**  
- 运行轨迹日志  
- 重复调用样本  
- 预算器记录  
- 异常终止案例

**低风险优化路径**  
1. 先加步数和预算硬限制  
2. 再做重复动作检测  
3. 最后补明确终止条件和人工兜底

**易错点**  
- 只有 Prompt 没有系统护栏  
- 不记录执行状态  
- 没有超预算止损

## 5. Agent 的短期记忆如何实现？

**核心结论**  
短期记忆的目标，是让 Agent 在一个任务会话内保留必要上下文和中间决策，但不把所有历史都无差别塞回上下文。

**标准答案**  
我实现短期记忆时，通常会先区分哪些信息是任务当前必须保留的，比如用户目标、关键约束、已执行步骤、已获得证据和未解决问题。因为如果把所有对话和所有工具输出都回灌给模型，很快就会把上下文撑爆，还会引入噪音。工程上比较稳的做法，是维护结构化任务状态，比如目标摘要、已完成步骤、关键证据列表，再按需把最相关的信息送回模型。成熟团队会把短期记忆做成任务状态管理，而不是简单聊天记录堆积。

**高分回答**
短期记忆应区分事实源与模型摘要。原始用户输入、工具结果和审批记录作为不可变事件保存；模型真正使用的是结构化工作状态，包括目标、硬约束、计划版本、已完成动作、证据引用、待决问题和预算。摘要只能压缩内容，不能覆盖原始证据；重要数值、标识符、否定条件和授权信息应从可信来源直接注入，避免多轮摘要漂移。

上下文组装采用固定优先级：系统与安全约束、当前用户请求、任务状态、当前步骤所需证据，最后才是有限的近期对话。大工具输出存外部对象，仅把带 ID 的相关片段送入模型；每个状态更新做 schema 校验、版本控制和乐观并发，防止并行步骤互相覆盖。会话恢复从检查点与事件日志重建，并对过期凭证和权限重新验证。

评测要设计跨轮约束、任务中断恢复、上下文压缩和并发更新案例，观察约束保留率、事实失真率、恢复成功率、上下文 token、任务成功率和陈旧状态误用率。若摘要与原始证据冲突，以证据为准并标记重新规划，而不是继续沿用模型记忆。

**追问点答案**  
- 为什么不能把所有历史都塞回上下文？  
因为成本高、噪音大，还会降低模型注意力。  
- 短期记忆最该保留什么？  
目标、约束、已完成动作、关键事实。  
- 短期记忆一定要存数据库吗？  
短任务可存在会话层，但跨进程恢复、长任务、审批等待或并发执行需要持久化检查点与版本控制。

**原理补充**  
短期记忆本质上是在有限上下文预算下保留完成当前任务所需的最小状态集。

**架构扩展**  
成熟 Agent 系统通常会把记忆分层：对话文本只是原始数据，真正给模型用的是抽取后的结构化状态，这样更稳也更便宜。

**工程指标**  
- 上下文长度增长率：看记忆控制效果。  
- 任务续接成功率：看状态保留质量。  
- 冗余历史占比：看噪音水平。  
- 单任务 token 成本：看经济性。  
- 记忆丢失导致的任务失败数：看稳定性。

**证据链**  
- 会话状态结构  
- 上下文拼装日志  
- 任务失败样本  
- 成本看板

**低风险优化路径**  
1. 先定义最小任务状态  
2. 再做结构化摘要  
3. 最后控制上下文注入策略

**易错点**  
- 聊天记录原样全量回灌  
- 不区分关键信息和噪音  
- 没有任务状态结构

## 6. 长期记忆如何做召回与过期？

**核心结论**  
长期记忆要像知识库一样治理，重点是可召回、可更新、可过期，而不是把所有历史永久保存并默认可信。

**标准答案**  
我做长期记忆时，通常先定义哪些记忆值得长期保留，比如稳定偏好、常用配置、长期目标、历史产出，而不是把所有临时会话都存进去。因为长期记忆一旦脏了，后续每次召回都会污染决策。工程上比较稳的做法，是给记忆加类型、来源、时间戳、置信度和过期策略，召回时按任务相关性和新鲜度筛选。成熟团队会把长期记忆看成动态资产，而不是永不清理的沉淀池。

**高分回答**
长期记忆先按可信度分层：用户显式保存的偏好和配置是强事实，经过工具验证的历史事实次之，模型从对话推断出的偏好只是待确认假设。只有对未来任务有复用价值、用户有权保存且符合最小化原则的信息才写入；敏感信息默认不记忆，并提供查看、更正、删除和关闭记忆的入口。原始会话不是长期记忆本身。

每条记忆包含主体与租户、类型、内容、来源证据、创建与最后确认时间、置信度、有效期和替代关系。写入前去重或合并，冲突时保留版本并优先显式新事实，不能让旧推断覆盖用户最新配置。召回采用主体硬过滤后再按任务相关性、时间衰减、置信度和来源排序；模型上下文明确标注“确定配置”还是“可能偏好”，关键动作使用前再次确认。

过期既有 TTL，也有事件驱动失效，例如岗位、项目或权限变化立即撤销相关记忆。通过有记忆/无记忆对照集评估任务成功率提升、错误个性化率、过期记忆误用率、召回精度和用户纠正率；若某类记忆带来的误导高于收益，就停止自动写入。长期记忆是受治理的用户数据产品，不是无限增长的向量库。

**追问点答案**  
- 为什么长期记忆要过期？  
因为用户偏好、业务规则和上下文都会变化。  
- 召回时为什么要看新鲜度？  
旧记忆可能不再适用。  
- 长期记忆和 RAG 知识库一样吗？  
不一样；记忆通常绑定主体、可能是软事实且需要用户控制，知识库强调组织事实、版本和访问治理，两者应分库并标注可信度。

**原理补充**  
长期记忆的本质，是把“未来可能有价值的历史状态”以受控方式保留下来，并在需要时精准召回。

**架构扩展**  
成熟系统通常会把长期记忆和显式用户配置分开管理，因为配置是强事实，记忆更像软事实，两者的可信度不一样。

**工程指标**  
- 记忆召回命中率：看相关性。  
- 过期记忆误用率：看风险。  
- 记忆更新成功率：看治理质量。  
- 陈旧记忆清理比例：看新鲜度维护。  
- 召回后任务提升幅度：看真实价值。

**证据链**  
- 记忆存储结构  
- 召回日志  
- 过期规则  
- 误用案例

**低风险优化路径**  
1. 先限制长期记忆类型  
2. 再补时间戳和过期策略  
3. 最后做召回评估和清理机制

**易错点**  
- 什么都长期保存  
- 记忆无过期策略  
- 召回不看新鲜度和来源

## 7. 如何限制 Agent 权限范围？

::: tip L2 深度训练
把最小权限、执行端鉴权、审批和参数冻结组合起来：[Agent 生产化与工具安全](/deep-dives/ai-architecture/02-agent-production-safety)
:::

**核心结论**  
限制 Agent 权限的关键，是默认最小权限、按任务授予临时能力，并把高风险动作放到人工审批或系统二次校验之后。

**标准答案**  
我限制 Agent 权限时，通常不会给它一个“全能账号”，而是按任务目标只开放必要工具、必要参数范围和必要数据访问。因为 Agent 一旦有写操作、支付、删除、外发消息这类能力，风险就从“答错”升级成“做错事”。工程上比较稳的做法，是让 Agent 使用短时令牌、能力白名单、参数约束和审批节点，高风险动作必须二次确认。成熟团队不会把权限问题交给 Prompt 约束，而是用身份和授权体系真正收紧执行面。

**高分回答**
Agent 本身不持有长期高权限凭证。入口先确认发起用户与租户，策略服务根据当前任务签发短时、可撤销、范围化的能力令牌，只允许指定工具、资源、动作、参数范围和次数。工具执行端必须独立鉴权，模型产生的角色、tenant_id、收件人或资源 ID 都不能被信任；网络出口、文件路径和数据行级权限也要在基础设施层约束。

我会把工具按只读、可逆写入、高风险不可逆动作分级。只读也要防敏感数据外泄；可逆写入使用 dry-run、幂等键和回滚；付款、删除、对外发送、权限变更在执行前生成冻结的变更摘要，由有资格的人审批，审批令牌只对该参数哈希有效。工具结果经过脱敏和最小化后才回到模型，避免 Agent 借助低权限工具组合出高权限效果。

验收用越权资源、参数替换、审批后篡改、工具链权限提升和凭证过期等攻击用例，要求越权执行为零，并监控拦截率、误拒率、令牌寿命、高风险审批覆盖和撤权生效 P99。应急时能一键撤销 Agent 身份或单个工具能力，且审计可还原用户授权、模型意图、策略决策和实际副作用。

**追问点答案**  
- 为什么 Prompt 里写“不要乱做”不够？  
因为权限控制最终要靠系统，不靠模型自觉。  
- 什么动作最该人工审批？  
资金、数据删除、对外发送、高权限变更。  
- 临时授权为什么更好？  
因为可把权限绑定任务、工具、资源、参数、次数和有效期，并能在任务结束或异常时立即撤销。

**原理补充**  
权限限制的本质，是把 Agent 从“可能执行任何事”收束成“只能在受控边界内执行少数事”。

**架构扩展**  
成熟平台会让每个 Agent 身份、工具权限和租户范围都能被审计回放，这样出现问题时能追到“谁在什么上下文下被允许做了什么”。

**工程指标**  
- 高风险动作审批覆盖率：看保护力度。  
- 越权调用拦截率：看控制效果。  
- 临时凭证使用率：看最小权限落地程度。  
- 权限误配导致的事件数：看治理质量。  
- 审计可追踪率：看事后可控性。

**证据链**  
- 权限模型  
- 工具白名单  
- 审批记录  
- 审计日志

**低风险优化路径**  
1. 先做工具白名单  
2. 再补短时授权和参数约束  
3. 最后把高风险动作接审批

**易错点**  
- 给 Agent 宽泛系统权限  
- 把权限控制寄托在 Prompt 里  
- 高风险动作无审批

## 8. Tool schema 设计的关键点有哪些？

**核心结论**  
Tool schema 设计的关键是语义明确、参数可验证、默认值可控、错误可解释，帮助模型“更容易做对”。

**标准答案**  
我设计 Tool schema 时，通常会优先保证字段含义足够清楚，让模型知道什么时候该填、填什么、哪些值合法。因为很多函数调用错误不是模型能力不够，而是 schema 自己就模糊，比如字段名太抽象、可选项太多、默认值含义不清。工程上比较稳的做法，是用少量必要参数、明确枚举、示例说明和输入约束把 schema 做得尽量收敛。成熟团队会把 schema 当作“给模型看的 API 设计”，所以比给人看的接口还要更显式。

**高分回答**
好的 schema 先做到工具职责单一且命名可区分，避免多个工具描述高度重叠让模型猜测。字段使用业务语义名称，描述单位、格式、时区、是否含边界和缺省行为；必填尽量少，用 enum、oneOf、长度与数值范围收窄空间，并表达条件依赖。例如“退款”应接收订单标识和业务原因，而不是让模型填写任意账户与金额。

安全相关值不放进模型可控 schema：用户、租户、权限范围和服务凭证由执行上下文注入。危险操作把预览和提交拆成两个工具，提交引用服务端生成的 plan_id 与参数哈希。输出同样需要 schema，区分成功、部分成功、可重试错误、业务拒绝和未知结果，并限制返回长度、敏感字段与不可信文本，避免工具输出污染上下文。

Schema 变更采用显式版本和兼容窗口，用黄金调用集、非法参数、边界值、描述歧义与历史轨迹回放做契约测试。上线观察工具选择混淆矩阵、字段补全率、校验失败率、修复后成功率和误执行率。若模型经常错填，先简化工具边界或拆分 schema，而不是堆更多示例掩盖接口设计问题。

**追问点答案**  
- 为什么字段命名这么重要？  
因为模型主要依赖名字和描述理解参数语义。  
- 参数越多越好吗？  
不是；应保留完成意图所需的最小参数，身份和权限字段由系统注入，互斥或高歧义参数优先拆成不同工具。
- 为什么要有错误可解释性？  
因为 Agent 需要据此决定重试还是放弃。

**原理补充**  
Schema 的本质，是把工具调用空间压缩到模型更容易正确搜索的范围内。

**架构扩展**  
成熟平台会把 schema 规范和代码生成、回归测试打通，避免不同团队工具风格过于发散，影响 Agent 稳定性。

**工程指标**  
- Schema 参数合法率：看设计质量。  
- 模型填参错误率：看易用性。  
- 可选参数误用率：看复杂度。  
- 工具回归问题数：看维护压力。  
- 错误后恢复成功率：看异常语义质量。

**证据链**  
- Schema 文档  
- 调用错误样本  
- 回归测试报告  
- 代码生成模板

**低风险优化路径**  
1. 先减少非必要参数  
2. 再补清晰描述和示例  
3. 最后做 schema 级回归

**易错点**  
- 参数名过于抽象  
- schema 太复杂  
- 工具风格各自为政

## 9. 多轮任务状态机如何建模？

**核心结论**  
多轮任务状态机要先定义阶段、转移条件和异常出口，确保 Agent 不会在多轮交互里失去任务边界。

**标准答案**  
我建多轮任务状态机时，通常先把任务拆成几个明确阶段，比如目标确认、信息收集、执行、校验、收尾，并定义每个阶段何时进入、何时退出、失败后去哪。因为多轮任务最大的问题之一就是“聊着聊着目标漂了”。工程上比较稳的做法，是让系统维护显式状态，而不是让模型自己隐式记住所有进度。成熟系统会在关键节点做状态落盘和人工接管点，这样即使中途失败也能恢复。

**高分回答**
我会把业务状态与会话文本分开。业务状态机包含 task_id、目标与约束、当前状态、计划版本、已确认输入、待审批动作、工具结果引用和预算；典型状态是待澄清、已规划、执行中、待审批、校验中、完成、失败、取消。每次转移都由明确事件触发，声明前置条件、幂等处理、超时、补偿与可见输出，模型只能建议事件，状态存储负责判断转移是否合法。

长任务采用事件日志加持久化检查点，写入时使用版本号防止并行分支覆盖。外部调用记录 request_id 和结果状态；崩溃恢复后先查询副作用是否已发生，再决定继续、补偿或转人工。审批等待和用户补充信息属于可持久化暂停状态，恢复时重新校验权限、数据时效和预算，不能盲用旧上下文。

测试不仅覆盖主路径，还要覆盖重复事件、乱序回调、用户取消、审批过期、工具未知结果、进程重启和补偿失败。指标包括非法转移拦截数、卡住任务年龄、恢复成功率、重复副作用率、各状态停留 P95 和端到端完成率。状态数应保持最小可解释，只有在恢复、权限或监控语义不同的时候才新增状态。

**追问点答案**  
- 为什么状态机要显式化？  
因为隐式状态难调试、难恢复、难审计。  
- 多轮任务为什么容易漂移？  
因为上下文越来越长、目标和约束容易被稀释。  
- 中断恢复为什么重要？  
因为任务可能跨进程、审批和外部回调；恢复时必须基于持久化状态判断副作用是否发生，而不是从聊天记录重新猜测。

**原理补充**  
状态机的本质，是给开放式 Agent 行为加上一条可验证的进度主线。

**架构扩展**  
成熟平台经常把 Agent 的会话状态和业务状态分开管理：前者服务模型交互，后者服务系统可控和恢复，这样更稳。

**工程指标**  
- 状态转移成功率：看流程稳定性。  
- 异常中断恢复率：看韧性。  
- 状态漂移导致的失败数：看边界控制。  
- 平均多轮完成时长：看效率。  
- 人工接管率：看自动化成熟度。

**证据链**  
- 状态机定义  
- 会话回放  
- 中断恢复日志  
- 异常样本

**低风险优化路径**  
1. 先定义少量关键状态  
2. 再明确异常出口和恢复点  
3. 最后做可视化和审计

**易错点**  
- 完全依赖模型隐式记忆状态  
- 没有异常出口  
- 状态和业务事件混在一起

## 10. Agent 失败重试策略如何设计？

::: tip L2 深度训练
区分明确失败与未知结果，用幂等键和状态查询恢复：[Agent 生产化与工具安全](/deep-dives/ai-architecture/02-agent-production-safety)
:::

**核心结论**  
失败重试要按失败类型分层处理，能重试的不一定都该马上重试，有副作用的动作更不能盲目重放。

**标准答案**  
我设计 Agent 重试策略时，通常先把失败分成几类：临时性故障、参数错误、权限问题、工具超时、业务拒绝。因为不同类型的失败后续动作完全不一样。工程上比较稳的做法，是临时故障才做限次重试，参数类错误优先纠偏，权限和业务拒绝则应该终止或转人工；对有副作用的操作，还要先靠幂等键和审计保证不会重复执行。成熟系统不会把“失败就再来一次”当成默认策略。

**高分回答**
第一步是区分明确失败和未知结果。连接建立前失败通常可安全重试；收到校验错误、权限拒绝或业务冲突应纠正或终止；请求超时却不知道服务端是否完成时，绝不能直接重放副作用操作，而要用幂等键查询执行状态。服务端需持久化幂等键、请求摘要和首次结果，同一意图重复提交返回同一结果，参数不同则拒绝冲突。

瞬时限流、网络抖动和明确的 5xx 使用有限次数的指数退避与 jitter，并遵守服务端 retry-after；参数错误只允许基于结构化错误修正有限次数；权限和不可重试业务错误直接转用户或人工。每个任务设置总重试预算，断路器阻止依赖故障演变成重试风暴。写操作采用查询状态、补偿或人工核对，不能假设所有动作都可回滚。

运行记录 attempt、错误类别、幂等键、依赖请求 ID 和最终决策，监控首次成功率、重试恢复率、放大系数、未知结果积压、重复副作用和死信年龄。故障注入要覆盖超时前成功、超时后失败、回调丢失和服务重启，验证系统最终能收敛到成功、明确失败或人工处理，而不是无限重试。

**追问点答案**  
- 为什么副作用动作不能盲重试？  
因为可能造成重复创建、重复扣费或重复通知。  
- 临时性失败怎么识别？  
通常看超时、网络抖动、短暂 5xx 这类信号。  
- 纠偏和重试有什么区别？  
纠偏改变错误参数或计划；重试保持同一意图再次执行。副作用请求即使纠偏后也要生成可审计的新版本，并继续受幂等和审批约束。

**原理补充**  
重试治理的本质，是区分“偶发失败”和“设计上就不该继续”的情况。

**架构扩展**  
成熟系统常把 Agent 重试与工具幂等、死信队列、人工接管一起设计，因为它本质上已进入分布式可靠性问题范围。

**工程指标**  
- 重试成功率：看收益。  
- 误重试率：看策略质量。  
- 有副作用操作重复执行数：看风险。  
- 转人工比例：看兜底效果。  
- 死信堆积量：看异常收敛情况。

**证据链**  
- 失败分类日志  
- 幂等记录  
- 重试轨迹  
- 死信样本

**低风险优化路径**  
1. 先给失败做分类  
2. 再限制重试只覆盖临时故障  
3. 最后补幂等和人工兜底

**易错点**  
- 所有失败统一重试  
- 有副作用动作无幂等  
- 没有失败分类和审计

## 11. 人工审批节点何时必须加入？

**核心结论**  
当任务涉及高金额、高权限、外部发送、不可逆操作或合规高风险时，人工审批节点通常是必须的，不应完全自动化。

**标准答案**  
我决定是否加人工审批时，通常先看三个问题：这个动作有没有不可逆副作用、有没有高合规风险、出错后的业务代价有多大。像转账、删数据、对外发函、批量权限变更、合同发送这类动作，即使 Agent 判断再好，也不适合完全自动闭环。工程上比较稳的做法，是让 Agent 做准备和建议，人来确认最后一步，这样既保留效率，也保留责任边界。成熟系统不会把人工节点视为“自动化不够高级”，而会把它看成风险吸收层。

**高分回答**
我会用风险矩阵而不是凭感觉决定审批：影响范围、金额或数据敏感度、可逆性、外部可见性、法规要求和模型置信度共同决定动作等级。资金划转、生产删除、权限提升、代表用户对外发送、法律承诺和大规模变更属于强制审批；只读查询、可验证且可撤销的小范围动作可自动执行。风险阈值由业务与合规负责人配置，模型无权自行降级。

审批必须放在最后一个不可逆副作用之前。Agent 先完成信息收集和 dry-run，生成包含目标、证据、差异、影响范围、风险、回滚方案和精确参数的执行计划；系统冻结 plan_id、参数哈希、数据版本与有效期。审批人必须具备相应权限，批准令牌仅对该计划有效，任何参数或上下文变化都要求重新审批，防止“批的是 A，执行成 B”。高风险操作可采用双人复核和职责分离。

审批超时、拒绝和撤回都要成为显式状态，不能默认通过；紧急绕过必须单独授权并完整审计。指标包括强制审批覆盖率、审批后参数变更拦截率、驳回率、审批耗时、误执行事件和自动化节省时间。若审批人经常无法判断，问题通常是证据与差异展示不足，而不是应该取消审批。

**追问点答案**  
- 人工审批是不是意味着系统不成熟？  
不是，高风险场景下这是成熟治理的一部分。  
- 审批最适合放在哪一步？  
放在真正产生高代价副作用之前。  
- 审批信息应该展示什么？  
还要展示精确参数、影响范围、数据版本、可逆性和回滚方案，并冻结参数哈希，变更后必须重新审批。

**原理补充**  
人工审批的本质，是把模型推荐和最终责任承担分开。

**架构扩展**  
成熟团队通常会让 Agent 自动整理上下文、证据和建议动作，把人工审批从“人工搜信息”变成“人工做判断”，这样效率和安全性才能兼顾。

**工程指标**  
- 高风险动作审批覆盖率：看治理完整性。  
- 审批平均耗时：看流程效率。  
- 审批后驳回率：看 Agent 建议质量。  
- 无审批误执行事件数：看风险。  
- 审批负担占比：看流程设计是否合理。

**证据链**  
- 审批流定义  
- 高风险动作清单  
- 审批日志  
- 驳回样本

**低风险优化路径**  
1. 先定义必须审批的动作清单  
2. 再让 Agent 输出可审阅建议  
3. 最后优化审批体验和时延

**易错点**  
- 高风险动作无审批  
- 审批节点太早或太晚  
- 审批界面不提供充分证据

## 12. Agent 结果可解释性如何增强？

**核心结论**  
增强可解释性的关键，不是让模型写一大段“我为什么这么想”，而是保留任务目标、关键证据、工具轨迹和最终结论之间的对应关系。

**标准答案**  
我增强 Agent 结果可解释性时，通常会优先保留结构化过程信息，比如它调用了哪些工具、依据了哪些文档、为什么选择了这个动作，而不是简单要求模型自由解释。因为自由解释很容易听起来合理，但不一定真实反映执行过程。工程上比较稳的做法，是把计划、关键证据、工具结果和最终输出做成可回放结构，让人能快速知道“它做了什么、为什么、基于什么”。成熟系统里的解释能力更像审计视图，而不是模型作文。

**高分回答**
我不会暴露或依赖模型的隐藏思维过程，而是记录可验证的决策工件：用户目标与约束、计划版本、每次工具调用的结构化意图、参数摘要、授权决策、工具结果 ID、关键证据和最终副作用。最终输出中的每个关键结论绑定证据或执行结果，失败则指出停在哪个阶段、哪些事实仍未知。

解释按受众分层：用户看到简洁的步骤、来源和可执行结果；审批人看到差异、风险与回滚；运维和审计人员看到带时间、版本、trace_id 和策略决策的完整轨迹。自然语言摘要只能由这些事实生成，不能替代事实日志。日志中的敏感参数、凭证和原始个人数据需要脱敏与权限隔离，解释能力不能制造新的泄漏。

通过轨迹回放验证工具调用与实际副作用是否一致，用主张—证据抽检验证解释忠实度，并记录缺失事件率、回放成功率、问题定位时间和审计通过率。若模型声称“已完成”但没有对应成功事件，服务端必须拒绝该状态；这比让模型写更长的理由更能提升可信度。

**追问点答案**  
- 为什么自然语言解释不一定可靠？  
因为它可能是事后生成的合理化表述。  
- 最关键的可解释信息是什么？  
是目标、授权决策、工具动作、真实结果、证据和最终结论的可回放关系，而不是不可验证的自然语言自述。
- 可解释性为什么重要？  
因为高风险任务必须能复盘和追责。

**原理补充**  
可解释性的核心，是让人能验证输出依据，而不是只看一个听起来合理的说法。

**架构扩展**  
成熟平台会把可解释性能力与审计、审批和复盘工具打通，这样出问题时可以快速定位是检索错、规划错还是工具执行错。

**工程指标**  
- 关键步骤可回放率：看解释完整度。  
- 人工复盘耗时：看解释效率。  
- 解释与真实轨迹一致率：看可信度。  
- 高风险任务带证据输出比例：看治理质量。  
- 审计复核通过率：看可用性。

**证据链**  
- 执行轨迹  
- 工具调用日志  
- 引用与证据列表  
- 审批/复盘记录

**低风险优化路径**  
1. 先保留结构化执行轨迹  
2. 再给结果绑定关键证据  
3. 最后做面向人类的审计视图

**易错点**  
- 只让模型口头解释  
- 不保留执行轨迹  
- 解释与证据断裂

## 13. 多 Agent 协作如何拆职责？

**核心结论**  
多 Agent 协作要按目标和能力拆职责，让每个 Agent 的输入、输出和权限边界足够清晰，避免互相重叠和重复思考。

**标准答案**  
我拆多 Agent 职责时，通常不会按“一个 Agent 干一切”去拆，而是看任务里是否天然存在不同角色，比如规划、检索、执行、校验、汇总。因为多 Agent 一旦边界不清，很容易出现重复调用、互相推诿或者输出冲突。工程上比较稳的做法，是给每个 Agent 明确职责、工具白名单和交付格式，并尽量让它们通过结构化结果协作，而不是自由聊天。成熟团队会先证明单 Agent 不够，再引入多 Agent，而不是把多 Agent 当默认高级形态。

**高分回答**
我只在任务存在可独立评测的专业子域、不同权限边界，或需要并行处理时拆多 Agent；上下文太长本身未必是理由，先尝试单 Agent 加工具和固定工作流。常见模式是一个确定性 orchestrator 管理任务图，规划、检索、领域分析和验证角色各自只有最小工具权限。执行高风险动作不应由多个 Agent 投票决定，而仍由策略服务和人工审批控制。

每个角色有明确输入 schema、输出 schema、完成定义、预算和错误语义，通过共享任务存储或消息传递交接，不自由互聊。交付物携带来源、置信度和版本；协调器负责依赖、超时、去重和冲突升级。验证 Agent 不能只重复生成者的上下文和提示，否则会形成相关性错误；关键结果优先用规则、工具或独立证据验证。

先以单 Agent 为基线，在相同任务集上比较多 Agent 的端到端成功率、子任务失败传播、冲突率、协作步数、P95、成本和人工返修率。若只是增加 token 和协调失败，没有带来分片并行、专业质量或权限隔离收益，就回退单 Agent。多 Agent 的价值来自职责与故障域分离，不来自角色数量。

**追问点答案**  
- 为什么多 Agent 不是越多越好？  
因为协作成本和调试难度会快速上升。  
- 什么时候值得上多 Agent？  
当子任务能独立定义与评测、需要不同权限或可并行执行，且相对单 Agent 基线有可量化收益时。
- 协作输出为什么要结构化？  
因为结构化更容易校验和串接。

**原理补充**  
多 Agent 协作的本质，是把复杂任务切成职责清晰、可单独治理的子问题。

**架构扩展**  
成熟平台常见的模式是“主控 Agent + 专家 Agent”，而不是多 Agent 自由对话。这样更容易收敛，也更利于权限和成本治理。

**工程指标**  
- 子任务完成率：看拆分质量。  
- Agent 间冲突率：看职责清晰度。  
- 端到端任务成功率：看整体收益。  
- 平均协作步数：看协同成本。  
- 调试和回归成本：看维护压力。

**证据链**  
- Agent 职责图  
- 输入输出契约  
- 失败协作样本  
- 成本数据

**低风险优化路径**  
1. 先证明单 Agent 的边界已不足  
2. 再拆最清晰的两个角色  
3. 最后补协作协议和审计

**易错点**  
- 为了复杂而复杂地上多 Agent  
- 职责边界重叠  
- Agent 之间自由对话无约束

## 14. Agent 的成本上限如何控制？

**核心结论**  
控制 Agent 成本上限要同时限制步数、工具次数、上下文增长和模型档位，并在任务级设置预算止损点。

**标准答案**  
我控制 Agent 成本时，通常不会只看单次模型调用价格，而是看整个任务生命周期的总成本，因为 Agent 的主要风险是“每一步都不贵，但最后跑了太多步”。工程上比较稳的做法，是给任务设 token、调用次数、工具预算和时间上限，并在超预算时触发降级、转人工或直接终止。成熟团队还会按任务价值分层，高价值任务允许更高预算，低价值任务必须更快收敛。这样做能避免 PoC 阶段看起来很酷，上线后成本完全失控。

**高分回答**
成本上限要在执行前分配，并在运行时逐步扣减。预算至少包括输入输出 token、模型调用次数、工具费用、检索/重排次数、墙钟时间和最大步骤；根据任务价值、风险和租户套餐设硬上限，同时保留平台与租户级日/月配额。规划器看到的是剩余预算和可选能力价格，不能自行修改预算。

运行时先执行低成本、高信息增益动作；重复查询、无进展步骤和上下文膨胀会被预算器阻断。上下文使用结构化状态与外部证据引用压缩，确定性步骤不用大模型，昂贵模型或工具只在复杂度分类与质量门槛需要时触发。接近上限时按预定义策略缩小搜索、跳过可选重排、返回部分结果或转人工，但不得降低权限校验、审批和审计。

成本指标以“每个成功任务成本”为核心，同时看 P50/P95、预算耗尽率、无效步骤成本、降级后成功率和任务价值/成本比。预算要用生产轨迹分布校准，而不是拍脑袋；极少数长尾任务可单独提额审批。若大量任务耗尽预算，应修复循环、工具失败或任务拆分，而不是持续提高上限。

**追问点答案**  
- 为什么 Agent 成本容易失控？  
因为它的成本是动态累积的，不是单步固定。  
- 降级方式有哪些？  
还可返回带检查点的部分结果、跳过非关键精排或等待用户补充；安全校验、权限和审批不能作为成本降级项。
- 为什么要按任务价值分层预算？  
因为不同任务的投入产出不同。

**原理补充**  
成本控制的本质，是给开放式决策链路设置经营边界。

**架构扩展**  
成熟平台通常会把成本控制和任务编排绑在一起，让 Agent 从设计上知道自己“最多可以花多少资源解决这个问题”。

**工程指标**  
- 单任务总成本：看核心控制目标。  
- 步数长尾分布：看失控风险。  
- 超预算终止率：看止损触发。  
- 降级后完成率：看经济性与效果平衡。  
- 成本异常告警数：看治理及时性。

**证据链**  
- 成本日志  
- 步数统计  
- 预算配置  
- 超预算案例

**低风险优化路径**  
1. 先建立任务预算模型  
2. 再限制步数和工具次数  
3. 最后补降级和告警

**易错点**  
- 只看单步调用价格  
- 无预算上限  
- 成本异常没有自动止损

## 15. 计划错误时如何纠偏？

**核心结论**  
计划错误纠偏的关键，是尽早发现偏离目标的信号，并让系统能回到上一个稳定检查点，而不是一路错到底。

**标准答案**  
我做计划纠偏时，通常会在关键步骤加入检查点，比如目标是否仍成立、当前证据是否支持下一步、是否已经偏离用户约束。因为 Agent 最大的问题不是犯错一次，而是沿着错误计划不断放大。工程上比较稳的做法，是把阶段性验证、异常分支和人工确认点提前埋好，让系统在发现偏离时能重规划、回滚部分状态或直接转人工。成熟团队不会把纠偏完全留给模型自我反思，而会用系统信号帮助它更早收敛。

**高分回答**
我会把计划写成带前置条件、预期观察、成功判据、预算和副作用等级的结构化步骤，而不是一段自然语言。每步执行后由系统检查：是否获得预期信息、目标与用户约束是否仍满足、状态是否前进、剩余预算能否完成。工具错误、证据冲突、状态无进展或关键假设被推翻都会触发重新规划，而不是让模型凭“自我反思”决定一切。

纠偏优先从最近无副作用检查点重算未执行部分，已确认事实继续复用但标注版本；已产生副作用时先查询真实状态，再按补偿契约处理，不能假装回滚。新计划必须说明变更原因、废弃哪些假设和新增风险，并继续受原权限与预算限制。连续两次重规划仍无进展、涉及高风险动作或补偿失败时转人工。

测试通过故意注入错误工具结果、过期证据、用户约束变化和部分成功，观察偏离检测召回率、误触发率、发现前浪费步数、纠偏后成功率、重复副作用和人工接管时延。计划版本、触发信号和状态差异要可回放，才能区分最初规划错误、环境变化和执行故障。

**追问点答案**  
- 为什么 Agent 会一路错到底？  
因为早期错误可能污染后续所有决策。  
- 检查点最适合放哪？  
还应放在外部事实获取后、阶段完成处和不可逆动作前，检查点要持久化状态与证据版本，而不是只保存对话摘要。
- 纠偏一定要自动吗？  
不一定，高风险场景应允许人工接管。

**原理补充**  
纠偏的本质，是让错误从“持续扩散”变成“局部收敛”。

**架构扩展**  
成熟 Agent 系统通常会把计划和执行解耦，这样才能在执行中发现偏离后重新计划，而不是整条链路推倒重来。

**工程指标**  
- 计划纠偏成功率：看恢复能力。  
- 偏离后损失步数：看止损能力。  
- 人工接管率：看复杂度。  
- 错误计划放大导致的失败数：看风险。  
- 重规划触发率：看动态性。

**证据链**  
- 执行轨迹  
- 检查点日志  
- 偏离样本  
- 人工接管记录

**低风险优化路径**  
1. 先定义关键检查点  
2. 再加重规划和回退能力  
3. 最后补人工接管入口

**易错点**  
- 没有阶段验证  
- 错误发生后无回退路径  
- 高风险任务强制自动纠偏

## 16. 工具超时与降级怎么处理？

**核心结论**  
工具超时处理的重点，是让 Agent 区分“可重试、可替代、可跳过、必须终止”四类动作，并为每类准备不同降级路径。

**标准答案**  
我处理工具超时时，通常先按工具角色分层。像搜索、检索、推荐这类辅助工具，超时后往往可以降级为更简单结果、缓存结果或跳过；但像支付、审批、提交这类关键动作，超时则必须谨慎，优先查询状态而不是盲重试。工程上比较稳的做法，是给每个工具定义超时阈值、重试策略和降级方式，并要求 Agent 在超时后显式说明当前状态，而不是假装工具成功了。成熟系统会把工具超时视为分布式故障治理，而不是 Agent 的小异常。

**高分回答**
每个工具都要有调用契约：延迟预算、只读或副作用等级、幂等能力、状态查询接口、可替代方案和数据新鲜度要求。端到端 deadline 向下游传播，单工具超时必须小于剩余任务预算，并预留收尾时间；不能让各层独立设置超时后叠加成失控长尾。并行工具使用有界并发和取消传播，任务结束后及时终止无用请求。

只读辅助工具超时可在质量标记下使用版本兼容的缓存、备用数据源或跳过；关键事实缺失则拒答。副作用工具超时属于“结果未知”，先以 request_id 或幂等键查询状态，确认未执行后才能重试；无法确认时进入待核对状态或人工处理。备用工具必须具有相同权限、语义和审计能力，否则所谓降级可能改变业务含义或扩大数据边界。

运行时区分超时、限流、熔断和业务拒绝，采用有限重试、jitter、断路器与每依赖隔离舱，防止故障扩散。评测注入慢响应、成功但响应丢失、备用源陈旧和半成功，观察 P99、超时恢复率、未知状态收敛时间、重复副作用、降级质量损失和任务完成率。对用户明确展示完成、部分完成或待确认，绝不把超时包装成成功。

**追问点答案**  
- 为什么关键写操作超时不能盲重试？  
因为可能已经执行成功，只是结果没返回。  
- 什么工具最适合降级？  
只读且有语义等价备用源的辅助工具；涉及关键事实或副作用时，宁可返回待确认也不能用低可信结果冒充成功。
- 为什么要显式暴露当前状态？  
因为人和系统都需要知道结果是否可信。

**原理补充**  
超时治理的本质，是把“不知道是否成功”这类灰状态处理清楚。

**架构扩展**  
成熟平台通常会给工具打上“只读/写入/高风险”标签，超时和降级策略也跟着标签走，这比每个 Agent 自己猜更稳。

**工程指标**  
- 工具超时率：看稳定性。  
- 降级触发成功率：看韧性。  
- 因超时误重试导致的副作用数：看风险。  
- 状态不明事件数：看治理成熟度。  
- 降级后任务完成率：看实际效果。

**证据链**  
- 工具分类  
- 超时日志  
- 状态查询记录  
- 降级案例

**低风险优化路径**  
1. 先按工具风险等级分层  
2. 再定义超时和降级策略  
3. 最后补状态查询和审计

**易错点**  
- 所有工具统一超时处理  
- 写操作超时直接重试  
- 超时后状态不透明

## 17. 如何评估 Agent 任务成功率？

**核心结论**  
Agent 成功率不能只看“有没有返回结果”，要看目标是否达成、步骤是否合规、成本是否可接受、是否留下副作用问题。

**标准答案**  
我评估 Agent 任务成功率时，通常会把“成功”定义得比普通问答更严格。因为 Agent 不是只回答一句话，它可能涉及检索、计划、工具调用、执行和确认。工程上比较稳的做法，是为每类任务定义明确成功标准，比如任务是否完成、结果是否正确、工具调用是否合法、是否超预算、是否需要人工补救。成熟团队会把自动评测和人工抽检结合起来，避免只用单一指标掩盖问题。

**高分回答**
成功必须按任务类型定义可验证的完成判据。例如“创建工单”不仅要生成文本，还要正确选择客户、写入一次、返回真实工单 ID 且符合权限；“故障分析”要覆盖关键证据、给出可执行结论并避免未经授权的变更。我会把结果分为完全成功、可接受的部分成功、安全拒绝、可恢复失败和有害失败，避免二元指标掩盖问题。

评测集来自真实任务分布和线上失败，包含多轮澄清、工具异常、权限受限、注入、无解和副作用案例。自动检查确定性状态、schema、工具日志、证据和预算，规则无法判定的开放式质量由盲审或校准后的评审器评估；高风险样本必须人工抽检。主指标是端到端任务成功率，护栏包括越权/有害动作、重复副作用、人工返修、P95 完成时间和每成功任务成本。

还要看分阶段漏斗：意图理解、计划、工具选择、参数、执行、验证分别成功多少，从而定位瓶颈。发布时固定回归集、影子流量和小流量 canary，对各任务切片设最低门槛；总体均值提升但高风险切片退化不能上线。线上用户纠正、取消、人工接管和后续返工回流为新回归样本。

**追问点答案**  
- 为什么有结果不等于成功？  
因为结果可能错、越权、超预算或留下脏状态。  
- 自动评测能完全替代人工吗？  
不能；确定性副作用可自动验真，开放式结果需要校准评审，高风险与新型失败仍要人工抽检和仲裁。
- 成功率为什么要分任务类型？  
因为不同任务的完成标准不同。

**原理补充**  
任务成功率的本质，是把 Agent 的“目标完成质量”变成可量化指标。

**架构扩展**  
成熟团队会把成功率拆成主指标和护栏指标，比如主指标看完成率，护栏指标看越权、成本和副作用，这样更接近真实生产价值。

**工程指标**  
- 任务完成率：看主目标。  
- 结果正确率：看质量。  
- 合规执行率：看安全性。  
- 超预算比例：看经济性。  
- 人工返修率：看真实可用性。

**证据链**  
- 任务评测集  
- 审计日志  
- 成本数据  
- 人工修正记录

**低风险优化路径**  
1. 先定义各任务成功标准  
2. 再建立自动评测和抽检  
3. 最后按失败类型持续改进

**易错点**  
- 只有“是否返回结果”一个指标  
- 不看成本和副作用  
- 不区分任务类型

## 18. 如何防止 prompt 注入影响工具调用？

**核心结论**  
防止 prompt 注入的重点，是把外部输入视为不可信数据，并用权限隔离、工具白名单、内容净化和二次校验保护执行链路。

**标准答案**  
我防 prompt 注入时，通常不会把外部检索内容、网页内容或用户上传文档直接当成“可信指令”。因为注入攻击最典型的做法，就是把恶意命令藏在看似普通内容里，诱导 Agent 越权调用工具。工程上比较稳的做法，是把系统指令和外部内容明确分层，限制外部内容不能直接改变工具权限；同时对高风险动作加白名单、参数校验和人工审批。成熟团队会把注入防护当成整条 Agent 执行链路的安全问题，而不是只靠一条“忽略恶意提示”的 Prompt。

**高分回答**
Prompt 注入难以仅靠文本识别彻底消除，所以设计目标是即使模型被诱导，攻击者也无法获得高价值副作用。系统策略、用户指令和外部网页/邮件/文档必须在数据模型中分层；外部内容只作为带来源的不可信数据，不能改变工具列表、权限、审批或系统目标。解析器限制内容大小和类型，并隔离其中的链接、隐藏文本和指令式片段，但过滤只是辅助层。

真正的边界在执行端：最小权限令牌、网络与域名 allowlist、参数化工具、敏感数据流向策略、写操作 dry-run 和人工审批。Agent 不能读取凭证，也不能将检索到的秘密发送到外部目标；当工具参数来自不可信内容时进行来源标记和策略检查。工具输出同样是不可信 observation，不能因为来自内部连接器就升级为指令。

我会构建直接注入、间接网页注入、跨工具数据外传、编码混淆、审批绕过和多轮持久化攻击集，验证有害工具调用成功率、敏感数据泄漏率、误拦截率和检测/撤销时间。审计记录内容来源、模型意图、策略拦截和实际执行；出现异常可立即撤销工具能力。即使检测漏掉文本，最小权限和副作用门禁仍应阻断攻击链。

**追问点答案**  
- 为什么外部内容默认不可信？  
因为网页、文档和用户输入都可能包含恶意指令。  
- 工具白名单为什么重要？  
因为它能把即使被诱导后的执行范围也限制住。  
- Prompt 注入能完全消除吗？  
不能承诺完全消除，因此必须假设模型可能受影响，用执行端授权、数据流控制和审批限制攻击半径。

**原理补充**  
Prompt 注入问题本质上是“数据和指令边界被混淆”导致的控制权劫持。

**架构扩展**  
成熟平台会把检索内容、用户内容、系统策略和工具权限分别治理，避免任一层被注入后直接拿到执行能力。

**工程指标**  
- 注入攻击成功率：看核心风险。  
- 越权调用拦截率：看防护效果。  
- 高风险动作审批覆盖率：看止损能力。  
- 对抗样本通过率：看鲁棒性。  
- 安全回归问题数：看长期稳定性。

**证据链**  
- 红队测试报告  
- 工具审计日志  
- 注入样本库  
- 权限策略记录

**低风险优化路径**  
1. 先把外部内容全部视作不可信  
2. 再用白名单和审批隔离高风险工具  
3. 最后持续红队和回归

**易错点**  
- 把检索内容直接当系统指令  
- 工具权限过宽  
- 只靠 Prompt 防注入

## 19. Agent 日志与审计怎么做？

**核心结论**  
Agent 日志和审计要同时覆盖目标、输入、工具调用、证据、权限上下文和最终动作，做到可回放、可追责、可定位。

**标准答案**  
我做 Agent 日志与审计时，通常会先明确哪些信息是必须保留的：任务目标、用户上下文、关键 Prompt 版本、调用过的工具、输入输出、错误信息、审批节点和最终结果。因为 Agent 的问题往往不是一步错，而是多步链路叠加，所以没有全链路日志几乎没法复盘。工程上比较稳的做法，是对高风险任务保留更细颗粒度的审计，对一般任务保留摘要级日志，并注意敏感信息脱敏。成熟系统里的审计不是为了“多留点日志”，而是为了出事后能快速回答“发生了什么、为什么会这样、还会不会再发生”。

**高分回答**
我会用统一 task_id、trace_id 和 step_id 串起用户请求、计划版本、模型/提示/工具版本、状态转移、授权决策、审批、工具调用、幂等键、证据 ID、成本和最终副作用。日志记录结构化意图和可验证事件，不依赖模型自述；每次写操作保存请求摘要、实际执行主体、目标资源、前后差异与结果状态，才能回答谁授权、系统为何允许、实际改变了什么。

调试日志与合规审计分层：调试可采样并较短保留，审计事件不可随意修改、按风险设留存期和访问审批。Prompt、工具输出可能含凭证、个人数据和商业秘密，采集前先字段级脱敏或令牌化，原文仅在确有必要的受控存储中保留；日志访问本身也要审计。内容版本或哈希保证回放时知道当时看到的材料，而不是读取已经变化的页面。

通过日志 schema 契约、事件数量对账和故障注入验证完整性，定期抽样回放高风险任务。指标包括关键事件完整率、版本可定位率、回放成功率、敏感字段泄漏数、审计查询时延和事件落盘延迟。日志不可用时，高风险副作用应暂停或进入保守模式，因为无法审计本身就是控制失效。

**追问点答案**  
- 为什么 Agent 审计比普通接口日志更复杂？  
因为它通常涉及多步推理和多工具交互。  
- 为什么要记录 Prompt 版本？  
因为行为变化可能来自 Prompt 而不是模型。  
- 审计和调试日志一样吗？  
不一样；调试日志可采样和短期保留，审计事件要求完整、受保护、可证明且其访问行为也被审计。

**原理补充**  
审计的本质，是把开放式 Agent 行为转成可验证的系统事件序列。

**架构扩展**  
成熟平台通常会把 Agent 日志与审批、权限、计费、回放工具整合在一起，这样平台治理能力才真正完整。

**工程指标**  
- 全链路日志完整率：看覆盖度。  
- 高风险任务可回放率：看治理质量.  
- 敏感字段脱敏率：看合规性。  
- 问题定位平均耗时：看可用性。  
- 审计缺失事件数：看风险。

**证据链**  
- 日志 schema  
- 审计平台截图/记录  
- 回放结果  
- 脱敏策略

**低风险优化路径**  
1. 先统一 Agent 审计字段  
2. 再建立高风险任务回放能力  
3. 最后补敏感信息脱敏和留存策略

**易错点**  
- 只记录最终结果不记录过程  
- 不记录版本和权限上下文  
- 审计日志含敏感信息无脱敏

## 20. 如何把 Agent 从 PoC 做到生产可用？

::: tip L2 深度训练
用状态机、工具契约、预算、审计与任务评测完成生产闭环：[Agent 生产化与工具安全](/deep-dives/ai-architecture/02-agent-production-safety)
:::

**核心结论**  
把 Agent 从 PoC 做到生产，关键是补齐约束、评测、权限、观测、成本和回退能力，让它从“能跑”变成“能控、能审、能收敛”。

**标准答案**  
我把 Agent 从 PoC 推到生产时，通常不会直接扩大流量，而是先补齐几类生产能力：任务成功标准、工具权限控制、失败重试与人工兜底、日志审计、成本预算、回归评测和灰度发布。因为 PoC 阶段最容易忽略的，恰恰是生产里最要命的部分。工程上比较稳的做法，是先把 Agent 限制在小范围、低爆炸半径场景，用真实指标证明价值，再逐步放大。成熟团队不会因为 PoC 演示顺滑就认为它已经适合上线，而会把生产化当成一次系统工程。

**高分回答**
我先收窄一个价值明确、任务边界清晰且失败可恢复的场景，定义基线人工流程、端到端成功标准、SLO、每成功任务成本和绝不能发生的安全事件。PoC 只证明能力可行；生产门槛还包括真实分布评测、工具契约、状态持久化、幂等与超时、最小权限、审批、预算、审计、人工接管、版本管理和降级回滚。

发布路径是离线黄金集与红队、历史轨迹回放、影子执行、内部只读试点、低风险 canary，再逐级开放可逆和高风险能力。影子阶段不产生副作用；写能力开始时使用 dry-run 和严格审批。每次发布固定模型、提示、schema、策略和工具版本，设置任务成功率、越权事件、重复副作用、P99、人工返修率与成本护栏，触发即暂停并恢复上一稳定版本。

组织上明确工具、策略、评测、值班和业务结果的负责人，建立事故响应、kill switch、权限撤销、死信处理和定期演练。只有在样本量充分的真实流量中持续达到质量与经济性门槛，才扩大自动化范围。平台化发生在多个场景证明需要共同控制面之后，不能先建通用 Agent 平台再寻找问题。

**追问点答案**  
- 为什么 PoC 和生产差这么多？  
因为 PoC 关注能不能做，生产关注能不能长期稳定做。  
- 生产化最重要的一类能力是什么？  
是可验证、可限制、可恢复的控制闭环：成功标准、执行端权限、状态与幂等、审计、人工接管和快速回滚缺一不可。
- 为什么要先低风险场景试点？  
因为真实任务复杂度和失败模式远比 demo 多。

**原理补充**  
生产化的本质，是把不确定行为纳入工程治理框架，而不是单纯追求更强能力。

**架构扩展**  
成熟平台通常把 Agent 生产化分成几个阶段：PoC 验证、内测、低风险业务试点、灰度上线、平台化复用。这样每一步都有明确验收标准，而不是一次跨越式上线。

**工程指标**  
- 试点任务成功率：看基本可用性。  
- 人工兜底率：看稳定性。  
- 审计覆盖率：看治理完整度。  
- 单任务成本：看规模化可行性。  
- 版本回归通过率：看持续迭代能力。

**证据链**  
- 试点报告  
- 评测和回归数据  
- 审计记录  
- 灰度发布记录

**低风险优化路径**  
1. 先补治理和评测能力  
2. 再在低风险场景试点  
3. 最后逐步灰度和平台化

**易错点**  
- PoC 成功就直接全量上线  
- 缺权限、审计和回退  
- 没有清晰生产验收标准
