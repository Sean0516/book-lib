# AI模块1 - LLM基础与模型选型 - 标准答案库

## 1. 你如何做基础模型选型？

::: tip L2 深度训练
从任务、风险、SLA 和单次成功成本建立选择矩阵：[模型路由、SLA 与成本](/deep-dives/ai-architecture/03-model-routing-cost)
:::

**核心结论**  
基础模型选型的关键，不是追最强模型，而是根据业务任务、质量要求、延迟预算、成本上限和治理要求，选当前阶段最合适的模型组合。

**标准答案**  
我做基础模型选型时，通常先看业务目标到底是什么，是通用问答、代码生成、RAG 问答、工具调用，还是复杂推理。因为不同任务对指令跟随、长上下文、结构化输出和幻觉容忍度的要求差异很大。比较稳的做法，是先把候选模型缩小到几类，再从质量、延迟、成本、可用性和安全能力几个维度做压测和评测，而不是只看榜单分数。工程上我会优先给核心场景定义最低可接受标准，比如任务成功率、TTFT、TP99 和单请求成本，再决定用单模型还是多模型路由。成熟架构师谈模型选型，不会停在“这个模型更强”，而会继续讲为什么它适合当前业务约束。

**高分回答**
我先把“模型更好”改成业务契约：任务类型和输入分布、可接受错误、证据/工具要求、TTFT 与端到端 P99、可用性、数据边界和单次成功成本。例如客服 RAG 要求有据回答率≥95%、错误引用≤1%、P95<3s，而不是追通用榜单第一。用真实线上分层样本建立黄金集，覆盖高频、长尾、拒答、对抗和多语言；盲评候选模型在完整 Prompt/RAG/工具链中的任务成功率，并同时压测并发、限流、结构化输出、上下文退化和供应商故障。成本按成功请求计算：低单价但重试/人工复核多可能更贵。先选满足硬门槛的最简单单模型，只有任务质量分层、成本或容灾收益被数据证明时引入路由。采用版本固定、灰度、影子流量与回滚，模型升级重新跑回归。最终记录决策矩阵、未选方案和触发复审条件，持续看线上漂移、失败样本、人工接管和单位业务结果成本。

**追问点答案**  
- 为什么不能只看公开榜单？  
榜单的任务、语言、Prompt、评分与线上分布不同，还不覆盖限流、延迟、安全和工具链；它只能用于初筛，最终必须以版本化业务评测和生产影子证据决策。
- 模型选型为什么要先定义任务成功标准？  
因为没有标准就无法比较“够不够用”。  
- 一开始就要上多模型吗？  
不一定，很多场景先用单模型更容易验证和治理。

**原理补充**  
模型选型的本质，是把“模型能力”映射到“业务任务成功率”，再用工程约束筛掉不适合的候选项。

**架构扩展**  
真实项目里，模型选型最常见的误区是先拍脑袋决定供应商，再倒推业务要配合它。成熟团队会先用代表性样本做任务级评测，验证在真实 Prompt、真实上下文、真实链路下的表现，再决定模型路线。

**工程指标**  
- 任务成功率：看业务可用性。  
- TTFT/TP99：看交互体验。  
- 单请求成本：看规模化可行性。  
- 输出结构化成功率：看工程稳定性。  
- 模型可用性：看供应商稳定度。

**证据链**  
- 代表性评测集  
- 多模型对比实验结果  
- 压测报告  
- 成本测算表

**低风险优化路径**  
1. 先定义业务成功标准  
2. 再小范围评测 2 到 3 个候选模型  
3. 最后按结果决定是否引入路由和兜底模型

**易错点**  
- 只看榜单和宣传材料  
- 没有任务级评测  
- 一上来就做复杂多模型架构

## 2. 闭源模型和开源模型如何取舍？

**核心结论**  
闭源模型通常在综合能力和上线速度上更有优势，开源模型则在可控性、定制化和长期成本上更有空间，关键看业务阶段和治理需求。

**标准答案**  
我在闭源和开源模型之间做取舍时，通常先看目标是快速验证业务价值，还是已经进入大规模、强合规、强定制阶段。如果是 PoC 或上线初期，闭源模型往往能更快给到稳定能力，减少训练、部署和调优成本；但如果业务对数据边界、定制能力、部署位置或者长期成本特别敏感，开源模型的吸引力会明显上升。之所以不能简单说“闭源更强”或“开源更自由”，是因为两边的代价结构完全不同。工程上我会同时比较能力、运维复杂度、供应商锁定风险和总拥有成本，而不是只看 API 单价。成熟架构师会把这件事讲成阶段选择，而不是立场选择。

**高分回答**
我按能力门槛、控制面和总拥有成本选，而非意识形态。闭源 API 通常能更快验证价值，平台负责推理、弹性和升级，但要接受数据出域、限额/价格/模型行为变化、审计透明度和退出风险；开放权重可本地部署、定制和固定版本，却需承担 GPU 容量、量化/服务、补丁、安全评估、值班和人才。先用同一业务黄金集比较质量、TTFT/P99、吞吐、上下文与工具能力，再算 12–36 个月 TCO：API token、网络、缓存、人工复核，对比 GPU 利用率、冗余、工程与运维。合规先明确数据能否发送、训练保留、地域、删除和审计；开放许可证也要检查商用与衍生限制。架构通过模型适配层、可移植 Prompt/评测和数据导出降低锁定，但不追求虚假的无差别切换。可采用敏感/稳定高量任务自建、复杂长尾闭源的组合，并对两边都设计故障兜底、灰度升级和退出演练。

**追问点答案**  
- 开源模型一定更便宜吗？  
只有稳定负载、较高 GPU 利用率和足够运维能力时才可能更省；需计入峰值冗余、闲置、量化质量损失、升级、安全、值班与机会成本，并比较“单次成功任务”而非每 token。
- 闭源模型最大风险是什么？  
供应商锁定、价格波动和能力变更不完全可控。  
- 可以混用吗？  
可以，很多团队会用闭源跑主流程、开源做特定能力或本地部署。

**原理补充**  
这类选择本质上是在“能力现成度”和“可控性”之间做平衡。

**架构扩展**  
成熟团队很少把闭源和开源看成二元对立，而是会保留迁移和混用空间。比如对高价值主链路先用闭源快速上线，同时逐步积累数据和评测体系，为后续开源替代留出可能性。

**工程指标**  
- 模型能力差距：看业务效果。  
- 单位请求总成本：看经济性。  
- 上线和维护人力投入：看组织负担。  
- 数据出域比例：看合规风险。  
- 供应商依赖度：看长期可控性。

**证据链**  
- 成本测算  
- 合规要求清单  
- 真实评测结果  
- 运维复杂度评估

**低风险优化路径**  
1. 先按阶段明确优先级是速度还是可控性  
2. 再做小范围双路评测  
3. 最后保留切换和混用能力

**易错点**  
- 用意识形态替代选型  
- 只看 API 价格不看总成本  
- 不评估退出路径

## 3. 长上下文能力在业务中如何评估？

**核心结论**  
长上下文能力不能只看支持多少 token，更要看在长输入下是否还能稳定检索关键信息、保持指令一致并控制延迟和成本。

**标准答案**  
我评估长上下文能力时，通常不会只看模型标称的上下文窗口，而是会用真实长文档、长对话和多段引用任务去测它的有效表现。因为很多模型虽然能“塞进去”，但在超长输入下会出现前后信息利用不均、关键信息遗漏或者延迟成本急剧上涨的问题。工程上比较稳的做法，是先定义几个典型任务，比如长合同问答、多轮客服总结、长代码库分析，再测召回准确率、结构化输出稳定性和成本曲线。成熟团队不会把长上下文当成万用解法，而会同时评估 RAG、摘要分层和分段推理是不是更划算。

**高分回答**
我区分标称窗口、可接受输入和有效上下文。按真实任务构造不同长度与位置的数据集：单针/多针关键信息放在开头、中间、末尾，加入相似干扰、跨段推理、时间顺序和“证据不存在”样本；测答案正确、证据定位、遗漏/错误引用和拒答，而非只看能否返回。每个长度档记录 TTFT、总时延、token 成本、超限/截断和并发吞吐，画质量—长度—成本曲线，识别 lost-in-the-middle 与长输入导致的指令漂移。与 RAG、层级摘要、map-reduce 和混合方案在同一集上比较；长上下文适合需要全局关系且材料规模受控，RAG适合大语料更新与可溯源，但有召回风险。输入先去重、按权限过滤，摘要要保留来源并评估信息损失。生产监控长度分桶成功率和成本，达到退化点自动检索/分段或请求澄清，不能静默截断。

**追问点答案**  
- 为什么上下文窗口大不等于效果好？  
因为模型未必能稳定利用长输入里的关键信息。  
- 长上下文和 RAG 是替代关系吗？  
长上下文减少检索漏召回但成本高且会注意力退化；RAG扩展知识规模和更新性却依赖召回。常见组合是先检索缩小候选，再把相关完整章节放入较长上下文并引用验证。
- 最该测什么？  
真实任务下的信息命中率、延迟和成本。

**原理补充**  
长上下文的本质问题，不是“能否接收输入”，而是“是否还能高质量使用输入”。

**架构扩展**  
很多团队最开始会把所有文档直接塞进模型，后面很快发现成本高、性能不稳、效果还不一定最好。成熟方案往往会先做检索或摘要压缩，再把最相关信息送进上下文。

**工程指标**  
- 长输入任务成功率：看真实效果。  
- 关键片段命中率：看信息利用能力。  
- TTFT/TP99：看时延代价。  
- 单请求 token 消耗：看成本。  
- 长输入退化幅度：看性能边界。

**证据链**  
- 长文任务评测集  
- 命中与遗漏案例  
- 成本和延迟曲线  
- 对比试验结果

**低风险优化路径**  
1. 先用典型长任务做评测  
2. 再比较长上下文、RAG 和摘要链路  
3. 最后按收益选择真正需要的方案

**易错点**  
- 只看上下文窗口大小  
- 不测长输入退化  
- 用超长上下文替代所有检索设计

## 4. 模型幻觉风险如何量化？

**核心结论**  
幻觉风险不能只凭主观感觉判断，要结合任务类型定义“错误但看起来像真”的标准，再用样本评测和线上反馈做量化。

**标准答案**  
我量化幻觉风险时，通常先定义什么叫幻觉，因为不同任务的判断标准不一样。像事实问答、合同引用、医疗或金融建议这类任务，幻觉往往是生成了没有依据或与事实冲突的信息；而在创意写作里，同样的输出不一定算问题。工程上比较稳的做法，是先构建带标准答案或带证据链的数据集，统计无依据回答率、错误引用率、事实冲突率，再结合线上人工抽检和用户纠错反馈持续跟踪。成熟团队不会把“模型有时会编”当成抽象担忧，而是会把幻觉拆成任务级风险指标。

**高分回答**
我先建立任务级错误分类：与权威事实冲突、证据不支持、伪造引用/实体、遗漏关键限定，以及本应拒答却高置信作答；创意生成则不按同一标准。黄金集覆盖可回答、不可回答、冲突来源、过期事实和长尾，高风险样本按损失加权。评测拆成检索召回、证据充分性和生成忠实度，人工盲标配清晰 rubric 与一致性抽检；自动 judge 只做规模化筛查，需用人工集校准偏差。指标至少报告 claim-level unsupported rate、错误引用、事实冲突、拒答 precision/recall 与严重度加权损失，不能用一个“幻觉率”掩盖分母。线上对引用做可点击校验，抽样高风险回答、收集用户纠错并回放；模型无法知道时明确不确定或转人工。RAG、约束输出和验证器能降低风险但不能消除，最终用事故/人工接管和业务决策错误验证保护是否有效。

**追问点答案**  
- 为什么要按任务类型定义幻觉？  
因为不同业务对错误的容忍度和定义不一样。  
- 幻觉能完全消除吗？  
不能保证为零；应按损失分层，降低发生概率、限制回答范围、提供证据与拒答/人工升级，并让错误可检测、可追踪和可恢复。
- 线上怎么发现幻觉？  
靠用户反馈、抽检、引用校验和样本回放。

**原理补充**  
幻觉本质上是模型在概率生成中超出证据边界做了高置信度输出。

**架构扩展**  
成熟产品不会只要求模型“少幻觉”，而是会通过 RAG、引用溯源、拒答策略和高风险问题拦截，把高代价错误控制在业务可接受范围内。

**工程指标**  
- 无依据回答率：看直接风险。  
- 错误引用率：看证据一致性。  
- 用户纠错率：看真实使用风险。  
- 高风险场景拒答命中率：看保护能力。  
- 幻觉导致投诉/事故数：看业务影响。

**证据链**  
- 标注评测集  
- 人工抽检记录  
- 引用校验结果  
- 用户反馈工单

**低风险优化路径**  
1. 先定义高风险任务的幻觉标准  
2. 再做离线评测和线上抽检  
3. 最后补证据链和拒答机制

**易错点**  
- 不区分任务类型统一谈幻觉  
- 只有主观感受没有量化指标  
- 发现问题后没有闭环改进

## 5. 推理成本如何估算到单请求？

**核心结论**  
单请求成本要按输入 token、输出 token、缓存命中、重试率和上下游附加成本一起算，而不是只看模型报价表。

**标准答案**  
我估算单请求成本时，通常会先把链路拆开，看一次请求到底包含哪些消耗：输入 token、输出 token、检索或 embedding 开销、重试和回退、缓存命中与否，必要时还要加网关和审计等附带成本。因为如果只拿官方价格乘 token 数，往往会低估真实成本。工程上比较稳的做法，是先按典型请求和高峰请求分别建样本，测平均、P95 和极端情况，再算到日活或业务量规模。成熟团队通常会把成本估算直接接到监控和限额系统里，不会等账单出来才发现超预算。

**高分回答**
API 模型的直接成本可写为 `Cin×输入token + Cout×输出token`，但生产单位成本还要加 embedding/检索、rerank、缓存、编排与网关、重试、fallback、内容审核、日志存储和人工复核；自建再加入 GPU 时长、利用率、峰值冗余、网络、工程和值班。按任务、租户、模型版本和长度分桶记录平均/P95/P99，区分缓存命中与未命中，并以成功业务结果为分母：`每次成功成本 = 总链路成本 / 成功且无需返工的任务数`。例如便宜模型 20% 请求升级到大模型且 5% 重试，不能用首调单价比较。容量预测用流量×长度分布×并发与供应商限额，做价格、命中率、输出膨胀和高峰敏感性分析。预算门禁包括 max output、上下文裁剪、请求/租户配额和成本异常告警，但不能通过截断导致质量下降；联合看任务成功、延迟和成本。账单与内部用量定期对账，版本升级重新校准估算。

**追问点答案**  
- 为什么平均成本不够？  
长上下文、长输出、重试与升级集中在尾部，平均值会隐藏单请求上限和峰值现金流；需看 P95/P99、最大预算、分任务分布以及成功结果成本。
- 缓存为什么会显著影响成本？  
因为重复问题和相似请求能大幅减少 token 消耗。  
- 重试要不要算进成本？  
要，生产环境里它经常不是小头。

**原理补充**  
成本估算的本质，是把模型调用从“感觉贵不贵”变成可预测、可控制的经营指标。

**架构扩展**  
成熟系统会区分实验成本和稳定运营成本。PoC 阶段可以接受贵一点，但一旦进入规模化，就必须把请求分层、缓存、降级和配额一起设计进去。

**工程指标**  
- 平均单请求成本：看整体经济性。  
- P95 成本：看长尾风险。  
- 缓存命中率：看优化收益。  
- 重试率：看隐藏成本。  
- 成本预算偏差：看估算质量。

**证据链**  
- Token 使用日志  
- 典型请求样本  
- 缓存统计  
- 成本看板

**低风险优化路径**  
1. 先做典型请求成本画像  
2. 再补缓存和配额  
3. 最后按业务线做成本归因和治理

**易错点**  
- 只看模型单价  
- 不算长尾和重试  
- 没有成本监控

## 6. 何时需要多模型路由？

::: tip L2 深度训练
设计可解释的路由信号、升级门槛和能力降级矩阵：[模型路由、SLA 与成本](/deep-dives/ai-architecture/03-model-routing-cost)
:::

**核心结论**  
多模型路由适合在任务差异大、成本压力明显、质量要求分层或需要供应商兜底时引入，而不是为了架构好看提前复杂化。

**标准答案**  
我判断要不要上多模型路由时，通常先看单模型是不是已经出现明显不匹配，比如简单请求用大模型太贵、复杂请求小模型又扛不住，或者对可用性要求高需要跨供应商兜底。之所以不能一开始就做路由，是因为它会增加评估、监控、回归测试和故障排查复杂度。比较稳的做法，是先把请求按任务复杂度分层，再让简单任务走便宜快模型，复杂任务走高质量模型，高风险链路保留兜底模型。成熟架构师谈路由时，不会只说“更灵活”，而会补上治理代价和触发条件。

**高分回答**
只有单模型在质量、成本或可用性上出现可量化分层时我才引入路由。例如分类/改写走小模型，复杂推理走强模型，供应商故障走能力受限兜底。先用黄金集标注任务、难度、风险和最小能力，路由信号只用调用前可得信息，如意图、长度、语言、租户/SLA；不能用模型回答正确与否这种事后特征。规则或轻量分类器先粗分，低置信/高风险直接升级；回答后可用结构/证据验证触发一次受预算控制的 escalation，避免无限级联。路由策略版本化、可解释，影子运行比较“总是强模型”基线，统计误降级与不必要升级。跨供应商 fallback 要处理 Prompt/schema/安全策略差异、限额相关性和幂等，故障时可能选择降级功能而非盲切。验收任务成功、严重错误、P95、单次成功成本、路由混淆矩阵、升级率和回归复杂度；净收益小于治理成本则保持单模型。

**追问点答案**  
- 多模型路由最大代价是什么？  
评测、回归和观测复杂度会上升。  
- 什么场景最适合先做路由？  
任务差异明显、成本压力大或要跨供应商兜底的场景。  
- 路由是不是越细越好？  
不是。每增加一条分支都扩大评测矩阵和故障组合；从可解释的 2–3 层开始，只有某类流量规模和质量/成本收益显著时再细分。

**原理补充**  
多模型路由的核心，是让不同请求匹配不同能力和成本曲线。

**架构扩展**  
成熟团队通常不会一上来做复杂路由树，而是从“简单请求 vs 复杂请求”这种最明显的二分开始，先验证收益，再逐步精细化。

**工程指标**  
- 路由后平均成本下降幅度：看收益。  
- 复杂任务成功率：看质量是否被保住。  
- 路由误分配率：看策略准确性。  
- 兜底命中率：看异常保护。  
- 多模型回归问题数：看治理压力。

**证据链**  
- 任务分层评测  
- 路由策略文档  
- 成本/质量对比结果  
- 供应商可用性记录

**低风险优化路径**  
1. 先验证单模型是否已明显不匹配  
2. 再做粗粒度分层路由  
3. 最后补回归和观测体系

**易错点**  
- 为了“高级”提前上路由  
- 路由规则过细  
- 没有统一评测与回归

## 7. 小模型何时可替代大模型？

**核心结论**  
小模型适合替代大模型的前提，是任务边界清楚、上下文可控、质量要求可量化，并且小模型在真实样本上已经达到业务可接受水平。

**标准答案**  
我判断小模型能不能替代大模型时，通常先看任务是不是足够聚焦，比如分类、摘要、结构化抽取、固定流程问答、规则明确的代码改写。因为这类任务对通用推理能力要求没有那么高，小模型更容易通过 Prompt、微调或工作流设计达到要求。之所以不能只看离线分数，是因为线上容错率、长尾输入和异常样本才是真正会出问题的地方。工程上比较稳的做法，是让小模型先接低风险或高频简单请求，再看质量和成本是否真正拉开差距。成熟团队通常把“小模型替代”当成本效优化项目，而不是信仰选择。

**高分回答**
我将替代定义为在同一端到端任务、同一 guardrail 下非劣。候选任务应边界稳定、输出可验证、风险低且量大，如分类/抽取；复杂开放推理和稀有长尾先保留强模型。用生产分层样本双盲比较任务成功、严重错误、校准/拒答、结构合法、P95 和单次成功成本，设非劣阈值，例如成功率下降≤0.5 个百分点且严重错误不升。先 shadow，再只路由低风险/高置信 cohort，低置信、异常长度或验证失败升级大模型；记录升级原因以发现边界漂移。Prompt 优化后再判断是否需要微调，微调还要计入数据、训练、版本与遗忘风险。线上监控输入分布、长尾失败和人工返工，定期用新样本回归；若回退率太高，理论单价优势会消失。替代成功需成本/延迟收益兑现且业务质量守住，不是模型参数更小。

**追问点答案**  
- 小模型最大的优势是什么？  
成本、速度和部署灵活性。  
- 小模型最大的风险是什么？  
复杂推理和长尾任务能力不足。  
- 是否一定要微调？  
不一定。先用清晰 schema、示例、检索和确定性后处理验证上限；当稳定重复错误且有高质量数据、Prompt 无法解决时再微调，并保留基座对照和回退。

**原理补充**  
替代判断的核心，不是参数大小，而是任务复杂度和业务容忍度是否匹配。

**架构扩展**  
成熟系统常见的做法是“大模型兜复杂，小模型扛规模”，比如把分类、打标签、初筛这类任务下沉给小模型，把真正难的问题留给大模型。

**工程指标**  
- 小模型任务成功率：看可替代性。  
- 平均/长尾成本差：看经济收益。  
- 响应时延改善幅度：看体验收益。  
- 高风险场景回退率：看边界稳定性。  
- 长尾失败样本比例：看风险面。

**证据链**  
- 双模型对比评测  
- 成本与时延报表  
- 长尾失败样本  
- 回退记录

**低风险优化路径**  
1. 先挑边界清楚的任务试点  
2. 再双跑对比  
3. 最后逐步扩大替代范围

**易错点**  
- 还没评测就盲目降级到小模型  
- 拿简单样本证明复杂能力  
- 缺少回退机制

## 8. 指令跟随能力如何验证？

**核心结论**  
指令跟随能力要通过多样化、带约束的任务集验证，重点看模型是否稳定遵守格式、边界和禁止项，而不是只看单次 demo 效果。

**标准答案**  
我验证指令跟随时，通常会设计一组覆盖真实业务的任务，里面包含输出格式限制、禁止回答项、角色设定和多条件指令组合。因为模型最容易在条件变多或上下文复杂时偏离要求。工程上比较稳的做法，是统计结构化输出成功率、规则违背率和多轮一致性，并加入一些对抗样本，比如干扰信息、冗余指令和冲突约束。成熟团队不会拿几条顺滑样本做结论，而是会看模型在复杂边界条件下是否仍然可控。

**高分回答**
我把指令拆为优先级、内容、格式、禁止项和工具边界，构造单约束、组合约束、冲突、长上下文、多轮漂移与 prompt injection 样本；系统/开发者指令应高于不可信文档和用户内容。评分不能只看 JSON 可解析，还要看 schema 语义、必填/枚举、禁止内容泄漏、任务完成和在信息不足时是否澄清/拒绝。确定性规则校验格式与 policy，人工或经校准的 judge 评语义，失败按“漏约束、优先级错误、过度拒答、被注入”分类。工具调用/高风险动作还需服务端权限与参数校验，不能把模型服从当安全边界。对每个模型、Prompt 和模板版本跑回归，shadow/灰度观察线上规则违背和修复率。设置严重约束零容忍或硬门禁，同时报告正常任务成功率，防通过全部拒答获得高安全分。

**追问点答案**  
- 为什么结构化输出成功率很重要？  
因为很多生产链路需要稳定被程序消费。  
- 对抗样本为什么要加入？  
因为真实用户输入不会总是干净标准。  
- 多轮一致性要测什么？  
还要测试早期指令被压缩/截断、用户改口、工具结果冲突和会话污染时，模型能否保持高优先级约束并正确更新可变偏好。

**原理补充**  
指令跟随的本质，是模型在复杂输入下保持对约束条件的稳定执行。

**架构扩展**  
在客服、审批、代码生成和工具调用场景里，指令跟随其实比“语言华丽”更重要。成熟系统会把格式、边界和拒答行为都纳入回归测试。

**工程指标**  
- 格式遵守率：看结构化稳定性。  
- 规则违背率：看可控性。  
- 多轮漂移率：看上下文稳定性。  
- 对抗样本通过率：看鲁棒性。  
- 回归波动幅度：看版本稳定性。

**证据链**  
- 指令跟随评测集  
- 结构化输出日志  
- 对抗样本结果  
- 版本回归报告

**低风险优化路径**  
1. 先把业务高频约束转成评测样本  
2. 再加入对抗和多轮测试  
3. 最后接入版本回归门禁

**易错点**  
- 用少量 demo 判断模型可控  
- 只测格式不测边界行为  
- 升级模型后不回归

## 9. 多语言能力如何评测？

**核心结论**  
多语言能力评测不能只看翻译效果，要看不同语言下任务成功率、术语一致性和文化语境适配能力。

**标准答案**  
我评测多语言能力时，通常不会只用通用翻译题，而是直接用目标业务任务去测，比如中文客服、英文知识问答、日语摘要、跨语言检索。因为业务里真正关心的不是语言表面流畅，而是任务有没有完成、术语有没有跑偏、引用有没有错。工程上比较稳的做法，是为每种重点语言准备代表样本，测任务成功率、专业术语准确率和输出风格一致性。成熟团队会特别关注小语种和混合语境，因为这些地方最容易出现隐性退化。

**高分回答**
我先按用户量、业务价值和风险把语言分层，不追求每种语言样本相同。黄金集由母语领域专家创建/复核，覆盖意图、专业术语、礼貌/文化语境、方言、代码混排、跨语言检索和安全拒答；不能把英文集机器翻译后当真值，因为难度和文化风险会被抹平。每种语言分别测任务成功、事实/引用、术语一致、格式、毒性/偏见与拒答，并看语言识别和切换是否正确；报告最差组和置信区间，不用总体平均掩盖小语种。RAG 同时评估 query 翻译、跨语 embedding 召回和答案语言，定位是检索还是生成退化。低资源语言可回退翻译→强语言模型→回译，但要测信息损失、隐私和延迟。上线按 locale 监控人工转接、投诉和版本回归，术语库版本化；质量不足的语言明确能力边界或人工服务。

**追问点答案**  
- 为什么不能只做翻译测试？  
因为翻译好不等于业务任务做得好。  
- 多语言评测最容易漏什么？  
术语一致性和混合语言场景。  
- 是否所有语言都要同等投入？  
不需要等量，但高风险低流量语言也不能因样本少忽略；按用户影响、法规和错误损失确定最低门槛、样本量和人工兜底。

**原理补充**  
多语言能力的本质，是模型在不同语言环境下仍能稳定完成同一业务目标。

**架构扩展**  
国际化产品最容易踩坑的点，是英文效果很好，其他语种却没人持续监控。成熟团队会按业务优先级建立重点语言看板，而不是只依赖主观感受。

**工程指标**  
- 各语种任务成功率：看能力差异。  
- 专业术语准确率：看业务可用性。  
- 混合语言错误率：看边界场景。  
- 用户满意度按语种分布：看真实体验。  
- 版本升级后的语种退化数：看回归风险。

**证据链**  
- 分语种评测集  
- 术语库  
- 用户反馈按语言分类  
- 版本对比结果

**低风险优化路径**  
1. 先锁定重点语种  
2. 再按业务任务建样本  
3. 最后做语种级回归和看板

**易错点**  
- 用英语效果代表全部语言  
- 不维护术语库  
- 不关注混合语言输入

## 10. 函数调用能力如何评估稳定性？

**核心结论**  
函数调用稳定性要看模型能否稳定选对工具、填对参数、遵守 schema，并在异常场景下安全降级。

**标准答案**  
我评估函数调用能力时，通常会重点看四件事：工具选择是否正确、参数字段是否完整、类型和枚举是否符合 schema、失败场景下会不会胡乱编造调用。因为生产里真正的问题往往不是“会不会调”，而是“有没有稳定可控地调”。工程上比较稳的做法，是准备一批带标准工具决策的数据集，同时加入歧义请求、缺参请求和不该调工具的样本，统计调用正确率、参数合法率和误调用率。成熟团队不会把 function call 当成黑盒魔法，而是会用 schema 约束、后置校验和审计日志把它收进工程体系。

**高分回答**
我把稳定性拆成“该不该调—调哪个—参数语义—执行结果—失败恢复”。评测集覆盖相似工具、无需工具、缺参数、歧义、多步依赖、权限不足、工具超时/错误、恶意工具输出和重复提交；工具选择与参数字段做精确/语义评分，高风险误调用按更高损失加权。schema 只保证形状，服务端还需鉴权、租户/对象范围、业务校验、幂等键、速率和最大影响面；删除/支付等动作展示具体参数并人工确认。模型不得根据工具返回的文本改写系统规则，输出视为不可信数据。执行器设截止时间、有限重试和错误分类：可恢复时重试/换工具，缺参时澄清，不确定时不调用；副作用调用不能盲重试。记录 plan、工具版本、参数摘要、结果、确认和 trace 以回放。验收选择/参数正确、误调用、澄清、端到端完成、重复副作用、P95 与故障恢复，并在版本升级后全矩阵回归。

**追问点答案**  
- 为什么误调用率很重要？  
因为错调工具比不调更危险。  
- schema 约束为什么不够？  
即使 JSON 合法，也可能操作错租户、金额或资源；schema 不提供权限、业务不变量、幂等和用户意图确认，这些必须由确定性执行层保证。
- 什么场景必须拒绝调用？  
信息不足、高风险动作或权限不满足时。

**原理补充**  
函数调用的核心，不是“模型会输出 JSON”，而是“模型在任务理解上做对了工具决策”。

**架构扩展**  
在工单系统、审批流、数据库查询和自动化执行场景里，函数调用稳定性几乎直接决定事故风险。成熟系统通常会把工具调用放进审计、权限和回放链路里统一治理。

**工程指标**  
- 工具选择正确率：看决策质量。  
- 参数合法率：看调用可执行性。  
- 误调用率：看风险水平。  
- 拒绝调用准确率：看保守性。  
- 调用失败后的恢复率：看韧性。

**证据链**  
- 工具调用评测集  
- Schema 校验日志  
- 调用回放记录  
- 异常案例库

**低风险优化路径**  
1. 先限制工具集合和 schema 复杂度  
2. 再做离线评测和后置校验  
3. 最后补权限、审计和失败降级

**易错点**  
- 只测 happy path  
- 函数调用失败后无校验  
- 高风险动作无权限和审计

## 11. 模型升级如何灰度发布？

::: tip L2 深度训练
把模型、Prompt、检索和工具版本纳入联合发布门禁：[AI 评测与发布门禁](/deep-dives/ai-architecture/04-evaluation-release-gates)
:::

**核心结论**  
模型升级灰度发布的关键，是先控制流量范围，再持续比较质量、时延、成本和异常行为，而不是直接全量替换。

**标准答案**  
我做模型升级灰度时，通常会先保留旧模型作为基线，再让新模型在小流量或特定租户、特定任务上试跑。因为模型升级的问题不只是接口通不通，而是输出质量、格式稳定性、工具调用行为和成本曲线都可能变化。工程上比较稳的做法，是先做离线回归，再做在线双跑或影子流量，确认任务成功率、拒答率、TTFT、单请求成本这些指标没有明显退化后再逐步放量。成熟团队不会把模型升级看成普通版本升级，而会把它当成行为变化治理。

**高分回答**
我把模型、Prompt、采样参数、RAG 索引/embedding、工具 schema 和策略组成不可变 release bundle，先在版本化黄金集做任务、严重错误、安全、结构、延迟和成功成本回归。通过后用脱敏生产影子流量双跑，旧模型服务用户，新模型记录差异；非确定输出采用规则、经校准 judge 与人工抽样，不做字符串比较。上线按内部用户→低风险租户/任务→逐级权重放量，每个 cohort 同期对照旧版，设置最小样本、置信区间和 guardrail；高风险错误零容忍或硬门槛。模型非确定性要求看分布和最差切片，不能只看平均。命中阈值先停放量，再回退完整 bundle；供应商版本若不可固定，则通过网关快照行为并提高影子监测。完成后保留少量 canary、更新失败样本与决策文档。指标包括严重错误差、成功率、拒答/工具行为、P95、成本、人工转接和回滚时间。

**追问点答案**  
- 为什么模型升级要保留旧模型基线？  
因为没有基线就无法判断新模型是否真的更好。  
- 双跑的价值是什么？  
还能发现真实分布和供应商限额问题；但会增加成本且不能观测用户反馈，因此需脱敏、采样，并在后续小流量灰度验证真实业务结果。
- 什么指标一退化就该暂停放量？  
任务成功率、结构化成功率、高风险错误率这类核心指标。

**原理补充**  
模型升级的本质，是对“行为分布变化”做可控验证，而不是简单替换版本。

**架构扩展**  
成熟系统往往会把模型版本、Prompt 版本、工具 schema 和知识库版本一起纳入灰度，因为真正影响效果的通常不是单个模型版本。

**工程指标**  
- 任务成功率差值：看质量变化。  
- 格式/工具调用成功率差值：看工程稳定性。  
- TTFT/TP99 差值：看性能变化。  
- 单请求成本差值：看经济影响。  
- 高风险异常率：看上线风险。

**证据链**  
- 离线回归报告  
- 影子流量结果  
- 模型版本对比看板  
- 异常样本记录

**低风险优化路径**  
1. 先做离线回归  
2. 再小流量双跑  
3. 最后逐步放量并保留快速回滚

**易错点**  
- 把模型升级当普通配置变更  
- 没有旧模型基线  
- 只看平均指标不看高风险异常

## 12. 模型版本回滚策略怎么设计？

**核心结论**  
模型回滚策略要预先定义回滚触发条件、回滚粒度和依赖项一致性，不能等出问题时再临场决定。

**标准答案**  
我设计模型回滚策略时，通常先回答三个问题：什么情况下必须回滚、回滚到哪一版、回滚时哪些相关配置也要一起回退。因为模型效果问题往往不只是版本本身，还可能和 Prompt、工具 schema、检索链路一起耦合。工程上比较稳的做法，是把模型版本、Prompt 模板和关键配置做版本化管理，并在发布系统里保留一键回切能力，同时定义高风险指标阈值。成熟团队不会只准备“回滚模型”这一个动作，而是准备“回滚整条推理配置组合”。

**高分回答**
回滚单位是已验证的 release bundle，而非一个模型名；记录精确模型/端点、Prompt、路由、检索与索引版本、工具 schema、安全策略和采样参数。发布前验证旧 bundle 仍可用、配额足够、数据/工具契约向后兼容，并为供应商停止旧版本准备第二供应商或能力降级。触发分为自动硬门槛（越权工具、严重安全/事实错误、结构大面积失败）和人工综合判断（质量/成本缓慢退化），均设最小样本与限次防抖。回切按租户/任务/权重渐进或紧急全量，保持请求幂等，避免已执行工具重复；回滚后确认业务成功、成本、队列与缓存污染恢复，不仅看 API 200。保留异常输入、完整 trace 和变更时间线，补入回归。季度演练旧模型不可用、配置不兼容和供应商故障，量化检测、决策、生效、业务恢复及回滚后失败率。

**追问点答案**  
- 为什么回滚不能只切模型？  
因为相关 Prompt、工具和知识链路可能也变了。  
- 什么情况应该立刻回滚？  
高风险错误、结构化大面积失败、核心任务成功率明显下滑。  
- 回滚后还需要做什么？  
还要验证用户结果、工具副作用和缓存/会话是否恢复，处理灰度期间产生的数据，并确认旧 bundle 的容量和安全性，而非只切路由后宣布完成。

**原理补充**  
回滚设计的本质，是让系统在行为异常时迅速回到已知稳定状态。

**架构扩展**  
成熟 AI 系统的回滚，通常和特征开关、租户灰度、任务路由一起配合，而不是只能全量开关式切换。

**工程指标**  
- 回滚触发到生效时长：看止损速度。  
- 回滚后恢复率：看回滚有效性。  
- 误回滚次数：看阈值设计质量。  
- 回滚所需人工步骤数：看自动化程度。  
- 相关配置一致性错误数：看版本治理能力。

**证据链**  
- 发布与回滚记录  
- 版本配置清单  
- 异常监控告警  
- 根因分析文档

**低风险优化路径**  
1. 先统一版本化管理  
2. 再定义回滚阈值和一键回切  
3. 最后把异常样本纳入回归

**易错点**  
- 只记录模型版本不记录关联配置  
- 回滚全靠手工  
- 没有预定义回滚触发条件

## 13. Prompt 变更如何做回归测试？

**核心结论**  
Prompt 变更回归测试要像代码回归一样管理，核心是样本集、版本化、自动比对和高风险场景门禁。

**标准答案**  
我做 Prompt 回归时，通常会先把 Prompt 当成正式版本资产，而不是随手改的文本。因为 Prompt 一旦接入生产，哪怕改一句话，也可能影响格式、工具选择、拒答边界甚至成本。工程上比较稳的做法，是维护分场景样本集，对比变更前后的任务成功率、结构化输出、拒答策略和成本指标，同时把高风险场景单独设门禁。成熟团队不会让 Prompt 修改靠主观感觉上线，而会把它纳入完整的评测和灰度流程。

**高分回答**
Prompt 与模板变量、few-shot、系统指令、输出 schema、模型/参数一起版本化，变更走 diff、owner 和原因说明。测试集由生产分布、高频任务、历史失败、不可回答、注入、长上下文、多轮和高风险样本组成，并固定数据快照防测试漂移。确定性检查 schema、引用、禁词和工具参数；语义质量用明确 rubric 的人工/校准 judge，多次采样评估方差。比较前后切片而非单一总分，同时守任务成功、严重错误、过度拒答、token、P95 和成功成本；允许某类改善牺牲另一类时必须显式风险接受。通过后 shadow 与小流量 A/B，监控真实转接/转化和投诉，可一键回退 Prompt bundle。每次线上事故转成最小复现样本，但保留独立 holdout 防止对测试集过拟合。

**追问点答案**  
- 为什么 Prompt 也要版本化？  
因为它会直接改变模型行为。  
- 回归测试最该覆盖什么？  
还要覆盖长尾、不可回答、指令冲突/注入、工具副作用与多轮状态，并保留未参与调优的 holdout，防 Prompt 只记住回归集。
- Prompt 变更也要灰度吗？  
对核心链路来说，通常要。

**原理补充**  
Prompt 回归的本质，是验证“行为控制逻辑”变更后是否仍然满足业务约束。

**架构扩展**  
成熟系统常常不是模型版本频繁变化，而是 Prompt 和流程不断迭代，所以 Prompt 版本管理和回归能力，实际上是 AI 工程化的基本盘。

**工程指标**  
- Prompt 版本回归通过率：看质量稳定性。  
- 核心任务成功率变化：看业务影响。  
- 高风险场景退化数：看上线风险。  
- 平均 token 变化：看成本影响。  
- Prompt 回滚次数：看变更质量。

**证据链**  
- Prompt 版本库  
- 回归样本集  
- 前后对比结果  
- 灰度记录

**低风险优化路径**  
1. 先把 Prompt 纳入版本管理  
2. 再建立最小回归样本集  
3. 最后给高风险链路加门禁和灰度

**易错点**  
- 把 Prompt 修改当“小改不需要测”  
- 没有固定回归样本  
- 回归只看平均效果

## 14. token 限额如何做预算治理？

::: tip L2 深度训练
从 token 单价升级到单次成功任务成本和预算燃烧率：[模型路由、SLA 与成本](/deep-dives/ai-architecture/03-model-routing-cost)
:::

**核心结论**  
Token 限额治理要把预算拆到业务线、租户、场景和用户层，并配合告警、降级和审计，而不是只做一个总额度。

**标准答案**  
我做 token 预算治理时，通常会先按业务和调用场景拆预算，因为不同功能的价值密度完全不同。像高价值客服助手、代码 Copilot、内部知识问答，它们的预算策略就不该一样。工程上比较稳的做法，是给业务线和租户设总额度，再给单请求、单用户、单时间窗口设上限，并在接近预算时触发告警、限流、降级或改用低成本模型。成熟团队不会把 token 成本当成财务事后问题，而会把它做成实时可控的产品和平台治理能力。

**高分回答**
我先让每次调用带业务/租户/功能/实验标签，计量输入、输出、缓存、重试、fallback 和检索开销，预算以“每成功任务成本”和业务价值制定。治理有四层：单请求限制上下文/输出与循环次数；用户/租户用速率和日/月配额；业务线用预算与燃烧率；平台用供应商总限额和异常熔断。阈值像错误预算分多窗口告警：短期突增防循环/攻击，长期偏差用于容量与产品决策。降级矩阵按价值预定义：压缩/检索上下文、缓存、切小模型、降低非核心频率或转异步；不能静默截断导致错误，高风险任务宁可拒绝/人工。预算借用和例外有 owner、期限与审计。成本看板同时展示任务成功、延迟和人工返工，防通过降低质量省钱。用预算预测偏差、异常检测时间、超额、降级质量损失、单位业务结果成本和未归因比例验收。

**追问点答案**  
- 为什么不能只设一个总预算？  
因为无法精细定位问题，也容易误伤高价值场景。  
- 接近预算时可以怎么降级？  
还可提高缓存、缩小检索候选、转批处理或暂停低价值功能；每种降级需质量门槛和用户提示，高风险任务不能静默换成不满足要求的模型。
- 为什么还需要审计？  
因为异常消耗往往来自某些特定租户或请求模式。

**原理补充**  
预算治理的核心，是把成本控制前移到请求层和场景层。

**架构扩展**  
成熟平台会把 token 预算和计费、权限、限流一起打通。否则即使有预算看板，也无法在消耗失控时及时止损。

**工程指标**  
- 预算使用率：看消耗水位。  
- 超额请求占比：看治理效果。  
- 异常 token 峰值事件数：看风险。  
- 降级触发率：看保护策略使用情况。  
- 成本归因覆盖率：看可见性。

**证据链**  
- Token 消耗日志  
- 配额配置  
- 告警记录  
- 业务/租户成本看板

**低风险优化路径**  
1. 先建立成本归因  
2. 再做分层限额和告警  
3. 最后补降级和审计能力

**易错点**  
- 只有总预算没有细粒度配额  
- 没有异常告警  
- 超预算后缺少自动止损

## 15. 如何控制首字延迟（TTFT）？

**核心结论**  
TTFT 优化的重点，是减少请求前置耗时、缩短模型启动准备链路，并避免把可缓存和可并行的步骤都串行化。

**标准答案**  
我优化 TTFT 时，通常先把链路拆开，看请求在进入模型前是不是已经花了很多时间，比如鉴权、检索、Prompt 拼装、工具准备、队列等待。因为很多团队误以为 TTFT 全是模型问题，实际上前置链路经常占掉很大一部分。工程上比较稳的做法，是先把能缓存的缓存、能并行的并行、能预热的预热，再看模型侧的并发、上下文长度和流式输出配置。成熟系统对 TTFT 的优化，不只是换更快模型，而是优化整条推理路径。

**高分回答**
我定义用户 TTFT 为提交到首个“有意义且可展示”token，使用 trace 拆 DNS/网关、鉴权、限流、队列、检索/rerank、Prompt 构建、供应商排队、prefill 和首包传输，并按模型、输入长度、缓存命中和并发分桶。先优化最大段：独立步骤并行，连接/TLS 复用，静态系统 Prompt 与检索结果缓存，去重/压缩上下文，预热实例与容量，避免模型队列 head-of-line；但依赖安全鉴权的步骤不能错误并行或跳过。流式只改善感知，不降低完成时间，首包不能是空白占位；若答案需验证，可先展示状态而非流出未经审核高风险内容。投机生成或小模型草稿需验证质量与额外成本。设实时队列与批处理隔离、超时和降级。验收端到端与各段 P50/P95/P99、首个有效 token、完成时间、取消率、质量和单位成本，防只优化 TTFT 却让吞吐/答案恶化。

**追问点答案**  
- 为什么 TTFT 不只是模型问题？  
因为前置检索、拼接、排队都会拉长首字时间。  
- 流式输出为什么能改善感知体验？  
它缩短感知等待并允许用户提前阅读/取消，但不减少总计算；高风险内容若需完整验证不能直接流式展示，应采用缓冲、分段审核或进度反馈。
- 长上下文为什么会伤 TTFT？  
因为输入处理和计算开销会明显上升。

**原理补充**  
TTFT 的本质，是用户从发起请求到看到第一段有效响应的整体等待时间。

**架构扩展**  
成熟团队会同时区分“感知 TTFT”和“技术 TTFT”，前者关乎用户体验，后者便于定位瓶颈。很多优化其实不需要改模型，只要把检索和拼接流程做顺就能见效。

**工程指标**  
- 端到端 TTFT：看用户体验。  
- 前置链路耗时：看非模型瓶颈。  
- 模型排队时长：看并发压力。  
- 输入 token 长度分布：看上下文成本。  
- 流式首包率：看体验优化效果。

**证据链**  
- 链路 tracing  
- TTFT 分段监控  
- 请求上下文长度数据  
- 并发队列统计

**低风险优化路径**  
1. 先拆分 TTFT 耗时来源  
2. 再优化检索、缓存和并行链路  
3. 最后再调模型和资源

**易错点**  
- 只盯模型本身  
- 不拆分链路耗时  
- 把所有前置步骤串行执行

## 16. 批处理和实时推理如何分层？

**核心结论**  
批处理和实时推理要按 SLA、任务形态和资源使用方式分层，避免互相抢资源、互相拉低体验。

**标准答案**  
我做批处理和实时推理分层时，通常先看任务有没有明确的时延要求。像搜索问答、Copilot、在线客服这类交互式任务，更适合走实时通道；像离线摘要、批量打标签、知识库预处理，则更适合走批处理通道。之所以要分层，是因为两类任务在并发模型、资源调度和失败容忍度上完全不同。工程上比较稳的设计，是把实时链路优先保障低延迟，把批量任务放到可控窗口和可中断队列里，并且在高峰期允许批处理让路。成熟平台会把这件事当资源治理问题，而不是简单的接口分类。

**高分回答**
我按业务 deadline、交互性、可中断/重试和数据新鲜度分实时、准实时、离线，而不是只按接口。实时请求走低排队、有并发保留和 P95/P99 SLO 的池；批任务进入持久队列，按 deadline/价值调度，支持 micro-batching、checkpoint、幂等与断点续跑；准实时可用小批次平衡吞吐和延迟。GPU/供应商配额可共享物理容量，但逻辑队列、并发、预算和优先级隔离，实时高峰可抢占尚未执行的批任务，不能中断已产生外部副作用的步骤。批量失败使用有限重试、DLQ 和对账，结果带模型/Prompt/数据版本以便重放；积压按最老任务年龄和目标清空时间扩容。容量在 N-1 或供应商限流下仍保障实时，批处理有完成窗口与降级策略。验收实时 P99/拒绝、批吞吐/成本、deadline 达成、积压年龄、抢占恢复和资源利用，防实时空闲时批池仍浪费。

**追问点答案**  
- 为什么不能共用一个调度池？  
可以共享底层资源提高利用率，但必须有独立队列、保留容量、优先级和抢占边界；完全同池会产生 head-of-line blocking，完全物理隔离又可能长期闲置。
- 批处理更适合什么优化？  
批量并行、低峰执行、成本优先。  
- 实时链路最该关注什么？  
TTFT、TP99 和稳定性。

**原理补充**  
分层的本质，是让不同 SLA 的任务匹配不同资源和调度策略。

**架构扩展**  
成熟 AI 平台往往会先把服务分成实时、准实时和离线三层，再分别设计限流、缓存和资源队列，这样在高峰期才不会互相拖垮。

**工程指标**  
- 实时链路 TP99：看体验。  
- 批处理吞吐：看资源效率。  
- 资源抢占冲突次数：看分层效果。  
- 高峰期实时成功率：看优先级保护。  
- 批任务积压量：看调度健康度。

**证据链**  
- 任务分类清单  
- 调度策略文档  
- 高峰期资源监控  
- 队列积压数据

**低风险优化路径**  
1. 先按 SLA 分类任务  
2. 再拆分实时和批量资源池  
3. 最后补优先级和高峰让路策略

**易错点**  
- 实时和批处理混跑  
- 不区分体验指标和吞吐指标  
- 高峰期没有优先级保护

## 17. 温度、top_p 在生产怎么设置？

**核心结论**  
温度和 top_p 的生产设置要服务任务目标，越偏确定性任务越应收紧随机性，越偏创意任务越可以适度放开，但要保持可回归。

**标准答案**  
我在生产设置温度和 top_p 时，通常先看任务是要稳定、可重复，还是要多样、发散。像结构化抽取、SQL 生成、工具调用、合同问答这类任务，我一般会把随机性收紧，优先保证格式和结论稳定；而像营销文案、创意写作这类任务，才会适度提高随机性。之所以不能拍脑袋设，是因为这些参数会直接影响结果稳定性、回归难度和线上投诉率。工程上比较稳的做法，是按任务类型设默认参数，并把参数也纳入版本管理和回归测试。成熟系统不会用一套参数打天下。

**高分回答**
温度重塑 token 概率分布，top_p 截断到累计概率质量，两者同时大幅变化会让原因难归因，因此先固定一个、对另一个做小网格实验。抽取、工具和事实问答优先低随机性并用 schema/验证器，但低温不等于事实正确或完全确定，服务端实现、模型版本和并行也可能造成差异；创意任务用适度随机并通过人工偏好与安全护栏评估多样性。每类任务的配置与模型/Prompt 一起版本化，在黄金集多次采样，报告任务成功、结构、安全、self-consistency、distinctness、成本与输出长度；选择满足质量门槛的最简单参数，不凭“更像人”判断。若要多候选，明确 n-best 的额外成本和选择器偏差。上线小流量看用户选择、编辑率与投诉，设置最大输出和回退。模型升级后重新搜索，旧参数不保证仍适配。

**追问点答案**  
- 为什么确定性任务要低温度？  
低温可减少采样方差，但不能替代 schema、事实验证和业务校验，也不保证完全可复现；应以端到端成功率而非参数直觉验收。
- top_p 和温度都要调吗？  
通常可以先固定一个，再小步调另一个。  
- 参数改动也要回归吗？  
对核心链路来说，要。

**原理补充**  
这两个参数本质上是在控制生成采样的随机性和候选分布范围。

**架构扩展**  
成熟团队会把采样参数和任务标签绑定，而不是写死在代码里。这样模型升级或任务扩展时，参数治理更容易跟上。

**工程指标**  
- 同样输入输出一致率：看稳定性。  
- 结构化成功率：看工程可用性。  
- 用户主观满意度：看体验。  
- 回归波动幅度：看可控性。  
- 高风险输出率：看安全边界。

**证据链**  
- 参数实验记录  
- 回归测试结果  
- 任务分类配置  
- 用户反馈

**低风险优化路径**  
1. 先按任务分层参数  
2. 再做小范围 A/B 试验  
3. 最后把参数纳入版本管理

**易错点**  
- 全场景共用一套采样参数  
- 参数变化不做回归  
- 为了“更像人”牺牲稳定性

## 18. 模型安全能力如何纳入选型？

**核心结论**  
模型安全能力要作为选型维度前置考虑，重点看拒答边界、越狱鲁棒性、工具调用安全性和合规配套能力。

**标准答案**  
我把安全能力纳入选型时，通常不会只看模型“会不会回答敏感问题”，而是看它在真实业务链路里能不能和安全策略协同工作。比如在高风险问题下是否会越权输出、在工具调用场景里是否容易被 prompt injection 带偏、是否支持审计和安全配置。之所以安全要前置，是因为后补安全往往代价更高，而且有些风险和模型能力本身强相关。工程上比较稳的做法，是把安全对抗样本、越狱测试和高风险任务拒答能力一起纳入评测，不让安全只停留在合规条款上。成熟选型会把安全看成生产能力，而不是附加项。

**高分回答**
我先做场景威胁模型：数据泄露、越狱/有害内容、间接 prompt injection、工具越权、成员推断与供应商数据使用，按业务损失设门槛。评测集含正常高风险请求、合法边界请求、编码/多语言/多轮越狱、不可信检索文档、恶意工具结果和跨租户尝试；同时测攻击成功与过度拒答，防“全拒绝”拿高安全分。模型侧拒答只是纵深防御，确定性系统仍负责输入/输出策略、权限、租户隔离、数据最小化、工具参数/影响面、人工确认和审计。选型还检查数据保留/训练政策、地域、加密、删除、事件响应、版本通知和可固定性。红队结果按严重度加权并保留未知攻击预算，版本/Prompt/工具变化持续回归。上线采用最小权限、canary 和 kill switch，监控越权尝试、敏感输出、误拒、工具副作用与安全事件 MTTR。

**追问点答案**  
- 为什么安全能力不能后补？  
因为有些风险来自模型本身的行为特性。  
- 安全能力评测该测什么？  
越狱、敏感问答、工具越权、合规日志能力。  
- 仅靠外层审核够吗？  
不够；外层分类器也会被绕过，模型拒答也会失效。需要数据边界、确定性权限与业务校验、最小工具能力、输出审核和人工升级组成纵深防御。

**原理补充**  
安全能力的本质，是模型在面对恶意输入和高风险场景时的可控性。

**架构扩展**  
成熟团队选模型时，往往会专门留出安全评测集，不会把安全交给最后的网关统一兜底，因为真实风险经常发生在模型理解和调用决策层。

**工程指标**  
- 越狱成功率：看鲁棒性。  
- 高风险请求拒答率：看安全边界。  
- 工具越权误调用率：看执行风险。  
- 安全审计字段完整率：看可追踪性。  
- 安全回归问题数：看长期稳定性。

**证据链**  
- 安全评测集  
- 对抗测试报告  
- 工具调用审计日志  
- 合规要求清单

**低风险优化路径**  
1. 先把安全样本纳入选型评测  
2. 再验证模型与外层防护协同  
3. 最后持续做红队和回归

**易错点**  
- 选型只看效果不看安全  
- 安全测试只测普通输入  
- 工具调用场景缺少对抗验证

## 19. 供应商锁定风险如何规避？

**核心结论**  
规避供应商锁定的关键，不是完全不用外部能力，而是在接口、评测、Prompt 和路由层保留替换空间。

**标准答案**  
我规避供应商锁定时，通常会先承认锁定风险不可完全消除，但可以控制。比较稳的做法，是把业务侧和供应商 API 之间加一层统一抽象，让模型调用、参数映射、日志和审计尽量通过自有网关或 SDK 统一收口。之所以这样做，是因为一旦 Prompt 模板、调用协议、工具 schema 和评测体系都深度绑定某一家，后续切换成本会急剧上升。工程上还要维护跨模型的评测基线和回退方案，这样新供应商引入时才能快速验证。成熟架构师不会为了避免锁定而放弃效率，而是会保留合理的迁移通道。

**高分回答**
我先识别真正锁定点：专有 API/tool schema、Prompt 行为、embedding 空间、微调权重/数据、批处理与缓存语义、日志/评测和合同数据导出。用薄网关统一鉴权、限额、观测与通用请求模型，业务依赖自己的任务契约；但允许通过显式 capability 接口使用高价值专有能力，避免最低公分母损失。Prompt、工具、知识源与黄金集归自己且版本化，向量原文/元数据可重建，重要日志可导出。每季度让替代模型跑影子回归，估算达到门槛所需改造、TCO 和迁移 RTO；关键链路可预置第二供应商，但需验证故障是否相关、配额、数据合规与行为差异，不能假装瞬时无损切换。合同明确版本通知、数据保留/删除、价格与退出。锁定是有意 trade-off：专有收益大于退出成本可接受，但记录触发退出的价格、SLO、合规或路线图阈值。

**追问点答案**  
- 统一抽象层的价值是什么？  
降低供应商 API 差异直接渗透到业务层。  
- 为什么评测基线也能降低锁定？  
因为它让替代方案可快速验证。  
- 会不会为了可迁移牺牲最佳能力？  
会。应保持通用核心和显式专有扩展，不为理论切换屏蔽高价值能力；通过 ADR 量化专有收益、迁移成本和退出触发条件。

**原理补充**  
锁定风险治理的本质，是把“替换成本”从不可控变成可评估、可计划。

**架构扩展**  
成熟团队往往不会在第一天就做多供应商复杂架构，但会先统一调用入口、日志和评测，这样业务跑起来之后仍然有换路的空间。

**工程指标**  
- 业务侧直接依赖供应商 API 的比例：看耦合度。  
- 替代模型验证周期：看迁移弹性。  
- 跨供应商回归通过率：看兼容性。  
- 供应商专有能力依赖数：看锁定程度。  
- 故障切换成功率：看兜底能力。

**证据链**  
- 调用抽象层设计  
- 评测基线  
- 供应商对比报告  
- 切换演练记录

**低风险优化路径**  
1. 先统一调用入口  
2. 再建设跨模型评测基线  
3. 最后演练替换与故障切换

**易错点**  
- 业务直接耦合供应商接口  
- 没有跨模型评测基线  
- 完全不演练替代路径

## 20. 如何建立模型选型决策文档？

**核心结论**  
模型选型决策文档要能说明问题、方案、对比、结论和后续策略，而不是只记录“选了哪个模型”。

**标准答案**  
我写模型选型决策文档时，通常会包含五部分：业务目标和约束、候选模型清单、评测方法和结果、最终选择与取舍、上线和回滚策略。因为如果文档只写“我们选了某某模型”，过几个月团队很难知道当时为什么这么选，也无法判断现在要不要重做决策。工程上比较稳的做法，是把质量、时延、成本、安全、供应商风险和替代空间都记录成可复查的结论，并注明当时的样本和假设。成熟团队会把选型文档当成活文档，在模型升级或业务变化时持续更新。

**高分回答**
我用 ADR + 证据附件记录：业务任务与用户、硬门槛/SLA/数据边界、流量和成本假设；候选模型精确版本、Prompt/RAG/工具配置；黄金集来源、切片、评分 rubric、人工一致性与压测环境；质量/严重错误、延迟、可用、单次成功成本、安全和 12–36 月 TCO 对比。结论写清为什么选择、拒绝了什么、残余风险、补偿控制和不确定性区间，不用加权总分掩盖硬门槛。还包含 rollout bundle、灰度/回滚、供应商故障与退出方案、owner、评审者和决策日期。设置复审触发：任务分布、价格、模型版本、法规、SLO 或失败率超过阈值；评测数据和脚本可复现，文档链接线上 dashboard。更新用 superseded ADR 保留历史，而非覆盖当时依据。文档价值是让下一次决策可审计和快速重验，不是证明选型永远正确。

**追问点答案**  
- 为什么决策文档不能只写结论？  
因为后续复盘和替换时需要知道当时的依据。  
- 文档里最关键的是什么？  
真实评测依据和取舍说明。  
- 选型文档多久该更新？  
使用事件触发而非固定日历为主：模型/Prompt/知识链路变化、任务分布或风险等级变化、价格/SLO/法规越阈值时重评；同时定期确认假设仍成立。

**原理补充**  
决策文档的本质，是把临时判断沉淀成组织可复用的技术资产。

**架构扩展**  
成熟组织通常会把模型选型文档和上线门禁、回滚方案、供应商评估放在一起，这样它就不只是文档，而是完整治理入口。

**工程指标**  
- 选型文档覆盖率：看治理完整度。  
- 文档与实际版本一致率：看维护质量。  
- 重大变更复用历史决策次数：看资产价值。  
- 选型返工率：看决策质量。  
- 文档更新滞后时长：看治理及时性。

**证据链**  
- 选型决策文档  
- 评测数据附件  
- 上线与回滚记录  
- 复盘记录

**低风险优化路径**  
1. 先统一选型文档模板  
2. 再把评测和取舍写实  
3. 最后让文档进入版本和评审流程

**易错点**  
- 文档只有结论没有依据  
- 决策与实际版本脱节  
- 文档写完不更新
