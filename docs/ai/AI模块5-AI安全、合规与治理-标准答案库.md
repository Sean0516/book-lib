# AI模块5 - AI安全、合规与治理 - 标准答案库

## 1. Prompt Injection 的核心防线是什么？

::: tip L2 深度训练
把外部内容视为数据，并在模型外强制执行权限与副作用边界：[Agent 生产化与工具安全](/deep-dives/ai-architecture/02-agent-production-safety)
:::

**核心结论**  
Prompt Injection 的核心防线不是一句“忽略恶意指令”，而是把外部内容当成不可信数据，并用权限隔离、工具白名单、内容净化和结果校验多层防护。

**标准答案**  
我防 Prompt Injection 时，通常先把用户输入、网页内容、知识库文档、邮件正文这类外部内容全部视为不可信，而不是默认它们只是“普通上下文”。因为注入攻击本质上是在伪装成数据的内容里夹带指令，试图让模型改变原本的系统行为。工程上比较稳的做法，是把系统指令、业务规则和外部内容明确分层，限制外部内容不能直接影响工具权限和执行边界；同时对高风险动作增加白名单、审批和后置校验。成熟团队不会把注入问题交给 Prompt 单点处理，而会把它当成整条 AI 执行链路的安全问题。

**高分回答**
我会假设检测一定存在漏网，核心目标是模型即使被注入也无法完成高价值攻击。系统指令、用户意图和网页、邮件、RAG 文档、工具输出在消息与数据模型中明确分层；外部内容只作为带来源的不可信数据，不能改变工具列表、权限、审批和数据流策略。内容清洗、指令检测与引用隔离能降低风险，但不作为唯一安全边界。

执行端使用最小权限短时令牌、工具与域名 allowlist、参数化接口、敏感数据出域策略和副作用审批。模型不接触凭证，也不能把从一个工具读取的秘密发送到另一个未经授权的出口；高风险动作先 dry-run，审批绑定冻结参数。检索和工具内容中出现“忽略规则”时，即使分类器未识别，授权层仍应拒绝越权执行。

红队覆盖直接注入、网页/邮件间接注入、编码混淆、跨工具外传、持久记忆污染和审批绕过，衡量攻击链完成率、敏感数据泄漏率、越权调用、误拦截和撤权时延。每个事件记录内容来源、策略决策与实际副作用，失败样本进入发布门禁。Prompt 强化是纵深防御的一层，不是权限系统。

**追问点答案**  
- 为什么“忽略恶意提示”不够？  
因为模型可能仍然被上下文污染，系统级约束更可靠。  
- 为什么外部内容默认不可信？  
因为任何文档、网页或输入都可能夹带恶意控制意图。  
- 最有效的补强手段是什么？  
执行端最小权限和数据流控制最关键，再叠加参数校验、审批、内容隔离与持续红队，确保单层失效不会打通攻击链。

**原理补充**  
Prompt Injection 的本质，是攻击者把“数据”伪装成“控制指令”，试图劫持模型的决策路径。

**架构扩展**  
成熟 AI 平台通常会把系统 Prompt、用户 Prompt、检索内容和工具权限分层治理，并对检索链路和工具调用链路做独立审计，这样即使某一层被污染，也不至于直接放大成执行事故。

**工程指标**  
- 注入攻击成功率：看核心风险。  
- 越权工具调用拦截率：看防护效果。  
- 高风险动作审批覆盖率：看止损能力。  
- 内容净化命中率：看前置过滤能力。  
- 安全回归缺陷数：看长期稳定性。

**证据链**  
- 红队测试报告  
- 工具调用审计日志  
- 注入样本库  
- 权限策略记录

**低风险优化路径**  
1. 先把外部内容全量视为不可信  
2. 再做工具权限和执行边界隔离  
3. 最后持续红队演练和回归

**易错点**  
- 只靠 Prompt 抵御注入  
- 外部内容与系统指令混用  
- 高风险工具没有审批和审计

## 2. 工具调用越权如何防护？

::: tip L2 深度训练
从可信身份重新鉴权，而不是相信模型生成的授权声明：[Agent 生产化与工具安全](/deep-dives/ai-architecture/02-agent-production-safety)
:::

**核心结论**  
工具调用越权防护的关键，是让模型“即使被诱导也拿不到不该有的能力”，核心靠身份、权限和参数边界控制，而不是靠模型自觉。

**标准答案**  
我防工具调用越权时，通常会先把 Agent 和工具都放进统一身份与授权体系里，而不是默认只要模型选中了工具就可以执行。因为越权的本质问题不是“模型会不会想坏事”，而是系统有没有给它机会做坏事。工程上比较稳的做法，是按任务、角色、租户和动作类型限制可调用工具、参数范围和目标资源，对高风险写操作必须加审批和二次确认。成熟平台会把权限控制放在工具网关或执行层，而不是埋在 Prompt 里。

**高分回答**
工具网关从可信认证上下文获得用户、租户、角色与任务，不接受模型自行填写的身份声明。策略引擎同时判断主体、动作、资源、环境和参数范围，执行端再次鉴权；Agent 只拿到任务级短时能力令牌，限制工具、资源集合、调用次数和有效期。网络出口、文件路径和数据库行级权限也要收紧，防止多个低权限工具组合成越权能力。

工具按只读、可逆写入和高风险不可逆动作分级。写操作使用业务幂等键和 dry-run，高风险操作生成精确差异与影响范围，由合格审批人确认；审批令牌绑定 plan_id、参数哈希和数据版本，参数变化即失效。工具返回结果经过字段级脱敏，避免越权数据已经进入模型后再在展示层隐藏。

测试不仅验证正常角色，还包括伪造 tenant_id、对象 ID 枚举、参数替换、审批后篡改、缓存复用和跨工具权限提升。硬指标是越权执行与跨租户泄漏为零，同时监控误拒率、撤权生效 P99、审批覆盖和审计完整率；异常时可立即撤销 Agent 或单工具能力。

**追问点答案**  
- 为什么 Prompt 里写“不要执行危险操作”不够？  
因为真正的权限控制必须落在系统层。  
- 参数范围为什么也属于权限？  
因为授权不仅是“能否调用”，还包括可操作的资源、金额、数量、收件人和时间范围；这些必须由执行端校验。
- 什么工具最需要强权限约束？  
删除、支付、发消息、改权限、改配置等有副作用工具。

**原理补充**  
越权防护的核心，是让工具执行层对模型决策保持独立审查，而不是被模型直接驱动。

**架构扩展**  
成熟平台常见做法是“模型建议动作，执行层判定是否允许”，并通过最小权限、短时令牌和操作白名单把风险面收紧。这样即使模型被攻击，也只能在很小的范围内活动。

**工程指标**  
- 越权调用拦截率：看控制效果。  
- 高风险工具审批覆盖率：看治理强度。  
- 权限误配导致的安全事件数：看配置质量。  
- 临时权限使用率：看最小权限落地情况。  
- 工具执行审计完整率：看可追溯性。

**证据链**  
- 权限模型  
- 工具白名单  
- 执行日志  
- 审批记录

**低风险优化路径**  
1. 先给工具建立统一权限层  
2. 再限制高风险动作和参数范围  
3. 最后接审批和审计闭环

**易错点**  
- 工具权限和业务权限脱节  
- 高风险工具无参数约束  
- 只有调用日志没有审批链路

## 3. 数据泄露风险如何分级治理？

::: tip L2 深度训练
从数据分类、出口、工具副作用和事件影响面联合分级：[安全事件响应与 AI 边界](/deep-dives/security-compliance/04-security-incident-ai)
:::

**核心结论**  
数据泄露治理要先按数据敏感度和泄露后果分级，再决定脱敏、隔离、审批、审计和保留策略，不能所有数据一刀切。

**标准答案**  
我做数据泄露治理时，通常先把数据按敏感度分层，比如公开数据、内部数据、敏感个人信息、商业机密和高监管数据。因为不同数据泄露后的损失完全不同，如果不先分级，后面所有控制措施都会失真。工程上比较稳的做法，是对高敏感数据优先做最小暴露、脱敏、访问审批、日志审计和保留限制，对低风险数据则控制到业务可接受范围。成熟团队不会只问“有没有泄露”，而会问“哪类数据最危险、现在暴露在哪里、谁可以接触”。

**高分回答**
我会按数据类别、可识别性、法规义务、商业影响和暴露范围形成分级，例如公开、内部、机密、高度受限；分类结果绑定数据所有者、允许用途、区域、保留期和可使用的模型/供应商。风险不是只看字段名，同一地址在公开门店与个人住址场景等级不同，组合数据也可能提升可识别性。

随后画端到端数据流：采集、Prompt、RAG、向量、工具、缓存、日志、评测、人工审核和第三方出域。对高度受限数据优先不采集或本地确定性处理；确需使用时做字段最小化、令牌化、用途与行级授权、加密、短保留和审批，并要求供应商不用于训练、满足区域和删除传播。DLP 同时检查输入与输出，但误报处理不能成为永久绕过通道。

治理以高敏数据资产覆盖率、未登记数据流、出域量、越权访问、删除传播时长和事件影响面衡量，并定期用 canary 数据和红队验证监控。发生泄露时按数据等级决定隔离、密钥/令牌撤销、通知与取证。分级的价值是选择相称控制，不是给数据贴完标签就结束。

**追问点答案**  
- 为什么分级很重要？  
因为不同数据的治理成本和风险代价不同。  
- 哪类数据最该优先治理？  
个人敏感信息、商业秘密和监管重点数据。  
- AI 场景为什么更容易放大泄露风险？  
还会进入向量、评测、人工审核和工具输出，并可能被多轮上下文重新传播，因此必须管理完整数据生命周期和出域路径。

**原理补充**  
风险分级的本质，是把有限安全资源优先投入到高后果数据的防护上。

**架构扩展**  
成熟平台通常会给 AI 数据流做单独视图，明确哪些数据会进模型、会进日志、会进缓存、会出域，这比单纯在数据库层做分类更贴近真实风险。

**工程指标**  
- 高敏数据暴露面数量：看风险面。  
- 脱敏覆盖率：看基础控制效果。  
- 高敏数据出域调用比例：看外泄风险。  
- 数据访问审批覆盖率：看治理强度。  
- 泄露告警数：看风险态势。

**证据链**  
- 数据分级清单  
- 数据流图  
- 审计日志  
- 出域调用记录

**低风险优化路径**  
1. 先完成数据分级和数据流盘点  
2. 再给高敏数据做最小暴露和脱敏  
3. 最后补审批与异常告警

**易错点**  
- 不分级统一治理  
- 只关注存储层不看 AI 链路  
- 不记录数据出域路径

## 4. PII 在 AI 流程中如何脱敏？

::: tip L2 深度训练
覆盖输入、日志、缓存、向量库、供应商和删除传播：[数据保护与生命周期](/deep-dives/security-compliance/02-data-lifecycle)
:::

**核心结论**  
PII 脱敏要覆盖输入、日志、缓存、审计和训练样本全链路，并根据业务用途决定替换、遮蔽还是令牌化，而不是只在展示层做处理。

**标准答案**  
我处理 PII 脱敏时，通常先看数据在 AI 流程里的用途，是为了理解任务上下文，还是只是附带出现。如果对任务本身不关键，就优先在进入模型前做遮蔽或替换；如果必须保留语义关联，就更适合用令牌化或结构化占位。工程上比较稳的做法，是把脱敏做在统一入口层，并要求日志、缓存和评测集同步遵守相同规则，避免输入脱敏了，日志里却原样留着。成熟团队不会把脱敏理解成“前端遮一下”，而会把它看成数据处理链路的标准能力。

**高分回答**
先做数据最小化和用途判断：任务不需要的 PII 在采集入口直接删除；需要保留类型但不需要原值时使用占位符；跨段需要同一实体关联时使用租户隔离、可轮换的令牌化；只有确需恢复原值的受控流程才允许在独立 vault 中映射。不可逆哈希对低熵手机号或证件号并不安全，必须使用带密钥的方案并控制重识别权限。

检测结合结构规则、校验位、词典和上下文模型，覆盖 Prompt、附件、OCR、工具结果、日志、trace、缓存、向量元数据、评测集与人工工单。模型只看到最小化数据，必要的重新识别由独立服务在授权后的最终步骤完成，供应商和普通运维人员不可访问映射。脱敏规则版本化，删除请求传播到所有派生副本。

评测按姓名、联系方式、证件、账户、位置及组合识别分别统计召回率和误脱敏率，并加入多语言、分隔符、编码和图片样本。硬指标包括高风险 PII 出域为零、日志残留率、重识别审计覆盖与删除完成时限；误脱敏损害业务时，针对用途调整策略，而不是全局降低检测阈值。

**追问点答案**  
- 为什么展示层脱敏不够？  
因为模型、日志和缓存里可能已经泄露了。  
- 什么时候适合令牌化？  
适合需要稳定关联或受控恢复的场景；映射必须独立存储、按租户隔离、最小授权，并支持轮换和删除。
- 为什么评测集也要脱敏？  
因为评测数据本身也会被反复传播和复用。

**原理补充**  
PII 脱敏的本质，是在尽量保留业务可用性的前提下降低可识别性和泄露价值。

**架构扩展**  
成熟 AI 平台会把脱敏做成平台层能力，而不是要求每个业务自己实现，否则很快会出现口径不一致和漏网路径。

**工程指标**  
- 输入脱敏覆盖率：看入口防护。  
- 日志脱敏完整率：看落库安全性。  
- 评测集敏感残留率：看治理闭环。  
- 误脱敏率：看业务可用性代价。  
- PII 出域事件数：看核心风险。

**证据链**  
- 脱敏规则  
- 样本扫描报告  
- 日志抽检  
- 出域审计记录

**低风险优化路径**  
1. 先统一入口脱敏规则  
2. 再覆盖日志、缓存和评测集  
3. 最后优化令牌化和误脱敏控制

**易错点**  
- 只做前端脱敏  
- 评测和日志链路漏脱敏  
- 脱敏规则与业务语义冲突无人校准

## 5. 模型输出安全审核如何设计？

**核心结论**  
输出安全审核要分层设计：先做模型内约束，再做规则/模型审核，最后对高风险结果做人工确认，而不是寄希望于模型天然安全。

**标准答案**  
我设计输出安全审核时，通常会先按风险类型分类，比如涉政、涉暴、涉黄、歧视、违法建议、越权操作建议、敏感业务结论等。因为不同风险的识别方式和容忍度都不一样。工程上比较稳的做法，是让基础模型先承担第一层约束，再用专门的审核规则或审核模型做第二层判断，对高风险场景增加人工确认和拒答策略。成熟团队不会只在最后接一个“内容安全接口”就结束，而会把高风险场景前后链路都考虑进去。

**高分回答**
我会先按业务场景建立风险 taxonomy、严重度和允许动作，而不是使用一个全局“安全分”。输入侧识别高风险意图并选择受限能力；生成侧应用模型策略；输出侧用确定性规则、PII/DLP、领域校验和独立审核模型组合判断。涉及医疗、金融、法律、未成年人或真实副作用的结果，还需要证据验证、人工确认或明确转介。

流式输出是关键边界：若风险内容一旦展示就不可撤回，高风险场景应缓冲句子或完整结果后审核；低风险场景可增量审核并在触发时停止，但要接受少量内容已发送的残余风险。结构化工具参数与自然语言分别审核，安全文本不代表动作安全。拒绝响应也要提供安全替代，避免过度拦截让用户绕过系统。

评测集覆盖正常、边界、变体、对抗与上下文依赖样本，按风险类别统计漏判率、误杀率、严重度加权损失、审核 P99、申诉改判和人工负担。阈值按场景与风险成本校准，规则/模型版本进入发布门禁；重大漏判可一键切换保守策略并回放影响范围。

**追问点答案**  
- 为什么单层审核不够？  
因为单一层容易漏判或误判。  
- 审核模型和主模型分开有什么好处？  
能提供不同信号和独立版本治理，但并不天然独立；仍需规则、业务事实验证、人工复核和对抗评测共同兜底。
- 什么场景最适合人工复核？  
高监管、高敏感、高后果的输出。

**原理补充**  
输出安全审核的本质，是在生成之后再次确认结果是否落在业务允许边界内。

**架构扩展**  
成熟平台会把审核策略和业务场景绑定，比如对通用闲聊、企业知识问答和自动执行建议分别使用不同的安全门槛，而不是统一一把尺子。

**工程指标**  
- 高风险输出拦截率：看核心防护。  
- 误杀率：看可用性代价。  
- 人工复核命中率：看审核质量。  
- 审核延迟：看体验影响。  
- 审核漏判导致的事件数：看残余风险。

**证据链**  
- 审核规则库  
- 风险样本集  
- 审核日志  
- 人工复核记录

**低风险优化路径**  
1. 先按风险类型分层  
2. 再建立多层审核链路  
3. 最后对高风险场景接人工复核

**易错点**  
- 只靠主模型自我约束  
- 所有场景统一审核阈值  
- 不跟踪误杀和漏判

## 6. 多租户隔离在 AI 系统怎么做？

::: tip L2 深度训练
身份、检索、缓存、模型上下文、工具和审计都要绑定租户：[身份、授权与租户隔离](/deep-dives/security-compliance/01-identity-authorization)
:::

**核心结论**  
多租户隔离要覆盖 Prompt、知识库、缓存、工具权限、日志和审计，而不是只在 API 层加一个 tenant_id。

**标准答案**  
我做 AI 多租户隔离时，通常会先盘清租户数据会流经哪些环节：模型输入、知识检索、缓存命中、工具执行、输出日志和审计记录。因为真实泄露风险经常发生在“主链路看起来隔离了，但缓存或日志没隔离”。工程上比较稳的做法，是让租户标识贯穿整条调用链，并在知识库、缓存键、工具权限和配额系统里都按租户分层。成熟团队会把多租户隔离和权限裁剪一起治理，而不是拆开看。

**高分回答**
我会先按监管、敏感度、租户规模和噪声邻居风险选择共享行级、独立 namespace/索引或独立账户与集群。不是所有租户都要物理隔离，但隔离等级、升级条件、迁移和销毁流程必须明确。高敏租户还可使用独立密钥、网络出口、模型实例与日志存储，降低共享控制面故障的影响面。

tenant_id 由认证服务在入口绑定并签名传递，不能由用户或模型任意指定。它贯穿 Prompt 模板、会话状态、RAG 文档与过滤、缓存键、工具令牌、队列、配额、计费和审计；执行端每次从可信上下文重新鉴权。语义缓存和批处理尤其容易串租，私有内容不因查询相似而跨租户复用，离线任务写入目标也要做租户一致性校验。

隔离验收包含伪造/缺失 tenant_id、缓存命中、并发任务、备份恢复、日志检索、运维账号和租户删除，并将跨租户召回、输出与副作用设为零。运行中看每租户 P99、配额、资源占用和越权告警，既防数据串租，也防单租户拖垮共享服务；发生异常能按租户撤销凭证、隔离流量并回放证据链。

**追问点答案**  
- 为什么 API 层隔离不够？  
因为下游缓存、日志和检索层也可能串租户。  
- 哪些环节最容易漏隔离？  
缓存、日志和共享知识索引。  
- 高敏租户是否需要更强隔离？  
通常需要，可升级到独立索引、账户、密钥、网络或集群；强度应由合规与影响面决定，并预先设计迁移路径。

**原理补充**  
多租户隔离的本质，是任何路径都不应让不属于该租户的数据被感知、调用或复用。

**架构扩展**  
成熟平台通常会给租户隔离做独立审计和压测，因为“理论隔离”不等于“生产上没有串租风险”。

**工程指标**  
- 跨租户误召回率：看核心风险。  
- 缓存串租率：看共享层问题。  
- 审计完整率：看可追溯性。  
- 高敏租户独立资源占比：看隔离强度。  
- 越权访问告警数：看态势。

**证据链**  
- 租户隔离设计图  
- 缓存键规则  
- 审计日志  
- 隔离测试报告

**低风险优化路径**  
1. 先把租户标识贯穿全链路  
2. 再隔离缓存、知识库和工具权限  
3. 最后对高敏租户加更强隔离

**易错点**  
- 只在 API 层做隔离  
- 缓存共享不带租户维度  
- 缺少隔离审计和压测

## 7. 向量库里的敏感数据如何管控？

::: tip L2 深度训练
将向量、元数据、原文、缓存和删除索引纳入同一生命周期：[数据保护与生命周期](/deep-dives/security-compliance/02-data-lifecycle)
:::

**核心结论**  
向量库敏感数据管控要从“入库前分类、入库时最小化、检索时权限裁剪、存储时审计和加密”四个层面一起做。

**标准答案**  
我管向量库里的敏感数据时，通常先判断哪些内容根本不该入库，哪些可以入库但必须脱敏，哪些可以在更强隔离下存储。因为向量库最大的风险之一是大家以为“存的是向量，不是原文”，就低估了它的泄露价值。工程上比较稳的做法，是对入库内容做分类和最小化处理，对 metadata 和原文存储分别做权限和加密，并且在检索时执行严格权限过滤。成熟团队不会把向量库当成普通搜索索引，而会把它视作新的敏感数据承载层。

**高分回答**
Embedding、metadata、原文片段和它们的关联关系都属于派生敏感资产，向量并非匿名化结果。我会在入库前做用途与数据分类：不必要的 PII、密钥和高度受限内容直接排除；需要语义的内容先最小化或令牌化；保留 document_id、版本、租户、ACL、数据等级、来源与删除标记，支持授权检索和生命周期治理。

存储使用传输/静态加密、租户或等级化密钥、网络隔离和最小运维权限；查询在候选生成前执行服务端 ACL 预过滤，TopN 返回后再次鉴权。原文仓与索引可分离，引用通过受控接口按当前身份获取；备份、缓存、离线导出和调试快照使用同一等级控制。Embedding 服务出域前也要满足供应商用途、区域、保留和不训练约束。

更新与删除采用版本化索引和 tombstone：先阻断召回，再清除向量、metadata、原文、缓存与备份到期副本，并保存完成证据。测试成员推断、相似查询探测、ACL 后过滤缺口、备份恢复和删除传播，监控敏感入库拦截率、越权召回、出域量、孤儿向量与删除 P99。若索引不能表达必要权限，高敏数据就不应进入该索引。

**追问点答案**  
- 为什么向量也有敏感性？  
因为它保留语义与成员关系信号，并可通过 document_id、metadata 和近邻结果关联原文，不能视为天然匿名化。
- 什么数据最该避免入向量库？  
高敏个人信息、强监管数据和不必要的机密内容。  
- 只加密存储够吗？  
不够，检索时的权限边界同样关键。

**原理补充**  
管控向量库的本质，是把“可被检索和复用的数据”纳入正式的数据安全治理范围。

**架构扩展**  
成熟平台会把向量库和原文仓分开治理，并对高敏索引做更高等级的密钥、审计和访问审批，这样在检索层出现问题时仍有缓冲。

**工程指标**  
- 敏感数据入库前识别率：看前置治理。  
- 高敏向量索引访问审批覆盖率：看控制强度。  
- 检索权限过滤正确率：看运行时安全。  
- 高敏数据出域次数：看核心风险。  
- 向量库审计完整率：看可追溯性。

**证据链**  
- 入库分类规则  
- 向量库访问日志  
- 权限策略  
- 加密配置

**低风险优化路径**  
1. 先做入库前分类和最小化  
2. 再补检索权限和审计  
3. 最后对高敏数据做强隔离

**易错点**  
- 认为向量不算敏感数据  
- 高敏数据原样入库  
- 检索时不做权限过滤

## 8. RAG 场景中的权限绕过如何防止？

**核心结论**  
RAG 权限绕过防护的关键，是在召回前、召回中和生成前都做权限校验，避免“先看见，再隐藏”的风险模式。

**标准答案**  
我防 RAG 权限绕过时，通常会优先在召回阶段做权限过滤，因为一旦无权限内容进入上下文，后续即使不展示，模型也可能已经利用了它。工程上比较稳的做法，是让文档、chunk、租户和角色元数据全都进入过滤逻辑，并在生成前做二次核验，对高风险答案强制带引用和审计。成熟团队会把权限绕过视为检索层和生成层共同问题，而不是只靠前端展示控制。

**高分回答**
原则是无权证据永远不能进入模型上下文。入口将可信用户身份解析为租户、组、角色和属性，策略服务计算允许资源；检索层使用不可绕过的预过滤，而不是先召回再后过滤。文档和 chunk 携带内容版本与 ACL 版本，复杂继承规则若向量库无法安全表达，就回源授权服务校验或使用独立索引，不能为性能放宽权限。

生成前对 TopN 再次鉴权，引用打开和文件下载也重新校验；查询改写、重排、缓存、会话历史和多跳中间结果保持同一身份边界。撤权事件优先将资源标记不可检索，再异步更新索引与驱逐缓存；过滤后证据不足就拒答，不允许回退到未过滤召回。答案只显示用户有权知道的无答案状态，不能泄露某秘密文档“存在”。

安全测试覆盖直接对象引用、伪造租户/角色、组成员变化、ACL 继承、撤权延迟、缓存复用和多跳绕过，硬门禁是越权证据进入上下文为零。同时衡量误拒率与授权 P99，审计保存策略版本、决策、证据 ID 和检索快照，才能证明当时的回答边界。

**追问点答案**  
- 为什么“先召回后隐藏”危险？  
因为模型可能已经根据不该看的内容生成结论。  
- 生成前复核为什么必要？  
它是纵深校验，用于发现 ACL 版本滞后、过滤实现错误和动态撤权；但不能替代检索前预过滤。
- 什么场景最容易出问题？  
多租户、细粒度角色权限和共享知识库场景。

**原理补充**  
权限绕过问题本质上是“内容访问控制晚于模型消费”导致的失控。

**架构扩展**  
成熟平台通常会对高风险场景启用“强引用 + 权限回放”机制，确保任何答案都能追溯到用户在当时确实有权访问的证据。

**工程指标**  
- 越权召回率：看核心风险。  
- 生成前复核拦截率：看兜底效果。  
- 高风险答案引用覆盖率：看可审计性。  
- 权限变更生效延迟：看一致性。  
- 权限相关事故数：看治理质量。

**证据链**  
- 权限模型  
- 召回日志  
- 审计回放  
- 事故复盘

**低风险优化路径**  
1. 先把权限过滤前置到召回层  
2. 再补生成前二次校验  
3. 最后用强引用和回放闭环

**易错点**  
- 先召回后前端隐藏  
- 权限元数据不完整  
- 高风险场景无引用审计

## 9. 合规日志需要记录哪些字段？

::: tip L2 深度训练
记录检索来源、模型/Prompt、策略、工具参数、审批和副作用结果：[安全事件响应与 AI 边界](/deep-dives/security-compliance/04-security-incident-ai)
:::

**核心结论**  
合规日志要能回答“谁、在什么上下文下、对什么数据、通过什么模型/工具、产生了什么结果”，并支持后续审计和追责。

**标准答案**  
我设计合规日志时，通常会记录调用主体、租户、角色、请求目标、模型版本、Prompt 版本、知识版本、工具调用、权限上下文、关键结果和时间戳。因为 AI 系统的问题常常不是单次调用本身，而是过几周才被发现，这时如果日志不完整就几乎没法追。工程上比较稳的做法，是高风险场景保留更细粒度日志，一般场景保留摘要日志，并统一做脱敏和保留周期管理。成熟团队不会只把合规日志当“存一份访问记录”，而会把它设计成审计证据。

**高分回答**
合规日志要形成不可抵赖的事件链。公共字段包括 event/task/trace_id、时间、主体与服务身份、租户、用途、数据等级和区域；AI 上下文包括模型/参数、Prompt 与策略版本、知识快照和证据 ID；控制字段包括授权策略与结果、内容审核、审批人与参数哈希；执行字段包括工具、目标资源、幂等键、前后差异、状态和成本。原始思维过程不是审计要求，结构化意图与真实事件才可验证。

日志最小化保存：默认记录哈希、引用和脱敏摘要，只有法规或事故取证确需原文时才进入单独加密存储。审计事件追加写、时间同步、完整性保护并设职责分离，访问、导出和删除日志本身也要留痕。保留期按数据类型与法律依据设置，legal hold 通过有审批的例外流程执行，不能无限保留所有 Prompt。

通过 schema 契约和事件对账检查字段完整性，用高风险轨迹定期回放，监控关键事件缺失、落盘延迟、敏感残留、回放成功率和审计查询时长。若审计链路不可用，高风险副作用应暂停或进入保守模式，因为“先执行、以后补日志”无法证明合规。

**追问点答案**  
- 为什么要记录模型和 Prompt 版本？  
因为行为变化经常来自这两者。  
- 为什么日志也要脱敏？  
因为日志本身可能含有高敏输入和输出。  
- 合规日志和普通链路日志一样吗？  
不一样；链路日志服务排障且可采样，合规日志需完整、受保护、按法定期限保留并能证明授权和实际副作用。

**原理补充**  
合规日志的本质，是把 AI 系统的关键行为沉淀成可证明的审计事实。

**架构扩展**  
成熟平台通常会把合规日志、审批流、权限系统和告警系统打通，这样安全、法务和平台团队看的是同一套事实来源。

**工程指标**  
- 合规字段完整率：看证据充分性。  
- 日志脱敏覆盖率：看次生风险。  
- 审计查询成功率：看可用性。  
- 高风险场景日志留存覆盖率：看治理强度。  
- 日志缺失事件数：看风险。

**证据链**  
- 日志 schema  
- 审计查询记录  
- 脱敏规则  
- 保留策略

**低风险优化路径**  
1. 先统一高风险场景日志字段  
2. 再补版本、权限和工具上下文  
3. 最后治理脱敏和保留周期

**易错点**  
- 只记调用时间不记上下文  
- Prompt/模型版本缺失  
- 合规日志本身含敏感原文

## 10. 高风险回答如何做二次确认？

**核心结论**  
高风险回答二次确认的关键，是让模型输出从“直接给结论”变成“给建议 + 给依据 + 等确认”，避免把高代价决策一次性自动闭环。

**标准答案**  
我做高风险回答二次确认时，通常先定义哪些问题属于高风险，比如法律建议、医疗建议、财务建议、删除或发送类操作建议，以及会直接影响用户权益的判断。工程上比较稳的做法，是让系统在识别到高风险场景后，切换到“确认模式”：要求模型先给摘要、证据和风险提示，再由用户或审批人确认下一步动作。成熟团队不会让高风险输出直接变成执行结果，而是把二次确认设计成正式产品流程。

**高分回答**
我会用风险矩阵识别需要确认的场景：潜在伤害、金额与影响人数、不可逆性、监管义务、用户脆弱性和是否产生真实副作用。信息解释、专业建议与执行动作要区分；医疗急症、法律时限或投资交易不能仅靠用户点“确认”把责任转嫁给用户，应提供专业转介、强制人工审核或禁止自动执行。

确认前展示模型结论、关键证据及版本、不确定性、替代方案、预计影响和不可逆后果；执行类任务还展示精确参数、对象、金额和回滚方案。系统冻结 plan_id、参数哈希、数据版本和有效期，确认人必须通过重新认证并具备权限；确认后任何参数变化、超时或环境变化都触发重新评估。不能使用默认勾选、模糊按钮或“确认一切”的批量同意。

确认事件记录提出者、模型/策略版本、证据、确认人、冻结参数和实际结果，并支持撤销尚未执行的计划。评估覆盖率、驳回率、确认后变更拦截、误执行、高风险事故、确认时长和自动化收益；若用户机械确认率异常高，应改善风险呈现或提高独立审批级别，而不是把点击当作有效监督。

**追问点答案**  
- 为什么高风险回答不能直接给结果？  
因为错误代价太高，且用户往往会过度信任系统。  
- 二次确认界面最该展示什么？  
还需展示证据版本、不确定性、替代方案、精确参数、影响范围和回滚能力，并对确认后的参数做哈希冻结。
- 哪些场景特别需要二次确认？  
法律、医疗、财务、隐私和不可逆操作相关场景。

**原理补充**  
二次确认的本质，是把高风险 AI 输出从“自动决策”降级为“受监督建议”。

**架构扩展**  
成熟平台常把二次确认和审批流、审计日志一起做，这样出问题时可以完整回溯“模型说了什么、人确认了什么、最终做了什么”。

**工程指标**  
- 高风险回答二次确认覆盖率：看保护力度。  
- 确认后驳回率：看模型建议质量。  
- 未确认直接执行事件数：看风险。  
- 高风险任务事故数：看真实效果。  
- 确认流程耗时：看体验成本。

**证据链**  
- 高风险分类规则  
- 确认界面设计  
- 审批记录  
- 事故复盘

**低风险优化路径**  
1. 先定义高风险问题清单  
2. 再接确认模式和依据展示  
3. 最后把确认结果纳入审计

**易错点**  
- 高风险建议直接执行  
- 二次确认只有按钮没有依据  
- 高风险分类规则模糊

## 11. 内容安全与可用性冲突怎么平衡？

**核心结论**  
平衡内容安全和可用性的关键，不是选一边，而是按场景分级、按风险分层，让高风险场景更保守、低风险场景更顺滑。

**标准答案**  
我处理内容安全与可用性冲突时，通常会先按业务场景定义容忍度，比如公开社区、企业助手、内部工具、教育产品，这些场景对误杀和漏判的容忍度都不同。因为如果统一使用非常严格的安全策略，很多正常业务会被拦；但如果为了体验放太松，高风险场景又可能出大问题。工程上比较稳的做法，是按场景、用户类型和内容类别分层，结合申诉、人工复核和二次确认做动态平衡。成熟团队不会用一句“安全优先”掩盖产品设计问题，而会把权衡做成明确规则。

**高分回答**
我会把问题转成分场景的代价矩阵：不同风险类别的漏判后果、正常任务误杀损失、用户脆弱性、公开传播范围和可逆性。儿童、医疗急症或真实交易使用更保守阈值；企业内部代码讨论可在权限范围内允许更多专业内容。硬性法律与安全禁区由平台统一，业务只能在允许区间内调节，不能为了转化率关闭底线。

策略结果不应只有允许/拒绝，还可选择安全改写、限制细节、要求澄清、提供可靠来源、转人工或允许但禁止工具执行。拒绝要说明可理解的类别并给安全替代，申诉进入独立复核而不是直接放行。流式场景根据不可逆风险选择句级缓冲或完整审核；高风险输出和动作审核分开，内容合规不代表工具参数可执行。

用按类别和人群分层的正常、边界、对抗样本校准阈值，观察严重度加权漏判、误杀、任务完成率、申诉改判率、人工负担和审核 P99。策略变更通过影子评估与 canary，重大风险为硬门禁。目标不是最大化拦截率，而是在不可突破的安全下限上最大化合法任务完成。

**追问点答案**  
- 为什么不能统一一个安全阈值？  
因为不同业务场景的风险和体验要求不同。  
- 什么是误杀？  
正常内容被错误拦截。  
- 为什么人工复核有价值？  
它适合高后果或模型不确定的边界样本，并能产出校准数据；但审批人需要证据、标准和时限，不能成为无限人工队列。

**原理补充**  
这类平衡本质上是在“风险控制”与“用户任务完成率”之间找业务最优点。

**架构扩展**  
成熟平台通常会把审核策略做成可配置能力，而不是硬编码规则，这样不同业务线能在统一框架下做差异化治理。

**工程指标**  
- 误杀率：看可用性损失。  
- 漏判率：看安全风险。  
- 申诉率：看用户体验。  
- 人工复核通过率：看边界质量。  
- 用户任务完成率：看业务效果。

**证据链**  
- 审核策略配置  
- 误杀/漏判样本  
- 申诉数据  
- 场景分层规则

**低风险优化路径**  
1. 先按场景做风险分层  
2. 再差异化配置策略  
3. 最后通过申诉和复核持续校准

**易错点**  
- 所有业务共用同一阈值  
- 只看安全不看任务完成率  
- 没有申诉和复核出口

## 12. 数据保留与删除策略如何制定？

::: tip L2 深度训练
用用途、保留期、处理方清单、幂等任务和完成证据闭环：[数据保护与生命周期](/deep-dives/security-compliance/02-data-lifecycle)
:::

**核心结论**  
数据保留与删除策略要按数据类型、用途、合规要求和业务必要性分层，明确“为什么留、留多久、谁能删、删到什么程度”。

**标准答案**  
我制定数据保留与删除策略时，通常先区分原始输入、模型输出、日志、审计记录、评测样本和训练数据，因为这些数据的保留目的和合规要求都不一样。工程上比较稳的做法，是给每类数据定义保留周期、删除触发条件、脱敏要求和例外审批，并区分软删除、逻辑删除和物理删除。成熟团队不会把“以后可能有用”当成无限保留理由，而会要求每类保留都有业务和合规依据。

**高分回答**
我会建立数据生命周期矩阵：数据类别、主体与租户、用途/法律依据、数据所有者、存储与处理方、区域、默认保留期、删除触发、备份窗口和 legal hold 规则。Prompt、输出、会话、向量、缓存、评测、训练集、审计日志的用途不同，不能共享一个 TTL；保留期到期即自动处理，“以后可能有用”不是依据。

删除从目录服务生成全局 deletion_id，先阻断在线访问和召回，再以幂等任务传播到主库、对象存储、向量/搜索索引、缓存、评测与训练候选集和第三方处理方。备份通常不能逐条即时改写，应加删除清单，恢复时在重新上线前重放，并在备份生命周期结束后物理消失。已经训练进模型的数据若无法精确删除，要在使用前明确技术边界、隔离后续训练并评估重训或其他补救。

每个处理方返回完成证据，协调器对账未处理副本、重试与死信；legal hold 需有权限、范围、到期和审计，结束后恢复删除。指标包括到期删除率、超期残留、主体请求 P95、第三方确认、备份覆盖和恢复重放成功率。定期从数据地图抽样做端到端删除演练，不能只相信任务状态显示成功。

**追问点答案**  
- 为什么不同数据不能共用一个保留周期？  
因为用途和合规要求差异很大。  
- 删除为什么还要分类型？  
在线阻断、逻辑删除、密码学删除和介质物理清除的时效与保证不同，应按法律依据、恢复需求和备份能力明确承诺。
- 审计日志能随便删吗？  
通常不能，要看监管和追责要求。

**原理补充**  
保留与删除策略的本质，是在“未来价值”和“合规风险”之间建立明确边界。

**架构扩展**  
成熟平台通常会把数据生命周期管理做进底层存储和任务调度，而不是依赖人工清理，否则很快就会积累大量历史风险数据。

**工程指标**  
- 到期数据删除完成率：看执行力。  
- 超期数据残留率：看风险。  
- 删除请求处理时长：看合规响应能力。  
- 物理删除成功率：看强删除能力。  
- 例外保留审批覆盖率：看治理规范性。

**证据链**  
- 数据生命周期清单  
- 删除任务日志  
- 审批记录  
- 合规要求映射表

**低风险优化路径**  
1. 先完成数据分类和用途映射  
2. 再定义分层保留/删除策略  
3. 最后自动化执行和审计

**易错点**  
- 全部数据默认长期保留  
- 删除策略没有自动化执行  
- 不区分业务删除和物理删除

## 13. 训练数据来源合规如何证明？

::: tip L2 深度训练
记录来源、许可、用途、版本、处理链和可删除性证据：[数据保护与生命周期](/deep-dives/security-compliance/02-data-lifecycle)
:::

**核心结论**  
训练数据来源合规证明的关键，是保留来源、授权、用途和处理过程的完整证据链，而不是事后口头说明“数据应该没问题”。

**标准答案**  
我做训练数据来源合规时，通常会要求每批数据都能回答几个问题：从哪来、谁授权、允许做什么、有没有敏感信息、经过了哪些处理。因为一旦这些信息缺失，后续无论是客户审计、法务审查还是内部复盘都会非常被动。工程上比较稳的做法，是建立数据来源清单、授权记录、处理流水和样本抽检结果，把训练数据纳入正式资产治理。成熟团队不会把训练数据当“模型团队自己找的素材”，而会把它当合规资产。

**高分回答**
每个数据集都要有 datasheet 与不可变版本：来源 URI/系统与采集时间、提供方、权利或许可依据、允许用途与地域、主体同意/退出机制、敏感等级、保留期限、限制条款和负责人。公开可访问不等于允许训练，内部工单也不天然可用于新目的；授权范围必须由法务与数据治理按实际场景判断。

处理流水记录原始批次哈希、去重、质量过滤、PII/版权与安全扫描、人工标注供应链、删除清单和最终样本清单，生成 lineage 到训练运行、checkpoint 与发布模型。授权撤销或发现问题时，能定位受影响样本与模型，隔离数据并评估重训、下架或其他补救。合成数据也记录生成模型、种子数据权利与过滤，不能用“合成”掩盖来源问题。

发布门禁要求来源可追溯与授权覆盖达到既定阈值，来源不明和禁止用途样本为零；抽样复核、供应商审计和训练集 membership/canary 检查作为证据。指标包括 lineage 完整率、限制条款违反、删除传播时间、抽检缺陷和受影响模型定位时长。证明合规依赖证据链和适用法律判断，不是模型团队的一句保证。

**追问点答案**  
- 为什么来源证明这么重要？  
因为它直接关系到合法使用边界。  
- 内部数据是不是天然可训练？  
不是；还要验证收集目的、员工/客户告知与授权、合同、保密义务、区域和数据主体权利是否允许新的训练用途。
- 处理流水为什么也要保留？  
因为脱敏、过滤和清洗过程本身也是合规证据。

**原理补充**  
来源合规证明的本质，是把数据合法性从“主观相信”变成“客观可审”。

**架构扩展**  
成熟企业通常会把训练数据治理和数据平台、法务审批、模型版本管理连接起来，确保模型上线时能追溯用过什么数据。

**工程指标**  
- 数据来源可追溯率：看证据完整性。  
- 授权记录覆盖率：看合法性基础。  
- 高敏样本筛除率：看处理质量。  
- 来源不明样本占比：看风险。  
- 审计通过率：看治理成熟度。

**证据链**  
- 数据来源台账  
- 授权和合同记录  
- 处理流水  
- 抽检报告

**低风险优化路径**  
1. 先建立数据来源台账  
2. 再补授权和处理证据  
3. 最后把来源校验纳入上线门禁

**易错点**  
- 训练数据来源不留痕  
- 内部数据未经确认直接用  
- 只留结果不留处理过程

## 14. 第三方模型供应商风险如何评估？

**核心结论**  
第三方模型供应商风险评估要同时看能力、稳定性、合规、安全、价格波动和退出难度，而不是只看模型效果。

**标准答案**  
我评估第三方模型供应商时，通常会拆成几类风险：业务风险、可用性风险、安全与合规风险、成本风险和锁定风险。因为一个供应商今天能力强，不代表明天价格、条款、可用性和合规条件都稳定。工程上比较稳的做法，是做供应商尽调、合同审查、压测验证、隐私评估和替代路线评估，并给核心链路准备备用方案。成熟团队不会只做一次选型，而会持续监控供应商风险变化。

**高分回答**
我会先按拟发送的数据等级和业务关键性给供应商分级，再做尽调。安全维度看身份、加密、隔离、漏洞与事件响应、分包商；隐私合规看数据区域、保留、是否训练、删除与主体请求；工程维度看真实任务质量、限流、SLA、区域故障和版本变更；经营维度看价格、容量承诺、财务稳定、知识产权条款、赔偿与退出成本。

合同要求用途限制、分包商清单与变更通知、事件通知时限、审计权、删除证明、服务与支持目标、价格/模型退役通知和终止后的数据处置。架构上仍执行数据最小化、令牌化、网关出域策略与租户路由，供应商合规声明不能替代自己的控制。核心任务预先验证备用模型、提示/schema 兼容、容量与降级质量；没有等价备用时明确排队或停服边界。

上线后持续监控质量、P99、错误、费用、条款和安全公告，定期重新评估并演练切换与导出。指标包括单一供应商集中度、未批准出域、切换 RTO、备用质量差、账单偏差和整改逾期。双供应商并非强制，只有业务中断风险高于双栈成本时才值得投入。

**追问点答案**  
- 为什么模型效果不是唯一标准？  
因为长期可用性和合规风险同样会决定业务能不能跑下去。  
- 什么是锁定风险？  
包括专有 API、Prompt/tool 行为、数据迁移、评测重做、合同与区域限制，使替换时间和质量损失超出可接受范围。
- 是否一定要双供应商？  
不一定，但关键链路最好有备用预案。

**原理补充**  
供应商风险评估的本质，是判断外部能力是否能在你的业务边界内长期可控地被使用。

**架构扩展**  
成熟平台会把供应商风险和模型路由、成本治理、合规审计打通，而不是把供应商管理只交给采购或法务单独处理。

**工程指标**  
- 供应商可用性：看稳定性。  
- 价格波动幅度：看成本风险。  
- 合规变更频率：看政策风险。  
- 替代路径验证周期：看退出难度。  
- 关键链路单供应商依赖度：看集中风险。

**证据链**  
- 尽调报告  
- 合同和 SLA  
- 压测与演练记录  
- 替代方案评估

**低风险优化路径**  
1. 先完成供应商多维度评估  
2. 再统一接口和备用路径  
3. 最后持续监控条款和稳定性变化

**易错点**  
- 只看模型效果  
- 不评估退出难度  
- 关键链路无替代预案

## 15. 红队测试如何覆盖 AI 特有风险？

::: tip L2 深度训练
覆盖恶意文档、跨租户检索、工具越权、数据外发和预算耗尽：[安全事件响应与 AI 边界](/deep-dives/security-compliance/04-security-incident-ai)
:::

**核心结论**  
AI 红队测试要覆盖提示注入、越狱、越权工具调用、敏感数据泄露、错误高风险建议和多轮诱导，而不是只测传统 Web 攻击。

**标准答案**  
我做 AI 红队测试时，通常会按攻击目标分类设计样本：诱导模型突破系统限制、引导工具执行危险操作、从上下文中套取敏感信息、利用多轮对话逐步绕过边界。因为 AI 风险最特别的地方就在于它既有内容风险，也有执行风险。工程上比较稳的做法，是让红队样本覆盖不同输入源、不同模型版本和不同权限场景，并保留可复现记录和修复后回归。成熟团队会把红队测试做成持续机制，而不是上线前一次性过场。

**高分回答**
红队从威胁模型出发，列出资产、攻击者、信任边界和高价值后果。测试矩阵覆盖直接/间接 Prompt Injection、jailbreak、多轮与编码绕过、RAG 数据投毒和跨租户召回、敏感信息提取、长期记忆污染、工具参数越权、跨工具外传、重复副作用、成本/资源耗尽，以及高风险错误建议。既测模型内容，也测执行端即使模型失守能否阻断攻击链。

样本分通用对抗、业务特定、真实事故回放和自适应探索，覆盖文本、图片、文档、网页、邮件与工具输出。测试在隔离环境、假数据和 dry-run 工具中进行，记录模型/提示/策略/工具版本、随机性、完整轨迹与预期结果，保证可复现。高风险发现按影响面、可利用性和可检测性分级，先撤权或关闭能力，再修复根因。

每个修复样本进入不可删除的安全回归集，同时加入语义变体防止只记住字符串；发布门禁看严重度加权攻击成功率、敏感数据外传、越权副作用、误拦截和检测时长。定期由独立人员复测并演练响应。红队通过只证明已测范围，不能证明系统绝对安全，因此运行时仍需最小权限与监控。

**追问点答案**  
- 为什么 AI 红队不能只测单轮输入？  
因为很多攻击是多轮诱导逐步达成的。  
- 红队测试为什么要覆盖工具调用？  
因为执行风险通常比内容风险后果更大。  
- 红队测试结果怎么用？  
还要先按严重度止损、定位失效控制、指定负责人和期限，并用变体回归验证修复没有只适配单个攻击字符串。

**原理补充**  
AI 红队测试的本质，是系统性寻找模型在真实对抗条件下的边界漏洞。

**架构扩展**  
成熟平台通常会把红队样本分成通用样本、业务特定样本和事故回放样本，这样覆盖面和现实贴近度会更高。

**工程指标**  
- 红队攻击成功率：看暴露面。  
- 修复后回归通过率：看改进质量。  
- 多轮攻击命中率：看会话风险。  
- 工具越权样本拦截率：看执行防护。  
- 高风险样本覆盖率：看测试完整度。

**证据链**  
- 红队样本库  
- 测试报告  
- 修复记录  
- 回归结果

**低风险优化路径**  
1. 先建立高风险攻击样本库  
2. 再定期红队和修复闭环  
3. 最后把典型样本纳入常规回归

**易错点**  
- 红队只做一次性检查  
- 不覆盖工具和多轮对话  
- 修复后不纳入回归集

## 16. 安全基线如何自动化校验？

**核心结论**  
AI 安全基线自动化校验要覆盖配置、权限、日志、数据流和模型调用策略，避免团队靠人工记忆维持安全标准。

**标准答案**  
我做安全基线自动化时，通常会先把必须满足的安全条件清单化，比如是否开启日志脱敏、是否限制高风险工具、是否有租户隔离、是否保留审计字段、是否启用审批流。因为如果这些检查全靠人记，很快就会在多团队和多版本演进里失效。工程上比较稳的做法，是把基线写成 CI 检查、配置扫描、运行时告警和发布门禁。成熟团队不会把基线当静态文档，而会把它做成自动执行的守门机制。

**高分回答**
我会把基线按资产与生命周期写成 policy-as-code：模型/提示版本、工具 schema 与权限、数据出域和保留、RAG ACL、租户缓存、日志脱敏、审批、预算和 kill switch 都有机器可判定规则。规则包含适用范围、严重度、负责人、修复建议和证据要求；硬性合规与越权项不可由业务自行豁免，风险例外必须有审批、补偿控制和到期时间。

控制点分三层：CI 扫描配置、IaC、schema 和测试清单；发布门禁运行安全黄金集、注入/越权回归和权限契约；运行时持续检测配置漂移、异常出域、常驻高权令牌和审计缺失。仅做静态扫描无法证明动态授权和模型行为，需合成租户、canary 数据与 dry-run 工具进行端到端验证。

规则本身版本化、经过单元测试与影子运行，观察覆盖率、真正例/误报、违规修复时间、逾期例外、运行漂移和重大漏检。自动修复只用于明确可逆配置，高风险问题默认阻断并提供诊断。基线要连接资产清单与负责人，否则告警无人处理仍等于没有控制。

**追问点答案**  
- 为什么安全基线不能只写文档？  
因为文档无法保证每次发布都被执行。  
- 哪些检查最适合自动化？  
配置项、权限开关、日志字段、版本依赖等。  
- 自动化能替代人工审计吗？  
不能；自动化验证可表达的不变量，威胁建模、用途合法性、复杂业务风险和新攻击仍需人工判断与独立审计。

**原理补充**  
安全基线自动化的本质，是把安全要求转成机器可验证规则。

**架构扩展**  
成熟平台通常会把基线自动化嵌到模型发布、Prompt 变更、工具接入和知识库更新等关键节点，而不是只在系统初始化时检查一次。

**工程指标**  
- 基线自动校验覆盖率：看治理覆盖度。  
- 基线违规拦截率：看门禁效果。  
- 人工补救工单数：看自动化价值。  
- 误报率：看规则质量。  
- 发布前基线通过率：看整体健康度。

**证据链**  
- 基线规则库  
- CI/发布门禁记录  
- 告警日志  
- 违规整改记录

**低风险优化路径**  
1. 先把高频基础安全项规则化  
2. 再接入 CI 和发布门禁  
3. 最后补运行时巡检和告警

**易错点**  
- 基线只有文档没有自动检查  
- 检查只在上线前做一次  
- 不跟踪误报和漏报质量

## 17. 安全事件响应流程如何设计？

::: tip L2 深度训练
把安全硬门槛、能力关闭、审计取证和失败样本回归串联：[AI 评测与发布门禁](/deep-dives/ai-architecture/04-evaluation-release-gates)
:::

**核心结论**  
AI 安全事件响应要比传统故障响应多一层“模型与数据行为分析”，关键是快速止损、保留证据、隔离影响范围并持续回归。

**标准答案**  
我设计 AI 安全事件响应时，通常会先定义事件类型，比如数据泄露、越权工具执行、模型越狱、高风险错误建议、租户串扰。因为不同事件的止损动作不同。工程上比较稳的做法，是一旦触发事件，先暂停相关模型或工具能力、切换降级路径、冻结高风险日志和审计证据，再由安全、平台和业务一起做根因分析。成熟团队不会把 AI 安全事件当普通线上 bug，而会把模型行为、Prompt 版本、知识版本和权限状态一起纳入调查。

**高分回答**
响应预案先按数据泄露、跨租户访问、越权副作用、模型/知识投毒、危险输出和供应链事件定义严重度、值班角色、通知链与止损手册。发现后优先收缩能力：撤销令牌和工具、隔离租户/索引、关闭受影响版本、阻断出域、切到安全降级或暂停高风险流量；同时保存不可变审计、模型/提示/策略/知识版本、检索与工具轨迹，避免调查过程中破坏证据。

随后确定时间窗、受影响主体、数据类别、模型与派生副本、真实副作用和供应商范围。未知时按较高严重度处理，由安全负责事件指挥，平台执行技术隔离，业务评估用户影响，法务/隐私判断通知义务，供应商按合同协同。恢复必须通过定向修复、安全回归、权限复核和小流量 canary，不能只因错误率下降就重新开放。

事后完成根因与失效控制分析、用户/监管沟通、删除或补偿、样本回归和负责人明确的整改。定期演练注入导致外传、跨租户 RAG、审批绕过和供应商泄露，衡量发现时间、遏制时间、影响面定位、证据完整率、恢复时间和重复事件。Kill switch、联系人与取证路径必须在事故前验证。

**追问点答案**  
- 为什么止损优先级高于根因分析？  
因为持续放大的损失通常比误停机更危险。  
- 为什么日志和版本证据要优先冻结？  
因为后续复盘和追责都依赖这些证据。  
- AI 事件和普通接口故障最大区别是什么？  
还可能跨模型、提示、知识、工具和供应商形成概率性攻击链，影响范围必须从轨迹与版本中重建，而非只查代码异常。

**原理补充**  
安全事件响应的本质，是在不确定条件下迅速收缩风险，并为后续追责和修复保留完整证据。

**架构扩展**  
成熟平台通常会把 AI 安全事件演练纳入常规稳定性演练，因为没有演练过的响应流程到了真实事件时通常会非常慢。

**工程指标**  
- 安全事件发现到止损时长：看响应速度。  
- 证据保全完整率：看追溯能力。  
- 事件影响范围：看隔离效果。  
- 修复后回归通过率：看闭环质量。  
- 重复事件率：看治理成效。

**证据链**  
- 响应预案  
- 审计与版本日志  
- 事件工单  
- 复盘报告

**低风险优化路径**  
1. 先定义 AI 事件类型和止损手册  
2. 再建立证据保全和版本回放能力  
3. 最后定期演练和回归

**易错点**  
- 事件响应只关注业务恢复不关注证据  
- 不区分 AI 事件类型  
- 修复后不做回归验证

## 18. 如何做 AI 系统的最小权限原则？

::: tip L2 深度训练
按租户、工具、对象、动作和额度构建可审计权限契约：[Agent 生产化与工具安全](/deep-dives/ai-architecture/02-agent-production-safety)
:::

**核心结论**  
AI 系统的最小权限原则，就是默认不给模型、工具、知识库和租户链路多余能力，只按任务必要范围临时授予。

**标准答案**  
我做最小权限时，通常会从三个层次看：模型能看到什么、工具能做什么、租户和用户能触达什么。因为 AI 系统权限面往往比普通系统更大，它会同时涉及数据、工具和执行。工程上比较稳的做法，是按任务模板配置默认最小权限，对高风险路径采用短时授权和审批，并把所有权限变更纳入审计。成熟团队不会把最小权限只理解成账号权限，而是把它扩展到 Prompt、知识、工具和执行边界。

**高分回答**
我会从身份、数据、能力和时间四维定义权限契约：谁代表哪个用户/租户，能读取哪些知识和字段，能调用哪个工具对哪些对象执行什么动作，以及额度、次数和有效期。任务模板默认只有完成该任务所需的只读能力；需要写入时由可信策略服务签发短时、可撤销令牌，模型不持有长期凭证，也不能自行扩权。

授权在每个资源和工具执行端强制，tenant_id、目标资源和敏感参数来自可信上下文或再次校验。高风险动作采用 dry-run、参数冻结、职责分离审批和幂等；工具结果按字段最小化，网络出口和跨工具数据流同样受控。会话、缓存、长期记忆、日志和异步任务继承原授权范围，权限撤销事件立即驱逐相关状态。

权限使用可观测：记录授予理由、策略版本、实际使用和撤销，定期基于真实使用清理闲置能力。测试越权对象、参数替换、凭证过期、跨租户缓存和组合工具提权，要求越权副作用为零，并监控常驻高权身份、权限未使用率、撤权 P99、误拒率和审批负担。最小权限是降低爆炸半径，不是依靠用户每一步机械确认。

**追问点答案**  
- 为什么 AI 最小权限比普通系统更复杂？  
因为它同时控制内容访问和动作执行。  
- 短时授权为什么重要？  
因为它能减少误用和泄露窗口。  
- 默认最小权限会不会影响体验？  
会，因此低风险只读能力可预授权，高风险才逐步升级；通过任务模板、短时授权和清晰审批减少摩擦，而不是常驻大权限。

**原理补充**  
最小权限的本质，是把错误或攻击发生时的最大损失压缩到最小范围。

**架构扩展**  
成熟平台通常会让不同 Agent 模式、不同用户角色和不同任务模板自动继承不同权限配置，而不是所有人和所有场景共用一套大权限。

**工程指标**  
- 默认低权限任务覆盖率：看设计成熟度。  
- 临时授权使用率：看敏感操作收口情况。  
- 越权被拦截次数：看防护效果。  
- 权限变更审计完整率：看治理能力。  
- 高权限常驻账号比例：看风险面。

**证据链**  
- 权限策略表  
- 授权记录  
- 审计日志  
- 高风险操作清单

**低风险优化路径**  
1. 先按任务模板做默认最小权限  
2. 再引入短时授权  
3. 最后持续清理常驻高权限能力

**易错点**  
- 一套大权限覆盖所有场景  
- 权限变更无审计  
- 默认开放再慢慢收紧

## 19. jailbreak 检测与防护策略有哪些？

**核心结论**  
jailbreak 防护不能只靠关键词黑名单，应该结合输入分类、上下文约束、输出审核、权限限制和对抗样本回归多层治理。

**标准答案**  
我做 jailbreak 防护时，通常会把它看成持续对抗问题，而不是一次规则配置就能结束。因为用户会不断变换表达方式绕过简单拦截。工程上比较稳的做法，是在输入侧做风险分类和异常模式识别，在模型侧做边界约束，在输出侧做高风险内容审核，同时限制高风险工具和动作的执行权限。成熟团队会把 jailbreak 样本不断回流到回归集里，让防护体系随攻击手法一起进化。

**高分回答**
Jailbreak 目标是让模型违背内容或行为策略，Prompt Injection 则常利用外部数据劫持指令；生产中两者会组合，所以需要统一威胁模型。输入侧用规则、分类器和对话级信号识别角色扮演、编码、渐进诱导和已知攻击家族，但不能因命中就把所有类似正常请求永久拉黑。系统提示与策略做版本控制和最小暴露，输出侧按风险类别审核关键主张与工具意图。

真正的止损仍在模型外：高风险工具最小权限、参数校验、数据流控制、审批和执行后验证。即使内容审核漏判，模型也不能读取秘密或产生越权副作用。拒答使用安全且有帮助的替代路径，避免过度误杀；多轮会话中对风险累积和上下文污染重新评估，不能只检查最后一句。

持续红队生成语义变体、多语言、编码、多模态和长上下文攻击，按场景统计严重度加权成功率、漏判、误杀、工具攻击链完成率和用户任务完成率。新样本进入隔离的安全回归集，防止评测泄漏到提示；防护升级影子验证并可快速回滚。目标不是声称“检测所有 jailbreak”，而是降低成功概率并把残余影响限制到可接受范围。

**追问点答案**  
- 为什么黑名单不够？  
因为攻击可用语义变体、多轮、编码和多模态表达；黑名单只能作为高精度信号之一，还会误伤正常研究与安全讨论。
- 输入和输出都要做检测吗？  
要，输入防尝试，输出防漏网。  
- 为什么回归集必须持续更新？  
因为攻击方式会不断演化。

**原理补充**  
Jailbreak 本质上是利用模型对上下文边界的脆弱性，诱导其越过原本限制。

**架构扩展**  
成熟平台通常会把 jailbreak 检测和 Prompt Injection 一起治理，因为两者在真实系统里常常混合出现，并最终影响工具调用和高风险内容输出。

**工程指标**  
- jailbreak 成功率：看核心风险。  
- 输入检测命中率：看前置防护。  
- 输出漏判率：看后置防护。  
- 对抗样本回归通过率：看演进能力。  
- 高风险执行拦截率：看最终止损。

**证据链**  
- jailbreak 样本库  
- 检测规则/模型  
- 审核日志  
- 回归报告

**低风险优化路径**  
1. 先建立高风险 jailbreak 样本库  
2. 再做输入和输出双层防护  
3. 最后持续回流和演进检测体系

**易错点**  
- 只靠黑名单  
- 只有输入检测没有输出审核  
- 攻击样本不进入回归

## 20. 如何建设 AI 治理委员会与机制？

**核心结论**  
AI 治理委员会的价值，不是多一个审批组织，而是把模型风险、数据合规、业务价值和技术治理拉到同一张桌子上持续决策。

**标准答案**  
我建设 AI 治理委员会时，通常会把它定位成跨职能治理机制，而不是单纯安全委员会。因为 AI 决策往往同时涉及业务、法务、安全、平台、数据和运营，不是单个部门能独立拍板。工程上比较稳的做法，是定义清楚治理范围，比如高风险场景准入、模型上线门槛、供应商评估、数据使用审批和重大事件复盘，再明确谁负责什么、哪些事项必须升级到委员会处理。成熟组织不会让委员会包办所有事情，而是只处理高风险和高影响决策，日常执行仍由平台和业务流程承接。

**高分回答**
委员会需要正式 charter：目标、适用范围、风险分级、成员与表决/升级规则、会议节奏、证据要求和问责机制。成员至少覆盖业务所有者、AI/平台、数据、信息安全、隐私/法务、合规、运营与必要的领域专家；董事会或高管承担最终风险责任。委员会处理新高风险用途、重大例外、关键供应商、严重事件和政策变更，低风险标准项目由授权流程快速自助，避免成为审批瓶颈。

每个 AI 系统进入资产台账，指定业务与技术负责人，完成用途、数据流、模型/供应商、风险评估、评测、监控、人工监督和退出计划。治理决策必须落到 policy-as-code、发布门禁、权限模板、审计、预算和事件响应；例外有补偿控制、负责人和到期复审。模型、数据用途或影响人群发生重大变化时重新评估，而非一次审批永久有效。

委员会按季度查看高风险覆盖、评测/监控缺口、逾期例外、事件与近失、申诉、供应商集中度、价值与成本，并发布可审计决策记录。衡量决策落地率、风险整改周期、重复事件、低风险审批时长和被淘汰的无价值项目。治理成功不是会议数量，而是风险边界明确、控制可执行、责任可追溯且业务能在安全范围内更快上线。

**追问点答案**  
- 为什么 AI 治理不能只交给技术团队？  
因为很多问题同时涉及法务、业务和组织风险。  
- 委员会为什么不能管所有小事？  
因为会变成低效审批中心。  
- 治理机制最重要的输出是什么？  
还要把它们落实为资产台账、风险分级、机器门禁、例外到期、持续监控和可审计决策，而不是停留在会议纪要。

**原理补充**  
治理委员会的本质，是让高影响 AI 决策从局部最优变成组织层面的平衡决策。

**架构扩展**  
成熟企业通常会让委员会与平台规则、审计能力、发布门禁和事件响应联动，这样治理不只是会议决议，而能真正落到系统控制点。

**工程指标**  
- 高风险项目治理覆盖率：看机制作用范围。  
- 治理决策落地完成率：看执行力。  
- 重大事件复盘闭环率：看成熟度。  
- 供应商/数据审批周期：看效率。  
- AI 治理例外申请数量：看规则适配度。

**证据链**  
- 委员会 charter  
- 决策记录  
- 风险评审结果  
- 复盘和整改报告

**低风险优化路径**  
1. 先明确治理范围和角色  
2. 再定义高风险事项准入机制  
3. 最后把决策连接到平台控制点

**易错点**  
- 治理委员会只开会不落地  
- 所有事项都升级审批  
- 技术、法务、业务职责边界不清
